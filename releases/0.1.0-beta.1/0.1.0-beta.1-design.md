# Memoria 0.1.0-beta.1 — MVP system design (query/data substrate + WRITE/output)

Date: 2026-07-05
Status: beta.1 MVP design draft — implementation may start on unblocked
slices; §8.1 freeze blockers must be resolved before beta.1 release freeze.

> **North star:** Memoria is an opinionated, single-person knowledge-production
> tool — an AI-enhanced Zettelkasten utilizing the LLM-wiki framework.
> **Mission:** help the user **read** (accumulate / compile / curate knowledge)
> and **write** (compose / create new knowledge), with **code and text as the
> main outputs.** Writing and coding are the *output half* of the mission, not
> peripheral.

This design **implements** the governing
[0.1.0-beta.1-requirements.md](./0.1.0-beta.1-requirements.md) (MVP requirements,
evidence, technical decisions, acceptance, open questions) — *what must be true*
vs *how*.

**What this document is.** beta.1 is the first MVP that delivers Memoria's full
promise: the point where system functionality is *done* enough that a real
researcher can use it on a real corpus and the build-measure-learn loop begins.
The MVP is defined by **no deferred core**:
to defer an item you must *prove the user does not need it* for viable,
feedback-producing use — "future consumer / post-beta / hard to build" are
excuses, not proofs. **beta.1 = alpha.16's READ product flow + the query/data
substrate changes required for that flow + the WRITE/output additions required
for the MVP.** alpha.16 already ships the READ cycle and the existing
knowledge-production substrate ([alpha.16 design](../0.1.0-alpha.16/0.1.0-alpha.16-design.md));
this document specifies the beta.1 substrate changes and WRITE/output half, and
does not re-specify unchanged alpha.16 behavior.

*Facts-only grounding: every decision rests on a research finding, a documented
failure, or a fact stated in an active top-level release file — never
ADR-authority, prior-note authority, or a `resources/` file. The load-bearing
sources for this design are this file's primary-source links, the governing
requirements, the decision ledger, and the two top-level analysis files. Legacy
tags such as RR / ADV / B / A / owner are traceability labels only; implementers
must not treat them as justification unless the same claim is stated in an
active source above. Anything not settled by those facts is an explicit OPEN
RESEARCH QUESTION (§8), never an invented mechanism. For data structure and
query mechanics, [`data-structure-analysis.md`](./data-structure-analysis.md)
and [`query-mechanism-analysis.md`](./query-mechanism-analysis.md) govern this
design where they are more specific than the inherited alpha.16 wording.*

## 0. Relationship to alpha.16 (inherits the READ flow; implements beta.1 substrate additions)

beta.1 inherits alpha.16's READ product flow and current vault/operation
substrate. It does **not** treat the beta.1 query/data substrate as already
implemented: `passages`, FTS5, `sqlite-vec`, `concept_edges`, query-time
reindex, and same-transaction `check_status` cascades are beta.1 implementation
work (§8.4, §12.2). From alpha.16, load-bearing and unchanged here:

- The **two-graph vault** — catalog (SQLite, relational, objective) + ZK
  markdown notes/claims/hubs with **typed argument edges** (`supports` /
  `contradicts` / `extends`) forming the subjective knowledge graph, bridged by
  a `note` with `mode: work` after the delete-and-fold
  (`data-structure-analysis.md` Part 8).
- **Capture / extract / digest**, **retrieval / Ask on the inherited BM25
  baseline**, **gap analysis over both graphs** and the gap between them,
  integrity / provenance / **snapshot→apply→verify→commit + revert**, catalog /
  enrichment, security, cost. beta.1 then implements/replaces the query
  substrate with `passages` / FTS5 / `sqlite-vec` / `concept_edges`, RRF/MMR,
  and **verbatim passages** as the stored/retrieved unit (§12.2).

**The flawed beta.1's core error was deferring exactly this substrate** — the
argument graph, hubs, claims-with-context, gap-over-both-graphs — and treating
writing and code as afterthoughts. That deferral is not research-justified; the
requirements and technical-preservation decisions make the alpha.16 READ
product substrate inherited scope, not optional beta.1 product work. beta.1
still implements the query/data substrate changes named above, then adds the
surface that turns the READ flow into **text and code output**.

**The guardrails carry over unchanged and remain load-bearing in every section
below** — no synthetic index with a threshold; show the reasoning, not just the
verdict; propose-not-dispose + anti-anchoring; **no complementarity
assumption**; no-write-time-oracle; `checked` ≠ approved; deterministic
grounding for every number and citation;
graph *structural* integrity over single-node semantic validity. Every addition
in §§1–6 is subordinate to these requirements.

The MVP additions, and nothing else: beta.1 query/data substrate (§12.1–§12.2),
§1 writing module (→ **text** draft), §2 coding module (→ **code** artifact),
§3 compose/deep-work workspace, §4 output verification, §5 onboarding & the
usable bar, §6 feedback instrumentation — each consuming the inherited alpha.16
flow without re-specifying unchanged alpha.16 behavior.

## 1. The writing module

The requirements' named flow, made concrete: **argument graph → knowledge map /
canvas → outline → draft (TEXT output)**. Each stage below is a *proposal
surface*: the system proposes with its reasoning shown; the human disposes.

### 1.1 Argument graph (consumed, not built here)

The writing module reads the alpha.16 argument graph: **claims-with-context**
carrying TEMPO fields — `temporal_scope`, `tense`, `qualifier`/certainty,
originating section — because a claim's truth is determinate only relative to
these. Claims are connected by human-authored typed edges, with **saturation**
expressed as *honest coverage reasoning* — which claims lack a support, which
lack a counterpoint — **never a synthetic saturation score or band**. This is
the substrate the thread is drawn from.

### 1.2 Knowledge map / canvas — the ZK-like sketch connecting notes into a thread

The **canvas** is the project-scoped spatial sketch that connects the notes and
claims relevant to the thesis into a *thread*. Design, grounded:

- **Small-graph, analyst-only, and it must beat a tabular baseline.** The
  requirements scope canvas to the *project slice* of the argument graph, not
  the whole vault. It is offered alongside — never instead of — the
  faceted/tabular view, with the canvas-vs-table threshold measured in §8.2.
- **Positions persist; they are not regenerated.** "Disposable is safe for
  *data*, hazardous for spatial *layout*" — canvas node positions are governed,
  persisted state, reversible via the inherited snapshot/revert protocol.
- **Propose-not-dispose.** The system proposes *which* notes/claims belong on the
  canvas and *how* they connect (with the reasoning for each), drawing the
  relevant argument-graph slice; the human arranges the thread. Candidate
  surfacing uses the beta.1 query substrate: lexical search, first-class
  embeddings, structural predicates, and bounded local/global graph expansion
  over `work_graph_edges` and `concept_edges`, fused by RRF. Graph traversal is
  retrieval over checked/PI-confirmed structure only; it never creates edges or
  judges truth. Dense/graph activation as the default path remains fixture-gated
  against the BM25 baseline (`query-mechanism-analysis.md` Q3/Q8).
- Format: a plain `.canvas` (JSON-Canvas) file so it is editable and inspectable
  without the plugin (the keep-test, §3).

### 1.3 Outline — derived from the canvas

**DECISION: the outline is a one-time, re-runnable derivation from the canvas —
NOT a live two-way sync.**

- Direction is fixed by the beta.1 loop: canvas/thread → outline,
  one-directional.
- Live two-way reconciliation between a spatial graph and a linear document is
  unbounded machinery with no documented need; the **simplest-design guardrail**
  rejects it until a requirement proves it. The canvas is the source for outline
  derivation only; it is not the authority for the graph. The outline is
  *derived data*.
- Once derived, the outline is a **human-owned artifact**: edits stay in the
  outline and do not propagate back. Re-derivation is an **explicit, opt-in**
  action that supersedes the prior outline as a new version (reversible via
  snapshot/revert), warning that it discards outline-local edits.
- OPEN (§8): whether heavy outline editing creates real demand for
  outline→canvas back-propagation — to be *measured* (does the user re-derive,
  or diverge and never re-derive?), not assumed.

### 1.4 Draft — the TEXT output

The draft is composed section by section from the outline. It is the **output
artifact**, not a write into the knowledge graph. Grounding is dense here
because synthesis is where the research bites hardest:

- **Feed coherent slices, not top-k fragments.** Each outline node maps to its
  canvas notes/claims; synthesis **expands** them to coherent slices — whole
  distilled notes plus section-expanded anchors — before composing. This is a
  governing beta.1 requirement, and §11.2 adds primary-source support for the
  whole-note-if-budget else section-expanded-anchor policy.
- **Every number computed in code; every citation resolved deterministically.**
  A printed number is computed by the engine's code path, never by the model.
  Citations resolve against deterministic catalog/bibliography metadata to a
  **verbatim span**.
- **Warrant = an evidence-*set*, typed.** Each drafted claim carries its
  evidence set with type single-sentence / multi-sentence / multi-hop /
  **computed** / **implicit**. Computed claims have **no single source-span
  warrant**; they are complete when the referenced code artifact and output hash
  resolve and recompute, and are evidence-incomplete only when that code warrant
  is missing, unresolved, or failing. A span match is relabeled
  **citation-correct, not grounded**.
- **The evidence-set contract (minimal, durable).** A drafted claim's evidence
  set is an ordered list of **items**, each either a **source-span ref**
  (`work_id#^pNNNN`, or an existing `ev-<8hex>` pinned-evidence id) or a
  **code-warrant ref** (`code-artifact-id` + `output-sha256`, §2.1). The engine
  **derives** the `type` from the items — *single-span* (one span), *multi-span*
  (>1 span, one work), *multi-hop* (>1 works), *computed* (code-warrant, no
  span), *implicit* (no resolvable warrant) — and never asserts it. The `state`
  is *complete*, or *evidence-incomplete* iff the type is implicit, any source
  item fails to resolve, or a code-warrant item fails to resolve/recompute;
  incomplete routes to the PI (§4). Durable form: a trailing
  inline marker on the claim block (`%%ev: <id> type=… state=…%%`, the same
  pattern as `%%prov%%`) **plus** an `evidence_sets` DB row (id, block_ref,
  items_json, type, state, review_required, run_id) — inline for portability,
  DB for query/rebuild. `work_id#^pNNNN` is the stable reference syntax; it
  resolves through `passages.anchor` page metadata, not by requiring literal
  `^pNNNN` block markers inside `fulltexts/<work_id>.md`. State machine:
  (re)computed on draft/edit from its items;
  verification (§4.1) re-resolves each item and flips a broken set to incomplete.
  This is the smallest contract every drafted claim can carry, and it is what
  §4.1's citation/number checks and §8 Q2's routing measurement run on.
- **Abstain rather than fabricate.** Where the corpus does not support a passage,
  the draft emits an explicit gap/"Unknown" marker; faithfulness comes from
  deterministic grounding, not from the model *choosing* to say it doesn't know.
- **Prose, not a forced schema.** §11.4 names negation as a specific
  verification risk and supports the prose-plus-typed-evidence-set shape. The
  draft is natural-language output; structured extraction stays in the evidence
  and verification layers.
- **Propose-not-dispose.** The draft is a proposal with per-passage reasoning
  (which notes/claims/evidence back it); the human composes and judges. Drafting
  **does not auto-write knowledge back into the graph** (that violates
  no-write-time-oracle); promoting a draft insight into a permanent note is the
  same sparse, uncertainty-routed PI gate (§4); its workflow is **§1.5**.

### 1.5 Draft → knowledge-graph write-back

Drafting is a *producer* of notes, not only a consumer: it surfaces insights
that should become permanent notes (Zettelkasten/Matuschak — a manuscript is the
rearrangement of notes, and writing reveals notes that don't yet exist). The
write-back is a **PI-gated action, never automatic** (no-write-time-oracle):

- **Extract to an atomic, concept-oriented permanent note.** A first-class
  in-draft action turns a selected passage into a new `note` (claim/question) in
  the graph — atomic and concept-oriented, so it detaches from any single source
  (Matuschak evergreen notes).
- **Transclusion, not copy.** The draft keeps a **back-link / transclusion** to
  the new note — one canonical instance — never a pasted copy that silently
  diverges (single source, correct attribution).
- **Provenance + maturity.** The note carries a link to its source record (not
  inlined source text) and a **maturity marker** (`seedling`), distinguishing a
  raw draft-spawned extraction from a vetted permanent note.
- **Squeeze-point trigger.** The system *offers* extraction when a draft section
  accumulates many loose references (the "mental squeeze point") — an
  event-driven prompt, never an auto-write; the human decides.

Promoted notes re-enter gap analysis like any note, closing the post-draft loop.
**OPEN residual (§8 Q9):** the squeeze-point detection
threshold, and whether draft-spawned `seedling` notes need a distinct review
lane before they count as compose evidence.

### 1.6 Close the loop

Drafting surfaces new gaps — claims that abstained, passages with
evidence-incomplete warrants, sections with no counterpoint. These feed back
into alpha.16's **gap analysis over both graphs**, completing the beta.1 loop
(project → … → develop notes/claims → **re-run gap analysis**). Writing is thus
not a terminal export but a driver of the next READ iteration.

## 2. The coding module

Code is a **first-class project output** — the other half of "code and text as
the main outputs" — not the inert artifact the flawed beta.1 made it. Evidence
specific to code generation is thin in the corpus, so this section specifies
only what the facts support and pushes the rest to §8 rather than inventing
mechanism.

### 2.1 Two grounded roles for code

1. **Code as executable warrant.** Computed claims can have no single source
   span. Code fills that gap: a computed figure is warranted by the
   **re-runnable script that produced it**. This promotes "every number computed
   in code" to a first-class artifact — the code is the deterministic warrant
   for the class of claims the text warrant model cannot cover. A claim's
   evidence set (§1.4) may therefore point to a **code artifact + its output**,
   not only a source span.
2. **Code as deliverable.** A script, analysis, or tool the researcher ships as
   an output in its own right.

### 2.2 How code relates to the project and its claims

- Code artifacts are **project-scoped**, stored as plain files, and versioned
  through the inherited **single VaultWriter + snapshot→apply→verify→commit +
  one-click revert**. Each non-markdown code file is referenced from a typed
  markdown project/draft record; the source file is a Memoria companion
  artifact, not an OKF Concept document. Reversibility and provenance are
  unchanged from alpha.16.
- The link to claims runs through the **evidence-set** mechanism (§1.4): a
  computed claim references the code that computes it; regenerating the number
  re-runs the code (§4).
- **Propose-not-dispose applies**, with a sharp caveat: code review is a genuine
  bottleneck. The beta.1 answer is per-item verification tooling (diff,
  run-and-observe, tests) and triage — not throughput (§4).

### 2.3 What is deliberately NOT designed here

The full code-authoring workflow (a canvas/outline analog for code; language and
runtime scope; the code lane's agent posture and skills) is **not settled by
evidence** and is enumerated in §8. Most critically, **executing generated code
reopens the exec/exfiltration surface**: AgentDojo's primary finding is that
tool-using agents over untrusted data are vulnerable to indirect prompt
injection, so Memoria needs a real execution boundary, not just a prompt rule
([AgentDojo](https://arxiv.org/abs/2406.13352)).

**The minimal execution sandbox (committed).** Because code is a first-class MVP
output, execution cannot be left unresolved — acceptance depends on it. beta.1
commits the *minimal* sandbox as an enforcement contract, not a label: a code
run executes in the **credential-free engine-worker context**, with network
denied by the sandbox backend itself, not merely by lack of Guard access. The
Linux/WSL backend uses OS-level network namespace isolation through `bubblewrap`
(`--unshare-net`, or an equivalent verified invocation); any macOS or other
backend is not acceptance-eligible for code execution until it provides an
equivalent no-network boundary. The filesystem is scoped to the project's code
path plus a temp working dir (no vault access outside the project) plus an
explicit **secret read-denylist** — `~/.ssh`, `~/.aws`, the OS keychain,
provider-key files, and the derived DB — because scoping *writes* to the project
is **not** enough: a scoped-write + broad-read sandbox still lets an injected run
exfiltrate secrets by *reading* them. It runs **only user-approved commands**
per artifact, under a **wall-clock + memory + output cap** that kills on breach,
and staged so a failed or killed run **fails cleanly and never half-writes**
(§4.2). Hardening beyond this minimum (per-OS backend selection beyond
Linux/WSL, finer syscall/container isolation, per-language runtimes, adversarial
validation) is the open item (§8 Q1); code execution is part of acceptance only
on platforms with the verified no-network backend.

## 3. The compose / deep-work workspace

The deep-vs-task split is a beta.1 requirement: the compose / deep-work surface
for writing and coding stays distinct from the task-oriented curation surfaces
(inbox, gap review, link review, catalog health) inherited from alpha.16.

**The workspace is specified as five UI-framework-independent jobs** — *what* the
researcher needs, not *how* a plugin renders it, so beta.1 can commit the jobs
while the gate topology and host stay open (§8, Q11): (1) **Read** — source
queue, reading position, annotations, digest; (2) **Knowledge** —
claims/questions/tensions, hubs, graph integrity; (3) **Project** — question,
source scope, gap report, canvas, outline, draft, code; (4) **Review** — machine
writes, conflicts, export refusals, integrity incidents; (5) **Upkeep** — sync,
cost, backups, migrations, provider failures. The deep-work composition surface
below is the *Project* job; the others are inherited from alpha.16.

- **What it hosts:** the deep-work project artifacts — the **canvas, outline,
  draft, and code artifacts** — plus the co-PI as the active-assistance surface.
  It is where the human arranges the thread and composes; it deliberately does
  **not** surface the
  inbox/recommendation stream, which lives on the task-oriented gates
  (mindset separation).
- **Constraint — the keep-test holds.** The plugin must **not** be the only way
  in: the product must run with `vim`/`nano`/a file browser. Every deep-work
  artifact is a plain file (`.md` draft, `.canvas`, source files); the workspace
  is a convenience layer over files, never a gate on them. This also preserves
  the **no-daemon** posture.
- **What is grounded vs open.** The *deep-vs-task boundary* is committed for
  beta.1. The full **gate/workspace model** is **not**: asking, reviewing gates,
  returning to work, and handling code actions remain freeze blockers (§8).
  beta.1 fixes the deep-vs-task separation and the keep-test; it does not
  prematurely settle the gate topology.

## 4. Output verification

Verifying the produced text and code — the "verify" step of the cycle — stays
consistent with **no-write-time-oracle** and **graph structural integrity**.
Verification here is **never a semantic quality oracle**; the requirements'
HANS/PAWS evidence constraint says semantic checkers can be fooled by overlap and
wording. So verification produces **deterministic flags + PI routing**, and
proposes — it does not dispose.

### 4.0 Provenance labels — separate labels for separate facts

**No label ever means "true."** `human` (authored by the user); `machine`
(generated/transformed); `mc` / `span-matched` (matched to a source span by the
current checker = **citation-correct, not grounded**); `contradicted` (checker
found a likely contradiction); `unchecked` (no current check); `tainted`
(depends on machine/untrusted content — monotone, never clears); `stale`
(evidence/source changed after the check). The checks below act on these labels;
**semantic truth stays the user's judgment** (no-write-time-oracle).

### 4.1 Verifying TEXT

- **Deterministic citation/number checks (necessary, not sufficient).** Every
  cited claim's evidence set resolves: `work_id` resolves to `bibliography.bib`
  and the catalog row, DOI/identifier metadata exists when available, the source
  span resolves through `passages`, and every printed number *recomputes* in
  code and matches. These prove *span existence and arithmetic*, **not
  grounding**. They are gates on form; they never certify truth.
- **Unsupported means structurally unsupported, not false.** Export refusal uses
  this label only when the evidence contract is incomplete or broken: no
  evidence set, unresolved `work_id`/bibliography row, unresolved passage anchor,
  stale/retracted/superseded source, unresolved code warrant, or failing numeric
  recomputation. It does not mean Memoria has judged the proposition untrue.
- **Abstain / evidence-incomplete surfacing.** Passages that abstained or carry
  implicit/multi-hop warrants are flagged and **routed to the PI**.
- **Graph structural integrity.** The draft's references into the notes/claims
  graph must resolve (links, anchors, references) — *structural* checks, not
  per-node semantic validity.
- **Sparse, triaged, tooling-first.** Verification routes **only
  high-uncertainty / high-leverage** items to the PI, invests in per-claim
  tooling (side-by-side source, `work_id`+span diff), and treats a
  **max-items-per-session / triage threshold as a first-class constraint** — not
  a throughput target.

### 4.2 Verifying CODE

- **Verification = execution.** Code is verified by *running it and observing the
  output*, not by trusting the model's claim that it works — deterministic
  grounding over model assertion, the same principle as numbers-in-code. A
  computed figure is re-verified by re-running its warrant script.
- **Structural + reversible.** Artifacts are versioned and revertable
  (snapshot/revert); a failed run fails cleanly and never half-writes (the
  staged-writes / offline contract).
- Execution uses the **minimal committed sandbox (§2.3)** — no-network
  credential-free worker, project-scoped filesystem, approved-commands-only,
  resource-capped; hardening beyond it is §8 Q1.

### 4.3 Human + machine is not assumed to beat human XOR machine

Human-AI complementarity is an empirical result to earn, not a design truth.
Research on AI-advised decisions shows that explanations can increase acceptance
of AI recommendations regardless of correctness, simple explanations do not
reliably prevent overreliance, verification cost changes whether users inspect
AI advice, and reliance measures must be compared against baseline and
complementarity benchmarks ([Bansal et al. 2020](https://arxiv.org/abs/2006.14779);
[Buçinca et al. 2021](https://arxiv.org/abs/2102.09692);
[Vasconcelos et al. 2022](https://arxiv.org/abs/2212.06823);
[Guo et al. 2024](https://arxiv.org/abs/2401.15356)).

beta.1 therefore treats every human-machine gate as a measured intervention.
Where outcomes are adjudicable without creating a new feedback chore, calibration
compares at least four conditions: human-only / null workflow, raw machine
proposal, machine-visible-before-PI-judgment, and
PI-independent-first-then-machine-critique. "PI accepted" is a **disposition**,
not proof of truth and not the same fact as `checked`. For semantic/argument
judgments, the preferred review shape preserves independence first: show
evidence/source context and let the PI form or request an initial judgment
before revealing the machine's verdict or critique, unless the PI explicitly
asks for machine-first assistance. If the combined path does not beat the
relevant solo baseline after review cost, the gate stays shadow-first, is
retuned, or is removed.

## 5. Onboarding & the usable-by-a-real-researcher bar

The MVP bar (method): a real researcher uses it on a real corpus and produces a
grounded answer *and* begins producing text/code. Onboarding operationalizes
that bar.

- **Guided corpus seeding.** An onboarding wizard walks the full opening arc of
  the cycle: open a project (research question) → seed → first gap analysis over
  both graphs → first capture → first note → first Ask → first canvas/draft
  gesture. This turns the abstract cycle into a guided path.
- **Time-to-first-answer: install → grounded answer ≤ 30 min.**
  The contract is *grounded* — a real answer with resolvable citations — not a
  canned demo. beta.1 extends it to a **first WRITE gesture** (a canvas thread or
  a drafted paragraph) within the guided path, since WRITE is half the mission.
- **The sample corpus is REAL, never generated.** The onboarding corpus is a set
  of real, citable sources (papers/repos). Generated corpora are barred by the
  honest-corpus condition and the no-synthetic-index guardrail — a fabricated
  corpus cannot produce honest feedback.
- **Designed for low burden, because abandonment is the dominant failure mode.**
  Onboarding minimizes setup, derives feedback *implicitly* from gate
  dispositions rather than adding a feedback chore, and front-loads a fast
  payoff (the ≤30-min grounded answer). Which real sources ship as the seed
  corpus, and their licensing, is OPEN (§8).

## 6. Feedback instrumentation

"The feedback loop begins" is the definition of the MVP; §6 is what makes it
begin — **honestly, with no synthetic index**. The dashboard shows raw counts,
dispositions, and attribution events; it never rolls them into one "system
health" score.

- **Disposition capture, implicit.** Every recommendation already carries impact
  level, certainty level, consequences, and reasoning (propose-not-dispose). The
  loop records the **accept / reject / edit / defer / override disposition**,
  whether machine advice was visible before the PI judgment, and the reason
  shown at the time. Feedback is *implicit* from the gate the PI already operates
  — no separate rating burden.
- **Shadow-first calibration discipline.** Every tunable ships behind
  `production_enabled:false`; **no score changes routing, promotion, or
  visibility until it is grounded on recorded dispositions**. The loop
  *collects* dispositions to enable later calibration; it never auto-promotes a
  threshold into production.
- **LLM call-site eval ledger.** Every product-affecting LLM API call has a
  stable `call_site_id` and versioned variables: prompt template, model/provider,
  provider profile, runtime mode, decoding settings, context/retrieval policy,
  tool permissions, schema/validator, and cost/latency budget. `runtime_mode`
  is only `production` or `test`; `llm_profile` selects the endpoint/model.
  Production mode may use only the promoted production profile and may write to
  the real workspace. Test mode uses an isolated workspace plus either a local
  API endpoint or a budget-capped cheap remote profile. A candidate change stays
  shadow-only until its frozen eval suite beats the active baseline on the
  registered metric and does not regress any safety, grounding, schema, or cost
  blocker. Eval runs record dataset hash, grader versions, sampled outputs,
  token/cost/latency, runtime mode, provider profile, and the explicit promotion
  decision. Deterministic validators and reference checks are preferred; PI
  adjudication handles subjective cases; LLM-as-judge is a calibrated diagnostic
  only, never the sole authority for truth, grounding, or blocker promotion.
  Test-mode results prove wiring/schema/regression behavior; they prove
  production quality only when the same profile is the intended production
  target and the normal promotion gate passes.
- **Usage & load, because single-PI capacity is unvalidated.** Instrument
  items/session, time-to-decision, override rate, and lapse/abandonment — the
  single-PI sustainable load is an *unvalidated assumption*. Define a
  **throughput-exceeded fallback** (raise the uncertainty threshold; batch
  low-risk items).
- **Payoff attribution, event-based not scored.** "Improves knowledge work" has
  no settled beta.1 evidence. So attribution is a traceable chain of events —
  *recommendation → capture → note → claim → draft passage / code warrant* —
  compared against the **human-only / null workflow**, and against raw-machine or
  combined variants when the task has an adjudicable outcome. Honest events,
  never a payoff score.
- **Complementarity calibration.** Sample PI-vs-engine agreement, seeded probes,
  and adjudicable tasks to test whether the combined workflow beats the relevant
  solo baseline after review cost (§4.3). Measure overreliance, underreliance,
  override rate, and time-to-decision. If the combined path loses, surface it.
- **Guard the loop against itself.** A feedback loop trained on what the PI
  up-votes can suppress disconfirmations. So a **separate
  exploration/anomaly/diversity channel is reserved**, and **diversity coverage
  is measured as its own objective.** The design of that channel is OPEN (§8).

## 7. MVP acceptance — the definition of done

**beta.1 is accepted when a real researcher, on a real corpus, runs the complete
cycle end-to-end with no core item stubbed:**

1. **READ half (inherited, must run):** open project (question/thesis) → seed →
   gap analysis over **both** graphs → source → capture (immutable raw + content
   + metadata) → extract atomic annotations → develop notes/claims-with-context
   toward the thesis → re-run gap analysis.
2. **Query mechanism (beta.1 substrate, must run):** SQLite + FTS5 +
   `sqlite-vec` indexes `passages` as verbatim retrieval units; query-time
   mtime-gated reindex keeps files fresh without a daemon; verdict changes
   cascade to `passages.check_status` in the same transaction; `concept_edges`
   carries `links:` mirrors and PI-confirmed `tension` edges; targeted lookups
   apply structural filters before lexical/vector ranking; topic surfacing
   returns the bounded local/global neighborhood with denominators; RRF fuses
   lexical, semantic, and graph lists; BM25 stays the default activation gate
   until the pre-registered fixture spike beats it.
3. **WRITE half (this document, must run):** argument graph → **canvas** thread →
   derived **outline** → **draft (text)** grounded in coherent slices with
   deterministic numbers/citations and abstention; **code** artifacts as
   executable warrants and/or deliverables.
4. **Verify:** deterministic citation/number checks, evidence-incomplete/abstain
   routing, graph structural integrity, code-by-execution — sparse and triaged.
5. **LLM evaluation bar:** every production-enabled LLM API call has a versioned
   call-site contract, frozen eval suite, active baseline, and recorded
   promotion evidence for the active prompt/model/config/context/tool variables,
   with production/test runtime modes and a local-or-cheap test profile.
6. **Usable bar:** install → grounded answer ≤ 30 min on a **real** seed corpus,
   plus a first WRITE gesture on the guided path.
7. **Crash/recovery bar:** SQLite-local ACID holds, while DB+files provide
   observable crash consistency: every interrupted machine operation recovers to
   committed/consumable, retryable/pending, or failed/hidden, and no partial
   machine output is visible as checked.
8. **Loop begins:** dispositions, machine-advice exposure timing, usage/load,
   payoff-attribution events, complementarity calibration, and
   diversity-coverage measures are all instrumented and producing honest data.

**The anti-deferral rule (binding).** MVP scope is set by user NEED, not deferral
convenience. To move any item out of beta.1 you must **prove the user does not
need it** for viable, feedback-producing use. "Future consumer," "post-beta,"
and "hard to build" are **excuses, not proofs** (method). The argument graph,
hubs, claims, gap-over-both-graphs, the writing module, and the coding module are
all IN — the mission's output half cannot be stubbed and still carry a real user.

**What is legitimately out (proven not-needed for viable use), not deferred core:**
multi-user / team features (single-person tool by identity); cloud sync beyond
single-user-multi-device (local-first is a constraint, not the mission); a
rich **spatial-graph-at-scale** surface (canvas is a small project-thread
surface and must beat the tabular/faceted baseline before expansion); a **local
fine-tuned / QLoRA adapter** (the frozen local model is only the *test fixture*;
the live engine is an API, so frozen-model schema failures do not gate the MVP);
**two-way canvas↔outline sync** (§1.3, simplest-design). Each of these is
excluded by a fact, not by convenience.

## 10. Trade-offs

beta.1 trades **breadth for a complete loop** — no general web discovery, no
multi-user/team features, no autonomous agents; the mission is one researcher
producing knowledge (text and code) from a real corpus. It also trades
**machine certainty for honesty**: the system can capture, connect, retrieve,
cite, check structure, and refuse bad exports, but it **cannot know an argument
is true** — that is the user's judgment (no-write-time-oracle). The one
constraint that flips a deferral into scope is real alpha.16 use: if the
researcher cannot reach feedback-producing use without a missing piece, it moves
into beta.1; if a feature is merely elegant or historically desired, it waits
(the anti-deferral rule, §7).

## 11. Open-question research resolutions (2026-07-06 addendum)

Fresh, independently-sourced research was commissioned against six of §8's open
questions (six `deep-research` workflow passes, each with 3-vote adversarial
verification), per an explicit epistemic rule: prior material already
in `resources/` — including this project's own rethink/gap-analysis/redo
docs — is reference only, never a citable source, and every claim below had
to be independently re-verified against a primary source regardless of what
`resources/` already said. Resource write-ups remain archived for traceability;
only the load-bearing conclusions summarized here govern beta.1. §11.7 adds the
same primary-source discipline for LLM prompt/model evaluation, triggered by the
beta.1 LLM API-call requirement rather than by the original six open-question

### 11.1 Sandbox backend (resolves §8 Q1, in part)

The minimal sandbox (§2.3) is corroborated by direct precedent: Anthropic's
Claude Code sandboxing write-up states that effective agent sandboxing needs
both filesystem and network isolation, and that Claude Code uses Linux
`bubblewrap` and macOS Seatbelt as OS-level enforcement primitives. Three
concrete hardening rules follow from primary/reliable sources: (1)
"approved-commands-only" must be an **allowlist resolved to canonical paths**,
not a denylist/pattern match, because command-denylist fragility is empirically
documented in terminal-agent settings; (2) wrapper code invoking `bwrap` must
insert the `--` argument delimiter before the command, because CVE-2024-32462
was exactly a Flatpak/bubblewrap option-injection sandbox escape fixed by that
delimiter; (3) Apple's `sandbox-exec` is marked **deprecated** in its own man
page, so the macOS backend is a tracked risk, not an assumed-stable primitive.
Adversarial-validation methodology is sharpened, not just reaffirmed:
peer-reviewed evidence (NAACL 2025 Findings) shows adaptive, defense-aware
attacks bypassed all eight tested indirect-prompt-injection defenses — §8 Q1's
"adversarial validation" must be an adaptive red-team process, not a one-time
static attack-list pass.
Primary sources: [Anthropic Claude Code sandboxing](https://www.anthropic.com/engineering/claude-code-sandboxing);
[CmdNeedle](https://arxiv.org/abs/2606.15549);
[Flatpak CVE-2024-32462 advisory](https://github.com/flatpak/flatpak/security/advisories/GHSA-phv6-cpc2-2fgj);
[Apple `sandbox-exec` man page](https://keith.github.io/xcode-man-pages/sandbox-exec.1.html);
[Adaptive Attacks Break Defenses Against Indirect Prompt Injection Attacks on LLM Agents](https://arxiv.org/abs/2503.00061).
Reference archive: `resources/0.1.0-beta.1-sandbox-backend-research.md`

### 11.2 Coherent-slice construction (resolves §8 Q8)

Independent research substantially **validates** §1.4's existing
whole-note-if-budget-else-section-expanded-anchor policy rather than
overturning it. A real, quantified precedent (SCAR, arXiv:2606.16661) shows
whole-section retrieval winning on accuracy (0.968) but at far higher token
cost than selective anchor-expansion (0.928 accuracy, ~15% of the cost) —
confirming the budget-gated choice (not a blanket preference either way) is
the right shape. SCAR's own formalized expansion-decision threshold
(`S(c,n) > γ·cos(e_q,e_c)`) is a concrete refinement worth prototyping
against the current fixed-anchor heuristic in a later iteration, not a
reason to change the shipped default now.
Primary source: [SCAR: Semantic Continuity-Aware Retrieval for Efficient Context Expansion in RAG](https://arxiv.org/abs/2606.16661).
Reference archive: `resources/0.1.0-beta.1-coherent-slice-research.md`

### 11.3 Exploration/diversity channel (resolves §8 Q10; reconciles with the existing alpha.17 decision)

This research keeps alpha.17's 2026-07-06 decision ("MMR baseline,
facility-location behind a fixture gate, DPP deferred") but narrows the
evidence claim. MMR is an appropriate baseline because the original paper
defines a cheap user-tunable relevance/novelty tradeoff and explicitly
describes low-λ exploration followed by higher-λ focus. Facility-location's
classic support is the greedy guarantee for monotone submodular maximization,
so it belongs behind a fixture gate rather than being adopted without a
Memoria result. DPPs are real diversity models, but their machinery is heavier
than MMR for beta.1's first pass. The original MMR paper reports only a
five-user pilot for user preference, so it is not a generalizable proof that
Memoria users will act on diversity-oriented surfacing. This sharpens §7's
"exploration acted-on rate" measurement residual: whether the channel is used
at all is an empirical beta.1 question.
Primary sources: [Carbonell & Goldstein 1998 MMR paper](https://www.cs.cmu.edu/~jgc/publication/The_Use_MMR_Diversity_Based_LTMIR_1998.pdf);
[Nemhauser, Wolsey & Fisher 1978](https://doi.org/10.1007/BF01588971);
[Kulesza & Taskar 2012, Determinantal Point Processes for Machine Learning](https://arxiv.org/abs/1207.6083).
Reference archive: `resources/0.1.0-beta.1-exploration-diversity-research.md`

### 11.4 Structured-extraction ↔ free-composition boundary (resolves §8 Q13, in part)

The source-backed finding is narrower: **negation must be named as a specific
verification risk**, not assumed safe because a fact was "extracted correctly."
FactCC explicitly includes sentence-negation transformations in factual
consistency training data and reports that auxiliary source/summary span
extraction helps human verification. AlignScore supports chunk-level
context/claim-sentence alignment as a factual-consistency checking pattern,
and clinical/radiology NLP work treats negation/uncertainty as a specialized
extraction problem. Together these sources support Memoria's free-form prose
plus typed evidence-set marker: structured extraction verifies and explains
drafted prose after the fact; it does not become the substrate for composition
or truth judgment. **Important limit:** these are analogies from factuality and
clinical NLP, not direct studies of research-writing compose pipelines.
Primary sources: [FactCC](https://arxiv.org/abs/1910.12840);
[AlignScore](https://arxiv.org/abs/2305.16739);
[NegBio](https://arxiv.org/abs/1712.05898).

### 11.7 LLM call-site evaluation methodology

LLM API calls are empirical components, not design constants. Official provider
guidance converges on the same shape: define the task, run representative test
inputs, grade with explicit criteria or task metrics, compare runs when prompts
or models change, and inspect results before promotion. OpenAI's eval guidance
frames evals as a way to test model outputs against specified style/content
criteria, especially when trying or upgrading models; Anthropic's evaluation
tool emphasizes re-running the same test suite across prompt versions,
side-by-side comparison, quality grading, and prompt versioning; Google's model
evaluation docs describe prompt/ground-truth evaluation datasets and task
metrics; OpenAI's `simple-evals` notes that eval results are sensitive to
prompting and reports model results with explicit prompt settings. LLM-as-judge
research supports scalable preference approximation but also names position,
verbosity, and self-enhancement biases, so Memoria cannot use an LLM judge as
the sole promotion authority. The production/test split follows config practice:
credentials, backing services, and deploy-varying values stay in runtime
configuration rather than code constants.

Therefore beta.1 treats prompt text, model/provider, provider profile, runtime
mode, decoding settings, retrieval/context construction, tool permissions,
output schema, and cost/latency budget as **versioned variables**. A
production-affecting LLM call site must have a frozen eval suite and active
baseline before it can be enabled, and any candidate change must run
candidate-vs-baseline with recorded metrics and sampled outputs. `runtime_mode`
is a safety/write boundary (`production` or `test`); `llm_profile` selects the
endpoint/model. Test mode exists to make evals cheap and frequent: it uses an
isolated workspace with either a local API endpoint or a budget-capped cheap
remote profile. Those runs validate wiring, schemas, deterministic validators,
prompt regressions, and cost envelopes; they do not prove production quality
unless that same profile is the intended production profile and the promotion
gate passes. Grading uses deterministic validators and reference checks where
available, PI adjudication for subjective cases, and calibrated LLM judges only
as secondary diagnostics. This is the same empirical gate already used for
retrieval activation: measure the changed variable on the Memoria task, then
promote only with recorded evidence.

Primary sources: [OpenAI evals guide](https://platform.openai.com/docs/guides/evals);
[Anthropic evaluation tool](https://docs.anthropic.com/en/docs/test-and-evaluate/eval-tool);
[Google model evaluation docs](https://cloud.google.com/vertex-ai/generative-ai/docs/models/evaluate-models);
[OpenAI simple-evals](https://github.com/openai/simple-evals);
[Judging LLM-as-a-Judge with MT-Bench and Chatbot Arena](https://arxiv.org/abs/2306.05685);
[The Twelve-Factor App: Config](https://12factor.net/config).

- The retrieval store is SQLite + FTS5 + `sqlite-vec`, with a startup capability
  check for FTS5/extension availability.
- `passages` is the retrievable unit: verbatim text, not extracted facts. Its
  origins are `note-body`, `source-span`, `work-aspect`, and `db-mirror`.
  `mode` and `question_status` are mirrored onto note-body rows so Q1-style
  structural queries are real SQL predicates, not prose approximations.
- `passages.check_status` is copied for query speed, so every write path that
  changes verdict state must update matching `passages` rows in the same
  transaction. Query results never mix unchecked/quarantined rows into checked
  output without explicit labeling.
- `concept_edges` mirrors `links:` as `supports` / `contradicts` / `extends`,
  and stores PI-confirmed `tension` edges. `work_graph_edges` remains
  bibliographic only.
- Query-time mtime-gated lazy reindex uses `file_index_state`; no daemon or
  watcher is required.
- Search combines lexical, semantic, structural, and bounded local/global graph
  retrieval with RRF. Shape 1 targeted lookups may be result-limited; Shape 2
  topic surfacing returns the complete checked neighborhood within the stated
  reachability bound and relevance threshold, with denominators shown.
- Reranking and synthesis are presentation layers over the retrieved set. They
  may improve order or summarize a stated subset, but never hide the underlying
  retrieved context or certify truth.

### 12.3 OKF Compliance Contract

beta.1 must be **genuinely OKF-compliant**, not merely OKF-inspired.

- The Memoria export unit is a clean copy from a committed state of the vault
  root minus `.memoria/` and any still-root `journal/` runtime mirror; its
  markdown knowledge layer is the OKF bundle, with `bibliography.bib` and any
  non-markdown artifacts treated as Memoria companions. No lossy conversion
  pipeline is in scope; if the live/export unit is compliant, copy-paste export
  is enough.
- OKF conformance is the §9 bar from the attached v0.1 draft: every
  non-reserved `.md` file has a parseable YAML frontmatter block and a
  non-empty `type`; every reserved `index.md` and `log.md`, when present,
  follows its reserved structure. `index.md` is the generated progressive
  disclosure listing; it normally has no frontmatter, except the OKF-permitted
  bundle-root `okf_version: "0.1"` frontmatter if Memoria chooses to declare
  the target version. `log.md` is optional; if present, date headings use ISO
  `YYYY-MM-DD`.
- Every non-reserved `.md` file in the export unit is an OKF Concept document
  for **export shape**. This includes bundle files, `outline.md`, `draft.md`,
  root operational docs, and vault-visible `system/` docs/dashboards. That does
  **not** make every such file a checked Memoria knowledge-graph node: drafts,
  outlines, operational docs, and dashboards are export-readable artifacts unless
  a separate Memoria write path promotes their extracted claims into the checked
  knowledge graph. `bibliography.bib` is not an OKF Concept document; it is
  Memoria's portable citation companion and must carry `memoria_work_id` for
  each entry.
- Concept identity is path-derived. Notes and hubs use stable ID filenames;
  digests and fulltexts use `work_id`; each `projects/<slug>/` directory is a
  nested OKF knowledge bundle whose slug path participates in the OKF Concept
  ID. A Memoria-internal `id` field, if present, is producer-defined metadata,
  not the OKF identity.
- OKF-facing relationships and citations use standard Markdown links, preferring
  bundle-relative absolute links (`/path/to/concept.md`) for stability.
  Obsidian wikilinks may remain as local authoring affordances only when every
  relationship/citation an OKF consumer needs also has a standard Markdown link.
- Memoria keeps structured `links:` for validation, but the OKF-readable
  relationship kind is conveyed by surrounding prose. This prose narration is a
  Memoria strict-producer/lint rule, not an extra OKF §9 conformance condition:
  every structured `links:` target must be narrated in the Links section with
  its relation.
- OKF reserved body headings (`# Schema`, `# Examples`, `# Citations`) are
  conventional, not acceptance blockers. Memoria SHOULD use `# Citations` when
  a concept body makes claims from external material, but missing it is not an
  OKF conformance failure.
- `.canvas`, code files, raw source blobs, and raw datasets are Memoria
  companion artifacts, not OKF markdown Concept documents. When included in a
  project, they must be referenced from typed markdown records and must not be
  required for an OKF consumer to parse the markdown bundle. Raw dataset
  bundling remains an open freeze blocker in §8.
- `check_status` and verdicts never travel in frontmatter. Imported or copied
  OKF content re-enters Memoria as unchecked and passes through the normal gate.

### 12.4 Crash-Consistency / ACID Contract

beta.1 must be crash-consistent, but it must **not** claim ACID compliance for
the whole DB+files structure.

**Why not global ACID:** SQLite provides atomic transactions for the SQLite
database itself ([SQLite atomic commit](https://www.sqlite.org/atomiccommit.html);
[SQLite application-file format](https://www.sqlite.org/appfileformat.html)).
Memoria deliberately has more than one durable surface: SQLite, live markdown,
tracked companion files, git commits, blobs, projections, and direct human edits
from Obsidian or another editor. `os.replace` gives an atomic successful file
replacement on POSIX ([Python `os.replace`](https://docs.python.org/3/library/os.html#os.replace)),
but it does not make a group of file writes plus SQLite writes plus a git commit
one transaction. Git commits record selected file contents and advance history
([git-commit](https://git-scm.com/docs/git-commit)); they are checkpoint and
revert proof, not a transaction manager for SQLite state. `.memoria/memoria.sqlite`
is not git-committed, and direct human edits cannot join a Memoria transaction.

Therefore the beta.1 system-level contract is **observable crash consistency**:
after `workspace recover`, every machine operation is either fully committed and
consumable, retryable/pending, or failed/hidden. No read, query, verification, or
export path may observe torn machine output as checked or complete.

**Atomicity required for beta.1:** SQLite mutations are atomic inside one SQLite
transaction. Across DB+files, Memoria requires **operation atomicity at the read
barrier**, not physical all-or-nothing writes: every operation has a durable
intent record with idempotency key, base commit, footprint paths, payload
bytes/hashes or blob refs, and expected outputs before materialization. Recovery
must reconcile each non-terminal operation to committed, retryable, or failed.
If recovery collides with later human edits, recovered machine content is
diverted to a conflict artifact and an incident; it never overwrites human text.

**Consistency required for beta.1:** all externally visible surfaces must satisfy
their invariants before they are consumable. `checked` file outputs must have
matching materialized hashes and, for tracked files, a materializing commit.
`passages.check_status` must cascade in the same transaction as verdict changes.
Query results must not mix unchecked/quarantined rows into checked result sets.
`bibliography.bib` must be current enough for every checked citation/export
claim, by regeneration or freshness check. Derived indexes/projections are either
current, rebuilt on demand, or visibly stale/failed; never silently authoritative.

**Isolation required for beta.1:** Memoria has one workspace machine writer.
SQLite supplies DB transaction isolation; the workspace lock serializes Memoria
machine operations over the wider vault. Human editor writes are outside Memoria's
transaction boundary, so operations use precondition hashes, postcondition
verification, demotion, and requeue/conflict diversion instead of pretending full
isolation. A second concurrent writer/user is a design flip condition, not a
beta.1 tuning parameter.

**Durability required for beta.1:** SQLite uses durable transaction settings for
DB state; file writes use temp-write, fsync, atomic replace, and parent-directory
fsync; tracked human-visible files and `.memoria/journal-head` are committed to
git as checkpoint/revert evidence; exact payload bytes or blob refs needed for
replay are written before materialization. Backup/restore drills must cover the
whole operational vault state needed to recover — tracked files, `.git`,
`.memoria/memoria.sqlite` and WAL sidecars, blobs, materialization payloads,
configs, and generated projections or their rebuild path — not only the OKF
clean-copy export.
