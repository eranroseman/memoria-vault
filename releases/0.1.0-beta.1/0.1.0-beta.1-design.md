# Memoria 0.1.0-beta.1 — MVP system design (the WRITE/output additions)

Date: 2026-07-05
Status: beta.1 MVP design — alpha.16 + the MVP additions; the first usable product.

> **North star:** Memoria is an opinionated, single-person knowledge-production
> tool — an AI-enhanced Zettelkasten utilizing the LLM-wiki framework.
> **Mission:** help the user **read** (accumulate / compile / curate knowledge)
> and **write** (compose / create new knowledge), with **code and text as the
> main outputs.** Writing and coding are the *output half* of the mission, not
> peripheral.

**What this document is.** beta.1 is the first MVP that delivers Memoria's full
promise: the point where system functionality is *done* enough that a real
researcher can use it on a real corpus and the build-measure-learn loop begins
([method](resources/redo/00-redo-method.md)). The MVP is defined by **no deferred core**:
to defer an item you must *prove the user does not need it* for viable,
feedback-producing use — "future consumer / post-beta / hard to build" are
excuses, not proofs. **beta.1 = alpha.16 + the additions required for the MVP.**
alpha.16 already ships the entire READ cycle and the knowledge-production
substrate ([alpha.16 design](../0.1.0-alpha.16/0.1.0-alpha.16-design-claude.md)); this
document specifies **only the WRITE/output half** and does not re-specify alpha.16.

*Facts-only grounding: every decision rests on a research finding, a documented
failure, or a B/A/R verdict — never ADR-authority. Sources: RR =
[REVIEW-REFUTATIONS.md](../../design-history/sources/notes/REVIEW-REFUTATIONS.md);
ADV = beta.1 paper adversarial-review; B-# = [beta1-adopt-borrow.md](resources/redo/beta1-adopt-borrow.md);
A-# = [alpha15-borrow-adapt-reject.md](resources/redo/alpha15-borrow-adapt-reject.md); owner =
[owner-design-notes.md](resources/redo/sources/owner-design-notes.md). Anything not settled by
evidence is an explicit OPEN RESEARCH QUESTION (§8), never an invented mechanism.*

## 0. Relationship to alpha.16 (inherits everything; this doc specifies only the MVP additions)

beta.1 **inherits alpha.16 wholesale** and re-specifies none of it. From
alpha.16, load-bearing and unchanged here:

- The **two-graph vault** — catalog (SQLite, relational, objective) + ZK
  markdown notes/claims/hubs with **typed argument edges** (`supports` /
  `contradicts` / `extends`) forming the subjective knowledge graph, bridged by
  the source note (A-BORROW-3/4/5/12; owner R-1/R-2).
- **Capture / extract / digest**, **retrieval / Ask** (BM25+dense, RRF/MMR,
  **verbatim byte-exact spans** as the stored/retrieved unit — B-ADOPT-1/2),
  **gap analysis over both graphs** and the gap between them (A-BORROW-3; owner
  R-2), integrity / provenance / **snapshot→apply→verify→commit + revert**
  (B-ADOPT-3/4), catalog / enrichment, security, cost.

**The flawed beta.1's core error was deferring exactly this substrate** — the
argument graph, hubs, claims-with-context, gap-over-both-graphs — and treating
writing and code as afterthoughts. That deferral is not research-justified; the
research in several places *requires* keeping the substrate (A-BORROW cross-cutting
note). alpha.16 restores it, so beta.1 starts from a **complete production
substrate** and adds the surface that turns it into **text and code output**.

**The guardrails carry over unchanged and remain load-bearing in every section
below** — no synthetic index with a threshold (owner); show the reasoning, not
just the verdict (owner); propose-not-dispose + anti-anchoring (owner);
no-write-time-oracle, `checked` ≠ approved (A-BORROW-2; RR #1); deterministic
grounding for every number and citation (ADV L97; RR — Zou 2024); graph
*structural* integrity over single-node semantic validity (A-ADAPT-4; RR §2).
Every addition in §§1–6 is subordinate to these.

The MVP additions, and nothing else: §1 writing module (→ **text** draft), §2
coding module (→ **code** artifact), §3 compose/deep-work workspace, §4 output
verification, §5 onboarding & the usable bar, §6 feedback instrumentation — each
consuming the alpha.16 substrate above and re-specifying none of it.

## 1. The writing module

The owner's named flow, made concrete: **argument graph → knowledge map /
canvas → outline → draft (TEXT output)** (owner "the analyst/librarian
merger"). Each stage below is a *proposal surface*: the system proposes with
its reasoning shown; the human — the single owner of judgment — disposes.

### 1.1 Argument graph (consumed, not built here)

The writing module reads the alpha.16 argument graph: **claims-with-context**
carrying TEMPO fields — `temporal_scope`, `tense`, `qualifier`/certainty,
originating section — because a claim's truth is determinate only relative to
these (A-BORROW-4; RR L21). Claims are connected by human-authored typed edges,
with **saturation** expressed as *honest coverage reasoning* — which claims lack
a support, which lack a counterpoint — **never a synthetic saturation score or
band** (A-ADAPT-2; owner anti-synthetic-index). This is the substrate the
thread is drawn from.

### 1.2 Knowledge map / canvas — the ZK-like sketch connecting notes into a thread

The **canvas** is the project-scoped spatial sketch that connects the notes and
claims relevant to the thesis into a *thread* (owner R-11's canvas; owner "the
analyst/librarian merger"). Design, grounded:

- **Small-graph, analyst-only, and it must beat a tabular baseline.** Spatial
  layouts *degrade as density grows*; tabular/faceted is the validated default
  at scale (Hearst head-to-head studies; RR L23; A-ADAPT-3). The canvas is
  therefore scoped to the *project slice* of the argument graph, not the whole
  vault, and is offered alongside — never instead of — the faceted/tabular view.
- **Positions persist; they are not regenerated.** "Disposable is safe for
  *data*, hazardous for spatial *layout*" — a stable object position builds the
  mental model that drives novel-task success (Card & Moran; Hearst; RR L89).
  Canvas node positions are governed, persisted state, reversible via the
  inherited snapshot/revert protocol.
- **Propose-not-dispose.** The system proposes *which* notes/claims belong on the
  canvas and *how* they connect (with the reasoning for each), drawing the
  relevant argument-graph slice; the human arranges the thread. The retrieval
  that surfaces candidates is BM25+dense, **never graph traversal** — the typed
  edges are the *argument* layer, not the retrieval representation
  (A-ADAPT-5; RR L51, MemoryAgentBench put graph systems below BM25).
- Format: a plain `.canvas` (JSON-Canvas) file so it is editable and inspectable
  without the plugin (the keep-test, §3).

### 1.3 Outline — derived from the canvas

**DECISION: the outline is a one-time, re-runnable derivation from the canvas —
NOT a live two-way sync.**

- Direction is fixed by the owner's own framing ("outline derived from the
  canvas"): canvas → outline, one-directional.
- Live two-way reconciliation between a spatial graph and a linear document is
  unbounded machinery with no documented need; the **simplest-design guardrail**
  rejects it until a requirement proves it. The canvas is the structural source
  of truth; the outline is *derived data*.
- Once derived, the outline is a **human-owned artifact**: edits stay in the
  outline and do not propagate back. Re-derivation is an **explicit, opt-in**
  action that supersedes the prior outline as a new version (reversible via
  snapshot/revert), warning that it discards outline-local edits.
- OPEN (§8): whether heavy outline editing creates real demand for
  outline→canvas back-propagation — to be *measured* (does the user re-derive,
  or diverge and never re-derive?), not assumed.

### 1.4 Draft — the TEXT output

The draft is composed section by section from the outline. It is the **output
artifact**, not a write into the knowledge graph. Grounding is dense here
because synthesis is where the research bites hardest:

- **Feed coherent slices, not top-k fragments.** Each outline node maps to its
  canvas notes/claims; synthesis **expands** them to coherent slices — whole
  distilled notes plus section-expanded anchors — before composing. Top-k
  fragments degrade integrative answers (Hu 2026; ADV #6; B-ADOPT-10). This is
  the single most important synthesis rule and the direct answer to "writing
  must feed coherent slices."
- **Every number computed in code; every citation resolved deterministically.**
  A printed number is computed by the engine's code path, never by the model
  (abstention+numbers-in-code is a *pipeline* property: 37.1% → 70.2%, Zou 2024;
  ADV L97; B-ADOPT-8). Citations resolve against the catalog's deterministic
  metadata (Zotero CSL-JSON; B-ADOPT-7) to a **verbatim span** (B-ADOPT-1).
- **Warrant = an evidence-*set*, typed.** Each drafted claim carries its
  evidence set with type single-sentence / multi-sentence / multi-hop /
  **implicit** (FEVER; Buchmann & Gurevych; RR). Derived/computed claims (the
  classic "1960") have **no single verbatim warrant**; they enter an
  **evidence-incomplete / abstain** state and route to the PI. A span match is
  relabeled **citation-correct, not grounded** — up to 57% of RAG citations are
  post-rationalized (Wallat; B-ADOPT-9).
- **Abstain rather than fabricate.** Where the corpus does not support a passage,
  the draft emits an explicit gap/"Unknown" marker; faithfulness comes from
  deterministic grounding, not from the model *choosing* to say it doesn't know
  (Rao & Callison-Burch; RR). CoT actively degrades abstention (Shaikh 2023).
- **Prose, not a forced schema.** Constrained decoding to a rigid schema corrupts
  negation and polarity — the worst error class here (Shin 2021; RR). The draft
  is natural-language output; structured extraction (if any) stays upstream in
  the engines.
- **Propose-not-dispose.** The draft is a proposal with per-passage reasoning
  (which notes/claims/evidence back it); the human composes and judges. Drafting
  **does not auto-write knowledge back into the graph** (that violates
  no-write-time-oracle); promoting a draft insight into a permanent note is the
  same sparse, uncertainty-routed PI gate (§4), and its workflow is OPEN (§8).

### 1.5 Close the loop

Drafting surfaces new gaps — claims that abstained, passages with
evidence-incomplete warrants, sections with no counterpoint. These feed back
into alpha.16's **gap analysis over both graphs**, completing the owner's cycle
(project → … → develop notes/claims → **re-run gap analysis**). Writing is thus
not a terminal export but a driver of the next READ iteration.

## 2. The coding module

Code is a **first-class project output** — the other half of "code and text as
the main outputs" — not the inert artifact the flawed beta.1 made it. Evidence
specific to code generation is thin in the corpus, so this section specifies
only what the facts support and pushes the rest to §8 rather than inventing
mechanism.

### 2.1 Two grounded roles for code

1. **Code as executable warrant.** FEVER/Buchmann establish that *derived /
   computed* claims have no single verbatim source span. Code fills exactly that
   gap: a computed figure is warranted by the **re-runnable script that produced
   it**. This is the same doctrine as "every number computed in code" (Zou 2024;
   B-ADOPT-8) promoted to a first-class artifact — the code *is* the deterministic
   warrant for the class of claims the text warrant model cannot cover. A claim's
   evidence set (§1.4) may therefore point to a **code artifact + its output**,
   not only a source span.
2. **Code as deliverable.** A script, analysis, or tool the researcher ships as
   an output in its own right.

### 2.2 How code relates to the project and its claims

- Code artifacts are **project-scoped**, authored by a **code lane** (owner:
  writing/coding "have their own agents"), stored as plain files, and versioned
  through the inherited **single VaultWriter + snapshot→apply→verify→commit +
  one-click revert** (B-ADOPT-3/4). Reversibility and provenance are unchanged
  from alpha.16.
- The link to claims runs through the **evidence-set** mechanism (§2.1.1): a
  computed claim references the code that computes it; regenerating the number
  re-runs the code (§4).
- **Propose-not-dispose applies**, with a sharp caveat: **verification asymmetry**
  — auditing a code artifact "requires the same expertise as building it," and
  under deadline the review degrades to a cursory nod (Feng & Liu §5.5; RR). So
  code review is a genuine bottleneck; the answer is per-item verification
  *tooling* (diff, run-and-observe, tests) and triage — not throughput (§4).

### 2.3 What is deliberately NOT designed here

The full code-authoring workflow (a canvas/outline analog for code; language and
runtime scope; the code lane's agent posture and skills) is **not settled by
evidence** and is enumerated in §8. Most critically, **executing generated code
reopens the exec/exfiltration surface** that least-privilege cannot close
(InjecAgent: an exec/send tool is precisely what completes a two-step steal; RR
§5; AgentDojo tool-isolation fails in exactly Memoria's conditions). The
execution sandbox is a **security-critical open question (§8)**, not a detail to
hand-wave.

## 3. The compose / deep-work workspace

The owner's split is explicit design intent: **"the workspaces should clearly
separate deep work (e.g., the project document) from task-oriented work (e.g.,
reviewing links). Distinct mindsets; don't mix."** beta.1 adds the **compose /
deep-work workspace** as the surface for writing and coding, distinct from the
task-oriented curation surfaces (inbox, gap review, link review, catalog
health) inherited from alpha.16.

**The workspace is specified as five UI-framework-independent jobs** — *what* the
researcher needs, not *how* a plugin renders it, so beta.1 can commit the jobs
while the gate topology and host stay open (§8, Q11): (1) **Read** — source
queue, reading position, annotations, digest; (2) **Knowledge** —
claims/questions/tensions, hubs, graph integrity; (3) **Project** — question,
source scope, gap report, canvas, outline, draft, code; (4) **Review** — machine
writes, conflicts, export refusals, integrity incidents; (5) **Upkeep** — sync,
cost, backups, migrations, provider failures. The deep-work composition surface
below is the *Project* job; the others are inherited from alpha.16.

- **What it hosts:** the deep-work project artifacts — the **canvas, outline,
  draft, and code artifacts** — plus the co-PI as the active-assistance surface
  (owner: "Left pane = navigation. Co-PI = right pane"). It is where the human
  arranges the thread and composes; it deliberately does **not** surface the
  inbox/recommendation stream, which lives on the task-oriented gates
  (mindset separation).
- **Constraint — the keep-test holds.** The plugin must **not** be the only way
  in: the product must run with `vim`/`nano`/a file browser (the keep-test the
  flawed beta.1 reversed; B "DO NOT ADOPT" — Obsidian-plugin-as-required-UI).
  Every deep-work artifact is a plain file (`.md` draft, `.canvas`, source
  files); the workspace is a convenience layer over files, never a gate on them.
  This also preserves the **no-daemon** posture (ADR-95 rejection cited in B).
- **What is grounded vs open.** The *deep-vs-task boundary* is grounded owner
  intent and is committed for beta.1. The full **gate/workspace model** is
  **not**: the carried `spaces/` + `_nav` three-places model was rejected as
  authority-carried with no surfaced failure (A-REJECT-1), and the owner's notes
  re-open gates, left-pane-nav, and active-assistance-per-workspace as design
  tasks. This connects directly to the **OPEN workspace/gate question inherited
  from alpha.16** (§8). beta.1 fixes the deep-vs-task separation and the
  keep-test; it does not prematurely settle the gate topology.

## 4. Output verification

Verifying the produced text and code — the "verify" step of the owner's cycle —
consistent with **no-write-time-oracle** and **graph structural integrity**.
Verification here is **never a semantic quality oracle**: human+machine review
does not beat machine alone and a wrong machine verdict drags the reviewer
*below* baseline (Jacobs 2021; RR #1), NLI is high-precision/low-recall and
miscalibrated (RR §2), and richer explanations can *worsen* overreliance
(Jacobs; RR). So verification produces **deterministic flags + PI routing**, and
proposes — it does not dispose.

### 4.0 Provenance labels — separate labels for separate facts

**No label ever means "true."** `human` (authored by the user); `machine`
(generated/transformed); `mc` / `span-matched` (matched to a source span by the
current checker = **citation-correct, not grounded**); `contradicted` (checker
found a likely contradiction); `unchecked` (no current check); `tainted`
(depends on machine/untrusted content — monotone, never clears); `stale`
(evidence/source changed after the check). The checks below act on these labels;
**semantic truth stays the user's judgment** (no-write-time-oracle).

### 4.1 Verifying TEXT

- **Deterministic citation/number checks (necessary, not sufficient).** Every
  cited claim's evidence set resolves: citekey/DOI exists, the verbatim span is
  present in the source, every printed number *recomputes* in code and matches.
  These prove *span existence and arithmetic*, **not grounding** (ExpertQA:
  deterministic checks don't establish correctness; RR). They are gates on
  form; they never certify truth.
- **Abstain / evidence-incomplete surfacing.** Passages that abstained or carry
  implicit/multi-hop warrants are flagged and **routed to the PI** — the exact
  claims FEVER/Buchmann say cannot be auto-resolved.
- **Graph structural integrity.** The draft's references into the notes/claims
  graph must resolve (links, anchors, references) — *structural* checks, not
  per-node semantic validity (A-ADAPT-4; RR §2).
- **Sparse, triaged, tooling-first.** Auditing costs the same expertise as
  authoring and floods under load (Feng & Liu; RR); AutoResearchClaw found
  targeted intervention at a few checkpoints dominates dense per-item approval.
  So verification routes **only high-uncertainty / high-leverage** items to the
  PI, invests in per-claim tooling (side-by-side source, citekey+span diff), and
  treats a **max-items-per-session / triage threshold as a first-class
  constraint** — not a throughput target.

### 4.2 Verifying CODE

- **Verification = execution.** Code is verified by *running it and observing the
  output*, not by trusting the model's claim that it works — deterministic
  grounding over model assertion, the same principle as numbers-in-code. A
  computed figure is re-verified by re-running its warrant script.
- **Structural + reversible.** Artifacts are versioned and revertable
  (snapshot/revert); a failed run fails cleanly and never half-writes (the
  staged-writes / offline contract, B-ADOPT-12).
- The **execution sandbox** is the open security dependency (§2.3, §8).

### 4.3 The gate is not assumed to work

Per Jacobs 2021, "PI approves" is not automatically more correct than raw engine
output. beta.1 therefore **instruments a calibration check** — are PI-approved
outputs measurably more correct than un-reviewed ones? — feeding §6. If they are
not, the gate is theater, and that finding is a first-class output of the
feedback loop, not a failure to hide.

## 5. Onboarding & the usable-by-a-real-researcher bar

The MVP bar (method): a real researcher uses it on a real corpus and produces a
grounded answer *and* begins producing text/code. Onboarding operationalizes
that bar.

- **Guided corpus seeding.** An onboarding wizard walks the full opening arc of
  the cycle: open a project (research question) → seed → first gap analysis over
  both graphs → first capture → first note → first Ask → first canvas/draft
  gesture. This realizes the deferred walkthrough (B-BORROW-15) and turns the
  abstract cycle into a guided path.
- **Time-to-first-answer: install → grounded answer ≤ 30 min** (B-BORROW-15).
  The contract is *grounded* — a real answer with resolvable citations — not a
  canned demo. beta.1 extends it to a **first WRITE gesture** (a canvas thread or
  a drafted paragraph) within the guided path, since WRITE is half the mission.
- **The sample corpus is REAL, never generated.** The onboarding corpus is a set
  of real, citable sources (papers/repos). Generated corpora are barred by the
  honest-corpus condition (B-BORROW-15) and the no-synthetic-index guardrail —
  a fabricated corpus cannot produce honest feedback.
- **Designed for low burden, because abandonment is the dominant failure mode.**
  Data collection is "burdensome" and the action stage has "few efficacy
  studies" (Epstein 2020, 523-paper survey; RR); proactive personalized-relevance
  is intractable (Rogers 2006; RR). So onboarding minimizes setup, derives
  feedback *implicitly* from gate accept/reject rather than adding a feedback
  chore, and front-loads a fast payoff (the ≤30-min grounded answer). Which real
  sources ship as the seed corpus, and their licensing, is OPEN (§8).

## 6. Feedback instrumentation

"The feedback loop begins" is the definition of the MVP; §6 is what makes it
begin — **honestly, with no synthetic index** (the owner's single biggest
pitfall). The dashboard shows raw counts, dispositions, and attribution events;
it never rolls them into one "system health" score.

- **Disposition capture, implicit.** Every recommendation already carries impact
  level, certainty level, accept/reject consequences, and reasoning
  (propose-not-dispose; owner guardrail). The loop records the **accept/reject
  disposition** against that reasoning. Feedback is *implicit* from the gate the
  PI already operates — no separate rating burden (Rogers 2006; Epstein 2020).
- **Shadow-first calibration discipline.** Every tunable ships behind
  `production_enabled:false`; **no score changes routing, promotion, or
  visibility until it is grounded on recorded dispositions** (A-BORROW-8). The
  loop *collects* dispositions to enable later calibration; it never
  auto-promotes a threshold into production. This is the owner's
  anti-synthetic-index guardrail as an operational rule.
- **Usage & load, because single-PI capacity is unvalidated.** Instrument
  items/session, time-to-decision, override rate, and lapse/abandonment — the
  single-PI sustainable load is an *unvalidated assumption* measured only on
  scripted humans (AutoResearchClaw; RR). Define a **throughput-exceeded
  fallback** (raise the uncertainty threshold; batch low-risk items).
- **Payoff attribution, event-based not scored.** "Improves knowledge work" has
  no efficacy evidence (Epstein 2020; RR). So attribution is a traceable chain of
  events — *recommendation → capture → note → claim → draft passage / code
  warrant* — measured against an **unranked/unaugmented baseline** before any
  benefit is claimed (Epstein; Rogers). Honest events, never a payoff score.
- **Gate calibration (Jacobs check).** Sample PI-vs-engine agreement to test
  whether PI-approved outputs are actually more correct than raw engine outputs
  (§4.3; Jacobs 2021; RR #1). If not, surface it.
- **Guard the loop against itself.** A feedback loop trained on what the PI
  up-votes *suppresses disconfirmations* — the McNamara fallacy: consensus-priored
  models converge and down-rank anomalies (Bisht 2026; RR). So a **separate
  exploration/anomaly/diversity channel is reserved, not subject to any relevance
  ranker**, and **diversity coverage is measured as its own objective.** The
  design of that channel is OPEN (§8).

## 7. MVP acceptance — the definition of done

**beta.1 is accepted when a real researcher, on a real corpus, runs the complete
cycle end-to-end with no core item stubbed:**

1. **READ half (inherited, must run):** open project (question/thesis) → seed →
   gap analysis over **both** graphs → source → capture (immutable raw + content
   + metadata) → extract atomic annotations → develop notes/claims-with-context
   toward the thesis → re-run gap analysis.
2. **WRITE half (this document, must run):** argument graph → **canvas** thread →
   derived **outline** → **draft (text)** grounded in coherent slices with
   deterministic numbers/citations and abstention; **code** artifacts as
   executable warrants and/or deliverables.
3. **Verify:** deterministic citation/number checks, evidence-incomplete/abstain
   routing, graph structural integrity, code-by-execution — sparse and triaged.
4. **Usable bar:** install → grounded answer ≤ 30 min on a **real** seed corpus,
   plus a first WRITE gesture on the guided path.
5. **Loop begins:** dispositions, usage/load, payoff-attribution events, and the
   gate-calibration and diversity-coverage measures are all instrumented and
   producing honest data.

**The anti-deferral rule (binding).** MVP scope is set by user NEED, not deferral
convenience. To move any item out of beta.1 you must **prove the user does not
need it** for viable, feedback-producing use. "Future consumer," "post-beta,"
and "hard to build" are **excuses, not proofs** (method). The argument graph,
hubs, claims, gap-over-both-graphs, the writing module, and the coding module are
all IN — the mission's output half cannot be stubbed and still carry a real user.

**What is legitimately out (proven not-needed for viable use), not deferred core:**
multi-user / team features (single-person tool by identity); cloud sync beyond
single-user-multi-device (local-first is a constraint, not the mission); a
rich **spatial-graph-at-scale** surface (tabular/faceted is the validated default;
the canvas is small-graph and must beat that baseline — RR L23); a **local
fine-tuned / QLoRA adapter** (the frozen local model is only the *test fixture*;
the live engine is an API — RR correction 2026-06-21, so the frozen-model schema
failures do not gate the MVP); **two-way canvas↔outline sync** (§1.3,
simplest-design). Each of these is excluded by a fact, not by convenience.

## 8. Open research questions

Everything below is **not settled by evidence** and must be resolved before
beta.1 is finalized. Each is a genuine unknown grounded in a finding — none is a
placeholder for an invented mechanism. The owner explicitly asked for these.

1. **Code execution sandbox (security-critical).** Executing generated code
   reopens the exec/exfiltration surface least-privilege cannot close (InjecAgent;
   AgentDojo tool-isolation fails in Memoria's conditions; RR §5). What sandbox
   contains a run without a reachable outward-network/exec tool, and how does it
   coexist with the exfiltration-containment linter? Unresolved and blocking for
   §2/§4.2.

2. **Warrant model for derived/multi-hop/implicit claims.** Code-as-executable-
   warrant covers computed numbers, but the general evidence-set model
   (single/multi-sentence/multi-hop/implicit typing, the evidence-incomplete
   state, the PI-routing threshold) is unvalidated. What fraction of drafted
   claims route to the PI? Unknown until measured (FEVER; Buchmann & Gurevych).

3. **Sustainable single-PI verification load.** Auditing costs building-level
   expertise and floods under deadline (Feng & Liu); sustainable load is measured
   only on scripted humans (AutoResearchClaw; RR). The max-items-per-session
   threshold and triage routing need real-user calibration.

4. **Gate calibration (is the gate theater?).** Whether PI-approved outputs are
   actually more correct than raw engine outputs is unknown until §6 instruments
   it (Jacobs 2021; RR #1). If not, the review gate needs rethinking.

5. **Payoff / efficacy.** "Improves knowledge work" has no efficacy evidence
   (Epstein 2020). The proximal-outcome-vs-baseline measure is proposed but the
   actual benefit is unproven **by design** — starting this loop is the point of
   beta.1; the answer is genuinely open.

6. **Outline↔canvas back-propagation.** Committed one-way for the MVP (§1.3);
   whether heavy outline editing creates real demand for outline→canvas sync must
   be measured (does the user re-derive or diverge?), not assumed.

7. **Canvas scale threshold & position governance.** Spatial degrades as density
   grows and persisted positions build the mental model (Hearst; Card & Moran;
   RR L23/L89). The node count at which the canvas stops beating the tabular
   baseline, and how positions are governed as the underlying graph changes, are
   both open.

8. **Coherent-slice construction for synthesis.** "Feed coherent slices, not
   top-k fragments" is the principle (Hu 2026); *how* to build the slice per
   outline node — whole-note vs section-expanded-anchor boundaries, the token
   budget, the expansion policy — is an engineering/eval open question.

9. **Draft → knowledge-graph write-back.** The draft is an output artifact and
   does not auto-write into the graph (no-write-time-oracle). Whether and how
   PI-confirmed draft insights become permanent notes — the promotion workflow
   that would close the owner's post-draft gap-analysis loop — is undesigned;
   propose-not-dispose applies but the mechanism is open.

10. **Exploration/anomaly/diversity channel.** Reserving a channel outside the
    relevance ranker is required to stop the loop suppressing disconfirmations
    (Bisht 2026; RR), but its surfacing design and diversity-coverage metric are
    open.

11. **Workspace / gate topology.** The deep-vs-task boundary and the keep-test are
    committed; the full gate model (Gate 1/2/3, plugin-vs-editor surface,
    active-assistance-per-workspace, where the user asks for help) is rejected as
    authority-carried (A-REJECT-1; owner) and must be redesigned from
    job-to-be-done — the OPEN question inherited from alpha.16.

12. **Code lane agent model.** The owner assigns writing/coding "their own
    agents," but the code lane's posture, skills, and relationship to the
    librarian lanes are undesigned (owner's open profile-division question).

13. **Structured-extraction ↔ free-composition boundary.** NL-intermediate-then-
    deterministic-resolver avoids the negation corruption of rigid schemas (Shin
    2021), but where structured extraction upstream meets prose composition in
    the draft needs an explicit eval boundary.

14. **Real seed-corpus selection & licensing.** The onboarding corpus must be
    real citable sources, never generated (B-BORROW-15). Which sources ship, and
    the redistribution/licensing to bundle them, is an open practical question.

## 9. Acceptance tests

beta.1 is done when a real researcher, on a real corpus, can:

1. Run the full loop — project → seed → **gap over both graphs** → source →
   capture → extract → notes/claims → re-run gap — **at least twice**.
2. Capture from **Zotero, DOI/PDF, BibTeX, and GitHub**, land-then-resolve, with
   no data loss.
3. See a project graph with **claims (with context), questions, tensions, hubs,
   sources, and code artifacts**.
4. Produce a **canvas thread → derived outline → text draft** grounded in
   **coherent slices**, every number computed in code, every citation resolved
   to a **verbatim span**.
5. Produce or curate a **code artifact** with provenance whose check runs **by
   execution**.
6. **Ask** with citations and honest denominator/absence language.
7. Run **output verification** — deterministic citation/number checks,
   evidence-incomplete/abstain routing to the PI (sparse, triaged), graph
   structural integrity.
8. Have **export refuse** unresolved citations and unsupported/superseded/
   retracted claims (blocks, never edits).
9. **Revert a machine run** without changing human text; **backup, restore,
   migration, rebuild** all pass on a clean environment.
10. Reach **install → grounded answer ≤ 30 min** on a **real** seed corpus, plus
    a first WRITE gesture.
11. See the **feedback loop instrumented** — dispositions, usage/load,
    payoff-attribution events, the **gate-calibration (Jacobs) check**, and
    diversity coverage — producing honest data, no synthetic index.
12. Confirm the **keep-test**: the vault is usable as markdown + `.bib` if
    Memoria is removed.

## 10. Trade-offs

beta.1 trades **breadth for a complete loop** — no general web discovery, no
multi-user/team features, no autonomous agents; the mission is one researcher
producing knowledge (text and code) from a real corpus. It also trades
**machine certainty for honesty**: the system can capture, connect, retrieve,
cite, check structure, and refuse bad exports, but it **cannot know an argument
is true** — that is the user's judgment (no-write-time-oracle). The one
constraint that flips a deferral into scope is real alpha.16 use: if the
researcher cannot reach feedback-producing use without a missing piece, it moves
into beta.1; if a feature is merely elegant or historically desired, it waits
(the anti-deferral rule, §7).
