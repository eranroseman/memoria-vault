## 0. Three ruling ideas

Everything else follows from three sentences:

1. **Keep-test rules formats.** Everything the researcher must keep if Memoria
   dies is a markdown file or a PDF inside the vault, navigable in stock
   Obsidian. Everything else lives in a single-file database *outside* the
   synced tree — **derived state rebuildable from the vault; operational
   ledgers backed up and selectively exported, never claimed rebuildable**
   (§5).
2. **A single writer rules safety.** Exactly one code path — `VaultWriter` —
   ever writes vault files on behalf of machines. Every invariant (human-file
   protection, exfiltration neutralization, taint, the hash manifest) is
   enforced there once, never re-implemented per engine.
3. **Runs rule time.** Every machine write belongs to exactly one `run_id`.
   Snapshots, revert, cost, the Review queue, and incidents all key on runs. A
   run commits whole or rolls back whole — with one deliberate exception
   (per-item batch failures, §6) that the charter's safety property does not
   cover.

## 1. Host and process model

**One Obsidian plugin (five surfaces, two of them rich panes — §16) + one
engine binary (`mem` — the CLI and, via `mem serve`, the two-process local
daemon are the same artifact).**

- **The Obsidian plugin is the only researcher UI.** The researcher already
  lives in Obsidian; parity with the null workflow means zero app-switching,
  and the keep-test answer is that every artifact is plain markdown Obsidian
  navigates unaided. A standalone app would duplicate an editor we cannot beat.
- **The daemon is `mem serve`, supervised by the plugin with LSP lifecycle
  semantics (§16), resident while Obsidian is open.** PDF extraction, local
  embeddings, NLI, egress brokering, and the DB do not belong inside Electron;
  a separate OS process gives crash isolation and one network choke point.
  Discovery via a port-file + per-install token in the engine state dir (any
  client may spawn; the workspace write lock keeps one writer); plugin ↔
  daemon over loopback HTTP + WebSocket. The daemon **pushes write events**
  after every applied run and the plugin refreshes affected views — Obsidian's
  file watcher provably misses external writes, and a stale check badge is an
  honesty bug, not a cosmetic one. Initialize-first version/capability
  handshake (refuse on mismatch), parent-death watch, crash restart that gives
  up after five crashes in three minutes and surfaces at block loudness —
  never a silent retry loop. The daemon runs as **two processes** (§2): a
  *supervisor* holding the key, the DB, Guard, Ledger, and VaultWriter; and a
  per-run *engine worker* that is credential-free and socketless.
- **Work survives restarts.** The job queue is DB-persistent; batch-API digest
  jobs poll across sessions. Offline: catalog / search / notes / reading fully
  work; LLM jobs queue and fail cleanly, never half-write (all machine writes
  are staged, §6).
- **Surfaces → the lowest layer that carries them (§16): two rich panes;
  files, commands, and native primitives for the rest.** Capture = a staging
  `Memoria/inbox/` drop folder (files land freely; an "N files awaiting
  import" chip mints imports on the researcher's action — never silently) +
  "Add by DOI / Import Zotero" commands, with progress as a strip on the
  read-only inspector (no dedicated Activity pane); Reading = **Zotero's PDF
  reader for the primary read** (annotations sync back append-only with a
  processing card, §7), with `fulltext.md` as the in-vault evidence substrate
  — a normal markdown view keeping the select-then-click "New anchored note"
  action, which is also the full reading path for non-Zotero works; **Ask** = a
  right-sidebar pane (quote cards, stance chips, span links, gap list +
  denominator); Compose = plain markdown in `projects/` with
  `evidence.md`/`gaps.md` read as files plus embedded views (meaning-only
  projections may use native Bases; verdict-carrying ones come from the
  read-API — §15); **Review** = a dedicated main-pane view opened from a
  ribbon badge showing the pending count. Every surface keeps a no-plugin
  floor — Capture → folder/CLI, Review → run report + `mem revert-run`,
  Compose → files + `mem export`, Reading → stock markdown; Ask deliberately
  has none (below).
- **`mem` CLI = survival + ops + the daemon entry point**: `serve`,
  `capture`/`import`, `export`, `rebuild`, `backup`/`restore --verify`,
  `migrate`, `revert-run`, `cost`, `hat`. **No `mem ask`** — Ask is a judgment
  surface with no survival role, and the keep-test answer is stock-Obsidian
  navigability, not a CLI.

**Surface discipline.** Every notification routes through one loudness ladder —
quiet (logged, weekly digest), notice (pull-only surface), alert (push, never
blocks), block (hold until acknowledged) — assigned by one test: *does it
change what the researcher does in the next 30 minutes?* System health is
ambient: a status-bar indicator plus a read-only **inspector pane** (daemon
liveness, queue depth, recent runs, integrity verdict, budget state, and
**per-store backup status** — vault-git / DB / payloads each configured or
unconfigured, a report that warns before disk-death, never a gate), strictly
read/enqueue-only so debugging never grows a second write surface. Every
surface's empty state names the action that populates it and distinguishes
empty-because-new from producer-never-ran from structurally-can-never-match
(the silent-failure class: for every empty view, prove it can be non-empty).
Machine-written surfaces show "as of <computed_at>" with a **distinct
last-refresh-failed state** — failure is never presented as merely-old data —
and sort on payload event time, never file mtime (a run rewrites everything at
once).

**Review discipline.** Machine proposals (proposed flags, fixable chips,
digest claims) render as **honesty cards** — action, argument-for,
argument-against (the machine's strongest self-rebuttal, the
information-bearing field), what tipped it, coarse certainty — and **no
verdict line on a proposal** (a verdict there is vacuous and induces
automation bias; mechanical *findings* invert and lead with the verdict). Any
report needing dozens of same-shape decisions (the day-1 Zotero import, a
demotion cascade) renders as **one batch worklist** — per-row toggles plus a
single aggregate prompt, never N drip-fed queue items ("forty cards train
select-all-accept"; any item clearable without reading is a design smell).
Dispositions are captured at the review action itself — apply / reject /
defer, with accept-unedited distinguished from accept-after-edit — and machine
findings carry a **content-hash identity**: suppressed while an open item
exists, re-emitted only if the state cleared and re-broke. **A rejected engine
output is not just telemetry: it is auto-added as a candidate case to that
operation's regression fixture** (mirroring the per-incident security-fixture
ratchet, §10) — so the review workflow's sharpest signal, a human catching a
bad output, becomes a permanent guard against that failure mode recurring, not
only a tuning input. A full
awaiting-review queue **throttles new machine work** instead of piling
unreviewed output (the system is cognition-bound: scheduled runs are bounded
by triage batch size, not wall-clock or spend), and no propose-only surface
ships without a measured precision floor and a volume cap.

## 2. Component map

Supervisor process: **JobRunner** (persistent queue; mints `run_id`s; drives
the batch protocol), **Catalog** (DOI/PDF/BibTeX/URL/repo/Zotero → CSL-JSON +
citekey; OpenAlex enrichment + capped cited-by backfill; entity resolution;
annotation sync + the exact-quote anchor bridge; regenerates
`bibliography.bib`), **VaultWriter** (the sole machine writer; also
owns the exfiltration linter), **ProvenanceService** (span attribution,
monotone taint, human-span hash manifest), **Guard** (brokered egress + the
key), **Ledger** (event log, cost ledger, token previews, circuit breaker,
meter/cap), **SnapshotService** (git + `VACUUM INTO`, revert, verify-restore),
**MigrationRunner**, and a typed **DB-RPC surface** the worker calls for index
reads/writes (the worker owns no DB handle).

Engine worker process (spawned per run, credential-free, filesystem = staging +
read-only vault): **Extractor**, **DigestEngine**, **AskEngine**, **Retriever**,
**Checker**, **DiscoveryEngine** (snowball + monitor — deterministic,
identifier-only, stages its report notes like any engine). It talks to the
supervisor over a Unix domain socket / named pipe
with a per-spawn token, through exactly two IPC operations: **brokered egress**
and **DB-RPC**.

Plugin: **SurfaceViews** (typed pane shells, render only; every
list/table/card/worklist *inside* a shell renders through one shared
**ViewSpecRenderer** over the engine's `view-spec.v1` — §15),
**ProvenanceRenderer** (ambient badges/gutters; green requires the in-file
marker, the block's current hash, and the DB `checks` verdict to agree — the
marker alone is never authority, §4), **DaemonClient** (RPC + progress +
daemon supervision).

**Hard boundaries:** engines write to staging only → VaultWriter applies; all
egress passes Guard; every LLM call passes Ledger; nothing in the plugin writes
machine content.

**Operational contract** (the half that historically decayed): every job
carries an idempotency key (`kind:target:payload-hash`) so duplicate triggers
coalesce; retries are bounded and escalate to a visible human-decision state
(unblocking resets the counter; recurring failure archives with a reason —
never an infinite retry loop); JobRunner appends a heartbeat row only after
clean completion, so a missing or stale heartbeat *is* the detector for silent
non-execution; scheduled jobs may only produce logs, DB derived state, and
capped report notes written through VaultWriter as runs — they never modify
existing tracked content. Guard fails closed when its policy cannot load, and
engine workers refuse to start unless Guard and the linter are registered —
fail-closed only holds for a decision that is actually reached. Every denial
is surfaced with the matched rule, never silently retried; a deny-rate spike —
especially immediately after a PDF ingest — is surfaced as an injection signal
(§10). Plugin, daemon, and CLI exchange a version handshake and refuse on
mismatch (stale boundary code waves through what a newer runtime treats as
mutating), and standing negative deny-tests prove each boundary fires on the
installed build, not just in CI.

## 3. Vault layout and formats

```
<vault>/                          # the user's existing vault, untouched
  <existing notes>                # zero migration, zero markers required
  Memoria/
    inbox/                        # staging drop folder — imports mint only via
                                  # the awaiting-import chip (no silent minting)
    works/doe2023attention/
      record.md                   # catalog record; FULL CSL-JSON in a fenced block
      fulltext.md                 # anchored extracted text (machine, untrusted)
      digest.md                   # LLM digest (machine, untrusted, opt-in)
      source.pdf                  # original blob, sha256 recorded
    notes/                        # atomic + fleeting notes (human)
    projects/wm-review/
      project.md                  # question + outline (human)
      evidence.md                 # pinned evidence (machine)
      gaps.md                     # gap report (machine)
      draft.md                    # draft (human class, local-only by default)
      export/                     # export outputs (machine run)
    reports/                      # machine report notes: discovery digests,
                                  # monitor notes, generated queue/index notes
    bibliography.bib              # regenerated BibTeX (portable)
    local-only/                   # folder-level local-only marker
    system/                       # visible, not dot-named: Obsidian Sync
                                  # ignores hidden folders, and audit copies
                                  # must survive device loss (§16)
      manifest.jsonl              # exported human-span hash manifest (audit copy)
      incidents/                  # named incident reports (type: incident)
      metrics/                    # weekly events+costs CSV (evaluation data)
```

**File vs DB vs blob.** Files = records, fulltext, digests, notes, projects,
drafts, bibliography, incidents (everything keep-test). DB = chunks,
embeddings, indexes, links, anchors, events, costs, jobs, runs, manifest of
record. Blobs = PDFs (in vault, sha256 recorded) and embeddings (in-DB via
sqlite-vec). **Full CSL-JSON is serialized into `record.md`** as a fenced code
block (the derived DB is rebuildable, so bibliographic fidelity and
the Zotero re-import key cannot live only in the DB). Report-class notes
(`type: report` — discovery digests, monitor notes, generated queue/index
notes) are **never indexed for Ask and never citable as compose evidence** —
structurally enforced today by fulltext-only embedding (§5), stated here so
it survives any future indexing change.

**The `Memoria/` tree is an OKF v0.1 bundle.** The alpha.15 vault is already
fully OKF-conformant and beta.1 must not regress the exchange format.
Concretely: **every markdown file this design names carries a `type` from the
class enum below** (OKF §9.2) — including the machine-scaffolded human-class
files (`draft`, `ruled-out`, stamped at creation; the body stays human), the
revert-diverted `conflict` files (§6), and `incident` reports; `title` and
`description` ride as the recommended keys, and works carry `resource:` (the
DOI URL, or the pinned repo URL for repo works) — the OKF key for the
underlying asset. The `mem-*` provenance markers and researcher-set decisions
are producer-extension keys OKF consumers preserve and tolerate. The engine
generates the two **reserved filenames as read-only projections** — the
shipped alpha pattern ("generated by projections; edit source Concepts
instead"): `index.md` at the bundle root (carrying `okf_version: "0.1"`, the
one place OKF permits index frontmatter) and per directory, navigation-only
and meaning-only (verdict-bearing views stay on the read-API, §15); and a
root `log.md` projected from the run ledger, newest-first ISO-dated — OKF §7
is a free fit for runs-rule-time, and it puts a keep-test-readable run
history inside the vault at zero new state. Obsidian wikilinks remain the
in-vault link form — OKF §9 conformance does not constrain link syntax. Two
scoping rules keep the claim honest: `inbox/` is **staging, outside the
bundle** (raw artifacts of any format; conformance begins when import mints
the stub), and the bundle boundary is `Memoria/`, not the researcher's vault
— pre-existing notes outside it stay untouched by design (zero-migration),
while a freehand human file created *inside* `Memoria/` without frontmatter
surfaces as a one-tap frontmatter proposal — flags never block, so the
conformance guarantee is "every Memoria-created file, always; freehand files
converge via proposals."

**Frontmatter** appears only on Memoria-created files; user notes need none.
Absence of all `mem-*` markers = human — this is what makes pre-existing notes
zero-migration.

```yaml
---
type: work           # OKF-required key: work|fulltext|digest|note|evidence|gaps|
                     #   project|draft|ruled-out|export|report|incident|conflict
description: One-line summary       # OKF-recommended; feeds index projections
resource: https://doi.org/10.1234/abc   # OKF-recommended on works: DOI, or pinned repo URL
citekey: doe2023attention
title: "Attention and working-memory capacity"
year: 2023
doi: 10.1234/abc
source: zotero       # zotero|inbox|doi|bibtex|url|repo
reading: cite-only   # optional, researcher-set pass-1 disposition (skip|cite-only|read)
mem-prov: machine    # absent ⇒ human
mem-taint: untrusted # monotone; never removed once present
mem-flags: [abstract_only]     # deterministic distrust flags
mem-run: r_20260702T0413Z_7f3a
local-only: true     # optional; enforced in prompt assembly
superseded: true     # optional, researcher-set; Ask excludes by default (counted, overridable)
superseded-by: doe2024attention   # optional successor pointer
---
```

**Fulltext anchors:** one Obsidian block ID per paragraph, assigned once at
extraction, zero-padded, in emission order — plain text, so they survive
reopen, sync, and the death of the tool:

```
…capacity limits predict comprehension under load. ^p0042
```

## 4. Provenance encoding

**In-file: a frontmatter file-default plus trailing block markers. No sidecar
span maps** — sidecars rot the instant the user edits in stock Obsidian;
in-file markers travel with the text they describe and the hash manifest
catches anything that slips past them.

- **File default:** `mem-prov` + `mem-taint` in frontmatter. Absence = human.
- **Block exception** (mixed files — e.g. a researcher-merged digest block): a
  trailing Obsidian comment, invisible in reading view, visible in source mode,
  trivially parseable:
  ```markdown
  - Doe et al. show capacity predicts comprehension: "…verbatim…"
    ([[doe2023attention/fulltext#^p0042]]) %%prov:mc run:r_… t:u%%
  ```
  `prov ∈ h|m|mc`; `t:u` = untrusted-derived.
- **Two independent dimensions**: the **check state** badge
  (none = human, amber = machine-unchecked, green = **citation-correct
  (machine-checked)**) and the **trust class** dot (`t:u` = untrusted-derived)
  are rendered separately. The taint dot is monotone and permanent and **never
  turns green regardless of checking**. `mc` is deliberately named
  **citation-correct**, never "grounded/verified/safe": NLI entailment
  certifies that the cited anchor supports the sentence — it structurally
  cannot certify that the sentence was *derived from* the anchor (Wallat: up to
  57% of RAG citations are post-rationalized). The faithfulness gap is
  documented, not papered over; the byte-substring quote guarantee carries the
  real safety. UI copy reads "verified against anchors," never "safe/approved."
- **Transitions:** machines can never write `h`. `m → mc` only via Checker
  (NLI entailment or deterministic verification). Taint is set at creation for
  anything derived from untrusted origin and inherited by every artifact that
  consumes it.
- **The file marker is a claim; the verdict is DB state.** The `%%prov:mc%%`
  marker is display shorthand, never the authority — the verdict of record is a
  `checks` row (§5) carrying the block's `block_sha256` at the moment it was
  checked. `ProvenanceRenderer` paints **green only when the file marker, the
  block's *current* sha256, and the `checks` row all agree**; if a human edits
  the text under an `mc` marker the hash diverges and the block renders amber
  (`m`) and enqueues a re-check — a human edit changes meaning and can never
  silently inherit a machine verdict. A file-only restore or `mem rebuild`
  (which has no `checks` rows for freshly-restored files) therefore shows amber
  until the $0 local Checker re-earns each verdict — **fail-closed to
  unverified, never fail-open to a stale green**. This is what §11's
  "provenance verdicts DB-side" means concretely, and it is the reason
  `checks` is a derived table (§5): losing it costs recompute time, never a
  false green.
- **Invalidation is its own dimension** (taint can't see it; Memora: obsolete
  reuse is a named memory failure): a **researcher-set `superseded: true`** on
  a work marks it retracted/corrected/obsolete, optionally with a
  `superseded-by: <citekey>` successor pointer. Effect: Ask **excludes** its
  chunks by default, counts the exclusion in the printed denominator ("2
  superseded excluded"), and offers a per-Ask historical override under which
  every surfaced anchor carries a visible **superseded label**; export
  **refuses** on citing it (§8 — the one place the flag blocks, not down-ranks).
  The docs-history record is blunt here: last time the flag shipped
  without a retrieval consumer and was a live correctness bug for months —
  down-ranking is not a consumer. The flag stays researcher-set, but the
  **weekly citation audit also checks Crossref retraction/expression-of-concern
  status for corpus works** (the host is already allowlisted; corpus-scoped, so
  not external discovery) — hits surface as *proposed* flags with human-only
  disposition — and probes whether Ask resurfaces flagged items unlabeled.
- **The human span set** of a file = its blocks whose effective provenance is
  `h` (file default minus block exceptions). The manifest hash is
  `sha256( Σ NFC(block_text)+"\x00" over human blocks in order )` — recomputable
  from the file alone, which is what makes the whole manifest rebuildable.

## 5. Derived store

**One SQLite file per vault, WAL, FTS5 + sqlite-vec, at
`~/.local/share/memoria/<vault-id>/memoria.db`** — outside the synced tree,
per-device, transactional, single-file backup via `VACUUM INTO`. Consolidated
schema (`anchors` was defined twice; the reconciled table names
owners per column group, and `links` is added for the backlink graph two
mechanisms need):

```sql
meta(schema_version, vault_id, tool_version, model_tier, embed_model,
     embed_dim, embed_sha256, nli_model, nli_sha256, embed_wps)
     -- refuse-on-mismatch; weight hashes verified at load (a poisoned local
     -- checker would silently defeat never-green taint)
migrations(n PK, name, kind, applied_ts, git_pre_tag, db_snapshot, status)
works(citekey PK, doi, title, year, csl_json, status, pdf_sha256, distrust_flags)
chunks(id PK, citekey, anchor, anchor_end, text, sha256, tokens)  -- +chunks_fts (FTS5)
vec_chunks(chunk_id, embedding)                          -- sqlite-vec
anchors(citekey, anchor, page, char_start, char_end, bbox_json,
        status, text_sha256, PRIMARY KEY(citekey, anchor))
        -- Extractor writes geometry; ProvenanceService owns status/text_sha256
links(src_path, target, anchor)                          -- backlink graph (Retriever-owned)
runs(run_id PK, kind, started, ended, status, git_pre_tag, git_commit, usd_total)
jobs(id PK, run_id, kind, state, payload_json, error)
manifest(path PK, human_hash, run_id, ts)                -- manifest of record
checks(path, block_id, verdict, block_sha256, run_id, ts) -- mc-authority (§4): the verdict
        -- lives HERE, not in the file marker; green requires marker+current-block-hash+row agree
payloads(sha256 PK, provider, fetched_at, json)          -- content-addressed raw provider
        -- responses (OpenAlex/Crossref); operational-ledger class, backed-up not rebuilt (§7)
events(id PK, ts, verb, hat, payload_json)               -- CHECK verb IN (capture,
                                                         -- read, note, ask, compose, decide);
                                                         -- discovery rounds + intake provenance
                                                         -- ride capture payloads; entity and
                                                         -- triage decisions ride decide
costs(id PK, ts, run_id, purpose, model, prompt_v, in_tokens, out_tokens, usd, artifact_path)
incidents(id PK, ts, name, run_id, detail, resolved)
guard_log(id PK, ts, dest, run_id, pid, allowed, vault_text)  -- egress incl. denials;
                                                         -- vault_text tags any payload carrying vault/document text
edges(src_citekey, dst_key, kind, source, retrieved_at)  -- citation graph: outgoing refs,
                                                         -- capped cited-by, paper<->repo typed edges (§7)
entities(id PK, kind, label, ext_ids_json, status)       -- authors/venues; unresolved-by-default (§7)
entity_mentions(citekey, entity_id, raw_name)            -- resolution decisions live in events
annotations(zotero_id PK, citekey, quote, color, page, anchor, status,
        disposition, date_added)                         -- append-only Zotero sync; orphan
                                                         -- status + triage dispositions (§7)
discovery_seen(ext_id PK, first_seen, disposition)       -- snowball/monitor seen-rejected ledger (§7)
```

`mem rebuild` re-derives `chunks`/`chunks_fts`/`vec_chunks`/`links`/`manifest`,
**recomputes the `anchors` geometry map by re-running deterministic local
extraction against `source.pdf`** (anchors are otherwise DB-only,
breaking "100% rebuildable"), and **re-earns every `checks` verdict via the
local NLI Checker** — until it does, restored `mc` markers render amber (§4).
It prints an estimate first ("≈18 min, $0.00 —
local models only") and re-pays nothing, because extraction and digests are
files (and re-checking is the $0 local model). **Only `works.fulltext` (plus opt-in digest chunks) is embedded** —
notes and projects are hashed for the manifest but not embedded; embedding the
user's whole personal vault buys no beta.1 feature and creates an oversized DB.

**Stage state stays DB-side.** Machine pipeline state (`works.status`, jobs,
edges, entity status) is never materialized into frontmatter — the
machine-frontmatter pattern is a standing reject with three documented
generations of decay — and a **periodic reconciliation pass** re-derives it
from the artifacts themselves (digest exists ⇒ digested; fulltext exists ⇒
extracted), flagging divergence into Review rather than asserting truth by
construction. The only pipeline-adjacent frontmatter is **researcher-set,
meaning-bearing decisions** (`reading`, `superseded`), which meaning-only
doctrine permits. Bases-visible machine state renders through generated
queue/index notes and the read-API (§15), never per-work YAML.

**Two DB classes, stated (the ruling idea in §0 depends on this split).**
*Derived state* — chunks/FTS/vectors, links, anchor geometry, edges,
entities, `works.status`, `checks` verdicts — is regenerable from vault files
and providers via `mem rebuild` (fail-closed: a missing `checks` row renders
amber, never a stale green). *Operational ledgers* — `runs`, `jobs`, `events`,
`costs`, `incidents`, `guard_log`, and the `payloads` raw-provider-evidence
store — are **not rebuildable and are not claimed to be**:
they are preserved by `VACUUM INTO` backups (the content-addressed `payloads`
JSON sits beside `memoria.db`, same per-device data dir), and their safety-
and
evaluation-relevant slices are exported into the vault (`system/manifest.jsonl`,
`system/incidents/`, the weekly `system/metrics/` events+costs CSV — which
includes `decide` events, so no researcher judgment such as an entity
same-as decision lives only in the DB). Total DB loss therefore costs
derived-state recompute time plus ledger history since the last backup —
never vault content, never exported evaluation data.

## 6. Snapshot / revert substrate and the batch protocol

**Git for the vault (external git-dir) + `VACUUM INTO` for the DB** — both,
because they version different things. The git repo is Memoria-owned:
`git --git-dir=~/.local/share/memoria/<vault-id>/vault.git --work-tree=<vault>`
— an external git-dir means no collision with the user's own vault repo and no
`.git` churn in their file sync, yet still plain git for forensics.

**The batch protocol** (every run — the run-kind enum is
`capture|extract|digest|import|ask-materialize|export|migration|merge|revert|discovery|rename`,
adding `capture`, `export`, `discovery` — the monitor and snowball notes
are vault writes, so they are runs like any other — and `rename` for citekey
correction, §7):

1. **snapshot** — JobRunner mints `run_id`, quiesces VaultWriter, tags
   `pre/r_…` (git commit-all); `VACUUM INTO` a DB snapshot for import/migration
   kinds only.
2. **apply** — engines have staged their outputs; VaultWriter applies
   file-by-file under a per-file lock: **block-level** human-content protection
   (VaultWriter may modify only machine-class blocks and their
   trailing markers; human blocks are byte-immutable, which is what lets
   migration and Checker touch mixed files at all), exfiltration
   neutralization, local-only respect, marker stamping. **Stale-base check:** if
   a live file's sha256 ≠ its base at snapshot (the user typed during the run),
   that file is skipped and requeued — machine writes never clobber concurrent
   human keystrokes.
3. **verify** — ProvenanceService recomputes each touched file's `human_hash`
   **per file, immediately before vs after its own write, under the lock** (so
   a legitimate concurrent human edit is not a false incident); any
   mismatch ⇒ `git reset --hard pre/r_…`, incident `INC-r_…-manifest`, run
   `failed`.
4. **commit** — **pathspec-limited to the run footprint** (human
   edits made during the run stay uncommitted and are swept by the next run's
   pre-tag, so the inverse patch can never contain human hunks), trailer
   `Mem-Run: r_…`; export `manifest.jsonl`; enqueue Review items.

**Partial-success rule (the one deliberate exception to whole-run atomicity):**
inside a digest/extraction batch, per-item failures do **not** roll back the
run — successes commit, failures stay retryable, run status
`completed_partial`. Whole-run rollback is reserved for **invariant breaches**
(manifest mismatch). Atomicity protects the safety property (no human text
altered), not job luck; rolling back 32 good digests because 2 timed out would
be hostile and re-billable. `completed_partial` is a first-class `runs.status`.

**One-click revert** (Review, or `mem revert-run`): preview = the rendered
inverse diff of the run commit, dry-run-classified per file
(`clean-restore | delete-created | conflict`). Execution is itself a run: apply
inverse hunks; a file a later human edit overlaps is left untouched and the
inverse hunks go to `<name>.conflict.md` (`type: conflict` — a bundle concept
like any other) + an incident (never silent, never lost); then a **direct byte comparison** of captured human-block bytes vs
post-revert bytes — any difference aborts the revert. Cost rows are never
deleted; money spent stays on the meter. **Byte-identical human text has three
legs:** the original commit passed the manifest gate (its footprint provably
contained no human-span change), later overlapping human edits divert to
`.conflict.md`, and the runtime byte-check converts the argument into a check.
**Revert is DB-complete via reconciliation, not snapshots**: after the file
inverse-apply, the revert run enqueues and awaits a **footprint-scoped DB
reconciliation** that re-derives every derived table (§5: `works.status`,
chunks/FTS/vectors, links, anchors, edges, entities) from the post-revert
files — no `record.md` ⇒ no work row and no orphaned edges. The operational
ledgers (`runs`, `jobs`, `events`, `costs`, `incidents`, `guard_log`) are
**intentionally untouched by revert** — history appends, never erases — and
the revert appends its own run row. Per-run DB snapshots stay reserved for
import/migration kinds; snapshotting every run would be write amplification
in service of inverting ledgers the design defines as append-only.

## 7. Ingest (Capture → Extraction → Digest)

**Work lifecycle** (one `works` row; `status` scalar + `distrust_flags` array;
**partial states are legal resting states**): `new → cataloged → {no_pdf |
pdf_stored} → extracting → {extract_failed(reason) | extracted → indexed}`,
then opt-in `indexed → digest_queued → digest_submitted → {digest_failed |
digested}`. `indexed` is the **payoff state** — Ask runs over raw fulltext at
$0, so a grounded answer never waits on digests. `unsearchable` is a **derived
predicate** (no `fulltext.md`: `no_pdf` or `extract_failed`), *not* a stored
status — abstract-only works are extracted, indexed, and searchable with the
flag surfaced on evidence, honoring "flags never block".

**Inputs — land, then resolve.** Every input (file, ID, BibTeX entry, URL,
repo) lands **immediately as a catalog stub** with its raw artifact attached —
phase 1 never fails; resolution is a separately visible, retryable step
(failure is a triage stub, never data loss). Resolution is identifier-first:

- **IDs** (batch-pasteable DOI / arXiv): DOI → `citation.doi.org` content
  negotiation (registry-agnostic CSL-JSON across Crossref *and* DataCite;
  pinned metadata endpoint, redirects disabled); arXiv IDs map to their
  DataCite DOIs (`10.48550/arXiv.<id>`) — same channel, zero extra hosts.
- **PDF** (from staging or Zotero): DOI/arXiv regex over pages 1–2 (the
  Crossref regex covers ~97% of live DOIs); registry metadata fetched per
  candidate and accepted **only if the registry title fuzzy-matches the PDF
  text (≥0.95)** — untrusted PDF text *nominates* identifiers, a trusted
  registry *authors* metadata: injection-resistant with zero LLM calls. No
  identifier → Crossref `query.bibliographic` with validation scoring (the
  outbound title string is extracted from an untrusted PDF, so it is
  length-capped, stripped of URLs/markup, and journaled verbatim); below
  threshold → `needs_metadata`, a fixable chip, never a guess.
- **BibTeX file**: local parse; each entry a stub through the same ladder —
  the manager-agnostic import path; records-only until PDFs attach.
- **URL**: structured origins only — DOI landing pages, arXiv abs pages,
  GitHub repos. No per-site scrapers, ever (Zotero's ~600 translators need
  daily CI to stay alive; capture from the wild web stays Zotero's job).
- **GitHub repo** (local clone path, or URL via `api.github.com`): metadata by
  the `CITATION.cff → codemeta.json → Zenodo-DOI → API fields` chain,
  recording which level supplied each field; pinned HEAD commit SHA +
  retrieval date; README + docs become `fulltext.md` (already markdown —
  anchoring is trivial); code files stay inert (§8). **Paper and repo are two
  works joined by a typed edge, never merged.**

**Zotero import** (day-1): Zotero 7 local HTTP API
(`127.0.0.1:23119`, Guard-allowlisted) for CSL-JSON records (native → cheap
≥95% fidelity) + attachment paths; PDFs copied from Zotero storage via
filesystem, not network; the Zotero key stored in `csl_json.custom.zotero_key`
for idempotent upsert; re-import never clobbers a `record.md` whose hash
diverged from the last machine write (skipped, noted in Review). One import =
one run; extraction jobs enqueue as follow-on runs; digests are **not**
auto-queued. **PDF acquisition:** the Zotero-captured attachment first;
otherwise the work rests at `no_pdf` with a one-click **"capture in Zotero"
handoff** — Zotero is already the designated capture surface in the null
workflow, so this adds zero egress; a Memoria-side OA-PDF fetcher stays out
(an auto-fetch at import time is both an allowlist violation and a
zero-interaction beacon), with named OA hosts (arXiv, Europe PMC) the
recorded fallback if the handoff measurably fails.

**Import fragility rules** (each a documented historical failure): citekeys
are **pinned at import** — **inherited from Better BibTeX when the Zotero item
carries a pinned key** (one citation vocabulary across the researcher's old
and new notes; the key lives in Zotero's extra field, so it survives even if
BBT dies), else minted BBT-pattern-compatible in the same collision-checked
namespace; correction is a first-class revertible rename run
(directory rename + link/anchor/manifest fix-up in one run) — and work paths
derive via collision-safe encoding (two keys must never collide to one path on
a case-insensitive filesystem). Identity conflicts (conflicting
DOI/identifiers) always flag as `needs_metadata` and never silently merge;
auto-merge only on an exact identifier match. A failed import (no PDF, API
down, parser garbage) materializes a **visible triage stub** at the job's
give-up signal — no captured source is silently lost. After a bulk import
commits, the plugin **awaits Obsidian's metadata-cache idle** before surfaces
read (a 10k-file cold parse runs ≈ a minute; surfaces briefly rendering zero
results reads as data loss on day 1). `bibliography.bib` stays
Memoria-generated from stored CSL-JSON — complete across non-Zotero works
(repos, direct DOIs) — with the researcher's own BBT auto-export as an
optional cross-check, never the artifact of record.

**Enrichment (corpus-scoped, $0).** At import, one OpenAlex singleton per
work (by DOI/arXiv) fills **empty fields only** — the JabRef merge rule:
machine fills blanks, conflicts render as per-field left/right diffs through
Review, never auto-applied — and stores the outgoing reference list as
citation edges (§5 `edges`). **The raw provider response is preserved
verbatim** as a sha256-named JSON blob in `payloads` (§5), and every enriched
field / edge carries that hash: so a merge or normalization is replayable and
auditable, and when a provider later mutates its record the original evidence
the researcher's facts were derived from is still on disk — the retention
evaluation can audit provider-derived facts after the fact. The store is
operational-ledger class (backed up, not rebuilt); it holds distilled facts'
provenance, not a second copy of the vault. An idle background job backfills **capped
cited-by edges** (top-N citers per work, N a logged shadow knob; per-run
ceiling; no forward-of-forward expansion), and every consumer of those edges
stamps the cap ("among top-200 citers per seed"). Every enrichable field is
**three-valued** — present | absent-in-source | not-yet-fetched — and counts
render with provenance and as-of dates ("41 citations — OpenAlex,
2026-07-02"), never bare. No background metadata auto-refresh, ever; refresh
follows attention (work-page open). Citation-facing fields the
researcher's Zotero owns are corrected **at source** (deep link to the Zotero
item; the next sync confirms) rather than locked vault-side — a vault fork of
bibliographic truth would diverge from the keep-test bibliography. Payloads
are identifiers only; provider-returned abstracts/titles are author-controlled
text and cross into prompts only as fenced data and into the vault only
through the exfiltration linter.

**Entities (DB-side; no machine-minted pages).** Authors and venues derive
from CSL-JSON + ORCID into DB rows (§5); the graph is queries, never vault
sprawl — machine-spawned person/venue pages remain a standing reject.
Resolution policy, in order: a **name-incompatibility veto** blocks any merge
regardless of other signals; an **identifier conflict** (two ORCIDs) is a
hard cannot-link; shared ORCID + compatible names **auto-merges with recorded
provenance**; everything else rests as **`unresolved` by default** — counted
honestly in any consumer's denominator, and surfaced for adjudication **only
when a downstream consumer (a discovery seed, a work-page open — hubs and
mapping views are post-beta.1 consumers) actually
touches that entity**. Lazy adjudication at the moment of use, never a
standing clerical queue — the arithmetic: ~4–5 authors × 10 imports/week ≈ 45
mentions/week, and a day-one backfill mints thousands; a standing queue is
the week-3 abandonment mechanism wearing a rigor costume. Venue merges on
ISSN-L only. Researcher same-as / not-same-as decisions are recorded (events
+ DB) and never overwritten by any sweep; person pages exist only when the
researcher writes one.

**Extraction.** **pdfium alone** (BSD-licensed, native, embeddable,
matching the single-static-binary install; PyMuPDF's AGPL + Python contradicted
it, and the dual-engine fallback doubled the test surface for no charter
benefit). No OCR: scanned PDFs fail named as `extract_failed:needs_ocr`.
De-hyphenate → paragraph-segment (font-size/indent) → headings by font
percentile → linearized markdown with `^pNNNN` per paragraph. Before any
`fulltext.md`/anchor write the output must pass a cheap text-coherence gate,
failing closed to `extract_failed:garbled` — garbled text otherwise silently
poisons FTS, vectors, and quote verification. (Repo works bypass pdfium —
their docs are already markdown and linearize directly.) Deterministic
distrust flags: `truncated` (extracted pages < PDF pages, or >30% of pages
< 200 chars), `abstract_only` (< 600 words, or ≤2 pages for an
`article-journal`). Indexing (follow-on local job, $0): anchor-aligned chunks →
`chunks`/`chunks_fts`/`vec_chunks`.

**DigestEngine** (opt-in, always behind a dollar preview). Pinned prompt
`digest_prompt_v1`; input = record core fields + anchored fulltext clipped to
the price-table input ceiling by **one clipping algorithm**: abstract +
first/last sections + evenly-sampled middle, flag `input_truncated`.
Structured JSON output: `{summary, claims:[{text,kind,anchors[]}], methods,
key_terms}` with `kind ∈ {context, key_idea, method, outcome, limitation}`
(MASSW-aligned; `projected_impact` excluded — speculation, not extraction);
every cited anchor must appear verbatim in the input. Long papers are clipped,
not recursively condensed — single-pass clipping is honest and bounded;
recursive batch-condense earns its multi-call complexity only if the beta shows
the clip losing content (a deferred observation, not a launch assumption).
Deterministic validation before staging: schema check; claims citing a
non-existent anchor are dropped (`anchor_invalid`); >20% dropped ⇒
`digest_failed:anchor_hallucination`. Distrust inheritance:
truncated/abstract-only/input_truncated ⇒ the digest carries the distrust flag.
Cost ceiling from the **normative price table (§9)**; per-batch **circuit
breaker owned by Ledger** (computes and decides), executed by JobRunner
(cancels remaining items as `cancelled(budget)`, one canonical state name).
Batch protocol: one digest batch = one run via the provider
Message-Batches endpoint (50% price, applied in previews); poller survives
restarts; per-item partial success per §6. Checker then NLI-tests each claim:
the claim is **statement-converted** first (deterministic; a failed conversion
degrades to *unknown*, never a guess) and entailed against the **concatenation
of all anchors it cites**, not one at a time (multi-span support is real —
FEVER). Entailed → `mc`, else `m` with `nli_unsupported`; taint stays
`untrusted`. **The thresholds are not literals**: they are tuned on a small
pre-registered claim/anchor calibration set (a few dozen hand-labeled
corpus-like pairs — one afternoon, capped; Sainz: ~15-point swing from tuning
alone), seeded with the measured NLI failure classes — same-topic /
different-variable pairs, which NLI confidently mints as contradictions and no
threshold filters (route to *unknown*), and high-lexical-overlap /
opposite-meaning pairs — and then frozen; and the Checker must pass a **HANS/PAWS-style
high-overlap negation + direction-reversal probe set as a launch-blocking
acceptance test** — a floor, not a certificate (probes are adversarial, not
representative).

**Corpus seeding** (onboarding wizard): import/drop → auto extraction+indexing
($0, live trace) → an optional cheap **yes/no relevance prefilter** over the
seeding batch (skip works irrelevant to the live question before spending) →
"Digest N works: est. $X (batch)" single confirm →
**prompts the first Ask immediately, before digests return** (Ask needs only
`indexed`). This is what meets install → grounded answer ≤30 min. The wizard
also offers a **half-built sample corpus**: ~8 real, citable sources (never
generated — a faked seed models the exact anti-pattern the product prevents)
pre-distilled into a dense cluster with a planted tension, a named gap, and
2–3 items deliberately left unfinished for the researcher to complete; it
loads idempotently, removes cleanly by archive label, and doubles as the
end-to-end test fixture and the demo.

**Reading and annotation (Zotero-first).** The primary read happens where the
researcher already reads: Zotero's PDF reader. Annotations sync one-way,
append-only, keyed by Zotero annotation ID with a `dateAdded` watermark; each
lands with its verbatim quote, color, page, and a `zotero://open-pdf` deep
link, and is **bridged to the evidence substrate by exact-quote match** — a
verbatim hit in `fulltext.md` earns the annotation its `^pNNNN` anchor;
misses are counted per work as orphans ("exact quote not found verbatim"),
never silently dropped (fuzzy re-anchoring is Hypothesis-grade engineering,
deferred until a measured orphan rate demands it). The researcher-set
`reading` disposition (skip / cite-only / read) is set via a designated
Zotero tag/color or on the work page — meaning-bearing frontmatter, never
machine state. When a sync appends ≥N new highlights to a work (N a shadow
knob), the next vault open shows one dismissible **processing card** —
"Process 14 new highlights on [work]" — opening a batch triage worklist (§1
review discipline): per row **promote / link / drop / snooze**, promote
opening a pre-typed note stub whose machine pre-fill **stays machine-class
(`m` markers) until the researcher edits it — its taint is monotone and never
clears (§4)** — a promoted note with
zero human-typed characters is flagged in Review, because one-tap promotion
of machine-drafted sentences would launder untrusted-derived prose into the
human class. The research is unambiguous that initiation, not execution, is
where annotation-processing collapses — the card is the starter motor.
Per-work honesty line: "processed 9/24"; the global backlog surfaces only
contextually, never as a guilt inbox.

**Discovery (bounded citation-graph only; the charter's discovery exclusion
is amended accordingly).** Two mechanisms, both $0, both identifiers-only
egress (`guard_log.vault_text = false`), both structurally separate from Ask
— discovery output never enters gap or coverage denominators:

- **Snowball** (human-triggered): backward from stored reference edges +
  forward from the capped cited-by cache; depth ≤ 2, per-run candidate
  ceiling, ranking by transparent occurrence counting across multiple seeds
  (visible arithmetic, never a black-box recommender — that is the
  echo-chamber counter), every line cap-stamped ("appears on 6 paths from 4
  seeds, among top-200 citers per seed"). Results land as one capped digest
  note (10–20 items) with per-item provenance; dispositions import / reject /
  defer are logged; accepted items enter the normal import ladder; a
  persistent seen/rejected ledger stops resurfacing. Each run appends a
  PRISMA-S-style line to `events`; **continuation past 1–2 rounds is a human
  action**, framed by per-run gain ("this round: 3 new"). Coverage wording is
  index-relative — "declining yield within OpenAlex as of <date>" — never
  "of the literature."
- **Citation monitor** (scheduled): "N new works cite M of your catalog
  papers since last run," one capped machine note per schedule, written
  through VaultWriter's exfiltration linter — a poisoned citing-paper title
  is a **permissionless ingress channel** (T5, §10) and must land
  neutralized.

Intake provenance (gap-driven vs researcher-initiated) is logged from day 1
for the evaluation's payoff attribution.

## 8. Ask, Compose, Export

**Indexing.** Chunk = anchor (1:1); coalesce <40-token paragraphs forward,
window >480-token paragraphs at ≤320/64-overlap sharing the parent anchor.
Local pinned models, bundled by the installer, **never downloaded at runtime**:
embeddings `bge-small-en-v1.5` int8 ONNX (384-dim); NLI
`deberta-v3-base-mnli-fever-anli` int8 ONNX (the same instance Checker uses).
Pins live in `meta`; mismatch ⇒ refuse with `mem rebuild --embeddings`.
Incremental on every capture commit: re-chunk, diff by sha256, embed only
changed chunks, delete-then-insert in one transaction (a crash leaves the old
index intact). Note: **no digest-chunk rank boost** (a tuning knob
with unmeasured benefit in a pipeline whose selling point is "nothing to
tune"); evidence text is drawn from fulltext chunks only, always.

**Ask flow** (all local, $0): (1) **probe derivation** for stance —
deterministic subject-aux inversion of yes/no questions; wh-questions get a
"state as a claim" field; no probe ⇒ every item is `mentions`, never a
fabricated stance. (2) hybrid retrieval — FTS5 BM25 top-100 ∪ vector top-100,
fused by RRF (`Σ 1/(60+rank)`, no learned weights), per-work cap 8, MMR to 40
working chunks; chunks from `superseded: true` works are excluded by default,
counted in the denominator, and surfaced only under the per-Ask historical
override with the superseded label (§4). (3) verbatim quote = best 1–3 contiguous sentences ≤60 words,
**asserted to be a byte-substring of the anchored block** (fail ⇒ drop +
re-index). (4) stance via NLI (premise = quote, hypothesis = probe; thresholds are the
**calibration-set-tuned, frozen** values from §7, not literals; entail →
supports, contradict → contrasts, else `mentions` — and **`mentions` means
*unknown stance*, never "no contradiction"**; probability on hover, never
aggregated). Contrasts render as **"contradictions found *within the retrieved
set*"** — an empty contrasts list is never presented as absence of
disagreement (MRMR: contradiction retrieval is near chance on negation; an
empty list would be an implicit agreement meter, the thing the aggregate-score
exclusion exists to prevent). (5) evidence set (≤25, stable `ev-<8hex>` ids — **dash-form
block IDs**, so `^ev-9f2c41d8` is a valid Obsidian anchor). (6)
**gaps with a printed denominator** — coverage gaps itemized by cause, with
truncated/abstract_only works counted as **suspect, not covered**: "Searched
38 of 46 works fully — 3 unsearchable (2 failed extraction, 1 no PDF), 3
partial (truncated/abstract-only), 2 superseded excluded" — and content gaps
(question facets with
corpus df 0 or no surviving item); a printed **year-spread line** for the
retrieved anchors (the cheap temporal-coverage signal; TEMPO). Zero survivors
⇒ the banner reads **"nothing matched lexical/semantic search over N
documents"** — a searched-set claim, never corpus ontology (BRIGHT/HotpotQA:
reasoning-linked and second-hop evidence is missed at measured high rates; the
retriever cannot support an ontological "corpus has nothing"). A **"search
deeper" affordance** runs one deterministic second-hop expansion (entities
extracted from first-round hits re-queried) — opt-in per Ask, never a default
cost on the $0 path, no tuning surface. An Ask is an `events` row; nothing is
written unless the researcher pins.

**Optional paid synthesis** — a button labeled with its computed price
("Synthesize — $0.04"), never a default. Prompt = system + question + the
numbered evidence items, each **expanded to its enclosing section** — a
deterministic coherent slice, anchor citability preserved — rather than the
bare 1–3-sentence quote (Hu: top-k fragments measurably degrade the
integrative answers the payoff claim lives on; section expansion captures most
of the coherence gain at near-zero complexity; iterative re-retrieval is
**deferred** unless the paired ChatGPT counterfactual still wins on coherence).
Still quotes+citekeys+sections only — **never whole documents, notes, or
drafts, so local-only material is structurally absent**; ceiling from §9;
streamed usage feeds the circuit breaker. Output cites only `[En]`; uncited
sentences get a visible `⚠ uncited` flag (flagged, never blocked); **the
rendered synthesis carries the same per-quote stance chips and coverage
denominator as raw Ask — never a neutral narrative overview** (a smooth
unlabeled summary is the aggregate-meter failure wearing prose); Checker
upgrades entailed sentences to `mc`; "save to project" routes through the
normal run as machine blocks with `t:u`. Synthesis-shaped Asks are
**instrumented separately** so an exit verdict can attribute a payoff failure
to evidence granularity rather than grounding.

**Compose.** `project.md` (human: question + outline citing evidence/spans),
`evidence.md` (machine, one anchored block per pinned item, idempotent by id),
`gaps.md` (machine, regenerated wholesale, opens with the denominator),
`draft.md` (**human class, `local-only: true` by default** — machines never
write it; its blocks join the manifest), an optional `ruled-out.md` (human:
decision points with rationale — "considered X as the organizing frame;
rejected because…" — the dead-end memory the ChatGPT null workflow also lacks;
never machine-written, never auto-promoted), and inert code files (listed in
the panel, never executed, never indexed, never in prompts). **Export is a run**
(— it writes vault files, so it needs a snapshot to be revertible):
collect `[[citekey/…#^…]]` and `[@citekey]` citations → **the export
readiness gate refuses (never merely warns) on any of:** an unresolved
citation (citekey in `works`, anchor in `chunks`, falling back to a file
scan); a cited work flagged `superseded`/retraction-flagged (export is the
one place the flag must *block*, not down-rank — a submitted draft citing a
retracted work is the exact harm the flag exists to prevent); or a mapped
project claim with no `mc`/Checker-verified anchor **and** no explicit
gap-analysis missing-evidence marker (an unwarranted claim leaves the vault
only as a knowingly-marked gap, never silently). Each refusal names the
offending list with jump links. On pass → rewrite
span links to `[@citekey, para N]` → emit `export/draft.md` +
`references.bib` (from stored CSL-JSON, cited works only) → VaultWriter's
exfiltration linter re-runs over both. Its provenance: machine class, local-only
inherited from `draft.md`. (The gate blocks; it does not *edit* — softening a
superseded citation or filling a gap stays a human act, per no-write-time-
oracle.)

## 9. Cost machinery

BYOK; key in the OS keychain, never in vault/exports/logs (canary-drilled,
§10). **The normative price table lives in one place — Ledger — and every
component references it, never restates constants** (three
components had diverged). Each operation is bounded by a `max_tokens` ceiling so
its worst-case price is arithmetic, not hope. Token previews use a **local
deterministic estimator with a stated safety margin** (the
provider's production tokenizer is not public; the `max_tokens` ceiling supplies
the exact worst case, so previews are honestly labeled bounded estimates).
Batch API on ingestion (50% savings applied in previews; **cache savings
budgeted as zero**); one pinned model tier for the whole beta (a mid-beta tier
change is a release marker, never silent). Every run records the exact model id
**and prompt-template version** (`costs.prompt_v`); **any model or prompt pin
change re-triggers validation** of the affected engine against its fixture
before it runs on the real vault — a prompt edit is an unevaluated model swap
(Perez/Lu/Zhao: prompt variance dominates). Confirmation only when a single
action >$1 or the projected month crosses the cap. **Meter/cap enforced three
times:** at preview (refuse enqueue), pre-call (recheck), and the circuit
breaker (Ledger decides on streamed/polled usage, JobRunner executes
cancellation). Default soft budget $20/mo, researcher-set hard cap.
Unattributable spend is never zero: provider-reported usage that cannot be
tied to a run becomes an explicit **miss row**, and the breaker treats missing
usage data as over-budget evidence — a breaker computed over
silently-incomplete data fails exactly when a provider changes its reporting.

## 10. Security

Threat model: (T1) injection-steered output exfiltrating vault content via
markdown that fetches/resolves a URL; (T2) untrusted PDF/metadata text becoming
active content in Obsidian; (T3) key leakage into any artifact; (T4) engines
contacting non-allowlisted hosts; (T5) third-party content entering the vault
with no researcher action — the citation monitor's poisoned-title channel
(§7), neutralized at write by the exfiltration linter. **Injection
classifiers are excluded by charter — we contain outputs, we don't detect
intent** (>90% adaptive bypass is the documented state of the art).

- **Brokered egress, not a proxy.** Engines get **no sockets at all**; all
  egress is a typed IPC call to Guard, which performs the TLS request itself,
  attaches the key at the socket, streams usage to the breaker, and logs every
  request (allowed and denied) to `guard_log` — **not to `events`, whose verb
  set is a schema-enforced closed verb enum (§5)**.
  Compiled-in allowlist (no config surface), every entry requirement-driven
  with its own audit: the pinned LLM host; `api.crossref.org` (SBMV title
  fallback, retraction check); `api.openalex.org` (enrichment backbone +
  capped cited-by backfill + discovery — identifier-only payloads,
  `guard_log.vault_text = false`); `api.github.com` (repo import);
  `citation.doi.org` (registry-agnostic DOI content negotiation across
  Crossref and DataCite — **pinned metadata endpoint, redirect-following
  disabled**; general `doi.org` resolution stays out because redirect-follow
  lands on arbitrary hosts); Zotero localhost. **No OA-PDF fetcher** —
  acquisition goes through the Zotero capture handoff (§7), zero new
  egress. Any dest outside
  the enum, or any raw-URL field, is a logged denial with a typed `EgressDenied`
  error to the engine. The two scheduled egress consumers (cited-by backfill,
  citation monitor) are capped, journaled, and produce only DB derived state
  and capped report notes — exactly what §2's scheduled-jobs contract permits.
  Deferred named entries under the same entry-plus-audit rule: arxiv.org /
  europepmc.org (the recorded OA-PDF fallback, §7), Software Heritage, NCBI.
- **OS confinement: Linux network-namespace only**. The
  supervisor/worker split with brokered egress *is* the charter's "allowlist
  mechanism"; on Linux the worker additionally runs in a `CLONE_NEWNET`
  namespace with no interfaces — cheap, kernel-enforced, and literally the
  charter's "network namespace" wording. The seccomp fallback, macOS
  `sandbox-exec` profile, and Windows-Firewall/UAC flow are **cut** — two of
  them bought near-zero enforcement for real code and a 3-OS sandbox matrix.
  SECURITY.md states the per-OS truth plainly: on macOS/Windows, egress
  restriction is process-architecture discipline, not OS enforcement; on every
  OS the worker never holds the key and everything it writes passes the linter.
- **Exfiltration linter, VaultWriter-owned** (one owner), default-
  deny strip-never-flag, applied to every machine-writable class at apply and
  re-run over the whole bundle at export: the only surviving link shape is
  `https://doi.org/10.…` with no query/fragment/userinfo; every other URL →
  inline code span; `![…]` embeds get `!` escaped; raw HTML stripped;
  frontmatter serialized from typed fields only. **Untrusted-origin text is
  neutralized at vault-write for extracted fulltext and imported metadata too**,
  not just engine notes — a beacon in a PDF body must not survive into a
  rendered reading view.
- **Third-party plugin interference is a named corruption class:**
  format-on-save style plugins silently rewrite machine spans, flip markers,
  and trip rollback loops. Startup detects known-hostile plugins and settings,
  registers Memoria-owned paths as formatter-excluded where the plugin
  supports it, and audits plugin `data.json` files for secret-shaped fields
  before the vault snapshot captures them; the human-span manifest gate
  remains the structural backstop.
- **Prompt assembly:** untrusted text enters only inside nonce-delimited data
  blocks (nonce escaped inside the text so closure can't be forged; system
  prompt: data blocks are quoted material, never instructions). Vault text is
  **stripped of frontmatter, `%%prov%%` markers, taint tags, and CSL fences
  before it enters any prompt** — models quote machine metadata as source
  content; citations that fail to resolve degrade to plain text and log a
  resolution failure. Template
  standard — **construction hygiene, explicitly not a security control**
  (ordering is not a defense; containment carries security): instructions and
  output constraints restated *after* every untrusted block, stop sequences
  set, no chain-of-thought scaffolds over untrusted text (Shaikh: CoT degrades
  refusal), and list-shaped outputs use existence-gating plus an explicit
  abstain option (Agrawal: models invent non-trivial answers where none exist); **local-only
  exclusion is fail-closed** (frontmatter flag, `local-only/` residence, or any
  marked ancestor; unparseable frontmatter ⇒ treated as local-only; transitive
  pulls resolve the target's marker; exclusions counted in the coverage
  denominator); the key exists only in the keychain and Guard's memory, raw
  prompts are never persisted (Ledger stores a payload hash).
- **Keys** in OS keychain per platform (Keychain / DPAPI / Secret Service, with
  a mode-0600 fallback + logged warning on Linux). **One canary spec, owned by
  Security** (two incompatible canaries existed): `MEMORIA-CANARY-<8hex>`
  minted at install beside the key; weekly + post-migration, a drill injects it
  into synthetic output to prove the redaction path fires *and* scans all
  artifacts/DB/logs/exports for it and for key prefixes; any hit ⇒
  `INC-canary-<date>` and engines disabled.
- **SECURITY.md** (one maintainer): guarantees table, the allowlist, the
  data-flow statement (exactly what leaves the machine — non-local-only vault
  text in prompts, inherent to BYOK), the honest residual list (a compromised
  OS user; the LLM provider seeing non-local-only text; semantic persuasion of
  the researcher; on macOS/Windows, no OS egress enforcement), and a poisoned-
  fixture corpus run in CI. Fixtures are regression, not proof, and grow by one
  per incident.

## 13. Agent boundary and LLM allocation (2026-07-03)

Outcome of the control-surface-vs-standalone-agent design review (grounded in
ADR-125–130 and prior art: mem0/OpenMemory MCP and LangMem on the tool side;
Letta's v1 loop re-architecture, Khoj's stack weight, and the Rewind shutdown
on the standalone side; the MCP spec's servers-simple/hosts-own-the-loop
principles). Recorded so it is not re-proposed.

**Memoria is not an agent, and beta.1 adds none.** A resident reasoning loop
re-creates the exited Hermes failure with the loop in the framework's seat —
it becomes the thing that owns state, workflow, and the write boundary, and
every invariant in §0 must be threaded through a component that changes with
each model generation (ADR-125's recorded lesson; ADR-130 rejects the same
inversion at the UI layer). It also has no arithmetic worst-case price, which
§9 requires of every call, and it is an uncontrolled variable inside a
pre-registered experiment built on pinned prompts and frozen thresholds. Any
agent — a future first-party worker or the researcher's own (Claude Code,
Cursor) — is a **client under the ADR-130 contract**: read-API + request
envelope, universally quarantined, read-scoped, never granted
file/terminal/exec/send tools; chat additive, never the primary path. Agent
transports stay local/stdio for beta.1 (the alternatives file's assumption);
remote/OAuth remains deferred.

**The operation test — which LLM calls belong in the product.** Memoria runs
an LLM call directly iff it can be an *operation*: (1) pinned prompt + pinned
model, so it ships with a fixture and a seeded-error verdict; (2)
deterministically checkable output (schema, verbatim-anchor, byte-substring
checks); (3) an arithmetic worst-case price (`max_tokens` ceiling + a Ledger
price-table row, batchable); (4) per-item, corpus-scale, unattended shape.
Two litmus questions: *could it ship with a fixture and a seeded-error test?*
and *can its worst-case price be printed before it runs?* Failing either, the
call belongs to the researcher's agent — open-ended, interactive, cross-tool,
or voice-laden work, on their model and their bill, entering the vault only
as envelope writes that land unchecked like everyone else's.

**Applied to the beta.1 roster** (the call sites this design already carries,
now with the rule stated): the digest (§7), the seeding relevance prefilter
(§7), and paid synthesis (§8), plus the local pinned models (embeddings/NLI,
§8) as Memoria-owned machine judgment. Synthesis is **deliberately redundant**
with what a frontier agent could do better: the product copy exists for the
agent-free direct path and for instrumented payoff attribution; agent-side
synthesis over the same Ask evidence is welcome and unmeasured. The
redundancy is the direct-access contract working, not waste. New LLM features
must pass the same test — "research the literature for me" fails it
(open-ended, external), consistently with the amended discovery exclusion
(bounded citation-graph discovery is deterministic and identifier-only, §7 —
it is not an LLM call and does not enter this roster).

**Neither side, ever:** confers a verdict (checks promote; every LLM opinion
is a proposer — ADR-129), performs a rule-derivable write (engines write),
writes human-class text, or chooses an egress destination.

**Flip conditions, recorded:** synthesis-shaped Asks losing to the ChatGPT
counterfactual *specifically on integrative reasoning* even after section
expansion (→ a bounded multi-step compose operation — still pinned, priced,
and run-scoped — not a resident loop); agent-host/MCP fragmentation making
BYO clients unreliable (→ a first-party worker-client rises in priority);
disposition telemetry showing conversation as the de facto primary surface;
scope leaving single-researcher/local-trusted territory (→ a Memoria-owned
pinned-model host becomes a genuine *security* argument). In every flip the
loop gains standing as a client, never ownership of the write boundary.

## 15. Control surface: generative-UI disposition (2026-07-03)

Outcome of the generative-UI rethink over the plugin control surface.
Evidence base: SDUI at scale (Airbnb Ghost, Lyft, Adaptive Cards' schema-skew
decade); fixed-vocabulary components as a *security* boundary (Shopify
checkout extensions, Google A2UI's "UI as data, not code"); the MCP Apps /
OpenAI Apps SDK containment models (both presume an iframe sandbox); the
adaptive-UI literature (Gajos; Findlater & McGrenere; Cockburn's intramodal
expertise; Office 2000's adaptive-menu retreat); Obsidian's own statement
that plugins cannot be sandboxed. Recorded so it is not re-proposed. The
ruling idea: **the unit of generativity is the block, never the pane.**

- **Chrome is authored.** The five pane shells and the interaction-rich rails
  (Reading's select-then-anchor on CM6, Review's diff-and-revert, price-labeled
  confirms) stay hand-built typed code. The lockstep version handshake (§2)
  nullifies SDUI's ship-without-the-client payoff, so its versioning tax would
  be pure cost; one habituated daily expert sits exactly where the evidence
  says static layout wins; and "render known kinds only" is enforced by the
  type system rather than a runtime validator.
- **Projection content is server-driven.** Every list/table/card/worklist
  inside those shells renders through the one **ViewSpecRenderer** (§2) over
  engine-composed `view-spec.v1` (landed engine-side: table/card blocks,
  `check_status` per block *and* per row, graph-bound refs, untrusted text
  only as tagged `body_data`). Composition is deterministic and $0; a new
  projection becomes an engine change, not pane code. Tripwire: if the
  vocabulary creeps past ~two dozen kinds or panes degenerate into bespoke
  kinds anyway, the interpreter has become five panes wearing a schema —
  retreat to view-spec islands. Forcing move worth naming: meaning-only
  frontmatter keeps verdicts out of files, so native Bases views structurally
  cannot carry `check_status`; anything verdict-bearing must come through the
  read-API path. Bases remains the keep-test browsing layer for meaning-only
  projections.
- **Agent views land as data, not components** (post-beta.1; needs its own
  adapter ADR plus a deceptive-layout fixture family before shipping — the
  SusBench/Unit-42 preselection and hidden-affordance patterns are the probe
  set). An agent may submit a view-spec *instance* through the request
  envelope, validated like any write, landing unchecked, rendered by the same
  interpreter inside a mandatory proposed-view container: provenance banner,
  engine-clamped loudness, and an enqueue-only action vocabulary the producer
  cannot set. This is A2UI's client-owned-catalog stance mapped onto the
  quarantine path, and categorically not the rejected shared-mutable-state
  pattern — one-shot data through the queue, never a synchronized layer. It
  closes a real gap: agents can currently hand back only markdown.
- **Model-composed layouts stay deferred** behind the alpha.15 preconditions
  (constrained-decoding validation + an eval harness). Per-render composition
  on the hot path fails §13's operation test by construction ($0 path, no
  printable price); the only slot that could ever be admissible is riding an
  already-priced cold-path call — Synthesize choosing among *fixed* layout
  kinds at marginal token cost — and only with spatial stability across
  repeated similar questions (the Gajos/Findlater bar). If layout-choice
  accuracy is mediocre, abandon rather than tune.
- **Model-generated markup stays rejected.** Obsidian has no plugin sandbox
  and states it cannot add one; a plugin holds full filesystem/network/Node
  access while the vault holds untrusted PDF text — the lethal trifecta with
  a proven exfiltration channel (the cross-vendor markdown-image incidents).
  Every industry containment model for generated UI presumes the sandbox that
  is absent here. Revisit bar, deliberately high: a real Obsidian plugin
  permission/sandbox model with mobile iframe/CSP parity AND per-render cost
  passing the printable-price bar. Neither is on any roadmap.

**Flip conditions, recorded:** a second first-class client ships (mobile app,
web view — the Airbnb payoff condition becomes real; adopt strict spec
versioning and capability negotiation *before* that client, or Adaptive
Cards' silent-element-drop failure lands on the honesty contract, where a
dropped badge is a safety bug); the deceptive-layout fixtures fail against
engine clamping (restrict agent views to engine-templated shapes, or drop
them); the eval harness clears model-composed layouts with high accuracy and
spatial stability (admit them on priced cold paths only, never the $0 loop).

## 16. Obsidian integration disposition (2026-07-03)

Outcome of the integration rethink. Evidence base: the LSP architecture and
vscode-languageclient lifecycle norms (client owns the server; capability
flags; five-crashes-in-three-minutes give-up); rust-analyzer's
explicit-consent binary download and vscode-ripgrep's retreat from
postinstall downloads to bundled prebuilts; the community catalog's
three-asset limit and the May-2026 automated-scanning regime; the loved
files-first archetypes (Readwise's append-only contract, Omnivore's
shutdown-with-surviving-markdown, Excalidraw's degradation story) versus the
in-Electron-compute failure archetype (Smart Connections' in-vault 500MB
index and hangs, Copilot's index corruption and plaintext keys);
obsidian-git's "very unstable" mobile JS fallback; documented Obsidian
Sync/iCloud/file-watcher behavior. Recorded so it is not re-proposed. Ruling
idea: **integrate at the lowest layer that can carry the job** — files, then
native primitives, then plugin capabilities, then panes, then the daemon.

- **ADR obligation, named.** This host decision (plugin + daemon as the
  researcher UI) amends ADR-125 (standalone CLI + engine architecture) and
  ADR-130 (surfaces) and **requires that ADR before implementation begins**.
  Until it lands, `docs/reference/obsidian-plugins.md` stays exactly as
  written — "no plugin required; a Memoria plugin is a future adapter and
  needs its own ADR" is already the correct current-state form, and this
  document is a proposal, not product truth.
- **One artifact.** The engine binary is `mem`; the daemon is `mem serve`
  (§1). No separate `memoriad` executable: one distribution story, one
  version, and the CLI survival floor is by construction installed wherever
  the daemon runs.
- **Acquisition is consented, never bundled.** The catalog ships only
  `main.js`/`manifest.json`/`styles.css`, and under the automated-scanning
  regime "plugin silently fetches and executes a binary" is the most
  flaggable pattern in the ecosystem. Two sanctioned paths: a package
  manager (`uv tool install memoria`, winget/brew) or the rust-analyzer
  model — explicit user consent, checksummed, version-matched download from
  the release page. Windows builds are code-signed: SmartScreen reputation
  is per-identity, unsigned builds restart at zero every release, and EV no
  longer bypasses it.
- **LSP lifecycle, verbatim** (§1): port-file + per-install token discovery
  (Watchman/Chrome/Jupyter norms; Host-header validation against DNS
  rebinding), spawn on `onLayoutReady` if absent, initialize-first
  version/capability handshake, parent-death watch, crash-loop give-up
  surfaced at block loudness. Not novel engineering — transcription of the
  proven pattern.
- **Two rich panes, floors everywhere** (§1): Ask and Review are the only
  surfaces whose interactions (streaming retrieval with stance chips;
  classified diff with one-click revert) cannot be file-rendered. Capture,
  Compose, and Reading ride files, commands, native primitives,
  and the inspector. Every surface keeps a no-plugin floor except Ask, which
  by §1's own ruling has no survival role.
- **The plugin is catalog-clean and desktop-only.** `isDesktopOnly: true`;
  `onload` contains registrations only, all work in `onLayoutReady`;
  DeferredView-compliant; Vault API with `Vault.process()` for any file
  touch; no default hotkeys; README discloses the local daemon and LLM
  egress; BRAT is the beta channel — which is the charter's release-channel
  discipline, not an extra mechanism. Mobile is **keep-test mode by
  design**: synced markdown, PDFs, Bases catalog views, `mem-prov` markers
  in source mode — no JS-fallback daemon heroics (the obsidian-git lesson).
- **Write-event push, not watcher faith** (§1): the daemon notifies the
  plugin after every applied run and the plugin refreshes affected views.
  Obsidian's watcher provably misses external writes on real setups; a
  stale badge is an honesty-contract violation.
- **No dot-folder for artifacts that must survive** (§3): Obsidian Sync
  ignores hidden folders, so the manifest audit copy, incidents, and metrics
  live in visible `Memoria/system/`. Engine-private state stays outside the
  vault entirely (§5).
- **Plugin coexistence.** Memoria requires **no Obsidian community
  plugins** — a load-bearing property of the lowest-layer rule, not an
  accident. Two footnotes: BRAT, during the beta only (it *is* the
  release-channel discipline above); and core-plugin/app-version floors —
  Bases available (core 1.9+, Bases view API 1.10, which sets
  `minAppVersion`) and File Recovery left enabled (a free extra net beside
  the manifest gate and run revert). Companions live outside Obsidian:
  Zotero is an *app* reached over its local API — Zotero-first reading and
  the annotation bridge (§17) need no Obsidian-side Zotero plugin — and
  Better BibTeX is a *Zotero-side* plugin whose citekey namespace Memoria
  inherits (§17); Zotero Integration/Citations remain unnecessary, since
  `bibliography.bib` is engine-regenerated. Recommended beyond that: one
  sync authority of the user's choice, nothing else —
  Dataview/Templater/Local REST API/refresh pollers are superseded by the
  read-API, engine writes, and the write-event push. Discouraged (the §10
  corruption class, which startup detection covers): format-on-save
  plugins, bulk frontmatter/tag rewriters touching `Memoria/`, and a second
  stacked sync service. obsidian-git is tolerable but noisy — Memoria's
  snapshots live in their own external git-dir, so there is no collision,
  but user-side auto-commit captures mid-run file states as churn in
  *their* repo: one line of user education, not a ban.

**Flip conditions, recorded:** the review regime formalizes a ban on
consented binary downloads (→ package-manager-only acquisition; the plugin
becomes pure discover-and-connect, the Zotero archetype); mobile
capture/triage becomes a real daily-loop requirement (→ thin-client-to-
server, the Khoj archetype — the daemon reachable off-box — which is a
deployment decision, not a plugin heroic); Obsidian platform churn
(pane/CM6/Bases/DeferredView breakage) exceeds a solo maintainer's budget
(→ fewer panes, more native primitives); Bases gains stock-rendered
write-back actions (→ more surfaces become file-rendered and the pane count
shrinks further); signing logistics prove untenable (→ package-manager
channel only, accepting the install-step friction).

## 18. Alpha-regression resolutions (2026-07-04)

Outcome of checking the alpha.14/15 design invariants against this design
([0.1.0-beta.1-alpha-regression-review.md](0.1.0-beta.1-alpha-regression-review.md):
158 verified, 151 held). Six regressions were found; four were fixed in the
body, two are recorded decisions here so the divergence is named, not silent.

**Fixed in the body:**
1. **In-file `mc` is a claim, not authority** — the verdict of record is the
   DB `checks` row bound to the block's `block_sha256`; green requires marker,
   current hash, and row to agree; `mem rebuild`/restore fails closed to amber
   and re-earns via the local Checker (§2, §4, §5). This also resolves the
   §11 "provenance verdicts DB-side" contradiction.
2. **Export gate widened** — export now refuses (not warns) on a cited
   superseded/retracted work and on a mapped claim lacking both an `mc` anchor
   and an explicit gap marker; the gate blocks, never edits (§8).
3. **Raw provider evidence preserved** — a content-addressed `payloads` store
   (operational-ledger class) makes enrichment merges replayable and auditable
   after a provider mutates its record (§5, §7).
4. **Rejection→fixture loop** — a rejected engine output auto-adds a candidate
   regression case, mirroring the security-fixture ratchet (§1, §10).

**Recorded decisions (not fixed):**
5. **Backup-status doctor — accepted, light form.** The alpha "warn before
   disk-death" per-store report is added to the read-only inspector pane
   (§1) as a status line; the DB streaming-replication RPO is **declined**
   for beta.1 (the charter's backup/restore bar is met by `VACUUM INTO` +
   `restore --verify`; streaming replication is production hardening beyond a
   single-researcher first beta — revisit if backup gaps show in the beta).
6. **Frontmatter strict-reject validator — declined; OKF-permissive is
   deliberate.** Beta.1 keeps OKF v0.1's tolerate-unknown-keys consumption
   model (§3) rather than alpha's reject-undeclared-fields validator; machine
   writes stay conformant by construction (VaultWriter serializes from typed
   fields; closed `type` enum), and freehand human files converge via
   non-blocking proposals. The one alpha property worth reviving —
   confining *Memoria's own* extension keys to an `x.*` namespace — is
   **deferred** (it touches every frontmatter example for a meaning-layer-
   cleanliness gain, and OKF permits bare extension keys); revisit if the
   `mem-*` keys crowd the meaning layer in practice.

## Appendix A. The same design as a VS Code extension (2026-07-03)

Not scheduled work — recorded to show the §16 shape pays its M×N dividend,
and what the second client would actually cost. A VS Code extension is
**client #2 of the same daemon**, not a port.

- **Engine: zero changes.** `mem serve`, port-file + token discovery,
  initialize-first handshake, write-event push are host-agnostic; VS Code is
  the home turf of this lifecycle (the LSP pattern verbatim). Binary
  acquisition stays the single §16 story (package manager or consented
  download) — do **not** bundle per-platform VSIXes even though VS Code
  sanctions them; two distribution stories for one binary is how version
  skew starts.
- **Extract the renderer once.** §15 did the heavy lifting: pane *content*
  renders from `view-spec.v1` through one interpreter. ViewSpecRenderer +
  DaemonClient become a shared, host-free TypeScript package (no `obsidian`
  imports); the Obsidian ItemView shells and VS Code WebviewView shells are
  the only bespoke chrome per host.
- **The extension is four contributions:** commands (capture/import,
  anchored note from selection, revert — all through the same request
  envelope); two webview views (Ask, Review) hosting the shared renderer;
  `TextEditorDecorationType` provenance badges fed by the same read-API (the
  CM6 decoration logic does not port; it is the thinnest layer); and a
  `DocumentLinkProvider` resolving `[[citekey/fulltext#^p0042]]` and
  `^ev-…` anchors to file + offset — the one genuinely new piece, since
  Obsidian-flavored wikilinks and block IDs are inert text in VS Code and
  evidence links are the product's spine.
- **Deliberately skipped:** Bases-equivalent views (VS Code users read the
  markdown files — the keep-test already guarantees that works),
