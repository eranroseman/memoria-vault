### S14.1 — Platform split re-introduced on a new axis (security-guarantee strength) after alpha.14/15 flattened the runtime-location split

**LEAVE — deliberate and recorded** · verification: confirmed (citation audited) · confidence: high

The umbrella delta has no normative content beyond its four facets, and every facet is individually adjudicated in the design body: the Linux-only kernel egress layer and the macOS/Windows sandbox cut are recorded with rationale and an honest-residual line in s10; the per-OS keychain in s10 and Windows code-signing in s16; the desktop-only plugin and mobile keep-test mode in s16 with recorded flip conditions; s11 items (6) and (10) record the netns and host decisions as deliberate beta.1 scope. The alpha-line flattening this superficially resembles removed a runtime-location prerequisite (Hermes WSL2-only, ending in 'one install command, no external runtime, no always-on process'); B1 imposes no OS prerequisite — one `mem` artifact installs and runs cross-platform (s16 'One artifact') — and splits only guarantee strength and adapter tier, so the reason alpha flattened its old OS dimension is not re-opened.

- **Recorded:** 0.1.0-beta.1-design.md s10 L904-913 + L955-959 (Linux-only confinement, per-OS SECURITY.md residual), s10 L946-949 (keychain), s16 L1169-1171 (code-signing), s16 L1184-1191 + L1222-1232 (desktop-only plugin, mobile keep-test, flip conditions), s11 items (6) and (10)
- **History:** Alpha's only platform split was runtime-location: Hermes WSL2-only with Windows as the editing surface (ADR-26/55/64; history L565-579), whose documented costs were WSL2 as hard prerequisite, /mnt/c file-lock fights, the OneDrive hazard, and the networkingMode=mirrored bridge; alpha.14/15 removed it by dropping Hermes and shipping standalone installers (history L2022, L2190, L2302-2304). B1 preserves exactly the property that fix earned — single cross-platform artifact, no external runtime — so the motivating failure is addressed, not re-opened.

### S14.2 — Facet A: OS-enforced kernel egress (CLONE_NEWNET netns) is Linux-only; seccomp/macOS sandbox-exec/Windows-Firewall flows cut

**LEAVE — deliberate and recorded** · verification: confirmed (citation audited) · confidence: high

Design s10 records the decision explicitly and with its reasoning: the three non-Linux OS-enforcement flows are cut because two of them 'bought near-zero enforcement for real code and a 3-OS sandbox matrix,' and SECURITY.md is required to state the per-OS truth plainly, with 'on macOS/Windows, no OS egress enforcement' carried in the honest-residual list. The charter's prompt-injection bar (untrusted text cannot control vault egress, req L74) is met OS-independently by brokered egress — engines hold no sockets at all and Guard is the sole network path — and the Linux netns is a strict addition on top of that. Alpha never had OS-kernel enforcement on any OS, so nothing alpha earned is lost; s11 item (6) further records brokered egress + Linux netns as the enforceable form of the allowlist.

- **Recorded:** 0.1.0-beta.1-design.md s10 L904-913 ('OS confinement: Linux network-namespace only' with the cut and rationale) and L955-959 (SECURITY.md honest residuals); s11 back-pressure item (6)
- **History:** The alpha line's egress control was always application-level: the alpha.1 write-layer policy gate (ADR-22, permissions enforced per-profile at the write layer), alpha.9 egress-as-P0, alpha.11 channel-separation + allowed_network, alpha.14/15 no-agent-tool-grants — with no OS-kernel layer at any point in alpha.1-alpha.15 (gap report L146 twin bullet; history L1852 fail-closed provider allowlist as data-config over the manifest ceiling). There is no documented alpha failure behind cross-OS kernel enforcement to re-open; B1 keeps a strictly stronger application layer on every OS and adds kernel enforcement where it is cheap.

### S14.3 — Facet B: per-OS keychain for BYOK keys + code-signed Windows builds, both new in B1

**LEAVE — deliberate and recorded** · verification: confirmed (citation audited) · confidence: high

Both additions are recorded design decisions serving a charter bar. s10 specifies the per-OS keychain (Keychain / DPAPI / Secret Service, mode-0600 fallback with logged warning on Linux) and the invariant 'the key exists only in the keychain and Guard's memory,' directly implementing the charter Key-handling row (keys never enter vault content, backups, exports, or logs — req L65). s16 records Windows code-signing with its reasoning (SmartScreen reputation is per-identity, unsigned builds restart at zero every release) plus a recorded flip condition (package-manager-only channel if signing proves untenable). The delta is purely additive: alpha held no long-lived user key as a first-class secret, so no alpha invariant is displaced and nothing needs a divergence entry.

- **Recorded:** 0.1.0-beta.1-design.md s10 L946-949 (keychain, key-residence invariant), s16 L1169-1171 (code-signing) + L1231-1232 (flip condition); requirements charter Key handling L65
- **History:** In the Hermes era, keys lived in per-profile .env files seeded by the installer ('Hermes reads per-profile .env only... The installer must seed shared keys into each profile .env', ADR-27 Finding 6, history L360), and model routing was a Hermes/OpenRouter deployment concern (history L593); by alpha.15 the sole credential was the loopback bearer token MEMORIA_HTTP_TOKEN. Alpha never had, and never rejected, a keychain; B1's BYOK (design s13) creates the first first-class user secret and the keychain is strictly stronger than the .env precedent. No failure re-opened.

### S14.4 — Facet C: Obsidian plugin is desktop-only and is the only researcher UI — the 'own ADR / next wave' work alpha.15's scope boundary punted

**LEAVE — deliberate and recorded** · verification: confirmed (citation audited) · confidence: high

The host decision and its desktop-only scope are adjudicated in s16, which records both the rationale (the plugin fronts the daemon; PDF extraction, embeddings, NLI, egress brokering, and the DB sit in a separate OS process for crash isolation, not Electron; the obsidian-git mobile-JS-fallback lesson) and the exact obligation the alpha.15 scope gate imposed: the decision 'amends ADR-125 and ADR-130 and requires that ADR before implementation begins,' with docs/reference/obsidian-plugins.md unchanged until it lands. s11 item (10) separately records the host decision as new beta.1 scope. The substance of the alpha demotion is preserved, not reverted: the engine remains the product (one `mem` artifact), the plugin is a client of the read-API, and every surface keeps a no-plugin floor except Ask, which s1 rules has no survival role — so this continues the alpha.12-15 arc under its own recorded terms.

- **Recorded:** 0.1.0-beta.1-design.md s16 L1152-1158 (ADR obligation, named) and L1184-1191 (isDesktopOnly: true, catalog-clean); s11 back-pressure item (10); s1 L85-88 (no-plugin floors)
- **History:** Alpha.12 demoted Obsidian to an optional adapter on the principle 'the engine is the product... don't make Obsidian, Zotero, Hermes, or any adapter the architecture' (history L1814, ADR-122 era); alpha.14 dropped it as a required surface — 'product must run on vim/nano/file browser' (L1930, L2052); alpha.15 fenced plugin work behind 'its own ADR' and 'alpha.15 cannot close with Obsidian plugin implementation counted as release scope' (ADR-130, #1203/#1208, L2216). The motivating concern — an adapter owning state/policy/the write boundary and loss of the standalone floor — is addressed: B1 keeps the standalone engine with CLI floors and explicitly pays the required-ADR obligation before implementation.

### S14.5 — Facet D: mobile is keep-test mode by design — synced markdown + Bases catalog views, no daemon, no verdict-bearing views

**LEAVE — deliberate and recorded** · verification: confirmed (citation audited) · confidence: high

The mobile stance is recorded in s16 ('Mobile is keep-test mode by design: synced markdown, PDFs, Bases catalog views, mem-prov markers in source mode — no JS-fallback daemon heroics, the obsidian-git lesson') with an explicit recorded flip condition: if mobile capture/triage becomes a real daily-loop requirement, the move is thin-client-to-server (the Khoj archetype), a deployment decision, never a plugin heroic. The structural reason a daemon-less device renders only meaning-only views is itself recorded in s15 (meaning-only frontmatter keeps verdicts out of files, so native Bases structurally cannot carry check_status; anything verdict-bearing comes through the read-API). The write-side boundary is anchored in the charter's Deliberate Exclusion of concurrent multi-device writes (req L87), which is decisive under the calibration rules.

- **Recorded:** 0.1.0-beta.1-design.md s16 L1189-1191 + flip conditions L1224-1227; s15 L1093-1097 (forcing move); requirements charter Deliberate Exclusions L87 ('Concurrent multi-device writes')
- **History:** 'mobile' appears nowhere in the ~2400-line design history (grep over the full file: zero hits), confirming A15's zero design presence — there is no alpha 'mobile era' to revert to. The lineage B1 projects onto the device class is continued, not dropped: the alpha.12 keep-test (ADR-122 — markdown + portable bibliography survive the tool, history L2308) and the no-daemon CLI recovery surface (ADR-130, L2176); and the docs-history Reject table independently keeps mobile capture via network channels dead ('synced vault files the daemon ingests — never a network service'). The charter exclusion protects against the Obsidian-Sync last-write-wins data-loss risk; no failure re-opened.
