# alpha.15 → beta.1 gap analysis — read from the point of view of the beta.1 reports

Companion to [0.1.0-beta.1-alpha15-gap-analysis.md](0.1.0-beta.1-alpha15-gap-analysis.md).
That document enumerates every material A15→B1 delta. This one takes each of those
gaps, one bullet at a time, and analyzes it **against the rest of the release
folder** — the requirements charter, the governing design (incl. §11 alpha-plan
implications and §18 regression resolutions), the design-alternatives summary,
the two research-evidence corpora, the paper adversarial review, and the paper +
docs-history borrow/adapt/reject reports.

Descriptive only — what the reports say about each gap. No pros/cons, no
recommendations. Section numbers mirror the gap analysis (§1–§16); each gap
bullet gets its own subsection.

**Lenses applied to every bullet** (only where a report speaks to it):

- **Recorded vs silent** — do the reports register the change as a deliberate,
  reasoned decision (charter Deliberate Exclusions; design §11/§17/§18; a
  regression verdict; a Borrow/Adapt/Reject row; an adversarial finding), or note
  it as an unrecorded drop?
- **Rationale** — the stated reason the reports give for the B1 side (research
  evidence, named papers, and the load-bearing doctrines: keep-test,
  single-writer, no-write-time-oracle, meaning-only frontmatter, three-host egress
  allowlist, adoption/abandonment-first, cost ceilings).
- **Regression status** — how [alpha-regression-review](0.1.0-beta.1-alpha-regression-review.md)
  classifies the delta: held / strengthened / one of the 14 recorded-divergences /
  one of the six regressions **R1–R6** (with severity and the §18 resolution).
- **Re-homing / rejection** — whether the docs-history or paper B/A/R corpus
  treats the item as a Borrow, Adapt, or Reject, and the reason.
- **Adversarial pressure** — any [paper-adversarial-review](0.1.0-beta.1-paper-adversarial-review.md)
  finding that attacks or defends the B1 choice, and its adjudication.
- **Silence / conflict** — where no report addresses the gap, or reports disagree.

Legend, as in the source: **A15** = alpha.15 · **B1** = beta.1.

Method: each section was analyzed by one agent reading the full report corpus, then
independently re-verified by a second agent that re-opened every cited file to
confirm each attribution (quote, §, R-number, table row) and fill misses.

## 1. Host / runtime / distribution

### 1.1 — Obsidian re-introduced as the *required* researcher UI

**RECORDED, not silent.** The B1 side is stated as a first-class design decision, not an unnoticed drift. `design §1` opens the host section with "**One Obsidian plugin … + one engine binary**" and the flat assertion "**The Obsidian plugin is the only researcher UI.**" `design §11` enumerates the reversal explicitly as back-pressure item (10): "the **host decision itself** — beta.1 is an Obsidian plugin + local daemon, which the alpha.15 plugin design anticipates but does not yet center," and classes it as "**new alpha.16 scope**." `design §16` names the ADR debt: "This host decision (plugin + daemon as the researcher UI) **amends ADR-125** (standalone CLI + engine architecture) **and ADR-130** (surfaces) and **requires that ADR before implementation begins**," adding that until it lands, `docs/reference/obsidian-plugins.md`'s "no plugin required; a Memoria plugin is a future adapter and needs its own ADR" "is already the correct current-state form, and **this document is a proposal, not product truth**." So the design records that A15's "no plugin required" is still the live product truth and B1 is a proposal to change it.

**RATIONALE.** `design §1` grounds the choice in the abandonment-first / keep-test doctrines: "The researcher already lives in Obsidian; parity with the null workflow means **zero app-switching**, and the keep-test answer is that every artifact is plain markdown Obsidian navigates unaided. A standalone app would **duplicate an editor we cannot beat**." `requirements` bakes Obsidian into the experiment's baseline: beta.1 is a "retention experiment" against "the null workflow: Zotero + Obsidian + ad-hoc ChatGPT," and Success Claim 1 (Parity) is measured as "note-writing is **no slower than plain Obsidian**." `research-evidence` supplies the survivor argument: Obsidian is one of the two category survivors ("'file over app'," "plain local files, zero exit cost"), and "**The null workflow (Zotero + Obsidian + ad-hoc ChatGPT) is composed of the category's proven survivors, which sets the trust bar Memoria must never fall below.**" `design-alternatives` records that adoption-first **Design A won** ("strongest focus on the dominant risk: abandonment … made Ask the everyday payoff moment"), the doctrine that motivates living inside Obsidian.

**REGRESSION STATUS.** `alpha-regression-review` does **not** classify the Obsidian-as-required-UI pivot as one of the six regressions (R1–R6 concern the `mc` verdict-authority marker, export gate, provider-payload store, rejection→fixture loop, backup-status doctor, and the frontmatter validator — none is the host). Its method "default[s] to *not-a-regression* … treating any delta recorded in §11–§17 or the charter's exclusions as deliberate," so a §11/§16-recorded host change is a deliberate delta by construction. The "Checked and intact" sweep records the plugin-centric invariant "**the read-API as the sole read path**" among invariants verified present or strengthened. The 14 recorded-divergences it names (agent/co-PI machinery, kanban/spaces, hub/tag graph, claim-context fields, qmd, provider attention routing) do not individually list the host pivot, but its sweep sentence explicitly buckets all §11–§17-recorded deltas as "**recorded** divergences … **not silent drops**."

**RE-HOMING / REJECTION.** `docs-history B/A/R` **rejects** community-plugin UI assembly (Reject row: "Community-plugin UI assembly for load-bearing surfaces") on the ground "**The Memoria plugin is the only UI**," citing the alpha.3 matrix where "every load-bearing plugin was single-maintainer risk." It also re-homes the plugin keep-test constraint (**Adapt** row "UI keep-test + graph-view constraints"): "Memoria plugin surfaces are **additive and never disable core Obsidian surfaces, so a dead plugin degrades to stock Obsidian** (session-level keep-test)." The "Patterns in what was deleted" section frames the whole exercise as the required discipline: "When an architecture is retired, its surface-level policies **need explicit re-homing**." `design §16`'s "Plugin coexistence" rule reinforces the single-UI stance from the other side — "**Memoria requires no Obsidian community plugins** — a load-bearing property of the lowest-layer rule, not an accident" (BRAT during the beta and core Bases / File Recovery the only footnotes).

**ADVERSARIAL PRESSURE / SILENCE.** `paper-adversarial-review` is **silent** on the host choice — all eight findings (BLOCKER→MEDIUM) attack the Checker, the retriever's "corpus has nothing" claim, the doubly-blind contrasts list, `mc` semantics, supersession, synthesis granularity, overreliance, and the evaluation's reverse-funnel, and the Survivors list contains no host/plugin entry. `design-alternatives` is likewise **silent** on host: Designs A/B/C differ on adoption vs integrity vs evaluation weighting, and none of its attack-table rows concerns which application hosts the UI. `paper-borrow-adapt-reject` does not touch the host question.

### 1.2 — A resident two-process daemon (`mem serve`), where A15 had none

**RECORDED, not silent.** `design §1` states it directly: "**The daemon is `mem serve`, supervised by the plugin with LSP lifecycle semantics (§16), resident while Obsidian is open,**" and "The daemon runs as **two processes** (§2): a *supervisor* holding the key, the DB, Guard, Ledger, and VaultWriter; and a per-run *engine worker* that is credential-free and socketless." `design §2` splits them into the "Supervisor process" and "Engine worker process (spawned per run, credential-free …)." `design §16` records the acquisition and lifecycle as a deliberate transcription — "**LSP lifecycle, verbatim** … Not novel engineering — transcription of the proven pattern" — and collapses the artifact count: "**One artifact.** The engine binary is `mem`; the daemon is `mem serve` (§1). **No separate `memoriad` executable.**" `design §11` item (10) lists the "local daemon" as part of the recorded host decision / new alpha.16 scope.

**RATIONALE.** `design §1` gives the isolation argument: "PDF extraction, local embeddings, NLI, egress brokering, and the DB **do not belong inside Electron**; a separate OS process gives **crash isolation and one network choke point**," with the workspace write lock enforcing "one writer." This ties to §0 ruling ideas 2 ("A single writer rules safety") and 3 ("Runs rule time"). `design §16` cites the "**in-Electron-compute failure archetype** (Smart Connections' in-vault 500MB index and hangs, Copilot's index corruption and plaintext keys)" against Electron-resident compute, and "obsidian-git's 'very unstable' mobile JS fallback" against a JS daemon. `design §13` records that "**Agent transports stay local/stdio for beta.1** (the alternatives file's assumption); remote/OAuth remains deferred," matching the loopback-only daemon; `design-alternatives`'s attack row supplies that assumption ("stdio/local-only MCP assumptions for beta.1"). The charter-level basis for a single writing host is the `requirements` Deliberate-Exclusion of "Concurrent multi-device writes" (reason: "**Sync conflict handling is a data-loss risk outside the first beta claim**"); the "single-writer and runs-rule-time assume exactly one writing host" rationale is the `docs-history B/A/R` Reject row, re-homed below.

**REGRESSION STATUS.** `alpha-regression-review` does not flag adding the daemon as a regression; the daemon-adjacent invariants are swept as held/strengthened — "**the single-writer + staging + block-level human-content protection spine**," "**cross-store recoverability / observable atomicity with paired rollback**," "**brokered fail-closed egress with the compiled allowlist**," "**per-operation model pins with test/live modes**," and "the **request-envelope integrity fields**." The one regression touching this surface, R5 (backup-status doctor, **Low**), attaches a per-store status line to the read-only inspector pane (§1) rather than the daemon itself, and its §18 resolution is "**accepted, light form**" (the DB streaming-replication RPO "**declined**" for beta.1).

**RE-HOMING.** `docs-history B/A/R` treats the daemon's operational contract as a high-value re-home from deleted docs: the Headline notes "beta.1's operational layer is unspecified today but **fully designed in deleted docs**: JobRunner lifecycle (idempotency keys, retry ceilings, heartbeat-on-success), Guard fail-closed liveness with negative deny-tests … hostile-plugin defense," and Recommended-action 6 says "**Add a memoriad operational contract section**" — matching `design §2`'s "Operational contract (the half that historically decayed)." (Terminology nuance: the docs-history report uses the old daemon name "memoriad," which `design §16` explicitly retires — "No separate `memoriad` executable.") Borrow row "Read-only inspector pane" (adr/84) re-homes the daemon status surface ("daemon liveness, queue depth, recent runs …"), and Borrow row "Hostile-plugin defense" re-homes "**Daemon/plugin startup detects known-hostile plugins**." Reject row "Multi-device / cross-vault writers" reaffirms a single daemon: "If it ever returns: sync plain files only, **one memoriad, second device is a reader**."

**ADVERSARIAL PRESSURE / SILENCE.** `paper-adversarial-review` is **silent** on the daemon; no finding or Survivor addresses process architecture. `paper-borrow-adapt-reject` is *not* silent on the process split: its Reject of "Credentialed, socketed LLM orchestrator" affirms "**engines stay credential-free and socketless**," and its Reject of "Machine hypothesis/idea generation" notes "Gottweis's supervisor + credential-free workers **already parallels memoriad**" — both bearing on §2's supervisor / credential-free-worker split, though neither addresses the resident-daemon host decision itself. `design-alternatives` introduces no resident daemon in its attack table and is otherwise silent on it.

### 1.3 — CLI demoted from product to survival/ops floor; `ask` dropped from the CLI

**RECORDED, not silent.** `design §1` defines the demotion verbatim: "**`mem` CLI = survival + ops + the daemon entry point**: `serve`, `capture`/`import`, `export`, `rebuild`, `backup`/`restore --verify`, `migrate`, `revert-run`, `cost`, `hat`. **No `mem ask`** — Ask is a judgment surface with no survival role, and the keep-test answer is stock-Obsidian navigability, not a CLI." The surface list confirms Ask is the one surface with no non-plugin floor: "**Ask deliberately has none (below)**," restated in `design §16`: "Every surface keeps a no-plugin floor **except Ask, which by §1's own ruling has no survival role**."

**RATIONALE.** The load-bearing doctrine cited is the keep-test: survival is "**stock-Obsidian navigability, not a CLI**," consistent with `requirements` Success Claim 4 (Survival) — "the vault remains useful as **plain markdown plus standard bibliography files** if Memoria dies" — which locates survival in files, not in `mem`. `design §16` justifies Ask being pane-only: "Ask and Review are the only surfaces whose interactions … **cannot be file-rendered**." The retained CLI verbs map to `requirements` Operations rows (Install, Upgrade/migration, Backup/restore, Rebuild, Cost control); `mem hat` corresponds to the `design-alternatives` attack response "Separate researcher-hat events from maintainer work."

**REGRESSION STATUS.** `alpha-regression-review` does **not** list the CLI demotion or the `ask` drop among R1–R6, nor among the named recorded-divergences. It records supporting invariants as intact — "**the proposer-never-authority rule and act/ask/log routing (not an approval queue)**," "the **read-API as the sole read path**," and the keep-test ("**markdown + BibTeX survive the tool**") — but is otherwise **silent** on whether removing `mem ask` weakens any alpha invariant.

**RE-HOMING / SILENCE.** No `docs-history B/A/R` or `paper-borrow-adapt-reject` row addresses the CLI verb set or the removal of an Ask CLI; the closest is the docs-history Borrow row "Deterministic test tiers" ("plugin coverage **IS** product coverage when the plugin is the only UI"), which supports Ask being plugin-only but does not discuss the CLI demotion. `paper-adversarial-review` is **silent**. Thus, apart from `design §1`'s own statement, the reports do not independently pressure or defend dropping `ask` from the CLI.

### 1.4 — Plugin distribution + binary-acquisition story (catalog-clean, consented download, code-signed Windows)

**RECORDED, not silent.** `design §16` specifies the distribution model in full: "**Acquisition is consented, never bundled.** The catalog ships only `main.js`/`manifest.json`/`styles.css`, and under the automated-scanning regime 'plugin silently fetches and executes a binary' is **the most flaggable pattern in the ecosystem**. Two sanctioned paths: a package manager (`uv tool install memoria`, winget/brew) or the **rust-analyzer model** — explicit user consent, checksummed, version-matched download from the release page. **Windows builds are code-signed**: SmartScreen reputation is per-identity, unsigned builds restart at zero every release." It also fixes the plugin shape: "**The plugin is catalog-clean and desktop-only.** `isDesktopOnly: true` … README discloses the local daemon and LLM egress; **BRAT is the beta channel**." `design Appendix A` extends the same single-story rule to a hypothetical VS Code client: "Binary acquisition stays the single §16 story … do **not** bundle per-platform VSIXes … two distribution stories for one binary is how version skew starts."

**RATIONALE.** The doctrine is "**integrate at the lowest layer that can carry the job**" plus one-artifact/one-version distribution. `design §16` grounds the specifics in named prior art: "rust-analyzer's explicit-consent binary download and vscode-ripgrep's retreat from postinstall downloads to bundled prebuilts; the community catalog's three-asset limit and the **May-2026 automated-scanning regime**." Recorded **flip conditions** anticipate tightening: "the review regime formalizes a ban on consented binary downloads (→ package-manager-only acquisition …)" and "signing logistics prove untenable (→ package-manager channel only, accepting the install-step friction)." `requirements` Operations "Install" sets the envelope this acquisition flow must fit: "Clean-machine install reaches first value from a quickstart in **under 30 minutes**." `research-evidence` supports consented flows and channel discipline (Go telemetry opt-in reversal, Homebrew pre-consent notice; a version identifier embedded in every provenance record; a low-ceremony beta-channel release loop), and `design-alternatives` graft 6 supplies the release-channel discipline ("unreleased builds touch only the sandbox vault; only a tagged beta build touches the real corpus").

**REGRESSION STATUS.** `alpha-regression-review` does **not** classify the plugin-distribution / binary-acquisition story as a regression; no alpha invariant about the installer or distribution surfaces in R1–R6 or the recorded-divergence list. The nearest ops/robustness regression, R5, concerns per-store backup-completeness reporting, not distribution, and was resolved in light form in §18.

**RE-HOMING.** `docs-history B/A/R` re-homes packaging discipline: the **Borrow** row "Doc & release epistemics" carries "**packaging hygiene** (CI installs the built wheel into a clean venv and smokes every entry point; SHA-pinned artifacts; explicit rollback; installer re-runs idempotent and secret-preserving)," and the **Adapt** row "Upgrade reconcile over a live vault" says "**Keep the classification, drop the machinery**." The Reject of "Community-plugin UI assembly" reinforces the catalog-clean single-plugin shape. These bear on the acquisition story only as re-homed packaging/upgrade discipline; none prescribes the specific consented-download vs code-signing mechanics `design §16` chooses.

**ADVERSARIAL PRESSURE / SILENCE.** `paper-adversarial-review` is **silent** on distribution, code-signing, and binary acquisition. `paper-borrow-adapt-reject` and `design-alternatives` are **silent** on the acquisition mechanism (design-alternatives graft 12 only defers "continuous egress machinery … until there is a real support surface," which is adjacent but not about plugin acquisition; the `requirements` Deliberate-Exclusion "Support-org machinery" — "SLAs, polished debug bundles, and continuous observability wait until there is a support surface" — is the charter-level form of that same deferral).

### 1.5 — §11 self-records the host/runtime pivot as new alpha.16 scope on top of alpha.15

**RECORDED — this bullet is the design's own self-labeling.** The gap paraphrases `design §11` directly. §11's verbatim text: item (10) "the **host decision itself** — beta.1 is an Obsidian plugin + local daemon, which the alpha.15 plugin design **anticipates but does not yet center**," followed by "**Items (1)–(3) and (10) are new alpha.16 scope**; (4)–(8) sharpen mechanisms alpha.15 already builds; (9) is a process artifact to land during alpha.16." The section title frames the whole pivot as "**back-pressure, not redesign**," even as the gap-analysis frames it as a re-introduction/addition — a framing difference worth noting: the design calls it back-pressure that alpha.15 "anticipates," while the gap document scores it as "a beta.1 addition on top of alpha.15." `design §16` sharpens the same admission that this is prospective, not current, product truth: it "**requires that ADR before implementation begins**" and "**this document is a proposal, not product truth**," with A15's "no plugin required" still standing until the ADR lands.

**RATIONALE.** The stated reason for treating it as scope-to-land (not a settled build) is the ADR-obligation and one-artifact discipline in §16, plus the abandonment-first / keep-test doctrines already cited in 1.1. No additional rationale is attached to the "new alpha.16 scope" label beyond §11's sequencing of items (1)–(10).

**REGRESSION STATUS.** This bullet is where the regression-review's **method** is decisive. `alpha-regression-review` states it "**treat[s] any delta recorded in §11–§17 or the charter's exclusions as deliberate**," and its sweep conclusion is that such items are "**recorded** divergences in §11–§17 or the charter's exclusions, **not silent drops**." Because the host pivot is exactly a §11-recorded (and §16-recorded) delta, the review's framework places it in the deliberate-divergence bucket rather than among the six regressions — i.e., the reports treat the pivot as recorded, not silent, by design.

**RE-HOMING.** `docs-history B/A/R` "Patterns in what was deleted" supplies the meta-rationale for why §11's explicit recording matters: prior surface-level policies "**died as collateral of architecture retirements they merely rode on**" and "**need explicit re-homing, or this archaeology has to be re-run**." §11 is that explicit-recording act for the host decision; no single docs-history row targets the §11 pivot itself.

**ADVERSARIAL PRESSURE / SILENCE.** `paper-adversarial-review`, `paper-borrow-adapt-reject`, and `design-alternatives` are all **silent** on §11's scoping of the host pivot; none contests or endorses the "new alpha.16 scope" classification. The only cross-report tension is the framing one noted above (design's "back-pressure, not redesign" and "proposal, not product truth" versus the gap-analysis's "re-introduces / addition"), which the reports do not otherwise adjudicate.

## 2. Vault structure / bundle layout

### Bullet 1 — `Memoria/` nested as a subfolder inside an unrelated vault (two sibling roots) vs A15's single `knowledge/` bundle-as-workspace

**RECORDED vs SILENT — the nesting is recorded and reasoned, but the specific "two-sibling-roots" framing is only implicit.** The design records the subfolder model directly. `design §3` opens its layout block with `<vault>/ # the user's existing vault, untouched`, `<existing notes> # zero migration, zero markers required`, and the `Memoria/` subfolder beneath it. The prose states the boundary explicitly: "the bundle boundary is `Memoria/`, not the researcher's vault — pre-existing notes outside it stay untouched by design (zero-migration)" (`design §3`). The charter ratifies this as scope: the `requirements` "Knowledge format" row requires "The `Memoria/` tree is an OKF v0.1-conformant bundle: required typed frontmatter on every Memoria-created file plus generated `index.md` projections — **parity with the alpha.15 vault, no regression of the exchange format**." So B1 treats `Memoria/` (not the whole vault) as the OKF bundle that must match A15's `knowledge/` bundle.

**RATIONALE — the nesting is a consequence of the host pivot plus the zero-migration/keep-test doctrines.** The change is driven by the recorded host decision: `design §11(10)` lists "the **host decision itself** — beta.1 is an Obsidian plugin + local daemon, which the alpha.15 plugin design anticipates but does not yet center," and marks items (1)–(3) and (10) as "new alpha.16 scope." Because the researcher's real Obsidian vault already exists, `design §3`'s "zero migration, zero markers required" and "**Frontmatter** appears only on Memoria-created files; user notes need none. Absence of all `mem-*` markers = human — this is what makes pre-existing notes zero-migration" (`design §3`) are the load-bearing reasons the tree lives *beside* unrelated notes rather than owning the root. The keep-test doctrine anchors the bundle claim: `design §3` "The alpha.15 vault is already fully OKF-conformant and beta.1 must not regress the exchange format," reinforced by the survival Success Claim 4 in `requirements` ("the vault remains useful as plain markdown plus standard bibliography files if Memoria dies"). Research backing for the zero-migration / keep-test doctrine appears in `research-evidence`: the PKM-abandonment findings (line 87: Zettelkasten users "declaring PKM bankruptcy," the Medium account "deleting 1,500 of 4,500 Obsidian notes" because the vault "was turning into a graveyard of notes") and the "file over app" keep-test evidence (lines 309–312, kepano: "the working format IS the exit format… you can directly interact with the same exact files in other apps," and the beta.1 implication "open the vault in stock Obsidian, confirm notes/pages/hubs are readable").

**REGRESSION STATUS — not flagged as a regression; OKF-bundle conformance recorded as HOLDING.** `alpha-regression-review` does not list the nesting among R1–R6. Its "Checked and intact" sweep names "the finding that started this pass — **OKF bundle conformance**" as present/holding, and "the **keep-test** (markdown + BibTeX survive the tool)" as holding. The layout shift from `knowledge/`-as-workspace to `Memoria/`-as-subfolder is therefore treated as a preserved-format outcome, not a divergence.

**RE-HOMING / REJECTION — one tension in `docs-history`.** `docs-history` Reject row #125 ("Git-as-enforcement + standards-conformance substrates … OKF bundles") rejects OKF as an internal model — "OKF conformance was maintained overhead with zero external consumers … never let a standard become the internal model" — while recording "Adopt named subsets at boundaries (CSL-JSON, W3C selectors)." This sits against `requirements` and `design §3`, which keep `Memoria/` as an OKF v0.1 bundle for exchange-format parity; the reconciliation the design uses is that OKF is the bundle/exchange claim (with `index.md`/`log.md` projections and the class enum) rather than the internal data model.

**ADVERSARIAL PRESSURE — none.** No `paper-adversarial-review` finding (1–8) attacks the subfolder/two-root layout; the Survivors list (line 100) endorses the adjacent "in-file automatic provenance + ambient rendering" rather than the folder placement.

**SILENCE / CONFLICT.** No enumerated report addresses A15's `folders.yaml categories = [knowledge]` mechanism or its removal in B1 — the `folders.yaml` construct is not mentioned anywhere in the report corpus. No report discusses the "two sibling roots" collision surface (e.g., what happens if the user already has a `Memoria/`); the reports speak only to the bundle-boundary and zero-migration guarantees, not to root-adjacency mechanics.

### Bullet 2 — Per-work folder `works/<citekey>/` co-locating `record.md` (CSL-JSON) + `fulltext.md` + `digest.md` + `source.pdf`; catalog re-materialized as markdown

**RECORDED vs SILENT — fully recorded and explicitly reasoned.** `design §3`'s layout block shows `works/doe2023attention/` holding `record.md` ("catalog record; FULL CSL-JSON in a fenced block"), `fulltext.md` ("anchored extracted text (machine, untrusted)"), `digest.md` ("LLM digest (machine, untrusted, opt-in)"), and `source.pdf` ("original blob, sha256 recorded"). The re-materialization of the catalog record as markdown is stated with its reason: "**Full CSL-JSON is serialized into `record.md`** as a fenced code block (the derived DB is rebuildable, so bibliographic fidelity and the Zotero re-import key cannot live only in the DB)" (`design §3`, "File vs DB vs blob"). The same paragraph fixes the class split: "Files = records, fulltext, digests, notes, projects, drafts, bibliography, incidents (everything keep-test). DB = chunks, embeddings, indexes, links, anchors, events, costs, jobs, runs, manifest of record. Blobs = PDFs (in vault, sha256 recorded) and embeddings (in-DB via sqlite-vec)."

**RATIONALE — keep-test + rebuildable-DB + mergeable-text doctrines, with named research.** The load-bearing reason is that the DB is derived/rebuildable, so anything that must survive the tool cannot live only in SQLite (`design §3`). `research-evidence` supplies the mergeability argument at line 292 ("anything Memoria stores outside markdown will be last-write-wins or corrupted under any sync … the canonical layer (notes, catalog records, provenance) belongs in mergeable text files") and the file-over-app keep-test at lines 309–312 ("every data class … should be explicitly labeled 'canonical file' or 'derived/rebuildable' … anything that fails that classification is a design bug"). The CSL-JSON choice is a named survivor: `paper-adversarial-review` Survivors (line 101) lists "Zotero CSL-JSON over LLM metadata (Rao & Callison-Burch 2026)" and "durable anchored fulltext (Wang & Luan 2026)" among decisions "attacked from multiple angles and failed to refute." `paper-borrow-adapt-reject` Borrow rows #15–16 and the Immediate-design-feed digest engine spec (#6) back the co-located `digest.md`'s MASSW five-aspect schema and recursive batch-condense.

**REGRESSION STATUS — not a regression; the DB→markdown catalog move is treated as convergence, and a related provider-evidence gap (R3) was fixed.** `design §11` records the convergence with A15 on "SQLite authority + markdown keep-set + blobs"; `alpha-regression-review` does not list the `record.md` re-materialization among R1–R6. The one adjacent regression it does raise is **R3** ("Provider evidence is content-addressed and replayable," severity **Med**): B1's blob inventory was "closed to PDFs + embeddings … re-derivation is live re-fetch, not replay." Per `design §18(3)` and `alpha-regression-review` Resolution, R3 was **fixed** with "a content-addressed `payloads` store (operational-ledger class)" (§5/§7) — this concerns raw provider responses beside the DB, not `record.md` itself.

**RE-HOMING / REJECTION — `docs-history` records the deliberate reversal to co-located PDFs and the CSL-JSON boundary-subset adoption.** Reject row #123 ("Zotero anti-patterns") states "beta.1 deliberately reversed to `works/<citekey>/source.pdf` for the keep-test" and rejects the Zotero-owns-the-PDF deep-link and `.bib` auto-export patterns ("The `.bib` file is a second source of truth with built-in staleness and sync races"). Reject row #125 records the CSL-JSON-as-named-boundary-subset adoption. Borrow row #40 ("Citekey stability") ties the folder to identity — "unpinned keys … orphan the `works/<citekey>/` folder, every wikilink, and the BibTeX keep-test output" — and requires "Pin the citekey at import."

### Bullet 4 — Prose "operation test" replaces per-operation `method_class`/capability-manifest declaration

**RECORDED vs SILENT.** The prose operation test is recorded as a decision-of-record in `design` §13 ("The operation test — which LLM calls belong in the product"), with four criteria — "(1) pinned prompt + pinned model, so it ships with a fixture and a seeded-error verdict; (2) deterministically checkable output (schema, verbatim-anchor, byte-substring checks); (3) an arithmetic worst-case price (`max_tokens` ceiling + a Ledger price-table row, batchable); (4) per-item, corpus-scale, unattended shape" — and two litmus questions ("could it ship with a fixture and a seeded-error test?" and "can its worst-case price be printed before it runs?"). §13 states the rule is generative ("New LLM features must pass the same test") and applies it to the fixed roster. The substance A15's manifest carried — per-operation pins + mandatory fixture re-validation — survives in `design` §9: "Every run records the exact model id **and prompt-template version** (`costs.prompt_v`); **any model or prompt pin change re-triggers validation** of the affected engine against its fixture before it runs." The *replacement of the `method_class` declaration / capability-manifest registry* is not itself narrated as a delta in the design; §13 presents the operation test as the governing rule without naming what it displaces.

**RATIONALE.** The operations-as-product-code posture (against vault-resident capability manifests) is the recorded reason on the B1 side. `docs-bar` Reject ("keep dead") row "Prompts/operations as vault-resident data (fabric pattern library, capability Concepts with trust hashes)": "Alpha.15 decided operations are product code with pins, test modes, and calibration — vault-resident prompts escape all of it (untestable, unpinnable) … the Concept variant duplicated code-truth into data and needed drift doctors to survive." `docs-bar` Adapt row "Named epistemic operations + reading lenses" restates the shipping form ("ship as product-code operations with per-op pins and disposition telemetry, never as vault-resident prompts"). The operation test's price-arithmetic criterion is tied to the cost doctrine: `design` §13 routes any call lacking "an arithmetic worst-case price" to the researcher's agent, consistent with §9's `max_tokens` ceilings, and argues a resident loop "has no arithmetic worst-case price, which §9 requires of every call, and it is an uncontrolled variable inside a pre-registered experiment built on pinned prompts and frozen thresholds." `paper-bar` Borrow rows "Model id + prompt version in the run ledger" (Feng & Liu 2026) and "Deterministic-first cascade with tier logging" (CiteTracer — "log which tier decided each mc promotion") supply the ledger/logging substance the manifest had provided.

**REGRESSION STATUS.** `reg` classifies the substance as held: its "Checked and intact" sweep lists "**per-operation model pins with test/live modes**" among invariants verified present or strengthened. `method_class` as a named declaration is not among R1–R6 and is not named in the six enumerated recorded divergences, so `reg` neither flags the manifest-registry drop as a regression nor logs it as an explicit divergence — it affirms the pin/test-mode function held.

**RE-HOMING / REJECTION.** The `method_class` idea is re-homed as an **Adapt**, not a Reject: `docs-bar` Adapt row "Declared method class per operation" — old form "`method_class: deterministic|hybrid|generative` in skill frontmatter" → "**Trivially enforceable now that operations are product code: declare the class as operation metadata and have runner --mode test assert zero LLM calls for deterministic ops** — cost regressions become structural, not accidental." `docs-bar` Borrow row "Engine output contracts" re-homes the operation-contract discipline: "Schema-constrained decoding at temperature 0 with explicit closed label sets and a mandatory N/A abstention … **Generator ≠ checker**: the operation that generates never verifies its own output, ideally under a different model pin." `docs-bar` Reject row "Prompts/operations as vault-resident data" rejects only the vault-resident *capability Concept* form, not the product-code operation registry.

**ADVERSARIAL PRESSURE.** No `paper-rev` numbered finding attacks the operation test or the `method_class` replacement; the design's operation-vs-agent boundary is defended structurally in the `paper-rev` Survivors list ("Agentic tool loops: Rabinovich & Anaby-Tavor 2025, τ-bench") and in `paper-bar` Reject rows for agent runtimes ("Agent runtimes / multi-agent frameworks") and credentialed orchestrators ("Credentialed, socketed LLM orchestrator") plus `docs-bar` Reject "Agent-executed code loops," but none pressures the choice of a prose test over a declared `method_class`.

**SILENCE / CONFLICT.** There is a mild conflict the reports leave unresolved: `docs-bar` (Adapt line 95) recommends *keeping* `method_class` as enforceable operation metadata with a test-mode assertion, whereas `design` §13 adopts a prose operation test and neither declares a `method_class` field nor stands up a capability-manifest registry — the cost-regression-becomes-structural goal `docs-bar` attaches to `method_class` is instead met in `design` §9 via per-run `model_id`+`prompt_v` recording plus pin-change fixture re-validation. No report explicitly records the decision to replace the per-operation `method_class`/manifest declaration with the §13 prose test; the substitution is legible only by comparing `design` §13/§9 (operation test + ledger-enforced pins) against the `docs-bar` Adapt row that would have revived the declared class.

## 7. Catalog / bibliographic authority

### 7.1 Bibliographic authority location, primary key, and `bibliography.bib` generation (SQLite/`work_id` → `record.md` CSL-JSON/citekey)

**RECORDED vs SILENT — recorded, across four design sections plus the §17 information-flow resolution.**

- The `record.md`/full-CSL-JSON decision is stated verbatim in design §3: the layout block labels `record.md` as "catalog record; FULL CSL-JSON in a fenced block" (design l.197), and the File-vs-DB-vs-blob rule reads "**Full CSL-JSON is serialized into `record.md`** as a fenced code block (the derived DB is rebuildable, so bibliographic fidelity and the Zotero re-import key cannot live only in the DB)" (design l.222–226). That sentence is the recorded rationale for taking the fidelity/re-import authority out of the DB-only position.
- The citekey-as-primary-key move is recorded in the §5 schema: `works(citekey PK, doi, title, year, csl_json, status, pdf_sha256, distrust_flags)` (design l.374) — citekey is the primary key and `csl_json` is *also* carried in the row, so the DB is a rebuildable projection of the `record.md` authority, not the authority itself.
- `bibliography.bib` generation from stored CSL-JSON is recorded twice: "`bibliography.bib` stays Memoria-generated from stored CSL-JSON — complete across non-Zotero works (repos, direct DOIs) — with the researcher's own BBT auto-export as an optional cross-check, never the artifact of record" (design §7 l.584–587), and the export path emits "`references.bib` (from stored CSL-JSON, cited works only)" (design §8 l.838–839).
- The information-flow resolution §17.6 records this as a settled divergence: "**BBT citekeys inherited, generator kept** — one collision-checked key namespace across Zotero and non-Zotero works; `bibliography.bib` remains Memoria-generated and complete, the BBT export an optional cross-check" (design l.1261–1264).

**RATIONALE — the keep-test doctrine, stated explicitly.** The load-bearing reason in the reports is that a rebuildable DB cannot be allowed to be the *only* home of bibliographic fidelity or the Zotero re-import key (design §3 l.222–226). This is anchored in the keep-test evidence: research-evidence frames the keep-test as "portable markdown, files survive the tool… bibliographic records in a portable standard… nothing the researcher authored or the machine wrote into the record may exist only in an opaque store" (research-evidence l.48, l.101), and separately establishes that interchange export "(BibTeX/CSL-JSON for the catalog) documented as lossy and NOT a backup" (research-evidence l.308) — i.e., a portable bibliography is required *in addition to* the whole-vault backup. The charter itself elevates this to a pass/fail Success Claim: "**Survival:** the vault remains useful as plain markdown plus standard bibliography files if Memoria dies" (charter Success Claim 4, l.22–23) — the requirement the `record.md`-CSL-JSON + engine-generated `bibliography.bib` pairing satisfies. IF-research reaches the same verdict from the reference-manager ecosystem: "Treat the standard bibliography export (.bib or CSL-JSON) itself as the keep-test artifact… Do not invent a proprietary catalog store as the only machine-readable record" (IF-research l.161–163, verdict "adopt"), and doi.org content negotiation is chosen precisely because it "return[s] the keep-test-native formats (BibTeX/CSL-JSON) directly" (IF-research l.200–202). The citekey-as-key half rests on the citekey-stability doctrine: docs-history's Borrow row "Citekey stability" says "Pin the citekey at import… unpinned keys drift on metadata correction and orphan the `works/<citekey>/` folder, every wikilink, and the BibTeX keep-test output. Citekey correction is a first-class revertible run" (docs-history l.40), which design §7 implements verbatim as pinned-at-import (design l.567–570) + "correction is a first-class revertible rename run" (design l.575).

**REGRESSION STATUS — not classified as a regression; adjacent keep-test invariant verified held/strengthened.** The alpha-regression-review does not list the SQLite→`record.md` authority shift or the `work_id`→citekey primary-key change among the six confirmed regressions R1–R6, nor among the fourteen recorded divergences (of which six are named — "agent/co-PI machinery, kanban/spaces, hub/tag graph, claim-context fields, qmd, provider attention routing" — regression-review l.78–81). The closest positive verdict is in the "Checked and intact (swept, holding)" sweep, which lists "the keep-test (markdown + BibTeX survive the tool)" among invariant families "verified present or **strengthened**" (regression-review l.64, l.72–73). Note the tension worth flagging under SILENCE/CONFLICT below: design's own §11 does *not* describe this as a reversal — it lists B1 as **converging** with alpha.15 on "SQLite authority + markdown keep-set + blobs" (design §11 l.965). The reports thus treat SQLite as still-authoritative for the derived store, with `record.md` CSL-JSON scoped to the fidelity + re-import key "that cannot live only in the DB" — a narrower framing than the gap bullet's "moves bibliographic authority back into markdown." The nearest recorded regression, R3, is adjacent but distinct: it concerns raw *provider* payload replayability, fixed by a content-addressed `payloads` store (regression-review R3, l.41; design §18(3) l.1290–1292), not the `record.md` catalog authority.

**RE-HOMING / REJECTION — one Reject row directly ratifies the B1 choice; one Borrow row supplies the citekey rule; one Reject row licenses CSL-JSON as a boundary format.**
- docs-history Reject "Zotero anti-patterns" is the on-point row: it kills the BBT ".bib auto-export interface" because "The .bib file is a second source of truth with built-in staleness and sync races — beta.1 chose the local API for this reason" (docs-history l.123). That is the recorded reason `bibliography.bib` is engine-generated rather than an ingested BBT export.
- docs-history Borrow "Citekey stability" (l.40) re-homes the pin-at-import + revertible-rename mechanism that makes the citekey safe as the primary key.
- docs-history Reject "Git-as-enforcement + standards-conformance substrates" ends with the governing rule "Adopt named subsets at boundaries (CSL-JSON, W3C selectors); never let a standard become the internal model" (docs-history l.125) — CSL-JSON is sanctioned specifically as a boundary/interchange format.
- docs-history's "Patterns in what was deleted" adds the procedural warning that governs the enrichment side of this record: "The project repeatedly builds the enforcement half and loses the consumption half… A mechanism without a named consumer decays into decoration" (docs-history l.136), the reason the design pairs `record.md`/CSL-JSON with named consumers.

**ADVERSARIAL PRESSURE — none against; one Survivor entry defends the direction.** No numbered adversarial finding (BLOCKER/CRITICAL/HIGH/MED) attacks the `record.md` CSL-JSON authority, the citekey primary key, or engine-generated `bibliography.bib`. The adversarial-review Survivors list positively defends the adjacent choice: under "Capture and thesis mechanics," "**Zotero CSL-JSON over LLM metadata (Rao & Callison-Burch 2026)**" is listed among decisions "the corpus attacked from multiple angles and failed to refute" (adversarial-review l.89, l.101). paper-BAR's Borrow row for the same paper adds the mechanism: metadata correction runs "as a distinct revise-against-Zotero/Crossref pass (78.3% fully correct, 0.8% regression)" (paper-BAR l.21) — i.e., authoritative-record reconciliation, not vault-side LLM authoring, is the sanctioned correction path.

**SILENCE / CONFLICT.**
1. **The "work_id → citekey primary" reversal is not corroborated as a reversal by any report.** No beta.1 report describes the alpha `work_id`/provider-ID-primary state; design §11 instead claims *convergence* on "SQLite authority" (l.965). The gap bullet's "A15 made SQLite the authority with `work_id` primary" is therefore asserted by the gap doc but neither confirmed nor denied by the reports, which frame the DB as an authoritative-but-rebuildable derived store with `record.md` holding only the two things that "cannot live only in the DB" (design l.222–226).
2. **The winning design overrides best-practice research on *how much* metadata to mirror.** IF-research recommends keeping vault-mirrored metadata "a machine-owned, regenerable cache kept minimal (citekey, ids, title)" and treating the *BBT/CSL export* (not a vault file) as the keep-test artifact (IF-research l.104, l.161–163, l.983–989). The superseded IF-design followed this literally: a minimal "machine header" catalog page and "`references.bib` — BBT auto-export — the keep-test bibliography artifact" (IF-design l.53, l.62, l.223). The system design instead serializes **full** CSL-JSON into `record.md` and makes the **engine-generated** `bibliography.bib` the artifact of record. §17.6 records the resolution of the generator half (generator kept; BBT export demoted to optional cross-check) but does **not** explicitly reconcile the "full CSL-JSON in `record.md`" choice against IF-research's "keep the mirror minimal" recommendation — a divergence the reports leave un-narrated.
3. **Identity-law divergence, unreconciled.** IF-research (l.104) and IF-design (l.32, "Identity law: filename = Better BibTeX citekey; identity = frontmatter `zotero-key` + `doi`… Provider IDs are revalidatable cache keys, never primary keys") make the immutable Zotero item key + DOI the identity and the citekey merely the filename. The winning design makes citekey the DB **PK** (design l.374) and demotes the Zotero key to `csl_json.custom.zotero_key` "for idempotent upsert" (design §7 l.557). §17 records §17.6 (citekeys inherited) but never restates or overrides IF-design's "identity law," so the citekey-as-PK-vs-citekey-as-filename difference is a silent divergence between the superseded input and the governing design.

### 7.2 Zotero re-centered as primary reading/capture surface (optional → day-1 import, Zotero-first reading, "capture in Zotero" handoff)

**RECORDED vs SILENT — recorded as a deliberate, high-priority beta.1 move in the charter, §7, §11, §16, and §17.1.**

- The charter fixes Zotero into the baseline the beta is measured against: the experiment tests "whether Memoria beats the current null workflow: **Zotero + Obsidian + ad-hoc ChatGPT**" (charter l.9–11), the Parity success claim is "capture is no slower than Zotero" (charter Success Claim 1, l.15), and the Capture requirement enumerates "PDF, DOI/arXiv ID, BibTeX-file, **Zotero**, and GitHub-repo import" (charter l.29). eval-protocol operationalizes it: the null-workflow baseline is "capture volume/timing reconstructed from **Zotero `dateAdded`**" (eval-protocol §2 l.28–30).
- §11 records the re-centering as the top back-pressure item: B1 back-pressures the alpha plans with "(1) **Zotero import at fidelity** — the single largest missing item on the path to beta.1" (design §11 l.974–975).
- The information-flow resolution §17.1 records Zotero-first reading as an adopted outcome: "**Reading is Zotero-first** — the primary read is Zotero's PDF reader with append-only annotation sync, the exact-quote anchor bridge, and the processing card (§7); `fulltext.md` is demoted to the evidence substrate (§1) and remains the full path for non-Zotero works" (design l.1241–1244).
- §7 records the day-1 import mechanics (Zotero 7 local HTTP API at `127.0.0.1:23119`, Guard-allowlisted; CSL-JSON records + attachment paths; PDFs copied from Zotero storage via filesystem; Zotero key stored in `csl_json.custom.zotero_key` for idempotent upsert; design l.554–559), the Zotero-first reading/annotation pipeline (design l.697–720), and the PDF-acquisition rule: attachment first, "otherwise the work rests at `no_pdf` with a one-click **'capture in Zotero' handoff** — Zotero is already the designated capture surface in the null workflow" (design l.561–563).
- §16 records the plugin-coexistence consequence: "Zotero is an *app* reached over its local API — Zotero-first reading and the annotation bridge (§17) need no Obsidian-side Zotero plugin" (design l.1207–1208).

**RATIONALE — adoption/abandonment-first + parity, plus the egress-allowlist and keep-test doctrines.**
- Adoption-first: beta.1 is "a retention experiment… It must prove or falsify whether Memoria beats the current null workflow" (charter l.9–11). Reading where the researcher already reads, and measuring capture against Zotero, follows directly.
- Zotero's survival/embedding evidence: research-evidence establishes "Zotero's 20-year survival formula: own one narrow, unavoidable job embedded in the researcher's existing obligations, with immediate per-use payoff" and that "the null workflow (Zotero + Obsidian + ad-hoc ChatGPT) is composed of the category's proven survivors, which sets the trust bar Memoria must never fall below" (research-evidence l.110–113, l.126). IF-research adds: "capture is a solved problem and is never why researchers quit… Zotero stays the single system of record and the vault only pulls" (IF-research l.977).
- Three-host egress allowlist / no-beacon: the "capture in Zotero" handoff is chosen because it "adds zero egress; a Memoria-side OA-PDF fetcher stays out (an auto-fetch at import time is both an allowlist violation and a zero-interaction beacon)" (design §7 l.561–567), reaffirmed in §17.3 "a Memoria-side OA-PDF fetcher stays out via the Zotero capture handoff" (design l.1250–1251).
- Layered read-only Zotero architecture and one-way sync: IF-research l.104 supplies the pattern the design adopts — the local HTTP API as the "sanctioned live-read surface," never `zotero.sqlite` reads, "metadata and annotations flow strictly Zotero→vault, and vault-authored content never syncs back," with role partition ("Zotero owns bibliographic truth, the vault owns intellectual content").
- keep-test caveat on the re-centering: docs-history records that B1 keeps the PDF blob in the vault rather than living on Zotero deep links — "beta.1 deliberately reversed to `works/<citekey>/source.pdf` for the keep-test" because "deep links make the corpus die with Zotero's install" (docs-history l.123). So Zotero is re-centered for the *reading experience*, but the corpus does not become Zotero-dependent for survival.

**REGRESSION STATUS — not a regression; treated as an addition/back-pressure, so unclassified by the regression sweep.** The alpha-regression-review does not list Zotero re-centering among R1–R6 or the fourteen recorded divergences (regression-review l.78–81). Because Zotero-first reading/annotation is an *addition* over alpha (the gap doc itself files reading/annotation under "new-in-B1," §16), it is not an alpha invariant that could "regress"; §11 books it as back-pressure priority (1) rather than a divergence (design l.974–975). The regression review's adjacent positive verdicts are the keep-test ("markdown + BibTeX survive the tool," strengthened, regression-review l.72–73) and "discovery-proposes-never-imports" (l.75) — neither contradicts Zotero re-centering.

**RE-HOMING / REJECTION — the Zotero-import surface is deliberately shaped by two Reject rows.**
- docs-history Reject "Zotero anti-patterns" both licenses and bounds the re-centering: it endorses the local API over the `.bib` interface and reverses the "Zotero-owns-the-PDF deep links" pattern to an in-vault `source.pdf` (docs-history l.123). So the *interface* to Zotero (local API, in-vault blob) is a deliberate re-home, not a wholesale adoption of the community Zotero-import pattern.
- docs-history Reject "Mandatory ingest interview" (l.124) keeps the re-centering friction-light — a required per-source LLM conversation is refused, consistent with the charter's "one initiating action and no mandatory per-item decisions" (charter Capture row, l.29).
- IF-research's recommended BBT-auto-export interface (IF-research l.983–989) is the one piece the design *overrode* (see §7.1 conflict #2): §17.6 keeps the generator and demotes BBT export to an optional cross-check.

**ADVERSARIAL PRESSURE — none against; Survivors defend the surrounding decisions.** No numbered adversarial finding attacks Zotero-first reading or the capture handoff. The adversarial-review Survivors list positively defends "onboarding as retention mechanism (Luger & Sellen 2016, Epstein et al. 2020, Brown et al. 2007)" and "Zotero CSL-JSON over LLM metadata (Rao & Callison-Burch 2026)" (adversarial-review l.101). Finding 5 (superseded/retracted works, HIGH, adversarial-review l.47) touches capture — a retracted paper "stays clean-provenance forever" (l.47) — but its remedy is a `superseded` flag + Crossref audit probe (adversarial-review Finding 5 recommendation l.55; design §4 l.339–353), not a critique of Zotero-centering. The containment-only security posture is a Survivor (adversarial-review l.91), and the Zotero handoff's zero-new-egress property (design l.561–567) is consistent with it.

**SILENCE / CONFLICT.**
1. **"Zotero optional" (the A15 side) is asserted only by the gap doc.** No beta.1 report quotes the alpha "Memoria does not require a live Zotero library" statement. What the reports do show is that B1 re-centers Zotero as *primary/default*, not strictly *mandatory*: non-Zotero capture paths (direct DOI/arXiv, PDF, BibTeX file, GitHub repo) remain first-class (design §7 l.529–552), `fulltext.md` "remains the full path for non-Zotero works" (design §17.1 l.1244; §1 l.79), and §16 treats Zotero as a companion *app* reached over its local API that lives *outside* Obsidian (design l.1206–1208), not an embedded hard dependency (Zotero Integration/Citations plugins "remain unnecessary, since `bibliography.bib` is engine-regenerated," design l.1210–1211). The charter — the authority on what beta.1 leaves out — does **not** list Zotero among its Deliberate Exclusions (charter l.82–89); it instead names Zotero a required Capture input (l.29) and the null-workflow baseline (l.11). The reports therefore support "Zotero re-centered as primary reading/capture surface" but not a claim that Zotero became a hard requirement.
2. **One superseded-input divergence, resolved.** IF-design's PDF-acquisition rule allowed an allowlisted OpenAlex `best_oa_location` fetch chain (arxiv/PMC/doi.org) *before* falling back to the Zotero handoff (IF-design l.84). The winning design tightens this: the Zotero handoff is the fallback and "a Memoria-side OA-PDF fetcher stays out," with named OA hosts only "the recorded fallback if the handoff measurably fails" (design §7 l.561–567; resolved in §17.3, design l.1250–1252). This is recorded, not silent, but it is a divergence from the superseded input that centers Zotero *more* than IF-design did.

## 8. Discovery / gap analysis

### No resident Co-PI agent — the agent role externalized to the researcher's own client under the ADR-130 contract

The one bullet in §11 asserts, in the gap analysis's own terms: **A15** ships an ADR-130 read-API **plus a Co-PI**, with operations as product code; **B1** states "Memoria is not an agent, and beta.1 adds none," makes any agent (first-party or the researcher's Claude Code/Cursor) a **client** under the ADR-130 contract (read-API + request envelope, quarantined, read-scoped, never file/terminal/exec/send), with chat additive, never primary; ⇒ the gap that B1 ships **no resident Co-PI** and externalizes the agent role to the researcher's own client (alpha15-gap-analysis §11). Multiple reports independently record this delta as a deliberate, recorded divergence rather than a silent drop (developed below).

**Recorded, not silent — and named as such.** The B1 side is an entire dedicated decision section: design **§13 "Agent boundary and LLM allocation (2026-07-03)"**, whose header states it is the "Outcome of the control-surface-vs-standalone-agent design review … Recorded so it is not re-proposed" (design §13). The gap's headline phrase is verbatim there: "**Memoria is not an agent, and beta.1 adds none.**" (design §13). The client-contract sentence is also verbatim: "Any agent — a future first-party worker or the researcher's own (Claude Code, Cursor) — is a **client under the ADR-130 contract**: read-API + request envelope, universally quarantined, read-scoped, never granted file/terminal/exec/send tools; chat additive, never the primary path." (design §13). The alpha-regression-review independently classifies the delta as deliberate: in its "Checked and intact" sweep it lists "Fourteen further deltas (**agent/co-PI machinery**, kanban/spaces, hub/tag graph …) verified as **recorded** divergences in §11–§17 or the charter's exclusions, **not silent drops**" (alpha-regression-review, "Checked and intact"). Design §16 adds the ADR bookkeeping: the plugin+daemon host decision "amends ADR-125 … and **ADR-130 (surfaces)** and **requires that ADR before implementation begins**" (design §16, "ADR obligation, named").

**Rationale the reports give for the B1 side.** Design §13 gives the primary reasoning, resting on named load-bearing doctrines:
- *Single-writer / framework-inversion (ADR-125 Hermes lesson).* "A resident reasoning loop re-creates the exited Hermes failure with the loop in the framework's seat — it becomes the thing that owns state, workflow, and the write boundary, and every invariant in §0 must be threaded through a component that changes with each model generation (ADR-125's recorded lesson; ADR-130 rejects the same inversion at the UI layer)" (design §13).
- *Cost-ceiling doctrine (§9).* A resident loop "has no arithmetic worst-case price, which §9 requires of every call" (design §13).
- *Pre-registered-experiment integrity.* It "is an uncontrolled variable inside a pre-registered experiment built on pinned prompts and frozen thresholds" (design §13).
- *The operation test* draws the boundary: Memoria runs an LLM call directly iff it is an "operation" — pinned prompt+model shipping with a fixture/seeded-error verdict, deterministically checkable output, an arithmetic worst-case price (`max_tokens` + Ledger row), per-item/corpus-scale/unattended; the two litmus questions are "could it ship with a fixture and a seeded-error test?" and "can its worst-case price be printed before it runs?" Failing either, "the call belongs to the researcher's agent — open-ended, interactive, cross-tool, or voice-laden work, on their model and their bill, entering the vault only as envelope writes that land unchecked like everyone else's" (design §13). The charter's discovery exclusion is the worked example: "'research the literature for me' fails it (open-ended, external)" (design §13; charter Deliberate Exclusions row "Open-ended external search and recommendation," requirements).
- *No-write-time-oracle / proposer-never-authority (ADR-129), single-writer, egress.* "Neither side, ever: confers a verdict (checks promote; every LLM opinion is a proposer — ADR-129), performs a rule-derivable write (engines write), writes human-class text, or chooses an egress destination" (design §13).
- *Deliberate redundancy of Memoria's own synthesis with a frontier agent.* "Synthesis is **deliberately redundant** with what a frontier agent could do better: the product copy exists for the agent-free direct path and for instrumented payoff attribution; agent-side synthesis over the same Ask evidence is welcome and unmeasured. The redundancy is the direct-access contract working, not waste" (design §13). ## 12. Security / OS confinement

### OS-level egress enforcement is Linux-only; seccomp / macOS `sandbox-exec` / Windows-Firewall+UAC profiles are cut

**RECORDED vs SILENT — recorded, and explicitly as a cut, in the design (not in the charter's exclusions table).** The B1 side of this bullet is stated verbatim in design §10 under "**OS confinement: Linux network-namespace only**" (`design.md` lines 904–913). The design records the enforcement floor ("on Linux the worker additionally runs in a `CLONE_NEWNET` namespace with no interfaces — cheap, kernel-enforced, and literally the charter's 'network namespace' wording") and then names the deletions outright: "The seccomp fallback, macOS `sandbox-exec` profile, and Windows-Firewall/UAC flow are **cut** — two of them bought near-zero enforcement for real code and a 3-OS sandbox matrix." The per-OS disclosure is also a recorded, load-bearing sentence, not an omission: SECURITY.md "states the per-OS truth plainly: on macOS/Windows, egress restriction is process-architecture discipline, not OS enforcement" (§10, lines 911–913), and the §10 honest-residual list (lines 957–959) carries "on macOS/Windows, no OS egress enforcement" as one of the named residual risks. §11 re-lists it: item (6) "**brokered egress + Linux netns** as the enforceable form of the network allowlist" (lines 982–983), placed in the (4)–(8) band that "sharpen mechanisms alpha.15 already builds" (lines 989–990) — i.e. the design frames the Linux-netns form as a sharpening of an existing allowlist mechanism, not a new scope item. The recording locus is the design body only: the OS-sandbox cut does **not** appear in the charter's Deliberate Exclusions table (`requirements.md` lines 80–89), whose sole security-adjacent row is "Support-org machinery — SLAs, polished debug bundles, and continuous observability wait until there is a support surface" (line 89), which speaks to observability/support machinery, not the seccomp/`sandbox-exec`/Windows-Firewall matrix.

**RATIONALE.** The design's stated reason the structural guarantee survives the cut is that OS sandboxing was never carrying the guarantee: "The supervisor/worker split with brokered egress *is* the charter's 'allowlist mechanism'" and "on every OS the worker never holds the key and everything it writes passes the linter" (§10, lines 904–913). The load-bearing doctrine is brokered egress — "Engines get **no sockets at all**; all egress is a typed IPC call to Guard" (§10, line 882) — over a "Compiled-in allowlist (no config surface), every entry requirement-driven with its own audit" (lines 887–903), what the docs-history report repeatedly names "the three-host egress allowlist" (`docs-history-B/A/R` lines 99, 108, 115; the design's actual enumeration is wider: pinned LLM host, `api.crossref.org`, `api.openalex.org`, `api.github.com`, `citation.doi.org` redirect-disabled, Zotero localhost). The reason OS enforcement is not the security spine traces to the research: `research-evidence.md` line 320 states the 2025–2026 convergence that "prompt injection is unsolved… Every credible shipped defense is therefore architectural," and line 325 casts the whole posture as cutting "leg 3" of the lethal trifecta ("engines that read extracted fulltext must have no network capability beyond the fixed LLM API endpoint"); the SECURITY.md honest-residual framing is prescribed at lines 354–357 ("honest residual-risk warning… what distinguishes 'a beta with a security posture' from 'a beta with security vibes'"). Design-alternatives graft #12 supplies the closest recorded rationale for deferring the broader machinery: "Keep safeguards that protect irreplaceable data; defer support SLAs, debug-bundle polish, and **continuous egress machinery until there is a real support surface**" (`design-alternatives.md` lines 54–56) — though this graft is worded around support-org observability machinery rather than the OS-sandbox matrix specifically.

**REGRESSION STATUS — not a regression; the surviving mechanism is swept as held/strengthened.** The alpha-regression review's "Checked and intact (swept, holding)" list explicitly clears the egress spine: "brokered fail-closed egress with the compiled allowlist and the exfiltration linter" is among the invariants "verified present or **strengthened**" (`alpha-regression-review.md` lines 64, 73–74). The Linux-only limitation is **not** one of the six confirmed regressions (R1 block-hash, R2 export gate, R3 payloads, R4 rejection fixtures, R5 backup doctor, R6 frontmatter strictness), and it is not among the named examples of the "Fourteen further deltas… verified as **recorded** divergences" (lines 78–81, which enumerate "agent/co-PI machinery, kanban/spaces, hub/tag graph, claim-context fields, qmd, provider attention routing"). Design §18 (the recorded alpha-regression resolutions, `design.md` lines 1274–1312) confirms this from the design side: it lists only the six R1–R6 outcomes, none of which touches brokered egress, the OS-sandbox cut, or the netns floor; the sole §18 item that even references §10 security is R4's "rejection→fixture ratchet… mirroring the security-fixture ratchet (§1, §10)" (line 1294). Note a naming trap: the review's only use of the phrase "confinement is deferred" (line 101) is the R6 resolution about the frontmatter "`x.*` extension-namespace confinement" (the word "confinement" appears only there and at line 49, both frontmatter-scoped), **not** OS confinement; the report never classifies the OS-sandbox cut as a regression or a divergence line item at all — it is subsumed under the brokered-egress sweep.

**RE-HOMING / REJECTION.** docs-history re-homes the Guard/allowlist substrate as a **Borrow** ("beta1"): "Guard liveness + fail-closed contract" (`docs-history-B/A/R` line 35) — "Guard fails closed when its policy cannot load; workers refuse to start if Guard/linter is not loaded… Researcher customization of the egress allowlist is tighten-only: may add denies, never widen past the shipped baseline" (sourced to alpha.9 hermes-clean-slate-design.md, policy-mcp.md) — which design §2 (Component map) reflects verbatim ("Guard fails closed when its policy cannot load, and engine workers refuse to start unless Guard and the linter are registered," lines 178–179). The compiled-allowlist choice is defended by a **Reject** row: "Capability enumeration escape hatches (generic REST passthrough; denylist + drift doctor)" is rejected because "any-URL wrappers are exactly the exfiltration channel the broker closes; enumerated denylists trail the runtime forever. New providers are named allowlist entries with their own audit" (line 120). Two further Reject rows enforce the same allowlist as structural: "External channels (Telegram… Todoist)… Violates the three-host egress allowlist and default-deny linter" (line 115) and the nightly-discovery loop (line 108). The recommended-actions list adds "Guard fails closed on policy-load failure, workers refuse to start without it, denials are surfaced never swallowed, plus a deny-spike-after-ingest security signal and standing negative deny-tests" (line 153). No docs-history or paper B/A/R row addresses the **per-OS sandbox profiles** themselves (seccomp / `sandbox-exec` / Windows Firewall) — the re-homing rows are all about Guard, the broker, and the allowlist, never the OS-enforcement layer that was cut.

**ADVERSARIAL PRESSURE.** The paper adversarial review lists "**Containment-only security; injection classifiers excluded**" in its **Survivors** — "Decisions the corpus attacked from multiple angles and failed to refute" (`paper-adversarial-review.md` lines 87–91), citing Zhan et al. 2024 (InjecAgent), Greshake 2023, Kang 2023 ("100% filter bypass"), Branch 2022 ("every detection metric fails"), Debenedetti 2024 (AgentDojo), Andriushchenko 2025, Perez & Ribeiro 2022 ("injection success scales *with* capability"), Shaikh 2023. It records one residual audit that pressures the containment design (not the OS layer): "verify no single engine both reads untrusted text and holds a capability pairing sufficient to exfiltrate" (line 91), plus the supply-chain corollary to "pin and hash the deberta-mnli and bge weights." No adversarial finding attacks the Linux-only limitation or argues for restoring the macOS/Windows sandbox profiles.

**SILENCE / CONFLICT.** No report defends re-adding the cut OS-sandbox profiles or disputes the design's "near-zero enforcement for real code" claim about them. The evaluation-protocol does not instantiate any per-OS egress-enforcement test: its only security-relevant instrumentation is the `guard_log` ledger and its weekly export to `Memoria/system/metrics/` (`evaluation-protocol.md` line 46), with deny reasons logged as day-1 signals (line 50); there is no calibration/audit row exercising the Linux-vs-macOS/Windows enforcement gap.

### VaultWriter-owned exfiltration linter + canary drill + per-OS keychain as centered mechanisms

**RECORDED vs SILENT — recorded in detail; one internal framing conflict with the gap doc's "new" label.** All three mechanisms are specified in design §10. The linter: "**Exfiltration linter, VaultWriter-owned** (one owner), default-deny strip-never-flag, applied to every machine-writable class at apply and re-run over the whole bundle at export: the only surviving link shape is `https://doi.org/10.…` with no query/fragment/userinfo; every other URL → inline code span; `![…]` embeds get `!` escaped; raw HTML stripped; frontmatter serialized from typed fields only," extended to "extracted fulltext and imported metadata too" (`design.md` lines 914–922); it is one of the four single-writer invariants named in the design's opening principle 2 ("exfiltration neutralization," lines 29–32) and re-runs over citing-paper titles in §7 (line 742). The keychain: "**Keys** in OS keychain per platform (Keychain / DPAPI / Secret Service, with a mode-0600 fallback + logged warning on Linux)" (lines 948–949). The canary: "**One canary spec, owned by Security** (two incompatible canaries existed): `MEMORIA-CANARY-<8hex>` minted at install beside the key; weekly + post-migration, a drill injects it into synthetic output to prove the redaction path fires *and* scans all artifacts/DB/logs/exports for it and for key prefixes; any hit ⇒ `INC-canary-<date>` and engines disabled" (lines 949–954) — the parenthetical "two incompatible canaries existed" records a de-duplication decision. **Conflict with the gap doc's framing:** the gap bullet calls these "new in B1" ("A15: no exfiltration-linter-at-vault-write detailed"; the Appendix lists "per-OS keychain + exfiltration linter + canary" under "New in B1, not in A15"), but design §11's convergence list states the design "**Converges** with alpha.15 on… the exfiltration linter as mechanized don't-amplify" (lines 965–972), and the regression review sweeps it as an *already-present* invariant (below). The reports thus treat the linter as an existing-and-strengthened mechanism rather than genuinely new — a divergence from the gap doc's "new" characterization.

**RATIONALE.** The linter's rationale is the single most load-bearing item in the security research: `research-evidence.md` lines 338–341, "**Kill markdown image/link exfiltration — the single most-exploited leak channel in LLM tools that render markdown. Every major product shipped a fix**," with the Obsidian-specific mechanism spelled out — "Obsidian's preview auto-fetches remote images, so a machine-written note containing an attacker-supplied image URL IS a working zero-click exfiltration channel," answered by "a deterministic linter on engine output — cheap, testable, and closes the one channel that has actually been exploited in the wild." Research-evidence's own short checklist (`research-evidence.md` line 320) names exactly items (2) "a deterministic output linter stripping/flagging external URLs" and (4) "API key in the OS keychain, never in the vault, invisible to engine contexts, verified absent from exports." Keychain rationale is `research-evidence.md` lines 362–365 (Electron safeStorage; Claude Code's shipped "macOS Keychain when available… file permissions on Windows and Linux"; the Linux `basic_text` hardcoded-password failure mode — answered by the design's "mode-0600 fallback + logged warning on Linux"). The charter binds all three to Trust-scope bars: "Prompt injection — Untrusted text cannot control destination, tool choice, or vault egress" (`requirements.md` line 74), "Canary checks — Known key/canary strings do not appear in logs, exports, or model-bound payloads" (line 76), and Operations "Key handling — API keys never enter vault content, backups, exports, or logs" (line 65).

**REGRESSION STATUS — held/strengthened; not among R1–R6.** The alpha-regression review's "Checked and intact" sweep names "the exfiltration linter" directly as verified "present or **strengthened**" alongside "brokered fail-closed egress with the compiled allowlist" (`alpha-regression-review.md` lines 64, 73–74). Neither the linter, the canary, nor the keychain is one of R1–R6, and none appears among the fourteen recorded divergences (lines 78–81); design §18's six recorded resolutions (`design.md` lines 1274–1312) likewise leave all three untouched.

**RE-HOMING / REJECTION.** docs-history Borrows the ingestion-side threat framing that the linter operationalizes: "Ingestion-side threat model" (`docs-history-B/A/R` line 36, "beta1 / process") — "Name the dominant threat: a poisoned PDF whose legitimate read-and-reason path IS the attack — no egress allowlist stops it; treat extracted fulltext as adversarial input to every LLM call… adversarial PDF fixtures (planted 'ignore instructions' canary) as permanent test cases… Guard/egress deny-rate spike — especially right after a PDF ingest — surfaced as an injection signal" (reflected in the §10 poisoned-fixture corpus, lines 959–961). The Reject row at line 120 ("any-URL wrappers are exactly the exfiltration channel the broker closes") defends the linter/broker pairing. The paper B/A/R **Adapt**-class row is the most direct treatment: "Privacy as measured floor, not retention lever" (Coopamootoo & Groß 2017; Acquisti et al. 2015) prescribes "score trust incidents against objective events (**egress-log audits, canary drills**), render the no-OS-enforcement statement so it can't read as a reassurance badge, and surface 'this text leaves your machine' at the priced button" (`paper-borrow-adapt-reject.md` line 37) — endorsing the canary-drill and egress-audit mechanisms and the honest no-OS-enforcement disclosure.

**ADVERSARIAL PRESSURE.** The linter's owner survives adversarial attack: "**Single-writer VaultWriter, human-span sha256 pre==post gate, block-level read-only**" is in the Survivors list (`paper-adversarial-review.md` line 92; Bernstein 2010 Soylent "~30% of external edits to human prose were bad; verify caught only 50%", Yao 2024 τ-bench, Ruan 2024 ToolEmu 23.9% risky-action floor, Blumenthal & Clark 2001 endpoint check). The AgentDojo residual (line 91) — "verify no single engine both reads untrusted text and holds a capability pairing sufficient to exfiltrate" — is precisely the property the credential-free/socketless worker plus the write-time linter implement. The one supply-chain corollary the same Survivors entry raises ("pin and hash the deberta-mnli and bge weights," line 91) is itself a charter Trust-scope bar: "Model supply chain — Bundled local model weights (embeddings, NLI) are hash-pinned and verified at load; a poisoned checker must not be able to silently pass claims" (`requirements.md` line 78). No adversarial finding attacks the linter, canary, or keychain.

#### Facet A — egress guarantee is Linux-only (macOS/Windows OS-sandbox profiles cut)

**RATIONALE — one-way append-only is the settled community pattern; the `dateAdded`/ID watermark is the import-only-new mechanism.** `if-research` is dense here: "Keep sync strictly one-way Zotero→vault … do not sync vault-authored content back. ### Exact-quote → `^pNNNN` anchor bridge

**RECORDED vs SILENT — recorded as a deliberately *rescoped* mechanism.** `design §7`: each annotation "is **bridged to the evidence substrate by exact-quote match** — a verbatim hit in `fulltext.md` earns the annotation its `^pNNNN` anchor." The `^pNNNN` anchors are the extraction-time block IDs — "**Fulltext anchors:** one Obsidian block ID per paragraph, assigned once at extraction" (design §3), produced when extraction yields "linearized markdown with `^pNNNN` per paragraph" (design §7 extraction). The geometry lives in the `anchors(citekey, anchor, page, char_start, char_end, bbox_json, status, text_sha256, …)` table (design §5), rebuildable by re-running deterministic extraction (`mem rebuild` "recomputes the `anchors` geometry map by re-running deterministic local extraction against `source.pdf`"). The *rescoping* is explicitly logged: `if-design` §6 "an exact-quote anchor only … TextPosition/fuzzy re-anchoring is Hypothesis-grade engineering and goes to the deferral ledger," recorded in `if-design`'s own "Minors folded" note ("anchors reduced to exact quote + page + deep link with fuzzy re-anchoring deferred").

**RATIONALE — exact-quote-only trades recall for cost, grounded in Hypothesis's anchoring-stack cost; anchors are a keep-test-safe layer.** `if-design`'s Evidence trail compresses it: "Hypothesis → explicit orphans; its anchoring stack's cost → exact-quote-only in beta.1," with the underlying Hypothesis evidence in `if-research` (W3C multi-selector fuzzy anchoring + explicit orphan states, verdict adopt; annotation-lesson summary: every frictionless-capture tool "produces the same 'highlight graveyard' failure when processing is left implicit"). The **keep-test** doctrine is the reason anchors are a layer over plain markdown, not a compiled artifact: `paper-bar` Reject row 71 (Liu et al. 2026, ARA, "Machine-first vault restructuring"): "Take the measured benefit (bindings, **anchors**) as a layer on human markdown; refuse demoting the narrative to a 'compiled view' — it breaks the keep-test." Multi-span support is a stated property: `paper-bar` Adapt row 45 (Thorne 2018, FEVER) "Checker entails against the concatenation of all anchors a sentence cites, not per-anchor," echoed in `design §7` ("multi-span support is real — FEVER").

**REGRESSION STATUS — not a regression; the anchor infrastructure is *reused by* R1's fix.** The `anchors` table's `text_sha256` is the pattern the regression review recommends binding provenance to: R1's minimal fix is "Bind each `mc` to its block-text hash (**the `anchors.text_sha256` pattern**) + a DB check row" (regression-review, R1), and the adopted `design §18.1` resolution binds `mc` to the block's `block_sha256`. So the anchor layer is treated as sound infrastructure, not a regressed invariant. The A15 note-side `anchors` field (gap §3) is the "claim-context fields" recorded divergence, not a regression.

**RE-HOMING / REJECTION — anchor stability is a Borrow (citekey-rename fix-up).** `docs-bar` Borrow row 40 ("Citekey stability"): citekey correction is "a first-class revertible run (directory rename + **link/anchor/manifest fix-up** atomically)," so anchors survive citekey renames — mirrored in `design §7` ("directory rename + link/anchor/manifest fix-up in one run"). Fuzzy re-anchoring is filed as a deferral: `design §7` defers it ("fuzzy re-anchoring is Hypothesis-grade engineering, deferred until a measured orphan rate demands it"); the written revive-trigger lives in `if-design`'s deferral ledger ("**Fuzzy/TextPosition re-anchoring** — trigger: orphan rate exceeds X%").

**ADVERSARIAL PRESSURE — marker error-proneness (paper-bar, above) is the only direct pressure.** `paper-adversarial-review` does not attack the anchor bridge; its `mc`-grounding finding (#4, "mc reads as 'grounded,' but the Checker can only certify citation-correctness") pressures how anchored citations are *labeled* downstream, not how the annotation bridge assigns `^pNNNN`.

**SILENCE / CONFLICT — none recorded.** No report disputes exact-quote-only; the deferral of fuzzy matching is stated with a numeric trigger rather than left silent.

### The "processing card" batch triage

**RECORDED vs SILENT — recorded as the "starter motor," in the body and the resolutions.** `design §7`: "When a sync appends ≥N new highlights to a work (N a shadow knob), the next vault open shows one dismissible **processing card** — 'Process 14 new highlights on [work]' — opening a batch triage worklist (§1 review discipline): per row **promote / link / drop / snooze** … The research is unambiguous that initiation, not execution, is where annotation-processing collapses — **the card is the starter motor**." It is named in `design §17.1` ("the processing card (§7)") and in `design §1` ("annotations sync back append-only with a processing card, §7"). The anti-laundering guard is co-recorded: "promote opening a pre-typed note stub whose machine pre-fill **stays machine-class (`m` markers) until the researcher edits it** … a promoted note with zero human-typed characters is flagged in Review, because one-tap promotion of machine-drafted sentences would launder untrusted-derived prose into the human class" (design §7).

**RATIONALE — initiation-collapse research + adoption/abandonment-first; the anti-laundering guard is a checks-only-promotion / injection defense.** `if-design`'s Evidence trail (the "Annotate/notes" entry) compresses the annotation-pipeline literature: "Doto + Ahrens → collapse the pipeline; **initiation, not execution, is the collapse point → the processing card**"; "Matuschak → contextual resurfacing"; "Westenberg → anti-mausoleum instrumentation"; "n=405 RCT → machine asks, human writes → attributed pre-fill, unauthored flag." `if-design` §6 states the mechanism as "The starter motor, new — the collapse point is initiation, not execution … Without this, the resurfacing loop … is downstream of the very notes it is supposed to cause." The **adoption/abandonment-first** doctrine drives it (`design-alternatives`: Design A won partly because it "kept capture/annotation friction measurable"; `research-evidence` makes reuse rate the death signal — "captured but never revisited (the 'graveyard')"). The anti-laundering flag ties to the checks-only-promotion / **no-write-time-oracle** rule the regression review lists as intact (machine pre-fill never becomes human-class without a human edit; taint monotone, never clears — design §7/§4) and the charter **Prompt injection** Trust row; `if-design` §6 records the rationale verbatim ("one-tap promotion of machine-drafted sentences derived from adversarial document text would launder injectable prose into the human-owned region").

**REGRESSION STATUS — not a regression; it is an instance of the review discipline the review confirms is *not* an approval queue.** The processing card is the one point where gap §5 ("B1 re-introduces a review/disposition surface that A15 had removed") and §16 meet. `alpha-regression-review` affirms this surface is compliant, not a regressed approval gate: the intact sweep lists "the proposer-never-authority rule and **act/ask/log routing (not an approval queue)**." **RE-HOMING / REJECTION — a direct Borrow, plus a rejected alternative.** `docs-bar` Borrow row 28 ("Batch worklists for high-cardinality decisions"): "one table-backed worklist … never N queue items ('forty cards train select-all-accept') … **Applies directly to … the day-1 Zotero import**," sourced to `tutorials/04-build-a-reading-batch.md` — the processing card is one rendering of this borrowed pattern. `if-design` §0a (the plugin's one interaction pattern) makes the pattern shared: "annotation triage" is one instance of "a generic actionable-queue note format … a new queue costs a template, not a surface," and its shipped acceptance test is "highlight-to-typed-stub under 60 seconds." Rejected alternatives around it: `docs-bar` Reject row 124 (mandatory ingest interview, "duplicates annotations→notes") and Reject row 117 (graduated pre-write approval gate — "beta.1 replaced pre-write approval with revertible runs + provenance + taint + auto-rollback").

**ADVERSARIAL PRESSURE — the `if-design` attack resolutions, not the paper review.** The recorded adjudications sit in the input design §17 adopted: `if-design`'s attack resolutions list "**No processing initiation trigger** → sync-triggered dismissible card ('Process 14 new highlights on [work]') wired to the annotation-sync event (§6)," and "**Pass-1 friction-or-fiction** → decisions captured via a designated Zotero tag/color plus explicitly-labeled inferred states." `paper-adversarial-review` does not address the processing card; its overreliance finding (#7, "Ambient provenance cannot carry the anti-overreliance load") pressures the *marker* layer the promoted stubs render, and the charter's **Overreliance** Trust row ("attributed only to vigilance-independent mechanisms … never to the researcher noticing a marker") is the constraint the `unauthored`-flag-in-Review (a mechanism, not a marker) is designed to satisfy.
