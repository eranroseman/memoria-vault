# Memoria 0.1.0-beta.1 — MVP Requirements, Evidence & Technical Decisions

Date: 2026-07-05

Status: **governing.** This document is the MVP requirements, evidence
constraints, normative corrections, technical preservation decisions,
acceptance checks, open research questions, and exit verdict for beta.1. The
detailed system design that **implements** it is
[0.1.0-beta.1-design.md](./0.1.0-beta.1-design.md); this document names *what
must be true*, the design names *how*. beta.1 inherits alpha.16 — see
[../0.1.0-alpha.16/0.1.0-alpha.16-requirements.md](../0.1.0-alpha.16/0.1.0-alpha.16-requirements.md).
(Consolidates the former requirements/evidence companion and the
technical-preservation recommendations; originals archived under
`resources/superseded/`.)

## 1. Role Of Beta.1

Beta.1 is the first version where Memoria delivers the stated promise for a
real user on a real corpus:

- read: accumulate, compile, curate, inspect, and retrieve knowledge;
- write: compose/create new text and software requirements artifacts from that
  knowledge.

Beta.1 inherits alpha.16's READ cycle and knowledge-production substrate. It
adds the WRITE/output half: text drafting, SRD production, output verification,
usable onboarding, and feedback instrumentation. Executable code output is
post-beta.1.

Beta.1 is not "feature complete forever." It is complete enough that a real
researcher can use it as the primary workflow and the product can improve from
observed feedback.

## 2. MVP Loop

Beta.1 must run this loop end to end:

1. Open a project with a question, thesis, or output goal.
2. Seed it from existing sources and notes.
3. Run gap analysis over both the catalog graph and knowledge graph.
4. Source or capture new material to fill gaps.
5. Extract, digest, annotate, and process into human-owned notes/claims.
6. Re-run gap analysis and refine the project.
7. Build a project thread from claims/evidence into a canvas or equivalent map.
8. Derive an outline from that thread.
9. Draft text from coherent evidence slices, not isolated top-k fragments.
10. Create or curate a Software Requirements Document (SRD) for future
    implementation.
11. Verify citations, numbers, evidence sets, SRD traceability, provenance, and
    graph structure.
12. Export or refuse export with explicit unresolved gaps.
13. Record feedback, incidents, cost, accept/reject dispositions, and fallback to
    the null workflow.

Beta.1 succeeds only if the loop is repeatable without returning to Zotero +
Obsidian + ad-hoc ChatGPT as the primary workflow.

## 3. Normative Corrections From The Redo Comparison

Adopt these corrections into the detailed beta.1 design.

1. Beta.1 should not re-specify alpha.16. It should inherit alpha.16 and specify
   only the WRITE/output additions.
2. Writing and SRD production are not peripheral. They are the beta.1 output
   half of the mission.
3. The writing flow should be concrete: argument graph -> project canvas/thread
   -> outline -> draft -> verification/export.
4. Canvas is a small project-slice working surface, not a whole-vault graph UI
   and not the authority for the graph.
5. Outline generation is one-way and explicit: canvas/thread -> outline.
   Two-way live canvas/outline sync is out until real use proves it.
6. Drafting must use coherent slices: selected notes/claims plus expanded source
   sections. Top-k fragments alone are not enough for synthesis.
7. Every drafted claim must carry an evidence set, not just a citation string.
   Evidence sets can be single-span, multi-span, multi-hop, implicit, or computed.
8. Printed numbers must be source-backed, deterministically computed by the
   engine, or marked evidence-incomplete; model prose must not invent arithmetic.
9. The code-side beta.1 output is an SRD: requirements, non-requirements,
   acceptance criteria, constraints, risks, and open questions traced to checked
   knowledge or explicit PI judgment.
10. Generated source files, code execution, executable warrants, and sandboxed
    code verification are post-beta.1.
12. Review gates are not assumed to work. Beta.1 must measure whether reviewed
    outputs are actually better than unreviewed outputs.
13. Feedback should be event/disposition based, not a synthetic "quality" score.
14. Onboarding needs a real corpus and a fast grounded payoff. Generated demo
    corpora cannot validate the product.

## 4. Evidence Constraints

The alpha.16 evidence companion covers the storage, metadata, retrieval, NLI,
prompt-injection, and single-case evaluation facts. Beta.1 adds these derived
constraints:

| Evidence fact | Consequence |
| --- | --- |
| The mission explicitly includes both reading and writing. | Beta.1 must include text output and an SRD for future software work. A read-only beta does not satisfy the mission. |
| Obsidian/Markdown/JSON Canvas are inspectable file formats. Sources: [Obsidian storage docs](https://help.obsidian.md/Files+and+folders/How+Obsidian+stores+data), [JSON Canvas spec](https://jsoncanvas.org/spec/1.0/). | Drafts, canvases, SRDs, `note mode: work` records, and evidence files must remain usable without Memoria. |
| Zotero and DOI/Crossref/OpenAlex provide metadata/citation inputs with different reliability. Sources: [Zotero API](https://www.zotero.org/support/dev/web_api/v3/basics), [Crossref content negotiation](https://www.crossref.org/documentation/retrieve-metadata/content-negotiation/), [OpenAlex paper](https://arxiv.org/abs/2205.01833). | Citation export and output verification must resolve against deterministic metadata, preserve conflicts, and never let model prose author bibliography facts. |
| HANS/PAWS-style failures show that semantic checkers can be fooled by overlap and wording. Sources: [HANS](https://arxiv.org/abs/1902.01007), [PAWS](https://arxiv.org/abs/1904.01130). | Verification can check structure, spans, links, arithmetic, and suspicious contradictions. It cannot certify truth. |
| Indirect prompt injection can use retrieved untrusted content to steer tool-using systems. Source: [Greshake et al. 2023](https://arxiv.org/abs/2302.12173). | Draft/SRD generation must not give untrusted source text authority over writes, network, shell, export target, or secrets. |
| LLM application behavior changes with prompts, models, context, and sampling settings, so reliable use requires task-specific evals and regression comparison. LLM judges can approximate human preference at scale, but published work documents position, verbosity, and self-enhancement biases. Sources: [OpenAI evals guide](https://platform.openai.com/docs/guides/evals), [Anthropic evaluation tool](https://docs.anthropic.com/en/docs/test-and-evaluate/eval-tool), [Google model evaluation docs](https://cloud.google.com/vertex-ai/generative-ai/docs/models/evaluate-models), [OpenAI simple-evals](https://github.com/openai/simple-evals), [Zheng et al. 2023](https://arxiv.org/abs/2306.05685). | Every product-affecting LLM call site must have a versioned prompt/model/config record, a frozen eval suite with task metrics, and a promotion gate before a changed variable affects production behavior. LLM-as-judge may aid diagnosis only when calibrated; it is never the sole authority for truth, grounding, or blocker promotion. |
| Runtime values such as credentials and backing service handles vary by deploy and should be separated from code. Source: [Twelve-Factor App config](https://12factor.net/config). | LLM runtime mode and provider profile are runtime configuration: `production` uses the promoted production profile; `test` uses an isolated workspace plus local API or budget-capped cheap remote profile. |
| Single-user/self-use evaluation requires baseline and repeated measurement before benefit claims. Sources: [Neustaedter and Sengers 2012](https://antle.iat.sfu.ca/wp-content/uploads/NeustaedterSengers_2012DIS_AutobioDesignMethodology.pdf), [WWC single-case design guide](https://ies.ed.gov/ncee/wwc/Docs/ReferenceResources/wwc_scd.pdf). | Beta.1 must instrument real use against a null workflow and must allow adopt/redesign/kill outcomes. |
| Pre-release software can require backup/forward-migration discipline. Sources: [SemVer](https://semver.org/), [Zotero beta builds](https://www.zotero.org/support/beta_builds). | Backup, restore, rebuild, and migration drills are beta prerequisites, not polish. |

## 5. Required Beta.1 Capabilities

### Writing

The writing module must:

- consume alpha.16 claims, `note mode: work` records, hubs, evidence pins, and
  gap reports;
- propose a project thread/canvas from selected claims and evidence;
- derive an outline from that thread;
- draft section-by-section from coherent evidence slices;
- mark machine-authored text until accepted or rewritten;
- preserve human ownership of judgment and final prose;
- refuse export on unresolved citations, structurally unsupported claims
  (missing or broken evidence set, span, bibliography row, or numeric
  provenance), stale/superseded evidence, evidence-incomplete passages, or
  unresolved accepted gaps.

### Coding

The coding module must:

- create a project-scoped SRD as plain markdown;
- link SRD requirements to project claims, source rationale, constraints, and
  explicit PI judgment;
- record requirements, non-requirements, acceptance criteria, risks, and open
  questions;
- mark missing requirement evidence as evidence-incomplete or an SRD gap;
- allow the SRD to be exported with project provenance.

### Verification

Verification must separate facts:

- citation exists;
- quoted span exists;
- bibliography entry resolves;
- graph link/anchor resolves;
- number recomputes;
- SRD requirement traces resolve;
- content is machine-authored, human-authored, unchecked, stale, tainted, or
  structurally verified.

No label means "true." Semantic judgment remains the user's responsibility.

### Feedback

Beta.1 must record:

- accept/reject dispositions on recommendations;
- incidents: wrong citation, wrong answer, data loss, cost surprise, prompt
  injection hit, unsupported draft claim, SRD gap;
- time/cost for capture, reading, notes, ask, compose, SRD, export;
- fallback to null workflow;
- whether reviewed outputs outperform unreviewed/raw outputs;
- whether the tool produced a text/SRD artifact worth keeping.

### LLM Call Evaluation

Every LLM API call that can affect a user-visible recommendation, draft,
verification route, SRD action, retrieval synthesis, or export decision must:

- declare a stable `call_site_id`, purpose, prompt template version, model/
  provider, provider profile, runtime mode, decoding settings, context/retrieval
  policy, tool permissions, schema/validator, and cost/latency budget;
- separate `runtime_mode` from `llm_profile`: `production` mode may use only the
  promoted production profile and may write to the real workspace; `test` mode
  uses an isolated workspace and a local API endpoint or budget-capped cheap
  remote profile;
- have a frozen eval suite before `production_enabled:true`: representative
  task inputs, expected properties or ground truth where available, graders/
  deterministic validators, and known failure cases from incidents;
- use deterministic validators and reference/ground-truth checks where possible;
  human/PI adjudication for subjective cases; LLM-as-judge only as a calibrated
  diagnostic, never as the sole authority for truth, grounding, or blocker
  promotion;
- record each eval run with candidate variables, baseline variables, dataset
  hash, grader versions, pass/fail metrics, latency, token/cost data, and
  sampled outputs;
- require an explicit promotion decision when any prompt, model, retrieval
  policy, tool permission, schema, temperature/top-p, or relevant budget changes;
- stay shadow-only or fallback to the previous version when the candidate does
  not beat the registered bar or regresses any safety/grounding blocker metric.
- treat test-mode results as wiring/schema/regression evidence. They can promote
  production behavior only when the same `llm_profile` is the intended production
  target and the normal promotion gate passes.

## 6. Open Research Questions Before Freezing Beta.1

These questions are not excuses to remove MVP scope. They are blockers to
specific mechanisms, separated from measurements and already-ratified
implementation work.

The empirical collection sequence lives in
[`empirical-use-action-plan.md`](./empirical-use-action-plan.md).

### Freeze blockers

1. **SRD contract.** Code execution is descoped; beta.1 still needs the minimal
   SRD sections, traceability rules, and verification checks.
2. **Evidence-set review UI.** The durable schema is committed; beta.1 still
   needs the PI review surface for multi-hop/implicit evidence sets and SRD
   gaps.
3. **Seed corpus.** Licensing mechanics are settled for PMC Commercial-Use-
   Allowed and arXiv CC BY, CC BY-SA, or CC0; beta.1 still needs the actual
   citable source list and fetch/bundle rule.
4. **Workspace / gate topology.** The five workspace jobs are committed; beta.1
   still needs the concrete path for asking, review gates, return-to-work, and
   SRD actions.
5. **Export target.** Export refusal is committed; beta.1 still needs the first
   useful output target.
6. **Multi-device topology.** Single-user local-first is committed; beta.1 still
   needs the write/read topology.
7. **Raw dataset bundling.** Catalog-by-reference is understood; beta.1 still
   needs the rule for raw dataset files: referenced blob store, packaged
   companion, or out of scope.
8. **Template-driven creation.** The CLI must consume templates consistently,
   including the `note mode: work` creation path.
9. **Schema/docs drift prevention.** The OpenAlex renames are resolved; beta.1
   still needs generation or lint to stop schema docs drifting again.

### Checks before implementation start

Only these checks block implementation start or the first implementation slice:

- LLM eval harness preregistration: before enabling or changing any
  product-affecting LLM call site, record its call-site contract, baseline
  variables, fixture/eval cases, metrics, runtime-mode/profile split, and
  promotion bar;
- retrieval fixture preregistration: before building activation logic for
  dense/graph retrieval, record the BM25 baseline, fixture corpus, metrics, and
  adoption bar; run the fixture after the substrate exists;
- seed corpus source/license verification: before implementing onboarding around
  a bundled/fetched corpus, name the actual sources and verify their
  redistribution or fetch rule.

### Validation protocols

These are measurements beta.1 must run after the relevant mechanism exists, not
unresolved design mechanisms or reasons to delay implementation:

- PI review load;
- gate calibration;
- payoff vs. null workflow;
- one-way canvas → outline residual demand;
- canvas-vs-table threshold;
- draft write-back squeeze-point/review-lane behavior;
- exploration acted-on rate and diversity coverage;
- retrieval default activation, with BM25 remaining default until the
  pre-registered spike beats it;
- LLM call-site regression, with prompt/model/context/sampling/tool changes
  remaining shadow-only until the registered eval beats the active baseline and
  no blocker metric regresses;
- provider conflict rates and topic/facet ownership quality.

### Resolved or prerequisite, not open

- Query mechanics are governed by `query-mechanism-analysis.md`; the substrate is
  built, while default activation is fixture-gated.
- Coherent slices are whole-note-if-budget else section-expanded-anchor.
- Draft write-back is extract-to-atomic-note, transclusion, source-link, and
  `seedling`.
- Extract/compose is prose-first with structured closed-vocab slots and typed
  evidence sets.
- Markdown `work` is deleted; `source-note` folds into `note` as `mode: work`.
- `.memoria/` migration and OKF identity compliance are implementation
  prerequisites, not research questions.

## 7. Beta.1 Acceptance Checks

Beta.1 passes when:

1. A real project completes the source -> note -> gap -> source -> draft loop at
   least twice.
2. Zotero, bibliography, DOI/PDF, GitHub, and local-file capture all work or
   fail into visible triage states.
3. The project graph contains works / `note mode: work`, claims/questions/
   tensions, hubs/facets, evidence pins, gaps, draft sections, and SRD
   requirements.
4. Ask answers with cited spans, denominators, and honest absence language.
5. The query mechanism runs over a real corpus: `passages` + FTS5 +
   `sqlite-vec` + `concept_edges`; query-time reindex; same-transaction
   `check_status` cascades; filter-before-rank targeted lookup; bounded
   topic-surfacing neighborhoods with denominators; RRF fusion; BM25 default
   preserved unless the pre-registered spike beats it.
6. Gap analysis reports source gaps, argument gaps, and graph mismatch gaps.
7. A text draft is produced, audited, corrected, and exported or refused for
   explicit reasons.
8. An SRD is produced or curated, linked to project claims/rationale, verified
   for traceability/scope, and exported with provenance.
9. Export refuses unresolved citations, structurally unsupported claims,
   stale/superseded evidence, and unresolved SRD gaps.
10. Crash-at-boundary recovery proves the system-level contract: SQLite is ACID
    locally, DB+files are not claimed globally ACID, and every interrupted
    machine operation recovers to committed/consumable, retryable/pending, or
    failed/hidden with no torn output visible as checked.
11. A machine run can be reverted without altering human text.
12. Backup, restore, rebuild, and migration pass in a clean profile.
13. Feedback instrumentation is producing disposition, incident, cost, time,
    complementarity-calibration, and null-workflow comparison data.
14. LLM call-site evaluation is operational: every production-enabled LLM API
    call has a versioned call-site contract, frozen eval suite, recorded
    baseline, production/test runtime-mode split, local-or-cheap test profile,
    and promotion evidence for the active prompt/model/config.
15. If Memoria is removed, the user can still read the corpus and project state
    as OKF-compliant Markdown, including each `projects/<slug>/` subfolder as a
    nested knowledge bundle, plus PDFs/source files, code files, `.canvas`, and
    `bibliography.bib`.

## 8. Exit Verdict

Beta.1 exits with one of three verdicts:

- **adopt:** the loop beats the null workflow enough to keep using;
- **targeted redesign:** a named mechanism failed and has a bounded fix;
- **kill:** the null workflow won.

The verdict must be based on recorded events and artifacts, not optimism about
future versions.

## Technical preservation decisions

Which mechanisms are output/MVP-specific (preserved for beta.1) versus inherited
from alpha.16's substrate — the technical layer of the requirements above. The
inherited substrate is the "adopt now" set in
[../0.1.0-alpha.16/0.1.0-alpha.16-requirements.md](../0.1.0-alpha.16/0.1.0-alpha.16-requirements.md).

## Decision Rule

Beta.1 should not re-specify alpha.16's substrate. It should inherit alpha.16
and add only what is required for the first usable read/write MVP: text output,
SRD output, output verification, onboarding, and feedback instrumentation.

## Inherited From Alpha.16

Beta.1 should assume these are already present:

- `run_id` for machine writes, cost, review, incidents, and revert;
- single `VaultWriter`;
- crash-consistent operation recovery: ACID inside SQLite, observable
  committed/retryable/failed atomicity across DB+files;
- block-level human text protection;
- supervisor/worker split;
- Guard-brokered egress;
- Ledger cost authority;
- derived vs operational DB split;
- content-addressed provider payload store;
- Zotero-fidelity import;
- anchored fulltext Ask with denominators;
- VaultWriter-owned exfiltration linter;
- minimal backup/restore/rebuild/migration substrate.

If any of these are missing, beta.1 should not rebuild them differently. They
should be backfilled into alpha.16's substrate first.

## Preserve For Beta.1

| Mechanism | Recommendation | Reason |
| --- | --- | --- |
| Writing flow: argument graph -> project canvas/thread -> outline -> draft | Preserve | This is the concrete WRITE path from knowledge graph to text output. |
| Small project canvas, not whole-vault graph UI | Preserve | Canvas is useful for a project thread; whole-vault spatial graphs do not earn their complexity. |
| One-way outline derivation | Preserve | Canvas -> outline is enough for MVP. Two-way live sync is unbounded machinery until use proves otherwise. |
| Coherent-slice drafting | Preserve | Drafting should use selected claims/notes plus expanded source sections, not isolated top-k snippets. |
| Query mechanism | Preserve, gated on BM25 default activation | `passages`, FTS5, `sqlite-vec`, `concept_edges`, filter-before-rank queries, bounded topic surfacing, and RRF are beta.1 substrate; BM25 remains default until the pre-registered spike beats it. |
| Evidence-set model | Preserve | Drafted claims need single-span, multi-span, multi-hop, or implicit warrants. A citation string is not enough. |
| Numbers deterministic or incomplete | Preserve | Model prose must not invent or silently transform numeric claims. |
| SRD as code-side output | Preserve | beta.1 produces requirements, acceptance criteria, constraints, risks, and open questions instead of executable code. |
| Code execution | Descope from beta.1 | Generated source, executable warrants, code-run ledgers, and sandboxed execution are post-beta.1. |
| Output verification labels | Preserve | Separate facts: citation exists, span exists, number provenance exists, stale, tainted, unchecked. No label means true. |
| Full export refusal gate | Preserve | Beta.1 export must refuse unresolved citations, structurally unsupported claims, stale/superseded evidence, and unresolved SRD gaps. |
| Deep-work vs task-work split | Preserve | Writing/SRD work needs a focused surface distinct from review queues and maintenance. |
| Feedback instrumentation | Preserve | Beta.1 is where the feedback loop begins; dispositions, incidents, time, cost, and fallback events must be recorded. |
| LLM call-site eval ledger and runtime modes | Preserve | Prompt, model, retrieval context, tool, schema, sampling, and provider profile are product variables; beta.1 must test cheaply in `test` mode but empirically compare production candidates to a registered baseline before promotion. |
| Gate calibration | Preserve | Review gates are not assumed to improve correctness; beta.1 must measure whether they do. |
| Real-corpus onboarding | Preserve | Generated demo corpora cannot validate the product. Beta.1 needs a real citable seed corpus and first WRITE gesture. |

## Open Before Beta.1 Freeze

These are beta.1 freeze blockers, not optional scope:

1. SRD contract and verification checks.
2. Evidence-set review UI for multi-hop/implicit warrants and SRD gaps.
3. Actual seed-corpus source list and fetch/bundle rule.
4. Workspace / gate topology for asking, review, return-to-work, and SRD
   actions.
5. First export target.
6. Multi-device write/read topology.
7. Raw dataset bundling rule.
8. Template-driven concept creation, including `note mode: work`.
9. Schema/docs drift prevention.

Measured during beta.1, but not design blockers: PI review load, gate
calibration, payoff vs. null workflow, canvas threshold, draft write-back
residuals, exploration acted-on rate, retrieval default activation, provider
conflicts, and topic/facet ownership.

Before implementation starts, complete only retrieval fixture preregistration,
LLM eval harness preregistration, and seed corpus source/license verification
named in §6. Product measurements run after the relevant mechanism exists.

## Beta.1 Acceptance Additions

Beta.1 should not close unless these output-specific checks pass:

1. A real project produces a draft from claims/evidence and either exports or
   refuses with explicit reasons.
2. Every drafted claim has an evidence-set state.
3. Every printed number is computed or marked unresolved.
4. The query mechanism passes its fixture: `passages` stay fresh, checked-state
   filtering is correct, targeted structural queries filter before ranking,
   topic surfacing reports bounded denominators, and BM25 remains default unless
   the registered spike beats it.
5. Export refuses unresolved citations, structurally unsupported claims,
   stale/superseded evidence, and unresolved SRD gaps.
6. An SRD is created or curated with traced requirements, non-requirements,
   acceptance criteria, constraints, risks, and open questions.
7. SRD verification checks traceability, acceptance criteria, and scope.
8. Production-enabled LLM call sites have frozen eval suites, production/test
   runtime modes, local-or-cheap test profiles, and recorded promotion evidence
   for their active prompt/model/config variables.
9. The user can inspect draft, canvas, SRD, evidence, gaps, and bibliography as
   plain files if Memoria is removed.
9. Feedback data exists for accept/reject dispositions, incidents, time, cost,
   fallback to null workflow, and gate calibration.

## Boundary

Preserve beta.1's output mechanics, not all beta.1 implementation choices. In
particular, do not treat "resident daemon while Obsidian is open" as a required
beta.1 truth. The required truth is narrower: one engine, one writer, process
isolation, write-event consistency, and a UI that does not become a second
authority.
