# Memoria 0.1.0-beta.1 — MVP Requirements, Evidence & Technical Decisions

Date: 2026-07-05

Status: **governing.** This document is the MVP requirements, evidence
constraints, normative corrections, technical preservation decisions,
acceptance checks, open research questions, and exit verdict for beta.1. The
detailed system design that **implements** it is
[0.1.0-beta.1-design.md](./0.1.0-beta.1-design.md); this document names *what
must be true*, the design names *how*. beta.1 inherits alpha.16 — see
[../0.1.0-alpha.16/0.1.0-alpha.16-requirements.md](../0.1.0-alpha.16/0.1.0-alpha.16-requirements.md).
(Consolidates the former requirements/evidence companion and the
technical-preservation recommendations; originals archived under
`resources/superseded/`.)

## 1. Role Of Beta.1

Beta.1 is the first version where Memoria delivers the stated promise for a
real user on a real corpus:

- read: accumulate, compile, curate, inspect, and retrieve knowledge;
- write: compose/create new text and code artifacts from that knowledge.

Beta.1 inherits alpha.16's READ cycle and knowledge-production substrate. It
adds the WRITE/output half: text drafting, code artifacts, output verification,
usable onboarding, and feedback instrumentation.

Beta.1 is not "feature complete forever." It is complete enough that a real
researcher can use it as the primary workflow and the product can improve from
observed feedback.

## 2. MVP Loop

Beta.1 must run this loop end to end:

1. Open a project with a question, thesis, or output goal.
2. Seed it from existing sources and notes.
3. Run gap analysis over both the catalog graph and knowledge graph.
4. Source or capture new material to fill gaps.
5. Extract, digest, annotate, and process into human-owned notes/claims.
6. Re-run gap analysis and refine the project.
7. Build a project thread from claims/evidence into a canvas or equivalent map.
8. Derive an outline from that thread.
9. Draft text from coherent evidence slices, not isolated top-k fragments.
10. Create or curate code artifacts as executable warrants or deliverables.
11. Verify citations, numbers, evidence sets, code execution, provenance, and
    graph structure.
12. Export or refuse export with explicit unresolved gaps.
13. Record feedback, incidents, cost, accept/reject dispositions, and fallback to
    the null workflow.

Beta.1 succeeds only if the loop is repeatable without returning to Zotero +
Obsidian + ad-hoc ChatGPT as the primary workflow.

## 3. Normative Corrections From The Redo Comparison

Adopt these corrections into the detailed beta.1 design.

1. Beta.1 should not re-specify alpha.16. It should inherit alpha.16 and specify
   only the WRITE/output additions.
2. Writing and coding are not peripheral. They are the output half of the
   mission and must be in beta.1.
3. The writing flow should be concrete: argument graph -> project canvas/thread
   -> outline -> draft -> verification/export.
4. Canvas is a small project-slice working surface, not a whole-vault graph UI
   and not the authority for the graph.
5. Outline generation is one-way and explicit: canvas/thread -> outline.
   Two-way live canvas/outline sync is out until real use proves it.
6. Drafting must use coherent slices: selected notes/claims plus expanded source
   sections. Top-k fragments alone are not enough for synthesis.
7. Every drafted claim must carry an evidence set, not just a citation string.
   Evidence sets can be single-span, multi-span, multi-hop, implicit, or computed.
8. Printed numbers must be computed by code, not by model prose.
9. Code has two MVP roles: executable warrant for computed claims, and project
   deliverable.
10. Code verification means running code and recording output. Model claims that
    code works are not verification.
11. Executing generated code is security-critical. beta.1 commits a **minimal
    sandbox** (no-network credential-free worker, project-scoped filesystem
    **+ secret read-denylist**, approved-commands-only, resource-capped — design
    §2.3), which makes code execution acceptance-eligible; hardening and
    adversarial validation before freeze remain open.
12. Review gates are not assumed to work. Beta.1 must measure whether reviewed
    outputs are actually better than unreviewed outputs.
13. Feedback should be event/disposition based, not a synthetic "quality" score.
14. Onboarding needs a real corpus and a fast grounded payoff. Generated demo
    corpora cannot validate the product.

## 4. Evidence Constraints

The alpha.16 evidence companion covers the storage, metadata, retrieval, NLI,
prompt-injection, and single-case evaluation facts. Beta.1 adds these derived
constraints:

| Evidence fact | Consequence |
| --- | --- |
| The mission explicitly includes both reading and writing, with code and text as main outputs. | Beta.1 must include text output and code output. A read-only beta does not satisfy the mission. |
| Obsidian/Markdown/JSON Canvas are inspectable file formats. Sources: [Obsidian storage docs](https://help.obsidian.md/Files+and+folders/How+Obsidian+stores+data), [JSON Canvas spec](https://jsoncanvas.org/spec/1.0/). | Drafts, canvases, source notes, evidence files, and code artifacts must remain usable without Memoria. |
| Zotero and DOI/Crossref/OpenAlex provide metadata/citation inputs with different reliability. Sources: [Zotero API](https://www.zotero.org/support/dev/web_api/v3/basics), [Crossref content negotiation](https://www.crossref.org/documentation/retrieve-metadata/content-negotiation/), [OpenAlex paper](https://arxiv.org/abs/2205.01833). | Citation export and output verification must resolve against deterministic metadata, preserve conflicts, and never let model prose author bibliography facts. |
| HANS/PAWS-style failures show that semantic checkers can be fooled by overlap and wording. Sources: [HANS](https://arxiv.org/abs/1902.01007), [PAWS](https://arxiv.org/abs/1904.01130). | Verification can check structure, spans, links, arithmetic, and suspicious contradictions. It cannot certify truth. |
| Indirect prompt injection can use retrieved untrusted content to steer tool-using systems. Source: [Greshake et al. 2023](https://arxiv.org/abs/2302.12173). | Draft/code generation must not give untrusted source text authority over writes, network, shell, export target, or secrets. |
| Single-user/self-use evaluation requires baseline and repeated measurement before benefit claims. Sources: [Neustaedter and Sengers 2012](https://antle.iat.sfu.ca/wp-content/uploads/NeustaedterSengers_2012DIS_AutobioDesignMethodology.pdf), [WWC single-case design guide](https://ies.ed.gov/ncee/wwc/Docs/ReferenceResources/wwc_scd.pdf). | Beta.1 must instrument real use against a null workflow and must allow adopt/redesign/kill outcomes. |
| Pre-release software can require backup/forward-migration discipline. Sources: [SemVer](https://semver.org/), [Zotero beta builds](https://www.zotero.org/support/beta_builds). | Backup, restore, rebuild, and migration drills are beta prerequisites, not polish. |

## 5. Required Beta.1 Capabilities

### Writing

The writing module must:

- consume alpha.16 claims, source-notes, hubs, evidence pins, and gap reports;
- propose a project thread/canvas from selected claims and evidence;
- derive an outline from that thread;
- draft section-by-section from coherent evidence slices;
- mark machine-authored text until accepted or rewritten;
- preserve human ownership of judgment and final prose;
- refuse export on unresolved citations, unsupported claims, stale/superseded
  evidence, evidence-incomplete passages, or unresolved accepted gaps.

### Coding

The coding module must:

- create project-scoped code artifacts as plain files;
- link code artifacts to project claims, source rationale, dependencies, and run
  instructions;
- treat code as either executable warrant or deliverable;
- run only approved commands inside the chosen sandbox;
- record command, environment assumptions, output, exit status, and unresolved
  risk;
- allow code artifacts to be exported with README/provenance.

### Verification

Verification must separate facts:

- citation exists;
- quoted span exists;
- bibliography entry resolves;
- graph link/anchor resolves;
- number recomputes;
- code runs and output matches the recorded expectation;
- content is machine-authored, human-authored, unchecked, stale, tainted, or
  structurally verified.

No label means "true." Semantic judgment remains the user's responsibility.

### Feedback

Beta.1 must record:

- accept/reject dispositions on recommendations;
- incidents: wrong citation, wrong answer, data loss, cost surprise, prompt
  injection hit, unsupported draft claim, failed code run;
- time/cost for capture, reading, notes, ask, compose, code, export;
- fallback to null workflow;
- whether reviewed outputs outperform unreviewed/raw outputs;
- whether the tool produced a text/code artifact worth keeping.

## 6. Open Research Questions Before Freezing Beta.1

These questions are not excuses to remove MVP scope. They are blockers to
specific mechanisms.

1. **Code sandbox — hardening (minimal committed).** The minimal sandbox —
   no-network + **secret read-denylist** + project-write-scope + resource caps —
   is committed (design §2.3); OPEN is hardening beyond it and adversarial
   validation before it is trusted for untrusted-influenced code.
2. **Evidence-set model — validation (minimal schema committed).** The durable
   schema is committed (design §1.4: items, engine-derived type, complete/
   incomplete state, inline marker + DB row); OPEN is the empirical PI-routing
   fraction and the review-UI for multi-hop/implicit sets.
3. **PI review load:** how many routed verification items can one user sustain
   per session before review becomes theater?
4. **Gate calibration:** do accepted/reviewed outputs actually become more
   correct than raw outputs?
5. **Retrieval stack:** on the user's real corpus, does FTS5+vector beat FTS5
   alone on answer quality, citation precision, latency, and cost?
6. **Provider conflicts:** what field-level conflict rates appear among Zotero,
   Crossref, OpenAlex, Semantic Scholar, DOI metadata, and local PDFs?
7. **Topic/facet ownership:** should topics be human-owned with machine
   candidates only, or are provider topics accurate enough for gap discovery?
8. **Canvas threshold:** when does a canvas stop beating a table/faceted view for
   a project slice?
9. **Draft-to-knowledge write-back — designed (design §1.5):** extract-to-atomic-
   note (PI-gated), transclusion not copy, source-link + `seedling` marker,
   squeeze-point trigger. OPEN residual: the squeeze-point threshold and whether
   `seedling` notes need a distinct review lane.
10. **Export target:** which output format matters first: Markdown, Pandoc,
    LaTeX, DOCX, Google Docs, or repository README/package?
11. **Multi-device:** does real beta use need more than one write-capable device,
    or is one writer plus read-only secondary devices enough?
12. **Seed corpus:** which real, citable onboarding sources can be bundled or
    fetched without licensing problems?

### Beta.1 open-question resolutions (2026-07-06 addendum)

Fresh, independently-verified research (design.md §11) updates three of the
above; full evidence lives in `resources/0.1.0-beta.1-*-research.md`:

- **Q1 (sandbox)** stays open, but is no longer unstructured: three concrete,
  evidenced hardening rules now exist (allowlist-by-canonical-path over
  denylist; the `--` argument delimiter for any bwrap-wrapping code; track
  `sandbox-exec`'s Apple-deprecated status as a real macOS risk) and
  adversarial validation is now specified as adaptive, not a static
  attack-list pass.
- **Q5 (retrieval stack)** is no longer answerable by more research — the
  literature contains no verified direct comparison of hybrid dense+sparse
  fusion vs. BM25-only. A runnable spike protocol (pinned baseline, treatment,
  metrics, concrete adoption bar) now exists at
  `resources/0.1.0-beta.1-retrieval-fusion-spike-protocol.md`, replacing "do
  more research" with "run this spike."
- **Q12 (seed corpus)** stays open on the core ask (no specific ~8 sources
  named yet), but the licensing *mechanics* are now settled: PMC's
  Commercial-Use-Allowed tier only (its own `cc0[filter]` mixes in unrelated
  public-domain content); arXiv papers only when the author selected CC
  BY/CC0 specifically (arXiv hosting alone is never sufficient permission).
  DOAJ and Semantic Scholar remain fully unresearched — a follow-up pass is
  still needed to name actual candidates.

## 7. Beta.1 Acceptance Checks

Beta.1 passes when:

1. A real project completes the source -> note -> gap -> source -> draft loop at
   least twice.
2. Zotero, bibliography, DOI/PDF, GitHub, and local-file capture all work or
   fail into visible triage states.
3. The project graph contains works, source-notes, claims/questions/tensions,
   hubs/facets, evidence pins, gaps, draft sections, and code artifacts.
4. Ask answers with cited spans, denominators, and honest absence language.
5. Gap analysis reports source gaps, argument gaps, and graph mismatch gaps.
6. A text draft is produced, audited, corrected, and exported or refused for
   explicit reasons.
7. A code artifact is produced or curated, linked to a project claim/rationale,
   checked by execution, and exported with provenance.
8. Export refuses unresolved citations, unsupported claims, stale/superseded
   evidence, and unresolved code risks.
9. A machine run can be reverted without altering human text.
10. Backup, restore, rebuild, and migration pass in a clean profile.
11. Feedback instrumentation is producing disposition, incident, cost, time,
    gate-calibration, and null-workflow comparison data.
12. If Memoria is removed, the user can still read the corpus and project state
    as Markdown, PDFs/source files, code files, `.canvas`, and `bibliography.bib`.

## 8. Exit Verdict

Beta.1 exits with one of three verdicts:

- **adopt:** the loop beats the null workflow enough to keep using;
- **targeted redesign:** a named mechanism failed and has a bounded fix;
- **kill:** the null workflow won.

The verdict must be based on recorded events and artifacts, not optimism about
future versions.

## Technical preservation decisions

Which mechanisms are output/MVP-specific (preserved for beta.1) versus inherited
from alpha.16's substrate — the technical layer of the requirements above. The
inherited substrate is the "adopt now" set in
[../0.1.0-alpha.16/0.1.0-alpha.16-requirements.md](../0.1.0-alpha.16/0.1.0-alpha.16-requirements.md).

## Decision Rule

Beta.1 should not re-specify alpha.16's substrate. It should inherit alpha.16
and add only what is required for the first usable read/write MVP: text output,
code output, output verification, onboarding, and feedback instrumentation.

## Inherited From Alpha.16

Beta.1 should assume these are already present:

- `run_id` for machine writes, cost, review, incidents, and revert;
- single `VaultWriter`;
- block-level human text protection;
- supervisor/worker split;
- Guard-brokered egress;
- Ledger cost authority;
- derived vs operational DB split;
- content-addressed provider payload store;
- Zotero-fidelity import;
- anchored fulltext Ask with denominators;
- VaultWriter-owned exfiltration linter;
- minimal backup/restore/rebuild/migration substrate.

If any of these are missing, beta.1 should not rebuild them differently. They
should be backfilled into alpha.16's substrate first.

## Preserve For Beta.1

| Mechanism | Recommendation | Reason |
| --- | --- | --- |
| Writing flow: argument graph -> project canvas/thread -> outline -> draft | Preserve | This is the concrete WRITE path from knowledge graph to text output. |
| Small project canvas, not whole-vault graph UI | Preserve | Canvas is useful for a project thread; whole-vault spatial graphs do not earn their complexity. |
| One-way outline derivation | Preserve | Canvas -> outline is enough for MVP. Two-way live sync is unbounded machinery until use proves otherwise. |
| Coherent-slice drafting | Preserve | Drafting should use selected claims/notes plus expanded source sections, not isolated top-k snippets. |
| Evidence-set model | Preserve | Drafted claims need single-span, multi-span, multi-hop, implicit, or computed warrants. A citation string is not enough. |
| Numbers computed in code | Preserve | Model prose must not invent or silently transform numeric claims. |
| Code as executable warrant | Preserve | Computed claims often have no single source span; re-runnable code can be the warrant. |
| Code as deliverable | Preserve | Code is part of the mission's output half, not an inert attachment. |
| Code execution verification | Preserve, gated on sandbox | Code is verified by running it and recording output, not by trusting generated prose. |
| Code execution sandbox | **Minimal committed (design §2.3)**; harden + validate before freeze | The minimal no-network, project-scoped **(with a secret read-denylist)**, approved-commands sandbox closes the network *and* secret-read exfiltration paths and makes code execution acceptance-eligible; finer isolation and adversarial validation are the remaining beta-critical work. |
| Output verification labels | Preserve | Separate facts: citation exists, span exists, number recomputes, code ran, stale, tainted, unchecked. No label means true. |
| Full export refusal gate | Preserve | Beta.1 export must refuse unresolved citations, unsupported claims, stale/superseded evidence, and unresolved code risks. |
| Deep-work vs task-work split | Preserve | Writing/coding need a focused surface distinct from review queues and maintenance. |
| Feedback instrumentation | Preserve | Beta.1 is where the feedback loop begins; dispositions, incidents, time, cost, and fallback events must be recorded. |
| Gate calibration | Preserve | Review gates are not assumed to improve correctness; beta.1 must measure whether they do. |
| Real-corpus onboarding | Preserve | Generated demo corpora cannot validate the product. Beta.1 needs a real citable seed corpus and first WRITE gesture. |

## Open Before Beta.1 Freeze

These are beta.1 open research questions, not optional scope:

1. **Sandbox hardening** — the minimal sandbox (design §2.3) is committed; what
   hardening (finer isolation, per-language runtimes) and adversarial validation
   are needed before relying on it for untrusted-influenced code?
2. **Evidence-set UI/validation** — the durable schema (design §1.4) is
   committed; what is the review-UI for multi-hop/implicit sets, and the measured
   PI-routing fraction?
3. How many verification items can one user sustain before review becomes
   theater?
4. Does PI-reviewed output measurably beat raw engine output?
5. At what project-slice size does canvas stop beating a table/faceted view?
6. How do accepted draft insights become permanent notes without an automatic
   write-time oracle?
7. Which export target matters first: Markdown, Pandoc, LaTeX, DOCX, Google
   Docs, or repository/package output?
8. Which real citable seed corpus can be bundled or fetched without licensing
   problems?

## Beta.1 Acceptance Additions

Beta.1 should not close unless these output-specific checks pass:

1. A real project produces a draft from claims/evidence and either exports or
   refuses with explicit reasons.
2. Every drafted claim has an evidence-set state.
3. Every printed number is computed or marked unresolved.
4. Export refuses unresolved citations, unsupported claims, stale/superseded
   evidence, and unresolved code risks.
5. A code artifact is created or curated as either executable warrant or
   deliverable.
6. Code verification runs in the selected sandbox and records command, context,
   output, exit status, and risk.
7. The user can inspect draft, canvas, evidence, gaps, code, and bibliography as
   plain files if Memoria is removed.
8. Feedback data exists for accept/reject dispositions, incidents, time, cost,
   fallback to null workflow, and gate calibration.

## Boundary

Preserve beta.1's output mechanics, not all beta.1 implementation choices. In
particular, do not treat "resident daemon while Obsidian is open" as a required
beta.1 truth. The required truth is narrower: one engine, one writer, process
isolation, write-event consistency, and a UI that does not become a second
authority.
