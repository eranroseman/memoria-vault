# Memoria 0.1.0-alpha.16 — Requirements, Evidence & Technical Decisions

Date: 2026-07-05

Status: **governing.** This document is the requirement set, external-evidence
constraints, normative corrections, technical adopt/defer decisions, and
acceptance checks for alpha.16. The detailed system design that **implements**
it is [0.1.0-alpha.16-design.md](./0.1.0-alpha.16-design.md); this document
names *what must be true*, the design names *how*, and the design is checked
against the acceptance below. (Consolidates the former requirements/evidence
companion and the technical-adoption recommendations; their originals are
archived under `resources/superseded/`.)

## 1. Role Of Alpha.16

Memoria's stated mission is to help one user read and write knowledge, with
code and text as the main outputs. For alpha.16, the implementable slice is the
READ cycle plus the knowledge-production substrate that beta.1 will use for
writing and code output.

Alpha.16 must prove that a real project can move through:

1. create a project around a question or thesis;
2. seed from Zotero, bibliography, DOI/arXiv/PMID, PDF, GitHub, or local file;
3. land immutable raw artifacts and bibliographic metadata without data loss;
4. extract readable full text with distrust flags;
5. create an opt-in machine digest with source anchors;
6. process annotations into human-owned notes, claims, questions, or tensions;
7. connect the source catalog to the human knowledge graph through source-notes;
8. run gap analysis over both the catalog graph and the note/claim graph;
9. answer corpus-grounded questions with cited spans and honest absence language;
10. preserve all keep-worthy material as readable files if Memoria disappears.

Alpha.16 is done only when this loop runs on one real project and machine output
is never presented as verified truth.

## 2. Normative Corrections From The Redo Comparison

Adopt these corrections into the detailed alpha.16 design.

1. `work.md` should not exist as the human/source bridge. Split it into:
   `record.md` for objective catalog projection, and `sources/<slug>.md` for
   the human source-note.
2. `record.md` is not the source of human interpretation. It carries stable
   source metadata, CSL-JSON/BibTeX-survival material, identifiers, source
   hashes, and import state.
3. `source-note` is the bridge between the objective catalog graph and the
   subjective knowledge graph. It is where the user's interpretation begins.
4. `digest.md` is opt-in, machine-authored, untrusted, and anchor-checked. It
   may assist reading, but it is not a human note and not citation truth.
5. The system has two graphs, not one: an objective relational catalog graph and
   a subjective markdown knowledge graph with human-owned argument edges.
6. Use stable `work_id` as identity. Citekeys are display/citation attributes
   because citation keys can change when metadata changes.
7. Retrieval should store and return anchored verbatim spans. Digests can enrich
   retrieval, but summaries must not replace source spans.
8. Gap analysis must compare three things: source coverage, argument coverage,
   and mismatch between the catalog graph and the knowledge graph.
9. Machine writes must use one staged writer with revert. Structural checks can
   verify links, spans, hashes, and references; they cannot verify truth.
10. Alpha.16 should defer full writing and coding modules to beta.1, but it must
    leave their substrate in place: claims, source-notes, hubs, projects,
    evidence pins, gap reports, and structural verification.

## 3. Evidence Constraints

| Evidence fact | Consequence |
| --- | --- |
| Obsidian stores notes as plain Markdown files in local vault folders, and nested vaults are discouraged. Source: [Obsidian storage docs](https://help.obsidian.md/Files+and+folders/How+Obsidian+stores+data). | Keep durable user artifacts as plain files. Do not make a plugin-only product. Do not hide the user's corpus inside an opaque app store. |
| JSON Canvas is an open JSON format with nodes and edges. Sources: [Obsidian Canvas docs](https://help.obsidian.md/Plugins/Canvas), [JSON Canvas spec](https://jsoncanvas.org/spec/1.0/). | Canvas may be a projection for a small project graph, not the authority for the graph. |
| Zotero exposes a local API and supports collection/item reads and export formats. Source: [Zotero API docs](https://www.zotero.org/support/dev/web_api/v3/basics). | Prefer Zotero for capture, PDFs, metadata, and annotation source-of-record behavior. |
| Zotero Connector is the documented reliable path for web/PDF capture; direct PDFs can become standalone attachments needing metadata. Source: [Zotero adding items](https://www.zotero.org/support/adding_items_to_zotero). | Land first, resolve later. Metadata failure is a visible repair state, not data loss. |
| Zotero PDF annotations can carry page links and citations, while built-in annotations live in Zotero's database unless exported. Source: [Zotero PDF reader docs](https://www.zotero.org/support/pdf_reader). | Import annotation quote/page/deep-link metadata; do not assume PDF files contain all annotation state. |
| Better BibTeX can auto-export BibTeX, and citation keys can change unless pinned/fixed. Sources: [BBT auto-export](https://retorque.re/zotero-better-bibtex/exporting/auto/), [BBT citation keys](https://retorque.re/zotero-better-bibtex/citing/). | `bibliography.bib` is a survival artifact; `work_id` is identity. |
| DOI/Crossref content negotiation can return CSL-JSON/BibTeX but mappings can be lossy. Source: [Crossref content negotiation](https://www.crossref.org/documentation/retrieve-metadata/content-negotiation/). | Preserve provider payloads and conflicts. Never ask an LLM to author bibliographic truth. |
| Crossref recommends filtering, caching, cursors, backoff, and responsible API use. Source: [Crossref REST API tips](https://www.crossref.org/documentation/retrieve-metadata/rest-api/tips-for-using-the-crossref-rest-api/). | Provider access needs cache, retry/backoff, rate limits, and reproducible request logs. |
| OpenAlex is an open scholarly graph with broad coverage but mixed metadata quality. Sources: [OpenAlex paper](https://arxiv.org/abs/2205.01833), [coverage study](https://arxiv.org/abs/2401.16359). | Use OpenAlex for enrichment/discovery candidates, not as sole metadata authority. |
| SQLite FTS5 supports local full-text search; sqlite-vec supports local vector search. Sources: [SQLite FTS5](https://www.sqlite.org/fts5.html), [sqlite-vec](https://alexgarcia.xyz/sqlite-vec/). | Ship local lexical search first; add vector retrieval only if it beats the baseline on the user's corpus. |
| HANS and PAWS show lexical-overlap shortcuts in NLI/paraphrase systems. Sources: [HANS](https://arxiv.org/abs/1902.01007), [PAWS](https://arxiv.org/abs/1904.01130). | NLI may propose flags; it must not certify truth. High-overlap/negation probes are required. |
| Indirect prompt injection lets untrusted retrieved content steer LLM-integrated tools. Source: [Greshake et al. 2023](https://arxiv.org/abs/2302.12173). | Source text is untrusted data. Models get no write, shell, network, or export authority. |
| Single-case/self-use evaluation needs baseline and repeated measurement to support claims. Sources: [Neustaedter and Sengers 2012](https://antle.iat.sfu.ca/wp-content/uploads/NeustaedterSengers_2012DIS_AutobioDesignMethodology.pdf), [WWC single-case design guide](https://ies.ed.gov/ncee/wwc/Docs/ReferenceResources/wwc_scd.pdf). | Record null-workflow baseline, incidents, repeated use, and fallback events before claiming improvement. |

## 4. Required File Contract

The companion endorses this minimal file contract:

```text
Memoria/
  works/<work_id>/
    record.md
    fulltext.md
    digest.md
    source.pdf
  sources/<slug>.md
  notes/<slug>.md
  hubs/<slug>.md
  projects/<slug>/
    project.md
    evidence.md
    gaps.md
  bibliography.bib
  system/
```

`record.md` owns objective source facts. `fulltext.md` owns extracted anchored
text. `digest.md` owns untrusted machine distillation. `sources/<slug>.md` owns
human interpretation of a source. `notes/<slug>.md` owns human claims/questions.
`hubs/<slug>.md` owns human topic salience plus machine candidates below a
boundary. Project files own project-specific question, evidence, and gap state.

SQLite owns derived indexes, graph queries, check rows, run logs, and provider
payload metadata. Derived SQLite state must rebuild from files plus provider
payload backups. Operational ledgers that are not rebuildable must be backed up
and surfaced honestly.

## 5. Alpha.16 Acceptance Checks

Alpha.16 passes when:

1. A real project can be created and seeded from Zotero plus at least one direct
   DOI/PDF/GitHub path.
2. Every source lands as an immutable artifact or visible triage stub.
3. One PDF becomes `source.pdf`, `record.md`, `fulltext.md`, and optional
   `digest.md` with resolvable anchors.
4. Five real annotations become human-owned notes or claims.
5. At least one source-note links a catalog work to the knowledge graph.
6. A hub or faceted topic view organizes real claims without machine-authoring
   the user's prose.
7. Gap analysis reports source gaps, argument gaps, and catalog/knowledge graph
   mismatches with denominators.
8. Ask returns anchored spans and says only what the searched corpus found.
9. A machine write can be reverted without altering human text.
10. The vault remains useful as Markdown plus PDFs plus `bibliography.bib` after
    disabling Memoria.

## 6. Deferred To Beta.1

Alpha.16 deliberately does not finish:

- the full writing module: canvas/thread, outline, draft, export gate;
- the full coding module: code artifacts as executable warrants/deliverables;
- code execution sandboxing;
- final workspace/gate topology;
- publication-grade exports;
- broad external discovery beyond bounded provider/citation lookup;
- multi-user or team workflows.

These are deferred only because beta.1 is the first full read/write MVP. They
are not optional to the product promise.

## Technical adopt/defer decisions

Which beta.1-draft mechanisms alpha.16 adopts now versus defers — the technical
layer of the requirements above, governing the design's mechanism choices.

## Decision Rule

Adopt a beta.1 technical mechanism in alpha.16 if it protects the READ cycle,
source ingest, machine writes, Ask, provenance, cost, security, or future
migration. Defer it if it only becomes necessary for full text/code output.

## Adopt In Alpha.16

| Mechanism | Recommendation | Reason |
| --- | --- | --- |
| `run_id` as the unit for writes, cost, incidents, review, and revert | Adopt now | Provenance and reversibility need this from the first machine write. Retrofitting run identity later would make early writes second-class. |
| Single `VaultWriter` | Adopt now | Digests, reports, gaps, fulltext, and exports all need one machine-write path that protects human text. |
| Block-level human protection | Adopt now | Mixed human/machine files are unavoidable. Human spans must be byte-immutable to machine runs. |
| Supervisor/worker split | Adopt now, without resident-daemon commitment | Keep the safety split: supervisor owns DB/key/Guard/Ledger/VaultWriter; per-run worker is credential-free and socketless. Residency remains a separate host decision. |
| Guard-brokered egress | Adopt now | Alpha.16 already touches Zotero, DOI/Crossref, OpenAlex, GitHub, and LLM providers. Workers should not own sockets or keys. |
| Ledger as cost authority | Adopt minimal form now | Paid digests or synthesis need price table, token ceilings, run attribution, and circuit-breaker behavior before real use. |
| Derived vs operational DB split | Adopt now | Derived state can rebuild; operational ledgers cannot. The design must say which is which before data exists. |
| Content-addressed provider payload store | Adopt now | Metadata/enrichment facts must be replayable and auditable after providers mutate records. |
| Zotero-fidelity import | Adopt now | Alpha.16's READ cycle depends on real source capture, metadata, PDFs, and annotations. |
| Fulltext-first Ask with denominators | Adopt now | Ask must work over anchored source text before digests are trusted. Absence language must be searched-set language. |
| VaultWriter-owned exfiltration linter | Adopt now | Untrusted PDFs, metadata, digests, and report notes enter the vault in alpha.16. Neutralization belongs at the write boundary. |
| Minimal version/migration/backup runner | Adopt now | Version the vault contract, support backup/rebuild, and prove clean-profile restore before beta. Full migration drills can mature by beta. |
| Export substrate as a run | Adopt minimal form now | A minimal `mem export` can exist if it is revertible and structurally checked. Full output gating belongs to beta.1. |

## Do Not Adopt In Alpha.16

| Mechanism | Defer To | Reason |
| --- | --- | --- |
| Resident daemon while Obsidian is open | Open host decision | Preserve process split and write-event ideas, but residency is not proven necessary for alpha.16. |
| Full writing module: argument graph -> canvas -> outline -> draft | beta.1 | This is the WRITE/output half of the MVP. Alpha.16 must only supply the substrate. |
| Full coding module and code lane | beta.1 | Code as executable warrant/deliverable is output scope. |
| Code execution sandbox | beta.1 | Required only when generated/project code is executed. |
| Full output export gates | beta.1 | Alpha.16 can do minimal structural export; beta.1 owns draft/code readiness. |
| Full onboarding / time-to-first-answer product flow | beta.1 | Alpha.16 should log use and incidents; beta.1 owns MVP usability. |
| Broad discovery or open web research | Later unless required | Alpha.16 only needs bounded provider/citation discovery. |

## Alignment With Current Designs

This recommendation matches the alpha.16 companion: alpha.16 is the READ cycle
plus knowledge-production substrate, while full writing/coding are deferred to
beta.1. It also matches the detailed alpha.16 design's adoption of the
supervisor/worker split, Guard, Ledger, VaultWriter, provider payloads, anchored
fulltext Ask, and denominator-first gap analysis.

One wording detail to preserve: "export" appears in alpha.16 only as a minimal
structural output. The beta.1-level export gate is different: it verifies
draft/code output readiness and should not be pulled into alpha.16.

## Alpha.16 Acceptance Additions

Alpha.16 should not close unless these technical substrate checks pass:

1. Every machine write has a `run_id`.
2. Every machine file change goes through `VaultWriter`.
3. A run can be reverted without changing human text.
4. Guard logs allowed and denied egress; workers have no direct provider keys.
5. Ledger records cost-bearing calls and blocks over-budget work.
6. Provider payloads are stored content-addressed and referenced by enrichment.
7. `mem rebuild` restores derived search/check state or fails closed.
8. Ask cites anchored fulltext spans and prints coverage denominators.
9. Export, if present, is a run and can be reverted.
10. A clean-profile backup/restore/rebuild smoke test passes.
