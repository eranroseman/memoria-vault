# Memoria 0.1.0-alpha.16 Technical Adoption Recommendations

Date: 2026-07-05

Status: companion recommendation. This document records which technically
superior parts of the beta.1 implementation design should be adopted in
alpha.16. It does not replace
[0.1.0-alpha.16-design-claude.md](./0.1.0-alpha.16-design-claude.md) or
[0.1.0-alpha.16-design-codex.md](./0.1.0-alpha.16-design-codex.md).

## Decision Rule

Adopt a beta.1 technical mechanism in alpha.16 if it protects the READ cycle,
source ingest, machine writes, Ask, provenance, cost, security, or future
migration. Defer it if it only becomes necessary for full text/code output.

## Adopt In Alpha.16

| Mechanism | Recommendation | Reason |
| --- | --- | --- |
| `run_id` as the unit for writes, cost, incidents, review, and revert | Adopt now | Provenance and reversibility need this from the first machine write. Retrofitting run identity later would make early writes second-class. |
| Single `VaultWriter` | Adopt now | Digests, reports, gaps, fulltext, and exports all need one machine-write path that protects human text. |
| Block-level human protection | Adopt now | Mixed human/machine files are unavoidable. Human spans must be byte-immutable to machine runs. |
| Supervisor/worker split | Adopt now, without resident-daemon commitment | Keep the safety split: supervisor owns DB/key/Guard/Ledger/VaultWriter; per-run worker is credential-free and socketless. Residency remains a separate host decision. |
| Guard-brokered egress | Adopt now | Alpha.16 already touches Zotero, DOI/Crossref, OpenAlex, GitHub, and LLM providers. Workers should not own sockets or keys. |
| Ledger as cost authority | Adopt minimal form now | Paid digests or synthesis need price table, token ceilings, run attribution, and circuit-breaker behavior before real use. |
| Derived vs operational DB split | Adopt now | Derived state can rebuild; operational ledgers cannot. The design must say which is which before data exists. |
| Content-addressed provider payload store | Adopt now | Metadata/enrichment facts must be replayable and auditable after providers mutate records. |
| Zotero-fidelity import | Adopt now | Alpha.16's READ cycle depends on real source capture, metadata, PDFs, and annotations. |
| Fulltext-first Ask with denominators | Adopt now | Ask must work over anchored source text before digests are trusted. Absence language must be searched-set language. |
| VaultWriter-owned exfiltration linter | Adopt now | Untrusted PDFs, metadata, digests, and report notes enter the vault in alpha.16. Neutralization belongs at the write boundary. |
| Minimal version/migration/backup runner | Adopt now | Version the vault contract, support backup/rebuild, and prove clean-profile restore before beta. Full migration drills can mature by beta. |
| Export substrate as a run | Adopt minimal form now | A minimal `mem export` can exist if it is revertible and structurally checked. Full output gating belongs to beta.1. |

## Do Not Adopt In Alpha.16

| Mechanism | Defer To | Reason |
| --- | --- | --- |
| Resident daemon while Obsidian is open | Open host decision | Preserve process split and write-event ideas, but residency is not proven necessary for alpha.16. |
| Full writing module: argument graph -> canvas -> outline -> draft | beta.1 | This is the WRITE/output half of the MVP. Alpha.16 must only supply the substrate. |
| Full coding module and code lane | beta.1 | Code as executable warrant/deliverable is output scope. |
| Code execution sandbox | beta.1 | Required only when generated/project code is executed. |
| Full output export gates | beta.1 | Alpha.16 can do minimal structural export; beta.1 owns draft/code readiness. |
| Full onboarding / time-to-first-answer product flow | beta.1 | Alpha.16 should log use and incidents; beta.1 owns MVP usability. |
| Broad discovery or open web research | Later unless required | Alpha.16 only needs bounded provider/citation discovery. |

## Alignment With Current Designs

This recommendation matches the alpha.16 companion: alpha.16 is the READ cycle
plus knowledge-production substrate, while full writing/coding are deferred to
beta.1. It also matches the detailed alpha.16 design's adoption of the
supervisor/worker split, Guard, Ledger, VaultWriter, provider payloads, anchored
fulltext Ask, and denominator-first gap analysis.

One wording detail to preserve: "export" appears in alpha.16 only as a minimal
structural output. The beta.1-level export gate is different: it verifies
draft/code output readiness and should not be pulled into alpha.16.

## Alpha.16 Acceptance Additions

Alpha.16 should not close unless these technical substrate checks pass:

1. Every machine write has a `run_id`.
2. Every machine file change goes through `VaultWriter`.
3. A run can be reverted without changing human text.
4. Guard logs allowed and denied egress; workers have no direct provider keys.
5. Ledger records cost-bearing calls and blocks over-budget work.
6. Provider payloads are stored content-addressed and referenced by enrichment.
7. `mem rebuild` restores derived search/check state or fails closed.
8. Ask cites anchored fulltext spans and prints coverage denominators.
9. Export, if present, is a run and can be reverted.
10. A clean-profile backup/restore/rebuild smoke test passes.
