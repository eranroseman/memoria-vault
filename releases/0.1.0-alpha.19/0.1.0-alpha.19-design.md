# Memoria - 0.1.0-alpha.19 Design

Date: 2026-07-07

Status: implementation design for the next alpha slice. alpha.19 is a delta on
the current `main` implementation and a pull-forward from
[beta.1](../0.1.0-beta.1/0.1.0-beta.1-design.md), not a second beta design.

## 0. Purpose

alpha.19 should make the two largest beta.1 implementation gaps real enough to
test:

1. the **dense/vector retrieval substrate** - `fulltexts/`, `passages`, FTS5,
   `sqlite-vec`, `concept_edges`, lazy reindex, same-transaction status cascade,
   and fixture-gated hybrid retrieval;
2. the **full code-output lane** - project code artifacts, sandboxed execution,
   code-run records, rerun verification, and computed evidence.

The target is not to finish beta.1. alpha.19 deliberately skips the beta.1 UI,
seed-corpus, multi-device, raw-dataset, and release-freeze questions. It creates
the substrate those later decisions depend on.

## 1. Requirements

alpha.19 is done only if these hold:

- A fresh vault uses `fulltexts/`, not `fulltext/`, everywhere. Old spelling may
  be tolerated only as an import/read compatibility alias during this alpha, not
  as the emitted shape.
- Queries run over a unified local store: SQLite rows, FTS5 full text,
  `sqlite-vec` vectors, and structural graph predicates.
- Dense/vector retrieval exists as a substrate, but **BM25 remains the default
  answer path** until the preregistered fixture beats it.
- Query freshness does not require a daemon. Reads perform mtime-gated lazy
  reindex, and verdict/status changes cascade to indexed passages in the same
  SQLite transaction that changes the verdict.
- Code is a Memoria output artifact, not a chat transcript. It is a plain
  project companion file with a typed markdown record, provenance, approved run
  command, execution result, and export path.
- Code execution is enabled only on a platform with a passing sandbox proof.
  Unsupported platforms fail closed.
- Computed evidence is first-class. A computed claim is complete only when its
  code artifact and output hash resolve and recompute.
- All new tables and file references have one rebuilding rule, one verification
  rule, and one crash-recovery state. No second source of truth.

## 2. Evidence

- SQLite FTS5 is an SQLite virtual table module for full-text search, supports
  `MATCH`, relevance ordering, and external-content/contentless table patterns;
  it may be compiled in or loadable, so alpha.19 must use a startup capability
  check rather than assume availability:
  [SQLite FTS5](https://www.sqlite.org/fts5.html).
- `sqlite-vec` provides a local SQLite vector-search extension with `vec0`
  virtual tables and pure SQL `CREATE`, `INSERT`, and `SELECT`; it needs no
  server:
  [sqlite-vec](https://alexgarcia.xyz/sqlite-vec/).
- Bubblewrap can create a new network namespace with only loopback visible, but
  its README also warns that the protection level is determined by the exact
  arguments and mounts supplied. That makes a proof test mandatory, not optional:
  [containers/bubblewrap](https://github.com/containers/bubblewrap).
- Python `subprocess` does not invoke a shell by default, and its docs warn that
  explicit `shell=True` shifts shell-injection responsibility to the caller.
  alpha.19 code runs therefore use argv arrays and forbid shell strings:
  [Python subprocess security considerations](https://docs.python.org/3/library/subprocess.html#security-considerations).

Current implementation facts, used only for gap sizing:

- Retrieval is BM25-only and the vector replacement bar is explicitly not
  cleared in
  [`retrieval_substrate.py`](../../../main/src/memoria_vault/runtime/retrieval_substrate.py).
- The schema has `evidence_sets`, but evidence types are only `single-span`,
  `multi-span`, `multi-hop`, and `implicit`; no `computed` type exists in
  [`schema.sql`](../../../main/src/memoria_vault/runtime/schema.sql) or
  [`evidence.py`](../../../main/src/memoria_vault/runtime/evidence.py).
- The code lane is not implemented as a product lane. Direct `execute_code`
  tool access is hard-denied in
  [`hook.py`](../../../main/src/memoria_vault/runtime/policy/hook.py), which is
  correct policy containment but not code-output functionality.
- The emitted bundle spelling is still `fulltext/` across the template, docs,
  tests, and runtime path checks.

## 3. Target Design

### 3.1 Bundle naming

`fulltexts/` is the emitted root. `fulltext` remains the document `type` because
the type names the artifact kind, not the directory plurality. The plural change
is path-level only:

```text
digests/<work_id>.md
fulltexts/<work_id>.md
projects/<slug>/...
```

The installer, template schema, trusted-writer scopes, search index, policy
rules, docs, fixtures, and tests all move together. An alpha.19 compatibility
reader may accept `fulltext/` in existing throwaway vaults, but no alpha.19
writer emits it.

### 3.2 Query/data substrate

Add the minimum schema that lets all retrieval axes join in one SQLite query:

- `passages`: the canonical retrievable unit. Columns include `passage_id`,
  `origin`, `concept_id`, `work_id`, `path`, `anchor`, `page`, byte offsets,
  `text_sha256`, `text`, `check_status`, `mode`, `question_status`,
  `source_mtime_ns`, and `indexed_at`.
- `passage_fts`: FTS5 index over `passages.text`, keyed by `passage_id`.
- `passage_vec`: `sqlite-vec` `vec0` table keyed by `passage_id`; embeddings are
  derived cache and may be deleted/rebuilt.
- `file_index_state`: one row per indexed file path with mtime/hash/status so a
  query can refresh stale files without a daemon.
- `concept_edges`: checked/PI-confirmed concept-to-concept edges:
  `supports`, `contradicts`, `extends`, and `tension`. `work_graph_edges`
  remains bibliographic.
- `retrieval_fixture_runs`: local fixture evidence for lexical, vector, hybrid,
  and graph retrieval variants.

Invariants:

- `passages` is derived from vault markdown and catalog mirrors. Rebuild may
  delete and repopulate it.
- `passages.check_status` is copied for query speed. Every status-changing write
  path updates matching passage rows in the same transaction.
- Query execution performs structural filters before ranking.
- RRF fuses lexical, vector, and graph candidate lists because raw BM25 and
  vector distances are not comparable scores.
- Topic surfacing returns denominators: "shown N of M checked passages within
  bound B," not only top-k.
- `sqlite-vec` absence fails the dense substrate capability check; it does not
  break the BM25 fallback path.

### 3.3 Retrieval activation

alpha.19 builds the dense/vector substrate, not a new default claim. The default
answer path remains BM25 unless the committed fixture shows a better result.

The fixture records:

- corpus hash and fixture inputs;
- query set and expected passage/work IDs;
- metrics: recall@k, MRR, citation-support precision, latency, and empty-result
  honesty;
- compared variants: BM25, FTS5, vector-only, FTS5+vector RRF, and
  FTS5+vector+graph RRF;
- promotion rule: the candidate must beat the active baseline and regress no
  blocker metric.

### 3.4 Code-output lane

Code artifacts live inside the project as companion artifacts referenced from a
typed markdown record:

```text
projects/<slug>/
  code/
    <artifact-id>.md          # type: code-artifact, OKF-readable record
    <artifact-id>/
      src/...                 # companion code files
      outputs/...             # generated outputs, hash-recorded
```

Add DB rows:

- `code_artifacts`: artifact ID, project path, record path, source directory,
  declared purpose (`warrant` / `deliverable` / both), approved command argv,
  declared inputs, declared outputs, dependency notes, and current status.
- `code_runs`: artifact ID, run ID, command argv, cwd, sanitized environment
  names, input hashes, output hashes, stdout/stderr hashes, exit status,
  timeout/resource result, sandbox backend/profile hash, started/ended times,
  and state.

Execution contract:

- The command is an argv list, never a shell string.
- The runner accepts only approved commands declared on the artifact record.
- The runner supplies an explicit minimal environment and never passes provider
  keys or local secret paths.
- The sandbox mounts the project code directory and declared inputs read-only
  except for a dedicated output directory.
- The Linux/WSL backend is `bwrap` with network namespace isolation, no home
  mount, canonical path allowlist, secret read denylist, timeout, max output
  bytes, and resource caps. If the proof fails, execution is unavailable.
- Every run records enough data to rerun or explain why rerun is impossible.

### 3.5 Computed evidence and schema references

Evidence items become a typed union:

- `source-span`: `work_id#^pNNNN` resolved through `passages.anchor` metadata;
- `evidence-set`: nested `ev-<8hex>`;
- `code-warrant`: `code_run_id`, `artifact_id`, expected output path or stdout
  selector, and `output_sha256`.

`computed` is a derived evidence-set type, not a user-supplied label. A set is
computed when its warrant is code-only or when the claim's load-bearing support
is a code-warrant item. Its state is:

- `complete` when the referenced artifact exists, the code run succeeded, the
  output hash matches, and rerun verification passes under the same declared
  inputs/profile;
- `evidence-incomplete` when the artifact/run/output is missing, failed,
  stale, unverifiable, or sandbox-unavailable.

No claim becomes true because code ran. Code execution verifies the structural
fact "this output was produced by this command from these inputs."

## 4. Acceptance

alpha.19 acceptance is one runnable implementation checkpoint:

1. `fulltexts/` is the emitted bundle root in templates, docs, runtime path
   policies, search fixtures, and tests; no writer emits `fulltext/`.
2. Fresh schema contains `passages`, `passage_fts`, `passage_vec`,
   `file_index_state`, `concept_edges`, `code_artifacts`, `code_runs`, and
   computed-capable `evidence_sets`.
3. Capability checks prove FTS5 and `sqlite-vec` availability or fail dense
   retrieval closed while BM25 still runs.
4. Query-time reindex updates stale passages without a daemon.
5. A verdict/status change updates `passages.check_status` in the same
   transaction.
6. The retrieval fixture is preregistered and run; BM25 remains default unless
   the candidate clears the bar.
7. A project creates a code artifact, runs it in the verified sandbox, records
   stdout/stderr/output hashes, reruns it, and verifies the recorded output.
8. Network and secret-read denial tests fail closed for the code runner.
9. A computed evidence marker backed by a verified code run is complete; the
   same marker with a missing/changed output becomes evidence-incomplete.
10. Crash-at-boundary tests prove query indexes and code runs recover to
    committed/consumable, retryable/pending, or failed/hidden.

## 5. Out Of Scope

- Making dense/vector/graph retrieval the default without fixture evidence.
- Automated idea/tension/gap invention.
- Non-Linux/WSL code execution unless that platform has its own passing sandbox
  proof.
- Two-way canvas/outline sync.
- Multi-device writes, raw dataset bundling, and the real beta.1 seed corpus.
- A polished beta.1 review UI. alpha.19 may expose CLI/read-API affordances only.
