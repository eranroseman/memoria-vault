# ExecPlan - Implement 0.1.0-alpha.19

Living plan on the scratch branch. Design authority:
[0.1.0-alpha.19-design.md](./0.1.0-alpha.19-design.md). When this plan and the
design disagree, fix the plan.

## 0. Metadata

- **Task:** implement alpha.19: `fulltexts/`, dense/vector query substrate,
  schema references, computed evidence, and the full supported-platform
  code-output lane; then run the whole-docs audit and cut the alpha.19
  source-install checkpoint.
- **Implementation worktrees:** one branch/worktree per slice under
  `~/memoria-vault/worktrees/alpha19-<slice>` off `origin/main`.
- **Release/checkpoint:** `0.1.0-alpha.19`.
- **Non-goals:** beta.1 UI, seed corpus, multi-device topology, raw dataset
  bundling, non-Linux/WSL code execution without a platform proof.

### Contracts and invariants

Cross-cutting constraints the workstreams above must satisfy — a slice that
lands without its invariant is incomplete, not "mostly done":

- **Vector embedding (WS-B/D).** `passage_vec` stores its `embedding_model_id`
  and `vector_dim`; a vector's cache key is `(text_sha256, embedding_model_id)`,
  so a model or dimension change invalidates the cache and forces a rebuild.
  Distance metric is cosine, declared on the index. Tests use a deterministic
  hash-based embedder — no network, no model download — and the retrieval
  fixture pins the `embedding_model_id` + `vector_dim` it was built with. Absent
  embedding deps, or a model/dim mismatch, fails the dense-substrate capability
  check closed; BM25 still answers. The production `embedding_model_id` and
  `vector_dim` are an architectural choice — record them in
  `releases/0.1.0-alpha.19/decisions.md` before WS-D promotes any candidate.
- **sqlite-vec packaging (WS-B/D).** `sqlite-vec` is an **optional extra**
  (`[project.optional-dependencies].vector` in `pyproject.toml`), never a core
  `dependencies` entry and never referenced in `scripts/install.sh`. Dense
  retrieval loads it at runtime through the capability check; the core install
  stays `pydantic-ai-slim` + `PyYAML`. `SELECTED_RETRIEVAL_SUBSTRATE` stays
  `bm25`. The load-bearing guard is
  `tests/test_retrieval_substrate.py::test_packaged_search_has_no_vector_dependency`
  — core deps and the installer must stay disjoint from
  `{sentence-transformers, sqlite-vec, sqlite_vec}`; that test must remain green.
  Making it a core dep instead is an architectural choice → record it in
  `decisions.md` first.
- **Status cascade (WS-C).** Every `check_status` write goes through one shared
  state-layer cascade helper; `passages.check_status` is never updated ad hoc.
  A caller-coverage test enumerates all writer paths (state, worker, catalog)
  and asserts each routes through the helper, so no path silently drifts the
  copied status.
- **Code runner availability (WS-G/H).** A single gate —
  `runtime/code/runner.py::execution_availability(vault) -> Availability` — is
  the only path any caller (policy hook, worker, CLI) uses to check whether
  product code may run. It returns `unsupported` (with a reason) until the WS-G
  sandbox proof passes AND the WS-H run ledger, rerun verification, and
  export-warrant refusal are all present. F/G/H may land in separate PRs, but
  execution stays unavailable until the whole lane is closed. A negative test
  proves that with the WS-G proof passing but WS-H absent,
  `execution_availability` still returns `unsupported` and no product code runs
  — no half-enabled runner.
- **Code-warrant ordering (WS-E before WS-H).** WS-E adds the `code-warrant`
  evidence item before WS-H builds `code_runs` and the verification API.
  Therefore every `code-warrant` set is `evidence-incomplete` until WS-H lands:
  with no run table/API to resolve `code_run_id` + `output_sha256`, the set can
  never reach `complete`. WS-E ships and tests exactly that incomplete state; a
  warrant only becomes completable once WS-H's verification path exists.

## 2. Execution Order

1. A first. The path rename is mechanical but touches many tests; doing it
   first prevents every later slice from landing new `fulltext/` references.
2. B and C next. The query substrate is not useful until freshness and status
   cascade are correct.
3. D after B/C. Hybrid ranking over stale or wrongly checked rows is worse than
   no hybrid ranking.
4. E before H. Code runs cannot warrant claims until evidence knows how to
   reference them.
5. F/G/H together but in small PRs: records, runner, verification.
6. I with each PR: every code slice co-lands its own reference-doc update.
7. J after A-I land: one whole-docs audit over the full `docs/` surface (not a
   re-run of the per-PR sync — it catches cross-page drift, Diátaxis
   misplacement, and coverage gaps the slice PRs missed).
8. K last: cut the checkpoint only once A-J are merged, CI is green on `main`,
   and no milestone issue is `Readiness: Blocked`.

## 3. Concrete steps

Every code slice (WS-A..H) follows the same shape; only the edit targets and the
slice proof change. Steps are idempotent: a re-run from a clean `origin/main` is
always safe.

### 3.1 Shared setup — once per slice (`AGENTS.md` §1)

```bash
git -C ~/memoria-vault/main fetch origin
git -C ~/memoria-vault/main worktree add \
  ~/memoria-vault/worktrees/alpha19-<slice> -b feat/alpha19-<slice> origin/main
cd ~/memoria-vault/worktrees/alpha19-<slice>
npm ci --silent            # per-worktree cspell/markdownlint for the commit hooks
```

### 3.2 Edit targets and proof per slice

`(new)` marks a file the slice creates; every other path exists on `main` today.
The proof command is the smallest failing-first test that turns green when the
slice is done.

| WS | Edit targets | Slice proof |
| --- | --- | --- |
| A | `vault-template/.memoria/` folder schema, `runtime/paths.py`, `runtime/trusted_writer.py`, `runtime/search_index.py`, `runtime/policy/paths.py`, `docs/`, `tests/` | `rg -n "fulltext/" src vault-template docs tests` shows no new hits; `pytest tests/ -k fulltext` |
| B | `runtime/schema.sql` (new tables + `PRAGMA user_version = 8`), `runtime/state.py` (`SCHEMA_VERSION = 8`), `pyproject.toml` (`[project.optional-dependencies].vector`) | `pytest tests/ -k "schema or retrieval_substrate"` — fresh-init at v8, v7 rejected, no-vector-dep test still green |
| C | `runtime/indexing.py` `(new)`, `runtime/state.py` (shared cascade helper) | `pytest tests/ -k "reindex or cascade"` |
| D | `runtime/retrieval.py` `(new)`, `runtime/search_index.py`, `tests/fixtures/` retrieval fixture | `pytest tests/test_retrieval_substrate.py` — fixture compares BM25/FTS5/vector/RRF; BM25 stays default |
| E | `runtime/evidence.py`, `runtime/schema.sql` | `pytest tests/ -k evidence` — `code-warrant` set is `evidence-incomplete` (WS-H absent) |
| F | `runtime/code/records.py` `(new)`, `runtime/projections.py` | `pytest tests/ -k code_artifact` |
| G | `runtime/code/runner.py` `(new)`, `runtime/policy/hook.py` | `pytest tests/ -k "sandbox or runner"` — network/secret/shell/escape canaries fail closed; unsupported platform refuses |
| H | `runtime/code/runs.py` `(new)`, `runtime/evidence.py`, export gate | `pytest tests/ -k "code_run or export"` — rerun verifies, export refuses a broken warrant |

Expected output is the observable acceptance claim in §4, not a pasted
transcript for unbuilt code. Each slice's PR pastes its *actual* run transcript
into the PR body as the evidence.

### 3.3 Rollback (any slice)

```bash
cd ~/memoria-vault/main
git worktree remove ~/memoria-vault/worktrees/alpha19-<slice>   # refuses if dirty
git branch -D feat/alpha19-<slice>                              # only once confirmed unmerged
```

All new query/index/vector state is **derived**: drop the new tables and re-run
`rebuild` from vault markdown — never a lossy path. The schema bump is additive
DDL, so re-running `_init` is idempotent.

### 3.4 PR checklist (every slice)

- [ ] branched off fresh `origin/main` in its own worktree (`AGENTS.md` §1)
- [ ] failing-first test for the slice's behavior committed before the fix
- [ ] `python scripts/verify pr` green
- [ ] the slice's own reference-doc update co-landed (WS-I)
- [ ] no new **core** dependency, no migration function, no back-compat shim
      (pre-1.0 alpha owes neither)
- [ ] `gh pr create --base main --fill`; `gh pr checks <n> --watch`

## 4. Validation

Run the smallest useful checks per PR, then the full gate before release:

- path/folder tests for `fulltexts/`;
- schema smoke: fresh-init creates every new table at `user_version = 8`, and a
  v7 vault is rejected by `state.py::_init` (no migration — delete-and-rebuild);
- packaging: `test_packaged_search_has_no_vector_dependency` stays green
  (`sqlite-vec` is an optional extra, absent from core deps and `install.sh`);
- FTS5 and `sqlite-vec` capability checks;
- retrieval fixture report comparing BM25, FTS5, vector, hybrid, and graph
  variants;
- stale-file lazy reindex test;
- status-cascade transaction test;
- code-runner canary tests: no network, no secret read, no shell command,
  timeout, max-output cap, and filesystem escape (symlink, `..` path traversal,
  and write-outside-the-output-dir all denied);
- vector fixture determinism: the hash-based test embedder reproduces identical
  vectors across runs, and a model_id/dim mismatch invalidates the cache;
- computed evidence complete/incomplete tests;
- crash/recovery tests for index writes and code-run materialization;
- `python scripts/verify pr` before merge.

Docs audit (WS-J):

- `python scripts/checks/docs_doctor.py docs`,

  `python scripts/checks/docs_doctor.py --vault-links` clean;

## 6. Risks

- `sqlite-vec` is a new runtime dependency. Keep vectors derived and rebuildable;
  BM25 remains the fallback.
- `bwrap` is argument-sensitive. Treat the proof tests as the feature gate, not
  as optional security decoration.
- The path rename is wide. Do it mechanically, with tests, before touching
  retrieval internals.
- Docs audit is deferred to after A-I land, so per-PR docs debt can accumulate.
  WS-I keeps each slice's own reference page current; WS-J only sweeps
  cross-page drift and coverage. Skipping WS-I does not make WS-J cheaper.
- release-please is paused (`workflow_dispatch`-only, manifest pinned to
  `0.0.0`). Merging to `main` cuts no tag; WS-K hand-writes the CHANGELOG entry
  and version bump, exactly as alpha.18 did.
