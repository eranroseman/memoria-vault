## 0. Identity — what alpha.17 is

alpha.16 gave the researcher the **READ** half: capture → digest → notes/claims →
gap analysis → grounded Ask, over the two-graph substrate. alpha.17 adds the
first half of **WRITE** — **producing verified TEXT output and closing the loop
back into the graph** — plus the two cross-cutting instruments the mission needs
early (an anti-suppression exploration channel and gate-efficacy measurement).

**The ruling scoping principle (owner directive): include everything we can build
now; defer only what we can't.** Applied honestly, the deferred items split into
**two independent chains** (corrected 2026-07-06; see §6 and the
[host/stack evaluation](0.1.0-alpha.17-host-stack-evaluation.md) — an earlier
draft of this section wrongly traced everything to one root):

- **In alpha.17 (design-resolved + host-neutral):** the writing spine — the
  **argument-graph slice**, its **linear outline view**, and the **draft** —
  draft→graph write-back, text output verification, a minimal compose surface
  (files + read-API + CLI), the exploration/anomaly channel, and gate-calibration
  instrumentation.
- **Deferred, chain (a) — security gate, client-independent:** code
  **execution** needs sandbox hardening + adversarial validation. This has
  nothing to do with which client renders the UI (code never runs in the
  client's process either way).
- **Deferred, chain (b) — client/adapter choice + topology design:** the
  **spatial canvas view** and the **full multi-gate workspace rendering**.
  The client/adapter half is now **recommended** (a thin Obsidian plugin,
  §6) but not yet built; the workspace-topology half remains an undesigned
  owner job-to-be-done. See §6.

**Why text escapes both chains — and where the boundary really is.** The slice,
outline, and draft are **plain files** editable in any editor and over the
**read-API + CLI**; they make no commitment to a client or a sandboxed runtime.
What defers is anything that must be *rendered* (the spatial canvas, workspace
panes — chain b) or *executed* (code — chain a). The **keep-test holds
throughout**: every artifact is a plain file usable in `vim`/`nano`/a file
browser.

All alpha.16 guardrails carry unchanged: **no-write-time-oracle** (checked ≠
approved), **no synthetic index**, **propose-not-dispose**, **graph *structural*
integrity** (not per-node semantic validity), single **VaultWriter** +
snapshot→apply→verify→commit + one-click revert.

## 1. The writing module — text output

The owner's named flow, made concrete: **argument graph → canvas → outline →
draft (TEXT)**. Every stage is a *proposal surface*: the system proposes with its
reasoning shown; the human — sole owner of judgment — disposes (beta.1 §1).

### 1.1 Argument graph (consumed, not built here)

The writing module reads the alpha.16 argument graph: **claims-with-context**
carrying TEMPO fields (`temporal_scope`, `tense`, qualifier/certainty,
originating section — a claim's truth is determinate only relative to these),
connected by human-authored typed edges. Saturation is expressed as **honest
coverage reasoning** (which claims lack support, which lack a counterpoint) —
**never a synthetic saturation score** (anti-synthetic-index). This is the
substrate the project slice is drawn from.

### 1.2 The project slice and its two views

The structural unit of a writing project is the **project slice of the argument
graph** — the member notes/claims, their typed connections, and their ordering.
The slice is a **plain-file, editor/CLI-editable artifact, fully host-neutral**;
it is the single structural truth, and **canvas and outline are two *views* of
it**, not two artifacts with a derivation between them:

- **Outline = the linear view (ships now, host-neutral, primary).** A markdown
  list of the slice in order — the surface the human actually arranges and the
  input to the draft. Nothing about it needs a host.
- **`.canvas` = the spatial view (emit-only in alpha.17).** The engine can *emit*
  a JSON-Canvas projection of the slice for the small-graph case; **interactive
  spatial viewing/editing — drag-to-arrange, persisted/revertible positions —
  needs a host and defers** (§6). The **~20-node** line (node-link readability
  collapses above it, matrix/list wins — Ghoniem 2004; Hearst 2009) governs
  **whether emitting the spatial projection is worthwhile at all**, not a
  "fallback" — the list is primary either way.
- **Membership is proposed, not forced.** The system *suggests* which notes/claims
  belong on the slice and how they connect (with per-item reasoning); it never
  forces tags/edges upfront (premature formalization makes users reject the
  system — Shipman & Marshall 1999). Candidate retrieval is **BM25+dense, never
  graph traversal** — typed edges are the *argument* layer, not the retrieval
  representation.
- **On-disk grammar (minimal).** The slice's canonical storage **is** the
  project's `outline.md` — there is no separate slice file. It is an ordered
  Markdown list, one member per line:
  `- {note-id} [{relation}] — {reasoning}`
  - `{note-id}`: the member note/claim's stable ULID `id` (never a path or
    title — alpha.16's identity rule; a path/title can change without
    breaking the reference).
  - `{relation}` (optional): the type of this item's connection to the item
    above it in sequence, drawn from the **same fixed relation set the
    argument graph already uses** — `{supports, contradicts, extends}`
    (alpha.15/16 `LINK_RELATIONS`) — no new taxonomy. Omitted means "ordered,
    no asserted relation."
  - `{reasoning}` (optional): free text — the system's proposal reasoning
    when suggested, or the human's own note once arranged.
  - **Order** is the list's own line order in the file — no separate numeric
    field to keep in sync.
  - **Versioning:** `outline.md` is a normal tracked artifact under the
    existing snapshot→apply→verify→commit + one-click revert protocol
    (alpha.16 §4) — no new mechanism.
  - **The `.canvas` emission (below) renders *this same list*** as JSON-Canvas
    nodes/edges — one node per line, edges from the `{relation}` annotations —
    so canvas and outline are provably one file and its projection, not two
    files a sync step must keep aligned.

### 1.3 Arranging the slice — no separate derivation

The human arranges by editing the slice's **membership, order, and connections**
(in any editor or via CLI). **The outline *is* that ordering** — there is no
canvas→outline derivation to run, no re-derivation step, and no "this discards
your outline edits" warning, because the linear and spatial views project the
*same* slice. This deletes the machinery beta.1 §1.3 carried (a one-way
re-derivation between two artifacts) once the spatial view is deferred.

The owner's `argument graph → canvas → outline → draft` flow is **preserved**:
the canvas survives as the (deferred) spatial *view* of the same slice, not a
deleted concept — it returns interactive when the host lands.

### 1.4 Draft — the TEXT output

The draft is composed section by section from the outline. It is the **output
artifact**, not a write into the graph. Synthesis is where the research bites
hardest, so the rules are concrete (beta.1 §1.4 + rethink Class B / Q8):

- **Feed coherent slices, not top-k fragments.** Each outline node maps to its
  member notes/claims; the **slice builder** emits the **whole note if it fits the
  per-node token budget**, else the **section-expanded anchor** with the
  section/document header prepended; the **strongest anchor at slice top or
  bottom** (never padding the middle — U-shaped "Lost in the Middle", Liu 2023);
  the **total capped** well under the model window (context is non-monotonic —
  Databricks 2024); a cheap **Self-Route** falls to fragments only for narrow
  factual nodes (Li 2024). Notes are indexed at section/whole-note granularity
  (LongRAG).
- **Numbers: engine-deterministic recompute now; analysis-computed defers.** A
  printed number is **either** produced by the **engine's own deterministic code
  path** — counts, ratios, date/interval arithmetic over catalog/graph data, no
  sandbox — in which case verification (§2) recomputes it and it must match;
  **or** it requires **running analysis code**, in which case alpha.17 has **no
  engine-checkable warrant for it** (the code-warrant defers with execution, §6),
  so the claim enters **evidence-incomplete → PI** rather than shipping. The model
  never produces a printed number (abstention + numbers-in-code is a pipeline
  property: 37.1% → 70.2%, Zou 2024). Citations resolve against the catalog's
  deterministic metadata to a **verbatim span**.
- **Warrant = a typed evidence-*set*, with a concrete storage contract.** Each
  drafted claim carries an ordered evidence set of **items**, each a
  **source-span ref** (`citekey#^pNNNN` or an existing `ev-<8hex>` pinned-evidence
  id). *(The **code-warrant** item variant — a computed claim warranted by a
  re-runnable script — defers with code execution, §6.)*
  - **ID:** `ev-<8hex>` is **minted once, opaque/random** at creation and written
    into the inline marker — **not** content-derived, so editing the set's items
    does not change its id or break references (the "stable ID" requirement).
    8 hex chars (32 bits) is a real collision surface at single-vault scale over
    years of use, so minting is **check-and-retry**: generate, scan existing
    markers (the canonical source, §durable form below) for a match, regenerate
    on collision — the same pattern short random IDs always use (Git's
    abbreviated hashes, nanoid), no length change needed at this vault's scale.
  - **Durable form / source of truth:** a trailing inline marker on the claim
    block — `%%ev: <id> type=… state=… review=…%%`, the same pattern as
    `%%prov%%` — is **canonical**. The `evidence_sets` DB row *(id, block_ref,
    items_json, type, state, review_required, run_id)* is **derived state**:
    `rebuild` reconstructs it by scanning markers and resolving items
    (consistent with alpha.16 §4's derived-store split). `block_ref` is the
    claim block's anchor `<file>#^blk-<id>`, reusing alpha.16's `#^` anchor
    scheme.
  - **Derived, never asserted — two independent signals, not one overloaded
    one.** The engine derives `type` — *single-span* / *multi-span* /
    *multi-hop* / *implicit* — from the items. `state` is **strictly about
    resolution**: *complete* if every item resolves, else *evidence-incomplete*
    (missing/broken item — **routes to the PI**, §2). `review_required` is a
    **separate derived boolean**, true iff `type` ∈ {*implicit*, *multi-hop*},
    **independent of `state`** — a fully-resolved multi-hop set is
    `state: complete` (nothing is missing) but `review_required: true` (a
    resolved chain across multiple works is not the same claim as a
    resolved single span; span-matching cannot verify a multi-hop reasoning
    chain, so it routes to the PI regardless of resolution, §2). Keeping these
    separate avoids calling a fully-resolved set "incomplete," which would
    otherwise violate §2's own provenance-labels rule ("no label ever means
    more than it says"). A span match is labeled **citation-correct, not
    grounded** (≤57% of RAG citations are post-rationalized — Wallat).
  - **Export behavior:** `%%ev%%` markers are internal provenance, **never
    published** — on export each resolvable item is rendered as a formatted
    citation (Pandoc/markdown) and the marker is stripped. Export **blocks** on
    any `state: evidence-incomplete` set, and on any `review_required: true`
    set that the PI has not yet disposed (§2/§5's existing disposition
    capture — no new tracking mechanism); once disposed, a `review_required`
    set exports on its resolved items like any other.
- **Abstain rather than fabricate.** Where the corpus does not support a passage,
  the draft emits an explicit gap/"Unknown" marker; faithfulness comes from
  deterministic grounding, not the model *choosing* to say it doesn't know (Rao &
  Callison-Burch; CoT degrades abstention — Shaikh 2023).
- **Prose, not a forced schema.** The draft is natural language; constrained
  decoding to a rigid schema corrupts negation and polarity — the worst error
  class here (Shin 2021). Semantic content is generated as prose first; structured
  extraction (IDs, links, tags, citekeys — closed-vocabulary, machine-checkable
  slots only) is a **second pass**, never at the point of semantic commitment
  (rethink Q13).
- **Propose-not-dispose.** The draft is a proposal with per-passage reasoning; the
  human composes and judges. Drafting **does not auto-write knowledge back** into
  the graph — that is the §1.5 gated action.

### 1.5 Draft → knowledge-graph write-back

Drafting is a *producer* of notes, not only a consumer (a manuscript is the
rearrangement of notes; writing reveals notes that don't yet exist —
Zettelkasten/Matuschak). The write-back is a **PI-gated action, never automatic**
(no-write-time-oracle) (beta.1 §1.5 + rethink Q9):

- **Extract to an atomic, concept-oriented permanent note.** A first-class
  in-draft action turns a selected passage into a new `note` (claim/question) in
  the graph — atomic and concept-oriented, detached from any single source.
- **Transclusion, markdown-neutral, not copy.** The draft keeps a **plain relative
  markdown link** to the one canonical note — `[title](../notes/<id>.md)` — as the
  durable back-link (resolves in any editor; keep-test), **plus an optional
  graceful-degrade embed marker** `%%embed: <id>%%` a host *may* render as an
  inline transclusion. It is **never** a host-specific `![[…]]` transclusion that
  breaks portability, and never a pasted copy that silently diverges.
- **Provenance + maturity.** The note carries a **`source:` link** field (a link to
  the source record, never inlined source text) and **`maturity: seedling`**
  frontmatter, distinguishing a raw draft-spawned extraction from a vetted
  permanent note.
- **Squeeze-point trigger.** The system *offers* extraction when a draft section
  accumulates many loose references (the "mental squeeze point") — an
  event-driven prompt, never an auto-write.

Promoted notes re-enter gap analysis like any note, closing the post-draft loop.
*Measurement residual (§7):* the squeeze-point detection threshold, and whether
`seedling` notes need a distinct review lane before counting as compose evidence.

### 1.6 Close the loop

Drafting surfaces new gaps — claims that abstained, evidence-incomplete warrants,
sections with no counterpoint. These feed back into alpha.16's **gap analysis over
both graphs**, so writing drives the next READ iteration rather than being a
terminal export.

## 2. Output verification — TEXT

The "verify" step of the cycle, consistent with **no-write-time-oracle** and
**graph structural integrity**. Verification is **never a semantic quality
oracle** — human+machine review does not beat machine alone and a wrong machine
verdict drags the reviewer *below* baseline (Jacobs 2021). It produces
**deterministic flags + PI routing**, and proposes; it does not dispose
(beta.1 §4).

- **Provenance labels — no label ever means "true."** `human`; `machine`;
  `mc`/`span-matched` (= **citation-correct, not grounded**); `contradicted`;
  `unchecked`; `tainted` (depends on machine/untrusted content — monotone, never
  clears); `stale` (evidence/source changed after the check). Semantic truth
  stays the user's judgment.
- **Deterministic citation/number checks (necessary, not sufficient).** Every
  cited claim's evidence set resolves: citekey/DOI exists, the verbatim span is
  present in the source, and **every engine-computed deterministic number
  recomputes and matches** (analysis-computed numbers were already routed to
  evidence-incomplete upstream, §1.4). These prove *span existence and
  arithmetic*, **not grounding** — gates on form, never a certificate of truth.
- **Abstain / evidence-incomplete / review-required surfacing.** Passages that
  abstained, or whose evidence set is `state: evidence-incomplete` (unresolved
  — implicit with no resolvable item, or analysis-computed, §1.4), or whose
  evidence set is `review_required: true` (implicit or multi-hop **by type**,
  regardless of resolution) are flagged and **routed to the PI** — the exact
  claims that cannot be auto-resolved (incomplete) or auto-trusted
  (review-required) respectively.
- **Graph structural integrity.** The draft's references into the notes/claims
  graph must resolve (links, anchors, references) — *structural* checks, not
  per-node semantic validity.
- **Sparse, triaged, tooling-first.** Auditing costs the same expertise as
  authoring and floods under load (Feng & Liu). Verification routes **only
  high-uncertainty / high-leverage** items to the PI, invests in per-claim tooling
  (side-by-side source, citekey+span diff), and treats a **max-items-per-session /
  triage threshold as a first-class constraint** — not a throughput target.
- **Export-readiness gate (extends alpha.16 §7 minimal export).** Export **runs**
  on a clean draft — stripping/converting `%%ev%%`/`%%prov%%` markers to
  citations (§1.4) — and **refuses**, with an explained refusal, when
  `evidence-incomplete` claims, un-disposed `review_required` claims, or broken
  structural references remain (§1.4's export-behavior rule). Targets start
  at **markdown + Pandoc**, expanded on measured demand (rethink Q14).

*(Code verification-by-execution — beta.1 §4.2 — defers with code execution, §6.)*

## 3. Minimal compose surface

The owner's split is design intent: **separate deep work (the project document)
from task-oriented work (reviewing links) — distinct mindsets, don't mix**
(beta.1 §3). alpha.17 realizes the **Project** deep-work job **host-neutrally**,
not as a rendered workspace:

- **What it delivers, host-neutral:** the deep-work text artifacts — **slice,
  outline, draft** — as **plain files**, plus the **read-API** (Ask, evidence, gap
  report) alpha.16 already exposes, plus **CLI operations** for arrangement (edit
  slice membership/order/connections, derive/emit views, run verification,
  export). **No rendered UI is required.**
- **What defers with the host:** any *pane/workspace rendering* — a laid-out
  compose surface, the inbox/recommendation stream separation as a visual gate —
  is a host concern deferred to §6, exactly like the spatial canvas. The mindset
  separation is honored structurally (the task-oriented streams are not mixed into
  the Project CLI/read-API surface), independent of any host.
- **Constraints:** the keep-test holds (files first, the surface is a convenience
  over them, never a gate on them); the **no-daemon** posture is preserved.

The five UI-framework-independent jobs (Read / Knowledge / Project / Review /
Upkeep) remain the requirement; alpha.17 realizes **Project** as files + read-API
+ CLI and inherits the others from alpha.16. The *full* gate topology is §6.

## 4. Exploration / anomaly channel

A feedback loop trained on what the PI up-votes suppresses disconfirmations (the
McNamara fallacy: consensus-priored models converge and down-rank anomalies).
alpha.17 reserves a **separate exploration/anomaly channel not subject to the
relevance ranker** (beta.1 §6 + rethink Class D / Q10). It ships now because its
*data* is host-neutral, and it must exist *before* the loop can start suppressing:

- **A separate channel, produced as data.** A read-API query yields the
  coverage/contrary set — computed *separately from the ranker* — so the
  separation itself is host-neutral and ships now. Its **salient, labeled
  *placement*** — which the research says governs whether diverse items are
  *noticed* at all (relegated items go unseen — Munson & Resnick 2010) — is a
  **host-rendering concern that completes with the host**. alpha.17 delivers the
  channel, its per-item *why*, and the "consider the contrary" framing
  (André 2009; Lord 1984); it **flags that noticing-efficacy cannot be fully
  validated until the salient presentation lands** (honest boundary, not a
  silent gap).
- **Populated by coverage/diversity, not relevance** — submodular
  facility-location coverage (provable (1−1/e)) or DPP over citation-graph breadth
  (co-citation, Swanson-style cross-silo bridges); MMR as the cheap baseline. This
  **reuses alpha.16 §7 bounded citation-graph discovery**, not a new recommender.
- **A first-class contrary-evidence lane** — the local NLI already surfaces
  REFUTES within the retrieved set; extend it as a standing "contrary evidence"
  channel (SciFact framing). **An empty contrary list is never presented as
  agreement** (guardrail).
- **Measured as its own objective, validated by action** — intra-list diversity +
  subtopic/coverage + honestly-labeled serendipity (unexpectedness × usefulness),
  **never a synthetic score presented as truth**. Validated by whether the
  researcher **acts on** the channel; if measurement shows it is never acted on,
  it is theater and gets thinned (same discipline as the gate, §5).

*Honest caveat (carried from the rethink):* the filter-bubble harm is genuinely
debated (Bakshy 2015; Flaxman 2016) — this channel is **insurance, presented as
debated, not settled fact**, and no controlled A/B yet proves "separate region vs
blended" for a *noticing* outcome, so it is measured, not assumed.

## 5. Gate-calibration instrumentation

The load-bearing risk of the whole system is that **human review is theater** —
human+AI is frequently *worse* than the better of the two alone, losing exactly
when the AI is more accurate (Vaccaro 2024, n=106, g=−0.23). alpha.17 starts
measuring this **a release early**, because the WRITE half is the first place the
PI reviews machine *prose* at volume (rethink Class A / Q3-Q5; beta.1 §4.3/§6).
It ships now because it is pure instrumentation over the existing gate —
host-neutral, no new surface:

- **Standing seeded-error probe — with a structural containment invariant.**
  Inject a known low rate of deliberately-wrong machine proposals into the normal
  Review stream; measure the reviewer's catch rate and false-alarm rate (d′).
  **If seeded errors aren't caught above chance, the gate is theater** — surfaced,
  not hidden (Green & Chen reliability test; Jacobs incorrect-recommendation arm).
  **Containment (enforced, not by convention):** a probe item carries a **sentinel
  the VaultWriter refuses to persist** and **never receives a real `run_id` or
  commit**, so it **cannot be written back, exported, promoted, or left in durable
  graph state**; it exists only in the review display and is removed once its
  disposition is recorded. The *disposition* is kept for calibration; the probe
  *content* never enters the vault. **Opt-in dogfood mode only** until proven
  harmless.
- **Agreement re-judging, both directions.** Periodically re-judge a sample of
  accept *and* reject decisions against ground truth; track the **correct→wrong
  flip rate** (automation bias) vs wrong→correct. Healthy gate: wrong→correct ≫
  correct→wrong. Apply-rate bands (≳90% or ≲20%) flag rubber-stamping/disengagement
  — but only the re-judged sample is decisive (accept/override rate alone is
  uninterpretable — Felisberto 2024).
- **No persuasive explanations.** Proposals show the machine's **self-rebuttal**
  (argument-against) + coarse certainty, never a polished rationale — explanations
  raise acceptance without raising accuracy (Bansal 2021), and an
  interpretable-but-wrong explanation does the most damage (Jacobs).
- **Load caps as first-class constraints.** Review in **short bounded batches**
  (≤~60–90 min, ≤~a few dozen units — defect detection falls after 60–90 min,
  SmartBear/Cisco; vigilance decays in 15–30 min), never an open streaming
  approve-queue.
- **Disposition capture, implicit.** The loop records accept/reject against the
  proposal's existing impact/certainty/reasoning — **no separate rating chore**
  (Rogers 2006; Epstein 2020). Every tunable ships behind
  `production_enabled:false`: **no score changes routing, promotion, or visibility
  until grounded on recorded dispositions** (anti-synthetic-index as an operational
  rule).

The **ABAB efficacy protocol** (gate-on/off reversed ≥3×, ≥5 measurements/phase;
abandonment tracked) is the *analysis* run over this data; alpha.17 lays the
instrumentation, and the full n=1 efficacy verdict is a beta.1-completion output
(§6).

## 6. What alpha.17 defers (tracked debts, each with its gate)

Each is a debt against the MVP with a named gate — not a silent deferral.
**Correction (2026-07-06):** an earlier version of this section, and the
preceding 17/18-merge discussion, claimed all deferred items trace to *one*
root ("the host/stack decision is unmade"), including that code execution
"runs in the host's process model." The
[host/stack evaluation](0.1.0-alpha.17-host-stack-evaluation.md) §0 found that
coupling wrong: code never runs in the rendering client's process regardless
of which client renders the UI, so it does not depend on the client/adapter
decision. There are **two independent chains**, not one:

- **Code execution (Blocker 1 — security gate, OS/platform axis only, no
  client dependency).** Running project code (the executable-warrant and
  code-deliverable roles, beta.1 §2) needs a **hardened sandbox + adversarial
  validation** before it faces a real corpus, because ingested untrusted
  content can influence a run (InjecAgent; AgentDojo). The *minimal* sandbox is
  designed (no-network credential-free worker, project-write-scope **+ secret
  read-denylist** `~/.ssh`/`~/.aws`/keychain/provider-keys/DB,
  approved-commands, resource caps — rethink Class C), but its per-OS backend
  (bubblewrap / Seatbelt / WSL2) and the InjecAgent/AgentDojo-style adversarial
  validation are real, uncertain work — a client-independent dependency: the
  client only ever enqueues a code-run operation and reads back the result via
  the read-API, exactly like every other operation. *Path:* build + validate
  the sandbox; then the code lane, the **code-warrant evidence item** (§1.4),
  and code-by-execution verification land together — and analysis-computed
  numbers (§1.4) stop routing to evidence-incomplete.
- **The spatial canvas view + full multi-gate workspace rendering (Blocker 2 —
  client/adapter choice + workspace-topology design, now partially resolved).**
  The evaluation recommends **Obsidian as the markdown editor of choice plus a
  logically-thin first-party plugin** rendering `view-spec.v1` over the
  existing `memoria serve` loopback HTTP transport as the client — no new
  engine-side contract, no independent write path (enqueue-only, same as every
  client), and it gets Obsidian's already-maintained cross-platform desktop
  story and native `.canvas` renderer for free (canvas position edits become
  foreign edits through the existing quarantine/scan path), at the honest cost
  of ceding fine-grained canvas-interaction control to Obsidian's renderer. A
  local web app is architecturally identical (same transport, same view-spec,
  same enqueue-only writes) and remains available later for headless/remote
  use. What remains genuinely open is the **workspace/gate topology** itself
  (the gate
  triad, deep-vs-task surfaces beyond the minimal files/CLI Project §3,
  left-nav/right-pane-Co-PI, active-assistance placement) — an **undesigned**
  owner job-to-be-done; both alpha.15's carried model and beta.1's
  five-surface model are rejected as authority-only inputs (alpha.16 §7), and
  the evaluation's research (Gerrit/GitHub review mechanics, iA Writer's
  deep-work rationale, the contested left-nav/right-Co-PI convention) informs
  but does not settle that design task. **Q6 (does heavy outline editing
  diverge from the spatial canvas?) stays deferred with the interactive
  canvas** — the evaluation found canvas/outline should be two views of one
  shared project slice (already corrected in §1.2/§1.3), which narrows Q6 to
  "does the client's rendering of that shared slice ever diverge," not
  resolved simply by picking a client. *Path:* build the recommended thin
  Obsidian plugin; run the workspace-topology design pass (independent of the
  client build, per the evaluation's §2b).

**No single "host/stack" root gates both of the above** — code execution is
gated by sandbox validation alone; the workspace/canvas is gated by the
client-adapter choice (now recommended) plus the still-open topology design.
Text output does not depend on either, which is why it ships in alpha.17
regardless.

**Inherited alpha.16 follow-ons that remain open** (unchanged by alpha.17):
sqlite-vec fusion behind the retrieval benchmark gate (alpha.16 §5); fuzzy
annotation re-anchoring behind a measured orphan rate; the QLoRA adapter as a
tracked contingency; advanced backup hardening. **Deferred to beta.1 completion,
not lost:** the full onboarding wizard + ≤30-min time-to-first-answer bar
(beta.1 §5), the complete feedback dashboard + ABAB efficacy verdict (beta.1 §6),
and the end-to-end MVP acceptance run on a real corpus (beta.1 §7).

## 7. Open questions & measurement residuals

alpha.17 invents no mechanism to fill a gap. It **starts collecting** the data
these need (each behind `production_enabled:false` until grounded):

- **Coherent-slice per-node token budget** — the threshold above which a note is
  section-expanded rather than passed whole (rethink Q8) — tuned on the real
  corpus.
- **Spatial-projection worthwhileness** — at what density (the ~20-node line) it
  is worth *emitting* a `.canvas` projection at all (Q7). *(The interactive
  "do users diverge from the canvas" half, Q6, is host-deferred — §6.)*
- **Evidence-set PI-routing fraction** — what share of drafted claims land
  `evidence-incomplete` (unresolved, including analysis-computed numbers) vs.
  `review_required` (implicit/multi-hop by type, §1.4) (Q2) — a measurement,
  not a target; the two are tracked separately since they route for different
  reasons.
- **Squeeze-point threshold** — when to offer draft→note extraction, and whether
  `seedling` notes need a distinct review lane (Q9).
- **Gate efficacy (the big one)** — the seeded-error d′, both-directions flip
  rate, and the ABAB verdict (§5) — the honest answer to "is the gate theater,"
  resolved by measurement, not assumption.
- **Exploration acted-on rate** — whether the channel is used or is theater (§4).

**Two independent things gate the next version** (§6): **sandbox validation**
unblocks code execution alone; the **workspace-topology design pass**
(the client/adapter half is now recommended, not a blocker) unblocks the
spatial canvas + full workspace. Neither blocks the other, and neither blocks
alpha.17 — text output ships, unblocked, regardless of both.

## 8. Acceptance — the text-output loop, end-to-end, host-neutral

alpha.17 is accepted when, **on a real corpus, via files + the read-API + CLI (no
host UI required)**, the text-output loop runs end-to-end with no core item
stubbed (extends the alpha.16 acceptance list):

1. **Slice + outline** — the system proposes a project-slice (member notes/claims
   + connections) with per-item reasoning; the human arranges membership, order,
   and connections **via the outline (list) in any editor or the CLI**; the engine
   can **emit** a `.canvas` spatial projection (past the ~20-node line) —
   **viewing/editing the canvas is not required and defers with the host**.
2. **Draft (text)** — composed from coherent slices (whole-note-if-budget else
   section-expanded anchor, top/bottom placement, capped); **every
   engine-computed deterministic number recomputes and matches, and any number
   requiring analysis-code execution routes to evidence-incomplete rather than
   shipping**; every citation resolves to a verbatim span; unsupported passages
   **abstain**; each claim carries a typed evidence set with a **mint-once
   `ev-<8hex>` id**.
3. **Verified** — deterministic citation/number checks pass or route to the PI;
   `evidence-incomplete` and `review_required` claims are both surfaced;
   structural graph references resolve; export **runs on a clean draft**
   (markers stripped/converted to citations) **and refuses**, with an
   explained refusal, on an unclean one.
4. **Write-back** — a draft passage promotes to an atomic `maturity: seedling`
   note via a **relative-link back-link (+ optional embed marker), not a copy**,
   with `source:`-link provenance, and **re-enters gap analysis**.
5. **Exploration channel** — on the **acceptance fixture** (a seed corpus
   constructed, like the gate-calibration seeded-error probe in §5, to contain
   at least one genuine coverage/contrary candidate), the channel surfaces it
   with a *why* (its salient placement is noted as host-completing). This is
   not assumed to hold on every real corpus: a corpus with no genuine
   contrary/diverse candidate is expected to show an **honest empty channel**,
   not a fabricated item — consistent with §4's guardrail that an empty
   contrary list is never presented as agreement. Acceptance is the fixture
   behavior; the empty case on a real corpus is a defined, correct outcome, not
   a failure.
6. **Gate instrumentation** — the **contained** seeded-error probe
   (non-persistable, no `run_id`), both-directions re-judging, and disposition
   capture are live and producing honest data (no synthetic score).

*(No separate alpha.17 requirements/governing doc is created here per the
"design only" scope, and this is a **proposal**, not a ratified plan — accepted
choices route to `scratch/releases/0.1.0-alpha.17/decisions.md` (§header). If we
split governance out later, this §8 is the seed for its acceptance list — kept
single-authoritative to avoid the drift alpha.16 §7-era design warned about.)*

## 9. Trade-offs

- **Whole-note slices cost tokens** — mitigated by the budget threshold + a
  Self-Route to fragments for narrow factual nodes. Flips toward fragments only
  where measurement shows the budget is routinely blown.
- **The spatial canvas defers** — the honest cost of the recommended client not
  being built yet, and of workspace topology staying an open design task.
  The human arranges the slice via the linear outline (editor/CLI) until the
  thin Obsidian plugin lands; deleting the canvas→outline derivation (§1.3)
  makes this *simpler*, not just deferred. **If the spatial canvas matters
  enough to the owner, that is the concrete argument for prioritizing the
  plugin build now** rather than after alpha.17 — the same gate as the 17/18
  merge question, now narrowed to a build-order decision rather than an open
  evaluation.
- **Write-back as a gated action adds friction** — the deliberate price of not
  laundering machine prose into the human note graph (no-write-time-oracle).
- **Measuring the gate spends real attention on seeded fake items** — kept at a
  low rate, contained so they can never corrupt the vault (§5), and counted as
  calibration, not work; the alternative is not knowing whether the gate does
  anything.
- **Deferring code + spatial-canvas + workspace means alpha.17 delivers half the
  WRITE mission** — the deliberate, honest consequence of **two separate,
  independent gates**: code defers on unfinished sandbox validation; canvas +
  workspace defer on an undesigned topology (the client/adapter half is now
  recommended, §6). Merging them in would either block ready text output on
  security hardening and an unresolved design, or ship code/workspace unsafe
  and undesigned. Text output shipping now, unblocked by either gate, is the
  correct call.
