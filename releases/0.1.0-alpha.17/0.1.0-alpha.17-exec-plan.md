## 1. Purpose

Before alpha.17: alpha.16 ships the READ half — capture → digest →
notes/claims → gap analysis → grounded Ask — over the two-graph substrate, via
files + CLI + a host-neutral read-API. Writing and code stay out of scope.

alpha.17 lands the first half of "WRITE," on a real corpus, **via files + the
read-API + CLI, with no host UI required**:

- **Project-slice + outline** (design §1.2/§1.3): the system proposes a
  project-scoped slice of the argument graph with per-item reasoning; the
  human arranges membership and order by editing `outline.md` directly;
  connections are never re-declared there — they're computed as the
  intersection of each member note's own argument-graph `links:` with current
  slice membership, recomputed on every read; the engine can emit a `.canvas`
  spatial projection past the ~20-node line (interactive canvas
  viewing/editing stays excluded, with the host track).
- **Draft composition** (design §1.4): coherent-slice synthesis (whole-note if
  it fits the token budget, else a section-expanded anchor with its header
  prepended; the strongest anchor at the top or bottom, never the middle;
  the total capped; a cheap Self-Route to fragments for narrow factual nodes);
  every engine-computed deterministic number recomputes and matches; any
  number that would need analysis-code execution routes to
  evidence-incomplete rather than shipping; unsupported passages abstain
  rather than fabricate; each drafted claim carries a typed, mint-once
  `ev-<8hex>` evidence set.
- **Text-output verification + the export-readiness gate** (design §2):
  deterministic citation/number checks, structural graph-reference checks,
  sparse/triaged PI routing on a load-capped budget; export **runs** on a
  clean draft (markers converted to citations) and **refuses**, with an
  explained refusal, on an unclean one.
- **Draft→graph write-back** (design §1.5): a first-class in-draft action
  promotes a selected passage to an atomic, concept-oriented note — no new
  frontmatter verdict field; draft-origin lives in the existing DB
  `check_status: unchecked` default and journal provenance — via a plain
  relative-link back-link (never a copy, never Obsidian's `![[…]]`),
  re-entering gap analysis.
- **The exploration/anomaly channel** (design §4): a coverage/contrary query
  over the existing citation graph, computed separately from the relevance
  ranker, with an honest empty result on a real corpus that has nothing to
  surface — never a fabricated candidate.
- **Gate-calibration instrumentation** (design §5): a standing seeded-error
  probe with structural containment (no real `run_id`/commit — ever), plus
  both-directions disposition re-judging, all behind
  `production_enabled:false`.

Observable outcome: on a real corpus, a researcher opens a project slice
(`memoria slice …`), arranges `outline.md` in any editor, composes a draft
(`memoria draft …` / `memoria compose …`) whose numbers and citations are
provably grounded or explicitly flagged, verifies it, exports it — refusing
cleanly if it isn't ready — promotes an insight back into a permanent note,
sees an honest exploration/contrary slate, and gets real (not synthetic)
calibration data on whether the review gate does anything. All of this from
plain files, the CLI, and the read-API — no rendered UI, no plugin, no code
execution.

Define terms on first use in every slice: **evidence set** (an ordered list of
source-span or pinned-evidence items backing a drafted claim), **source-span
ref** (`work_id#^pNNNN` — never a citekey, which is mutable), **block anchor**
(`<file>#^blk-<id>`, the claim block's own address), **marker** (the trailing
`%%ev: …%%` inline annotation that is the canonical, durable form), **slice**
(the project-scoped set of argument-graph members + their order), **outline**
(the slice's linear, host-neutral, human-edited view — `outline.md`),
**computed connection** (a typed edge shown only because it exists in a
member's own `links:` AND both endpoints are in the slice — never stored in
the outline itself), **`state` vs. `review_required`** (state is strictly
about item resolution; review_required is a separate signal, true iff the
evidence type is implicit or multi-hop, independent of resolution), **seeded-
error probe** (a deliberately-wrong proposal injected into the review stream to
measure whether the reviewer catches it), **disposition** (the PI's
accept/reject decision on a reviewed item), **squeeze-point** (the moment a
draft section accumulates enough loose references that extraction-to-note is
offered).

### PR-A1 — Evidence-set text + anchor primitives

Genuinely new (see §2's verified baseline facts — no marker parser, no `#^`
anchor scheme exists anywhere today). Builds three things: (1) the
`%%ev: <id> type=… state=… review=… items=<ref1>|<ref2>|…%%` inline-marker
parse/serialize layer — a trailing marker on a claim block, canonical source of
truth, carrying the actual items (not just the derived `type`/`state`/
`review_required` fields, which by themselves would give `rebuild` nothing to
reconstruct `items_json` from); (2) the block-anchor scheme — source-span
refs `work_id#^pNNNN` (`work_id` is alpha.16's stable identity; never a
citekey, which changes under Zotero metadata repair) and block refs
`<file>#^blk-<id>` for a claim block's own address — mint + resolve functions;
(3) the `ev-<8hex>` ID: minted once, opaque/random, **not** content-derived (so
editing a set's items never changes its id), with check-and-retry on collision
(scan existing markers — the canonical source — regenerate on a hit; 8 hex
chars is a real but small collision surface at single-vault scale, and
retry-on-mint is the standard mitigation for short random IDs, the same
pattern Git's abbreviated hashes and `nanoid` use).

Decision point → `decisions.md`: whether to repurpose the reserved, currently-
unused `anchors: list` field on `note.yaml`/`digest.yaml`, or keep the marker
fully separate. Recommend keeping them separate — the marker is the sole
canonical store; reusing `anchors:` would create two overlapping mechanisms
for the same fact.

Test: `tests/test_alpha17_evidence_markers.py` — round-trip parse/serialize,
ev-id collision-and-retry, ref-shape validation (reject a citekey-shaped
source-span ref).

### PR-A2 — Evidence-set derived store

Genuinely new (§2: no `evidence_sets` table exists; `schema.sql` ends at
`user_version = 5`). Depends on A1 (the marker format A2's `rebuild` scans).
**No migration function** — per §0's policy (pre-1.0 alpha, no
back-compat/migration path; the answer to a schema change is delete the
sandbox and reinstall), this is a plain `CREATE TABLE IF NOT EXISTS
evidence_sets(...)` block added to `schema.sql`, with the file's trailing
`PRAGMA user_version = 5;` bumped to `6` and `state.py::SCHEMA_VERSION`
bumped to match — no destructive rename/copy/guard logic, since there's no
prior `evidence_sets` data to preserve and no obligation to upgrade an
existing vault in place. Adds the `evidence_sets(id, block_ref, items_json,
type, state, review_required, run_id)` table. `type` (single-span/multi-span/
multi-hop/implicit), `state` (complete iff every item resolves, else
evidence-incomplete — strictly about resolution), and `review_required` (a
**separate** derived boolean, true iff type ∈ {implicit, multi-hop},
independent of `state`) are all derived from the items, never asserted — this
split exists specifically so a fully-resolved multi-hop set is `state:
complete` (nothing missing) but `review_required: true` (a resolved chain
across multiple works still isn't the same warrant class as a resolved single
span, and calling it "incomplete" would be a false label). `evidence_sets` is
**derived state**: `rebuild` reconstructs it by scanning A1's markers and
re-resolving each item — the marker, not the DB row, is the source of truth.

Decision point → `decisions.md`: confirm `user_version = 6` (the next integer;
recommend yes, no reason to skip).

Test: `tests/test_alpha17_evidence_sets.py` (derivation correctness, the
state/review_required independence case) + a schema-version-bump test
(a v5 fixture DB gains `evidence_sets` and reaches `user_version = 6` after
`_init`, beside the existing `state.py` schema tests) + `tests/test_schemas.py`
for the schema-mirror docs.

### PR-B — Project slice + outline

Extends `knowledge.py::_note_edges`/`_argument_component`: generalize the
member-set parameter from "the argument component" to "the outline's current
membership," so a connection between two slice members is computed — shown
iff it already exists as a typed edge in a member's own `links:` frontmatter
AND both endpoints are currently in the slice, recomputed on every read, never
duplicated into `outline.md`. This is why §1.1's "argument graph is consumed,
not built here" holds: asserting a brand-new `supports`/`contradicts`/
`extends` relation is an edit to the argument graph itself, not a project-
local invention.

Reuses `render_project_argument_canvas`/`write_project_argument_canvas`
directly for the `.canvas` emission — same node/edge shape, driven by the
computed connections above rather than the full argument component, gated by
whether emitting a spatial projection is worthwhile at all (the ~20-node line;
node-link readability collapses above it — the list/outline is primary
either way, this line only decides whether a `.canvas` is *also* worth
generating).

Builds the `outline.md` on-disk grammar: an ordered Markdown list, one member
per line (`- {note-id} — {reasoning}`), `{note-id}` is the member's stable
ULID (never a path or title), `{reasoning}` optional free text, order is the
list's own line order (no separate numeric field). Membership is proposed via
**BM25** candidate retrieval — never graph traversal, since typed edges are
the argument layer, not the retrieval representation; dense/vector retrieval
stays out of scope for alpha.17, behind alpha.16's own unresolved retrieval-
benchmark gate (see design §1.2's correction). Adds `read_slice`/
`read_outline` to `engine/api.py`, additive to `engine-read-api.v1`.

Test: `tests/test_alpha17_slice_outline.py` + extend `tests/test_knowledge.py`.

### PR-C — Draft composition + the minimal compose surface

Depends on {A1, A2, B}. This is where the writing spine and the host-neutral
surface land together — deliberately, since design §8's acceptance runs "via
files + the read-API + CLI," and a compose surface built without the CLI verbs
would leave that acceptance unmet. Builds the coherent-slice builder per

(citekeys, IDs, tags — closed-vocabulary slots only) as a strictly second
pass, never at the point of semantic commitment. Each drafted claim attaches
an A1/A2 evidence set.

existing `run_operation`/`write_new_concept` shape) and new CLI verbs — either

### PR-D — Text-output verification + export-readiness gate

Depends on C. Extends `project_export_readiness` (currently: `paper_plan`
required fields + thesis + ≥1 checked support) with two new checks: any
evidence set at `state: evidence-incomplete`, and any `review_required: true`
set the PI has not yet disposed — both must clear (or be explicitly disposed)
before the gate reports `ready`. Builds the deterministic citation/number
checks (each item's `work_id` resolves to a catalog record — citekey/DOI only
rendered at export time, the sole point a citekey ever appears; verbatim span
present in source; every engine-computed deterministic number recomputes and
matches — proving span existence and arithmetic, never grounding); the
provenance-label vocabulary (`human`/`machine`/`mc` or span-matched — labeled
citation-correct, not grounded/`contradicted`/`unchecked`/`tainted` — monotone,
never clears/`stale`; no label ever means "true"); structural graph-reference
checks; sparse/triaged PI routing with a max-items-per-session load cap as a
first-class constraint, not a throughput target. Export **runs** on a clean
draft — stripping `%%ev%%`/`%%prov%%` markers and rendering each resolvable
item as a formatted citation — and **refuses**, with an explained refusal,
when unresolved evidence-incomplete or un-disposed review_required claims, or
broken structural references, remain.

**Resolves the CLI export collision — with a target-specific gate, not a
shared monolithic one.** Three surfaces exist (§2): the generic top-level
`export`, `work export`, and `project export`. Extend `project export` with a
`--draft` disambiguator so one command exports either the existing paper-plan
target or the newly-composed draft. Add no fourth verb. Leave the top-level
`export` and `work export` untouched — they serve a different, generic
purpose and are not project-scoped readiness gates.

**Correction, 2026-07-06 (see decisions.md):** an earlier draft of this plan
said the draft target routes through "the same single readiness gate" as the
paper-plan target, meaning the existing `project_export_readiness` unmodified.
Verified directly: that function's paper-plan branch requires
`PAPER_PLAN_REQUIRED_FIELDS` — `target`, `audience`, `research_question`,
`central_contribution`, `gap_statement`, `claim_evidence_map`, `figure_plan`,
and more — none of which describe whether a *draft* is clean. Routing the
draft target through the identical check unmodified would refuse a
structurally-clean draft for missing paper-plan fields that have nothing to do
with it. Fix: `project_export_readiness` (or a target-aware sibling) takes an
explicit `target: "paper_plan" | "draft"` and checks only the requirements
that target actually has — paper-plan keeps its existing
`PAPER_PLAN_REQUIRED_FIELDS` + thesis/support checks unchanged; the draft
branch checks only this slice's own evidence-set `state`/`review_required`
and structural-reference requirements. One CLI verb, one shared `{ready,
status, missing}` return shape, two independent requirement sets — not one
requirement set stretched over two artifacts.

Test: `tests/test_alpha17_verification.py` + extend `tests/test_cli.py` with
an explicit refusal-transcript case.

### PR-F — Exploration / anomaly channel

Genuinely new (§2: `retrieval_substrate.py` has only FTS5 fixtures; no
diversity/coverage/DPP code exists anywhere). Independent of every other
slice — needs only the existing read-API and `work_graph_edges`, so it can
start immediately and land in any order relative to A–E. Builds a
coverage/diversity query over the citation graph — computed *separately* from
the relevance ranker — with a recommended default of MMR as the cheap
baseline, submodular facility-location coverage behind a fixture gate (only if
the fixture shows it's worth the complexity), DPP deferred until a fixture
justifies it. Extends `integrity.py`'s existing NLI REFUTES detection into a
standing "contrary evidence" lane. Each surfaced item carries a per-item *why*
(the "consider the contrary" framing). Measured as its own objective, never a
synthetic score. Acceptance is **fixture-backed**: a seed corpus constructed
with at least one genuine coverage/contrary candidate must surface it; a real
corpus with no genuine candidate is expected to show an honest empty channel —
that is a defined, correct outcome, never something to fabricate around.

## 6. Idempotence and recovery

- **Safe to re-run:** each slice is a small PR off `origin/main` in its own
  worktree; re-running a slice's tests is always safe. The `evidence_sets`
  schema bump (A2) is itself idempotent — `CREATE TABLE IF NOT EXISTS` plus a
  `PRAGMA user_version` bump, run unconditionally on every `_init`, no
  migration function or guard needed (§0, §2). Because `evidence_sets` is
  **derived** state (A1's markers are canonical), recovery from a partial or
  corrupted schema-init run is always: re-run `rebuild` against the markers
  — never a lossy path, since the markers themselves were never touched by
  the schema change.
- **Rollback:** `git worktree remove ~/memoria-vault/worktrees/alpha17-<slug>`;
  revert the slice's commit. For the schema, drop `evidence_sets` and re-run
  `rebuild` from the markers.
