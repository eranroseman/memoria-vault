# Host/stack first-principles evaluation

Date: 2026-07-06. **Status: evaluation, not yet a decision.** This document
produces the analysis the owner asked for ("stand up the host/stack
first-principles evaluation"). If a recommendation here is ratified, it becomes
a dated Y-statement in `scratch/releases/0.1.0-alpha.17/decisions.md` per
AGENTS.md *Decision records* — nothing below is ratified by being written.

**Method:** requirements/constraints from the actual corpus → prior art
(cited, external research) → clean-slate recommendation → trade-offs and flip
conditions → correction to the alpha.17 design where this evaluation changes it.

---

## 0. A correction owed first: the "host/stack" framing in the alpha.17 design was imprecise

[0.1.0-alpha.17-design.md](0.1.0-alpha.17-design.md) §6 and the earlier
17/18-merge discussion both said "all deferred items trace to one root: the
host/stack decision is unmade" and specifically that code execution "runs in
the host's process model." **That coupling is wrong, and this evaluation
corrects it before proceeding:**

- Code execution has **two** real dependencies: an **OS/platform axis** (the
  sandbox backend genuinely differs — bubblewrap/Seatbelt/WSL2 — this is real
  and unrelated to what follows) and a **client-UI-tech axis**. The client-tech
  axis is **spurious**. Code never runs in the rendering client's process
  regardless of which client renders the UI, or whether execution is
  internally sandboxed (beta.1/alpha.17's design) or externally delegated
  (alpha.15's ADR-125 stance, which alpha.17 has not resolved against beta.1's
  reintroduced internal sandbox — that divergence is a separate open question,
  not settled by this doc and not leaned on here). Either way, the client only
  **enqueues an operation request** and later reads the result via the
  read-API/view-spec — exactly like every other operation. **Code execution is
  independent of the client/adapter decision below.**
- What genuinely *is* host-dependent is narrower than "everything code and
  workspace": (a) **which client/adapter technology** renders the UI, and
  (b) the **workspace/gate topology** (the information architecture a
  rendered client would present) — and (b) is largely independent of (a) too
  (see §2).

This narrows §6's deferred bundle to: **client/adapter choice + workspace
topology design**, gated on each other only loosely, and **not** on code
execution. §5 below restates the corrected dependency graph and the concrete
edit to make in the alpha.17 design.

## 1. What's already settled — out of scope here unless new evidence overturns it

The owner's original concern — "Obsidian/SQLite chosen as defaults without
evaluating fit" — is real and is exactly why this evaluation exists. But not
everything under "host/stack" is actually unevaluated. alpha.15 already ran a
first-principles pass on the **engine-side architecture** (six consolidation
decisions, cited against Jacobs 2021/Feng & Liu/AutoResearchClaw/InjecAgent/
AgentDojo — the historic chapter names sources, not just assertion) and it is
**implemented today**, verified directly against the live source tree:

- **Standalone engine, no framework/daemon dependency.** `uv tool install
  memoria`; `memoria serve` starts transports on demand; no resident process
  required ([15-alpha.15.md](../../../main/design-history/15-alpha.15.md) §2,
  §8).
- **Engine/api is the sole logic layer; every UI is a client.** Verified in
  `src/memoria_vault/engine/api.py` — `VIEW_SPEC_VERSION = "view-spec.v1"`,
  `READ_API_VERSION = "engine-read-api.v1"` are real, shipped constants, not
  aspirational. Three transport skins exist today: CLI (`--json`),
  `src/memoria_vault/runtime/mcp_transport.py` (FastMCP stdio), and
  `src/memoria_vault/runtime/http_transport.py` (loopback HTTP, token-authed).
- **Declarative, fixed-vocabulary view-specs** (table/card). Agent-generated
  components and shared mutable agent↔UI state are **rejected** design history
  (design-history §8, "framework-hosted UX (CopilotKit-class) rejected …
  inverts the boundary via shared mutable agent↔UI state").
- **All writes are enqueue-only** from any client — a UI "never writes
  concepts, verdicts, frontmatter, or commits" (design-history §8, carried from
  the older ADR-121 rule).
- **Per-client/per-agent read scoping** is enforced in `engine/api` before any
  transport sees a payload (design-history §8).
- **SQLite as source-of-truth for records/operations/verdicts** was itself a
  **re-decision on the merits** in alpha.15, explicitly *overturning* an older
  "SQLite: never the source of truth" stance (design-history §2) — this is the
  **counter-example** to "chosen without evaluating fit," not an instance of
  it. Not reopened here.
- **Obsidian's role today is "optional plain-Markdown editor," not a plugin
  host.** Verified directly in the live docs: `docs/explanation/obsidian/
  README.md` — *"Obsidian is optional in alpha.16. The required human surface
  is the `memoria` CLI plus Markdown files."* `docs/design/design-system.md` —
  *"Alpha.16 does not ship CSS snippets or editor-plugin link styling; optional
  editor adapters may render the same design tokens later, but they are not
  part of the baseline."* `docs/explanation/obsidian/home.md` — *"A
  plugin-rendered start page would be opaque to the CLI/runtime checks and
  would make Obsidian a required product surface."* The heavy Hermes/Obsidian-
  plugin/Kanban-board stack (7-layer architecture, 12 vendored community
  plugins, an Obsidian-MCP policy plugin) that the owner's own notes flagged as
  "adopted without evaluating fit" **was already torn out** in alpha.15, under
  an explicit negative gate: *"alpha.15 cannot close with Obsidian plugin
  implementation counted as release scope"* (design-history §0, §8) — and
  future editor adapters were explicitly required to clear their **own**
  evaluation (design-history §8, §11: *"future editor adapters now require
  their own ADR rather than inheriting an alpha.15 commitment"*). **This
  evaluation is that owed adapter evaluation.**

**Owner flag (not a blocking question — Auto Mode):** the above is treated as
settled-on-the-merits per alpha.15's cited literature pass and verified against
the live implementation. Say so if any of it should be reopened; nothing below
assumes it can't be.

## 2. The two decisions that are genuinely open

### 2a. Client/adapter: which technology renders `view-spec.v1`?

Given the engine emits a **fixed** declarative vocabulary (table/card) over
**existing** transports (CLI/MCP/loopback HTTP), the discriminating test for
any candidate is:

> **Does this candidate force a change to an engine-side contract, or does it
> only render the existing vocabulary over an existing transport?**

A candidate that fails this test (forces the engine to grow a new transport,
a new write path, or a richer component vocabulary just to be renderable) is
reopening the settled layer, not adapting to it — treat that as a strong
signal against the candidate, not a reason to change the engine.

**A framing correction, made after owner feedback on an earlier draft of this
evaluation:** "thin" (as in "a thin Obsidian plugin") means **logically**
thin — no business logic, no independent write path, enqueue-only, all state
and verdicts stay in the engine. It does **not** mean UI-thin. A logically-thin
client can still render the full rich multi-pane workspace (§2b) — the two are
independent axes, not in tension. And because any non-Python client (a browser
tab, an Obsidian plugin's JS/TS running in Obsidian's own Electron process) is
a *different process* from the Python engine, it must reach the engine over
the **existing loopback HTTP transport** regardless of which shell hosts it —
so "thin plugin" and "local web app" are **the same architecture** (JS client →
loopback HTTP → Python engine → `view-spec.v1`, enqueue-only), differing only
in which shell the JS runs inside. This is why the discriminating test applies
identically to both, and why choosing one over the other is a shell/workflow
decision, not a re-litigation of the settled transport.

### 2b. Workspace/gate topology: what does a rendered client actually show?

The five UI-framework-independent jobs (Read / Knowledge / Project / Review /
Upkeep) are already committed (beta.1 §3; carried into alpha.17 §3). What is
undesigned is the concrete layout: gate triad, deep-vs-task surfaces beyond the
minimal Project one, left-nav/right-Co-PI, active-assistance placement — this
is an **owner job-to-be-done design task**, not a technology choice
(design-history's own alpha.15 rejection of both the carried `spaces/`/`_nav`
model and beta.1's five-surface model as "authority-only inputs" already
established this must be derived fresh, not inherited).

**(In)dependence of 2a and 2b:** mostly independent — the topology is "what
pages/sections and what's in them," which most candidates in §3 can render.
The one real coupling: a client that **cannot render a spatial view at all**
(a TUI) forecloses part of the topology (the canvas) rather than merely
deferring it — that's a §3 finding that feeds back into §2b, not a reason to
solve them jointly from scratch.

## 3. Client/adapter candidates, evaluated

Each candidate is run through the discriminating test, then scored against
Memoria's actual constraints: keep-test (no required plugin), no-daemon,
cross-platform including WSL2, single-user distribution simplicity, whether it
can render a small spatial canvas view (committed design, alpha.17 §1.2,
emit-only for now), and maintenance burden for a single-owner tool.

### Obsidian plugin (thin — logically, not visually)

**Owner correction applied (2026-07-06):** an earlier draft of this evaluation
read alpha.15's negative gate ("alpha.15 cannot close with Obsidian plugin
implementation counted as release scope") as a rejection of Obsidian as a
future host, and scored the candidate against the pains of the **thick**
Hermes-era stack (agent runtime, Kanban board, and enforcement all living
inside Obsidian). The owner clarified the actual intent: alpha.15 was a
**sequencing** decision — stand up the standalone engine first (CLI-complete,
not CLI-only), *then* reintroduce Obsidian as the markdown editor of choice
plus a **thin** plugin as the user surface of choice. `design-system.md`
already said as much before this correction: *"optional editor adapters may
render the same design tokens **later**"* — this evaluation is that later
adapter's evaluation, and design-history §11's "future editor adapters require
their own ADR" is exactly the sequencing gate the owner describes, not a
rejection.

**Discriminating test: passes cleanly, once scoped to a thin plugin.**
Obsidian's JS/TS runs in its own Electron process, separate from Memoria's
Python engine — so a plugin **cannot** call `engine/api` in-process; it must
reach the engine the same way a browser tab would, over the existing
token-authed **loopback HTTP transport** (`requestUrl()` reaches localhost
with no CORS restriction on desktop — verified against Obsidian's own
TypeScript API docs). Rendering table/card is straightforward (`ItemView` +
hand-rolled DOM from the fetched JSON). No new engine-side contract, no new
write path — the plugin only ever enqueues, exactly like every other client.

- **The historical pains were the *thick* stack's, and don't recur for a
  logically-thin renderer.** The 12-vendored-community-plugin supply chain,
  the Obsidian-MCP TLS/boundary confusion, the Kanban board's rigidity, and the
  "Bases has no auto-refresh" materialized-cache workaround were all costs of
  the *old* architecture, where the agent runtime, task state, **and
  enforcement** (a separate MCP policy-gate plugin) lived inside Obsidian. None
  of that recurs here: the standalone engine already owns all state,
  enforcement, and verdicts (§1); a thin plugin is a single first-party
  renderer with no Bases dependency (it fetches and renders its own JSON, on
  its own refresh schedule) and no board/lane model to be rigid about.
- **Security, precisely scoped.** Obsidian's own security page is right that
  it "cannot reliably restrict plugins to specific permissions" — but for a
  **first-party** plugin Memoria authors and ships, that's the **same trust
  model as the CLI**: audit Memoria's own code once, same as always. The
  residual that *is* genuinely Obsidian-specific and worth one honest line: any
  *other* community plugin the researcher chooses to install shares the same
  process and privilege, and could in principle read the vault or the loopback
  token. Memoria's actual threat model is **untrusted ingested papers**, not
  untrusted plugins the single researcher deliberately chose to install — so
  this is a real but narrow residual, not a structural regression of the
  engine's own boundary (enqueue-only, read-scoping, credential-free workers
  are unchanged and un-bypassable from the plugin).
- **Native canvas rendering is an asset, not a gap.** The earlier draft flagged
  "no programmatic Canvas API" as a minus. Reframed: Memoria doesn't need to
  *build* canvas UI at all — it emits a `.canvas` (JSON Canvas) file and
  Obsidian's own **already-built, polished native canvas view** renders it for
  free, which is more than the emit-only scope alpha.17 §1.2 currently commits
  to. When (later) the user drags a node, Obsidian writes the position change
  directly to the file — a **foreign edit**, handled by the same keep-test /
  quarantine / workspace-scan path already decided for any out-of-band PI edit
  (design-history §4; the alpha.5-era lesson that "a base-board drag writes
  through Obsidian directly, caught post-hoc" is precisely this path) — fully
  consistent with reversibility via snapshot/revert. **Trade:** Memoria cedes
  fine-grained control of the canvas *interaction* model to Obsidian's native
  renderer in exchange for not building one — a real, honest trade, named in §6.
- **Cross-platform is already solved, not a risk to mitigate.** Obsidian ships
  and maintains macOS/Windows/Linux (and mobile, moot here since a phone can't
  reach a same-machine loopback server) itself — the WSL2 localhost-
  reachability friction that dogs the local-web-app candidate below is a
  non-issue for a desktop app the researcher already runs.
- **"Thin" is a logical property, not a visual one.** A logically-thin plugin
  (no business logic, no independent write path, enqueue-only) can still
  render the **full rich multi-pane workspace** (§2b) — thin-logic and
  rich-UI are independent axes, not in tension.

**Verdict: recommend as the primary client**, per the owner's stated intent
and confirmed on the merits above.

### Local web app (same architecture, different shell)

**Discriminating test: passes cleanly** — architecturally **identical** to the
Obsidian plugin above (JS client → loopback HTTP → Python engine →
`view-spec.v1`, enqueue-only); the only difference is which shell hosts the
JS. `memoria serve` already starts a token-authed loopback HTTP server on
demand — a browser client simply *uses* that transport, adding no new one.
Direct architectural precedent: Jupyter Server (token-authed by default,
"access to the server means access to running arbitrary code" — the same
threat-model reasoning Memoria's Guard/credential-free-worker posture already
uses), Streamlit, Gradio — all "`pip install X` → `X run` → opens
`localhost:PORT`" — the exact shape of `memoria serve`.

- **Canvas rendering:** well-precedented in this category generally
  (Cytoscape.js/`ipycytoscape`, React Flow, D3) — but unlike Obsidian, a web
  app has no native canvas renderer to lean on; it would have to build
  interactive canvas UI from scratch.
- **Cross-platform/WSL2 — a real friction point that doesn't apply to the
  chosen primary.** Streamlit, Gradio, and Jupyter all have **documented, real
  WSL2 localhost-reachability issues** in their own issue trackers. Mitigation
  is known and cheap (`--no-browser` printing the URL), but it's friction a
  desktop Obsidian plugin simply doesn't have.
- **Maintenance:** no third product's plugin API or release cadence to track;
  the UI is Memoria's own code, versioned with Memoria — this remains a genuine
  point in its favor if it's ever built.

**Verdict: not the primary surface, but not rejected either** — it's the same
architecture as the recommended plugin, just a different shell, and remains
available later for headless/remote/non-Obsidian use under the same
per-adapter evaluation rule (design-history §11) that this document just
satisfied for Obsidian.

### VS Code extension

**Discriminating test: passes, with a structural cost.** A webview can render
anything (arbitrary HTML/JS — canvas rendering is trivial, concrete precedent:
`viewX-vscode` uses Cytoscape.js in a webview), and it only *talks* to
Memoria's existing HTTP transport (via the extension host relaying
`postMessage`s) — no new engine-side contract.

**Verdict: reject as primary — structurally excludes non-VS-Code users**, which
directly contradicts "the required human surface is the `memoria` CLI plus
Markdown files" (keep-test). Real friction found: **no CSP by default** on
webviews (an open, acknowledged VS Code issue), and direct webview→localhost
`fetch()` is frequently blocked as CORS in practice — the reliable pattern
requires an extra indirection layer (extension host relays JSON to the
webview). WSL2 adds real topology complexity here too: webviews always render
in the **local** UI process even when the extension's logic half runs
remotely, so a WSL2-hosted Memoria backend needs explicit port-forwarding
(`asExternalUri`), not a clean cross-platform story. **Possible later, optional
adapter** for developer-users who want code-artifact review inside their
editor — not core, and not gating anything.

### Terminal UI (Textual or Bubble Tea)

**Discriminating test: passes cleanly** for table/card (Textual's built-in
`DataTable` is a strong match for the vocabulary; either framework talks to
Memoria over the existing HTTP transport or, for Textual specifically,
in-process since both are Python — no new engine contract either way).

**The decisive finding: neither framework has a spatial-canvas primitive.**
Confirmed directly against both ecosystems — Textual's third-party
`textual-canvas` only does character-cell pixel drawing, not a node-link/free
layout; no comparable Bubble Tea widget exists either. **A TUI cannot render
Memoria's committed small-project-slice canvas view, full stop** — not "later,"
structurally never, short of falling back to exporting/opening an image or
another app (which the design already anticipated as the "emit-only" fallback,
§1.2, but a TUI *client itself* could never grow past that).

Genuine strengths worth weighing anyway: **Textual specifically** is
Python-native (zero cross-language bridge — could call `engine/api` in-process
*or* over HTTP), has a lean ~6-package dependency footprint, and is the
**cheapest, most CLI-aligned option** — a natural incremental step up from
today's "CLI `--json` only" posture rather than a leap to a full browser
GUI. One real asymmetry: Textualize Inc. (the company) wound down in 2025;
the maintainer has committed to continuing solo and calls it "mature and
battle-tested," while Bubble Tea's backing (Charm) remains actively funded —
a maintenance-risk data point, not disqualifying, and moot if Textual is used
(Python-native fits Memoria; Bubble Tea would add a Go toolchain for no
offsetting benefit here).

**Verdict: a legitimate minimal/incremental option, not the primary
recommendation** — because it forecloses the canvas requirement rather than
deferring it. Flagged explicitly in §6 as the "if you want to stay maximally
minimal" alternative.

### Custom desktop app (Tauri or Electron)

**Discriminating test: passes** (both just embed a webview and call the
existing HTTP transport) **but multiple independent 2025–2026 sources
explicitly call this "overkill"** for an app that only renders a fixed
table/card view over local HTTP — the two most-cited comparisons in current
research make that judgment directly, not just this evaluation's inference.

Concretely: Tauri's Rust core would need its documented **sidecar** mechanism
to run the separate Python `memoria` process — a genuine three-language stack
(Rust + JS + Python) for an app whose only job is rendering a browser tab's
worth of content locally. Electron avoids the third language (Node can spawn
Python directly) but bundles a full Chromium (85–150MB+ vs. Tauri's
single-digit-to-low-tens-of-MB, per the Hoppscotch migration case study) and
inherits browser-scale CVEs requiring app rebuilds to patch (two 2025 critical
Electron CVEs found, one on CISA's KEV list). Tauri's Linux story has its own
rough edge (WebKitGTK sometimes missing from distro repos; build-on-oldest-
target-glibc requirement). A plain browser tab against `memoria serve` already
gets ~90% of the rendering benefit with none of this packaging cost.

**Verdict: reject.** Revisit only if a hard requirement for native OS
integration emerges (systray, native file-drag-in, OS-level notifications)
that a browser tab genuinely cannot satisfy — no such requirement exists in
the current corpus.

## 4. Workspace/gate topology — research-grounded input to the owner's design task

This section does **not** attempt to settle the topology unilaterally — per
§2b and the design-history's own repeated finding, that's the owner's
job-to-be-done design, not a technology decision. It reports what the research
says, so that design task starts from evidence rather than a blank page.

- **Job-based navigation is well-precedented and supports the five-jobs
  commitment.** Obsidian's own "Workspaces" feature exists exactly to switch
  full layouts "depending on your task" (its own help docs). Linear states
  directly that Triage is deliberately excluded from normal views "since
  triage is considered to be outside the normal workflow" — direct precedent
  for keeping Review/Upkeep visually and navigationally separate from Project.
- **Deep-work (Project) vs. task-work (Review/Upkeep/Read-queue) separation has
  strong, explicit rationale, not just intuition.** iA Writer's own design
  writing makes the specific, citable claim that writing and formatting draw
  on different cognitive resources ("language processing and motor
  coordination" vs. "spatial processing") and that switching between them
  mid-task is costly, citing Csikszentmihalyi on flow — this is the strongest
  evidence found that *task type*, not merely visual clutter, justifies
  keeping Project distinct. Superhuman's founder writing makes the same case
  for triage specifically: single-item-at-a-time focus with no visible queue
  during triage protects flow. This directly supports beta.1 §3 / alpha.17
  §3's existing mindset-separation design — treat it as reinforced, not open.
- **Review-queue mechanics have concrete precedent to borrow, but the
  "honesty card" itself is not established prior art.** GitHub's per-file
  "Viewed" checkbox (an inbox-zero progress mechanism) and Gerrit's tiered
  `-2..+2` label model (non-authoritative agreement vs. authoritative
  submit-gate vs. absolute block) are concrete, borrowable mechanics for laying
  out the Review job. But the specific mechanism alpha.17 §5 already commits to
  — a proposal showing argument-for + argument-against (self-rebuttal) +
  coarse certainty as one artifact — **was not found as a documented pattern
  anywhere.** The closest relatives are Deliberative-AI (Chiang et al. 2024)
  and Cognitive Forcing Functions (Buçinca et al. 2021) research, which
  studies *adjacent* mechanisms (structured disagreement discussion; forced
  engagement with explanations) but not this exact artifact, and flags a real
  risk: forced engagement with explanations measurably *reduces* usability
  even as it cuts overreliance, with benefits concentrated in high-Need-for-
  Cognition users. **This reinforces, rather than undercuts, why alpha.17 §5's
  seeded-error probe is load-bearing** — the honesty card is a novel,
  unvalidated mechanism, and the probe is the only way to learn whether it
  actually works for this specific reviewer rather than assuming it does
  because it sounds well-motivated.
- **Left-nav / right-Co-PI is a legitimate pattern, but genuinely contested —
  treat as a hypothesis to validate, not a settled convention.** GitHub
  Copilot Chat and Cursor are real, current precedent for right-side AI
  placement, but the research surfaced meaningful counter-signal: Microsoft
  closed a user request to change Copilot Chat's side with **no design
  rationale offered** (an engineering convenience, not a considered choice);
  Cursor's own 2025–2026 product direction is explicitly moving *away* from
  "chat as a side panel" toward agents as first-class workspace objects;
  Notion's flagship AI pattern is **inline**, not a side panel, with reviewers
  calling Notion's *also-present* side-panel chat a source of user confusion,
  not a strength; Microsoft 365 Copilot's new design is an explicit hybrid
  (side pane for sustained work, inline canvas invocation for point tasks,
  under "progressive disclosure"). A synthesis piece surveying seven observed
  AI-placement patterns frames the choice as **role-dependent**, not
  aesthetic: a right-side panel signals "on-demand expert assistance during
  focused work" specifically — which does map reasonably well onto Memoria's
  Co-PI (explicitly read-only, proposal-only, exposing plan/sources/
  assumptions/unknowns/pending-writes before any enqueue, per the standalone
  engine's carried agent contract) — but the research does not let this
  evaluation call it settled. **Recommend keeping left-nav/right-Co-PI as the
  working hypothesis for the Project job specifically** (it fits the
  "sustained focused work + on-demand assistance" role the synthesis piece
  names), **but validate it rather than assume it**, especially for the
  Review/Upkeep task-oriented gates where the sustained-focus rationale is
  weaker and an inline/ambient pattern (closer to Notion's) may fit better.
- **Spatial canvas + linear outline as companion views: the research
  cross-validates the alpha.17 redesign, and flags what the original beta.1
  design got wrong.** Scrivener's binder/corkboard/outliner and Miro's
  card↔table sync are the cleanest precedents found, and both share one
  property: **there is no "authoritative" side** — both views read/write the
  *same* underlying ordered set of items; rearranging in either updates the
  other because it's the same data, not a derivation. Obsidian Canvas's
  file-cards work the same way (a canvas node referencing a note is the *same
  file*, not a copy). This is exactly what alpha.17 §1.2/§1.3 already redid —
  killing the one-way "canvas→outline derivation" beta.1 originally specified
  in favor of **outline and canvas as two views of one project slice** — so
  treat this research as **convergent confirmation of an already-made
  correction**, not new work. Milanote is the useful counter-example: its
  spatial view is canonical and its linear export is a one-way, static
  snapshot — structurally closer to what beta.1's original one-way derivation
  would have produced, and the reason that design was corrected.

## 5. Recommendation

1. **Client/adapter: reintroduce Obsidian as the markdown editor of choice plus
   a logically-thin first-party plugin as the primary rendering surface**,
   confirming the owner's stated sequencing intent on the merits (§3). The
   plugin renders `view-spec.v1` over the existing `memoria serve` loopback
   HTTP transport — no new engine-side contract, no independent write path,
   enqueue-only like every other client. It gets Obsidian's already-maintained
   cross-platform desktop story and its native `.canvas` renderer for free
   (position edits flow through the existing foreign-edit/quarantine path),
   at the honest cost of ceding fine-grained canvas-interaction control to
   Obsidian's own renderer, and of sharing Obsidian's process/privilege with
   whatever other community plugins the researcher chooses to install (a real
   but narrow residual — Memoria's threat model is untrusted ingested papers,
   not user-chosen local plugins).
   - **"Thin" is logical, not visual:** the plugin can render the full rich
     multi-pane workspace (§2b) while holding no business logic and no
     independent write path.
   - **A local web app is the same architecture, not a rejected one** — same
     transport, same `view-spec.v1`, same enqueue-only writes, different shell.
     It remains available later for headless/remote/non-Obsidian use under the
     same per-adapter evaluation rule this document just satisfied for
     Obsidian; its one real advantage over the plugin (no third-party host to
     track) doesn't outweigh Obsidian's native canvas + already-solved
     cross-platform story for the *primary* surface.
   - **A Textual TUI is a legitimate cheaper alternative** if the owner instead
     wants to defer the rendered client further and stay closer to today's
     CLI-`--json`-only posture — flagged explicitly because it forecloses the
     canvas requirement rather than merely deferring it, which is the actual
     trade-off, not a defect to hide.
2. **Workspace/gate topology remains the owner's job-to-be-done design task**
   (§2b), now informed by §4: the five-jobs split and deep-vs-task separation
   are reinforced by real cognitive-cost research (not just intuition); Review
   can borrow concrete queue mechanics (Gerrit's tiered labels, GitHub's
   viewed-checkbox) while treating the honesty-card artifact itself as
   unvalidated (which is exactly why alpha.17 §5 measures it); left-nav/
   right-Co-PI is a defensible starting hypothesis for the Project job
   specifically, not a settled convention to apply uniformly; canvas/outline
   should stay two views of one shared slice (already corrected in alpha.17).

## 6. Trade-offs and flip conditions

- **The plugin costs real build effort** a TUI or "stay CLI-only" would not.
  Flips toward Textual-TUI-first if the owner decides the interactive spatial
  canvas is not worth gating a client on right now — nothing here forecloses
  building a TUI *now* and the plugin *later*, since both are equally valid
  clients over the same fixed transport and neither requires the other.
- **Memoria cedes fine-grained canvas-interaction control to Obsidian's native
  renderer** — the direct cost of getting that renderer for free. Position
  edits become foreign edits handled by the existing quarantine/scan path, not
  a first-class Memoria-authored interaction. Flips toward a custom-built
  canvas (in the web-app alternative) only if Obsidian's native canvas proves
  too limited for what the researcher actually needs from it — unmeasured
  today, so not a reason to build a custom renderer up front.
- **Other community plugins share Obsidian's process and privilege** with
  Memoria's own plugin — a real but narrow residual, since the threat model is
  untrusted ingested papers, not plugins the researcher deliberately chose to
  install. Flips to a concern worth acting on only if a specific credible
  local-plugin threat is identified; nothing in the current corpus names one.
- **Deferring the local-web-app alternative has low cost, since it isn't
  foreclosed** — the identical transport and `view-spec.v1` contract mean it
  can be built later for headless/remote/non-Obsidian use without redoing any
  engine-side work, under the same per-adapter evaluation rule this document
  just satisfied for Obsidian (alpha.15's standing negative gate is reaffirmed
  here, not reopened).
- **Left-nav/right-Co-PI may need to flip to an inline/ambient pattern**
  specifically for Review/Upkeep if validation (once a client exists) shows
  users don't reach for a side panel during quick triage — the research
  already names this as the weaker fit; build the client so this is a UI-layer
  change, not an engine one (consistent with §1's decoupling).
