# Memoria 0.1.0-alpha.18 — data-structure normalization

Date: 2026-07-07
Status: alpha.18 design — the schema/frontmatter/layout normalization recommended
by `scratch/memoria-data-structure/data-structure-analysis.md`, on the released
alpha.17 engine.

> **What this release is.** alpha.18 is a **normalization** release: it fixes the
> real name collisions, redundant SSOTs, and dead schema surface in the *existing*
> schema, and lays the on-disk layout the query architecture will later build on.
> It changes the *shape and names* of what exists; it does **not** build the query
> engine or the new derived query-substrate tables that engine reads. Every change
> traces to a specific finding in `data-structure-analysis.md` (cited by Part /
> G-number / collision-id), never to ADR-authority.

**Design authority for this release:**
`scratch/memoria-data-structure/data-structure-analysis.md` (the analysis) and its
companion `query-mechanism-analysis.md` (the query design the deferred half serves).
When this design and the analysis disagree, the analysis wins; fix this design.

---

## 0. Scope — the IN / DEFER cut line (the load-bearing decision)

The analysis has two halves, kept in two documents by the project owner's own
direction: **how to structure the data** (this document's subject) and **how to
query it** (`query-mechanism-analysis.md`). alpha.18 implements the first half in
full and **defers the second half — and the new derived tables that exist only to
serve it — to a named follow-on release** (`query-architecture`, below).

**IN — the normalization (all of it ships in alpha.18):**

1. **OpenAlex collision renames** — the six-collision audit (analysis Part 8, C1–C6):
   `source_id`→`work_id`, `source_type`→`item_type`, `journal_events`→`event_log`
   (+`journal_id`→`event_id`), `concept_type='source'`→`'work'`,
   `relation_type='source'`→`'published_in'`; C4 (`Concept` vs OpenAlex's deprecated
   `Concept`) stays a documented permanent collision, glossary-mitigated only.
2. **Delete-and-fold** (analysis Parts 2/4/5, G0-continued; owner-ratified, beta.1
   design item 20): delete the markdown `work` Concept type + `work.yaml`; delete
   `source-note` + `source-note.yaml`, folding it into `note` as `mode: work`; drop
   `person`/`organization`/`venue` from `concept_type` entirely (the free-join
   entity-resolution decision, Part 4).
3. **Bundle-root layout** (analysis Parts 4/6/7): the five bundle roots become
   `notes`/`hubs`/`projects`/`digests`/`fulltext` (dropping `works`/`sources`); a
   new `fulltext` **frontmatter** `type:` (a `type:` conformance label with its own
   `fulltext.yaml` validation schema — **not** a `concepts.concept_type` CHECK value,
   and no `concepts` DB row, exactly like `system`/`dashboard`/`eval-task`);
   `templates/`, `eval/`, `patterns/_preamble.md`, and `journal/` move into
   `.memoria/`; `type:` conformance labels on the loose root/`system` docs.
4. **Frontmatter/schema** (analysis Part 5): the note `mode` collapse to
   `[claim, question, definition, work]` (with `certainty: hypothesized`); a
   `required_when` schema primitive; `todo: list` on the four surviving authored
   types; the G0 survival-copy field deletions (`citations`, and `work.yaml`'s
   `csl_json`/`identifiers`/`source_sha256` — moot once `work.yaml` is deleted); the
   G4 `evidence_set` frontmatter-field deletion (marker+table stays); the `item_type`
   enum; the `topics` reconciliation; `id: str` (= `work_id`) on `digest`/`fulltext`.
5. **`analyze_gaps` free-join** (analysis Part 4): traverse `work_graph_edges` by raw
   `target_id` for `authorship`/`institution`/`published_in`, delivering cross-paper
   recognition and gap-discovery over the OpenAlex-enriched corpus with no new table.
6. **Schema-documentation drift prevention** (analysis Part 8): a lint/CI check that
   cross-validates every field/enum claim in `docs/reference/frontmatter.md` and
   `document-types.md` against the live schema YAML — the mechanism that would have
   caught the `source_type`-enum and `creation.form` drift the analysis found.
7. **Migration plumbing** (analysis Part 8 §migration, G3): delete
   `_migrate_v4_to_v5`; edit `schema.sql` directly to its final shape; bump
   `user_version`/`SCHEMA_VERSION` 6→7. **No `_migrate_v6_to_v7`.**

**DEFER — the query substrate + engine (named follow-on: `query-architecture`):**

The new derived tables `passages`, `passages_fts` (FTS5), `passages_vec` (sqlite-vec
`vec0`), `file_index_state`, and `concept_edges`, together with their populate /
reindex / embed machinery, the Q5 `check_status` cascade into `passages`, the
`^pNNNN`→structured-page-metadata source-span change, and the retrieval engine
itself (RRF fusion, filter-before-rank, dual-level local+global expansion,
reranking, grounded synthesis, gap-visibility/juxtaposition queries) — all defer
**together**, as one coherent release, for three reasons:

- **They are caches and an engine with no consumer until that engine exists.**
  Building an embedding pipeline (`vec0`) or a derived-passage sync loop before any
  reader queries it is exactly the speculative machinery the project's
  simplest-design guardrail rejects.
- **The analysis itself keeps them separate** — `query-mechanism-analysis.md` is its
  own document with its own six-phase migration; Part 8's tables are "schema deltas
  *for the query architecture*," not standalone data structure.
- **The seam is clean.** alpha.18 leaves the *authored* schema correct and normalized;
  `query-architecture` adds *derived* tables on top of correct names, in one schema
  bump, with the engine that reads them — no half-built cache stranded between
  releases.

**One boundary case, decided explicitly rather than left to the executor —
tension recordability.** `concept_edges` (which carries both the `links:` mirror and
PI-confirmed `tension` edges) is a query-substrate table and defers with the rest.
Its consequence — a PI-confirmed tension has no persisted home until
`query-architecture` — is **accepted for alpha.18**, because (a) the tension *value*
in `concept_edges` is inseparable from the table's own derived `links:`-mirror half
and the reindex loop that maintains it, and splitting one CHECK-value out into its
own early release would strand exactly the kind of one-off machinery this cut avoids;
and (b) tension recordability is a *beta.1* foundation, and both alpha.18 and
`query-architecture` land before beta.1, so nothing downstream is delayed by keeping
them together. If the owner wants tension recordability sooner, the single-slice
alternative is named in §7 as the one pull-forward candidate.

## 1. The normalization, by group

Each change below is stated as a concrete schema/code/frontmatter edit. Naming is
given in *target* form; the exact old→new mapping and the confirmed companion
code sites live in analysis Part 8 and are not re-derived here.

### 1.1 Renames with no CHECK rebuild (C1, C2, C6)

Plain column/table renames — SQLite `ALTER TABLE … RENAME COLUMN`/`RENAME TO`, or,
per §2, direct edits to `schema.sql`'s final shape (nothing is deployed to migrate):

- `catalog_sources.source_id` (PK) → `work_id`; `enrichment_runs.source_id` →
  `work_id`; `field_provenance.source_id` → `work_id`; `work_aspects.source_id` →
  `work_id`; `note.yaml`'s optional `source_id: str` → `work_id: str`. These name the
  same identifier as `work_graph_edges.work_id` (confirmed: `integrity.py:685`
  `s.source_id = e.work_id`; `state.py:1004` `stable_work_id = _source_id(work_id)`).
- `source_type` → `item_type` on the frontmatter surface, gaining the enum block
  `[paper, dataset, repository, web-page, report]` — closing both the never-shipped
  alpha.10 enum drift **and** the OpenAlex `Source.type` collision, and matching the
  DB column `catalog_sources.item_type` live code already reads
  (`integrity.py:2069-2074`). The DB column keeps its unconstrained CSL-passthrough
  vocabulary; the frontmatter enum is the coarser PI-facing one — same name,
  deliberately different value domains (analysis Part 8).
- `journal_events` → `event_log`; `enrichment_runs.journal_id` → `event_id`. The
  ledger is an append-only audit log, not a bibliographic journal; `journal_id` is
  written nowhere today (confirmed — carries only its `DEFAULT ''`), so zero
  populated-value risk. Its two triggers are dropped and recreated on `event_log`.

### 1.2 Concept-type normalization + delete-and-fold (C3, G0-continued)

`concepts.concept_type`'s CHECK list goes from 15 values to **10**:
`work`, `digest`, `note`, `hub`, `project`, `capability`, `operation`, `skill`,
`adapter`, `workflow`.

- `concept_type='source'` (the catalog row) → `'work'`. Its only writer is
  `state.py:794`'s literal `_upsert_concept_mirror_conn(conn, concept_id, "source",
  "db")` — DB-store-only, no authored vault content, so the rename is a code+CHECK
  edit with zero content cost.
- The markdown `work` Concept type and `work.yaml` are **deleted** (G0 — its
  remaining surface is a bare identity wrapper `work_id` already covers; confirmed no
  code path requires `works/<work_id>/record.md`). This frees the word `work` for the
  catalog row above.
- `source-note` and `source-note.yaml` are **deleted**, folded into `note` as
  `mode: work` (analysis Part 5). Its fields move onto `note.yaml` (§1.4).
- `person`/`organization`/`venue` are **dropped** from `concept_type` entirely — zero
  writers ever (the "declared but dead" finding, G2); the free-join in §1.5 delivers
  their real requirement without a dedicated type. `integrity.py`'s
  `catalog/entities/{owner_type}-{owner_id}.md` path builder and its
  `{authorship:person, institution:organization, source:venue}` rosetta dict are
  retired as dead scaffolding.
- `work_graph_edges.relation_type='source'` → `'published_in'` (already means "the
  venue this work was published in," per `integrity.py:675-687`; renamed so the
  correct meaning stops sharing a spelling with the document-sense `source`).
  Companion: `integrity.py:675-679` dict key and `:687` IN-list.

`concepts` and `work_graph_edges` each carry a CHECK constraint, so their edits are
`CREATE TABLE`-shape changes (§2), not `ALTER … ADD CHECK` (SQLite has none).

### 1.3 Bundle-root layout (Parts 4/6/7)

- **`BUNDLE_ROOTS`** becomes `("notes", "hubs", "projects", "digests", "fulltext")`
  — dropping `works`/`sources`. Four code sites move in lockstep (confirmed by grep,
  not assumed): `projections.py:13`'s `BUNDLE_ROOTS`; `.memoria/schemas/folders.yaml`
  (`bundle_roots:`/`categories:`, kept identical to `projections.py`);
  `search_index.py`'s `_bundle_roots()` **own separate hardcoded set** (updating
  `folders.yaml` alone does not reach it); `operations.py`'s digest path
  `f"works/{source_id}/digest.md"` → `f"digests/{work_id}.md"`. Not exhaustive — a
  full `"works/"`/`"sources/"` string-literal sweep is a code task, these four are
  the verified starting points.
- **New `fulltext` frontmatter type — NOT a DB `concept_type`.** `fulltext/<work_id>.md`
  is an OKF-conformant reproduction (analysis Part 1 §3.1 correction) carrying
  `type: fulltext`, `id: str` (= `work_id`, not a ULID), `title`/`tags`/`links`; body
  stays the `## Page N` headings extraction already produces. It gets a
  `.memoria/schemas/types/fulltext.yaml` schema for frontmatter validation, but
  `fulltext` is **not** added to `concepts.concept_type`'s CHECK (which stays at the
  ten values in §1.2) and a `fulltext` file gets **no** `concepts` DB row — the
  analysis decouples OKF concept-document-hood from Memoria's internal Concept-hood,
  and `fulltext` is grouped with `system`/`template`/`eval-task`/`dashboard` on the
  frontmatter-label-only side of that line, not with the DB-tracked ten. (Contrast
  `digest`, which *is* one of the ten DB-tracked values — the two are deliberately
  asymmetric.) Adding `fulltext` to the CHECK would either force a spurious DB row or
  reject valid `fulltext` docs.
- **Move into `.memoria/`**: `templates/`, `eval/`, `patterns/_preamble.md`,
  `journal/` (analysis Parts 6/7). This touches the sites already inventoried in
  beta.1 design item 17 (`linter/detectors.py` SCAFFOLD_PREFIXES, policy `deny_write`
  lists, `.pre-commit-config.yaml`, 4 test files, 2 docs, 1 skill,
  `refresh-test-vault.sh`).
- **`type:` conformance labels** on the loose export-unit docs (analysis Part 1
  §3.1, item 7): `type: system` on root/`vocabulary`/`_preamble` docs, `type:
  dashboard` on `system/dashboards/*`, `eval-task` already present on eval fixtures.
  These are OKF-conformance labels only, **not** added to `concept_type`'s CHECK.
  `system/templates/{note,hub,project}.md` keep their real `type:` (it's the value
  the template copies onto new files) — left exactly as-is.

### 1.4 note.yaml and the surviving frontmatter schemas (Part 5)

- **`mode` enum** → `[claim, question, definition, work]` (the mode-collapse:
  `hypothesis` is `mode: claim` + `certainty: hypothesized`; `tension` is
  `mode: claim` + `links: {contradicts: [...]}` — neither is a separate mode).
- **`certainty`** gains `hypothesized`.
- **`required_when`** — a new `schema.py` field primitive: a field required
  conditionally on another field's value (`claim`→`claim_text`, `question`→
  `question_status`, `work`→`work_id`), decidable from frontmatter alone (the
  existing `required_any` is disjunctive-across-fields and doesn't fit).
- **Fold source-note's fields onto `note.yaml`**, relevant only when `mode: work`:
  `work_id` (required-when), `description`, `item_type` (the enum, §1.1);
  reconcile `source-note.topic` (singular/unchecked) into `note.topics`
  (plural/checked). **Not folded — dropped, per Part 5's own field table:** `citekey`
  (duplicates `bibliography.bib`, G0) and `project` (duplicates a project's own
  backlinks). `item_type` is **kept** — it is a PI *override input* to
  `catalog_sources.item_type` (file→DB, `state.py:691`), not a G0 survival copy.
- **`todo: list`** on `note`/`hub`/`project`/`digest` (the four surviving authored
  types; `work.yaml`/`source-note.yaml` no longer exist).
- **G0 field deletions** — `citations` from `note`/`digest`/`hub`; the export-
  completeness check (every catalog source referenced by a `checked` Concept present
  in `bibliography.bib`) replaces the per-file `check_citation_survival`/
  `check_citation_payload`.
- **G4 field deletion** — `evidence_set` frontmatter field from note/digest (the
  `%%ev%%` marker + `evidence_sets` table pair is the durable source; the field was
  the third, redundant copy). *This aligns with, and does not disturb, beta.1's
  evidence-set contract, which already uses marker+table, not the frontmatter field.*
- **`digest`/`fulltext`** carry `id: str` (= `work_id`), not `id: ulid` — a named
  exception to the universal core, since both are machine-generated and identified by
  the work they reproduce.

### 1.5 `analyze_gaps` free-join (Part 4)

Extend `analyze_gaps` (`knowledge.py`) to traverse `work_graph_edges` by raw
`target_id` for `relation_type IN ('authorship','institution','published_in')`,
alongside the existing `references`/`related` queries — delivering cross-paper
author/institution/venue recognition and gap-discovery over the entire
OpenAlex-enriched corpus with zero new schema, since `target_id` already holds the
resolved OpenAlex ID. The fuller `catalog_entities` table stays specified-but-unbuilt
(Part 4) until a real dedup or entity-browsing need appears.

### 1.6 Schema-documentation drift prevention (Part 8)

Build the cheaper of the analysis's two options: a **lint/CI check** that
cross-validates any field/enum claim in `docs/reference/frontmatter.md` and
`document-types.md` against the live `.memoria/schemas/types/*.yaml`. This catches
the recurring drift class the analysis found twice (alpha.10's `source_type` enum and
`creation.form` block, both documented-as-landed, neither shipped). Generating the
docs *from* the schema is the richer alternative, deferred unless the lint proves
insufficient.

## 2. Migration policy — delete-and-reinstall, no migration function

Consistent with the analysis's G3 correction **and** the project's own unbroken
precedent (no migration path since alpha.11; alpha.14 *deleted* a migration shim;
alpha.17's exec-plan restates it): there is nothing deployed to migrate — a repo-wide
grep this session found **zero** files anywhere using `type: work` or
`type: source-note`. So:

- **Delete `_migrate_v4_to_v5`** (`state.py:1245`) — dead code protecting a v4/v5
  database that has never existed outside a disposable test fixture (G3).
- **Edit `schema.sql` directly** to its final normalized shape — the renamed CHECK
  lists, columns, and table names land as edits to the `CREATE TABLE`/`CREATE
  TRIGGER` statements, not as an `INSERT…SELECT`/`CASE` rebuild (there are no rows to
  remap).
- **Bump** `schema.sql`'s `PRAGMA user_version` 6→7 and `state.py`'s `SCHEMA_VERSION`
  6→7 as a version label; simplify `_init`'s tolerance set
  (`state.py:1237`, today `{0, 5, 6}`).
- **No `_migrate_v6_to_v7`.** Per G3's own "what to add now, cheaply": write the rule
  (a schema change to a table with *real installed rows* ships as a numbered `ALTER`
  migration) so the ladder is built correctly when the first real vault exists — not
  before.

The only "migration path" for a local sandbox/dev vault is: delete it and
reinstall/rebuild fresh (`scripts/sandbox/refresh-test-vault.sh`).

## 3. Deferred, and why it is safe to defer (the `query-architecture` seam)

The follow-on release, named here so alpha.18's cut is legible, not to design it:
`passages`/`passages_fts`/`passages_vec`/`file_index_state`/`concept_edges` + their
sync loop; the Q5 `check_status` same-transaction cascade into `passages` (a release
blocker *for that release*, per the query-doc review — it ships in the same phase as
`passages` itself, never after); the source-span page-anchor change (the page number
`capture.py::_extract_pdf_pages` already computes flows into `passages.anchor` as
structured metadata — no inline `^pNNNN` written, retiring `_source_span_pages`'s
raw scan and `knowledge.py`'s fake `#^p0001` placeholder); and the retrieval engine
(RRF, filter-before-rank, dual-level expansion, reranking, synthesis, gap-visibility
and juxtaposition primitives) per `query-mechanism-analysis.md`'s six-phase plan.

**Why deferring is safe.** alpha.18 leaves every *authored* name and shape correct.
The deferred tables are all *derived* from that authored layer — rebuildable at any
time, disposable per the project's own `.memoria/memoria.sqlite` rule. Nothing in
alpha.18 has to be revisited when they land; they attach to correct names in one
clean schema bump alongside the engine that reads them. The one accepted consequence
(tension has no persisted home until then) is bounded — see §0's boundary case.

## 4. Trade-offs

- **A large mechanical surface, landed at once.** The `source_id`→`work_id` rename
  alone touches ~67 confirmed sites; the bundle-root rename touches four
  path-owning sites plus an unbounded string-literal tail; folding `source-note`
  rewrites every `type: source-note` file (of which there are currently zero). This
  is real churn, mitigated by the zero-deployment status (no data to preserve) and by
  the drift-prevention lint (§1.6) catching doc/code divergence mechanically. The
  alternative — phasing the renames across several releases with aliases — would cost
  more (compat shims the project explicitly refuses pre-1.0) for no benefit at this
  stage.
- **Normalization without the payoff, this release.** alpha.18 fixes names and shapes
  whose *value* (queryability) only materializes in `query-architecture`. The payoff
  is deferred; the prerequisite is not. This is the honest sequencing — a query engine
  built on colliding names would have to be renamed anyway.
- **`item_type` name/value asymmetry** — the frontmatter enum and the DB column share
  a name but hold different vocabularies (coarse PI-facing vs CSL passthrough). Stated
  so a reader doesn't assume same-name-means-same-values.
- **C4 (`Concept` collision) is documented, not eliminated** — glossary-mitigated
  only; a permanent, accepted lexical collision (analysis Part 8 Open Questions).
- **Tension recordability waits for `query-architecture`** — §0's boundary case; the
  one pull-forward candidate is named in §7.

## 5. Acceptance — the definition of done

alpha.18 is accepted when, on a fresh disposable vault built from the edited schema:

1. **Names normalized.** No `source_id`/`source_type`/`journal_events`/
   `concept_type='source'`/`relation_type='source'`/markdown-`work`/`source-note`/
   `person`/`organization`/`venue` remain anywhere in schema, code, or frontmatter
   schemas — verified by grep against the diff.
2. **Delete-and-fold complete.** `work.yaml`/`source-note.yaml` are gone; `note.yaml`
   carries `mode: work` and the folded fields; `concept_type` CHECK has exactly the
   10 values; the bundle roots are `notes`/`hubs`/`projects`/`digests`/`fulltext`.
3. **Frontmatter validates.** The `mode` collapse, `certainty: hypothesized`,
   `required_when`, `todo`, `item_type` enum, and `digest`/`fulltext` `id: str` all
   pass `schema.py::validate_frontmatter`; G0/G4 fields are rejected.
4. **Free-join works.** `analyze_gaps` surfaces a shared-author/institution/venue gap
   over an OpenAlex-enriched fixture with no new table.
5. **Drift lint fires.** The schema-doc cross-validation check fails on a seeded
   field/enum mismatch and passes on the aligned docs.
6. **Migration policy honored.** `_migrate_v4_to_v5` is deleted, no `_migrate_v6_to_v7`
   exists, `user_version`/`SCHEMA_VERSION` are 7, `_init` builds a fresh vault clean.
7. **Docs aligned** (§6) and **release cut** (§6).

**Negative scope gate:** no slice adds `passages`/`passages_fts`/`passages_vec`/
`file_index_state`/`concept_edges`, an embedding/reindex loop, a retrieval-fusion/
rerank/synthesis path, or a `_migrate_v6_to_v7` — those are `query-architecture`.

## 6. Docs-review and release (explicit, per the plan's requirement)

**Docs-review** is its own landing unit (the alpha.15/17 precedent — docs is a named
slice, not folded silently into code PRs):

- **Update** the ~18 doc pages that reference renamed names (grep-confirmed),
  led by `docs/reference/frontmatter.md`, `document-types.md`, `glossary.md`,
  `on-disk-layout.md`, `ingest.md`, `prompt-operations.md`, and the
  `explanation/knowledge/*` set — every `source_id`/`source-note`/`journal_events`/
  `works/`/`sources/` reference to its normalized form.
- **Create** a `fulltext` document-type reference entry and a short migration/rename
  note (`docs/reference/` or a design-history page) recording the old→new mapping.
- **Verify:** `python3 scripts/checks/docs_doctor.py docs`, plus the new drift-lint
  (§1.6) as the standing guard against re-divergence.

**Release** — versioning, notes, decision routing:

- Package `0.1.0a16`→`0.1.0a18` (or `a17` if alpha.17's package bump is still
  pending — confirm at release time; the *schema* is already at alpha.17's shape).
- Route the ADR-worthy decisions (delete-and-fold, the bundle-root rename, the
  `required_when` primitive, the drift-lint mechanism) to
  `releases/0.1.0-alpha.18/decisions.md`.
- Release notes: the normalized-names summary + the delete-and-reinstall instruction
  for any local dev vault. `release-please` is `workflow_dispatch`-only (pre-tag); the
  release gate records a manual dispatch or an explicit stay-pre-tag decision.

## 7. Open questions

1. **The one pull-forward candidate — tension recordability.** If the owner wants a
   PI-confirmed tension to have a persisted home before `query-architecture`, the
   minimal single slice is: add `concept_edges(concept_id, relation_type,
   target_concept_id, discovered_at)` with only `tension` populated (wire
   `surface_tensions`'s PI-confirmation step to write the row), leaving the
   `links:`-mirror half and the reindex loop for `query-architecture`. Recommended
   **default: no** (keep the query-substrate tables together), but it is a clean,
   self-contained pull-forward if wanted.
2. **`item_type` enum vocabulary.** `[paper, dataset, repository, web-page, report]`
   is the alpha.10-documented list; whether `podcast`/`preprint`/etc. belong is a
   content decision the analysis left open.
3. **`concept_id` prefix convention.** `catalog/sources/<id>` (`state.py:775`,
   `integrity.py:222`) is independent of the `concept_type` value; whether it becomes
   `catalog/works/<id>` for consistency is not answered by the OpenAlex audit and is
   deliberately not decided here.
4. **Residual `integrity.py` bare `type == "source"` checks** (`:235,330,1976,2005`,
   and `:2012`'s synthetic `type: source` dict) — flagged by the analysis as possibly
   dead code, not traced to ground; named for the executor to resolve or confirm-dead
   during S2, not chased into a full code audit here.
