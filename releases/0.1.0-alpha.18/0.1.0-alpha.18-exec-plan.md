# ExecPlan — alpha.18 data-structure normalization (delta on the released alpha.17 engine)

Living plan. Self-contained enough for a fresh agent to continue without chat
history. Tactical progress here; durable decisions to
`releases/0.1.0-alpha.18/decisions.md`, product state to issues.

Design authority: `releases/0.1.0-alpha.18/0.1.0-alpha.18-design.md`, which in turn

## 1. Purpose

Before alpha.18: the released schema carries six OpenAlex name collisions, two
redundant SSOTs (the markdown `work` type duplicating the catalog row; `source-note`
as its own type), three declared-but-dead `concept_type` values, a dead
`_migrate_v4_to_v5`, and a recurring schema-doc drift class — all catalogued in
`data-structure-analysis.md` (Parts 3/4/5/8, G0–G5, C1–C6).

alpha.18 normalizes all of it: correct, collision-free names; one SSOT per fact; the
bundle-root layout the query architecture will build on; the collapsed `note` mode
model; and a lint that keeps the docs from re-diverging. It does **not** build the
query engine or the tables it reads — those are `query-architecture` (design §0/§3).

### PR-S6 — Schema-documentation drift-prevention lint (Part 8 §1.6)

**Genuinely independent of S1–S5, because the lint is name-agnostic** — it validates
*alignment* (does every field/enum claim in the docs match whatever the live schema
YAML currently declares?), not a fixed set of names. That is the whole point: it must
stay green on the pre-rename state **and** on the post-rename state, catching only a
doc that has *drifted* from the schema. So its own unit tests run against **synthetic
fixtures** (a tiny throwaway schema YAML + a matching doc and a mismatching doc), never
against the repo's real names — which is what lets it land at any point in the S1–S8
sequence without encoding either the old or the new names.

Concretely: a `scripts/checks/` script + CI wiring that parses
`docs/reference/frontmatter.md` and `document-types.md` for field/enum claims and
diffs them against the live `.memoria/schemas/types/*.yaml`; fail on mismatch. Landing
order note: if S6 lands *before* the rename slices, it is green against today's
(old-name) schema+docs and then guards S1–S8 as they move schema and docs together; if
it lands *after*, it is green against the new names. Either way it never asserts a
specific name — only that the two artifacts agree. (Recommended: land S6 early as a
standing guard, so a rename slice that updates the schema but forgets a doc page fails
CI immediately.)

Test: `tests/test_alpha18_schema_doc_lint.py` — synthetic-fixture-driven: fires on a
seeded schema/doc mismatch, passes on an aligned pair; **no dependency on the repo's
own field names**, so the test's own correctness is unaffected by whether S1–S5 have
landed.

## 6. Idempotence and recovery

Every slice is a schema/code edit on a disposable vault — re-runnable by deleting the
sandbox and rebuilding (`scripts/sandbox/refresh-test-vault.sh`). No slice writes
irreversible state; the only durable artifact is the merged diff, reverted by normal
git. There is no data migration to be partially-applied, by design (§2).
