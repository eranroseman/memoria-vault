# Memoria 0.1.0-alpha.20 design - API contract spine and surface expansion

Date: 2026-07-07. **Status: proposal** - scratch design note, not an accepted
release record.

Base: alpha.19 standalone engine plus the CLI/HTTP/MCP audit performed against
the current implementation. The main conclusion carries forward unchanged:
Memoria should remain a standalone engine with thin control surfaces. Alpha.20
should make those surfaces contractual and drift-checked before building a
larger agent application.

---

## 0. Identity

Alpha.20 is the **API contract spine and surface expansion** release.

It does not replace the engine, move state into an editor, or introduce a
dedicated agent app. It makes the existing three control surfaces precise enough
for an Obsidian plugin now and a VS Code extension later:

- **CLI:** human and script surface.
- **Local HTTP:** editor/plugin surface.
- **MCP stdio:** agent/tool surface.

The design rule is: one engine contract, three adapters. Expansion means making
engine behavior available through the right thin surface, not adding new state
owners. If a behavior cannot be described once and exposed consistently, it is
not ready to be an integration surface.

---

## 1. Requirements

### Product requirements

- Memoria remains usable as a standalone local engine with no editor dependency.
- Obsidian integration is a thin local adapter, not a second source of truth.
- VS Code integration should reuse the same HTTP contract later.
- Users may bring their agent of choice through MCP.
- All machine writes enter the request envelope and remain review/check gated.
- Editor plugins should render engine views and queue operations, not edit
  Concepts, projections, SQLite, or journal rows directly.

### Engineering requirements

- The required package stays small; MCP remains an optional extra.
- HTTP remains loopback-only for alpha.20.
- MCP remains stdio-only for alpha.20.
- Specs and implementations must not drift silently.
- Read scope must be enforceable before payloads reach adapters.
- Authored body text returned to agents remains tagged as untrusted data.
- `--json` CLI output remains stable enough for scripts.
- Alpha.20 must define the minimal empirical event shape needed to measure
  beta.1 blocker decisions without logging raw note/source body text.

### Non-requirements

- No remote hosted API.
- No OAuth.
- No multi-user server.
- No dedicated agent app.
- No browser CORS surface unless a real plugin needs it and the threat model is
  updated.
- No retrieval default promotion beyond BM25.
- No full canvas/deep-work workspace.
- No code execution or code-authoring UX; beta.1's code-side output is an SRD.
- No analytics dashboard or custom survey system.
- No onboarding seed corpus, feedback dashboard, multi-device sync, or raw
  dataset packaging.

## 2. Current audit summary

### Already sound

- `memoria_vault.engine.api` is the right semantic boundary. It owns
  verdict-tagged reads, scoped projections, request-envelope writes, and
  untrusted-text tagging.
- `memoria_vault.runtime.http_transport` is correctly thin: stdlib loopback
  server, bearer token, selected engine reads, and one operation write route.
- `memoria_vault.runtime.mcp_transport` is correctly thin: optional FastMCP
  stdio server, required startup read scope, closed tool roster, and
  request-envelope writes.
- The CLI is the right baseline surface for humans and scripts, and its command
  roster is tested.

### Gaps to close in alpha.20

- The HTTP and MCP specs are manual docs, not generated or drift-checked from a
  shared contract.
- HTTP has no machine-readable OpenAPI document.
- HTTP returns `200` for unknown routes; that is poor adapter behavior.
- HTTP read scope is optional and per-request only; plugin sessions need a
  startup scope that can be narrowed but not widened.
- HTTP exposes less of the engine read API than MCP: no journal, project
  slice/draft, or exploration routes.
- MCP tools lack first-class descriptions and output schema metadata beyond the
  Python signatures.
- MCP exposes journal reads but not project slice/draft or exploration reads.
- CLI help is structurally correct but too bare for a user-facing product
  surface.
- Some CLI handlers still call lower-level runtime helpers directly. That is not
  always wrong, but the design target is that cross-surface behavior goes
  through `engine.api` or an explicit operation.
- `memoria new` and the shipped `.memoria/templates/` files do not yet have a
  small shared creation contract, so adapter authors cannot tell which fields
  are command inputs versus template defaults.
- Beta.1 blocker research needs a shared empirical event shape; alpha.20 does
  not yet define the fields, privacy rule, or drift check for disposition
  events.

## 3. Functionality by surface and expansion target

### 3.1 Engine API

The engine API is the deepest interface. It already exposes:

| Capability | Current functions | Expansion target |
| --- | --- | --- |
| Runtime status | `read_status` | Keep as-is; include in contract registry. |
| Operation catalog | `read_operations` | Add operation input-shape metadata when manifests can provide it. |
| Request state | `read_requests`, `read_request` | Keep list/detail split; add stable filter contract for status, operation, actor, and target path. |
| Attention | `read_attention`, `read_attention_card` | Keep table/card view specs; add standard action metadata so adapters know which buttons are valid. |
| Concepts | `read_concepts`, `read_concept` | Add pagination/filtering before large vaults make list views slow. |
| Work records | `read_work` | Add list/search Work read before editor plugins need local catalog browsing. |
| Journal | `read_journal`, `read_journal_event` | Keep scoped; expose through HTTP for editor diagnostics. |
| Project writing views | `read_slice`, `read_draft`, `compose_draft`, `verify_draft`, `promote_draft_passage` | Expose read views through HTTP/MCP; keep write actions request-envelope only. |
| Exploration | `read_exploration` | Expose as read-only through HTTP/MCP; do not make it a durable knowledge surface. |
| Generic writes | `run_operation` | Keep as the only cross-surface generic write. |
| Concept creation | `write_new_concept` | Define the minimal template-backed creation contract; keep external adapters on `operation_run` unless a named create action proves necessary. |
| Attention disposition | `resolve_attention` | Promote to a named engine action in the contract registry, implemented as request-envelope write. |

Do not add one typed engine wrapper per operation yet. The deep interface is
valuable because it hides request-envelope mechanics, not because it mirrors
every operation manifest with another function.

### 3.2 CLI

The CLI exposes the broadest surface:

| Area | Current commands | Expansion target |
| --- | --- | --- |
| Setup and diagnostics | `init`, `status`, `doctor`, `migrate`, `eval` | Add useful parser descriptions and stable JSON examples for common diagnostics. |
| Runtime servers | `serve --watch`, `serve --http`, `mcp` | Add HTTP startup read scope and a command to print the generated surface schema. |
| Work/source flow | `work add/import/enrich/digest/interview/update/export` | Keep CLI-first; expose only adapter-needed read/write primitives over HTTP/MCP. |
| Knowledge flow | `new`, `link`, `check`, `show`, `list`, `export` | Keep direct human ergonomics; define the `new` template contract and align shared read/write behavior with engine contracts. |
| Project writing | `project ask/trace/gaps/frame-paper/slice/compose/verify/resolve-evidence/promote/explore/export` | Make project read views available to HTTP/MCP, but keep project export and maintenance CLI-first. |
| Request/attention control | `request ...`, `attention ...` | Add adapter action metadata so UI buttons call the same request-envelope actions. |
| Maintenance | `workspace ...`, `steering`, `vocab`, `journal` | Keep most maintenance CLI-only unless an editor workflow proves it needs a route/tool. |

Expansion rule: CLI can stay broader than HTTP/MCP. It is the local operations
console, not just an adapter skin.

### 3.3 Local HTTP

The local HTTP transport currently exposes:

| Method | Path | Current role | Expansion target |
| --- | --- | --- | --- |
| `GET` | `/status` | Runtime status | Keep. |
| `GET` | `/operations` | Operation catalog | Add operation input metadata when available. |
| `GET` | `/requests`, `/request` | Request list/detail | Add filters already supported or added in engine API. |
| `GET` | `/attention`, `/attention/card` | Attention list/detail view specs | Add action metadata for plugin buttons. |
| `GET` | `/concepts`, `/concept` | Concept list/detail | Add pagination/filtering. |
| `GET` | `/work` | Work detail | Add Work list/search route. |
| `POST` | `/operation/run` | Generic request-envelope write | Keep as the only generic write route. |

HTTP should expand to parity with engine reads that editor adapters need:

- `GET /journal` and `GET /journal/event`
- `GET /project/slice`
- `GET /project/draft`
- `GET /exploration`
- `GET /openapi.json` or `GET /surface/schema`

HTTP should not expand into a remote product API in alpha.20. No OAuth, cookies,
CORS, SSE, WebSockets, or remote bind.

### 3.4 MCP

The MCP transport currently exposes:

| Tool | Current role | Expansion target |
| --- | --- | --- |
| `status` | Runtime status | Add description/output schema. |
| `operations` | Operation catalog | Add operation input metadata when available. |
| `requests`, `request` | Scoped request reads | Add filters matching engine contract. |
| `attention`, `attention_card` | Scoped attention reads | Include action metadata for human-confirmable next steps. |
| `concepts`, `concept` | Scoped Concept reads | Add pagination/filtering. |
| `work` | Scoped Work detail | Add Work list/search only after engine adds it. |
| `journal`, `journal_event` | Scoped journal reads | Keep; align route parity with HTTP. |
| `operation_run` | Generic request-envelope write | Keep as the only write tool. |

MCP should add the missing engine read tools:

- `project_slice`
- `project_draft`
- `exploration`

MCP should not add raw filesystem, SQLite, shell, browser, or network tools.
Resources/prompts/sampling remain deferred until a specific host workflow proves
they reduce friction without widening authority.

### 3.5 Adapter/UI surface

Obsidian and later VS Code should treat HTTP as the UI integration API and MCP
as the agent integration API.

| Adapter need | Surface | Expansion target |
| --- | --- | --- |
| Render dashboards/worklists | HTTP reads with view specs | Use engine view specs; do not parse SQLite. |
| Show note/work/project detail | HTTP read detail routes | Add missing project and Work browse routes. |
| Queue an action | `POST /operation/run` or named request-envelope action | Never write Memoria-owned files directly. |
| Agent reads scoped context | MCP | Start one MCP server per useful scope. |
| Agent proposes writes | MCP `operation_run` | Keep PI review/check path unchanged. |

The dedicated agent app remains out of scope until adapter testing proves the
plugin + MCP split fails.

## 4. Target design

### 4.1 One contract registry

Add one boring, data-only registry under the engine boundary:

```text
src/memoria_vault/engine/surface_contract.py
```

Each entry describes one public surface action:

```python
{
    "id": "concepts.list",
    "summary": "List scoped Concept summaries.",
    "engine": "read_concepts",
    "kind": "read",
    "scope": "optional-read-scope",
    "params": {
        "concept_type": {"type": "string", "default": ""}
    },
    "http": {"method": "GET", "path": "/concepts"},
    "mcp": {"tool": "concepts"},
    "cli": {"command": "memoria list"},
    "response_version": "engine-read-api.v1"
}
```

This registry is not a dispatcher. It is a contract source for:

- drift tests,
- HTTP OpenAPI generation,
- MCP tool descriptions,
- reference-doc tables,
- CLI help summaries where the command maps directly to engine API behavior.

The implementation still calls plain Python functions. No framework, no plugin
system, no dependency.

### 4.2 CLI

Keep `argparse`. The package intentionally avoids Click/Typer.

Alpha.20 CLI work:

- Add concise parser descriptions for top-level and nested commands.
- Keep `--json` as the script contract.
- Keep `--actor pi|agent`, idempotency keys, and schedule IDs on mutation
  commands.
- Add drift tests for commands that are part of the shared surface contract.
- Do not force every CLI-only maintenance command into HTTP/MCP.

The CLI remains broader than HTTP/MCP because it is also the local maintenance
surface.

### 4.3 Local HTTP

Keep the current loopback HTTP server.

Alpha.20 HTTP work:

- Add `memoria serve --http --read-scope <path>` for supported editor sessions.
- Server startup scope is the maximum scope for that HTTP session.
- Per-request `read_scope` or `scope` may narrow the startup scope, never widen
  it.
- Keep unscoped HTTP available only as explicit local debug behavior.
- Return correct adapter statuses:
  - `401` unauthorized,
  - `400` malformed input,
  - `404` unknown route or hidden target,
  - `405` unsupported method,
  - `413` body too large.
- Emit an OpenAPI JSON document from the contract registry.
- Add HTTP routes for journal, project slice/draft, exploration, and surface
  schema reads.
- Keep one write route: `POST /operation/run`.

Do not add CORS, cookies, OAuth, SSE, or WebSockets in alpha.20.

### 4.4 MCP

Keep stdio FastMCP as an optional extra.

Alpha.20 MCP work:

- Generate or validate the tool roster from the contract registry.
- Add human-readable tool descriptions from the registry.
- Add scoped tools for project slice, project draft, and exploration.
- Keep startup `--read-scope` required and non-root.
- Keep write identity fixed by server startup `--actor`.
- Keep `operation_run` as the only write tool.
- Do not add MCP resources/prompts/sampling until a host workflow proves they
  reduce user friction without widening authority.

MCP is for agent choice. It is not the editor UI and not a raw workspace tool
server.

### 4.5 Obsidian plugin shape

The Obsidian plugin should be a thin local control surface:

- Read plugin settings: Memoria executable, workspace path, HTTP port, token
  source, read scopes.
- Start or attach to `memoria serve --http`.
- Render engine view specs for attention, Concepts, requests, and worklists.
- Render project slice/draft and exploration views once HTTP exposes them.
- Queue writes through `/operation/run`.
- Never write Memoria-owned Concepts, projections, journal rows, or SQLite.
- Provide buttons for common operations; leave agent execution to the user's MCP
  host of choice.

The plugin may read ordinary workspace markdown for display when Obsidian is
already the user's editor, but any Memoria-owned state transition still goes
through the engine.

The alpha.20 data-collection shape is specified in
[`obsidian-data-collection-plugin-design.md`](./obsidian-data-collection-plugin-design.md).
It starts with commands, a status bar item, settings, and one event modal; a
custom pane is deferred until real dogfood sessions show command/modal capture is
too slow.

### 4.6 Dedicated agent app

Do not build it in alpha.20.

Reconsider only if at least one of these becomes true:

- Obsidian plus MCP cannot provide understandable approval/review flows.
- Multiple editors need the same complex UI and duplicating it is worse than a
  shared app.
- User testing shows plugin + MCP setup is the adoption blocker, not engine
  capability.
- The app can remain a pure client over the same HTTP/MCP contracts, with no
  state ownership.

### 4.7 Empirical event contract

Add the smallest data-only event contract needed by the beta.1 empirical-use
plan. This is measurement plumbing, not a dashboard or analytics product.

```text
src/memoria_vault/engine/empirical_events.py
```

Each event records a user-visible workflow fact:

| Field | Requirement |
| --- | --- |
| `event_id` | Required client-generated UUID. |
| `event_type` | Required value from the event-type schema. |
| `timestamp` | Required ISO-8601 timestamp. |
| `session_id` | Required opaque local session ID. |
| `project_id` | Optional project/workspace identifier. |
| `surface` | Required value from the surface enum. |
| `workflow` | Required or optional by event type. |
| `item_type` / `item_id` | Optional object reference; no body text. |
| `variant` | Optional comparison value from the event schema. |
| `decision` | Optional value from the decision enum. |
| `duration_s` | Optional positive duration. |
| `outcome` | Optional value from the outcome enum. |
| `reason_code` | Optional value from the reason-code enum. |

Define the exact enum sets in `engine/empirical_events.py`:

| Enum | Values |
| --- | --- |
| `surface` | `cli`, `http`, `mcp`, `editor`, `manual` |
| `workflow` | `ask`, `capture`, `gap`, `evidence-review`, `canvas`, `draft`, `srd`, `export`, `session`, `connection`, `operation` |
| `decision` | `accept`, `reject`, `edit`, `defer`, `override`, `abandon` |
| `outcome` | `connected`, `queued`, `flushed`, `kept-artifact`, `fallback`, `exported`, `blocked`, `failed`, `stopped` |
| `reason_code` | `useful`, `not-useful`, `too-slow`, `missing-context`, `wrong-scope`, `duplicate`, `confusing`, `privacy`, `offline`, `external-tool`, `other` |

Required fields vary by event type:

| Event type | Required fields beyond base fields |
| --- | --- |
| `session.started` | `workflow` |
| `session.stopped` | `workflow`, `outcome`, `duration_s` |
| `http.connected` | `workflow`, `outcome` |
| `view.opened` | `workflow` |
| `operation.queued` | `workflow`, `outcome` |
| `disposition.recorded` | `workflow`, `decision`, `reason_code` |
| `fallback.recorded` | `workflow`, `outcome`, `reason_code` |
| `export.attempted` | `workflow`, `variant`, `outcome`, `reason_code` |

The validator is a strict allowlist schema: unknown fields are rejected before
an event is queued, sent, or stored.

Rules:

- Do not store raw source, note, draft, or SRD body text in empirical events.
- Store accepted events as journal events under `.memoria/journal/`, with the
  normal SQLite journal mirror.
- Add new event types only for blocker questions in the beta.1 empirical-use
  plan.
- Keep free-text notes out of the event contract; session diary notes remain
  local scratch artifacts.
- Expose event schema through the surface contract for documentation and drift
  checks, but do not expose a dashboard in alpha.20.

### 4.8 Empirical event operation

Add a real packaged operation for recording events:

| Field | Value |
| --- | --- |
| Operation ID | `empirical_event.record` |
| Manifest file | `empirical-event-record.md` after safe filename normalization |
| Input schema | `empirical_event.v1` |
| Output schema | `journal_event_ref.v1` |
| Allowed tools | `trusted_writer` |
| Allowed paths | `.memoria/journal/`, `.memoria/index/` |
| Storage target | append one validated event to the worker journal |

The operation must:

- validate payloads through `engine/empirical_events.py`;
- reject unknown fields rather than blacklist known-bad names;
- reject raw body-text/path-like fields because they are not in the schema;
- write only the validated event plus journal metadata;
- use `idempotency_key=empirical-event:<event_id>` when called through
  `operation_run`;
- return the stored journal event ID and event ID.

Tests must cover manifest loading, enum validation, event-type required fields,
unknown-field rejection, raw-body/path leakage attempts, journal storage, and
duplicate `event_id` replay through the idempotency key.

This contract lets the Obsidian proof adapter and CLI dogfood runs record
dispositions and fallbacks without pulling beta.1 workflow implementation into
alpha.20.

### 4.9 Deferred beta.1 areas

Alpha.20 may expose read state for these areas when the engine already has it,
but it must not claim the product workflow. These are beta.1 blockers:

| Area | Blocker before implementation belongs in beta.1 |
| --- | --- |
| Retrieval default promotion | A preregistered fixture must show FTS/vector/hybrid/graph beats BM25 without worse grounding, latency, cost, or absence language; crash tests must cover passage/vector rebuilds and status cascades. |
| Canvas workspace | The small project canvas must beat the tabular/faceted baseline; user-owned positions, non-destructive re-render rules, and canvas-to-outline derivation must be fixed. |
| SRD lane | Code execution is post-beta. beta.1 must define the SRD sections, traceability rules, and verification checks. |
| PI evidence review | The review surface for `multi-hop`, `implicit`, and SRD-gap evidence must define accept/reject effects on draft verification. |
| Onboarding seed corpus | Actual citable sources, licenses, redistribution/fetch rules, and the install-to-first-answer/write path must be named. |
| Feedback/complementarity dashboard | Alpha.20 defines the event contract only; dashboard, metrics, payoff attribution, and shadow-first promotion stay deferred until real events exist. |
| Deep-work/gate topology | The user path across ask, attention/evidence review, writing, and SRD actions must be decided. |
| Multi-device topology | The product must choose single-writer, read-only secondary devices, or multi-writer sync before implementation. |
| Raw dataset bundling | The product must choose referenced blobs, companion packages, or catalog-by-reference only. |

## 5. Migration plan

### Phase 1 - Contract spine

- Add `engine/surface_contract.py`.
- Add tests that compare:
  - HTTP routes to contract entries,
  - MCP tools to contract entries,
  - reference tables to contract entries where tables repeat route/tool names.
- Keep the first registry small: status, operations, requests, attention,
  Concepts, Work, journal, project slice/draft, exploration, and
  operation_run.
- Register the empirical event schema as documentation/test metadata, not as a
  remote analytics endpoint.
- Add the `empirical_event.record` operation manifest and runner path.
- Add a drift check that fails when documented event fields diverge from the
  schema.

### Phase 2 - HTTP hardening

- Add server startup read scope.
- Make request scope narrowing explicit.
- Fix HTTP status codes.
- Add OpenAPI JSON generation from the registry.
- Add HTTP routes for journal, project slice/draft, exploration, and surface
  schema reads.
- Add focused tests for unknown route, unsupported method, scope narrowing, and
  OpenAPI route coverage.

### Phase 3 - MCP polish

- Add tool descriptions from the registry.
- Add project slice, project draft, and exploration tools.
- Add tests for descriptions and roster drift.
- Keep optional dependency behavior unchanged.

### Phase 4 - CLI polish

- Add command descriptions/help text for the shared surface.
- Add a CLI command or flag that prints the generated surface schema for local
  inspection.
- Define the minimal `memoria new note|hub|project` template contract and add a
  drift check against `.memoria/templates/`.
- Keep command roster tests.
- Do not migrate to another CLI framework.

### Phase 5 - Obsidian adapter start

- Build the smallest plugin that can:
  - connect to local HTTP,
  - show status,
  - list attention,
  - show one Concept,
  - queue one operation.
- Emit empirical events for connect, view, queued operation, disposition, and
  fallback actions where the adapter already owns the user interaction.
- Use the minimalist data-collection plugin design as the alpha.20 adapter
  starting point.
- Stop there until user testing shows which workflow deserves more UI.

## 6. Acceptance checks

Alpha.20 is done when these checks exist and pass:

- Contract registry covers every supported engine surface action, HTTP route,
  and MCP tool.
- HTTP OpenAPI output lists the implemented routes and request parameters.
- HTTP exposes journal, project slice/draft, exploration, and surface schema
  reads.
- HTTP unknown route returns `404`, unsupported method returns `405`.
- HTTP startup read scope cannot be widened by a query parameter.
- MCP tool roster remains closed and tested.
- MCP exposes project slice, project draft, and exploration reads with startup
  scope applied.
- MCP package remains optional; bare `import memoria_vault` does not import MCP.
- CLI command roster remains tested, and shared commands have useful help text.
- `memoria new` field inputs and shipped templates agree through a focused drift
  check.
- Empirical event schema documents enums, required fields per event type, and
  strict unknown-field rejection.
- `empirical_event.record` validates payloads, stores accepted events in the
  journal, and deduplicates replay through
  `idempotency_key=empirical-event:<event_id>`.
- Reference docs for CLI, HTTP, MCP, and engine API agree with the registry.
- The Obsidian proof adapter can connect to a disposable workspace without
  writing Memoria-owned files directly, and can emit a disposition/fallback
  event without storing note/source body text.

## 7. Risks

| Risk | Mitigation |
| --- | --- |
| Registry becomes a second implementation | Keep it data-only; no dynamic dispatch. |
| HTTP starts becoming a remote API by accident | Keep loopback-only and document remote/OAuth as out of scope. |
| Plugin bypasses the engine because files are easy to edit | Adapter rule: display may read files; state transitions call the engine. |
| MCP agents over-trust returned note text | Preserve server instructions and `untrusted_text` payload tags. |
| OpenAPI generation grows into a framework project | Generate a static JSON object from registry data only. |
| Surface expansion turns every operation into a custom endpoint | Add named routes/tools only for stable reads and high-frequency UI actions; keep generic writes through `operation_run`. |
| Empirical events become analytics scope creep | Define only the event schema and smoke checks; defer dashboards and survey tooling. |
| Offline event replay duplicates observations | Require client-generated `event_id` plus `idempotency_key=empirical-event:<event_id>`. |

---

## 8. Recommendation

Build alpha.20 as a contract-hardening release:

1. Keep the engine-first architecture.
2. Add the smallest shared surface contract registry.
3. Expand HTTP to the engine reads editor adapters need.
4. Harden local HTTP for scope, status codes, and OpenAPI/schema discovery.
5. Improve MCP metadata and add missing scoped project/exploration read tools.
6. Polish CLI help and expose the generated surface schema.
7. Define the minimal `memoria new` template contract.
8. Define the minimal empirical event/disposition contract and record operation.
9. Start the Obsidian plugin as a thin HTTP client that can emit those events.
10. Defer the beta.1 product blockers and the dedicated agent app.

This is the shortest path that makes Obsidian integration real while preserving
the standalone engine and keeping VS Code integration cheap later.
