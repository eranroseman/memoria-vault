## 1. Purpose / big picture

alpha.20 makes Memoria's **three control surfaces contractual and drift-checked** so an Obsidian plugin can be built now and a VS Code extension cheaply later, **without** turning the local engine into a hosted API or adding a second state owner. The observable result: one data-only **surface-contract registry** describes each public action once, and the CLI, loopback HTTP, and MCP-stdio adapters are generated-or-drift-checked against it; HTTP gains a machine-readable OpenAPI document, correct adapter status codes, a startup read-scope that can only narrow, and the engine reads editor adapters need (journal, project slice/draft, exploration); MCP gains descriptions and the matching scoped read tools; the CLI gains real help and a `memoria new` template contract; a strict **empirical-event** contract plus an `empirical-event-record` operation lets a dogfood adapter record dispositions and fallbacks with no body text; and a **thin Obsidian proof adapter** connects to a disposable workspace, renders engine views, queues one operation, and emits those events — writing no Memoria-owned files directly. "Done" = every check in design §6 exists and passes (§5 below).

Non-goals (design §1, §3.3–§3.4, §4.6): no remote/hosted API, OAuth, CORS, cookies, SSE, WebSockets, multi-user server, or dedicated agent app; no retrieval-default promotion beyond BM25; no analytics dashboard. Those are beta.1 blockers (design §4.9) and stay deferred.

## 2. Context and orientation

**Vocabulary (plain language, first use):**

- **Engine API** — `src/memoria_vault/engine/api.py`, the single semantic boundary. It owns verdict-tagged reads, scoped projections, request-envelope writes, and tagging authored body text as untrusted. All three surfaces call it; they do not touch SQLite, journal rows, or Concept files directly.
- **Request envelope** — the review/check-gated path every *machine write* enters. Adapters never write Memoria-owned state; they enqueue an operation that flows through the envelope.
- **Surface / adapter** — CLI (humans + scripts), loopback HTTP (editor/plugin), MCP-stdio (agent/tool). "One engine contract, three adapters."
- **Read scope** — a path prefix that limits which workspace subtree a surface session may read. alpha.20 makes it a *startup* maximum for HTTP/MCP that a per-request scope may narrow but never widen.
- **Contract registry** — the new data-only file (design §4.1) listing each public surface action once (id, engine function, kind, scope, params, http/mcp/cli binding, response version). Not a dispatcher — a source for drift tests, OpenAPI generation, MCP descriptions, doc tables, and CLI help.
- **Empirical event** — a small allowlisted record of a user-visible workflow fact (design §4.7), stored as a journal event with no source/note/draft/SRD body text.
- **Operation manifest** — a Markdown file under `src/memoria_vault/product/capabilities/operations/` describing one packaged worker operation (id, input/output schema, allowed tools/paths).

**Current `main` state (verified 2026-07-07):**

- Python package `memoria-vault`; CLI entrypoint `memoria = memoria_vault.cli:main`. Runtime installs into `<workspace>/.memoria/.venv` from `pyproject.toml`.
- `src/memoria_vault/engine/api.py` already exposes the functions the design names: `read_status`, `read_operations`, `read_exploration`, `read_requests`, `read_request`, `read_attention`, `read_attention_card`, `read_concept`, `read_concepts`, `read_work`, `read_slice`, `read_draft`, `compose_draft`, `read_journal`, `read_journal_event`, `run_operation`, `write_new_concept`, `resolve_attention` (design §3.1). **No `surface_contract.py` or `empirical_events.py` exists yet** — both are new in alpha.20.
- Transports: `src/memoria_vault/runtime/http_transport.py` (stdlib loopback, bearer token, selected reads + one `POST /operation/run`) and `src/memoria_vault/runtime/mcp_transport.py` (optional FastMCP stdio, required startup read-scope, closed tool roster). Both are "correctly thin" per the audit (design §2) and are expanded, not rewritten.
- Operation manifests live in `src/memoria_vault/product/capabilities/operations/` (e.g. `resolve-attention.md`, `capture-source.md`). The new `empirical-event-record.md` manifest goes here.
- Templates: `vault-template/.memoria/templates/{note,hub,project}.md` — the three files `memoria new` must agree with (design §4.2, §4.4).
- Surface docs: `docs/reference/cli.md`, `docs/reference/local-http-transport.md`, `docs/reference/mcp-transport.md`, `docs/reference/read-api.md`. These are the reference mirrors that must agree with the registry (design §6 last check, AGENTS.md "Zero tolerated contradictions").
- Surface tests: `tests/test_cli.py`, `tests/test_http_transport.py`, `tests/test_mcp_transport.py`. New contract/drift tests extend this set.
- CI gates (AGENTS.md "Required CI checks"): `pr-policy`, `lint` (`python scripts/verify l0`: ruff, docs-doctor, status-doctor, agents-doctor, ruleset-doctor, github-doctor, schema/design drift), `python-selftest` (L1 pytest in `tests/`), `cspell`, `markdownlint`, `lint-config`, shell/PowerShell lint.

**The gap (design → build):** the surfaces are sound but *un-contracted* — specs are hand-written, HTTP has no OpenAPI and returns `200` for unknown routes, read-scope is per-request only, HTTP exposes fewer reads than MCP, MCP lacks descriptions and three read tools, CLI help is bare, `memoria new` has no shared creation contract, and there is no empirical-event shape. alpha.20 closes those gaps by *adding a contract and expanding through it* — no new state owners, no framework, no new required dependency.

### 3.1 Docs update + docs-review (WS-6), in detail

Docs change with the code that changes behavior (same-CL rule); a final sweep reconciles the cross-cutting reference pages against the registry. Diátaxis routing per AGENTS.md:

- **Reference** (`docs/reference/`): update `cli.md` (new/expanded help, the surface-schema print command, the `memoria new` contract), `local-http-transport.md` (startup `--read-scope`, narrowing rule, status-code table, OpenAPI endpoint, the new journal/project/exploration routes), `mcp-transport.md` (descriptions, the three new tools), `read-api.md` (any read-shape additions). Any table that repeats the registry's route/tool/command/field names is a **covered mirror**: it must name the registry as owner and be guarded by a drift check (AGENTS.md "Zero tolerated contradictions"). Document the empirical-event schema (enums, per-event-type required fields, strict unknown-field rejection) as reference, but **no dashboard** (design §4.7).
- **Explanation** (`docs/explanation/`): the one-engine-three-adapters contract; why HTTP stays loopback-only and remote/OAuth is out of scope; the read-scope-narrowing invariant; untrusted-text tagging preserved for MCP.
- **Design book** (`docs/design/`): why the registry is data-only (not a dispatcher/framework) and why generic writes stay on `operation_run`.
- **Cross-file:** update `vault-template/` doc absolute-URL refs if any surface reference URL moved; add nothing to `AGENTS.md` platform facts unless the alpha baseline line changes.
- **Gates (all green):** `python3 scripts/checks/docs_doctor.py docs` and `--vault-links`, `markdownlint`, `cspell`, `status-doctor`; Diátaxis quadrant fit (no mixed-purpose pages); apply `the-elements-of-style` prose pass; **run the docs-review playbook** (`.agents/playbooks/docs-review.md`) before the docs PR and report findings by severity.

## 5. Validation and acceptance

**The acceptance gate is design §6 — nothing else.** Each check is stated as an observable claim with the lowest-cost proof (focused test → CLI repro → disposable-vault run), via `python3 scripts/verify`.

- **Claim (registry coverage):** Given the registry, when the drift tests run, then every supported engine action, HTTP route, and MCP tool has exactly one registry entry and no orphan on either side.
  - **Prove with:** `python3 -m pytest tests/test_surface_contract.py`
- **Claim (OpenAPI):** Given a running `memoria serve --http`, when `GET /openapi.json` (or `/surface/schema`) is fetched, then it lists every implemented route with its request params.
  - **Prove with:** `python3 -m pytest tests/test_http_transport.py -k openapi` + a curl repro against a sandbox `serve --http`.
- **Claim (HTTP read parity):** Given `serve --http`, when journal / project slice / project draft / exploration / surface-schema routes are called, then each returns the engine read payload (not `404`).
  - **Prove with:** `python3 -m pytest tests/test_http_transport.py -k "journal or project or exploration or schema"`.
- **Claim (HTTP status codes):** Given `serve --http`, when an unknown route is requested then `404`; when a supported route is called with an unsupported method then `405` (and `401`/`400`/`413` on their triggers).
  - **Prove with:** `python3 -m pytest tests/test_http_transport.py -k "status or unknown_route or method"`.
- **Claim (scope cannot widen):** Given `serve --http --read-scope <sub>`, when a request passes a broader `read_scope`/`scope` query param, then the response is still limited to `<sub>` (never widened).
  - **Prove with:** `python3 -m pytest tests/test_http_transport.py -k scope`.
- **Claim (MCP reads + closed roster):** Given `mcp --read-scope <sub>`, when the tool roster is listed then it is closed and matches the registry, and `project_slice`/`project_draft`/`exploration` return scoped reads under `<sub>`.
  - **Prove with:** `python3 -m pytest tests/test_mcp_transport.py`.
- **Claim (MCP stays optional):** Given a bare interpreter, when `import memoria_vault` runs with the MCP extra absent, then no MCP/FastMCP import occurs.
  - **Prove with:** `python3 -m pytest tests/test_mcp_transport.py -k optional` (or a `-c 'import memoria_vault'` in an env without the extra).
- **Claim (CLI help + roster):** Given the CLI, when `--help` is shown for shared-surface commands then each has a useful description, and the command roster test still passes.
  - **Prove with:** `python3 -m pytest tests/test_cli.py`.
- **Claim (`memoria new` contract):** Given the `memoria new` field inputs and `vault-template/.memoria/templates/{note,hub,project}.md`, when the drift check runs then command inputs and template defaults agree.
  - **Prove with:** `python3 -m pytest tests/test_cli.py -k "new or template"` (or the dedicated drift check).
- **Claim (empirical schema):** Given `engine/empirical_events.py`, when an event with an unknown field, a missing per-type required field, or a raw body-text/path-like field is validated, then it is rejected before queue/store.
  - **Prove with:** `python3 -m pytest tests/test_empirical_events.py`.
- **Claim (`empirical-event-record`):** Given the operation via `operation_run`, when a valid event is recorded then it lands as a journal event under `.memoria/journal/` with the SQLite mirror, and a replay with the same `event_id` (`idempotency_key=empirical-event:<event_id>`) stores no duplicate.
  - **Prove with:** `python3 -m pytest tests/` for the operation (manifest load, enum validation, per-type required fields, unknown-field + raw-body/path rejection, journal storage, duplicate-replay).
- **Claim (proof adapter):** Given the Obsidian proof adapter against a disposable `~/memoria-vault/sandbox` workspace, when it connects, lists attention, shows a Concept, queues an operation, and emits a disposition/fallback event, then no Memoria-owned Concept/projection/journal/SQLite file was written directly by the plugin and the event carries no note/source body text.
  - **Prove with:** the adapter smoke run transcript + a post-run `git status`/journal inspection on the sandbox, pasted into §11.
- **Claim (docs agree):** Given the reference pages, when docs-review + `docs-doctor` run, then CLI/HTTP/MCP/engine-API reference tables agree with the registry and the drift checks are green.
  - **Prove with:** `python3 scripts/checks/docs_doctor.py docs` + `python3 scripts/verify l0` + the docs-review report.
- **CI:** `python3 scripts/verify pr` green; `lint`/`python-selftest`/`cspell`/`markdownlint`/`lint-config`/`pr-policy` pass on every PR.

## 6. Idempotence and recovery

- **Safe to re-run:** each workstream is one squash-merged PR → `git revert <sha>` restores prior state. `surface_contract.py` and `empirical_events.py` are additive data/schema modules; re-adding them is a no-op diff. The `empirical-event-record` operation is idempotent by contract (`idempotency_key=empirical-event:<event_id>`), so replaying a recorded event stores nothing new.
- **Sandbox only:** all runtime/adapter validation runs against a disposable workspace under `~/memoria-vault/sandbox` (AGENTS.md); re-running rebuilds it from a seed — never a personal vault.
- **Rollback:** `git worktree remove ~/memoria-vault/worktrees/alpha20-<ws>` drops a workstream's tree; `git revert` the merge undoes a landed workstream. The scratch plan/decisions live on `origin/scratch` and are deleted at release close, not merged to `main`.

## 9. Surprises & discoveries

- 2026-07-07 — **The design's manifest name for the empirical operation is impossible as written.** Design §4.8 states Operation ID `empirical_event.record` *and* manifest `empirical-event-record.md` "after safe filename normalization." But `safe_filename` (`src/memoria_vault/runtime/paths.py`) preserves `.` and `_` — it only replaces chars outside `[A-Za-z0-9._-]`. Manifests are loaded as `safe_filename(operation_id) + ".md"` (`runtime/capabilities.py:108`), and every shipped operation uses a hyphenated single-token ID (`operation_id: resolve-attention` → `resolve-attention.md`). So `empirical_event.record` would resolve to `empirical_event.record.md`, never `empirical-event-record.md`. Chosen fix: adopt the convention-consistent `empirical-event-record` for both the operation ID and the manifest — which is also what the design author's manifest name implies. Follow-up: reconcile design §4.8's Operation ID field to `empirical-event-record` (schema IDs `empirical_event.v1`/`journal_event_ref.v1` and the idempotency namespace `empirical-event:<event_id>` are separate and unaffected).
- 2026-07-07 — **The Obsidian plugin location is currently gated shut.** `scripts/checks/plugin_provenance_doctor.py` (run by `scripts/verify pr`) forbids `packages/memoria-obsidian`, `packages/obsidian-plugin`, and `src/**/obsidian_*`/`agent_client*`. Since alpha.20 deliberately starts the plugin, WS-5 must update that doctor + its test to permit `packages/memoria-obsidian/` while keeping the real invariant (no plugin payload inside the baseline vault template). Folded into WS-5 as its first step.
