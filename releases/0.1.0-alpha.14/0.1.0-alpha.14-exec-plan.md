## 1. Purpose

The alpha.14 design makes Memoria a standalone CLI + engine. The alpha.12 SQLite
working-state boundary (ADR-122) and alpha.13 DOI enrichment (ADR-123) already
landed on `main`, so this is not greenfield. The pivot alpha.14 lands is:

- `memoria` CLI is the product surface (the console script now exists; parity
  and runtime gates are still incomplete).
- SQLite is the OpenAlex-style graph authority; markdown Work, Author, OpenAlex
  Source, and Institution records are removed (capture currently dual-writes
  both).
- Full-text acquisition, first-order citation/topic graph import, embeddings,
  richer retrieval, gap analysis, and discovery candidates are baseline
  functionality because checked digests and useful project gaps depend on them.
- qmd is the required local search backend for checked-only keyword and semantic
  retrieval; SQLite remains the graph/ops authority.
- The runner is pydantic-ai over an OpenAI-compatible endpoint; Hermes and
  `direct_api`-as-a-product-runner are dropped.
- Obsidian, Zotero, MCP-as-required, installed profiles, and lane/fleet are
  removed from the baseline - not deferred.
- Operation manifests are file-per-capability; optional same-stem asset folders
  are allowed only for large prompts, schemas, or examples.

Observable outcome: a fresh checkout where `pip install -e .` yields a working
`memoria` command that runs the thin end-to-end cycle (`init` -> `work capture`
-> `work enrich` -> `work update` -> `work interview` -> `work digest` ->
`note propose/accept/link` -> `memoria ask` / `project ask` -> `project export`
-> `workspace recover`) with no Obsidian, Zotero, Hermes, MCP, or profile
environment, and `rg` proves those paths are gone from runtime and installer.

  production-workspace migration target. There is no backward-compatibility branch.

## 10. Interfaces And Dependencies

- Keep the CLI on stdlib `argparse`, matching the existing worker entrypoint
  style. No new CLI framework.
- Reuse the existing SQLite queue/journal in `state.py`; do not add an event
  framework or workflow engine.
- pydantic-ai is runner scope only. It is the sole operation runner; it does not
  gate Work capture, graph persistence, or enrichment.
- The existing template provider config is conversion input. Runtime provider
  config lives at `<workspace>/.memoria/config/providers.yaml`; the runtime never
  reads provider policy from a repository checkout or template directory.
- Operation manifests live at `operations/<id>.md`; `operations/<id>/` is only an
  optional asset folder for oversized prompt/schema/example files. A directory
  without the matching Markdown file must not load.
- Optional external surfaces use `adapter` capability Concepts under
  `capabilities/adapters/`; `mcp` is not a capability type.
- Optional adapter policy hooks import `memoria_vault.runtime.policy.hook` from
  the installed workspace package. Their policy source is the workspace-local
  actor policy file, not lane overrides or a profile tool registry. The shipped
  workspace skeleton must not carry `vault-template/.memoria/mcp/`, make
  `.memoria/mcp` a runtime import path, ship `.memoria/lane-overrides/`, or ship
  `.memoria/tool-registry.yaml`.
- The `.memoria/operations/` subsystem (26 modules) was ported wholesale into
  `runtime/` in PR 6 and the vacated template operation home was deleted in #1118.
  `runtime/` is the single operation home; nothing runs from `.memoria/operations/`.
- Generated projections (`references.bib`, indexes, `knowledge/_views/index.md`,
  `capability-index.json`) must name their SQLite/Concept owner and be covered by a
  drift check.
- Security controls in design section 9 and the MCP-only sandbox guarantee must
  survive the Hermes/MCP/profile removal - re-read ADR-32/46/28/27 before cutting
  the sandbox plumbing.
