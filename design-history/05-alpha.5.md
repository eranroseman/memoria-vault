## alpha.5 — The Project gate (the fourth navigation gate becomes real)

alpha.5 fills ADR-70's deferred fourth-gate slot and ADR-68's empty `projects/` folder with a working **Project gate**: a bounded research inquiry that drives an argument toward a *defended or falsified* thesis and — the headline job — **knows when to stop** (saturation), inverting the treadmill where a dashboard's implicit purpose is to keep generating work. The release was driven by one converged design note (`project-starter.md`) whose central bet is that the gate's *logic* must be a deterministic Operation in the same trust class as the contradictions/knowledge-map views (auditable, replayable, bias-free) while its *inputs* stay PI-authored or agent-proposed-then-gated — never an LLM judging truth. It ships the deliberately narrowed **expanded v1 cut** (everything deterministic or free), pairs the gate with a coordinated ADR housekeeping sweep (three new ADRs, one recast, two rejections, one supersession), and clears cheap unblocked backlog slices (graded-loudness block gate, attention instrumentation, hub-handoff tier 2, PubMed/OA ingest, a visual design system). The WS-0 spikes de-risked the two structural forks up front (the Bases custom-view API and the maturity threshold), so the data model was never gated on rendering tech.

### Vault / concepts

**What:** Two net-new note types — `project` and `thesis` — added under `src/.memoria/schemas/types/{project,thesis}.yaml` + `src/system/templates/{project,thesis}.md`, both `category: projects`, `gated: false`. `project` carries `slug`, `scope_topics`, `inquiry` (PICO map), `finer` map, `output_mode: thesis|survey`, `question_version`, `question_log`, and `active_thesis`; lifecycle is only `[current, archived]`. `thesis` carries the full chain `proposed → provisional → current → retracted → archived` with `promotion_gate: current` and `initial_lifecycle: proposed`. — **Why:** the design's schema-grounded correction (§4): a `claim` cannot host a provisional position because `claim.yaml`'s lifecycle is `[current, retracted, archived]` — claims are *born canonical* (ADR-49/50), so "a provisional claim" is a schema contradiction. Forcing a thesis canonical "manufactures the premature-thesis / sunk-cost artifact this design exists to avoid" (ADR-78 Context; `project-starter.md` §4). The question and thesis are split into two artifacts because they are "different parts of speech" — interrogative+scoped vs. declarative — so a changed question triggers a staleness cascade while a changed thesis is ordinary hypothesis revision (ADR-78 Alternatives).

**What:** Project *status* is not stored — it is derived from the active thesis's lifecycle and `output_mode` (there is no project status field). — **Why:** follows ADR-10's precedent of "deriving state from structure rather than adding a parallel status field"; one lifecycle field absorbs output-mode visibility, project status, falsification-as-result, and mode transition (`project-starter.md` §4). A `retracted` thesis plus the subgraph that refuted it is a *finished* project, not a dead end (ADR-77/78 Consequences).

**What:** Folder renames `notes/source → notes/sources` and `notes/index → notes/indexes` (`schemas/folders.yaml` `homes` + `skeleton`; `.gitkeep` moves; all skill `write_scope`/descriptions, link-color CSS, and map skills updated). New homes `project: projects`, `thesis: projects`, `worklist-item: system/worklists`; new skeleton dirs `projects/_template/{code,drafts,exports}` and `system/worklists`. — **Why:** the project scaffold needs a per-project workspace shape (`projects/<slug>/` with code/drafts/exports), created by `start-project.js` (ADR-77 Consequences). The `source`/`index` pluralization is a consistency cleanup carried alongside the new folders; inferred housekeeping, not separately documented.

**What:** `worklist-item` note type added (`system/worklists/`, `worklists.base` with By-worklist/By-decision/By-group views) — `decision: [proposed, include, exclude, maybe, archived]`, `worklist`, `item_ref`, `rank`, `group`. — **Why:** materializes ADR-54's batch-worklist surface (folds #555 "batch worklist surface"); the ADR-audit found ADR-54 shipped only as a fleeting note + aggregate card, not the per-row Bases surface the prose promised (`adr-update.md` §1.2) — this closes that gap: one aggregate work-prompt per batch, each row a `worklist-item` note.

**What:** Schema field additions on existing types: `claim.yaml` and `thesis.yaml` `links` gain an optional `warrant` on `supports` entries (comment-level, not enforced); `paper.yaml` gains `ingest_status: [tier0, enriched, complete, needs-human]`; `source.yaml` gains `evidence_level: [cebm-1…cebm-5, ungraded]`; `gap.yaml` gains `gap_type: [additive, conflict, fragility, structural, unstated-warrant, refutation]` plus derived-cache fields (`impact`, `on_path`, `saturation_state`, `graph_maturity`, `computed_at`). — **Why:** `warrant` is reserved as an *optional attribute only* (Toulmin's "frequently left unstated" inferential rule) but deliberately NOT mandated — §13.2/ADR-79 rule it "the single biggest labor sink in the design for the least deterministic payoff," gameable as a tier-2 presence check, so v1 reserves the shape and leaves the unstated-warrant gap detector off. `ingest_status` fixes the ADR-30 audit finding that the field was "written by engine code but not declared in `paper.yaml`, so it escapes schema validation" (`adr-update.md` §1.2). `evidence_level` (CEBM) is "the one genuinely mechanical bridge between reliability and impact" — a rankable enum that can feed impact triage (`project-starter.md` §11). `gap_type` generalizes the ADR-51 `gap` card beyond "missing-source need" to the full missing-vs-contested taxonomy (§6, §16).

### Gate / provenance / the deterministic engine

**What:** New Operation `src/.memoria/operations/processing/project/structural_impact.py` (~800 lines) — the whole gate spine. It reads the thesis-rooted `supports`/`contradicts` subgraph, computes the connected component, articulation points (Tarjan DFS), and per-note **impact** via lost-reachability-from-thesis (falling back to degree for non-articulation nodes), and writes ONE generated `projects/<slug>/project-gate-index.md` with `computed_at`/`stale` and per-note derived values, embedding a machine-readable JSON block between HTML comment markers. — **Why:** Bases "reads only frontmatter, never note bodies, and cannot traverse graphs" (§12), so impact/saturation/maturity *cannot* be Bases formulas — they need a deterministic traversal Operation. Materializing to a **single index-note** rather than per-note stamps is the WS-0 §13.4 verdict: because the `registerBasesView` API was verified usable (§D3), pay the custom-view cost once and honor the "write notes once" discipline / avoid the ADR-25 append-only audit write-amplification a per-component re-stamp would cause (`ws-0-spikes.md`; `project-starter.md` §12/§D4). The index-note also self-excludes from the scan (`generated_by: memoria-structural-impact`).

**What:** The **maturity threshold** is hard-coded conservative: a thesis component is `mature` only when connected with `>= 5` addressed relations including `>= 1 supports` AND `>= 1 contradicts` (`MATURITY_RELATION_THRESHOLD = 5`); below that it is `immature`/`cold-start` and the payload returns `displayed_confidence: below-threshold`. — **Why:** §13.1, resolved in WS-0 and recorded in ADR-77: this is "the gate's trust switch… the parameter the gate's safety rests on, not a knob to tune after launch." Set too low and the gate "asserts confidence on a sparse graph — the automation-bias failure the whole design exists to prevent"; too high and it never terminates. "Lowering this threshold is a safety change, not a preference" (`project-starter.md` §13.1; `ws-0-spikes.md` §13.1). The `>=1 contradicts` requirement encodes Popper: a thesis corroborated only by accumulated `supports` "looks complete but is intellectually untested" (§5).

**What:** **Saturation** gates on three conditions in code: (1) `maturity == mature`; (2) `high_open_gaps == 0` (no on-path gap with impact `>= HIGH_IMPACT_THRESHOLD = 2`); (3) a structural refutation floor (a `contradicts` edge touching the thesis) AND a PI-set `refutation_sufficiency` truthy stamp on the project/thesis. `cold-start` forces `saturation_state: unknown`. — **Why:** this is *theoretical* saturation / Dey's "theoretical sufficiency," not data saturation (§5). Naïve "all gaps below threshold" is "true on an empty project" (fires at birth) and "true on an unchallenged thesis." Conditions 1–2 are "fully structural… replaces an unprovable prediction with a checkable fact about topology"; condition 3 is the irreducible tier-3 judgment that "resists structural formulation," so v1 collapses it to a **one-click PI-asserted `refutation_sufficiency` stamp** rather than a standing honesty-card nag (§5, §19). Honest caveat recorded: the stamp is itself gameable — "an acceptable 'trust the solo PI' choice for v1."

**What:** **Gap taxonomy** in code: three confident, zero-authoring kinds ship (`additive`, `conflict`, `fragility` — `CONFIDENT_GAP_KINDS`); `structural`/`refutation` are computed but emitted as `ADVISORY_GAP_KINDS`; `unstated-warrant` is absent. — **Why:** §13.5 chose "three, not two and not six," discriminated by *authoring cost*: fragility is "free — a degree query, zero PI authoring" so the reviewers' proposed strip-to-two "drops this for nothing." Structural is deterministic-but-niche, refutation folds into saturation condition 3 as a stamp, and unstated-warrant needs the not-shipped `warrant` (§13.5 table). Advisory items are "collapsed, uncounted, on-demand, and *hidden* (not greyed) below the maturity threshold — so the dashboard never becomes a guilt ledger" (§19).

**What:** **Cold-start** two-regime triage: below maturity the Operation uses `scope_overlap` (project `scope_topics` ∩ note terms) rather than argument-structure impact. — **Why:** §8 — impact ranking "is weakest precisely when orientation is most needed," so the honest fix is to name the regime: scope-overlap is deterministic (via ADR-15 `project-hints` topics) and "never forces a tier-3 jump," and the dashboard displays low confidence instead of a confident-looking meaningless ranking.

**What:** **Graded-loudness block gate** — new `operations/lib/loudness.py` + `policy_mcp.py` now denies any mutating action to a review-gated path when open `loudness: block` + `lifecycle: proposed` inbox cards exist (`policy_rule: loudness.block.active`), returning the blockers; `alert`/`block` cards are pushed to Telegram (best-effort, logged to `loudness-push.jsonl`), `quiet`/`notice` stay pull-only. — **Why:** commit #549/#610 — loudness stops being pure display metadata and gains routing semantics: "open block cards pause delegation and review-gated promotion until the PI acknowledges them." This is the safety-valve that lets a structural invalidation (thesis supersession, question-revision cascade) reserve the loud channel — §12's rule that "staleness ≠ urgency; reserve the loud channel for invalidation."

**What:** Audit log entries now stamp `schema_version: 2` and `review_mode: blocking` on every append (`policy_mcp.py append_audit`). — **Why:** the ADR-audit's ADR-41 "stamping sliver" — "stamp `review_mode: blocking` + bump `schema_version` now (attribution is non-backfillable); advisory behavior stays deferred" (`adr-update.md` §2.3/§4.2). Provenance attribution can't be added retroactively, so the cheap stamp lands even while the advisory feature waits.

**What:** New `cron_heartbeat.py` writes `system/logs/cron-heartbeat.jsonl` (last-successful-run per job); cron wrappers (`*-cron.sh`) updated to call it. — **Why:** the "start-now instrumentation" principle (`adr-update.md` §3.3): deferred always-on triggers (ADR-63) "can never self-detect" without the log — the heartbeat lets a future "recorded cron miss with sleep already disabled" trigger fire from data rather than PI memory.

### Security

**What:** The MCP-only sandbox gains an active *block* dimension (loudness block gate above) and the audit trail gains `schema_version`/`review_mode` stamps; the deny path is asserted by test (#610 "policy gate deny assertion"). — **Why:** §12's corrected gating analysis: `policy_hook` gates agent/Hermes writes but a *human* base-board drag "writes through Obsidian directly, and ADR-49's Linter monitor catches it only post-hoc." ### Removed / retired (and why)

- **ADR-34 (Code-artifact autopilot) — rejected.** Structurally precluded by ADR-21's L3 autonomy ceiling; any code-lane execution is the separate #369 "define the code lane" decision, "not an autopilot" (`adr-update.md` §5). Coding is also explicitly out of scope for the Project gate (§10 — "code output has a different artifact lattice… and a different lane").
- **ADR-40 (Admin/forensic GUI) — rejected.** Forensic need met by CLI + dashboards now and the deferred ADR-58 read-only Inspector later; a GUI "would add a second un-gated write surface — declined on architecture, not tool maturity" (§5).
- **ADR-17 (Shared candidate frontmatter) — superseded by [50,51].** The normative schema never shipped; the live `type: candidate` is ADR-51's honesty card. The stale `candidate-note`/`candidate_status`/`99-system/` schema body was dropped (§5).
- **The shipped-wheel deployment spine (old ADR-76) — discarded.** Kept only as its "code-layer install mechanism"; dropped as the *shipped artifact* because it can't carry the authored two-thirds, adds a premature supply-chain trust surface + non-atomic `pip --upgrade` rollback, and left gate skew "an open execution blocker" (new ADR-76 Alternatives).
- **Explicitly deferred, each for a named reason (release-plan §5):** the full ephemeral test-env harness (ADR-77/78 of that effort — alpha.5 ships only the thin S5 validation slice); the **inference tier** (tier-3 semantic impact / any LLM judging the gate — "breaks the auditability the gate rests on"); a `warrant` *mandate* and the unstated-warrant gap; multi-thesis projects and auto-promotion of gated transitions; **ADR-61's nightly autonomous loop** — *not accepted to unblock the gate* because gap-pull already works as on-demand `find` via the unified Librarian (ADR-48, per ADR-37's supersession note), and ADR-61's own adoption conditions (always-on deploy, maintained `research-focus.md`, a written screening-plan) are unmet, its `assumes` stale (§11/§13.3).
