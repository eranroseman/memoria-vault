## alpha.12 — The SQLite state boundary: working-state becomes a real database, the keep-set stays plain markdown

**Theme:** alpha.12 relaxes the alpha.11 "files-are-SSOT everywhere" rule and installs a **single embedded SQLite database (`.memoria/state/memoria.sqlite`) as the source of truth for Memoria's *working state*** — the operation queue/request envelope, the hash-chained journal, the catalog of sources, the concept/output ledger, durable materialization payloads, and the machine-consumer read barrier — while the **keep-set (`knowledge/` + `capabilities/` + generated `references.bib`) remains git-tracked markdown**. The release was driven by a single sharp reframing ("the keep-test": *what must survive Memoria's death is markdown OKF + a portable bibliography; everything else is free to be SQLite*), and by a pre-implementation test harness whose **seven results (4 PASS / 3 FAIL, per `alpha12-test-results.md`)** forced concrete contract decisions before any engine code was written (a separate four-spike `run_spikes.py` set ran all-PASS). This is a storage/persistence/recovery release: the design also re-frames the whole system as a **CLI-first standalone integrity engine with pluggable triggers** (command / file-change / schedule) rather than an Obsidian app, though the shipped code lands only the engine spine and cross-store recovery, not the full trigger/CLI surface.

A caveat that runs through this changelog: the alpha.12 **design doc is far larger than what shipped in this commit range.** The design specifies an `entities`/`catalog_links` graph, recursive-CTE cascade-rollback, a git-tracked `journal-heads.jsonl` anchor, and a Turso backup adapter — but the shipped `state.py` lands a deliberately smaller 7-table schema + 1 view, plus cross-store recovery. I mark design-only items as such.

---

### Vault / concepts / schemas

- **What:** Every keep-set Concept type gained an optional `citations: list` field — `digest.yaml`, `hub.yaml`, and `note.yaml` each added `citations` to their `optional:` block (`vault-template/.memoria/schemas/types/{digest,hub,note}.yaml`), surfaced in `docs/reference/document-types.md` and `frontmatter.md`. **Why:** this is the load-bearing consequence of moving the catalog into SQLite. Once source identity lives in a database that is *not* part of the takeout promise, a note/digest/hub that only holds a `source_id` handle becomes meaningless the moment SQLite is dropped. The design's §1 self-sufficiency invariant demands the keep-set "still read as complete, grounded knowledge in any markdown/OKF tool" after deleting SQLite, so §3's **Cross-store reference rule** requires denormalizing a compact survival copy (`source_id`, `title`, `authors`, `issued`, and at least one of `doi`/`url`/`citekey`) into the file itself: "`source_id` is the live handle; the embedded Citation is the survival copy" (`0.1.0-alpha.12-design.md` §3). The cross-store spike's FAIL result ("Cross-store same-operation citation lifecycle") corroborated that citations must be first-class, not derived on demand.

- **What:** Two new workspace skeleton directories — `.memoria/state/` (holds `memoria.sqlite`) and `.memoria/audit/` (git-trackable audit anchors) — added to `folders.yaml` `skeleton:` (skeleton entries `32 → 34`, reflected in `docs/reference/configuration.md`), each with a tracked `.gitkeep`; `.gitignore` now ignores `.memoria/state/**` contents while keeping the directory. **Why:** SQLite needs a durable, per-machine, gitignored home distinct from the disposable worker mirrors (`index/`, `queue/`, `staging/`, `quarantine/`), because the DB is *authoritative* working state, not a rebuildable index — "runtime content is disposable" no longer applies to it (`vault-template/.gitignore`, `on-disk-layout.md`: "`state/` — SQLite working-state DB"). The `audit/` directory scaffolds §7's git-tracked journal-head anchor (tamper-evidence for the SQLite journal). **Note:** in this commit range `audit/` is only *created*; the `journal-heads.jsonl` write path described in design §7 is not yet wired (the anchor helper appears only in later versions).

- **What:** `concepts.concept_type` is constrained to a fixed 13-value enum in the DB — `source, digest, note, hub, capability, operation, skill, mcp, workflow, person, organization, venue, project` (`state.py` `_init`). **Why:** this is the DB half of the "generator ≠ checker / prevent-what-a-DB-can" split (design §4): concept typing becomes a `CHECK` constraint the worker transaction enforces, converting a class of type errors "from detection to prevention." The through-line remains `.memoria/schemas/types/*.yaml` as the Concept-field owner; the SQLite enum is a role adapter checked against it (§2/§5).

### Retrieval / store / persistence — the SQLite state boundary (the core of the release)

- **What:** A new 763-line module **`src/memoria_vault/runtime/state.py`** introduces the SQLite working-state boundary at `.memoria/state/memoria.sqlite` (`DB_REL`), created via `connect()` with `PRAGMA foreign_keys = ON`, initialized by an inlined `executescript` schema (`SCHEMA_VERSION` not yet externalized — the standalone `schema.sql` file arrives in a later release). Shipped tables: `operation_requests`, `journal_events`, `concepts`, `outputs`, `materialization_payloads`, `catalog_sources`, `derivations`; plus the `consumable_outputs` view. **Why:** alpha.11's "passive rebuilt index over files" could never *prevent* violations, only detect them post-hoc. The design's clean-slate DB role (§4) makes SQLite "the SSOT for catalog and ops, and the projection engine for search/derived views," giving four jobs: retrieval, catalog+ops SSOT, relational/graph queries, and **write-boundary constraints** (FK/UNIQUE/CHECK/DAG "inside the same worker transaction"). The driving reframe: the keep-test proved most of Memoria's state (queue, journal, catalog graph, flags) fails the "need-this-to-think-outside-Memoria" test, so it "is free to be SQLite" (§1). Rejected alternatives are named explicitly: SQLite-as-SSOT-for-the-keep-set, CRDTs, and a hosted primary DB are all rejected because "the keep-set remains plain markdown; CRDT/hosted-primary storage breaks the local single-user simplicity target" (§4); a Python `eventsourcing` framework is rejected because "SQLite transactions, ordinary tables, and a small recovery state machine are clearer for a single-user local tool" (§7). **Note (design vs shipped):** the design's `entities` and `catalog_links` tables did **not** ship in this range — the catalog is a single flat `catalog_sources` table; the source↔entity↔link graph is design-only for alpha.12.

- **What:** The **request envelope** is now a durable SQLite row. `state.request_envelope(...)` builds and `save_request()` persists a uniform envelope with `request_id`, `trigger_type` (CHECK-constrained to `command`/`file_change`/`schedule`), `operation_id`, `args`, `idempotency_key` (UNIQUE), `target_path`, `target_hash`, `causal_refs`, `actor`, `provenance`, and optional `schedule_id`; `operation_requests.status` is CHECK-constrained to `pending`/`running`/`done`/`failed` (`state.py` `_init`, `save_request`). The worker (`worker.py`) now writes this envelope on every `enqueue_trusted_write`/`enqueue_operation`, deduplicates against SQLite first (`state.request_job`), and claims the next job from SQLite (`state.next_pending_job` → `_claim_sqlite_job`), mirroring JSON files under `.memoria/queue/{pending,running,done,failed}` only for Obsidian compatibility. **Why:** design §6's **Trigger lifecycle — one request envelope**: "Every trigger creates the same durable operation request before the worker runs" so "the queue survives a crash" (the `requested` state "makes accepted work durable"). The §11 gates "Trigger unification" and "Request-envelope contract" require command/file-change/schedule triggers to persist identical fields and enter the same lifecycle. The `idempotency_key UNIQUE` constraint enforces the design's crash-safe, replay-safe queue. Making the queue a DB row rather than a directory of JSON files is what lets "SQLite decide what to resume, roll back, or re-promote" on crash (§4).

- **What:** The **journal is now an append-only, hash-chained SQLite table.** `journal_events(event_id, timestamp, event_type, machine, payload_json, prev_hash, row_hash UNIQUE)` with `prev_hash`/`row_hash` chaining (`GENESIS` seed, SHA-256 over the row) and two `BEFORE UPDATE`/`BEFORE DELETE` triggers that `RAISE(ABORT, 'journal is append-only')`. Every trusted-writer event is now dual-written: `_append_event` appends to the JSONL journal **and** `state.append_journal_event` (`trusted_writer.py`). **Why:** design §7 rejects verifiable-log servers (immudb / Trillian / Sigstore-Rekor) as "overkill and break local-first," choosing instead "a local append-only SQLite journal (`event_id`, `prev_hash`, `row_hash`, append-only triggers)" for "enough practical tamper-evidence for the single-user threat model." The **SQLite journal-integrity spike** proved append-only rows and hash chains "need an external git-tracked head anchor for tamper-evidence" (design Provenance / test-results), motivating the (design-specified) `audit/` anchor. The journal vocabulary (`requested/run/model_call/derived/check-fired/checked/materialized/resolved`) is carried as the contract; the "Journal vocabulary and queue/flag projection rebuild" spike PASSED, confirming the full vocabulary can rebuild queue/flag/status views (test-results §PASS).

- **What:** A **cross-store materialization protocol** with an explicit `materialization_status` state machine. `outputs(output_id, concept_type, store∈{db,file}, target_path, staging_path, check_status∈{unchecked,checked,quarantined}, materialization_status∈{none,pending,materialized,failed}, output_sha256, materialized_commit, failure_reason)`, paired with `materialization_payloads(output_id PK→outputs ON DELETE CASCADE, expected_sha256, payload_text)`. `stage_concept` now calls `state.record_file_output(...)` storing the **full staged payload text** plus its hash (rejecting empty payloads and hash mismatches); `_write_checked` calls `state.mark_checked(...)`; and `commit_writer_changes` calls `state.mark_materialized(rel, commit=<hash>)` for each committed path (`trusted_writer.py`). **Why:** this is the release's trickiest correctness surface. Design §4/§6 make the rule "**Transactions: SQLite first, file promotion second**" — a DB commit can land before git promotion finishes, so file-owned outputs stay `materialization_status: pending` and non-consumable until the coupled git commit records its hash and flips to `materialized`. The **persistence/crash-replay spike FAILED** with the decisive finding: "Hash-only derived events cannot replay materialization after staging is lost… hashes alone are insufficient" (test-results §FAIL "Persistence authority and crash replay"). That failure is what forced the `materialization_payloads` table to store the **actual payload bytes**, not just a hash — design §92: "Hash-only pending rows are invalid… the DB stores the staged payload bytes." The lifecycle spike also proved "`checked` must be a check result rather than a consumability state" (design Provenance), which is why `check_status` and `materialization_status` are two independent columns.

- **What:** **Crash recovery from durable payloads.** `state.recover_pending_materializations()` scans file-owned outputs that are `checked` but still `pending`; if the target file already exists on disk with the expected hash it flips to `materialized`, otherwise it replays the file from the durable `payload_text` (failing the row with a `failure_reason` if the payload is missing or hash-mismatched). Wired to a new `memoria worker recover` CLI command (`worker.py` `main`). **Why:** directly answers the §11 "Crash/materialization replay" gate and the WP1 accept criterion ("crash *after* DB commit replays file promotion from payloads; hash-only pending rows rejected"). It is the concrete mechanism behind design §6's "on crash, the worker resumes promotion from the durable materialization payload or marks the operation failed." **Note:** the shipped recovery is payload-replay only; the design's git-commit-verification step (`_committed_materialization_commit`) and `recover_running_requests` appear only in later versions.

- **What:** The **read barrier is now a SQL view.** `consumable_outputs` selects outputs where `check_status='checked'` AND (`store='db'` OR (`store='file'` AND `materialization_status='materialized'`)). **Why:** design §6 makes the read barrier "a *machine-consumer* guarantee": agents/retrieval/export consume only this view, and "`checked` alone is never enough for file-owned outputs." The **checked-barrier spike PASSED**, confirming machine output stays outside `knowledge/` until checked promotion while "PI live edit can exist in knowledge while machine consumers filter it out" (test-results §PASS) — the human is trusted and reads live content directly; the barrier gates machines, not people (design §6, carried from `why-review-gate-is-structural` / `why-human-gate` principles).

- **What:** A **`derivations(input_id, output_id, actor∈{pi,operation,integrity})`** edge table, populated by `record_file_output` from each output's declared inputs. **Why:** this is the rollback-graph substrate. Design §4/§6 want "cascade-rollback blast-radius and retraction propagation" as "recursive CTEs… not hand-rolled walks," and §4 job 3 makes "Concepts/Links/journal as tables" so "transitive closure becomes SQL." The **SQLite oracle + recursive-rollback spike PASSED** ("UNIQUE, FK, type enum, DAG check, and recursive blast-radius query all worked… blast radius spans DB and file-backed concepts via the journal projection," test-results §PASS). **Note (design vs shipped):** alpha.12 lands the `derivations` edges but **not** the recursive-CTE cascade-rollback operation itself — rollback still runs through the alpha.11 integrity/journal path in this commit range.

### Pipeline / ingest / catalog / enrichment

- **What:** The **catalog is now SQLite-authoritative.** `catalog_sources(source_id PK, concept_path, doi UNIQUE, title, resource, identifiers_json, citekey, csl_json, metadata_status∈{verified,partial,unverified,not-indexed}, check_status, content_hash, raw_hash)`. `capture_source` (`capture.py`) and `_write_checked` for `type='source'` (`trusted_writer.py`) now call `state.upsert_catalog_source(...)` on every checked source, and `compile_source_digest` (`operations.py`) upserts before deriving. **Why:** design §3/§4/§10: "SQLite is SSOT for catalog + ops. File SSOT is limited to the keep-set." The `doi UNIQUE` constraint is the "no duplicate `source_id`/DOI" prevention from §4. The alpha.11 model kept source identity in per-source markdown; the keep-test judged the *relational catalog graph* as working state that fails the survival test (§1), so it moves to SQLite while the source *concept file* survives as the projection.

- **What:** **`references.bib` generation flips to SQLite-first with a file fallback, and becomes atomic with bibliography-changing captures.** `render_references_bib` now reads `state.catalog_sources(vault)` when `state.has_catalog_sources()`, falling back to scanning `catalog/sources/**/source.md` only when the DB has no rows (`capture.py`). `capture_source` computes `_has_bibliography_fields(...)` on the checked source and, when true, calls `write_references_bib(commit=False)` and folds `references.bib` into the *same* git commit and journal `outputs` list (`capture.py`). Documented across `ingest.md`, `system-actions.md`, `system-artifacts.md`, `operations.md`. **Why:** the **catalog-recovery spike FAILED** — "Periodic export loses Memoria-owned catalog changes made after the last export… Alpha.12 needs atomic tracked export per catalog commit" (test-results §FAIL "Catalog recovery from committed artifacts"). That forced design §3's **bibliography atomicity** rule: "every transaction that changes BibTeX-visible source fields schedules `references.bib` as a file-owned materialization before the operation can complete" and the §11 gate "Bibliography atomicity." `references.bib` is the **bibliography takeout** — "enough for the user to take completed sources with the keep-set" (§1) — the accepted lossy portability tradeoff vs the full catalog graph (§9 Costs: "`references.bib` is portable but lossy relative to the catalog DB; this is accepted for alpha.12 simplicity").

- **What:** **Compact citations are embedded into machine-derived digests, hubs, and note candidates.** `compile_source_digest` computes `state.compact_citation(source_rel)` and, when present, writes `citations: [citation]` into both the digest frontmatter and every derived hub frontmatter (`operations.py`); `emit_note_candidates` propagates `citations` from the source row or the parent digest into each candidate note (`knowledge.py`). `compact_citation` builds the survival map from `csl_json` (authors via `_csl_authors`, date via `_csl_issued`) and picks one locator in priority `doi > url > citekey`. **Why:** implements the §3 cross-store reference rule at the point of generation — the machine writer is responsible for making its own output self-sufficient, so "this is what makes §1's self-sufficiency invariant hold for digests and hubs, not just notes."

### Gate / provenance / integrity checks

- **What:** A new integrity check **`integrity-citation-survival-check`**: runtime helper `integrity.check_citation_survival(...)` flags any `checked` note/digest/hub whose catalog-source references lack a valid compact `citations` payload, delegating the per-frontmatter rule to `state.check_citation_payload(...)` (which requires each citation map to carry `title`, `authors`, `issued`, and one of `doi`/`url`/`citekey`). Shipped as a new capability Concept (`capabilities/operations/integrity-citation-survival-check.md`, `required_checks: [citation-survival]`, `risk_class: medium`, `runner: local`, `model: deterministic-fixture`), registered in `ai-catalog.json` and `capabilities/index.md`, added to the worker's `INTEGRITY_SWEEP_OPERATIONS` and dispatch, and documented (the daily sweep now runs **nine** integrity checks, up from eight, per `operations.md`). **Why:** the citation-survival invariant is only useful if it is *enforced*. Design §3 states "Validators reject a keep-set source reference that has no matching Citation payload," and this check is the enforcement arm — a structural, deterministic check consistent with `why-deterministic-methods` (prevent/detect only what is machine-decidable; the compact-payload presence test is fully structural).

- **What:** The **seeded-error verdict gate re-baselined to alpha.12.** New fixture `vault-template/system/eval/alpha12-seeded-errors.json` (13 seeded cases across structural/blind/checkpoint classes: evidence-resolves, contradiction-link, link-target, source-metadata ×2, stale-as-current, unwarranted-claim, crafted-injection, fresh-uncorroborated-source, poisoned-span, quote-anchor), with the release bars pinned at `recall_min=1.0`, `false_positive_rate_max=0.0`, `rollback_completeness_min=1.0`, `residual_error_rate_max=0.0`, `checkpoint_value_rate_min=1.0`. The worker's `run-seeded-error-verdict` and `seeded_errors` now prefer the alpha12 bundle and fall back to alpha11 if absent (`worker.py`; `operations.md`, `system-actions.md`). **Why:** design §8 carries forward "the gateless model + the seeded-error verdict as the release gate to real use" — the engine must be *certified* before real use (exec-plan WP9). Re-versioning the bundle keeps the gate pinned to the alpha.12 engine while the fallback preserves backward compatibility during transition. This is the concrete instrument for the two carried-forward bets (integrity engine trustworthiness; compounding-wiki value).

### Profiles / runtime — engine reframing (design intent) vs shipped

- **What (design):** The system is re-framed from "Obsidian app + worker" to a **standalone CLI-first integrity engine** with three pluggable trigger types (command / file-change scan/watch / schedule) all producing one request envelope, a pydantic-ai typed-operation runner, and Obsidian/Zotero/Hermes/MCP demoted to *optional adapters, not architectural dependencies* (design §2, §5, §10). The Co-PI interview becomes engine state (`memoria ask/answer/resume`), never a pydantic-ai session or `pydantic-graph`. **Why:** design §2 — "The engine is the product… don't make Obsidian, Zotero, Hermes, or any adapter the architecture." Heterogeneous storage "makes the engine boundary load-bearing (no trigger or surface could touch files+DB+blobs directly and stay consistent)." Rejected: workflow engines (Temporal/Prefect/Dagster/Airflow — "the worker is one queue, not a DAG orchestrator"), LangChain/LlamaIndex ("heavy framework-owned state"), and external agent-runtime ownership ("borrow useful prompt/workflow patterns, not runtime ownership") (§5, §7).

- **What (shipped subset):** The worker gained the *scaffolding* of the trigger model: `enqueue_operation`/`enqueue_trusted_write` accept `trigger_type`, `causal_refs`, `actor`, `provenance`, `schedule_id`; new CLI subcommands `scan` (enqueues a `file_change` `observe-pi-edits` pass), `run-scheduled` (enqueues a `schedule` job), and `recover`; the integrity sweep now tags its jobs `trigger_type="schedule"` (`worker.py`). **Why:** these land the §11 "Trigger unification" and "Standalone CLI" contract surface incrementally — command/file-change/schedule all funnel through `state.request_envelope`. **Note (design vs shipped):** the full pydantic-ai runner, the `memoria ask/answer/resume` interview CLI, `watch`, and the standalone `memoria <operation>` binary are **not** in this commit range — only the worker-level envelope + trigger tagging shipped. The docs still describe the Obsidian control panel as the worker's front end and note the JSON queue is mirrored "for Obsidian compatibility" (`system-artifacts.md`).

### Distribution / evaluation / security

- **What:** `.gitignore` and folder skeleton updates (above) make the SQLite DB gitignored per-machine state, and the docs now state plainly that **SQLite backup is a first-class, separate concern** from the git-backed takeout set. `on-disk-layout.md`, `configuration.md`, `operations.md` all re-labeled alpha.11 → alpha.12 with the state/audit additions. **Why:** design §3 "Backup posture (kept honest)" and §9 "Costs": "SQLite is authoritative for Memoria working state, so a full Memoria backup needs ordinary SQLite backup/snapshot handling. Alpha.12 does **not** promise full working-state portability without SQLite; it promises the user's completed knowledge and bibliography can leave." This is the honest security/durability tradeoff of the whole release, made explicit rather than hidden.

- **What (design-only, deferred):** An **optional Turso/libSQL backup/sync adapter** for the SQLite working-state DB (esp. to preserve the rollback graph and audit if the local DB is lost). **Why:** design §3/§7/§10 admit it "adds provider auth, network behavior, restore UX, compatibility limits, and free-tier/PITR terms that must be rechecked," so it is "spike-gated," "not a baseline dependency," and "not part of the ownership promise." Open question §12 D2 leaves feasibility unresolved. Nothing Turso-related shipped in this range.

- **What:** Security guardrails **carried forward unchanged** and re-affirmed: MCP-only agent sandbox (no profile ever gets file/terminal/code_execution), checks/evals pin a certified model and never route through Hermes, PI-direct-access preserved (exec-plan Guardrails). **Why:** these are alpha.11 invariants the design §8 explicitly carries; the storage rewrite must not weaken them. The write boundary stays the single choke point — "Plugin/agent/editor surfaces never write storage directly" (§6), now even more load-bearing because a surface touching files+DB+blobs directly could not stay consistent (§2).

The exec-plan's cross-cutting task — "turn §10 into ADRs — storage / engine-lifecycle / triggers-CLI / operations-runner / MCP-deferral clusters; retire the superseded alpha.11 storage/stack/operations ADRs" — was **not executed** in these commits. **Why:** not documented in the commit bodies (they are single-line: "Implement alpha.12 state boundary (#1032)"). **Inferred:** the release shipped the engine spine and its contract gates first (Phase 0 → WP1-level work) and deferred the ADR authoring/retirement, consistent with the design being explicitly labeled a "working design" that "supersedes alpha.11 §6/§8/§11" at the doc level rather than the ADR level (design §0, Provenance). ### Removed / retired

- **What:** The **files-SSOT-everywhere rule and the "passive rebuilt index" DB role are retired.** alpha.11's "one decision (files vs DB)" per component and "a passive rebuilt index over files" are replaced by the two-decision model (SSOT location ⟂ projection flavor) with SQLite as an intentional SSOT (design §0 table, §3). **Why:** the keep-test showed the old rule over-paid: forcing catalog/ops into many small JSON/markdown files bought no ownership benefit (they fail the survival test) while blocking real prevention, transactions, and graph queries (design §0, §9 "Buys").

- **What:** **The clean-rebuild migration target (alpha.11 §20) is retired** — alpha.12 has "**no migration target**; non-sandbox use starts from a fresh workspace" (design §0, exec-plan). **Why:** design §0 relaxes constraints for implementation simplicity; a single-machine, sandbox-only alpha with no backward-compatibility promise (§8) does not need a migration path.

- **What (design-level rejections, i.e. retired-before-adoption):** event-sourcing frameworks (`eventsourcing`), verifiable-log servers (immudb/Trillian/Sigstore-Rekor), Dolt (git-for-data SQL), policy engines (OPA/Rego, Cerbos), and workflow engines — all explicitly evaluated and rejected for the integrity core (design §7). **Why:** each is "the wrong shape" for a single-user local tool: policy engines "gate *authorization*; our gates are *knowledge-integrity* checks"; Dolt "adds a versioned-SQL engine and merge semantics the single-machine design does not need"; verifiable-log servers "solve cross-org cryptographic tamper-*proofing*" that "break local-first." The chosen posture: "the integrity core is a *thin composition over solid primitives* (SQLite + git CLI + Pydantic + NetworkX), plus the domain semantics no library provides — which is exactly the part that is the moat" (§7).
