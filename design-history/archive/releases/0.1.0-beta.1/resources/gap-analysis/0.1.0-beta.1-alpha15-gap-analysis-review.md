Date: 2026-07-04. Independent review of [0.1.0-beta.1-alpha15-gap-analysis.md](0.1.0-beta.1-alpha15-gap-analysis.md) (the updated version). Findings below are grounded in a re-opened source with a quote; severity is high = a delta or load-bearing fact is wrong, medium = a misattribution/overstatement that misleads, low = a real but minor imprecision.

### 1. [MEDIUM · CONFIRMED] §6 — A15/misattribution
*bullet 4 (operations declare method_class / capability manifests)*  
**Doc claim:** gap doc §6 bullet 4 A15 side: 'operations declare `method_class` (deterministic/hybrid/generative) and runner policy or fail loading; operations are product-code capability manifests' (and Appendix lists '`method_class`' among A15-present/B1-removed items).

**Problem:** The named field `method_class` with the `deterministic|hybrid|generative` enum is an alpha.14 manifest field, not an alpha.15 one. It appears in ZERO alpha.15 sources and in 0 of the 47 shipped alpha.15 operation manifests. The alpha.15 consolidation replaced the alpha.14 method_class declaration with a runner-policy manifest: shipped A15 manifests fail to load on runner/sealing violations, not on a missing method_class. Attributing method_class to A15 pins the 'or fail loading' gate to a field the A15 end-state does not contain. The higher-level delta (declared product-code manifest -> prose 'operation test') is directionally correct; only the specific A15 field named is wrong.

**Evidence:** method_class is alpha.14: scratch/releases/0.1.0-alpha.14/0.1.0-alpha.14-design.md line 332 'Every operation declares `method_class: deterministic | hybrid | generative`.' (also alpha.14 release-plan line 355 lists 'operation `method_class`' as an alpha.14 OKF-bundle deliverable, and exec-plan lines 574/1289). | `grep -rn method_class` over scratch/releases/0.1.0-alpha.15/ and main/src returns 0 hits. | Shipped A15 manifest main/src/memoria_vault/product/capabilities/operations/frame-paper.md frontmatter carries `runner: {test, live}`, `allowed_tools/allowed_paths/allowed_network`, `risk_class`, `required_checks`, `io_schema`, `prompt_version` and NO method_class; 0 of 47 operation manifests carry method_class, 47/47 carry `runner:`. | ADR-129 (scratch/releases/0.1.0-alpha.15/adr/129-layered-machine-judgment.md) lines 89-96 define the runner policy: manifests 'declare untrusted fields' and 'a manifest violating this fails to load', 'each operation pins `runner.test` and `runner.live`... the manifest is the hard ceiling' — no method_class.

**Correction:** State the A15 side as: operations are product-code capability manifests that pin `runner.test`/`runner.live` (plus allowed_tools/paths/network, risk_class, required_checks, io_schema, prompt_version) and fail to load on a manifest/sealing violation. The `method_class: deterministic|hybrid|generative` enum was an alpha.14 field the alpha.15 consolidation dropped in favor of the runner-policy manifest; it is not the A15 end-state (also remove '`method_class`' from the Appendix A15-present list).

### 5. [MEDIUM · PLAUSIBLE] §1 — delta/overstatement
*bullet 2 (no internal daemon -> resident daemon)*  
**Doc claim:** Gap statement (line 15): 'B1 adds a resident daemon; A15 explicitly had none.'

**Problem:** A15's design already ships a `serve` backend that is launched while Obsidian is open and hosts a filesystem watcher 'while serve runs' (memoria serve --http, loopback + token). B1 reuses the same `serve` verb. So 'A15 explicitly had none' glosses over serve's existence and presents a serve/backend process as a wholesale B1 novelty. The accurate delta is narrower: A15's serve is on-demand and self-described 'not a daemon'; B1 promotes it to a plugin-supervised (LSP-lifecycle), resident-by-default, two-process (supervisor+worker) daemon. (Note the offsetting truth that keeps this PLAUSIBLE not CONFIRMED: A15 explicitly disclaims a daemon and residency, and states serve can be not-running, so 'A15 had no resident daemon' has a defensible reading — the overstatement is specifically in implying A15 had no serve/backend process at all.)

**Evidence:** alpha.15 design @09bf018f: L325 'no new daemon, because `serve` is already the on-demand process'; L1383-1384 '`memoria serve --http`, loopback + token) launched while Obsidian is open -- not a daemon (§5)'; L3149 '§5 has no internal daemon'; L327 'When `serve` is not running, detection is scan-triggered'. **Correction:** State the delta as: A15 already ships an on-demand `serve` backend (loopback HTTP + file watcher, launched while Obsidian is open) that it insists is 'not a daemon'; B1 keeps `serve` and makes it a plugin-supervised (LSP-lifecycle), resident-by-default, two-process (supervisor+worker) daemon. The genuine additions are plugin supervision, residency-by-default, and the supervisor/worker split -- not the existence of a serve/backend process, which A15 had.

### 6. [MEDIUM · PLAUSIBLE] §12 — A15/overstatement
*bullet 1 (OS confinement / egress)*  
**Doc claim:** standalone (Node ≥22 + qmd), platform-agnostic

**Problem:** The parenthetical names A15's standalone runtime as 'Node ≥22 + qmd' and omits Python entirely. The A15 CLI+engine runtime is Python 3.12+ (uv-distributed, Typer CLI, stdlib sqlite3); Node ≥22 is not the runtime — it is a baseline cost dragged in only by the qmd retrieval subprocess, and the A15 design explicitly earmarks it for removal (FTS5+sqlite-vec = 'no Node'). Foregrounding a subprocess dependency as 'the runtime' and dropping the actual Python runtime misframes A15's platform footprint in the security/confinement comparison. Not fabricated (shipped alpha.15 does carry Node ≥22 because qmd ships and was not replaced — the same doc says so at line 81), so the footprint claim is partly supported; the defect is the omission of Python and the elevation of qmd's Node dependency to the headline runtime.

**Evidence:** ADR-125 (scratch/releases/0.1.0-alpha.15/adr/125-standalone-cli-engine-architecture.md) line 36: 'Runtime: Python 3.12+; uv distribution ... Typer CLI; stdlib sqlite3 ... retrieval via qmd for alpha.15'. A15 omnibus (git 09bf018f:scratch/releases/0.1.0-alpha.15/0.1.0-alpha.15-design.md) line 169 'Language baseline: Python 3.12+', line 186 'qmd ... is the last Node dependency in the baseline', §10 lines 1935/1940/1947 'a Node ≥22 runtime in the baseline ... SQLite FTS5 + sqlite-vec removes all four ... one store, no Node ... qmd is replaced and Node leaves the baseline', line 2903 'Node stays only for the qmd subprocess'. Gap doc line 90 is the defective bullet; line 81 (same doc) already notes qmd ships in alpha.15.

**Correction:** A15's standalone runtime is Python 3.12+ (uv-distributed, Typer CLI, no-ORM stdlib sqlite3), per ADR-125; qmd is the shipped alpha.15 retrieval subprocess and it — not the CLI/engine — carries the Node ≥22 baseline dependency, a cost the design explicitly earmarks for removal via FTS5+sqlite-vec. The parenthetical should read Python 3.12+ (with qmd, and its Node ≥22 dependency, as the retrieval subprocess), not '(Node ≥22 + qmd)'.
