### S7.1 — Bibliographic fidelity authority moves to full CSL-JSON in record.md

**LEAVE — deliberate and recorded** · verification: confirmed (citation audited) · confidence: high

Design §3 states the decision with its rationale in the body: 'Full CSL-JSON is serialized into record.md as a fenced code block (the derived DB is rebuildable, so bibliographic fidelity and the Zotero re-import key cannot live only in the DB)', and §11 books the overall storage shape under Converges ('SQLite authority + markdown keep-set + blobs'). This is not the 'frontmatter-only catalog' ADR-124 rejected: everything ADR-124 said belonged in SQLite — durable worker state, provider payload provenance, full-text gates, projections — stays DB-side in beta.1 (§5: works.status, jobs, the payloads store, checks), and works.csl_json remains a rebuildable projection. Expanding the alpha.12 compact survival copy to full CSL-JSON strengthens the very survival property the compact copy existed to serve (charter Success Claim 4), while the relational working state that 'fails the survival test' stays in the DB.

- **Recorded:** design.md §3 L224-226 (decision + rationale), §11 L965 (Converges: 'SQLite authority + markdown keep-set + blobs'), §5 L429-444 (two-DB-classes split)
- **History:** Alpha.12's keep-test flip (ADR-122; history L1896, L1800) moved the catalog to SQLite because the *relational catalog graph* was working state failing the survival test, while denormalizing a compact citation survival copy into keep-set files after the cross-store spike FAILED (L1776, L1804). ADR-124 (L2042) rejected a frontmatter-only catalog because 'alpha.14 needs durable worker state, provider payload provenance, full-text gates, and generated projections better represented in SQLite.' Beta.1 addresses both reasons: all of that machinery remains DB-side; only per-work bibliographic fidelity and the re-import key move to markdown.

### S7.5 — Zotero-owned citation-facing fields corrected at source, not locked vault-side

**LEAVE — deliberate and recorded** · verification: confirmed (citation audited) · confidence: high

Design §7 L606-610 is an explicit decision-with-rationale in the body, naming the rejected alternative: fields the researcher's Zotero owns 'are corrected at source (deep link to the Zotero item; the next sync confirms) rather than locked vault-side — a vault fork of bibliographic truth would diverge from the keep-test bibliography.' The alpha position it partially reverses was driven by the alpha.14 standalone-CLI premise, which the beta.1 charter itself supersedes: Zotero is a required Capture input and the null-workflow baseline, and beta.1 is a plugin+daemon (§11 item 10), not a standalone CLI. The delta does not reintroduce live-on-demand external truth (ADR-124's other rejected alternative): the correction propagates via sync into a durable vault copy (full CSL-JSON in record.md), non-Zotero works stay wholly vault-owned, and re-import never clobbers a diverged record.md (§7 L558-559).

- **Recorded:** design.md §7 L606-610 (explicit 'rather than locked vault-side' decision with rationale); charter L11 + Capture row L29 (Zotero as required input / null-workflow baseline)
- **History:** ADR-124 (history L2042) made the catalog THE citation authority because 'Alpha.14 makes Memoria a standalone CLI/engine. A required Zotero library or live Better BibTeX auto-export would violate that', and rejected live API-on-demand metadata for drift. Beta.1 removes the standalone premise by charter, and avoids the drift failure because the vault always retains its own durable CSL-JSON copy — corrections flow through sync into it rather than being read live.
