# Beta.1 retrieval-fusion spike protocol

Date: 2026-07-06. Status: **protocol only — no code written in this pass.**
This is a runnable spec for a later implementation pass to execute; it is not
itself evidence, and it is not a decision that dense retrieval will ship.

Reference-only inputs consulted (background only, never cited as evidence
below): `scratch/releases/0.1.0-beta.1/resources/gap-analysis/0.1.0-beta.1-alpha15-gap-report.md`,
`resources/redo/beta1-adopt-borrow.md`, `resources/gap-analysis/adjudication/S10.md`
— these describe a prior, now-removed "qmd" cross-encoder tier and its
decision *not* to adopt sqlite-vec on the alpha.15 fixture. That mechanism no
longer exists in the codebase (confirmed below); this protocol re-derives the
plan from the current shipped code, not from that historical account.

## 1. Pin the real baseline first

**Confirmed via direct grep and read (2026-07-06), not assumed:**

- `grep -rn -i "qmd|cross.encoder|cross_encoder|rerank" main/src/memoria_vault/ --include="*.py"` (excluding tests) returns **zero hits**. No cross-encoder or "qmd" reranking tier exists anywhere in the current shipped code — whatever the resources/ history describes, it is gone from `main` today.
- `main/src/memoria_vault/runtime/retrieval_substrate.py:10-19`:
  ```python
  SELECTED_RETRIEVAL_SUBSTRATE = "bm25"
  RETRIEVAL_SUBSTRATE_VERDICT = {
      "selected": SELECTED_RETRIEVAL_SUBSTRATE,
      "cleared_replacement_bar": False,
      "date": "2026-07-02",
      "reason": (
          "The packaged product uses deterministic checked-only BM25 search. "
          "Semantic and hybrid retrieval remain future work until a local vector "
          "pipeline ships as product code."
      ),
  }
  ```
- **Two distinct harnesses exist today — do not conflate them:**
  1. `evaluate_fts5_fixture()` (`retrieval_substrate.py:34-67`) is a **synthetic, in-memory, unit-test-only** harness: it takes an arbitrary `{documents, cases}` fixture dict and scores BM25-only recall via an in-memory SQLite FTS5 table. It is called only from `tests/test_retrieval_substrate.py` — it has no connection to a real vault or the gold-task set. Useful as a structural precedent for a parallel dense-retrieval unit-test fixture, but **not** the source of the "does this help on Memoria's real corpus" answer.
  2. `main/vault-template/system/eval/` + `eval_score.py` (`main/src/memoria_vault/runtime/subsystems/telemetry/eval/eval_score.py`) is the **real, vault-level diagnostic gold set**: 8 hand-curated tasks (`find`×2, `verify`×3, `link`×2, `extract`×2) over 5 well-known papers (Transformer, BERT, ResNet, Adam, Dropout), each with `type: eval-task` frontmatter and a `references:` gold-citekey list. `memoria eval run --workspace <vault>` dispatches the tasks; `eval_score.py --vault <vault> --from-json results.json` scores `recall_at_k` (top-`k`, default `k=3` — `DEFAULT_K` at `eval_score.py:70`, "the gold rubrics' top-3"), `support_rate`, and `fama_clean`, appending one line to `system/metrics/eval/runs.jsonl` (`METRICS_RELDIR`/`RUNS_LOG` at `eval_score.py:68-69`). **This is the real corpus and the real metric the spike must run against.**

**Baseline to beat = the current `bm25` run of the real gold set**, not a
synthetic fixture number.

## 2. Corpus

`main/vault-template/system/eval/` as ingested into a disposable test vault
(the 5-paper corpus: Vaswani et al. 2017 Transformer, BERT, ResNet, Adam,
Dropout). **Known limitation to state up front, not hide:** n=8 gold tasks is
a small sample — treat results as directional, not statistically decisive.
Once the beta.1 real seed corpus (a separate, ~8-source open research topic)
lands, extend this spike to run over it too before drawing a final
conclusion — do not finalize the retrieval decision on the 5-paper fixture
alone if the seed-corpus work has landed by then.

## 3. Baseline (control) run

1. Ingest the 5 gold-set papers into a disposable vault (`memoria` capture/
   digest pipeline, unmodified).
2. `memoria eval run --workspace <vault>` — dispatches all 8 gold tasks
   against the shipped `bm25` substrate (`SELECTED_RETRIEVAL_SUBSTRATE`
   unchanged).
3. `python -m memoria_vault.runtime.subsystems.telemetry.eval.eval_score --vault <vault> --from-json results.json`
   — records the control line in `system/metrics/eval/runs.jsonl`: per-task
   `recall_at_k`/`support_rate`/`fama_clean`, plus the aggregate mean per
   metric (`eval_score.py::score_run`, `aggregate` block).
4. This control run is the number every treatment run must beat.

## 4. Treatment

sqlite-vec + a **named** local embedding model (to be chosen and named
explicitly in the implementation pass — e.g. a small on-device sentence
embedding model consistent with the project's no-network-dependency
posture), fused with the existing BM25/FTS5 substrate via **Reciprocal Rank
Fusion (RRF)**. The implementation pass must specify concretely:
- the RRF constant `k` (a common default is 60; state whatever value is
  actually used and why),
- the candidate-pool size per leg before fusion (e.g. top-20 BM25 + top-20
  dense before RRF re-ranks to the final top-`k`),
- and must add this as a **new, parallel eval path** — a dense/hybrid
  variant of dispatch that runs the same 8 gold tasks through the fused
  retriever, producing a comparable `results.json` shape so `eval_score.py`
  scores it identically to the control.

No such dense/hybrid path exists in the codebase today (confirmed above) —
building this parallel path is real, new implementation work the spike
requires, not a config flag flip.

## 5. Queries

Each of the 8 gold tasks' `## Input` query (natural-language, PI-phrased, per
`main/vault-template/system/eval/*.md`) maps directly to one retrieval query;
its `references:` frontmatter list is the gold citekey set `recall_at_k`
scores against. No new query set needs to be authored — reuse the existing
gold tasks so the control and treatment runs are scored identically.

## 6. Metrics

- **Primary metric: `recall_at_k`** (k=3, matching the gold rubrics'
  "top-3" pass/partial/fail bands already defined per-task).
- Secondary: `support_rate` (citations resolve to real catalog citekeys) and
  `fama_clean` (no exposed superseded claims) — both should hold steady or
  improve; a treatment that raises `recall_at_k` while degrading
  `support_rate`/`fama_clean` is not a clean win.
- Latency and cost per query (wall-clock for the embedding + fusion step
  vs. the BM25-only control) — the embedding model adds real per-query cost
  that must be weighed against any recall gain.

## 7. The yes/no adoption bar

**Dense/hybrid retrieval is adopted only if:**
- it beats the `bm25`-only control's aggregate `recall_at_k` by a clear,
  pre-registered margin (the implementation pass must fix this number before
  running the spike, not after seeing results — e.g. "≥1 additional gold
  task recall-hits, given n=8" is the kind of concrete, small-n-appropriate
  bar to set) — chosen before the run, not fit to it,
- **and** `support_rate`/`fama_clean` do not regress,
- **and** added latency/cost stays within a stated ceiling the implementation
  pass names explicitly.

**Otherwise dense/hybrid stays disabled** and `RETRIEVAL_SUBSTRATE_VERDICT`
in `retrieval_substrate.py` keeps `"cleared_replacement_bar": False` — matching
the project's established "each tier must beat a declared cheap baseline or
it is disabled" discipline (the same posture already encoded in that verdict
dict today). This bar is empirical and cannot be resolved by more literature
research — only by running this protocol.

## 8. Runnable shape (what the implementation pass builds and runs)

- New harness code: a dense/hybrid retrieval path parallel to the existing
  `evaluate_fts5_fixture`/BM25 dispatch, wired into `memoria eval run` so it
  can target either substrate for a given run.
- Commands (control already shown in §3; treatment mirrors it):
  ```
  memoria eval run --workspace <vault> --substrate bm25       # control (existing)
  memoria eval run --workspace <vault> --substrate bm25+dense # treatment (new flag, to be added)
  python -m memoria_vault.runtime.subsystems.telemetry.eval.eval_score --vault <vault> --from-json <results>.json
  ```
  (The `--substrate` flag does not exist yet — it is scope for the
  implementation pass, named here so the spike has a concrete target shape.)
- Both runs land in the same `system/metrics/eval/runs.jsonl`, tagged so the
  `eval-trend` dashboard (referenced in `system/eval/README.md`) can display
  them side by side without new tooling.

## 9. Scope guard

This document is a protocol only. No dense-retrieval code, no `--substrate`
flag, and no benchmark run exist as a result of this research pass — all of
§4 and §8's "new harness code" is future implementation work this protocol
specifies, not work already done.
