# Memoria — 0.1.0-alpha.12 exec-plan

Pairs with `0.1.0-alpha.12-design.md` (this dir). Builds the alpha.12 **SQLite-backed, CLI-first integrity engine**; carries the alpha.11 spine / cycle / eval forward. Sandbox-only (`Memoria-test`); **no migration target** (fresh workspace); no production install in scope.

## Goal
Ship the alpha.12 engine — **SQLite-SSOT working state + git-tracked keep-set**, **CLI-first** with command / file-change / schedule triggers, typed **pydantic-ai** operations — running the full ingest cycle (capture → interview → digest + hub compounding → notes), **gated by the seeded-error verdict**. Obsidian rides as an **optional adapter that preserves PI-direct-access**.

## The two bets (carried from alpha.11)
1. The integrity engine makes machine-written knowledge **trustworthy** (trace + check + cascade-rollback + read-barrier).
2. The compounding wiki (digest + hub) + ZK notes produce real **knowledge-production value**.

## Settled (design §10 — don't re-litigate)
- **SQLite-SSOT** for catalog + ops; **files-SSOT** = keep-set (`knowledge/`, `capabilities/`) + tracked `references.bib`.
- SQLite-durable working state + git-durable user files; cross-store via `materialization_status` + durable replay payloads + worker recovery (not a pretend-atomic store).
- Write lifecycle `requested → stage → validate → check → DB commit → file promote → project`.
- CLI-first standalone engine; triggers unify (command / file-change / schedule) into one request envelope.
- Schema-owned Concept model (`.memoria/schemas/types/*.yaml`) + role adapters (LLM / file / DB / MCP), versioned independently.
- Journal vocab `requested … checked … materialized … resolved`; consumable = `checked` (+ `materialized` for file outputs).
- `references.bib` materialized **atomically** with bibliography-changing catalog writes; drift-checked.
- pydantic-ai runner; checks pin a certified model; Hermes a contingency only.
- No event-sourcing framework; git CLI baseline; Turso/libSQL optional backup adapter.

## Deferred (post-alpha.12)
- BYO-agent Memoria-engine **MCP API** (engine built MCP-ready; existing vault/Obsidian MCP stays).
- **Writing + coding** modules.
- Multi-device **sync**; multi-writer.
- **Turso/libSQL adoption** (spike only); **pygit2/dulwich** (only after a failing git-CLI spike); **Hermes-as-library**.

## Guardrails (standing)
- **MCP-only agent sandbox** — no profile gets file / terminal / code_execution, ever.
- **PI-direct-access** — every feature reachable from the **Obsidian UI directly**; CLI-first is the engine's *internal* contract, **not** a substitute. The Obsidian adapter must expose the full cycle.
- Checks/evals pin a **certified model**; never route through Hermes.
- **Full coverage, no sampling.**
- **Own branch/worktree before any commit** (shared checkout — this dir has concurrent sessions).

- **Gates:** crash/materialization replay · bibliography atomicity · citation survival · schema-ownership drift · boundary compatibility (existing vault MCP intact, BYO API absent) · standalone CLI (no Obsidian) · trigger unification · request-envelope contract · file-change reaction parity · git/foreign-write path · Zotero scope (no annotations).

## Phase 1 — engine spine
**WP1 — SQLite schema + worker state machine + write lifecycle**
- Goal: one operation drives `requested → … → project`; SQLite coordinates, git promotes file outputs.
- Build: catalog/ops/journal/queue/flags/rollback schema; the worker; content-addressed staging + durable materialization payloads; the 7-step lifecycle; recovery.
- Accept: op completes end-to-end; crash *after* DB commit replays file promotion from payloads; hash-only pending rows rejected; drop DB → rebuild projections from SQLite + files.
- Depends: Phase 0.

**WP2 — Constraints/oracle + checks + read barrier**
- Goal: prevention in-transaction; machine file output consumable only when `checked` + `materialized`.
- Build: FK/UNIQUE/CHECK/DAG (staged-aware); the check suite; `check_status` / `materialization_status`; the `consumable_outputs` view.
- Accept: source+digest in one op validates (staged-aware); unchecked/unmaterialized never consumable; PI live edit visible to human, filtered from machine consumers.
- Depends: WP1.

**WP3 — Cascade rollback**
- Goal: revert any output + descendants across stores.
- Build: recursive-CTE blast-radius over the rollback graph; inverse SQLite txn + inverse git commit; machine-auto-revert / human-flag routing.
- Accept: rollback a source → derived digests/notes/hubs reverted or flagged; blast-radius spans DB- and file-backed Concepts.
- Depends: WP1, WP2.

## Phase 2 — triggers + operations
**WP4 — Triggers + request envelope + CLI**
- Goal: command / file-change / schedule all enqueue **one** request envelope; CLI is the primary surface.
- Build: the envelope (`request_id`, `trigger_type`, `operation_id`, `args`, `idempotency_key`, `target_path/hash`, `causal_refs`, `actor`, `provenance`, `schedule_id?`); `memoria <op>`, `memoria scan` (watch later), `memoria run-scheduled`; path/content/type classification; provenance-based loop suppression + foreign-write quarantine.
- Accept: all three triggers produce identical envelope + lifecycle; CLI runs with **no Obsidian** (no env vars, no Obsidian MCP); foreign commit on a generated path quarantined, on a keep-set path imported + re-checked.
- Depends: WP1, WP2.

**WP5 — Operations + runner (pydantic-ai) + interview**
- Goal: typed operations behind the runner; schema-owned model + adapters; CLI-addressable interview.
- Build: direct-API runner; adapters generated/checked vs `.memoria/schemas/types/*.yaml`; operation contract; `memoria ask / answer / resume` (suspend/resume = engine state, not pydantic session/graph).
- Accept: typed I/O round-trips; LLM-adapter bump doesn't touch file/DB shape; interview suspends + resumes via CLI.
- Depends: WP1.

## Phase 3 — catalog + knowledge
**WP6 — Catalog (SQLite tables + references.bib + Zotero)**
- Goal: catalog in SQLite; bibliography takeout atomic; import the ~800-paper library.
- Build: `sources` / `entities` / `catalog_links` tables; `references.bib` materialization (atomic with biblio-changing writes) + drift check; Zotero import (bibliography/content only, **no annotations**); `source_id` stability.
- Accept: import N → SQLite rows + materialized `references.bib` (drift-clean); a biblio-changing write can't complete until `references.bib` materialized; restore-from-SQLite-backup recovers the catalog graph.
- Depends: WP1, WP5.

**WP7 — Knowledge graph + ingest cycle**
- Goal: digest/note/hub + capture → interview → compound → notes.
- Build: the three node-kinds; compact self-describing `citations` (`source_id`, `title`, `authors`, `issued`, + one locator); the ingest spine (interview-gated digest; hub compounding 5–15 nodes); gap analysis over both graphs.
- Accept: ingest a source → interview → digest + hub updates, all checked + materialized + grounded; citation standalone-meaningful after dropping SQLite; gap re-analysis runs.
- Depends: WP2, WP5, WP6.

## Phase 4 — surfaces + gate
**WP8 — Surfaces (CLI primary + Obsidian adapter)**
- Goal: full cycle from CLI; Obsidian adapter preserves **PI-direct-access**.
- Build: CLI surface over the operations; the Obsidian adapter (Co-PI pane, the panels, read `knowledge/` directly, calls the engine); both over the worker boundary, never touching storage directly.
- Accept: **every alpha.12 feature reachable from Obsidian directly**; CLI parity; plugin smoke.
- Depends: WP5, WP7.

**WP9 — Verdict gate (seeded-error)**
- Goal: certify the engine before real use.
- Build: the seeded-error harness; baseline; the verdict bar (alpha.11 §14).
- Accept: seeded errors caught at the bar; engine certified; gateless-to-real-use criteria met.
- Depends: WP1–WP8.

## Cross-cutting
- **Security:** MCP-only sandbox enforced (gate plugin hard-deny); capability trust manifest.
- **Eval/baseline:** the verdict harness + baseline discipline (alpha.11 §14).
- **Turso spike (optional):** backup/restore for the SQLite working-state DB incl. the rollback graph + audit; recheck terms; **never a baseline dependency**.
- **ADRs:** turn §10 into ADRs — storage / engine-lifecycle / triggers-CLI / operations-runner / MCP-deferral clusters; retire the superseded alpha.11 storage/stack/operations ADRs.

## Critical path
`Phase 0 → WP1 → WP2 → {WP3, WP4 ∥ WP5} → WP6 → WP7 → WP8 → WP9`

## Top risks
- **Cross-store materialization recovery** (DB-commit-then-file-promote) — the trickiest correctness surface; covered by the WP1 crash test + the §11 replay gate.
- **CLI-first vs PI-direct-access** — the Obsidian adapter must expose the *full* cycle or the standing PI rule breaks; build it alongside WP8, not after.
- **Git-commit-per-op throughput** on bulk Zotero ingest (~800 papers) — batch commits; parallelize compute, serialize promote.
- **pydantic-ai structured-output reliability** on the local test model — the runner abstraction lets you swap; checks pin a certified model.
- **SQLite backup discipline** — working state (catalog/ops/rollback) is DB-only and *not* in the takeout promise; ordinary backup matters (Turso optional).
