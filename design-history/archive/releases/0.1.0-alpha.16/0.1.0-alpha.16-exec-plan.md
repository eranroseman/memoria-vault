## 1. Purpose / big picture

alpha.16 makes the **basic knowledge cycle run end-to-end on one real project**: open a project → seed from Zotero/DOI/PDF/GitHub → capture (immutable raw + content + metadata) → extract + opt-in digest → process annotations into claim/question notes → connect via source-notes and hubs → gap analysis over **both** graphs → Ask with cited spans + honest denominators → a minimal structured export — **without data loss and without presenting machine output as verified truth.** It is the READ cycle plus the substrate the beta.1 WRITE modules (writing, coding) will stand on. "Done" = the requirements acceptance checks pass on a real corpus (§5).

**The gap to alpha.16 (design → build):** the vault layout, identity, concept model, retrieval, and integrity all move from the alpha.15 shapes to the alpha.16 design. This is an **alpha.15-forward evolution**, not a rewrite: keep the Python engine, the `memoria` CLI, the seeded-error/dogfood test discipline, and the read-API idea; change the vault contract, add the two-graph model + source-notes + claim-context + hubs-as-tag-membership, FTS-first retrieval, the run/VaultWriter/revert substrate, Guard/Ledger/exfil-linter, and the Obsidian Ask/Review panes.

**Two naming/shape decisions the plan pins (defaults chosen; owner may override):**
1. **CLI stays `memoria`** (alpha.15-forward continuity; renaming the shipped command to the design's `mem` is a gratuitous breaking change). Read every `mem …` in the design as `memoria …` for alpha.16.
2. **Vault is the folder at the user-defined path** (owner-confirmed 2026-07-05): Memoria-born, never a `Memoria/` subfolder, no nesting — the folder created by `memoria init` *is* the vault (R-8; Obsidian discourages vaults-inside-vaults). Design §2 and the requirements file-contract are reconciled to this. WS-A implements it plus an importer for an existing alpha.15 vault (migration).

## 5. Validation and acceptance

**The acceptance gate is the requirements doc's two lists — nothing else** (the design carries no separate acceptance mirror). Realized as one runnable dogfood plus per-workstream L1 tests:

- **`test_alpha16_dogfood_checkpoint.py`** (extends the alpha.15 pattern) drives a real seed corpus on `~/memoria-vault/sandbox` through the full cycle and asserts each **functional check 1–10** and **technical substrate check 1–10** from the requirements. This is the release gate.
- **Per-workstream L1 tests** assert the mapped rows in §3's table (e.g. WS-B: every machine write carries a `run_id`; a run reverts leaving human bytes identical; `memoria rebuild` fails closed to `unchecked`).
- **Standing negative/​boundary tests** (WS-G): Guard denies a non-allowlisted host and logs it; a worker holds no provider key; the exfil linter neutralizes a beacon in an extracted PDF; sealing makes a manifest that interpolates raw source text fail to load.
- **Host-neutrality proof (WS-E/H):** the entire Ask/Review acceptance runs through `memoria ... --json` (read-API) with **no Obsidian**, proving acceptance does not depend on the open host/stack decision.
- **Retrieval gate (WS-E):** Ask acceptance passes on **FTS/BM25 alone**; sqlite-vec turns on only after a committed benchmark shows it beats the lexical baseline on the real corpus.
- **CI:** `python scripts/verify pr` green; `lint`/`python-selftest`/`cspell`/`markdownlint`/`pr-policy` pass on every PR.

## 6. Idempotence and recovery

- Each workstream is one squash-merged PR → `git revert <sha>` restores prior state; the schema bump and vault-template rewrite carry a **migration + `user_version`** so a re-run detects and no-ops.
- The dogfood runs against a **disposable sandbox vault** (`~/memoria-vault/sandbox`), never a real corpus; re-running rebuilds it from the seed.
- `memoria rebuild` and the backup/restore matrix (design §5) are themselves the recovery path and are tested (WS-A/B).
