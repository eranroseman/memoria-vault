### S4.1 — Universal frontmatter re-scoped from every-Concept to every-Memoria-created-file at the Memoria/ bundle boundary

**LEAVE — deliberate and recorded** · verification: confirmed (citation audited) · confidence: high

The beta.1 charter itself sets the bar at exactly this scope: the Knowledge-format row requires 'required typed frontmatter on every Memoria-created file' with OKF v0.1 bundle conformance and 'no regression of the exchange format' — the re-scope is charter-defined, not silent. Design §3 records the two scoping rules explicitly ('the bundle boundary is Memoria/, not the researcher's vault'; 'the conformance guarantee is every Memoria-created file, always; freehand files converge via proposals') and states the absence-of-mem-*-markers = human rule with its zero-migration rationale. The alpha reason for universality (consumers need guaranteed fields for standing filtering and links traversal) is served DB-side in beta.1, and the regression review verified OKF bundle conformance held. Vault-wide universality is structurally incompatible with beta.1's zero-migration commitment to the user's pre-existing notes.

- **Recorded:** Requirements charter, Required Product Scope 'Knowledge format' row (0.1.0-beta.1-requirements.md L36); design §3 L232-262 (bundle-boundary scoping rules and absence-of-markers rule with zero-migration rationale)
- **History:** Universality arrived at alpha.14 (PR #1157, commit 88b7f0f5): id/standing/links enforcement on every Concept because OKF Knowledge Bundle contracts, standing-based supersession filtering, and links traversal 'have no guaranteed field to read' without it (history L1916-1922). Alpha.15 retired #1157's field set the same day while keeping universality with meaning-only fields (L2086, L2290-2292). Beta.1 addresses the motivating need differently: filtering/traversal consumers are DB-side, OKF conformance holds at the bundle boundary, and the human-span manifest carries authorship safety — the regression review's sweep confirms the exchange format did not regress.

### S4.9 — In-file provenance/verdict layer (mem-prov/mem-taint/mem-flags/mem-run + %%prov%% block markers) re-added after alpha.15 eliminated all in-file verdict state

**LEAVE — deliberate and recorded** · verification: confirmed (citation audited) · confidence: high

The delta is doubly recorded: §4 designs the in-file marker layer with explicit rationale (sidecars rot under stock-Obsidian edits; in-file markers travel with the text; the hash manifest backstops), and §18(1) adjudicates the exact ADR-126 hazard — 'in-file mc is a claim, not authority': the verdict of record is the DB checks row bound to block_sha256, green requires marker + current hash + row to agree, and mem rebuild/restore fails closed to amber and re-earns via the local Checker. Both documented reasons ADR-126 rejected an in-file projection are answered: the forged-status foot-gun is neutralized because a hand-written or restored marker cannot render green without a matching DB row and current hash, and the hash-churn/rewrite hazard on human notes is structurally absent because markers exist only on machine-created content (absence of markers = human) so checking never rewrites a human file — a human edit under a marker demotes to amber and enqueues re-check. Taint stays monotone and never-green (§4), and mem-flags mirrors a DB column (works.distrust_flags) rather than replacing it.

- **Recorded:** Design §18(1) L1282-1286 (regression R1 resolution) + §4 L294-338 (no-sidecar rationale; marker-is-claim/DB-is-authority; fail-closed rebuild); §5 L384-385 (checks table as mc-authority)
- **History:** ADR-126 rejected even a generated-but-ignored in-file check_status projection as 'hash churn plus a forged-status foot-gun,' rooted in the deeper failure that 'check_status lived in frontmatter, so checking a note rewrote it, hand-edits forged status, and every consumer had to distrust the file it was reading' (history L2086, L2292). Beta.1 initially re-opened exactly this (regression review R1, HIGH: ProvenanceRenderer painted green straight from the file marker, contradicting §11) and then closed it with the checks-row + block-hash + fail-closed-to-amber mechanism — the failure is addressed, not merely tolerated.
