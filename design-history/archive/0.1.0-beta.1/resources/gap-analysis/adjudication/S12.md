## 12. Security / OS confinement

3 gaps — leave 3 · record 0 · change 0

### S12.1 — OS-enforced egress (Linux CLONE_NEWNET netns) is Linux-only; seccomp/macOS sandbox-exec/Windows-Firewall flows cut

**LEAVE — deliberate and recorded** · verification: confirmed (citation audited) · confidence: high

The cut is explicitly recorded with rationale in the design body: §10's 'OS confinement: Linux network-namespace only' bullet names the three cut mechanisms and why ('two of them bought near-zero enforcement for real code and a 3-OS sandbox matrix'), mandates SECURITY.md state the per-OS truth, and §10's SECURITY.md bullet carries 'on macOS/Windows, no OS egress enforcement' as a named honest residual; §11 back-pressure item (6) records brokered egress + Linux netns as the enforceable form of the network allowlist, among the items that 'sharpen mechanisms alpha.15 already builds'. On macOS/Windows beta.1 equals — never regresses — the alpha line, which had zero OS-kernel enforcement on any platform at any point (grep of the full history finds no seccomp/namespace/firewall mechanism), and the regression review sweeps brokered fail-closed egress as present/strengthened. One editorial nit outside this gap's scope: §10's claim that netns is 'literally the charter's network namespace wording' does not match the current charter, which contains neither 'network namespace' nor 'allowlist' — the recording of the cut stands regardless.

- **Recorded:** Design §10 'OS confinement: Linux network-namespace only' bullet (0.1.0-beta.1-design.md L904–913) + SECURITY.md residual list (L955–961); design §11 back-pressure item (6) (L982–983); regression review 'Checked and intact' sweep (0.1.0-beta.1-alpha-regression-review.md L73–74).
- **History:** Alpha's application-level egress boundary was earned through repeated application-layer fail-open failures, never through OS enforcement: the alpha.1 shell hook never fired on live writes (ADR-27/28, history L306–310); alpha.9 recorded egress allow-by-default as a live P0 with the poisoned-paper injection threat, plus the 'merged ≠ deployed' gate and a second fail-open on plugin registration (history L1400); alpha.14 #1103 hardened allowlist matching fail-closed (L1964); alpha.15 ADR-125 made fail-closed an architecture invariant — 'a gate that proceeds on its own failure is not a gate' (L2202). Beta.1 addresses that recurring lesson more strongly than alpha ever did: engines hold no sockets structurally, Guard fails closed on policy-load failure with workers refusing to start, and on Linux the kernel netns enforces below the application layer where every alpha failure lived; the honest per-OS residual mirrors alpha's own ADR-46 correction of 'completely sandboxed' over-claiming (L669).

### S12.2 — Write-time exfiltration linter + unified canary spec/drill + per-OS keychain (vs alpha's structural no-exfil, diagnostic-plane self-test, bare loopback token)

**LEAVE — deliberate and recorded** · verification: confirmed (citation audited) · confidence: high

Every facet of the delta is adjudicated in an authoritative record and all three mechanisms strengthen rather than drop invariants. The linter is recorded in the §11 convergence list ('the exfiltration linter as mechanized don't-amplify') and fully specified in §10 with one owner and write-time+export application; the regression review sweeps it as present/strengthened. The canary is an explicit §10 decision with rationale ('two incompatible canaries existed' → one spec owned by Security, weekly + post-migration drill, INC-canary incident response) serving the charter Trust row 'Canary checks'. The keychain is a §10 decision serving the charter Ops row 'Key handling', and its newness is governed by the recorded §13 BYOK/agent-boundary decision — the alpha line held no long-lived provider key to protect, so no precedent applies. The bullet's parenthetical instrumentation silence (evaluation protocol schedules no canary-drill acceptance test) is not a delta from alpha.15 — the drill's weekly + post-migration schedule and incident response are mandated in the governing design §10 itself, and the charter's evaluation scope owns retention measurement, not security ops.

- **Recorded:** Design §11 convergence list (0.1.0-beta.1-design.md L965–972, exfiltration linter); design §10 linter bullet (L914–922), keys/canary bullet (L948–954); design §13 BYOK/LLM-allocation record (L998–1058); charter Required Operations 'Key handling' row (0.1.0-beta.1-requirements.md L65) and Required Trust 'Canary checks' row (L76); regression review sweep (0.1.0-beta.1-alpha-regression-review.md L73–74).
- **History:** ADR-105 (alpha.8) built the content-light diagnostic plane with a golden-corpus redaction self-test because logs were 'the first place vault-derived content could land' and 'a forgotten debug flag plus a routine git push exfiltrates unpublished research' (history L1286); alpha.11 made no-exfil structural via channel separation — agents hold no network or write authority (L1741); alpha.15 ADR-125/130 kept no-agent-tool-grants with the loopback bearer token and documented the residual 'any local bearer credential is readable by a trusted-local agent' (L2204). Beta.1 addresses each motivating concern by generalizing, not dropping: the linter operationalizes no-exfil as containment-at-write over all machine-writable content including extracted fulltext, the canary drill extends the ADR-105 self-test from the diagnostic plane to all artifacts/DB/logs/exports, and the keychain closes the bearer-token-era readable-credential residual for the new BYOK long-lived key alpha never held.

### S12.3 — Third-party-plugin interference revived as a named security corruption class (structurally absent in alpha.14/15 after Obsidian was dropped)

**LEAVE — deliberate and recorded** · verification: confirmed (citation audited) · confidence: high

The revival is a recorded docs-history borrow, which counts as an authoritative record: the B/A/R Borrow table's 'Hostile-plugin defense' entry (target beta1) specifies exactly the §10 mechanism set — startup detection of known-hostile plugins/settings, registering machine-owned paths as formatter-excluded, auditing every plugin's data.json for secret fields before it enters backups — and design §10 implements it nearly verbatim with the human-span manifest gate named as structural backstop. The enabling delta (Obsidian back as required UI) is itself recorded as §11 back-pressure item (10), 'the host decision itself'. Even the no-alpha-precedent facets are covered: the data.json secret audit is inside the borrow entry's text, and the T3 key-leak coupling follows from the recorded §13 BYOK decision. Nothing is relitigated: the class returns only because the surface returns, and the defense is stronger than the alpha era's prohibit-or-detect-post-hoc posture.

- **Recorded:** Docs-history B/A/R Borrow table, 'Hostile-plugin defense' row (0.1.0-beta.1-docs-history-borrow-adapt-reject.md L44, target beta1); design §10 third-party-plugin bullet (0.1.0-beta.1-design.md L923–929); host pivot recorded as design §11 item (10) (L987–989).
- **History:** In the alpha.1–11 Obsidian/Hermes era the threat was real and handled by prohibition plus post-hoc detection: ADR-12 ruled obsidian-linter 'incompatible — do not install' because 'a deterministic tool still fails the architecture if it writes outside the Policy MCP / audit trail' (history L495); vault-hash-drift caught out-of-band edits only after the fact because Hermes hooks fail open (ADR-25/49, L495); ADR-74 (alpha.8) added the plugin-provenance lock against unprovable plugin artifacts (L1320). Alpha.14/15 dissolved the threat by removing Obsidian from the required product entirely (L2320). Beta.1 re-creates the surface via the host pivot and addresses the era's underlying failure (third-party writers bypass application gates; post-hoc-only detection) with proactive startup detection, formatter exclusion, the data.json secret audit, and the preventive human-span pre==post manifest gate with auto-rollback that the Obsidian era lacked.
