### S3.1 — Four typed-concept schemas replaced by flat file-class enum; per-type schema files dropped

**LEAVE — deliberate and recorded** · verification: confirmed (citation audited) · confidence: medium

The enum replacement is stated as an explicit decision in the design body (§3: every Memoria-created file carries a `type` from the closed class enum, grounded in OKF §9.2 typed frontmatter), and the load-bearing half of the delta — losing the reject-undeclared per-type validator — is adjudicated with rationale in §18(6): OKF-permissive toleration is deliberate, machine writes stay conformant by construction because VaultWriter serializes from typed fields in product code, which is where the per-type contract now lives. The alpha position was earned by type-zoo accretion ('the multiplicity itself caused drift and confusion'), a failure that cannot recur in a regime with no per-type schema files to accrete lifecycle fields and per-type rules into. The un-named residue — the schema files as artifacts — is plainly covered by §18(6), since validation was the files' function; the charter's knowledge-format bar (required typed frontmatter, OKF v0.1 conformance, no exchange-format regression) is met. Owner position R-6 (every md file a Concept with defined OKF frontmatter+body contracts; this drop a 'first sign beta.1 was sloppy') disagrees and is flagged for the dedicated re-examination pass, but it names no re-opened failure, lost safety invariant, or charter contradiction.

- **Recorded:** design.md §3 (L232-267: class enum stated with OKF §9.2 rationale) + §18(6) (L1303-1312: strict-reject validator declined, conformance-by-construction mechanism named); regression-review R6 + Resolution 2026-07-04 (L99-101)
- **History:** Per-type schemas ran continuously from the pre-alpha.1 Old Skeleton; the alpha.15 collapse to four types (ADR-126 as pointer) was earned by accretion damage — '12+ types across three bundles... each with folder homes, lifecycle fields, and per-type rules. The multiplicity itself caused drift and confusion' (design-history L2074, L2284-2286) — and closed validation guarded the meaning layer against undeclared-field drift. Beta.1 addresses the accretion failure structurally (no schema surface to accrete into; typed contracts live in VaultWriter's serialization code) and trades the strictness half deliberately per §18(6), so the motivating failure is addressed by different machinery, not left open.

### S3.5 — ULID identity dropped; identity re-derived from citekey + file path

**CHANGE — needs design change (severity: medium)** · verification: contested (1/3 skeptics dissent) · confidence: high

Disposition decided by owner hard ruling R-1 (owner-rulings 2026-07-05, L16-37): a mutable key cannot be identity, and the concrete failure case is native to beta.1's own design — BBT-derived citekeys change on metadata correction, so re-import of the same work either mints a duplicate works row/directory or defeats the idempotent upsert, which is why §7 already stores the Zotero item key as a second, stable key: two identity systems with the mutable one primary. The machine adjudication, preserved as a footnote per the ruling, would have been 'record': the drop is fully silent (not in R1-R6, the fourteen divergences, or §11), but §7's pin-at-import plus the revertible rename run do implement the docs-history citekey-stability borrow that documented the drift failure. The ruling's point stands on the merits regardless: pinning mitigates but does not address the alpha.14/15 reason (identity decoupled from mutable keys and paths so corrections never change identity), and beta.1 reverses 'citekey change no longer renames the Work' — a correction now cascades through every citekey-keyed table (works, chunks, anchors, edges, annotations, checks) and every vault path in one rename blast radius.

- **Recorded:** nowhere - silent (only embodied in design §5 L374 `works(citekey PK)` and §7 L569-577 rename-run; the drop itself is adjudicated in no authoritative record). Owner hard ruling: gap-analysis/owner-rulings-2026-07-05.md R-1 (L16-37)
- **History:** ADR-124 (alpha.14, design-history L2042) made stable Work IDs primary with citekeys as aliases because a standalone engine cannot hang identity on a desktop reference manager's mutable keys; ADR-126 added ULID id — 'identity ≠ filename ≠ title... so renames/moves don't break references' (design-history L2088). The documented failure behind pinning: 'unpinned keys drift on metadata correction and orphan the works/<citekey>/ folder, every wikilink, and the BibTeX keep-test output' (docs-history L40). Beta.1 adopts the pinning fix but re-mounts identity on the mutable key, so the decoupling reason is mitigated, not addressed — the owner's re-import case shows the residue is live.
- **Action:** Restore a stable machine identity (ULID/work_id-class) as the primary key for works and all internal references — the DB tables (works, chunks, anchors, edges, annotations, checks), the manifest, and anchor/citation resolution — with `citekey` demoted to a pinned human-facing alias; a citekey correction updates the alias and may rename the directory as presentation, but never changes identity, and re-import idempotency keys on the stable ID (zotero_key becomes a provider alias, not a second identity system). Record the decision in design §5/§7 and name the reversal in the §11 divergence list.
- **Verifier notes:** No charter row demands a stable machine ID: the owner's concrete failure case is mitigated inside required scope (pin-at-import, zotero_key idempotent upsert, exact-identifier auto-merge with needs_metadata on conflict — design §7 L554-579), the DB-side cascade lands entirely in the derived store the charter's Rebuild row (requirements L63) declares regenerable, and the vault-side rename cost persists under ULIDs anyway since the proposal keeps directory renames as presentation. An architecture-wide re-keying of seven tables, anchor resolution, and the manifest (which is path-keyed over non-work human files that have no work_id) is over-machinery for a one-researcher retention experiment (requirements L9-11), and the Upgrade row (L61, exercised lossless migrations) keeps a post-beta.1 work_id migration open at no risk. What the charter's own discipline does require is that the reversal stop being silent — the drop is recorded nowhere, violating the name-every-divergence practice the regression review enforces (alpha-regression-review L88-90), and the preserved machine adjudication itself reached 'record'. Owner hard ruling R-1 forces 'change' procedurally, but R-1 is a design-quality judgment outside the charter, and the charter is the spec.
