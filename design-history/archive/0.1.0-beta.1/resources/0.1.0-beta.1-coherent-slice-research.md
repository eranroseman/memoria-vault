be read as validating alpha.17's existing default without qualification.

## Findings

- **SCAR (arXiv 2606.16661, Langlois, Jun 2026) gives a concrete, formalized
  anchor-expansion policy**: a neighboring chunk `n` is appended to the
  context around retrieved chunk `c` if its expansion score exceeds a
  relative threshold tied to the chunk's own query-relevance — `S(c,n) >
  γ·cos(e_q,e_c)`, with a continuity-adjusted relevance term `S_{c,n} =
  cos(e_q,e_n) − λ·b_{c,n}` (`b` penalizing a chunk boundary that splits a
  coherent logical unit, e.g. a table header from its rows). (confidence:
  high, vote: 3-0, independently confirmed via PDF, HTML, and abstract-page
  fetches)
  - Evidence: verified directly against the paper's Section 4.2 ("Expansion
    Decision") by three independent voters, one reading the PDF directly via
    `pypdf`.
  - Source: arXiv:2606.16661
  - Beta.1 implication: this is a genuine, parameterized alternative to
    alpha.17's simpler "strongest anchor top/bottom, capped" heuristic — a
    continuity-aware threshold (γ, λ) could refine *when* to expand an
    anchor, worth prototyping against alpha.17's existing default rather than
    replacing it outright without a comparison.

- **The same SCAR paper's own efficiency table (N=160 boundary-fragmented
  queries) shows whole-section "Parent Retrieval" achieves the highest
  accuracy (0.968) but at far higher token cost (50.2 tokens/query-ish scale)
  than SCAR's own selective expansion (0.928 accuracy, ~7.84 cost) — a
  real, quantified accuracy/cost tradeoff between whole-section and
  selective-anchor-expansion.** (confidence: medium — the raw numbers are
  confirmed exact by all 3 voters, but flagged by the majority as a
  self-reported comparison from the paper proposing the cheaper alternative,
  so treat the framing with the caution due any single-paper benchmark; vote:
  2-1, with the dissent specifically about source strength, not a factual
  error)
  - Evidence: confirmed exact table values across independent PDF/HTML
    re-reads.
  - Source: arXiv:2606.16661, Table 3
  - Beta.1 implication: directly validates alpha.17's own accuracy-vs-budget
    tradeoff reasoning for choosing "whole-note if it fits the budget, else
    section-expanded-anchor" — whole-section genuinely does cost more but can
    measurably win on accuracy, so the budget-gated choice (not a blanket
    preference for either) is the right shape, consistent with a real
    quantified precedent.

- **Fragmented top-k retrieval measurably degrades downstream synthesis when
  it splits a coherent logical unit** (e.g. a table header from its rows, a
  definition's two halves) — SCAR's own problem statement documents this
  concretely. (confidence: high — majority-confirmed 2-1; the dissenting
  voter agreed the quote is accurate but flagged that the claim generalizes
  the specific example into a broader assertion the source doesn't itself
  make — treat the *mechanism* as confirmed, the *generality* as unconfirmed)
  - Evidence: confirmed against SCAR's Section 2 problem statement.
  - Source: arXiv:2606.16661
  - Beta.1 implication: the specific failure mode (bifurcated logical units)
    is real and worth testing for directly in Memoria's own slice-
    construction eval, rather than assuming all fragmentation degrades
    quality equally.

- **Standard/naive top-k RAG specifically fails on *global, corpus-wide*
  questions (e.g. "what are the main themes in the dataset?") because these
  are query-focused summarization tasks, not explicit retrieval tasks** — the
  GraphRAG paper's own stated motivation. (confidence: high, vote: 3-0,
  independently confirmed against the paper's abstract by all three voters)
  - Evidence: verified verbatim against the abstract of Edge et al. 2024
    (Microsoft Research), "From Local to Global: A Graph RAG Approach to
    Query-Focused Summarization."
  - Source: arXiv:2404.16130
  - Beta.1 implication: this is a narrower, well-scoped claim than the
    "GraphRAG substantially outperforms top-k RAG" claim refuted below — it
    supports building a *separate* mechanism for corpus-wide/thematic
    questions distinct from per-claim coherent-slice drafting, not a
    wholesale replacement of the slice-construction policy.

- **Chunking/boundary strategy has a statistically significant effect on
  RAG-based *code* completion quality; function-level chunking specifically
  underperformed by 3.57-5.64 percentage points Exact Match (Cliff's delta
  -1.0, a maximal non-overlapping effect) on RepoEval.** (confidence: high,
  vote: 3-0, independently confirmed via arxiv.org/abs and arxiv.org/html
  fetches plus a corroborating web search)
  - Evidence: verified against Wu, Gong, Jahangirova, Zhang (King's College
    London), "How Does Chunking Affect Retrieval-Augmented Code Completion?
    A Controlled Empirical Study."
  - Source: arXiv:2605.04763
  - Beta.1 implication: this is about *code* completion specifically, not
    prose synthesis — directly relevant if/when the (currently excluded)
    coding module lands, but should not be over-applied to the text-drafting
    slice-construction question this research was primarily scoped for.

- **RAPTOR departs from flat-chunk retrieval by recursively embedding,
  clustering, and summarizing chunks into a tree with multiple levels of
  summarization**, specifically to address the "most existing methods
  retrieve only short contiguous chunks... limiting holistic understanding"
  failure mode. (confidence: high, vote: 3-0, both voters independently
  quoting the same abstract passage)
  - Evidence: verified against Sarthi et al. 2024, "RAPTOR: Recursive
    Abstractive Processing for Tree-Organized Retrieval."
  - Source: arXiv:2401.18059
  - Beta.1 implication: a hierarchical-summary construction is a heavier
    alternative worth naming as a future option if alpha.17's simpler anchor-
    expansion approach proves insufficient on real drafting tasks, but is not
    itself evidence that the simpler approach is wrong.

- **LlamaIndex's production AutoMergingRetriever gates its leaf-to-parent
  consolidation on a threshold** (merges child nodes into their shared parent
  only once the fraction of children retrieved for that parent exceeds a
  configurable value), rather than merging unconditionally. (confidence:
  high, vote: 2-0, 1 errored)
  - Evidence: confirmed against LlamaIndex's own documentation.
  - Source: docs.llamaindex.ai (Auto Merging Retriever)
  - Beta.1 implication: a real, shipping precedent for "expand to the parent
    only when enough of the parent is already relevant" — a concrete,
    implementable rule alpha.17's design could adopt as its own expansion
    trigger if a simpler fixed-anchor heuristic proves too blunt.

- **On NarrativeQA (a dataset with dispersed/synthesis-style answers spread
  across a long document), larger retrieved context measurably helped more
  than on datasets with localized answers.** (confidence: high, vote: 2-0, 1
  errored)
  - Evidence: confirmed via the chunk-size sizing study.
  - Source: arXiv:2605.04763 / arXiv:2505.21700 (chunk-size sizing studies)
  - Beta.1 implication: supports a *task-adaptive* budget, not a fixed one —
    Memoria's slice construction may want a larger default budget
    specifically for synthesis-heavy draft sections vs. narrow factual
    lookups, echoing alpha.17's own "cheap Self-Route to fragments for narrow
    factual nodes" design.

## Explicitly refuted — do not restate

1. **"Long-context (i.e., large, coherent, non-fragmented context)
   consistently beats RAG's isolated-chunk retrieval on average task
   performance"** (vote 0-2) — too strong/general a claim; not supported as
   a consistent, unconditional result by the cited source.
2. **"Graph-based global/community-summary approaches... substantially
   outperform naive top-k vector RAG... providing direct empirical evidence
   that coherent, structured context beats isolated fragment retrieval for
   integrative writing"** (vote 0-3) — the raw win-rate percentages (72-83%
   comprehensiveness, 75-82% diversity) are accurately quoted from the real
   GraphRAG paper, but all three voters agreed the claim's broader inference
   overreaches what that specific comparison (community summaries vs. naive
   RAG on query-focused *summarization*) actually establishes about
   "coherent vs. fragmented" retrieval generally. The narrower, confirmed
   version of this finding is listed above ("Standard RAG fails on global
   questions").
3. **"Reordering retrieved chunks to match their original document position...
   substantially improves answer quality, and the gap widens as more chunks
   are retrieved"** (vote 0-3) — refuted; do not cite this ordering-effect
   claim as established.
4. **"A coherently-ordered, budget-limited context (48K tokens) beat feeding
   the model the entire long document (117K tokens)"** (vote 1-2) — refuted;
   do not use this specific 48K-vs-117K comparison as evidence for capping
   slice size well below "whole document."
5. **"Token budget size is a stronger driver of output quality than either
   chunk size or retriever choice"** (vote 0-3) — refuted as stated.
6. **"The generation model tolerates inclusion of lower-relevance/mixed-
   relevance material... expanding the context window... does not hurt and
   can help"** (vote 0-3) — refuted; do not use to justify looser relevance
   filtering when building a slice.
7. **"Replacing isolated top-k fragments with auto-merged parent context
   reduced 6 disconnected retrieved nodes down to 3 consolidated nodes"**
   (vote 0-2) — the specific 6-to-3 count claim is refuted; the underlying
   AutoMergingRetriever mechanism (confirmed finding above) still stands.

**Unverified — re-check before relying on these (rate-limit-errored votes,
not content disagreement):** summarizer performance drops on retrieved vs.
gold documents (arXiv:2212.10526); summarizer sensitivity to irrelevant
content vs. insensitivity to duplicates/order (same paper); multi-hop
datasets (HotpotQA/2WikiMQA/MuSiQue) as the specific failure mode for
fragment-based RAG (arXiv:2407.16833); the precise AutoMergingRetriever
merge mechanism description (LlamaIndex docs); "most existing
retrieval-augmented LM methods retrieve only short, contiguous chunks"
(arXiv:2401.18059, RAPTOR's own framing — likely true given the RAPTOR
finding above quotes similar language, but not independently re-confirmed
here).
