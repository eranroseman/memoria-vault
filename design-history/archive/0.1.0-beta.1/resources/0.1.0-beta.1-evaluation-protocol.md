# Memoria — 0.1.0-beta.1 evaluation protocol

Date: 2026-07-04
Status: **owning design artifact for the charter's Required Evaluation
Scope** ([0.1.0-beta.1-requirements.md](0.1.0-beta.1-requirements.md)). This is the protocol *skeleton*:
before day 1 the researcher instantiates it into the actual pre-registration
(named thresholds, phase dates, budgets), and that instantiated document is
frozen. Every numeric value below is a pre-registered guess until beta.1 data
calibrates it; every change after freeze is a dated amendment, and any
amendment touching calibration, pins, or thresholds opens a new arm.

## 1. Object and hypotheses

Beta.1 tests whether Memoria beats the null workflow — Zotero + Obsidian +
ad-hoc ChatGPT — for one real researcher on a real corpus. The charter's four
success claims map to instruments:

| Claim | Instrument |
| --- | --- |
| Parity (capture / note-writing / Ask attention premium) | passive event timing vs baseline history; relegation tracking |
| Payoff (one compose moment beats the counterfactual, ends in export) | paired same-day ChatGPT counterfactuals; export events |
| Tolerance (upkeep, verification time, spend within budgets; no trust incident) | weekly metrics CSV; `costs`; `incidents` (empty = pass) |
| Survival (vault useful if Memoria dies) | keep-test audit — a mechanical check, not an opinion |

## 2. Arms and baseline

- **Null-workflow baseline:** ≥1 week of passivized history before install —
  capture volume/timing reconstructed from Zotero `dateAdded`, note activity
  from Obsidian file history — so parity claims compare against measured
  behavior, not recollection.
- **Counterfactual procedure:** for every payoff-candidate question, the
  researcher archives a same-day ChatGPT answer to the same question (prompt +
  output, timestamped) *before* reading Memoria's synthesis. Comparison is
  qualitative, criteria named at pre-registration.
- **Amendments as arms:** a model-tier change, prompt-pin change, or threshold
  change mid-beta is a release marker and a new arm — results are never pooled
  silently across arms.

## 3. Instrumentation (passive; no per-item bookkeeping)

- **Event log:** `events` verbs `capture | read | note | ask | compose |
  decide` with `hat` (researcher/developer); discovery rounds and intake
  provenance ride `capture` payloads; entity and triage decisions ride
  `decide` (design §5).
- **Ledgers:** `costs` (per-run, model + `prompt_v`), `runs`, `incidents`,
  `guard_log`. Weekly export to `Memoria/system/metrics/` (events + costs
  CSV, including `decide` events) — the evaluation dataset survives DB loss
  (design §5).
- **Non-backfillable day-1 signals** (instrumented from install, because none
  can be reconstructed later): reviewer decision time; deny reasons;
  accept-unedited vs accept-after-edit; the "later found wrong" reversal
  event (stamped at event time); export refusals (scored as the citation gate
  *working*); intake provenance (gap-driven vs researcher-initiated);
  synthesis-shaped Asks flagged separately (payoff attribution: evidence
  granularity vs grounding); annotation orphan counts; processing-card
  dismiss-vs-open.
- **Disposition streams** read through two-sided bands (sustained accept ≳90%
  = rubber-stamping, ≲20% = mistuned generator — both indict the mechanism);
  any surfaced rate states *n* and degrades to "insufficient data" below a
  sample floor. Band values are placeholder knobs.

## 4. Calibration (pre-registered, then frozen)

- **Checker/stance NLI thresholds:** tuned before day 1 on a small
  hand-labeled claim/anchor set (a few dozen corpus-like pairs, one
  afternoon, capped) **seeded with the measured NLI failure classes** —
  same-topic/different-variable pairs (routed to *unknown*) and
  high-overlap/opposite-meaning pairs — then frozen (design §7).
- **Launch gate:** the Checker passes a HANS/PAWS-style high-overlap negation
  + direction-reversal probe set — a floor, not a certificate.
- **Pin-change rule:** any model or prompt pin change re-triggers fixture
  validation before touching the real vault, and is a dated amendment.

## 5. Weekly citation audit

Random sample of emitted citations, sized for a double-digit fabrication base
rate; recall/precision rubric; evidence-completeness sampling. Oversamples:
near-threshold NLI promotions; marginal works (truncated / abstract-only).
Standing probes: does Ask resurface `superseded`-flagged works unlabeled;
Crossref retraction/expression-of-concern status of cited corpus works
(design §4). Quarterly-cadence extras inside the beta window:
**gate-efficacy sampling** — re-check a sample of accepted items against
source (approved-error rate) and re-judge a sample of rejections
(false-reject rate; is review improving correctness or is it theater) — and
one **decoy re-citation probe** (plant an already-cited span in a decoy
document; re-citing the decoy exposes surface matching posing as grounding).

## 6. Abandonment, relegation, re-entry

Pre-defined and scoreable before day 1: the no-use window that counts as
abandonment (~14 days, pre-registered); fallback-to-null detection (Zotero
captures continuing while Memoria events stop); failed resumption (a re-entry
that produces no `note`/`ask`/`compose` event within its session).
**Relegation clusters** — failed/empty Asks followed by narrowing to
capture-only use — are tracked as the leading indicator *ahead of* the
abandonment trigger. Idleness clocks and any decay UI tick on vault-active
days, never calendar days: absence is the normal case and must not be scored
or displayed as failure on return.

## 7. Verdict

Qualitative researcher judgment is primary; event-log metrics are evidence
into it, never the verdict themselves. A missed engagement number alone maps
to **targeted redesign, never kill**. Any sequential/statistical verdict
arithmetic is pre-registered and anytime-valid (no optional stopping); a
verb-mix shift between project phases is an expected pattern, not drift.
Exit is three-verdict: **adopt** (retention demonstrated, continue to
beta.2), **targeted redesign** (named failing mechanism, bounded rework,
re-run), or **kill** (the null workflow won; record why, per claim). The
verdict document names, per success claim, which instrument carried it.
