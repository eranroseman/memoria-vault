## Findings

- **AgentDojo is a real, actively-used adversarial evaluation environment for
  exactly this threat class, not a static test suite.** "AgentDojo is not a
  static test suite, but rather an extensible environment for designing and
  evaluating new agent tasks, defenses, and adaptive attacks." Its threat
  model: "AI agents are vulnerable to prompt injection attacks where data
  returned by external tools hijacks the agent to execute malicious tasks."
  (confidence: high, vote: 3-0 and 2-0 across two related sub-claims)
  - Evidence: verified directly against the AgentDojo paper (arXiv 2406.13352,
    NeurIPS 2024 Datasets and Benchmarks Track) by independent WebFetch of the
    abstract and full text; corroborated by OpenReview and independent review
    sites.
  - Source: https://arxiv.org/abs/2406.13352
  - Beta.1 implication: this is the right evaluation harness class to
    validate Memoria's own minimal sandbox against — design §2.3's committed
    sandbox should be run through an AgentDojo-style adaptive evaluation
    before it's trusted with untrusted-influenced code, not just checked
    against a static attack list.

- **Adaptive, defense-aware attacks broke all eight tested defenses against
  indirect prompt injection on LLM agents, exceeding 50% attack success rate
  — and non-adaptive (static) evaluation systematically underestimates this
  vulnerability.** "Prior studies have shown that non-adaptive attacks can
  greatly underestimate a system's vulnerabilities, as defenses that seem
  robust under these attacks may be entirely compromised by adaptive ones."
  (confidence: high, vote: 3-0 across both sub-claims, each independently
  verified 3 times)
  - Evidence: verified directly against the peer-reviewed paper "Adaptive
    Attacks Break Defenses Against Indirect Prompt Injection Attacks on LLM
    Agents" (Zhan, Fang, Panchal, Kang — UIUC; NAACL 2025 Findings,
    aclanthology.org/2025.findings-naacl.395), which itself cites Athalye et
    al. 2018 ("Obfuscated Gradients Give a False Sense of Security") for the
    general adaptive-vs-static evaluation principle.
  - Source: https://info.arxiv.org/abs/2503.00061 (arXiv:2503.00061)
  - Beta.1 implication: **whatever validation Memoria runs against its
    sandbox must be adaptive, not a fixed attack list** — a defense that
    passes a static InjecAgent/AgentDojo-style benchmark once is not
    evidence it holds against an attacker who adapts to the specific
    defense. This directly sharpens design §8 Q1's "adversarial validation"
    requirement: it must be an adaptive red-team process, not a one-time
    pass/fail suite run.

- **Anthropic's own production Claude Code sandbox uses exactly the two
  backends this research question named for Linux/macOS: bubblewrap and
  Seatbelt** — "We've built this on top of OS level primitives such as Linux
  bubblewrap and MacOS seatbelt to enforce these restrictions at the OS
  level," and separately, "It uses operating system primitives to restrict
  the filesystem and network access of every Bash command Claude runs:
  Seatbelt, the built-in macOS sandbox, and bubblewrap on Linux and WSL2."
  (confidence: high, vote: 2-1 and 2-0 across two related claims — the
  dissenting vote on the first claim specifically challenged the *network*
  half as an overreach, not the OS-primitive/bubblewrap/Seatbelt fact itself;
  see the filesystem-model correction below)
  - Evidence: verified directly against Anthropic's own engineering blog and
    current product docs.
  - Source: https://www.anthropic.com/engineering/claude-code-sandboxing , https://code.claude.com/docs/en/sandbox-environments
  - Beta.1 implication: this is a directly comparable, currently-shipping
    precedent for exactly Memoria's use case (a coding-agent tool running
    potentially-untrusted, LLM-influenced code locally) — validates
    bubblewrap+Seatbelt as the right backend pair for Linux/macOS, consistent
    with design §2.3's direction.

- **Correction to a natural oversimplification: Claude Code's filesystem
  isolation is NOT a simple symmetric "directory-scoped" read+write boundary
  — read defaults to the entire computer (with exclusions), while write is
  scoped to the working directory.** A claim asserting a single
  "coarse-grained... directory-scoped boundary" covering both read and write
  was **refuted** by all three independent voters after checking Anthropic's
  detailed current docs (which supersede the simplified engineering-blog
  quote): "Read is NOT scoped to the cwd at all. The docs state: 'Default
  read behavior: read access to the entire computer, except...'" (confidence:
  high — unanimous refutation, 0-3)
  - Evidence: Anthropic's current sandboxing docs distinguish read policy
    (broad, exclusion-based) from write policy (scoped to cwd/project), which
    a simplified single-boundary description misses.
  - Source: https://code.claude.com/docs/en/sandboxing
  - Beta.1 implication: **design §2.3's own filesystem model must specify
    read and write scope separately and explicitly** — "project-scoped
    filesystem" is ambiguous in exactly the way this refuted claim was;
    Memoria's design should state plainly whether read access is restricted
    to the project path (tighter than Claude Code's default) or broader, and
    should not assume "scoped filesystem" reads as a single symmetric
    property.

- **WSL1 cannot run this class of sandbox at all — bubblewrap requires kernel
  features WSL1 lacks; WSL2 is required.** (confidence: high, vote: 3-0)
  - Evidence: confirmed against practitioner/vendor documentation of
    bubblewrap's Linux-namespace dependency and WSL1 vs. WSL2's differing
    kernel compatibility.
  - Source: (corroborated across the "Isolation mechanics" and "Practitioner
    implementations" search angles; primary confirmation via
    https://code.claude.com/docs/en/sandbox-environments)
  - Beta.1 implication: a concrete, packaging-relevant constraint — Memoria's
    Windows story must require and detect WSL2 specifically, not just "WSL,"
    and should fail with a clear message on WSL1.

- **Bubblewrap itself provides only low-level isolation primitives (mount
  namespaces, etc.) — it is not a complete, ready-made sandbox policy.** The
  calling application must configure the actual restriction policy (which
  paths are writable, what's blocked) on top of it, achieving unprivileged
  operation via Linux kernel user namespaces (no setuid binary required —
  setuid mode has been removed from bubblewrap). (confidence: high, vote: 3-0
  and 2-1 across two related sub-claims)
  - Evidence: confirmed against bubblewrap's own GitHub documentation.
  - Source: https://github.com/containers/bubblewrap
  - Beta.1 implication: Memoria's sandbox implementation work is not "install
    bubblewrap and you're done" — the approved-commands-only, project-scoped
    policy design §2.3 already specifies is real, necessary work on top of
    bubblewrap, not a config flag bubblewrap ships with.

- **A real, named bubblewrap-based agent-sandbox escape exists and is
  publicly documented: a path-resolution alias (`/proc/self/root/usr/bin/npx`
  resolving to the same binary a denylist rule was meant to block, without
  being pattern-matched as such) let Claude Code escape a bubblewrap sandbox
  (April 2026, reported by Leonardo Di Donato).** (confidence: high, vote:
  3-0)
  - Evidence: independently confirmed against the practitioner writeup
    describing the exact escape mechanism.
  - Source: (via the "Known escapes, CVEs, and limitations" search angle,
    a practitioner blog writeup)
  - Beta.1 implication: **denylist/pattern-matching approaches to
    "approved-commands-only" are exactly the class of control this escape
    defeated** — design §2.3's "approved-commands-only" policy must be an
    allowlist resolved to canonical paths (not a denylist or a
    string-pattern match against argv[0]), or it inherits this exact failure
    mode.

- **CVE-2024-32462 is a real, patched bubblewrap-adjacent sandbox escape, but
  its root cause was a caller-side bug (Flatpak's wrapper), not a flaw in
  bubblewrap's own namespace-isolation mechanism.** "Flatpak versions before
  1.10.9, 1.12.9, 1.14.6, and 1.15.8 allowed a malicious app to pass an
  arbitrary commandline through the D-Bus portal interface
  `org.freedesktop.portal.Background.RequestBackground`, which Flatpak then
  forwarded unsanitized into the bwrap invocation." (confidence: high, vote:
  2-0)
  - Evidence: confirmed against the CVE database entry.
  - Source: https://www.rapid7.com/db/vulnerabilities/redhat_linux-cve-2024-32462/
  - Beta.1 implication: the lesson generalizes directly — **any wrapper code
    Memoria writes around bubblewrap must insert the `--` argument delimiter
    before invoking `bwrap`**, or attacker-influenced input (e.g. a project
    name, a code-artifact argument) could inject `bwrap` flags like
    `--bind` and remount arbitrary host paths into the sandbox. (The specific
    claim that this was purely "a caller-side argument-injection bug, not a
    bubblewrap flaw" could not be independently re-verified this pass — all
    3 verification votes errored on rate limits — so treat that precise
    root-cause framing as plausible-but-unconfirmed, even though the CVE
    itself is real and confirmed.)

- **On macOS, Codex enforces its sandbox via Apple's Seatbelt/`sandbox-exec`
  with a dynamically generated, deny-by-default policy, invoked only through
  a hardcoded `/usr/bin/sandbox-exec` path specifically to prevent path-
  injection attacks, with `.git` directories carved out as read-only.**
  (confidence: high, vote: 3-0)
  - Evidence: confirmed against OpenAI's own Codex sandboxing documentation.
  - Source: https://developers.openai.com/codex/concepts/sandboxing
  - Beta.1 implication: "hardcode the binary path, don't resolve it from
    `PATH`" is a second concrete, transferable hardening pattern for
    Memoria's own sandbox invocation code, alongside the allowlist-not-
    denylist lesson above.

- **On macOS, process sandboxing is opt-in, not kernel-enforced by default —
  unlike iOS, a macOS process is only restricted if it actively enters the
  sandbox.** (confidence: high, vote: 3-0; source quality: secondary/wiki,
  but confirmed against current content)
  - Evidence: confirmed against a security-reference wiki's current macOS
    sandboxing documentation.
  - Source: (via the "Hardening configuration specifics" search angle,
    a macOS-security reference wiki)
  - Beta.1 implication: Memoria's own worker process must explicitly and
    verifiably enter the Seatbelt sandbox before executing any project code
    — there is no ambient OS-level guarantee that simply running on macOS
    provides isolation.

- **Apple's own `sandbox-exec` (Seatbelt) — the exact mechanism Anthropic's
  Claude Code and OpenAI's Codex both depend on for macOS — is marked
  DEPRECATED in Apple's own man page.** (confidence: high, vote: 3-0)
  - Evidence: confirmed directly against the current Apple man page text.
  - Source: (via "Hardening configuration specifics" and "Known escapes"
    search angles, cross-confirmed by two independent fetches)
  - Beta.1 implication: **a real, named long-term risk to the macOS backend
    choice** — design §8 Q1's "hardening" work should explicitly track
    Apple's deprecation notice and have a contingency (or at least a
    monitoring plan) for `sandbox-exec` being removed in a future macOS
    release, rather than treating it as a stable foundation.

**Explicitly refuted — do not restate:**
1. "The Docker Desktop VM under WSL2 runs as a genuine Hyper-V-isolated VM
   boundary, and gaining control of that VM does not by itself grant code
   execution on the Windows host" (vote 0-2/0-3 across two independent
   verification rounds) — refuted specifically because the cited source
   (Trend Micro's own writeup) documents the opposite: real, working
   VM-to-host escape techniques under exactly this WSL2/Docker Desktop
   configuration. Do not treat "it's a Hyper-V VM" as sufficient isolation
   for a Windows backend.
2. "The filesystem isolation model is coarse-grained: read/write is allowed
   anywhere inside the current working directory... a directory-scoped
   boundary, not a fine-grained per-file or read-only-by-default policy"
   (vote 0-3) — see the correction above; read and write have different
   scopes in the real (Claude Code) system this claim was drawn from.
