## alpha.3 — Deciding the UI-build vocabulary: navigation, capture, operations, and docs hygiene, plus the issue-tracking system that governs it

**Theme.** alpha.3 is the "UI build" release, but the code in this range is deliberately small: the work of the release is *deciding the terms the UI is built on* and *cleaning the ground under it*. Five proposed ADRs (69–73) settle the deterministic-layer name, the navigation model, structured capture, command surfacing, and documentation-reference conventions; a sixth (74, deferred) records a plugin supply-chain target; a large body of research reports supplies the derivations; a GitHub-Project-based issue-tracking system replaces the old label soup; and the only shipped source change extracts policy path/glob semantics into a real runtime package so three MCP servers stop copy-pasting the gate's matching logic. What drove it: an "open issues" review of the vault surfaced a pile of naming clashes, broken docs links, capture defects, and navigation ambiguity that had to be resolved *before* the UI work (and the engines→operations rename) could land without pinning links and terms to things about to move.

Note on scope: much of this range lives in ADRs (explicitly REFERENCE per ground rules) and in `docs/releasing/0.1.0-alpha.3/tmp/` research reports. Per the version hints these research reports *are* this release's authoritative design intent, so I treat them as source and cite the ADRs as corroboration.

---

### Vault / concepts

- **What:** The deterministic non-LLM layer is renamed as a *concept* from "engines" / "deterministic apps" to **Operations**, and given **four output-defined categories — Processing, Integrity, Cleanup, Telemetry** (ADR-69 `docs/adr/69-operations-layer-naming.md`; derived in `tmp/open-issues-research.md` Issue 2 and `tmp/naming-and-diataxis-audit.md`). Each leaf operation gets **one bare-verb token** (`ingest`, `search`, `cluster`, `verify`, `lint`) used identically as name, directory, module stem, and CLI verb. The eventual code tree becomes `src/.memoria/operations/{processing,integrity,cleanup,telemetry}/`. **Why:** the docs called one concept two names ("engine" and "app") — the DDD ubiquitous-language anti-pattern — and "engine" is not self-descriptive (game/search/rules engines abound); the five components also had no shared structure (Cluster logic stranded in `mcp/cluster_mcp.py`, Search had no directory, `sweeps/` mixed three kinds of work). The original alpha.3 design note's "processing/maintenance vs bookkeeping/housekeeping" split was rejected because it mis-cuts the space (maintenance ≈ housekeeping; bookkeeping is a metaphor). The four self-descriptive categories were chosen because they line up with two things already true: the MCP-facade boundary (Processing is agent-reachable; the rest are cron/CI-only) and the navigation model (ADR-69 Consequences). Rejected alternatives: keep "engine"/delete "app" (cheapest, kept only as churn fallback); two buckets Processing/Maintenance (loses dashboard-driving distinctions); three categories folding Cleanup into Integrity (only tempting because of the synonym overlap the rename removes); "service"/"tool" umbrella (implies always-on daemon; collides with "MCP tool").

- **What:** The rename is **decided now but sequenced as a later atomic refactor**, explicitly *after* the docs→source link convention (ADR-73) lands (ADR-69 "When this matters"). **Why:** the vocabulary must be settled so alpha.3 UI/navigation/dashboards/docs build on final terms, but the tree move has high blast radius (`engines/` is hardcoded in `tests/conftest.py`, `scripts/test.sh`, `scripts/e2e-smoke.sh`, `scripts/install.sh`, `tests/test_precommit_schema.py`, and across docs), so it is deferred to avoid pinning the doc-link sweep to paths about to move (`tmp/open-issues-research.md` Issue 3c). Cheap non-structural parts (delete "app" from prose, define "CI invocation") may land in alpha.3 ahead of the rename.

- **What:** The `.memoria/` invisibility invariant is restated as a hard rule: **`.memoria/` is hidden infrastructure the PI never opens; everything user-reachable lives in visible folders** (`system/`, `catalog/`, `notes/`, `inbox/`), and any workflow that shows the user a `.memoria/...` path is a design bug (`tmp/ui-design-research-report.md` §5). **Why:** the report found a docs↔implementation contradiction — Tutorial 03 Step 4 tells the user to open `.memoria/data/extracts/<citekey>.md`, a path that *does not exist* (ingest actually produces the visible `catalog/papers/<citekey>.md`), simultaneously violating the zero-contradiction rule and the never-touch-`.memoria` invariant. Fixes queued: correct the tutorial, audit the whole doc set for `.memoria/...` paths shown to users, and add a folder-model reference stating the invariant plainly.

- **What:** New in-tree **README contracts** define concept ownership at each `src/.memoria/` subtree: `schemas/README.md` declares schemas the single machine-readable vault contract (`types/` = frontmatter by note type, `folders.yaml` = type homes + gated prefixes + empty-folder skeleton, `calibration.yaml` = shared thresholds) and directs consumers to `engines/lib/schema.py` rather than re-parsing; `engines/README.md`, `profiles/README.md`, `mcp/README.md` similarly pin responsibilities (commit `7b7a28ba`). **Why:** to make ownership boundaries explicit and prevent duplication/divergence — e.g. schemas README requires dependency-free fallback constants to be *equality-tested* against the canonical files, and profiles README forbids profile files silently widening the capability/path ceilings owned by `tool-registry.yaml` and `lane-overrides/`. Inferred rationale (not documented as an essay): these codify the source-of-truth discipline the CLAUDE.md "folder is the source of truth" rule demands. Note the READMEs still say "Engines"/"engines" — the ADR-69 concept rename is not yet reflected in these files, consistent with the "decide now, rename later" sequencing.

### Profiles / runtime

- **What:** Policy path/glob semantics are extracted into a **real shipped Python package**, `src/.memoria/memoria_runtime/policy/` (new `__init__.py` + `paths.py`), exporting `ACTIONS`, `MUTATING_ACTIONS`, `REVIEW_GATED_PREFIXES`, `normalize_path`, `glob_to_regex`, `path_matches`, `within_scope`; `memoria_runtime/__init__.py` declares it an "Internal Python package for Memoria's shipped runtime" (commit `7b7a28ba`). The MCP servers keep working as direct executables by bootstrapping `sys.path` from the enclosing `.memoria` dir (see `mcp/README.md`: "reusable policy semantics belong in `../memoria_runtime/`, while engine logic belongs in `../engines/`"). **Why:** the same glob/path machinery was duplicated across servers — `patterns_mcp.py` carried its own copy of `REVIEW_GATED_PREFIXES` ("Mirrors policy_mcp.REVIEW_GATED_PREFIXES"), and `tasks_mcp.py` reached into `policy_mcp` as a module (`import policy_mcp`) purely to borrow `within_scope`. Concentrating one dependency-free implementation guarantees "the delegation ceiling and the write gate agree on semantics" (`tasks_mcp.py` comment). This is the layering discipline: transport thin, semantics shared, one system of record.

- **What:** `profiles/README.md` adds an operational contract: run `scripts/agents-doctor.py --write` after profile or lane changes to refresh the derived profile-policy matrix, and profile files "must not silently widen" the `tool-registry.yaml` / `lane-overrides/` ceilings (commit `7b7a28ba`). **Why:** to keep the derived profile↔policy view honest and prevent per-profile drift from the central capability/path contracts (inferred from the README text; part of the same "strengthen runtime boundaries" commit).

### Gate / provenance

- **What:** `policy_mcp.py` is slimmed by ~120 lines: `normalize_path`, `glob_to_regex`, `path_matches`, `within_scope`, and the `ACTIONS`/`MUTATING_ACTIONS`/`REVIEW_GATED_PREFIXES` constants move out to `memoria_runtime.policy` and are re-imported; `policy_mcp` keeps only the lane-policy model, `is_review_gated`, `load_gated_prefixes`, and the decision protocol, and re-exports the moved helpers via `__all__` for compatibility (commit `7b7a28ba`, diff on `src/.memoria/mcp/policy_mcp.py`). **Why:** de-duplication and boundary hygiene — the gate's *tested* glob machinery is now the single canonical implementation other servers consume, rather than each server re-deriving or importing the gate module. The eight-action vocabulary, the mutating-action set, and the review-gated `notes/claims/` + `notes/hubs/` prefixes are preserved verbatim in `paths.py`, so no gate *behavior* changed — this is a pure structural move (verified by the identical logic and the new `tests/test_runtime_policy.py`).

- **What:** `patterns_mcp.py` and `tasks_mcp.py` are re-pointed at the shared package: `patterns_mcp` imports `REVIEW_GATED_PREFIXES` from `memoria_runtime.policy` (deleting its local mirror), and `tasks_mcp` imports `within_scope` directly (deleting `import policy_mcp`) (commit `7b7a28ba`). **Why:** eliminate the copy of the review-gated prefixes ("used only when folders.yaml is unreadable") and the cross-server module import, so the fallback constant and the scope-matching logic can no longer diverge from the gate's — same divergence-risk argument as the extraction above.

### Pipeline / ingest / enrichment

- **What:** No pipeline code changed, but a **design decision about ingest's write path and documentation** is recorded: ingest's canonical, *visible* output is affirmed as `catalog/papers/<citekey>.md` (not `.memoria/data/extracts/`), and the ingest docs are to be **split by Diátaxis** — a procedural `reference/ingest` (purely how-to), a separate `explanation/.../ingest` (rationale: why the uncertainty floor, why automated-not-gated), with the sweeps content (`reconcile.py`/`retraction.py`) moved out of `reference/ingest` into its own reference (ADR-73 + `tmp/ui-design-research-report.md` §7). **Why:** the audit found `reference/ingest.md` mixed procedure and rationale *and* documented other operations (the sweeps) on the same page, violating the Diátaxis "work vs study" separation; the fix generalizes to "each complex operation gets a procedural reference + a rationale explanation, cross-linked with Django's link-don't-repeat discipline so each fact has one home" (ADR-73 Decision §3).

- **What:** A cluster of **capture/ingest defects** is logged as blocking the core loop (`tmp/ui-design-research-report.md` §9, `tmp/alpha3-triage.md` §A): fleeting notes not appearing in `fleeting.base` (D-1), URL capture not writing a candidate card to `inbox/` (D-2), the resolve-inbox-card button broken (D-4), Zotero capture broken (D-5). **Why:** these break Tutorials 03/etc. — the "core loop" — and per the triage's guiding principle a broken tutorial is the highest-priority class; they were mapped to issues #455–459/#449 at Priority `High` (`tmp/project-field-mapping.md`). The report mandates live reproduction before any is declared fixed (verify-independently rule). No fix shipped in this range — these are scoped, not resolved.

### Discovery

- **What:** **Structured capture through forms** is decided: human note creation goes through the **Modal Forms** plugin with typed fields and controlled-vocabulary controls (radio buttons, not a dropdown, for small fixed sets like the lifecycle state `proposed`/`current`/`archived`), with help text living **in the form, not the saved note** (ADR-71 `docs/adr/71-structured-capture-forms.md`; `tmp/ui-design-research-report.md` §2). **Why:** current creation is template-based free text — instructional text leaks into saved notes, there's no per-type required-field enforcement, and controlled vocabularies are unconstrained free text. Rejected: lint-only (detection not prevention; bad data exists transiently); form-only (client-side, bypassable by agents/API/bulk); in-panel enum widget like Better Properties (off-store, leans on undocumented Obsidian internals — sandbox pilot at most).

- **What:** The **deterministic Linter remains the single enforcement authority**, and **both the form's controlled vocabularies and the Linter's allowed-value checks derive from one per-type schema** in `.memoria/schemas/`; Memoria is declared **strict, not liberal** — an unknown value for a fixed-set field is rejected, not coerced (ADR-71 Decision). **Why:** agents, the API, and bulk edits all create notes outside any form, so the form cannot be the guarantee; defense in depth = prevention at entry (form) + detection everywhere (Linter), with one schema so the two layers can't diverge. Corroborated by ADR-49 (Linter is the integrity authority) and ADR-30 (the non-form ingest write path). This is why the report explicitly rejects Propsec as a system-of-record (`tmp/ui-design-research-report.md` §10, "Skip as system-of-record — Linter owns enforcement").

- **What:** A net-new discovery affordance is queued: a **"create linked claim note" button on source notes** (issue #470) and clarification that the Librarian proposes a candidate card while the PI/Co-PI creates the source note (`tmp/ui-design-research-report.md` §6). **Why:** aligns capture with the Studio/synthesis flow (Tutorial 05); implemented as a QuickAdd/Commander page-header button pre-linking a claim note to its source. Inferred: closes the gap between the honesty-card flow and what the tutorial claims.

### UI / navigation / dashboards / panes

- **What:** Navigation is decided as a set of **intent-named gates** on the existing workspace machinery (ADR-68), with **three core gates now** — an Action/"what needs me" gate, a Sources/Library gate, a Knowledge gate — and the **Project gate deferred to alpha.4**; the top level never exceeds five, and switching a gate changes only the left-nav, main dashboard, and where the Co-PI points — **never what an action does** (ADR-70 `docs/adr/70-navigation-gates-dashboards.md`; `tmp/open-issues-research.md` Issue 1, `tmp/ui-design-research-report.md` §3–4). **Why:** ADR-68 shipped the Desk/Library/Studio *shells* but left the *model* unsettled; alpha.3 is the UI build so it must be decided now. Rejected: four/five gates now (sits at the upper bound before Project even arrives — spend the slot on Project); a single unified home dashboard (violates "show only what this job needs," overcrowds).

- **What:** **System health is ambient, not a gate** — a status-bar indicator carries the ambient signal, anything *actionable* surfaces as a card in the Action queue (point-of-action), and a pull-based health detail view is reachable on demand (ADR-70 Decision). **Why:** a dedicated Maintenance gate was explicitly rejected because every cited authority (Google SRE, NN/g, Calm Technology) treats unconditional "everything not-green" surfacing as the *cause* of alert fatigue, and VS Code/GitHub/Datadog all decline to promote "problems" to a forced destination (ADR-70 Alternatives; `tmp/open-issues-research.md` Issue 1c). This prevents maintenance from ever becoming a forced "wall of yellow."

- **What:** **Dashboards are organized by Jobs-To-Be-Done** — each gate's dashboard answers that gate's question: the Action dashboard is strictly action-first (every card carries its next action; non-actionable items don't appear), while Library and Knowledge lean object-first for browsing (ADR-70). This maps directly onto ADR-69: Integrity findings → status bar + Action cards; Telemetry → pull dashboards; Cleanup → invisible; Processing → the knowledge itself. **Why:** to cap cognitive load (3 gates + progressive disclosure) and keep each surface answering one job (`tmp/open-issues-research.md` Issue 1d). Implementing issues #467/#375/#380/#145.

- **What:** **Base Board (kanban over Bases) is adopted only as a version-pinned sandbox pilot**, not a committed dependency (ADR-70; `tmp/ui-design-research-report.md` §4, §10). **Why:** native Bases has no board view yet and Base Board is "very young, churning" — architecturally aligned but risky, so pilot-not-commit. The classic Kanban plugin is rejected outright ("diverges from the Bases-first model").

- **What:** **Every Memoria action is reachable directly from the Obsidian UI without the Co-PI**; native hotkeys + command-palette pinning cover most surfaces, **Commander** is adopted to place commands in the ribbon/page-header/status-bar (the one genuine core gap), Slash Commander is optional for an in-editor `/` menu, and the **Co-PI remains additive** — the path only for things that genuinely need LLM–human dialogue (ADR-72 `docs/adr/72-command-surfacing.md`; `tmp/ui-design-research-report.md` §1). **Why:** some actions were reachable only by asking the Co-PI, violating the product principle that every feature is directly PI-accessible and making routine actions depend on an LLM round-trip. Rejected: Co-PI-only invocation (violates direct access); native-only (can't place a command in the ribbon — the exact gap Commander fills). Explicitly rejects **Shell Commands** (arbitrary local execution — violates the MCP-only sandbox) and Better Command Palette (unmaintained) for this role. Corroborated by ADR-48 (one Co-PI fronts everything); implementing issues #461/#380.

- **What:** Sitewide capitalization decision **co-PI → Co-PI**, with the boundary that "Co-PI" is the agent's *name* while "PI" remains an internal role never used as a form of address; Client Agent config set to display name "Memoria Co-PI", auto-export **on**, open-after-export **off**, export to a *visible* system folder (`tmp/ui-design-research-report.md` §6). **Why:** consistency defect flagged in the review, bounded by the existing don't-call-the-user-PI rule; export must respect the never-touch-`.memoria` invariant (§5).

### Distribution / eval

- **What:** ADR-74 (deferred) `docs/adr/74-pinned-obsidian-plugin-supply-chain.md` records a **target supply-chain model for bundled Obsidian plugins**: keep build artifacts committed under `src/.obsidian/plugins/` (installation stays network-independent), but add a versioned **lock manifest** (per plugin: ID, upstream repo, pinned tag/commit, source URL, artifact SHA-256, license, patched/unmodified/Memoria-authored status), a verifying updater, a strict ownership split (upstream `main.js`/`manifest.json`/`styles.css` are vendored; Memoria-authored `data.json`/`.example`/overlays never overwritten), a CI doctor proving lock↔artifact match, and explicit patch provenance (fork or reviewable patch, no editing minified output). **Why:** the repo already commits plugin artifacts (ADR-26 repo-as-install-unit, ADR-55 golden copy) but records no machine-readable provenance, so updates require manual discovery and CI can verify the vault image but "cannot prove where a changed `main.js` came from." Rejected: download-at-install (network-dependent, moves executable supply-chain risk into the install path — conflicts with ADR-26); community-plugin installer (removes batteries-included first run); Git submodules (Obsidian loads built artifacts not source trees; adds build friction); continue manual vendoring (present state; non-reproducible). Status **deferred** — target model only; current layout unchanged until accepted. It notes a checksum proves the repo holds the reviewed artifact but *not* that the upstream publisher was trustworthy — version selection stays a human decision.

- **What:** Operational docs that still invoked component `--self-test` CLI flags as runnable commands are updated to **pytest / live smoke-tests** (commit `063fe766`): `headless-test-plan.md` Part A + results table → `pytest tests/test_*.py`; `g9-spine-plan.md`, `g10-ingest-plan.md`, `release-candidate-runbook.md` → pytest component tests; `diagnose-a-denied-write.md` → a live `--decide` gate smoke-test. **Why:** the `--self-test` flags were removed when L1 tests moved to the repo-side pytest tree (ADR-44) and the scripts now *reject* `--self-test` (verified live: all five reject it) — the docs were prescribing a command that no longer exists, a docs↔implementation contradiction. Frozen alpha.1 records and the ADRs keep their historical references intentionally.

- **What:** The `eval/` folder is clarified as `system/eval/` — gold-task fixtures for the quarterly vault-eval (diagnostic, non-gating) — and flagged as a *discoverability*, not a missing-artifact, problem; metrics dashboards (`eval-trend`/`fleet-health`) reading not-yet-existing `system/metrics/...` must show graceful "no data yet" messaging rather than an error (`tmp/ui-design-research-report.md` §5, §9 D-8). **Why:** review questions ("what are the files in the eval folder?") revealed unlabeled visible folders; the metrics-empty state is expected until cron sweeps populate it and shouldn't read as a failure.

### Security

- **What:** Reaffirms the **MCP-only sandbox** as a hard boundary in two plugin decisions: Shell Commands rejected because arbitrary OS execution violates it (ADR-72), and `cli-rest-mcp` rejected because its single generic `execute` tool "defeats the named-tool write gate" and is redundant with the existing Local REST API MCP (ADR-31) (`tmp/ui-design-research-report.md` §10). Buttons Panel skipped for its JS-exec security surface; the in-use Buttons (shabegom) plugin flagged as "our highest-risk current dependency" with an exit plan to native Bases actions / Commander. **Why:** the named-tool write gate + audit chain is the security model — a plugin exposing a generic execute escape hatch or arbitrary shell would bypass the policy MCP that every mutating vault operation must pass through (`mcp/README.md`: "Every mutating vault operation must pass through the policy MCP and complete its audit record").

- **What:** ADR-74's deferred model adds a **CI provenance doctor** requirement (lock↔artifact match, one entry per enabled plugin, no undeclared executable artifact) as a security control over the bundled plugin binaries. **Why:** currently a changed minified `main.js` cannot be traced to an upstream release — an opaque binary supply-chain gap.

Currently ~77 advisories across ~70 links in 17 files; exit stays 0 (advisory) so CI is unaffected until the sweep + promotion to error (issue #464).

### Issue-tracking system (release/process design — the version's headline focus)

- **What:** A new **GitHub-Project-based issue-tracking model** is documented (`docs/releasing/0.1.0-alpha.3/tmp/issue-tracking-guide.md`, commit `63751936`): planning metadata moves off labels into **Project #1 single-select fields** — originally **Status, Area, Type, Priority**; later amended in alpha.8 to the live **Status** and **Readiness** fields only — with milestones carrying release phase, sub-issues carrying hierarchy, and labels kept minimal. Guiding rule: "each kind of metadata lives on the feature that models it best, never duplicated." **Why:** the pre-alpha.3 tracker was label soup (`enhancement`/`question`/`research`/`needs-scoping`) that overloaded search chips with planning state; the mental model is fixed as "the issue is the work; Project fields slice it; the milestone is when; the ADR is why."

- **What:** **Status finalized to exactly five states in board order** — Backlog · In progress · In review · Done · Deferred (commit `7581edb1`); `Needs scoping` and `Blocked` were **deliberately dropped**. **Why:** option order = board column order (left = active lifecycle read as progress, right = inactive Done/Deferred); the two dropped states "each overlapped Backlog / In progress and earned no distinct action" — capture that signal with a note or Priority instead. Deferred is defined as a *revisit-each-cycle* parking lot, not a graveyard. Operational caution recorded: re-add statuses via the UI because editing field options via API can wipe existing item values.

- **What:** **Type** taxonomy (Bug/Feature/Refactor/Docs/Research/Decision) mapped from retired labels, **Area** taxonomy of 9 subsystems (capture/ingest/knowledge/obsidian-ui/operations/docs-site/installer/integrations/agents), **Priority** reserved `High` for tutorial/core-loop blockers only. The `operations` Area name explicitly matches ADR-69 (with the four categories as sub-kinds named in the issue body). **Why:** to give each work item exactly one Type/Status/Priority and one-or-more Area, filterable in the Project, while `bug`/`documentation` survive as *labels* only because repo-wide `is:issue` search and bots (Dependabot/release-please) see labels but not Project fields — the one acknowledged trade-off.

- **What:** A **Field color scheme** is specified (commit `7581edb1`): semantic fields encode meaning (Status = flow, Priority = heat), categorical fields use distinct hues as tags; reserve Red = severity/breakage, Green = good/done, Gray = neutral/inactive. Concrete assignments given for Status/Priority/Type; Sub-issues progress bar set to **Purple** ("a partial warm bar reads as a warning; purple/blue/green read as neutral progress"); Area left on auto-color (9 areas > GitHub's 8-color palette). **Why:** to make the board legible at a glance and avoid color miscues (a warm progress bar misreading as an alert).

- **What:** A **per-issue field-population map** for all 74 open issues (`tmp/project-field-mapping.md`, commit `63751936`) with bulk-pass instructions; the one-time population was executed via the API in a single scripted batch (184 edits), and retired labels (`enhancement`, `question`, `research`, `needs-scoping`) are to be deleted after Type is populated. **Why:** operationalizes the migration; documents the CLI gotchas (Projects need a project-scoped PAT via `env -u GITHUB_TOKEN`; bulk milestone/label edits across issues you didn't author are hard-blocked by the harness external-write guard, so use the table view).

(The engines→operations *code/doc* rename itself is scheduled, not executed in this range — the in-tree READMEs still read "Engines".)
