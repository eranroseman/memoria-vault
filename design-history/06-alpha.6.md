## alpha.6 — Conformance closeout & the model-free test-env harness (ADR-80 Phase 1)

**Theme:** make the *accepted* decisions that silently didn't hold actually hold, then ship the roadmap's committed-next automated verification spine.

alpha.6 is an internal, untagged checkpoint (`status: complete`, `released: false`) built on two pillars. Pillar one (Track A) turns the findings of a full accepted-ADR audit — all 56 accepted ADRs read against the *built* artifacts, which found 8 partials and 0 unbuilt (`docs/releasing/0.1.0-alpha.6/tmp/adr-implementation-gap-analysis.md`) — into real conformance fixes: a runtime-security regression, two unenforced schema/query invariants, the missing Project-gate navigation surface, and cheap docs/template/supply-chain cleanups. Pillar two (Track B) delivers the long-pole net-new work: the **ephemeral test-env harness (ADR-80, Phase 1)**, a model-free L0–L4 golden path via seed→cassette that makes every future release re-runnable from a fresh clone. The driving pressure throughout was the audit's discovery that "accepted" had drifted from "true," plus the parallel finding that several *deferred* ADRs cited machinery that doesn't exist.

### Vault / concepts & schemas

- **What:** New `code-note` note type added — 22nd type. New schema `src/.memoria/schemas/types/code-note.yaml` (`category: projects`, `gated: false`, `initial_lifecycle: proposed`, lifecycle subset `proposed → current → archived`, required `title/lifecycle/title/project/agent/task/acceptance`, `agent` enum `[codex, claude-code, other]`), folder mapping `code-note: projects` in `folders.yaml`, and a starter template `src/system/templates/code-note.md`. Docs updated: `note-types.md` (21→22 types, "Projects (2)"→"(3)"), `frontmatter.md` lifecycle-subset table adds `code-note`. **Why:** the audit found ADR-07 (code-agent attachment) PARTIAL — "Fully wired (memoria-engineer profile, `claude-code`/`codex` external agents, `code`-lane override), but the ADR's 'Files affected' promises a `system/templates/code-note.md` starter template that does not exist." The plan chose create-over-drop as "the cheaper-to-defend choice and the profile/lane wiring already exists" (`tmp/exec-plan-0.1.0-alpha.6.md` WS-3 #627). The template is "the Engineer's handoff/provenance note for external coding agents under a project's `code/` scratch" (`note-types.md`).

- **What:** `thesis.yaml` gains optional `promoted_at: date` and `promoted_by: str`; `frontmatter.md` documents them as "Promotion provenance for a thesis. A `thesis` at `lifecycle: current` **must** carry `promoted_at`." **Why:** ADR-78 (thesis note type) was PARTIAL — "ADR-78 requires 'the schema must reject a born-`current` thesis' and a review-gated promotion to `current`. Neither holds: `validate_frontmatter` only checks enum membership, and `current` is a legal enum value, so a thesis authored directly at `lifecycle: current` validates clean." The fix keys enforcement on a **provenance marker** rather than history because "`validate_frontmatter` is stateless, so key on a marker, not history" (exec-plan WS-1 #621). Crucially, the alternative — adding `projects/` to `gated_prefixes` — was **rejected**: the policy gate is "path-based and content-blind … it can't tell *create-proposed* from *promote-to-current*, and it only intercepts agent `mcp_obsidian_*` writes (it misses PI promotions typed directly in Obsidian)"; a `projects/` prefix "would dry-run all `proposed`/`provisional` agent work too — which ADR-78 says to allow." Enforcement therefore lives in the **content-aware Linter/schema layer**, resolved formally in PR #647.

- **What:** `claim.yaml` gains optional `schema_version: int`; both claim templates bumped — `src/system/templates/claim.md` and `create-linked-claim.js` now emit `schema_version: 2`; documented in `frontmatter.md` and `note-types.md` ("Claim template version 2 includes `schema_version: 2`"). **Why:** ADR-10 (claim supersession) was PARTIAL — "the ADR's other guarantee — '`query` and `write` exclude superseded claims by default' — has no implementation … The promised `schema_version` bump on the claim template is also absent" (gap analysis §4). The bump versions the new default-read behavior so pre-bump notes are legible as read under the new default (release plan §6 known-limitation).

### Gate / provenance / query behavior

- **What:** Superseded claims are now excluded **by default** from claim query and write-assist surfaces. `link-claim.js` adds `isSupersededClaim()` and skips any `notes/claims/` file with a non-empty `superseded_by` when ranking link suggestions. `AGENTS.md` gains a standing instruction: "When querying or writing from claim notes, exclude claims with non-empty `superseded_by` by default. Include superseded claims only when the task explicitly asks for lineage, audit, or supersession history." Reflected in `frontmatter.md` and `note-types.md`. **Why:** closes the missing half of ADR-10. Design constraint recorded as a known-limitation: "this changes default read behavior, and notes authored before the claim `schema_version` bump are read under the new default" (release plan §6). Chosen as default-on-but-overridable so lineage/audit tasks can still opt in.

- **What:** Confirmed (no code change) that the deterministic Project-gate materialization is **not** a policy-gate regression. **Why:** the born-`current`/promotion enforcement decision raised the question of whether gating `projects/` would break gate materialization; the exec-plan retired it by verifying `structural_impact.py:743` "writes `project-gate-index.md` via direct `write_text` (zero obsidian-MCP), so it bypasses the policy gate entirely" (PR #647 body; exec-plan WS-1 #621).

- **What:** Runtime Obsidian-MCP TLS conformance re-verified and installer/profile HTTPS verification added (Track A #620, PR #651). Repo source already used `https://127.0.0.1:${OBSIDIAN_MCP_PORT}/mcp` + `ssl_verify: ${OBSIDIAN_MCP_SSL_VERIFY}`; alpha.6 adds fresh-deploy verification that generated profiles carry HTTPS + `ssl_verify`. **Why:** ADR-31 was the audit's single 🔴 finding — "the deployed runtime contradicts it … all five `~/.hermes/profiles/*/config.yaml` serve `http://…/mcp` with no `ssl_verify` … So `OBSIDIAN_API_KEY` rides unencrypted loopback in production" (gap analysis §1). The prior profile comment ("Hermes can't skip TLS verify … hence plain HTTP") was proven **wrong**: "Hermes feeds `ssl_verify` to httpx's `verify` (`mcp_tool.py:1355,1443`), which accepts a CA-bundle path, so the exported self-signed PEM verifies cleanly" — a runtime-config fix, no code change (PR #647). This is the one High-priority security blocker of the release; live-sandbox HTTPS smoke (S4) stayed tracked in release issues, not code.

### Distribution / evaluation — the test-env harness (ADR-80 Phase 1)

- **What:** New **model-free L0–L4 golden-path test harness** (Track B #586, PR #655). `scripts/test_env_harness.py` replays a version-controlled cassette (`fixtures/test-env/cassettes/alpha6-l4-golden-path.json`) against a disposable vault, driving deterministic Memoria operations and asserting artifact shape, schema validity, the Project gate, and the policy deny path. Cassettes match on **tool name + argument shape, not raw prompt text** ("catches wiring and schema drift while staying stable across prompt wording changes"). Wired into `scripts/e2e-smoke.sh`; pytest coverage added; new reusable procedure `docs/testing/plans/test-env-harness-plan.md`. An **opt-in `model-smoke` command** exists but is skipped unless `MEMORIA_MODEL_BASE_URL` is set. **Why:** ADR-80's own phasing — "Memoria's load-bearing logic is deterministic (Operations, the policy gate, ingest, schemas), so the model layer and the visual layer … buy the *last* increment of confidence, not the first." Phase 1 is "~20% of the cost, ~80% of the value." The key design move — L4 belongs in the model-free phase — was argued from ADR-29: "ADR-29 draws its cheap/expensive line at the L4/L5 boundary — L0-L4 are wiring, L5 is quality … The agent steps need a model only to emit a structurally-valid output, which a seed or a recording supplies, so the model is needed at record time, not run time" (commit aada1dff body; ADR-80 Phased-adoption §). The literature/online-research bet: "Gemma 4 12B is GA … official GGUF … served by `llama-server --jinja` over `/v1`," which resolved ADR-80's model-existence gate 2 and narrowed gate 3 to a ~5-min smoke (commit 06de3f66 body).

- **What:** The G3 `model-smoke` was fixed to use `tool_choice: "auto"` (not a forced function), `max_tokens` raised to 128, request timeout made configurable via `MEMORIA_MODEL_TIMEOUT` (default 90s) (commit 6348ca7f, PR #662). **Why:** "Several local OpenAI-compatible runtimes (e.g. Ollama serving Gemma) return empty output when a function is forced, but emit a correct tool_call under `tool_choice: auto` — so the smoke could never pass against the intended local model … a large model on CPU can take ~45s for a single tool call." Verified live: qwen2.5:7b on Ollama → PASS in 3.4s.

- **What:** New static **plugin provenance lock manifest** `src/.obsidian/plugin-provenance-lock.json` — pins version, upstream repo, `pinned_ref`, per-file SHA-256, license, and `local_patch_status` for the 12 vendored plugins; plus `tests/test_plugin_provenance.py` (Track A #585, PR #649). **Why:** ADR-74 (plugin provenance manifest) was deferred but its static-lock slice was "mechanical, no external dependency … 12 vendored plugins under `src/.obsidian/plugins/`, zero SHA/license manifest today" (cadence-review Tier A). Ships as a **precursor** while ADR-74 stays deferred: the manifest's own `notes` field records "Updater and CI provenance-doctor remain deferred; this lock pins the checked-in artifact hashes for manual audit" (release plan §6 known-limitation).
