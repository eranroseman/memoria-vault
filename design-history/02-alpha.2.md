## alpha.2 — The consolidation checkpoint: seven profiles → five, numbered folders → type-first, three layers → seven

**Theme.** alpha.2 is not a feature release — it is the *internal checkpoint* that ratifies the largest design reorganization in the project's history: the collapse of alpha.1's seven-specialist / numbered-folder / three-layer model into five posture-defined profiles, type-first category folders, and a seven-layer actor model, all sitting on deterministic engines behind the policy gate. **What actually landed in this commit (`0e9ed9dd`, #452) is documentation only** — it renames `docs/releasing/0.1.0/` → `0.1.0-alpha.1/`, adds the alpha.2 checkpoint plan, and repoints links; the git diff is otherwise empty. The design changes it *documents* shipped across the prior alpha.1→alpha.2 window (ADRs dated 2026-06-10/11/12; features such as workspaces v2 in `c6f1f5a5`, the eval scorer in `df25ea10`). I therefore report the consolidation the checkpoint consolidates, flagging shipped-vs-planned throughout, and note the doc-only mechanics of the commit at the end.

> Source-of-truth note: the authoritative intent is `docs/releasing/0.1.0-alpha.2/release-plan-0.1.0-alpha.2.md` and its README; the per-change *why* is mined from the superseding ADRs (46-49, 57, 68) and their Context/Alternatives sections. Verified shipped state: `src/.memoria/profiles/` (five profile dirs), `src/` top-level type-first folders, `src/.memoria/schemas/folders.yaml`, `src/system/dashboards/` (twelve dashboards).

### The checkpoint framing itself (what this release *is*)

- **What:** alpha.2 is declared an **internal checkpoint, not a formal release** — `released: false`, no tag, no GitHub Release, release-please manifest pinned at `0.0.0` (`release-plan-0.1.0-alpha.2.md` §6-7). Gates were renumbered from alpha.1's split G1-G8 (infrastructure) + G9-G11 (operability) down to a single **G1-G7** set, validated by stages **S0-S5** from a fresh clone (§2-3). — **Why:** alpha.1's plan had discovered that "green plumbing alone is not a release" and split *built* from *operable* (alpha.1 plan §2, the "core reframing"); alpha.2 folds that lesson into one gate list keyed to a live tracking issue rather than in-doc checkbox columns, "never a column here… a second list would drift" (§2, §4). The first *formal* cut is deliberately pushed to the beta (§8 roadmap). Alternative rejected: cutting a real alpha release now — rejected because the Project workspace is still empty and the UX surfaced by alpha.2 review needs an alpha.3 UI-build pass first (§8).

### Vault / concepts

- **What:** Top-level vault reorganized from **lifecycle-numbered folders** (`10-inbox/ … 50-deliverables/`, `90-assets/`, `95-archive/`) to **type-first category folders** `catalog/ notes/ projects/ inbox/ system/` plus hidden `.memoria/` and `.obsidian/` (ADR-47, shipped: `src/` tree, machine-read `src/.memoria/schemas/folders.yaml`). — **Why:** ADR-04's numbered folders "encoded lifecycle stage… which implied a pipeline," but the design update (D2/D24) found "the knowledge is a **network**, not a pipeline" (ADR-47 Context). Moving a note was a state transition that broke links and provenance; now "state changes are frontmatter edits, preserving links" and lifecycle direction lives in a `lifecycle` property (ADR-50), not a location. Alternatives rejected: keep numbered folders (every promotion is a link-breaking file move); topic folders (topics are facets, not locations — rejected since ADR-04). `90-assets/`/`95-archive/` were **retired** — derived artifacts are hidden runtime data and `archived` became a *state*, not a folder (ADR-47 Consequences).

- **What:** The entity/prose hybrid `paper-note`/`item-note` types were **split** into structured **catalog entities** (paper, person, organization, venue, dataset, repository — flat Bases-queryable frontmatter under `catalog/`) versus **source notes** (prose Zettelkasten under `notes/`) (ADR-49, D3). — **Why:** the hybrid was "a standing tension between structured bibliographic facts and interpretive prose" and "a recurring source of schema drift" (ADR-49 Context/Alternatives). The records remain the source of truth; "nothing reads a Base as data." Alternative rejected: SQLite/JSON/graph-DB store — "opaque, not git-diffable, outside the plain-text discipline."

- **What:** Type schemas consolidated into one canonical home **`.memoria/schemas/`**, read by the Linter, policy gate, installer, and tests; the type→folder map lives in `.memoria/schemas/folders.yaml` (ADR-47/49). — **Why:** to kill "scattered hardcoded field lists" — "every schema change is one YAML edit… picked up by all consumers" (ADR-49 Consequences); one machine-read source is the single authority for skeleton, gate, and tests (ADR-47 Consequences).

### Profiles / runtime

- **What:** Profiles cut from **seven** (`librarian, mapper, socratic, writer, verifier, coder, linter`) to **five posture-defined agents**: one conversational **co-PI** front plus four background lanes **Librarian / Writer / Peer-reviewer / Engineer** (ADR-48, shipped: `src/.memoria/profiles/memoria-{copi,librarian,writer,peer-reviewer,engineer}/`). — **Why:** ADR-02's seven specialists "created a real UX failure — 'who do I talk to?' — and fragmented the learning loop across agents that never conversed" (ADR-48 Context); several profiles shared one posture, forcing "artificial handoffs." The co-PI subsumes the old Socratic role and is the sole carrier of the self-improving loop (memory · /goals · skills). Mapper/Analyst folded into Librarian's four processing lanes (catalog · extract · link · map). Peer-reviewer was **deliberately kept separate** from Librarian — "the agent that synthesizes must not grade its own work" (separation of duties, ADR-48 Alternatives). Alternative rejected: co-PI + directly-conversable specialists ("reintroduces the confusion and splits the learning loop"). **Scope honesty:** ADR-48 records that the *original* plan shipped only co-PI + Librarian with the rest deferred to alpha.3, but "the PI expanded the scope mid-build — 2026-06-09 — so Writer, Peer-reviewer, and Engineer landed in v0.1.0-alpha.2 as well; their *Project-workspace workflows* still arrive with v0.1.0-alpha.3."

- **What:** **Linter and verification sweeps left the profile set** — they are now **engines, not agents** (ADR-48/46/57). — **Why:** "the 'is it an agent or an engine?' question is decided by posture + LLM judgment, not by invocation style; deterministic work never occupies a board lane" (ADR-46 Consequences). The old profile names Mapper/Socratic/Verifier/Coder/Linter "disappear from the installed set; their useful skills migrate into the consolidated lanes" (ADR-48 Consequences) — a **retirement**.

- **What:** Architecture reframed from ADR-01's **three layers** (board/workers/vault) to **seven layers** — PI · Interface · co-PI · Tasks · MCP · Engines · Vault — with three actor-kinds (PI human judgment, agents = posture+LLM, engines = deterministic) and one flow rule "decisions flow down, information flows up" (ADR-46, supersedes ADR-01). — **Why:** the three-layer model "conflated two distinctions… *where* things live (structure) and *who* acts (actor-kind)" and "hid the co-PI/Tasks split and the MCP boundary — the two things v0.1.0-alpha.2 builds against" (ADR-46 Context/Alternatives). Alternative rejected: two parallel cognitive+build models with a mapping table — "a 1:1 mapping was the smell that they were redundant; one good-enough model beats two perfect ones" (D42).

- **What:** Each agent = a shared layer (vault `AGENTS.md`) + a unique layer (`SOUL.md`, `skills/`, `config.yaml`, MCP wiring); `/personality` is now a **co-PI-only** affordance (specialist postures fixed by design) (ADR-48 Decision). — **Why:** "one conversational context compounds… instead of seven that reset; background lanes stay stateless propose-then-dispose executors" (ADR-48 Consequences). The board's lanes shrink to the four background agents — "there is no co-PI lane and no engine lane."

### Gate / provenance / trust

- **What:** New governing principle recorded: **engines write, agents judge** — "any write whose correct content is derivable by rule is performed by an engine (deterministic code), never by an LLM agent following an instruction" (ADR-57, new). — **Why:** the engine/agent split "always carried an implicit corollary… never recorded in one place" (ADR-57 Context). Driven by hard literature evidence that LLMs cannot supply **consistency** or **traceability**: production inference is not batch-invariant — "an experiment sampling one prompt 1,000 times at temperature 0 produced **80 distinct outputs**" (Thinking Machines Lab 2025), and multi-step agents show "stable tool *choice*, varying *arguments* — exactly the part that becomes file content" (arXiv:2605.28840). Alternatives weighed and rejected: tight-prompt mechanical writes ("no prompt restores batch-invariance"); lint-away variance ("the Linter is a monitor… not a substitute for writes being right"); pin providers/kernels for deterministic inference ("determinism without explainability"). Concretely in alpha.2: the ingest engine assembles records and appends the intake anchor while the Librarian fills only two judgment holes; `board_export`, `metrics_aggregate`, `golden`, sweeps, and the Linter write directly as trusted engine code (ADR-57 "How the rule lands").

- **What:** The write gate remains a **Python Hermes plugin** (`memoria-policy-gate`, ADR-28) over the **native Obsidian MCP** (ADR-31); the policy boundary is where allow-listing, write-scoping, and audit live (ADR-46). — **Why:** carried forward from the alpha.1 #58 fix — the earlier shell hook "never matched Hermes' real `mcp_obsidian_*` tool name (and shell hooks are consent-gated + fail-open)" (alpha.1 plan). alpha.2 makes the boundary's *role* explicit in the seven-layer model: "MCP is a policy gate, not an execution sandbox" (ADR-46 Decision).

- **What:** The Linter is redefined as **integrity monitor + commit gate**: a pre-commit `schema-check` gates git-tracked writes at commit (D50), and the cron/CI sweep monitors between commits, flagging drift as Inbox `flag`s (ADR-49). — **Why:** Bases "provide **no integrity guarantees** (no schema, no constraints)," so integrity had to come from elsewhere; a file-watcher write-gate daemon was **deliberately not built** — "the commit gate + sweep bound the risk acceptably" under the solo premise (ADR-49 Alternatives). Honest limitation stated: a bad live edit can serve a malformed record until the next sweep (accepted window).

- **What:** Security posture reworded from "completely sandboxed" to **policy-sandboxed via MCP** (ADR-46). — **Why:** an adversarial review found the stronger claim was over-claiming — "first-party agents get no process isolation (sufficient under the solo/local premise)… execution isolation is deferred until untrusted third-party code is run" (ADR-46 Decision/Consequences; red-team contradiction 3 also corrected the false "PI writes only through layers" claim — the PI edits the Vault directly).

### Pipeline / ingest / enrichment

- **What:** The deterministic ingest value loop (ADR-30) is carried into alpha.2 as an **engine** with the Librarian filling only judgment holes: capture → Tier-0/Tier-1 (S2 + OpenAlex + Crossref merge / extract / link) → two LLM judgments (classify constrained to `vocabulary.md`; comparative `[!brief]`) → gated multi-writes → `lifecycle: proposed`, `review_status: requested` (ADR-30/56/57; G3 in the alpha.2 plan). — **Why:** ADR-57 explicitly cites ADR-30's "scriptable-before-LLM" as the reinforced case — the engine "assembles records, builds relationships, appends the intake anchor; the Librarian fills only the two judgment holes." This was proven live end-to-end in alpha.1 (#100-#116) and is a G3 gate here.

- **What:** Inbox archival sweep and project-membership proposal from topic-overlap were wired in the window (commits `b5783d0a`, `1931631d` per ADR-15). — **Why:** sweeps are engine work that keeps the board/inbox bounded without agent involvement (consistent with ADR-57); project-hints propose membership deterministically for the future Project workspace.

### Retrieval / derived store

- **What:** Retrieval and schema extensions are explicitly **deferred** (ADR-65), as is the cluster MCP / BERTopic topic modeling (ADR-33) — `cluster_model_topics` "errors cleanly when the optional stack is absent"; graph/canvas tools work without it (alpha.2 plan §5-6). — **Why:** these are not on the checkpoint's critical path (single-machine consolidated system); deferring keeps scope to the five-profile / seven-layer core. Honest limitation flagged in §6.

### Discovery

- **What:** The **nightly discovery loop** (ADR-61) is **deferred** out of alpha.2 (plan §5). — **Why:** discovery/density-gated automation is a later phase; the checkpoint proves the *operable* ingest→review loop, not autonomous discovery (consistent with the "why-not-autonomous" stance the publication path leans on, ADR-20).

### UI / navigation / dashboards

- **What:** Workspaces reworked from two unshared layouts (Home, Library — plus a promised-but-unshipped "Project") to **three under one shared contract: Desk / Library / Studio**, with `home.md` rewritten as a **four-block control panel** (ADR-68, shipped `c6f1f5a5`). — **Why:** the old two layouts "had no shared contract" — the co-PI chat pane "appeared in neither," Library "opened on an empty tab," and the catalog base was mis-serialized as a `markdown` leaf (ADR-68 Context). The new contract pins the co-PI (`agent-client-chat-view`) in the right sidebar of *every* workspace so "the conversation partner travels with the mode," puts a real work file in every main pane, and fixes `.base` leaves to serialize as view type `bases`. **Studio replaces the planned Project workspace** (ADR-68 Decision) — since `projects/` ships empty, `research-focus.md` is the drafting anchor.

- **What:** Command **Buttons** added to `home.md` so quick actions no longer require a palette round-trip (ADR-68). — **Why:** "every quick action still required a palette round-trip" and `home.md` had grown prose clutter around its working parts (ADR-68 Context).

- **What:** Dashboard count grew from **eleven → twelve**, all Bases/Dataview-backed, and an eval-trend + skill-lifecycle dashboard were added (ADR-49, ADR-11 via `df25ea10`, ADR-43 dashboard-only via `01f2c3ff`; shipped `src/system/dashboards/`, four families: daily-glance, operational-health, structural-health, synthesis-agenda). — **Why:** G4 requires "all **twelve** dashboards render on real data" (alpha.2 plan §2, up from "eleven" in alpha.1 G4). The skill-lifecycle work chose a **dashboard-only** design — a lifecycle state machine was explicitly **rejected** (ADR-43, `01f2c3ff` commit subject). Honest limitation: `fleet-health` "is a prose placeholder until the observability aggregator lands" (§6).

### Distribution / installation / evaluation

- **What:** Installer now registers **five** profiles from `src/` (not seven), stages the golden copy, and is idempotent on re-run (G1/S3). — **Why:** direct consequence of ADR-48's 7→5 consolidation; the five `distribution.yaml` replace the alpha.1 seven (alpha.1 G8 referenced "the 7 `distribution.yaml`").

- **What:** A **deterministic rubric scorer** + system/metrics/eval log + trend dashboard shipped (ADR-11, `df25ea10`). — **Why:** the eval program instantiates the publication-measurement path (ADR-20/ADR-11); a deterministic scorer is engine work by ADR-57, not an LLM judge. `d1701057` also fixed an ADR-13 dashboard over-claim in the same window — part of the honesty sweep.

- **What:** No installable release is cut; release automation paused, manifest at `0.0.0` (plan §6). — **Why:** alphas are internal checkpoints; the first formal release-please tag is the beta (§8).

### ADRs (governance)

- **What:** Supersession map recorded — **ADR-46 supersedes ADR-01**; **ADR-47 supersedes ADR-04**; **ADR-48 supersedes ADR-02, 37, 42**; **ADR-49** and **ADR-57** are new; **ADR-68** is new (assumes 13, 48). ### Removed / retired (with reasons)

- **Profiles Mapper, Socratic, Verifier, Coder, Linter** — removed from the installed set (ADR-48). Mapper/Analyst → Librarian lanes; Socratic → co-PI posture; Linter + verification sweeps → **engines** (deterministic work must not occupy a board lane, ADR-46/57); useful skills migrated into the consolidated lanes.
- **Numbered folders `10-inbox/ … 50-deliverables/`, `90-assets/`, `95-archive/`** — retired (ADR-47). They implied a false pipeline and made promotions link-breaking file moves; `archived` became a state, derived artifacts became hidden runtime data.
- **The three-layer architecture (ADR-01)** and **hybrid `paper-note`/`item-note` types (ADR-49)** — superseded; both hid distinctions (co-PI/Tasks/MCP boundary; entity/prose) that the consolidation made explicit.
- **The old two-layout workspace set + the never-shipped "Project" layout** — replaced by Desk/Library/Studio (ADR-68); Studio stands in for Project until alpha.4.
- **A skill-lifecycle state machine** — rejected in favor of a dashboard-only design (ADR-43); state machinery was more than the stated need required.

**Net for alpha.2:** everything above is *shipped* except where marked deferred — the Project workspace and full Writer/Peer-reviewer/Engineer *project workflows* (`projects/` ships empty, alpha.4), cluster MCP (ADR-33), nightly discovery (ADR-61), multi-machine (ADR-63), native-Windows (ADR-64), retrieval/schema extensions (ADR-65), the messaging gateway, and a live `fleet-health` aggregator. The commit that bears the alpha.2 label is doc-only; the design substance shipped across the prior window and is *ratified* by this checkpoint.
