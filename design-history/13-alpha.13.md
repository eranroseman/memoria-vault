## alpha.13 — DOI catalog enrichment: capture stages, enrichment promotes

**Theme:** the multi-source scholarly enrichment subsystem lands as a worker capability on top of the alpha.12 SQLite catalog — DOI capture now stages a *durable unchecked* record and a separate `enrich-source` operation fetches Crossref/OpenAlex/Unpaywall, merges canonical CSL with per-field provenance, and is the gate that promotes a source to consumable `checked`.

**What drove this release.** The alpha.12 catalog made SQLite the working-state boundary, but `capture-source` could still promote a scholarly DOI to `checked` before any provider evidence had been fetched or reconciled — so bibliography output and retrieval could expose sources whose metadata, OA status, and *retraction state* were unverified (ADR-123 Context). alpha.13 splits capture from promotion, adds durable homes for raw content / provider payloads / provenance / blocked findings, and enforces provider etiquette through a declared allowlist. Riding alongside is a "CLI-first SSOT" documentation and decision-authority pass that makes the alpha.11–13 designs the implementation source of truth (retiring stale pre-reset ADRs and Hermes-first framing) plus a new installed-local-LLM testing tier. The cut is deliberately DOI-only and sandbox-only (`Memoria-test`), with no installed-vault migration target.

---

### Pipeline / ingest / enrichment

- **What:** Split DOI/ISBN capture from checked promotion — `capture-source` now *stages* an unchecked catalog row instead of writing a checked source. New `capture.stage_catalog_source()` writes an unchecked SQLite row + durable content/raw blobs under `.memoria/blobs/source-content/<source_id>/` and does **not** write `catalog/sources/<id>/source.md` or `references.bib`; `capture.source_requires_enrichment()` routes any input carrying a DOI/ISBN down this path, and `worker.py` dispatches accordingly (`src/memoria_vault/runtime/capture.py`, `worker.py`; manifest `vault-template/capabilities/operations/capture-source.md` gains `.memoria/blobs/source-content/` and the description "DOI/ISBN records stage for enrichment"). Non-enrichment inputs keep the alpha.12 checked-capture behavior. — **Why:** ADR-123 Decision: promoting DOI captures immediately was rejected "because checked retrieval and `references.bib` would expose sources before required provider and retraction evidence had been reconciled"; writing DOI source markdown during staging was rejected because "it recreates the catalog-frontmatter source-of-truth problem superseded by ADR-122 and leaves an unchecked source-shaped file for tools to accidentally consume." The design §1 "Capture boundary" table makes this the explicit contract.

- **What:** New `enrich-source` worker operation — the promotion gate (`src/memoria_vault/runtime/enrichment.py`, 568 new lines; new manifest `vault-template/capabilities/operations/enrich-source.md`; wired into `worker.py` and the operations table in `docs/reference/operations.md`). It loads provider config, fetches required DOI providers through the operation network allowlist, caches raw payloads, merges canonical CSL + external IDs + field provenance into SQLite, then marks the source `checked` only if required-provider and retraction checks pass. — **Why:** design §0/§3 Stage E and the exec-plan Goal: "every catalog add is enriched … merge with per-field provenance → check (incl. retraction) → commit through the alpha.12 lifecycle." Making enrichment a worker operation (not an agent tool) is a settled decision (design §8) enforcing the MCP-only sandbox — agents never get network or code-exec (ADR-32).

- **What:** Provider policy config `vault-template/.memoria/enrichment/providers.yaml` (new) — declares the `doi` branch `required: [crossref, openalex, unpaywall]`, per-provider `base_url`, `endpoint_template`, `timeout_seconds`, `rate_limit_per_second`, and `query_params` naming the `.env` variables (`OPENALEX_API_KEY`, `NCBI_EMAIL`) rather than secret values. `enrichment.provider_allowlist_issues()` fails closed if any enabled provider `base_url` is not covered by the manifest `allowed_network`. — **Why:** design §7 "One non-secret provider policy config … Actual mailto/email values and API keys live in profile `.env`, never in tracked config," and the hard rule that `providers.yaml` "is data config, not the enforcement boundary. The flat `enrich-source.md` manifest remains the hard ceiling." Keeps provider lists out of every adapter (§7) and keeps secrets untracked.

- **What:** Three provider adapters implemented inline (Crossref, OpenAlex, Unpaywall) — thin per-provider fetch + normalize, each writing a content-addressed blob under `.memoria/blobs/provider-payloads/`, a `provider_payloads` DB row, and an `api_call` journal event (provider, endpoint, request, `response_hash`, status, `latency_ms`) (`enrichment.py` `_provider_payload`/`_write_provider_blob`/`_normalize_provider`). No provider class hierarchy. — **Why:** design §8 "MVP starts with DOI required providers. Implement Crossref, OpenAlex, and Unpaywall first … No provider class hierarchy until duplicated code makes it smaller"; exec-plan "Six adapters + one reconciler; add an abstraction only when a 7th demands it." Each call as a journal `api_call` event satisfies §6 provenance ("every field is traceable to who supplied it").

- **What:** Deterministic field-authority merge + per-field provenance (`enrichment._merge_doi_source`) — Crossref is authoritative for title/authors/issued/DOI/container/type; Unpaywall for `oa_location`/license; OpenAlex supplies the OpenAlex ID and external IDs. Each canonical field records a `_provenance(field_path, value, winning_provider)` row via `state.replace_field_provenance()`. — **Why:** design §3 Stage C field-authority table and §8 "Field-authority merge with per-field provenance; high-authority conflicts → flag + `metadata_status`, never silent." Top risk in the exec-plan: "Provider disagreement — never silently pick a winner on a critical field."

- **What:** Conflict + retraction gate blocks promotion (`enrichment._is_retracted`, `_title_conflict`, `_blocked_status`) — a retracted/contested Crossref record → `blocked_status = "contested"`, a high-authority Crossref/OpenAlex title mismatch → `needs_human`; either sets `check_status = unchecked` + `metadata_status = unverified`, records a `check-fired`/failed integrity finding, and never materializes `references.bib`. — **Why:** design §6 "Retraction is never silently swallowed … blocked at the alpha.12 consumable read barrier unless explicit human resolution," and §3 Stage D conflict handling. ADR-123: "Provider failure, provider conflict, or retraction/contested evidence leaves the source unchecked … and writes a PI-reviewable attention projection under `inbox/`."

- **What:** PI-reviewable attention projections on blocked enrichment (`enrichment._write_attention_flag` → `inbox/`) — required-provider miss, provider conflict, and retraction each produce one attention file linking the source and the exact blocked check, committed via the trusted writer. — **Why:** design §3 "Human-confirmation cases produce an attention item that links the source row, candidate identifiers, provider evidence, and the exact blocked check. They do not silently become checked"; §9 "Human attention" gate. Because there is no PI catalog edit UI in this cut (ADR-121/106), attention projections are the only human-review surface.

- **What:** `references.bib` materialized from durable replay data *after* required providers pass, inside the same worker commit (`enrichment.py` renders via `capture.render_references_bib`, records the projection payload through `state.record_projection_output`, then commits). Rendering now reads checked SQLite catalog rows first, source Concepts only as fallback. — **Why:** design §1 Capture-boundary table and §9 "References materialization" gate: "citation metadata and the expected `references.bib` payload/hash are committed to SQLite first; the operation completes only after file promotion records the git commit" — the alpha.12 crash-recovery barrier. ADR-122 Consequence: "`references.bib` renders from checked SQLite catalog rows."

- **What (docs):** `docs/reference/ingest.md` rewritten around the capture/enrich split — new pipeline rows for "DOI/ISBN staging," "DOI enrichment," and "Provider policy"; read-barrier now excludes "unchecked staged catalog rows, provider payloads, source-content blobs"; deferred-work list swaps "identifier lookup" for "ISBN URL-depth enrichment … Semantic Scholar, PubMed, arXiv, graph enrichment, embeddings, and UI." — **Why:** cli-first-ssot plan §1 requires docs to "describe only implemented behavior"; the deferral list mirrors design §10 MVP cut.

### Gate / provenance / data model

- **What:** Four new SQLite tables on the alpha.12 §3 ER model (`src/memoria_vault/runtime/state.py`): `external_ids` (generic `owner_type/owner_id/namespace/value/source_provider/confidence`), `enrichment_runs` (`enrichment_status`, `required_provider_policy`, timestamps), `provider_payloads` (`run_id/provider/request_key/request_params_hash/status/raw_hash/raw_path/normalized_json/latency_ms`), and `field_provenance` (`source_id/field_path/value_hash/winning_provider/…`); plus two additive columns on `catalog_sources` (`content_path`, `raw_path`) via `_ensure_column`. New helpers `start/finish_enrichment_run`, `store_provider_payload`, `replace_external_ids`, `replace_field_provenance`, `record_projection_output`, `catalog_source`. — **Why:** design §5 data-model additions ("generic ID rows, not one schema column per provider" so future namespaces need no migration) and §2/§4 identity model. exec-plan interface note: "SQLite schema changes should be additive and idempotent … no production-vault migration target."

- **What:** Idempotency + provenance replay contract — enrichment run keyed `enrich-source:{source_id}:{hash(doi)}`, provider payloads keyed `(source_id, provider, request_key, request_params_hash)`, raw payloads content-addressed so identical inputs yield identical canonical CSL + provenance. — **Why:** design §6 Idempotency/Concurrency and §9 "Merge determinism: identical provider payloads → identical canonical record + provenance." Enables the "drop SQLite → re-enrich/rebuild" replay in the exec-plan WP5 accept criterion.

- **What:** Identity via external IDs, not name strings (`external_ids` + `_openalex_external_ids`, `_external_id`) — `authored_by`/entity edges carry DOI/OpenAlex/ORCID/ROR/ISSN namespaces. — **Why:** design §4 solves the alpha.12 `sources ↔ authors` gap: "global, stable disambiguation instead of fragile local string matching," preserving prior human merges on re-enrich (ties to the alpha.12 author-reconcile gate).

### Vault / concepts / on-disk layout

- **What:** New blob store directories + `.gitignore` and installer skeleton entries — `.memoria/blobs/provider-payloads/` and `.memoria/blobs/source-content/` (with `.gitkeep`), added to `vault-template/.gitignore` (payload contents ignored, skeleton dirs tracked) and to `folders.yaml` `skeleton`. — **Why:** design §5/§7 content-addressed cache: raw captured content and provider payloads "live in the content-addressed blob store under `.memoria/blobs/`; catalog rows store hashes/pointers" — cheap reproducible re-runs, large regenerable data never committed.

- **What:** New transient `inbox/` root — added to `folders.yaml` `transient_prefixes` and documented in `on-disk-layout.md` as "transient attention projections, not Concepts." — **Why:** ADR-119 (rewrite): "Inbox cards are not durable Concept types … They are generated attention projections over Concept and operational state." Gives blocked-enrichment attention items a declared home that converges to resolved/archived and leaves active views.

- **What (design recommendation, deferred):** design §1 records a clean-slate Concept model (records vs Concepts vs manifests vs projections; four Markdown Concept types `digest/note/hub/project`; universal frontmatter with `id/standing/links{}`), a role-first `knowledge/` keep-set (`digests/notes/hubs/projects`, `_views/`), and a kind-first directory-per-capability manifest layout (`capabilities/<kind>/<name>/capability.md`, `_generated/`). **None enforced in alpha.13.** — **Why:** design §8/§10 explicitly defer these: "Alpha.13 MVP keeps flat operation manifests … Keep current generated projection paths, including tracked `capabilities/ai-catalog.json`." The design note frames them as targets requiring "a separate structural ADR/PR before promotion," per the "simplest design that meets the requirement" discipline and to avoid migration churn on a DOI-only cut.

### ADRs

- **What:** New **ADR-122 "SQLite working state and catalog records sit behind the checked Concept boundary"** (`assumes [28,55,119,121]`, `supersedes [49]`). Runtime working state + catalog record state live in `.memoria/state/memoria.sqlite`; the DB is *not* a second live Concept store; Bases become a view/projection layer, not catalog authority; PI mutation stays enqueue-only (ADR-121), and CLI is inspect/verify/administer only, "not a PI action surface." — **Why:** Context: the alpha.12 worker "needs indexed catalog rows for materialization, idempotency, bibliography rendering, and later provider provenance, while Bases are only a view layer" — ADR-49's catalog-frontmatter-as-SSOT "is now too strong." Rejected alternatives: JSON queue files as SSOT (strandable), catalog markdown as SSOT (couples worker state to Obsidian-shaped files), promoting SQLite to the canonical Concept store (opaque to Obsidian/Git). This is the decision-authority prerequisite the cli-first plan required before relying on new behavior.

- **What:** New **ADR-123 "DOI catalog enrichment gates checked source promotion"** (`assumes [30,32,57,119,121,122]`) — codifies the capture/enrich split, the `providers.yaml` + manifest-allowlist enforcement boundary, blob caching, provenance rows, the retraction/conflict gate, `inbox/` attention, and post-pass `references.bib` materialization; scopes the MVP to DOI only. — **Why:** turns design §8 decisions into an accepted decision record; alternatives rejected are the same four from the design (immediate promotion, staging markdown, all-providers-required, CLI/DB PI edit path). Enumerates its own implementation files and contract tests (`tests/test_alpha13_enrichment.py`).

- **What:** **ADR-49 marked `superseded → ADR-122`** (`docs/adr/49-…`, README index) with an inline supersession banner; its `assumes` flipped `[47]→[119]`. — **Why:** ADR-122 Consequence: "ADR-49's catalog-frontmatter source-of-truth decision is superseded. Bases may still render catalog-shaped views … but they are projections/fallbacks, not the catalog authority."

- **What:** **ADR-119 rewritten** from "the type schema is the complete declarative contract that validates, generates, and is the single source" to **"Concept schemas own field, home, read-state, and form contracts"** (~250 lines cut/rewritten; `assumes [52,116]`). Names four contract owners (type-schema YAML, `folders.yaml`, `check_status` read-state, Modal Forms as UI-only), pins the alpha.12 Concept roster, declares inbox cards as projections (not Concepts), and says project theses need no dedicated type. — **Why:** Context: the document contract "used to be split across ADR prose, schema YAML, folder maps, templates, Modal Forms, and hand-coded detectors … old decisions about type names, lifecycle values, forms, and folder homes drift after the alpha.12 schema simplification." Consolidating the contract is what let the 10 pre-reset ADRs below be retired without losing the decision. `frontmatter.md` correspondingly drops the `scripts/gen-forms.py` authority claim.
