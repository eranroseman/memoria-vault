## alpha.4 — Plan-the-build checkpoint: the ExecPlan artifact, a settled packaging target, and the intent-gate PI surface

alpha.4 is an **internal checkpoint whose defining work is planning infrastructure, not shipped runtime**: it introduces the **ExecPlan** — a self-contained, resumable working document adapted from OpenAI Codex — and uses it to scope milestone #3 down to a buildable set; it settles (but defers) the runtime-as-a-wheel packaging target in ADR-76; and it lands an early wave of PI-surface polish that turns the three Obsidian workspaces into named **intent gates** (Desk / Library / Studio). The commit range `7581edb1..b1657d92` ends at the ExecPlan finalization, so the big implementation workstreams that ExecPlan describes (the `engines→operations` rename, ingest sources, native Windows) are *planned here and land in later PRs* — what actually ships in-range is the planning/ADR scaffolding plus the capture/dashboard/vocabulary improvements. Two clean-sheet design notes (an all-Linux test harness; the Project gate) were authored in this range and then deliberately pushed forward to alpha.5 as not-alpha.4 work.

### Planning & process (the headline change)

- **What:** Introduced the **ExecPlan** as a first-class working-document type — a new playbook (`.agents/playbooks/exec-plan.md`) + fill-in template (`.agents/templates/exec-plan.md`, 137 lines, 12 sections: Metadata → Purpose → Context → Plan → Concrete steps → Validation → Idempotence → Progress → Execution log → Surprises → Interfaces → Outcomes), registered in `.agents/README.md` with an `AGENTS.md` pointer and a new work-routing row (commit a760d0ef). **Why:** complex/multi-hour tasks had no home that survived context loss or handoff — "what was I doing and what is left" lived only in chat history. The playbook states the problem directly: a stateless agent or a novice must be able to "read it top to bottom and produce a working, observable result without asking for the next step." The approach was **adapted, not copied**, from OpenAI Codex ExecPlans and explicitly *reconciled* with three existing Memoria invariants: (a) split into playbook-vs-template to match how releases already separate the two; (b) the plan's Execution log holds only tactical/sequencing choices — architectural/product decisions still go to an ADR and are linked, preserving the ADR-only decision model; (c) an ExecPlan is a **working artifact, not a permanent record**, so instances live only in the tracked `docs/releasing/<version>/tmp/` and are deleted before the release closes (`_notes/` is gitignored and invisible to other agents, so a resumable plan can never live there) (a760d0ef commit body; `AGENTS.md` new "ExecPlans" section).

- **What:** Added the first ExecPlan instance — `docs/releasing/0.1.0-alpha.4/tmp/exec-plan-alpha.4.md` (347 lines) — inventorying all 25 open milestone-#3 issues, grouped into workstreams A–H, and recording the **2026-06-15 scope decision** (§2.2 declared authoritative) (commits 7dd0d09f, b1657d92). **Why:** the milestone was an ambiguous mix of buildable deliverables and deferred-ADR *tracking* issues; the plan's purpose is that every issue ends up "either **shipped and verifiable from the Obsidian UI** or **consciously deferred to a cadence-reviewed ADR** — with nothing left in the ambiguous middle." Honest scoping the plan surfaced: ~5 milestoned issues are pure ADR-cadence trackers (not deliverables); several PI-surface issues cite retired `docs/design/*.md` paths that must be repointed; and the full wheel migration stays deferred (only the tooling-only pyproject step is actionable) (7dd0d09f body).

- **What:** Finalized alpha.4 scope in the ExecPlan (§2.2), b1657d92. **Deferred out:** #527 HTTPS transport, #379/#559 calibration spec + score-* skills, #344 diversity reserve, #412 (ADR-62 harnesses, split to #560/#561), #370 (ADR-38 ratchet — alpha.4 keeps only shadow/report-only; enforcement → #562). **Added in:** the `engines→operations` rename (ADR-69) as a structural prerequisite (new Workstream H); #414 native Windows; #439 Mapper Tier 2; #443 contradiction-page fixes. **Why:** the 2026-06-16 revalidation against **live Hermes docs** changed the facts — Hermes now documents `ssl_verify` for HTTP/SSE MCP (unblocking #527) and native Windows support (accepting ADR-64) — so items previously deferred on missing-capability grounds became buildable, while calibration/enforcement work was split out to keep alpha.4 "free of blocked/deferred omnibus scope." The commit is careful to reframe native Windows as an **alpha.3 carry-over, not a v0.3 scope-up** (ADR-64's "revisit after alpha.3"), and flags #296's body as carrying a stale `v0.3` label to fix at source (b1657d92 second commit).

- **What:** Codified three cross-cutting **Working principles** in `AGENTS.md` (new section): *cover the whole scope* (no sampling/grep-as-read; verify sub-agent coverage), *zero tolerated contradictions* (docs must agree with each other and the implementation; sweep via the source-of-truth + change-impact maps and fix, don't log), *verify hard conclusions independently* (fresh agent + live-test the allowed/denied/fail-closed path; ground Hermes claims in `~/.hermes` docs first) (commit fcb912c9). **Why:** these rules "governed agent behavior only via one operator's private memory, not the repo" — externalizing them into the authoritative agent-policy home closes that gap. The maps are *referenced as the mechanism, not restated,* precisely to avoid the drift the zero-contradiction rule forbids. The same commit added a `.agents/playbooks/release.md` pointer to the `RELEASE_PLEASE_TOKEN` setup so the release entry point surfaces the silent-failure risk if the PAT lapses.

### Distribution / packaging

- **What:** Added **deferred ADR-76** — "Package the Python runtime as an installable wheel; vault holds data, not source" (`status: deferred`, `assumes: [44,46,69,73]`) plus the 349-line design note `install-a-real-package.md` (commits 0b388b16, bd039b7d). Target: a single-rooted `src/`-layout package `memoria` (subpackages `operations/`, `mcp/`, `runtime/policy/`, `lib/`), `pyproject.toml` as single source of truth (hatchling, `dynamic` version, three dependency tiers replacing three `requirements*.txt`, `[project.scripts]` console entry points), deploy via a SHA256-pinned wheel into `<vault>/.memoria/.venv` so `.memoria/` holds *only* data + venv, never source. **Why:** the source tree and deploy location are the *same dotted directory* — `src/.memoria` is both authored code and rsync target — forcing three standing costs the ADR enumerates: a **dotted, un-importable** top-level name; **two parallel import mechanisms that must agree** (8 `sys.path` entries in `conftest.py` + 11 `Path(__file__)` bootstraps in the deployed runtime + a 12th in the policy plugin); and a delete-pruned two-phase rsync deploy. Research issue #494 asked the question; the ADR settles the target but **defers execution** because it changes both the deploy mechanism and on-disk vault shape (real blast radius) and must not churn the tree before the `engines→operations` rename lands. It explicitly corrects a "common misconception" — that ADR-55's golden copy protects runtime *code* — proving `golden.py`'s manifest has **no `.memoria/` prefix**, so code integrity is the MCP-only sandbox + Git, and ADR-55 is untouched. The one hard prerequisite it names is a security property, not a chore: the **policy-gate version-skew** decision (lean: gate calls the running policy MCP over the wire — structurally skew-free — rather than installing a second policy copy that could drift and wave through a mutating call). Linked to cadence-tracking issue #521 (mirrors ADR-63↔#413).

- **What:** The **actionable packaging slice** is scoped in the ExecPlan (Workstream E) to a **tooling-only `pyproject.toml`** — `[tool.pytest.ini_options]` (`pythonpath`) + `[tool.ruff]` only, deleting the `conftest.py` `sys.path` block, keeping `requirements-dev.txt`, adding no `[project]` table. **Why:** ADR-76's staged migration says step-1 is "issue work, no ADR" and safe to land anytime, while the deployment flip is gated on the skew decision and the rename; the ExecPlan's **guard** is "do not add a `[project]` table or change the deployment mechanism early — that drags target-stage risk forward with no benefit."

- **What:** ADR-76's `## When this matters` names distribution to non-checkout operators as a trigger, and records the wheel **strengthens** ADR-63 (multi-machine: build-once/install-on-N beats rsync-from-checkout) and is **in tension with** ADR-26 (the install unit shifts from rsynced subtree to a built wheel). **Why:** to keep the deferred decision honest at each cadence review rather than re-litigating it.

### Pipeline / ingest / capture

- **What:** Ingest now **creates a proposed source-note stub under `notes/source/` for the PI to fill** as part of the catalog-enrich flow — the `capture-from-url.js`, `capture-from-zotero.js`, `catalog-source.js` prompts and the `catalog-enrich-record` SKILL were updated to add "create the proposed source-note stub … then enrich … propose the classification" (#512, commit 612c0af8). **Why:** captures were landing a paper entity under `catalog/papers/` but no source note, so a captured item never appeared in the reading pipeline — the fix "surfaces url/zotero captures in inbox" (companion fixes #499, #507) so the object the PI reads is actually created at ingest, not left implicit.

- **What:** Zotero capture switched from **Better BibTeX CAYW to the BBT JSON-RPC endpoint** — `capture-from-zotero.js` now POSTs `item.citationkey(["selected"])` to `/better-bibtex/json-rpc` instead of GETting the CAYW URL, parsing citekeys from the RPC response (#525, commit f615115b). **Why:** listed as a fix ("use bbt json-rpc for zotero capture") — the CAYW format returned inconsistent citation-key fields (`citationKey`/`citekey`) and title data; the JSON-RPC `item.citationkey` call is the stable, purpose-built selection API. **Why not documented for the title drop:** inferred — the RPC returns citekeys only, so `title` is now set empty and resolved downstream during ingest.

- **What:** Added **create-linked-claim** (#513, commit c5010254): a new QuickAdd user script `create-linked-claim.js` that, from an active `notes/source/` note, creates a schema-shaped claim note (`type: claim`, `maturity: seedling`, `sources: [<citekey>]`), links it back into the source's frontmatter and under "Worth distilling," and opens it; a "Create linked claim" button was added to `templates/source.md`. **Why:** distillation (source → atomic claim) was a manual, error-prone hand-edit; scripting it enforces the frontmatter schema and the source↔claim link at creation, matching the PI direct-access rule (reachable from the source note itself).

- **What:** Linter detectors gained **findings telemetry** — `detectors.py` added `--jsonl-out` (append one JSON row per finding, create an empty file on clean runs) and `append_findings_jsonl()`; the lint cron wires it (#524, commit 4f384912). **Why:** listed as "emit lint findings telemetry" — Linter runs were report-only with no persisted trail, so trends (e.g. drift accumulation) couldn't be measured; a JSONL log makes findings a queryable metrics stream (feeds the status-line/dashboards work planned in the ExecPlan).

- **What:** Renamed `scripts/refresh-retraction-watch.sh` → `retraction-refresh-cron.sh` (and updated ADR-32's "Files affected") (commit 20f2c506 area). **Why:** naming clarity — the `-cron` suffix marks it as the scheduled refresh job for the retraction sweep engine (`engines/sweeps/retraction.py`), consistent with the other `*-cron.sh` wrappers.

### ADRs (decisions governing the above)

- **What:** New **ADR-75** — "Use GitHub Project fields and release sub-issues for live work state" (`accepted`, `assumes: [45]`) (commit sequence around 5c829df3/012e376b). **Why:** ADR-45 had moved release readiness into a single "Release vX.Y" *checklist* issue, but that made each gate a checkbox rather than a work item with its own owner, evidence, and blocking discussion, and labels had accreted five meanings (type/area/priority/triage/bot). Decision: the **Memoria Issue Tracker** Project carries `Status`/`Area`/`Type`/`Priority` fields, labels stay minimal, and readiness becomes a **parent "Release vX.Y" issue with one sub-issue per gate/stage**. Rejected alternatives (recorded): keep the flat checklist (no per-gate evidence trail), use labels for the field schema (labels are search chips + bot hooks, not a planning schema), track state in markdown (reintroduces the drift class ADR-45 removed). This drove the parallel rewrites of `AGENTS.md` work-routing, `docs/releasing/README.md`, and the release-plan template (folder-per-version + thin README + `tmp/`; readiness in sub-issues; internal checkpoints use `status: complete`/`released: false`).

- **What:** Promoted **ADR-69, 70, 71, 72, 73** from `proposed` → `accepted` (dropping `nav_exclude`), dated 2026-06-14 (ADR README + individual diffs). **Why:** these are the alpha.3/alpha.4 design decisions (operations-layer naming; intent-named navigation gates; structured capture forms; command surfacing; docs-reference conventions) that the shipped UI/vocabulary/Commander work in this release implements — accepting them settles the vocabulary the code now builds on (ADR-69's own note: "the vocabulary decision is wanted now so UI work builds on settled terms"; the tree rename stays a separately-sequenced pass).

- **What:** ADR-73 promoted its `check_site_local_links` docs-doctor check from **advisory → blocking** ("`docs-doctor` blocks published→`src/` relative links"). **Why:** the published-site link hygiene sweep it was staging is complete, so the guard is now enforced rather than warn-only.

- **What:** Folded #194's borrowable ideas into existing deferred-integration registers rather than opening new issues — **ADR-58** names **Quartz** as the candidate static-HTML report renderer (and downstream renderer for the open-design deliverable flow), and **ADR-61** adds a packaged cross-lane **"deep research" macro** (after Khoj's scheduled-automation mode) built on the nightly discovery loop (commit cd74e03a, closes #194). **Why:** the comparative-repos survey surfaced two ideas that "belong in the existing deferred-integration registers rather than as new issues" — keeping them as ADR extension-notes prevents backlog bloat while preserving the provenance (the #194 survey) and the boundary discipline (an extension of the same loop, "not a separate runtime / competing system").

### Design notes authored here, then deferred to alpha.5

- **What:** Authored `test-env.md` — a **WSL2 process-testing environment** design (281 lines) — then moved it forward to `docs/releasing/0.1.0-alpha.5/tmp/` as a git rename (commits 0145dc3f then 7dd0d09f). It maps the test surface onto **ADR-29's L0–L5 pyramid**, running a WSL2 process-test env alongside the Windows production env: it refines L2b placement while explicitly **keeping the GUI-bearing tiers (L3/L4/L5) on Windows** (headless Obsidian on WSL2 via `xvfb-run` is rejected as heavy/flaky and not gating L2b). It specifies a local **Gemma 4 12B (GGUF Q4_K_M)** test model, which **supersedes the prior model decision** — flagged as a **process risk**: the superseding `MEMORIA_ENV` ADR "does not exist yet," so write it before adopting. **Why:** it keeps the interactive, GUI-dependent tiers on the native Windows + Zotero + Obsidian environment while making the deterministic/process tiers automatable in WSL2 — refining ADR-29's placement rather than replacing the environment. **Why moved:** the ExecPlan judged it "not alpha.4 work" and pushed it to alpha.5 planning (#530).

- **What:** Authored `project-starter.md` — the **Project gate** design (the fourth navigation gate deferred by ADR-70, the empty `projects/` from ADR-68; 678 lines) — then moved it to alpha.5/tmp (commits 0145dc3f, 7dd0d09f). Core model: **two graphs, not one** — a bottom-up descriptive knowledge map (Zettelkasten/MOC tradition) *and* a top-down Toulmin argument graph (`supports`/`contradicts`/`warrant` rooted at a thesis), where the **outline derives from the argument graph, not the topology** ("that single distinction is what moves the gate from 'organizes knowledge' to 'produces an argument'"); thesis-as-provisional-claim as the anchor; structural impact/saturation/termination as deterministic **Operations** in the same trust class as the contradictions dashboard. **Why:** keeps the gate inside the architecture's deepest invariant (agents propose, PI promotes, nothing auto-infers — ADR-10/48/51) — its *logic* is bias-free and replayable, never asking an LLM to judge. It states the **two-tier determinism rule** as its load-bearing honesty guard: a structural Operation can check that a thing is *articulated* (tier 2 — cheap, auditable, **gameable**) but not that it is *good* (tier 3 — PI or gated agent judgment); "determinism buys auditability of the computation, not correctness of the conclusion." **Why moved:** not alpha.4 work (#530); the Project gate build stays deferred, so Studio's dashboard still notes project drafting is out of scope.
