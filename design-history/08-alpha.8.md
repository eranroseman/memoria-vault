## alpha.8 — Runtime foundations, observability, and vocabulary discipline (a decisions-and-docs checkpoint)

alpha.8 is the "runtime-foundations & observability" checkpoint whose *implementation* is scoped to every `Readiness: Ready` tracker issue behind the ADR-76 package spine — but whose *shipped* content in this range is almost entirely design: a clean-slate telemetry re-derivation (three planes, ADRs 104-106), the graduation of the ADR-76 deployment spine and a batch of deferred ADRs to `accepted`, the retirement of the `deferred` ADR status in favor of issue-tracker readiness, the split of five bundle ADRs into eighteen single-decision ADRs (84-100), and two large vocabulary corrections — the navigation "gates → spaces" rename (ADR-101 superseding the just-authored ADR-82) and the `graph_maturity/saturation_state` field renames. The only behavior-bearing vault/runtime changes that actually land are the `src/gates/ → src/spaces/` move with its schema, the model-config templating for a local-model overlay, and the enum renames in the project structural-impact code; the rest is decisions recorded and a docs audit enforcing "history lives only in ADRs and `releasing/`." The pressure throughout is that two organizing vocabularies for one activity space, one word ("gate") carrying four load-bearing senses, and a logging posture that "grew signal-by-signal" were all standing signs of unfinished design that a checkpoint had to settle before feature work built on them.

### ADR lifecycle & governance

- **What:** The `deferred` ADR status was **retired entirely**; the lifecycle is now `proposed → accepted → superseded (or rejected)`, and scheduling/readiness moved to the linked GitHub issue's Readiness field (`docs/adr/README.md` "Status lifecycle" and "Mechanics"; `docs/adr/_template.md`; commit 4f524fce). ADR status now "records decision state only." — **Why:** the old rule ("`deferred` is revisited each release cycle, not gated") conflated *is it decided* with *is it ready to build*; separating them lets an accepted decision still carry an implementation issue at Readiness `Later`/`Blocked`/`Needs shaping`, so the ADR set stops doubling as a backlog (README "Scheduling is issue state, not ADR state").

### Vault / concepts & vocabulary

- **What:** **Navigation "gates" renamed to "spaces"; "gate" reserved for the approval checkpoint** (ADR-101, `accepted`, *supersedes ADR-82*). The four surfaces are now the Inbox · Library · Knowledge · Project **spaces**; note type `gate → space`, enum field `gate → space`, `src/gates/*.md → src/spaces/*.md`, schema `types/gate.yaml → types/space.yaml` (`type: space`, `category: spaces`, `space:` enum), `folders.yaml` map `gate: gates → space: spaces` and category list `gates → spaces`, Homepage/workspace startup target repointed to `spaces/inbox`, plugin-provenance lock digest re-pinned (commit 9aba2c65; ADR-101 Consequences). — **Why:** ADR-82 had just made the four surfaces canonical and called them "gates," but "gate" was already the architecturally central word for the structural human approval checkpoint (ADR-03) — a lexical audit found the approval sense outnumbering the navigation sense ~480:190, and "the gate" had become genuinely ambiguous (plus minor collisions: git pre-commit "commit gate," the Hermes "gateway"). The rightful owner (the industry-standard approval-gate sense) keeps the word; navigation takes the newer term. "Spaces" beat "views" because Obsidian already overloads "view" natively — reusing it would relocate the overload, not remove it (ADR-101 Context, Decision, Alternatives). The project *readiness* gate (`project-gate.*`) keeps "gate" — it is a genuine quality gate, now disambiguated from the Project space.
- **What:** **ADR-82 Four-gates canonical vocabulary** authored then immediately superseded — it retired the parallel **Compile/Compose "knowledge cycle"** vocabulary and its phase names, regrouped `how-to-guides/{compile,compose,curate}/ → {inbox,library,knowledge,project}/` (30+ guide renames), deleted `workflows/compile-and-compose.md`, and reconciled `knowledge-cycle.md` to "gates above, lanes beneath" (commit a7323a43; ADR-82). — **Why:** the system described one activity space with two overlapping-but-unaligned vocabularies — the JTBD gates the user actually navigates, and a Compile/Compose pipeline organizing the docs — plus a third partial "two modes" framing; carrying two organizing vocabularies "taxes the reader's cognitive model and is a standing sign of an unfinished design." A flow is a *process* decomposition, not a *jobs* decomposition, and it hid the high-frequency Inbox job — so the gates win and the cycle is removed outright, not demoted to a narrative (a narrative in its own vocabulary is still a second model to reconcile). The agent lanes are untouched — they are the agents' internal work breakdown, a different actor (ADR-82 Decision/Alternatives).
- **What:** Renamed project/gap/thesis schema enums: `graph_maturity → argument_stage` (with value `immature → developing`) and `saturation_state → evidence_saturation`, across `schemas/types/{project,gap,thesis}.yaml` and the computing code `operations/processing/project/structural_impact.py` (commit a7323a43). — **Why:** part of the one-JTBD-vocabulary terminology cleanup — the field names now say what they measure (argument development stage; evidence saturation) rather than graph-jargon, and `developing` reads as a stage rather than a judgment ("immature") (commit a7323a43 body).
- **What:** Documented the **Project Readiness field** and recorded the retirement of `Area`/`Type`/`Priority` GitHub Project fields (ADR-75 amendment note: "the live Project carries `Status` and `Readiness` only"; commits f2f0520a/ee2e753e/89588b3d; `docs/contributing/issue-tracking.md` rewrite). — **Why:** `Readiness` was added to "separate 'is it decided' from 'is it ready to build'" — the exact split that let `deferred` be removed from ADRs; `Area/Type/Priority` were retired as unused planning overhead. The decision to model planning state in Project fields (not labels/markdown) is unchanged (ADR-75 amendment).

### Gate / provenance / telemetry

- **What:** **Telemetry re-derived from first principles as three planes** (ADR-104, `accepted`; new `docs/explanation/architecture/telemetry-architecture.md`; commit c75a2290): **Audit** (forensic, content-free, append-only, per-write SHA-256 pairing), **Analytics** (content-free operational/publication events; derived metrics are pure reproducible projections, never a second source of truth), **Diagnostic** (why Memoria's own deterministic code broke — the previously *missing* plane). — **Why:** logging "grew signal-by-signal" and its *posture* was never decided as a whole; the three needs have opposite requirements (forensic permanence vs content-free analytics vs content-bearing debugging), so one substrate "applies the strictest rule to everything and still leaves debugging unserved" — which was the actual state (failures left only a stderr print). Rejected: one log family with levels (serves none), one merged event stream (a wash at n=1, and one bad line poisons everything), a correlation/trace ID (can't thread the unmodified Hermes runtime) (ADR-104 Decision/Alternatives).
- **What:** **Audit integrity substrate is Git, not in-file hash chaining** — Memoria explicitly does *not* add a per-entry `prev_hash` chain; per-write pairing stays, external anchoring (OpenTimestamps/RFC-3161) deferred (ADR-104). — **Why:** the vault is already a Git repo — commit history is a Merkle chain and an out-of-your-control remote is a genuine second trust domain; a linear in-file chain *forks* under multi-machine writes (ADR-63) and is recomputable by whoever owns the file, so it "adds complexity without a guarantee Git does not already give." Tamper-evidence is honestly framed as detective-not-preventive; anchoring against a distinct adversary would be "theater" under single-researcher scope (ADR-24) (ADR-104, telemetry-architecture.md).
- **What:** **Multi-machine partitioning recorded as the migration target** — when >1 machine writes, every Git-synced plane moves to per-machine files (a `machine` stamp in envelope + filename); until then single-machine files stay valid (ADR-104 "Multi-machine partitioning"). — **Why:** exactly one dispatcher runs per vault but the dispatching machine can change over time (ADR-63); per-machine files let two machines append across a Git sync without colliding. Recorded now "so the move is a refactor, not a redesign."
- **What:** **Content-light diagnostic plane** (ADR-105, `accepted`): Memoria's own Python MCP servers/Operations only; lives *outside the vault and outside Git* under `$XDG_STATE_HOME/memoria/diagnostics/`; content-light and deny-by-default for raw payloads (typed error code + SHA-256+length, never the payload); raw capture only via an ephemeral self-disarming session with a visible banner and no persistent "content=on" key; leveled/rotated/bounded; no remote telemetry — sharing is a single user-triggered redacted bundle guarded by a golden-corpus redaction self-test (commit c75a2290). — **Why:** this plane is "the first place vault-derived content could land in a log," and done naively "converts a content-never-logged system into one where a forgotten debug flag plus a routine `git push` exfiltrates unpublished research." The norm across comparable local tools (Obsidian plugins, VS Code, local AI runtimes) is ship-diagnostics-but-default-quiet-and-never-auto-exfiltrate. Rejected: content in `system/logs/` behind an allow-list (can't sanitize free-text error bodies / malformed lines whose payload *is* the content, and that tree is Git-pushed); a persistent content toggle (the classic forgotten-leak foot-gun for a single non-expert user, ADR-24) (ADR-105 Context/Alternatives).
- **What:** **Cost/disposition capture re-pointed to points Memoria controls** (ADR-106, `accepted`): per-card cost/tokens read from the **Hermes session store** (`hermes kanban show <id> --json` → `runs[].metadata.worker_session_id` joined to `~/.hermes/profiles/<lane>/state.db`), emitting Hermes's already-priced figures with provenance (`cost_source`, `pricing_version`); disposition captured at the **review action** (the board/QuickAdd accept/edit/reject), never inferred. Guarded by a pinned Hermes version + a fail-closed "Hermes cost doctor" (commit c75a2290; affects `src/.memoria/mcp/board_export.py`). ADR-20 gap re-pointed from "upstream Hermes limitation" to this capture-path fix (commit 89588b3d). — **Why:** the reference doc's claim that cost was empty because Hermes doesn't surface a card overlay "showed that explanation is misleading" — verification against installed Hermes v0.14.0 found it *does* compute and persist per-call cost, in the session store, reachable without modifying Hermes; the exporter simply queried the one endpoint that drops it. These signals can't be back-filled, so "every day they stay empty is permanently lost data" — the fix must not wait on an upstream change that isn't actually required. Rejected: wait for the overlay (indefinite empty signal); a local model-gateway proxy (re-captures priced data, adds a daemon in the inference hot path) (ADR-106 Context/Decision).
- **What:** **ADR-74 (pinned Obsidian plugin provenance) accepted** — repo now ships a static machine-readable `src/.obsidian/plugin-provenance-lock.json` recording plugin identity, upstream repo, pinned release/commit, artifact digest, license, patch status; a provenance doctor validates it (ADR-74's deferred→accepted transition landed in commit ffed6dab / #687; the lock file itself dates to alpha.6, c79f659c). — **Why:** bundled plugins (ADR-26/55 repo-as-install-unit) previously had no way to prove where a changed `main.js`/`manifest.json` came from; the lock closes the supply-chain gap so CI can verify provenance, while updater/download automation stays deferred behind the doctor being stable (release plan §5).

### Pipeline / ingest / enrichment

- **What:** **ADR-108 LiteParse** proposed (deferred): evaluates replacing the `pymupdf4llm` + OCR local-parse fallback in `extract.py` with LiteParse (LlamaIndex, Rust, Apache-2.0, local-only, no LLM, built-in Tesseract OCR, PDF+DOCX/XLSX/PPTX); keeps the subprocess `rlimit` sandbox and coherence checks; adoption gated behind a head-to-head trial on the real Zotero corpus (commit c2e321c9). — **Why:** PyMuPDF is AGPL-3.0 (the strictest license obligation in the dependency set) and a C parse surface with a CVE history (the reason ADR-30's sandbox exists); LiteParse drops the AGPL dep, swaps a memory-safe Rust core, and adds office-doc/OCR capability, and its design statement is "nearly a restatement of ADR-30/ADR-32." Deferred, not adopted: the local-parse tier is a last-resort fallback (most full text arrives pre-extracted), so the upside is bounded and "does not justify betting the deterministic spine on a library with no track record" (~3 months old) (ADR-108 Context/Proposal).
- **What:** **ADR-107 OKF import** — a foreign Open Knowledge Format bundle may enter as a new source type through the deterministic ingest pipeline (ADR-30), subject to the normal write/link gates (commit 8a476f27). — **Why:** Memoria's tolerance of OKF's "consumers MUST NOT reject" permissiveness stops at ingest — the gate, not OKF's permissive consumer rule, governs what reaches a canonical surface (ADR-107 Import).
