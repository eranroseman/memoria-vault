# Memoria — 0.1.0-alpha.13 design (catalog enrichment)

Date: 2026-06-30
Status: working design for `0.1.0-alpha.13`. Adds the **multi-source catalog enrichment subsystem** on top of the alpha.12 engine (SQLite-SSOT catalog, the worker/operation/journal boundary, the §3–§4 ER model). **Builds on, does not supersede, alpha.12.** Sandbox-only (`Memoria-test`); no installed-vault migration target. Clean-slate layout changes below are design targets and require a separate structural ADR/PR before promotion.
Method: derived from the alpha.12 catalog ER model + the requirement "every catalog item is enriched from multiple scholarly APIs (OpenAlex, Semantic Scholar, PubMed, Unpaywall, Crossref, arXiv)."

---

## 0. What alpha.13 adds
- A **capture/enrich split** for catalog adds: `capture-source` durably stages raw content + minimal metadata as unchecked, then `enrich-source` resolves identifiers → fetches required providers → merges one canonical record with **per-field provenance** → validates/checks (incl. **retraction**) → promotes the source to consumable `checked`.
- New tables on the §3 ER model: `external_ids`, `enrichment_runs` / `provider_payloads`, `field_provenance`; provider data populates author/venue/institution/topic `entities` + citation/OA edges in `catalog_links`.
- **Type-branched ingest:** DOI-keyed scholarly · ISBN-keyed book · URL/primary-source — each with its own provider set, all merging into the same canonical `sources` model.
- A clean-slate **Concept model**: records in SQLite, human-readable knowledge Concepts in Markdown, capability manifests, and generated projections isolated from authored truth.
- A clean-slate **capability manifest direction**: kind-first, directory-per-capability, generated mirrors isolated from source manifests. This is not an alpha.13 MVP prerequisite.

---

## 1. Where it lives (alpha.12 fit)
- `capture-source` remains the first **worker operation** in the ingest path, but in alpha.13 it only commits a durable, minimal, unchecked catalog record: raw/content hashes, initial identifiers, source row, `metadata_status = partial|unverified`, and `check_status = unchecked`.
- `enrich-source` consumes that staged source, writes provider evidence, and is the operation that promotes scholarly/book sources to consumable `checked` when required checks pass.
- For the alpha.13 MVP, `enrich-source` uses the current flat operation manifest:
  `capabilities/operations/enrich-source.md`. The existing loader and policy
  contract remain the owner until a structural manifest migration lands.
- Request/result schemas may be worker-side files referenced by the flat
  manifest. `capabilities/operations/enrich-source/capability.md` and adjacent
  sidecars are the clean-slate target, not required for DOI enrichment.
- **Sandbox respected:** the worker (trusted code) makes HTTP to a **declared network allowlist**; agents never get network or code-execution. Enrichment is a worker capability, not an agent tool.
- Writes still use the alpha.12 lifecycle `requested → stage → validate → check → DB commit → file materialize → project`, but the checked promotion happens after enrichment, not in the initial capture commit. **Each provider call is a journal `api_call` event** (provider, endpoint, request, `response_hash`, status, latency), so every field is traceable to who supplied it.

### Capture boundary

The checked/consumable boundary is deliberately narrow:

| Phase | Writes | May update `references.bib`? | Consumable? |
|---|---|---:|---:|
| `capture-source` | raw/content hash, minimal source row, initial identifiers, request/journal rows | No for DOI/ISBN. Source types with no required provider path may use the alpha.12 metadata check directly. | No |
| `enrich-source` required pass | provider payload hashes, external IDs, canonical CSL, provenance, source status | Yes, via the alpha.12 materialization barrier: DB rows store the expected payload/hash, then file promotion records the git commit before completion. | Yes |
| async enrichment | embeddings, citation graph, topics, volatile counts | No, unless canonical citation fields changed | Already consumable |

Do not let alpha.12 `capture_source()` promote a scholarly DOI/ISBN source to
checked before alpha.13 enrichment. If a source type has no required provider
path, it can use the existing alpha.12 metadata check directly.
`references.bib` is never part of the SQLite transaction itself; the worker
stores durable replay data and completes only after the file materialization is
recorded.

### Concept model (clean-slate recommendation)

Use four object classes. A Markdown **Concept** exists only when it has a
human-readable body worth opening in Obsidian:

| Class | Storage | Holds | Source of truth |
|---|---|---|---|
| Records | SQLite + journal | sources, people, organizations, venues, external IDs, provider payloads, field provenance | DB rows + append-only events |
| Concepts | Markdown under `knowledge/` | digests, notes, hubs, projects | frontmatter + body |
| Manifests | Markdown + adjacent JSON/schema files under `capabilities/` | operations, tools, skills, workflows | `capability.md` + sidecar contracts |
| Projections | generated files | Bases, indexes, AI catalog, `references.bib`, canvases, exports | generator inputs, never hand-edited |

Clean-slate Concept set:

| Type | Purpose | Frontmatter owns | Body owns |
|---|---|---|---|
| `digest` | Source's graph presence | `source_id`, check state, confidence, contested state, evidence pointers, typed links | machine summary, key findings, limits, contradictions, open questions |
| `note` | Atomic durable idea | `note_kind`, `claim_text`, sources/evidence, temporal qualifiers, typed links, topics, standing | claim/question/observation, evidence, reasoning, implications |
| `hub` | Curated map of an area | members, topics, links, confidence, contested state | scope, current shape, key members, tensions, gaps |
| `project` | Output-oriented working memory | thesis, scope, links, standing | direction, research questions, argument state, decisions, open gaps, outputs |

Universal Concept frontmatter:

```yaml
id: <stable-id>
type: digest | note | hub | project
title: <human title>
check_status: unchecked | checked | quarantined
standing: current | superseded | retracted | archived
description: <one-sentence summary>
links: {}
tags: []
```

Schema delta before enforcement:
- `id` becomes required only after a compatibility pass backfills existing
  Concepts from `source_id` for digests and from the canonical path slug for
  notes, hubs, and projects.
- `standing` defaults to `current` during migration; only then does it become a
  required enum (`current | superseded | retracted | archived`).
- `links` is a typed map for all Concepts. Existing digest `links: []` values
  migrate to `{}` before the linter rejects list-shaped links.
- `note_kind` starts as an optional open string; pin an enum only when UI/forms
  need filtering.
- Alpha.13 DOI enrichment does not depend on this schema migration.

Rules:
- Frontmatter is for identity, state, relations, provenance pointers, policy, and
  query fields. Body is for synthesis, rationale, instructions, and curation.
- `source`, `person`, `organization`, and `venue` are records, not Markdown
  Concepts. Their Obsidian presence comes through digests and generated views.
- `claim`, `question`, `definition`, and `observation` stay one `note` type with
  `note_kind` until validation or lifecycle genuinely diverges.
- `operation`, `tool`, `skill`, and `workflow` are capability manifests, not
  knowledge Concepts.
- Generated reports and indexes are projections. They do not become Concepts
  unless a PI edits them into durable knowledge.

Rejected clean-slate alternatives:
- **Keep every structured thing as Markdown.** Easy to inspect, but duplicates
  DB metadata and creates editable files for records that should be reconciled
  transactionally.
- **Split notes into many semantic types up front.** Clear names, but premature
  schema spread. Use `note_kind`; split only when enforcement needs differ.
- **Put provenance/status in folders.** Browseable, but breaks links and hides
  state from queryable metadata. Use frontmatter + journal.
- **Make generated artifacts Concepts.** Convenient, but confuses source with
  projection. Keep them generated until a human curates them.

### Knowledge folder structure (clean-slate recommendation)

Use a role-first keep-set. Folders encode artifact role, never topic, lifecycle,
author, provenance, or check state. This is the structural target; alpha.13 MVP
keeps existing generated paths unless a separate layout PR is scheduled:

```text
knowledge/
  index.md
  _views/
    knowledge.base
  digests/
    <source-id>.md
  notes/
    <atomic-note-slug>.md
  hubs/
    <topic-slug>.md
  projects/
    <project-slug>/
      project.md
      drafts/
      projections/
      exports/
      assets/
```

Rules:
- `digests/` is the graph presence of catalog sources: one checked/machine
  summary per source or source bundle, keyed by `source_id`.
- `notes/` holds atomic PI-meaningful claims, questions, observations, and
  synthesis fragments. Do not split this by topic or maturity.
- `hubs/` replaces topic folders. Topic structure lives in hubs, links, and
  controlled vocabulary fields.
- `projects/` is nested-only: `knowledge/projects/<project>/project.md` is the
  project Concept; drafts, exports, assets, and generated projections live next
  to it.
- `_views/` contains Bases/query definitions only. Generated project artifacts
  live under `projects/<project>/projections/`.
- Compatibility rule: alpha.13 keeps `knowledge/views/knowledge.base` until the
  folder schema, installer skeleton, Bases, docs, and tests move together.

### Capability manifest structure (clean-slate recommendation)

Use a **kind-first, directory-per-capability** layout. Kinds are product concepts,
not transport details. This is the target layout for a structural migration; the
alpha.13 MVP stays on flat operation Markdown:

```text
capabilities/
  README.md

  operations/
    enrich-source/
      capability.md
      input.schema.json
      output.schema.json
      examples/
    capture-source/
      capability.md
      input.schema.json
      output.schema.json

  tools/
    zotero-local-api/
      capability.md
    qmd-search/
      capability.md

  skills/
    docs-review/
      capability.md

  workflows/
    capture-enrich-digest/
      capability.md

  _generated/
    ai-catalog.json
    index.md
    capabilities.base
```

Rules:
- Authored truth is only `*/<capability>/capability.md` plus files next to it.
- `_generated/` holds catalog/index/Base projections and is never hand-edited.
- Use `tools/` for callable external/local tools; MCP is a transport detail, not
  a top-level capability kind.
- Domain groupings (`library`, `integrity`, `knowledge`) are generated views or
  tags, not source folders.
- Keep source manifests separate from runtime state. Runtime status lives in
  SQLite/journal/check outputs, not in `capability.md`.
- Capability manifest frontmatter is executable policy; body is purpose, usage,
  safety notes, examples, and failure modes. Required policy must not be buried
  in prose.

Manifest envelope:

```yaml
---
apiVersion: memoria/v1alpha1
kind: Operation
metadata:
  name: enrich-source
  title: Enrich source
  description: Resolve and merge scholarly provider metadata for one source.
  tags: [library, enrichment, alpha13]
spec:
  runner: local
  entrypoint: memoria_vault.runtime.enrichment:enrich_source
  permissions:
    tools: [trusted_writer, provider_fetch]
    files:
      read: [capabilities/]
      write: [journal/, references.bib, .memoria/blobs/]
    db:
      read: [catalog.sources, catalog.external_ids, catalog.provider_payloads]
      write:
        - catalog.sources
        - catalog.entities
        - catalog.catalog_links
        - catalog.external_ids
        - catalog.enrichment_runs
        - catalog.provider_payloads
        - catalog.field_provenance
    network:
      - https://api.openalex.org/
      - https://api.semanticscholar.org/
      - https://api.crossref.org/
      - https://api.unpaywall.org/
      - https://eutils.ncbi.nlm.nih.gov/
      - https://export.arxiv.org/
  io:
    inputSchema: ./input.schema.json
    outputSchema: ./output.schema.json
  effects:
    readOnly: false
    destructive: false
    idempotent: true
    approval: queue
  checks:
    required: [memoria-profile, source-metadata, retraction]
---
```

This is intentionally one extra folder level, not a taxonomy. The payoff is that
schemas, examples, prompt text, fixtures, and trust notes can travel with one
capability without creating later migration churn.

Flat-manifest compatibility:
- Alpha.13 `enrich-source.md` keeps the current operation fields (`type`,
  `operation_id`, `allowed_tools`, `allowed_paths`, `allowed_network`, `runner`,
  `io_schema`, `risk_class`, `required_checks`).
- `allowed_paths` remains file-only. DB access is authorized by the trusted
  worker entrypoint and schema migration, not by pretending `catalog/` is a
  writable filesystem path.
- The generated capability catalog is not tracked or shipped for alpha.13.
  `write_ai_catalog()` may render `capabilities/ai-catalog.json` on demand when
  explicitly invoked. `_generated/ai-catalog.json` is part of the later manifest
  layout migration.

## 2. Provider roles (who is authoritative for what)

| Provider | Key | Authoritative for | Etiquette / auth |
|---|---|---|---|
| **Crossref** | DOI | Registered bibliographic metadata, references, funders, license, container/ISSN | polite pool + `mailto`; no key; ~50/s |
| **OpenAlex** | DOI / OpenAlex ID | Disambiguated authors (ORCID), venues (ISSN), institutions (ROR), **topics**, **citation graph** (`referenced_works`), OA locations, `cited_by_count` | `mailto`; ~10/s, 100k/day |
| **Semantic Scholar** | DOI / arXiv / PMID | Abstract, **TLDR**, **SPECTER embedding**, citation **intent/influential** flags, fields-of-study, S2 author IDs | API key (1/s unauth) |
| **PubMed** (E-utils) | PMID | **MeSH** terms, biomedical abstract, pub types, grants, affiliations, PMCID | NCBI key for 10/s; idconv for PMID↔DOI |
| **Unpaywall** | DOI | **Best legal OA PDF** location, OA status/version/license | `email` param; ~100k/day |
| **arXiv** | arXiv ID | Preprint metadata, categories, versions, preprint↔published DOI link | gentle (~1 req/3s) |

Routing is conditional: PubMed only when a PMID resolves (or a biomed venue), arXiv only for preprints — never call a provider an item cannot key into.

### Tier-1 additions (fill real gaps; add by need)
- **Crossref Retraction Watch data + PubPeer** — *integrity-critical, on-thesis.* Retracted/contested status feeds `check_status` + a flag. (Partly reachable via Crossref `update-to`.)
- **ORCID + ROR** — authoritative resolvers for author / institution identity (OpenAlex/S2 give the IDs; these resolve them).
- **Open Library / Google Books (ISBN)** — books/monographs, which none of the six DOI-centric providers handle (essential for a humanities/historian library).
- **DataCite (+ Zenodo)** — datasets, software, many preprints (the non-Crossref-DOI half), pairing with the CFF/CodeMeta/DataCite citation standards.

### Tier-2 (situational / domain-conditional)
CORE (aggregated OA **full text**, not just a link) · bioRxiv/medRxiv (biomed preprints) · NASA ADS / INSPIRE-HEP (astro/physics) · OpenCitations / OpenAIRE / Crossref Event Data (citation/funding breadth) · Wikidata (entity linking / QIDs).

### Tier-3 (skip: cost/license, open equivalents exist)
Scopus, Web of Science, Dimensions, Altmetric, PlumX.

## 3. The pipeline
**Stage A — Resolve identifiers (type-branch).** From whatever the item has (DOI / arXiv / PMID / ISBN / title+authors / PDF / URL), build the canonical key set. **DOI is the master key**; backfill via cross-walk (NCBI idconv for PMID↔DOI↔PMCID, arXiv↔published-DOI, Crossref/OpenAlex title search → candidate DOI with a confidence threshold). No confident key → **flag for human confirmation, do not guess.** Branch:
- **DOI-keyed scholarly** → Crossref required; OpenAlex + Unpaywall required for alpha.13 unless explicitly configured as degraded; Semantic Scholar, PubMed, arXiv, Retraction Watch/PubPeer, ORCID/ROR are conditional or async.
- **ISBN-keyed book** → one successful Open Library or Google Books result is required.
- **URL / primary source** (a historian's web/archival materials, no identifier) → URL metadata extraction is required; Wayback availability is optional unless the capture policy requires archiving.

**Stage B — Fan-out fetch (concurrent, rate-governed).** Each provider has a thin **adapter** that fetches by key, caches the **raw payload as a content-addressed blob** under `.memoria/blobs/provider-payloads/`, stores the blob hash in the DB, and normalizes it into a common `ProviderRecord` (CSL-ish fields + external IDs + relations + `raw_hash`). A shared **token-bucket governor** enforces per-provider rate limits; per-provider timeout + bounded retry; **graceful degradation** — required provider failure leaves the item unchecked and queued for retry/human attention, optional provider failure does not block promotion.

**Stage C — Merge + reconcile (the heart).** A shared reconciler merges the `ProviderRecord`s by a **field-authority table** into the canonical `sources.csl_json`, recording **per-field provenance**:

| Field | Authority order |
|---|---|
| Title, authors(names), dates, container, license, references | Crossref → OpenAlex → PubMed → S2 |
| Author identity (ORCID/IDs), venue ISSN, institution ROR, topics, citation edges | OpenAlex → S2 |
| Abstract | PubMed → Crossref → S2 → OpenAlex; TLDR from S2 |
| MeSH / biomedical terms | PubMed |
| OA PDF location | Unpaywall → OpenAlex/S2 OA locations |
| Embedding | S2 (SPECTER) |
| Preprint version/category | arXiv |

Conflicts between high-authority providers on a critical field (title mismatch, divergent DOI) → a `check-fired` flag and `metadata_status = partial/unverified`.

**Stage D — Validate + check.** Schema (Pydantic adapter), identifier integrity, cross-provider consistency, and a **retraction/erratum check**. The MVP gate uses Crossref relation/update metadata; PubMed pub-types, Retraction Watch, and PubPeer harden the signal when available. Derive `metadata_status` from provider agreement/coverage, derive `enrichment_status` from run state, and reserve `check_status` for the alpha.12 read barrier.

**Stage E — Commit + project.** One worker operation writes `sources`, the populated `entities`/`catalog_links`, `external_ids`, and `field_provenance`; promotes `check_status` only if the gate passes; stores the durable `references.bib` payload/expected hash; then materializes `references.bib` and records the git commit before the operation completes. **Async backfill** (embeddings, full citation graph, topics, volatile counts) runs as a follow-up non-blocking operation so the source is usable as soon as the *required* providers land.

Human-confirmation cases produce an attention item that links the source row,
candidate identifiers, provider evidence, and the exact blocked check. They do
not silently become checked.

## 4. Identity reconciliation (solves the alpha.12 `sources ↔ authors` gap)
Authors are linked by **ORCID / OpenAlex Author ID / S2 author ID** (not name strings), venues by **ISSN-L**, institutions by **ROR**. So `catalog_links` `authored_by` edges carry external IDs → **global, stable disambiguation** instead of fragile local string matching. `csl_json` keeps the literal names; the entity graph carries the resolved identities. On re-enrichment, prior human merge/disambiguation decisions are preserved (FK-by-identity match), per the alpha.12 author-reconcile gate.

## 5. Data-model additions (on the alpha.12 §3 ER model)
- **`external_ids`**: generic ID rows, not one schema column per provider:
  `owner_type`, `owner_id`, `namespace`, `value`, `source_provider`, `confidence`,
  `verified_at`. Namespaces cover DOI, OpenAlex, S2, PMID, PMCID, arXiv, ISBN,
  DataCite, ORCID, ROR, ISSN-L, and future IDs without another migration.
- **`enrichment_runs`**: one run per source/enrichment request with
  `enrichment_status` (`pending`, `enriched`, `partial`, `failed`,
  `needs_human`, `contested`), `required_provider_policy`, `started_at`,
  `finished_at`, and the operation request/journal IDs.
- **`provider_payloads`**: per provider request —
  `run_id`, `provider`, `request_key`, `request_params_hash`, `status`,
  `fetched_at`, `raw_hash` (→ blob), `normalized_json`, `error`, `latency_ms`,
  `retry_count`, `ttl_until`, and optional provider version/rate-limit metadata.
- **`field_provenance`**: per canonical field —
  `source_id`, `field_path`, `value_hash`, `winning_provider`,
  `evidence_payload_id`, `alternatives_json`, `confidence`, and
  `conflict_status`.
- Naturally populated: author/venue/institution/topic `entities`; `cites` edges in `catalog_links` (among *your* sources only); `best_oa_location` on the source.

**No `catalog/` markdown folder.** The catalog is **SQLite-SSOT** (alpha.12), so enrichment writes only the DB tables (`sources` / `entities` / `catalog_links` / `external_ids`) + materializes `references.bib`; there is no per-source markdown to write or keep in sync. Raw captured content and provider payloads live in the content-addressed blob store under `.memoria/blobs/`; catalog rows store hashes/pointers. Consequences:
- **The `digest` is the source's sole presence in the knowledge graph** (1:1, in `knowledge/digests/`) — Obsidian shows one node per ingested source: the machine summary.
- **Link model:** keep-set Concepts (notes / hubs / digests) reference a source by **`source_id`** (engine-resolved, surfaced through the digest), **not** a `[[wikilink]]` to a markdown source file. They still embed the alpha.12 compact Citation survival payload (`source_id`, title, authors, issued, and a DOI/URL/citekey locator) so the keep-set remains useful without the DB.
- **PI surface:** ADR-121 makes PI mutation an in-Obsidian queued-control surface, so
  the new catalog CLI is **not** a PI surface. Pause it as operator/test tooling until
  the engine path is verified; after that, add an Obsidian UI that reads catalog rows
  and enqueues worker jobs for edits and human resolutions. The portable takeout stays
  `references.bib`.

This refines alpha.12 §3 and aligns with ADR-121 — the `catalog/` markdown folder is
removed; `references.bib` and the digest are the catalog's shipped projections; the CLI
is verification/operator-only until proven; PI catalog actions belong in Obsidian.

## 6. Integrity & provenance
- **Provenance:** each provider call → an `api_call` journal event; the cached payload makes re-merge deterministic.
- **Sandbox:** the worker does the HTTP against a declared allowlist; agents never get network/code-exec. Enrichment is a worker capability, not an agent tool.
- **Retraction is never silently swallowed:** a retracted/contested source may remain a structurally valid catalog row, but it is blocked at the alpha.12 consumable read barrier (`check_status != checked`) unless explicit human resolution records why it is allowed.
- **Idempotency:** operation key is
  `enrich-source:{source_id}:{normalized_identifier_hash}`; provider payloads are
  keyed by `(source_id, provider, request_key, request_params_hash)`. Raw payload
  hashes detect changed provider data; provider version/header data is optional
  evidence, not part of the idempotency contract.
- **Concurrency:** only one active enrichment run per source/normalized key may
  promote a source. Later re-enrichment runs append evidence and update canonical
  fields atomically.

## 7. Resilience & etiquette (do not cut)
- Concurrent fan-out + **per-provider token-bucket** rate limiting; per-provider timeout + bounded retry/backoff.
- **Content-addressed cache** of raw payloads → cheap, reproducible re-runs; respect cache TTL for volatile fields (`cited_by_count`, retraction status).
- **Graceful degradation:** required providers block checked promotion; optional enrichers async-backfill; a provider outage flags-for-retry or human attention, never deletes the captured item.
- **One non-secret provider policy config** for env-var names, rate limits, required-vs-optional provider sets, and the OA-only-PDF toggle. Actual mailto/email values and API keys live in profile `.env`, never in tracked config.

Required-provider policy:

| Source branch | Required for checked promotion | Optional / async |
|---|---|---|
| DOI scholarly | Crossref; OpenAlex + Unpaywall unless explicitly degraded | Semantic Scholar, PubMed, arXiv, Retraction Watch/PubPeer, ORCID/ROR, DataCite |
| ISBN book | Open Library or Google Books | the other book provider, Wikidata |
| URL / primary source | URL metadata extraction | Wayback availability/archive, Wikidata |

Keep this policy data-driven in one config file; do not bake provider lists into
every adapter. The planned owner is
`vault-template/.memoria/enrichment/providers.yaml`: it names provider branches,
timeouts, token buckets, required/optional roles, and the `.env` variable names
for `mailto` and keys.

## 8. Decisions (this iteration)
- **Enrichment is a worker operation, not an agent tool** (MCP-only sandbox; declared network allowlist).
- **Capture is provisional; enrichment promotes.** Alpha.13 capture persists raw/minimal unchecked state, and enrichment is the default checked/consumable gate for DOI/ISBN sources.
- **Concept boundary:** only `digest`, `note`, `hub`, and `project` are Markdown knowledge Concepts. Catalog entities are records; capabilities are manifests; Bases/indexes/catalogs/canvases are projections.
- **Alpha.13 MVP keeps flat operation manifests.** `enrich-source` is
  `capabilities/operations/enrich-source.md`; directory-per-capability manifests
  and `_generated/` mirrors are deferred to a structural manifest migration.
- **`tools/` replaces `mcp/` only in the clean-slate manifest model** because MCP
  is a transport, not the capability kind.
- **DOI is the master key; type-branch at resolution** (DOI / ISBN / URL-primary-source).
- **Field-authority merge with per-field provenance;** high-authority conflicts → flag + `metadata_status`, never silent.
- **Author/venue/institution identity via external IDs** (ORCID/ROR/ISSN/OpenAlex/S2), not strings.
- **Retracted/contested source → flagged and blocked from consumable `checked` unless human-resolved.**
- **Provider raw payloads cached as content-addressed blobs;** enrichment idempotent + re-runnable.
- **Required providers block checked promotion; optional enrichers async-backfill.**
- **MVP starts with DOI required providers.** Implement Crossref, OpenAlex, and Unpaywall first; add conditional provider functions as branches need them. No provider class hierarchy until duplicated code makes it smaller.
- **No `catalog/` markdown folder** — catalog is SQLite-SSOT only; `references.bib` is
  the portable projection, the `digest` is the source's graph presence, and PI catalog
  view/edit is paused until a verified engine path is exposed through Obsidian UI
  queued operations (refines alpha.12 §3; ADR-121).
- **Link model:** keep-set Concepts reference sources by engine-resolved `source_id`, not markdown wikilinks, and embed compact Citation survival data for takeout.
- **`knowledge/` subfolders kept** (`digests/` / `notes/` / `hubs/` / `projects/`) because they are Concept roles, not topics or lifecycle states.
- **Alpha.13 keeps current generated projection paths.** `knowledge/_views/` and
  `capabilities/_generated/` are structural targets, not MVP gates.
- **Project Concepts target nested-only paths** at
  `knowledge/projects/<project>/project.md`; any flat project files are
  compatibility-only migration inputs when the structural PR runs.
- **One `note` type with `note_kind`** until a second note family needs different validation, body shape, or lifecycle.

## 9. Pre-implementation gates
- **Capture boundary:** DOI/ISBN capture creates a durable unchecked source; only enrichment can promote it to consumable `checked`.
- **Concept boundary:** no `source`, `person`, `organization`, `venue`, `operation`, `tool`, `skill`, or `workflow` Markdown file is treated as a knowledge Concept; loaders route each object class through records, Concepts, manifests, or projections.
- **Concept schema:** alpha.13 does not enforce the universal frontmatter migration. If scheduled, the migration first backfills `id`, `standing`, and map-shaped `links`, then updates schemas/templates/Bases/docs/tests together.
- **Knowledge layout:** alpha.13 keeps existing generated paths. A later layout PR must move folder schema, installer skeleton, Bases, generated projections, policy/path fallbacks, docs, and tests together before rejecting old paths.
- **Capability layout:** alpha.13 loads `enrich-source` from the existing flat operation manifest. A later manifest migration must add loader/projection support before moving operations into directories.
- **Schema contract:** the queued request and worker result validate against the operation's declared input/output schema names or files.
- **Provider policy:** required-vs-optional providers are loaded from `vault-template/.memoria/enrichment/providers.yaml` and differ by DOI / ISBN / URL branch; secrets and mailto values come from profile `.env`.
- **Resolution:** DOI / ISBN / URL each produce the right key set; no-confident-key → flagged, not guessed.
- **Merge determinism:** identical provider payloads → identical canonical record + provenance.
- **Conflict:** high-authority disagreement → flag + `partial/unverified`, never silent.
- **Retraction:** a retracted DOI → flagged, never consumable-`checked`.
- **Provenance:** every field traces to a provider + a cached payload; every call journaled.
- **References materialization:** citation metadata and the expected `references.bib` payload/hash are committed to SQLite first; the operation completes only after file promotion records the git commit.
- **Idempotency/concurrency:** duplicate queued enrich requests collapse to one active run per source/key; re-enrich updates only changed fields, preserves history + prior merge decisions.
- **Human attention:** no-confident-key, provider conflict, required-provider outage, and retraction each create a reviewable attention item with evidence links.
- **PI surface:** alpha.13 does not accept a CLI-based PI workflow. Verify the new
  engine/CLI path first as operator/test tooling; PI view/edit/resolution must later
  ship as an Obsidian control-panel UI that enqueues worker jobs under ADR-121.
- **Etiquette:** rate-limit governor honored; `mailto`/keys present; degradation on provider failure/outage.
- **Identity:** ORCID/ROR/ISSN link entities; human merges preserved across re-enrich (ties to the alpha.12 author-reconcile gate).

## 10. Alpha.13 MVP cut

Ship the DOI path first:
- Capture creates an unchecked minimal source.
- Crossref, OpenAlex, and Unpaywall fetch and cache raw payloads.
- The reconciler writes canonical CSL, external IDs, field provenance, and
  a durable `references.bib` payload, then materializes the file before the
  operation completes.
- Retraction/contested checks block consumable `checked`.
- No PI catalog view/edit surface ships in the DOI MVP. The new CLI engine remains
  verification/operator tooling; PI-facing catalog actions wait for Obsidian UI.
- Optional providers and graph features can append evidence later.
- No knowledge folder migration, capability manifest migration, generated-path
  migration, or universal Concept frontmatter enforcement is required for the
  DOI MVP.

Defer ISBN, URL-primary depth, Semantic Scholar, PubMed, arXiv, full citation
graph, embeddings, topic expansion, and full-text acquisition unless a scheduled
alpha.13 issue explicitly requires them.

Deferred structural work:
- Directory-per-capability manifests (`capabilities/<kind>/<name>/capability.md`)
  plus `_generated/` mirrors.
- Generated knowledge views under `knowledge/_views/`.
- Universal Concept frontmatter enforcement (`id`, `standing`, map-shaped
  `links`).
- Project path rejection for flat project files after compatibility migration.
