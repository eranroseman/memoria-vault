# Memoria — 0.1.0-alpha.13 design (catalog enrichment)

Date: 2026-06-30
Status: working design for `0.1.0-alpha.13`. Adds the **multi-source catalog enrichment subsystem** on top of the alpha.12 engine (SQLite-SSOT catalog, the worker/operation/journal boundary, the §3–§4 ER model). **Builds on, does not supersede, alpha.12.** Sandbox-only (`Memoria-test`); no migration target.
Method: derived from the alpha.12 catalog ER model + the requirement "every catalog item is enriched from multiple scholarly APIs (OpenAlex, Semantic Scholar, PubMed, Unpaywall, Crossref, arXiv)."

---

## 0. What alpha.13 adds
- A **capture/enrich split** for catalog adds: `capture-source` durably stages raw content + minimal metadata as unchecked, then `enrich-source` resolves identifiers → fetches required providers → merges one canonical record with **per-field provenance** → validates/checks (incl. **retraction**) → promotes the source to consumable `checked`.
- New tables on the §3 ER model: `external_ids`, `enrichment_runs` / `provider_payloads`, `field_provenance`; provider data populates author/venue/institution/topic `entities` + citation/OA edges in `catalog_links`.
- **Type-branched ingest:** DOI-keyed scholarly · ISBN-keyed book · URL/primary-source — each with its own provider set, all merging into the same canonical `sources` model.
- A clean-slate **capability catalog layout**: kind-first, directory-per-capability, generated mirrors isolated from source manifests.

---

## 1. Where it lives (alpha.12 fit)
- `capture-source` remains the first **worker operation** in the ingest path, but in alpha.13 it only commits a durable, minimal, unchecked catalog record: raw/content hashes, initial identifiers, source Concept, `metadata_status = partial|unverified`, and `check_status = unchecked`.
- `enrich-source` consumes that staged source, writes provider evidence, and is the operation that promotes scholarly/book sources to consumable `checked` when required checks pass.
- Its capability lives as a self-contained directory, not a loose markdown file:
  `capabilities/operations/enrich-source/capability.md` plus adjacent
  `input.schema.json`, `output.schema.json`, and examples/fixtures when needed.
- **Sandbox respected:** the worker (trusted code) makes HTTP to a **declared network allowlist**; agents never get network or code-execution. Enrichment is a worker capability, not an agent tool.
- Writes still use the alpha.12 lifecycle `requested → stage → validate → check → DB commit → references.bib → project`, but the checked promotion happens after enrichment, not in the initial capture commit. **Each provider call is a journal `api_call` event** (provider, endpoint, request, `response_hash`, status, latency), so every field is traceable to who supplied it.

### Capture boundary

The checked/consumable boundary is deliberately narrow:

| Phase | Writes | May update `references.bib`? | Consumable? |
|---|---|---:|---:|
| `capture-source` | raw/content hash, minimal source row, initial identifiers, request/journal rows | No, unless an already-complete BibTeX/Zotero record satisfies the source-metadata check without enrichment | No |
| `enrich-source` required pass | provider payload hashes, external IDs, canonical CSL, provenance, source status | Yes, in the same transaction as the canonical catalog update | Yes |
| async enrichment | embeddings, citation graph, topics, volatile counts | No, unless canonical citation fields changed | Already consumable |

Do not let alpha.12 `capture_source()` promote a scholarly DOI/ISBN source to
checked before alpha.13 enrichment. If a source type has no required provider
path, it can use the existing alpha.12 metadata check directly.

### Capability catalog structure (Option B)

Use a **kind-first, directory-per-capability** layout. Kinds are product concepts,
not transport details:

```text
capabilities/
  README.md

  operations/
    enrich-source/
      capability.md
      input.schema.json
      output.schema.json
      examples/
    capture-source/
      capability.md
      input.schema.json
      output.schema.json

  tools/
    zotero-local-api/
      capability.md
    qmd-search/
      capability.md

  skills/
    docs-review/
      capability.md

  workflows/
    capture-enrich-digest/
      capability.md

  _generated/
    ai-catalog.json
    index.md
    capabilities.base
```

Rules:
- Authored truth is only `*/<capability>/capability.md` plus files next to it.
- `_generated/` holds catalog/index/Base projections and is never hand-edited.
- Use `tools/` for callable external/local tools; MCP is a transport detail, not
  a top-level capability kind.
- Domain groupings (`library`, `integrity`, `knowledge`) are generated views or
  tags, not source folders.
- Keep source manifests separate from runtime state. Runtime status lives in
  SQLite/journal/check outputs, not in `capability.md`.

Manifest envelope:

```yaml
---
apiVersion: memoria/v1alpha1
kind: Operation
metadata:
  name: enrich-source
  title: Enrich source
  description: Resolve and merge scholarly provider metadata for one source.
  tags: [library, enrichment, alpha13]
spec:
  runner: local
  entrypoint: memoria_vault.runtime.enrichment:enrich_source
  permissions:
    tools: [trusted_writer, provider_fetch]
    paths:
      read: [catalog/, capabilities/]
      write: [catalog/, journal/, references.bib]
    network:
      - https://api.openalex.org/
      - https://api.semanticscholar.org/
      - https://api.crossref.org/
      - https://api.unpaywall.org/
      - https://eutils.ncbi.nlm.nih.gov/
      - https://export.arxiv.org/
  io:
    inputSchema: ./input.schema.json
    outputSchema: ./output.schema.json
  effects:
    readOnly: false
    destructive: false
    idempotent: true
    approval: queue
  checks:
    required: [memoria-profile, source-metadata, retraction]
---
```

This is intentionally one extra folder level, not a taxonomy. The payoff is that
schemas, examples, prompt text, fixtures, and trust notes can travel with one
capability without creating later migration churn.

## 2. Provider roles (who is authoritative for what)

| Provider | Key | Authoritative for | Etiquette / auth |
|---|---|---|---|
| **Crossref** | DOI | Registered bibliographic metadata, references, funders, license, container/ISSN | polite pool + `mailto`; no key; ~50/s |
| **OpenAlex** | DOI / OpenAlex ID | Disambiguated authors (ORCID), venues (ISSN), institutions (ROR), **topics**, **citation graph** (`referenced_works`), OA locations, `cited_by_count` | `mailto`; ~10/s, 100k/day |
| **Semantic Scholar** | DOI / arXiv / PMID | Abstract, **TLDR**, **SPECTER embedding**, citation **intent/influential** flags, fields-of-study, S2 author IDs | API key (1/s unauth) |
| **PubMed** (E-utils) | PMID | **MeSH** terms, biomedical abstract, pub types, grants, affiliations, PMCID | NCBI key for 10/s; idconv for PMID↔DOI |
| **Unpaywall** | DOI | **Best legal OA PDF** location, OA status/version/license | `email` param; ~100k/day |
| **arXiv** | arXiv ID | Preprint metadata, categories, versions, preprint↔published DOI link | gentle (~1 req/3s) |

Routing is conditional: PubMed only when a PMID resolves (or a biomed venue), arXiv only for preprints — never call a provider an item cannot key into.

### Tier-1 additions (fill real gaps; add by need)
- **Crossref Retraction Watch data + PubPeer** — *integrity-critical, on-thesis.* Retracted/contested status feeds `check_status` + a flag. (Partly reachable via Crossref `update-to`.)
- **ORCID + ROR** — authoritative resolvers for author / institution identity (OpenAlex/S2 give the IDs; these resolve them).
- **Open Library / Google Books (ISBN)** — books/monographs, which none of the six DOI-centric providers handle (essential for a humanities/historian library).
- **DataCite (+ Zenodo)** — datasets, software, many preprints (the non-Crossref-DOI half), pairing with the CFF/CodeMeta/DataCite citation standards.

### Tier-2 (situational / domain-conditional)
CORE (aggregated OA **full text**, not just a link) · bioRxiv/medRxiv (biomed preprints) · NASA ADS / INSPIRE-HEP (astro/physics) · OpenCitations / OpenAIRE / Crossref Event Data (citation/funding breadth) · Wikidata (entity linking / QIDs).

### Tier-3 (skip: cost/license, open equivalents exist)
Scopus, Web of Science, Dimensions, Altmetric, PlumX.

## 3. The pipeline
**Stage A — Resolve identifiers (type-branch).** From whatever the item has (DOI / arXiv / PMID / ISBN / title+authors / PDF / URL), build the canonical key set. **DOI is the master key**; backfill via cross-walk (NCBI idconv for PMID↔DOI↔PMCID, arXiv↔published-DOI, Crossref/OpenAlex title search → candidate DOI with a confidence threshold). No confident key → **flag for human confirmation, do not guess.** Branch:
- **DOI-keyed scholarly** → Crossref required; OpenAlex + Unpaywall required for alpha.13 unless explicitly configured as degraded; Semantic Scholar, PubMed, arXiv, Retraction Watch/PubPeer, ORCID/ROR are conditional or async.
- **ISBN-keyed book** → one successful Open Library or Google Books result is required.
- **URL / primary source** (a historian's web/archival materials, no identifier) → URL metadata extraction is required; Wayback availability is optional unless the capture policy requires archiving.

**Stage B — Fan-out fetch (concurrent, rate-governed).** Each provider has a thin **adapter** that fetches by key, caches the **raw payload as a content-addressed blob** (in `raw/`, hash in the DB), and normalizes it into a common `ProviderRecord` (CSL-ish fields + external IDs + relations + `raw_hash`). A shared **token-bucket governor** enforces per-provider rate limits; per-provider timeout + bounded retry; **graceful degradation** — required provider failure leaves the item unchecked and queued for retry/human attention, optional provider failure does not block promotion.

**Stage C — Merge + reconcile (the heart).** A shared reconciler merges the `ProviderRecord`s by a **field-authority table** into the canonical `sources.csl_json`, recording **per-field provenance**:

| Field | Authority order |
|---|---|
| Title, authors(names), dates, container, license, references | Crossref → OpenAlex → PubMed → S2 |
| Author identity (ORCID/IDs), venue ISSN, institution ROR, topics, citation edges | OpenAlex → S2 |
| Abstract | PubMed → Crossref → S2 → OpenAlex; TLDR from S2 |
| MeSH / biomedical terms | PubMed |
| OA PDF location | Unpaywall → OpenAlex/S2 OA locations |
| Embedding | S2 (SPECTER) |
| Preprint version/category | arXiv |

Conflicts between high-authority providers on a critical field (title mismatch, divergent DOI) → a `check-fired` flag and `metadata_status = partial/unverified`.

**Stage D — Validate + check.** Schema (Pydantic adapter), identifier integrity, cross-provider consistency, and a **retraction/erratum check**. The MVP gate uses Crossref relation/update metadata; PubMed pub-types, Retraction Watch, and PubPeer harden the signal when available. Derive `metadata_status` from provider agreement/coverage, derive `enrichment_status` from run state, and reserve `check_status` for the alpha.12 read barrier.

**Stage E — Commit + project.** One worker transaction writes `sources`, the populated `entities`/`catalog_links`, `external_ids`, and `field_provenance`; promotes `check_status` only if the gate passes; materializes `references.bib` from the same canonical row; updates the index. **Async backfill** (embeddings, full citation graph, topics, volatile counts) runs as a follow-up non-blocking operation so the source is usable as soon as the *required* providers land.

Human-confirmation cases produce an attention item that links the source row,
candidate identifiers, provider evidence, and the exact blocked check. They do
not silently become checked.

## 4. Identity reconciliation (solves the alpha.12 `sources ↔ authors` gap)
Authors are linked by **ORCID / OpenAlex Author ID / S2 author ID** (not name strings), venues by **ISSN-L**, institutions by **ROR**. So `catalog_links` `authored_by` edges carry external IDs → **global, stable disambiguation** instead of fragile local string matching. `csl_json` keeps the literal names; the entity graph carries the resolved identities. On re-enrichment, prior human merge/disambiguation decisions are preserved (FK-by-identity match), per the alpha.12 author-reconcile gate.

## 5. Data-model additions (on the alpha.12 §3 ER model)
- **`external_ids`**: generic ID rows, not one schema column per provider:
  `owner_type`, `owner_id`, `namespace`, `value`, `source_provider`, `confidence`,
  `verified_at`. Namespaces cover DOI, OpenAlex, S2, PMID, PMCID, arXiv, ISBN,
  DataCite, ORCID, ROR, ISSN-L, and future IDs without another migration.
- **`enrichment_runs`**: one run per source/enrichment request with
  `enrichment_status` (`pending`, `enriched`, `partial`, `failed`,
  `needs_human`, `contested`), `required_provider_policy`, `started_at`,
  `finished_at`, and the operation request/journal IDs.
- **`provider_payloads`**: per provider request —
  `run_id`, `provider`, `request_key`, `request_params_hash`, `status`,
  `fetched_at`, `raw_hash` (→ blob), `normalized_json`, `error`, `latency_ms`,
  `retry_count`, `ttl_until`, and optional provider version/rate-limit metadata.
- **`field_provenance`**: per canonical field —
  `source_id`, `field_path`, `value_hash`, `winning_provider`,
  `evidence_payload_id`, `alternatives_json`, `confidence`, and
  `conflict_status`.
- Naturally populated: author/venue/institution/topic `entities`; `cites` edges in `catalog_links` (among *your* sources only); `best_oa_location` on the source.

**No `catalog/` markdown folder.** The catalog is **SQLite-SSOT** (alpha.12), so enrichment writes only the DB tables (`sources` / `entities` / `catalog_links` / `external_ids`) + materializes `references.bib`; there is no per-source markdown to write or keep in sync. Consequences:
- **The `digest` is the source's sole presence in the knowledge graph** (1:1, in `knowledge/digests/`) — Obsidian shows one node per ingested source: the machine summary.
- **Link model:** keep-set Concepts (notes / hubs / digests) reference a source by **`source_id`** (engine-resolved, surfaced through the digest), **not** a `[[wikilink]]` to a markdown source file.
- **PI surface:** the PI views/edits raw catalog records through a **plugin over the DB**; the portable takeout stays `references.bib`.

This refines alpha.12 §3 — the `catalog/` markdown folder is removed; `references.bib`, the digest, and the plugin are the catalog's surfaces.

## 6. Integrity & provenance
- **Provenance:** each provider call → an `api_call` journal event; the cached payload makes re-merge deterministic.
- **Sandbox:** the worker does the HTTP against a declared allowlist; agents never get network/code-exec. Enrichment is a worker capability, not an agent tool.
- **Retraction is never silently swallowed:** a retracted/contested source may remain a structurally valid catalog row, but it is blocked at the alpha.12 consumable read barrier (`check_status != checked`) unless explicit human resolution records why it is allowed.
- **Idempotency:** operation key is
  `enrich-source:{source_id}:{normalized_identifier_hash}`; provider payloads are
  keyed by `(source_id, provider, request_key, request_params_hash)`. Raw payload
  hashes detect changed provider data; provider version/header data is optional
  evidence, not part of the idempotency contract.
- **Concurrency:** only one active enrichment run per source/normalized key may
  promote a source. Later re-enrichment runs append evidence and update canonical
  fields atomically.

## 7. Resilience & etiquette (do not cut)
- Concurrent fan-out + **per-provider token-bucket** rate limiting; per-provider timeout + bounded retry/backoff.
- **Content-addressed cache** of raw payloads → cheap, reproducible re-runs; respect cache TTL for volatile fields (`cited_by_count`, retraction status).
- **Graceful degradation:** required providers block checked promotion; optional enrichers async-backfill; a provider outage flags-for-retry or human attention, never deletes the captured item.
- **One config** for `mailto`/email + per-provider API keys + rate limits + required-vs-optional set + OA-only-PDF toggle.

Required-provider policy:

| Source branch | Required for checked promotion | Optional / async |
|---|---|---|
| DOI scholarly | Crossref; OpenAlex + Unpaywall unless explicitly degraded | Semantic Scholar, PubMed, arXiv, Retraction Watch/PubPeer, ORCID/ROR, DataCite |
| ISBN book | Open Library or Google Books | the other book provider, Wikidata |
| URL / primary source | URL metadata extraction | Wayback availability/archive, Wikidata |

Keep this policy data-driven in one config file; do not bake provider lists into
every adapter.

## 8. Decisions (this iteration)
- **Enrichment is a worker operation, not an agent tool** (MCP-only sandbox; declared network allowlist).
- **Capture is provisional; enrichment promotes.** Alpha.13 capture persists raw/minimal unchecked state, and enrichment is the default checked/consumable gate for DOI/ISBN sources.
- **Capabilities use kind-first, directory-per-capability layout** (`operations/enrich-source/capability.md`), with generated mirrors under `_generated/`.
- **`tools/` replaces `mcp/` in the clean-slate catalog model** because MCP is a transport, not the capability kind.
- **DOI is the master key; type-branch at resolution** (DOI / ISBN / URL-primary-source).
- **Field-authority merge with per-field provenance;** high-authority conflicts → flag + `metadata_status`, never silent.
- **Author/venue/institution identity via external IDs** (ORCID/ROR/ISSN/OpenAlex/S2), not strings.
- **Retracted/contested source → flagged and blocked from consumable `checked` unless human-resolved.**
- **Provider raw payloads cached as content-addressed blobs;** enrichment idempotent + re-runnable.
- **Required providers block checked promotion; optional enrichers async-backfill.**
- **MVP starts with DOI required providers.** Implement Crossref, OpenAlex, and Unpaywall first; add conditional provider functions as branches need them. No provider class hierarchy until duplicated code makes it smaller.
- **No `catalog/` markdown folder** — catalog is SQLite-SSOT only; `references.bib` is the portable projection, the `digest` is the source's graph presence, and PI catalog view/edit is via a plugin over the DB (refines alpha.12 §3).
- **Link model:** keep-set Concepts reference sources by engine-resolved `source_id`, not markdown wikilinks.
- **`knowledge/` subfolders kept** (`digests/` / `notes/` / `hubs/` / `projects/`) — flattening considered and rejected: subfolders give free type-scoping + navigation and don't violate the by-role-not-store thesis (which was about DB-vs-files, not folder layout).

## 9. Pre-implementation gates
- **Capture boundary:** DOI/ISBN capture creates a durable unchecked source; only enrichment can promote it to consumable `checked`.
- **Capability layout:** `enrich-source` is loadable from the directory layout; generated catalog/index/Base files rebuild from manifests and are not hand-edited.
- **Schema contract:** `input.schema.json` and `output.schema.json` validate the queued request and worker result.
- **Provider policy:** required-vs-optional providers are loaded from one config and differ by DOI / ISBN / URL branch.
- **Resolution:** DOI / ISBN / URL each produce the right key set; no-confident-key → flagged, not guessed.
- **Merge determinism:** identical provider payloads → identical canonical record + provenance.
- **Conflict:** high-authority disagreement → flag + `partial/unverified`, never silent.
- **Retraction:** a retracted DOI → flagged, never consumable-`checked`.
- **Provenance:** every field traces to a provider + a cached payload; every call journaled.
- **References atomicity:** citation metadata and `references.bib` update in the same worker commit.
- **Idempotency/concurrency:** duplicate queued enrich requests collapse to one active run per source/key; re-enrich updates only changed fields, preserves history + prior merge decisions.
- **Human attention:** no-confident-key, provider conflict, required-provider outage, and retraction each create a reviewable attention item with evidence links.
- **Etiquette:** rate-limit governor honored; `mailto`/keys present; degradation on provider failure/outage.
- **Identity:** ORCID/ROR/ISSN link entities; human merges preserved across re-enrich (ties to the alpha.12 author-reconcile gate).

## 10. Alpha.13 MVP cut

Ship the DOI path first:
- Capture creates an unchecked minimal source.
- Crossref, OpenAlex, and Unpaywall fetch and cache raw payloads.
- The reconciler writes canonical CSL, external IDs, field provenance, and
  `references.bib` atomically.
- Retraction/contested checks block consumable `checked`.
- Optional providers and graph features can append evidence later.

Defer ISBN, URL-primary depth, Semantic Scholar, PubMed, arXiv, full citation
graph, embeddings, topic expansion, and full-text acquisition unless a scheduled
alpha.13 issue explicitly requires them.

## 11. Open questions
- **Migration shape** — convert all capability Concepts to directory-per-capability in one structural PR, or introduce the layout for `enrich-source` first and migrate existing operations after the loader supports both.
- **Generated path** — whether `_generated/ai-catalog.json` should replace the current root `capabilities/ai-catalog.json` in alpha.13 or remain a design target until alpha.14.
- **Title-match confidence threshold** — when to auto-accept a resolved DOI vs flag for human.
- **Citation-graph depth/cost** — how aggressively to pull `referenced_works`/citations (async + bounded).
- **Full-text acquisition** — store extracted text (CORE/OA PDF) or just the link? (ties to the ingest/digest text needs.)
- **Primary-source/web enrichment depth** — archive every URL via Wayback, or only on demand?
- **Re-enrichment cadence** — providers update (cited_by_count, retraction status); when to refresh.
- **Catalog PI surface** — how the DB-over plugin presents/edits source records, and whether the digest alone is enough source presence in the Obsidian graph.
