# Memoria — 0.1.0-alpha.12 design (architecture refinement)

Date: 2026-06-28
Status: working design for `0.1.0-alpha.12`. **Refines the storage, engine, and operation layers** of `0.1.0-alpha.11-design.md`; the knowledge model, gateless model, use-case loop, security model, evaluation, and milestones are **carried forward from alpha.11 unchanged** (referenced, not re-derived); the **integrity spine is carried forward in principle, but its write / read / rollback mechanisms are restated for the mixed markdown + SQLite storage model** (§6). This doc supersedes alpha.11 §6 (storage), §8 (stack), and §11 (operations).
Method: re-derived from requirements + best practice. **Three constraints relaxed vs alpha.11:** (1) **no multi-device sync** (single machine); (2) **ownership/portability scoped to the keep-set + bibliography export**, not the whole workspace; (3) **alpha.12 prioritizes implementation simplicity over full catalog/ops plain-text ownership**.

---

## 0. What changed from alpha.11, and why

| | alpha.11 | alpha.12 |
|---|---|---|
| Storage rule | files-SSOT everywhere + one ephemeral DB index | **the keep-test** decides SSOT per component (below) |
| The database | a passive rebuilt index over files | **SQLite SSOT for catalog + ops**; projection engine for search/derived views |
| Per-component model | one decision (files vs DB) | **two decisions** — (a) where is the SSOT; (b) what projection (ephemeral / tracked / on-demand) |
| Architecture framing | Obsidian app + worker | **integrity engine with pluggable surfaces** (Co-PI / MCP / CLI) |
| Operations | hand-rolled structured calls | **typed (pydantic-ai)** calls; the typed Concept model is the through-line |
| Sync | git-only, multi-device deferred | **no sync required** (single machine); optional Turso/libSQL backup adapter, not a primary sync design |
| Cutover | clean rebuild migration (§20) | **no migration target**; non-sandbox use starts from a fresh workspace |

The driver is a single sharp test that replaces the fuzzy "ownership feels important."

## 1. The principle: the keep-test

> **The thing you keep if Memoria dies must be markdown files in OKF, plus a portable bibliography export. Everything else is free to be SQLite.**

It's objective: not "is this valuable" but "does the user need this to keep thinking and writing outside Memoria." Only the **keep-set** must be precious markdown; bibliography leaves as BibTeX; Memoria's working state may be SQLite:

- **Keep-test pass → OKF markdown:** the knowledge graph (`note` + `hub` + `digest`) — your thinking *and* the machine wiki, both durable artifacts.
- **Config-as-code → OKF markdown:** `capabilities` — operations/skills/MCP/workflows are reviewed, signed, diffed, and **shared/imported as bundles**; plain text is the right (and safer) home even though they're Memoria-specific.
- **Bibliography takeout → tracked `references.bib`:** portable citation metadata generated from the catalog, materialized by the worker whenever bibliography-bearing catalog rows change, and drift-checked; enough for the user to take completed sources with the keep-set.
- **Relational / operational working state → SQLite SSOT:** the catalog, source/entity/link graph, operational journal, queue, flags, check status, rollback graph.
- **Pure projection → SQLite:** FTS/vector indexes and derived dashboard/query views; rebuilt from SQLite + files.
- **Blobs → a content-addressed file store:** `raw/` (PDFs etc.), gitignored; the DB holds only the hash + path.

**Invariant:** the user takeout set (`knowledge/` + `capabilities/` + materialized `references.bib`) is enough to continue the intellectual work outside Memoria for completed operations. Delete SQLite and the blobs and the notes/hubs/digests still read as complete, grounded knowledge in any markdown/OKF tool, with bibliography available as BibTeX. What is lost is Memoria's working state: detailed catalog graph, queue, flags, operational trace, check history, pending operations, and rollback machinery.

## 2. The layered architecture: an integrity engine with pluggable surfaces

```
Surfaces   │ Co-PI (Obsidian) · CLI · MCP server (BYO agent) — MCP-ready now, server deferred
           │   plugin/agent surfaces drive typed operations; PI markdown edits are the observed exception
Operations │ pydantic-ai: typed, validated outputs · model-agnostic runner
           │   the interview is an operation behavior (suspend/resume = engine state)
Engine     │ queue · write boundary · read barrier · promotion · cascade rollback · journal
           │   one uniform "checked knowledge graph" view + one write path, regardless of SSOT
Storage    │ files-SSOT (knowledge + capabilities md · references.bib export) · SQLite-SSOT (catalog + ops) · blobs (raw)
```

The **engine is the product** — the trustworthy-machine-written-knowledge guarantee. The agent/chat is commodity; don't bet on being a better agent than the one the user already has. Ship a **first-party Co-PI** (so the interview/grounding discipline is guaranteed); the operations carry **MCP-compatible typed schemas** so a power user will be able to drive Memoria from their own agent — but the **MCP server surface is post-alpha** (the engine is built MCP-ready now; alpha.12 ships Co-PI + CLI). Everything is safe because **the engine owns the write boundary** (agents emit write-requests; only the worker checks and promotes).

**The through-line that makes this cohere:** one **canonical domain model** per Concept, with **role adapters** projecting it to each layer — the MCP tool schema (surface), the pydantic-ai output type (operation), the write-request the engine validates (engine), and the SSOT shape (markdown frontmatter *or* SQLite row, storage). One model, four adapters that **version independently** — the external contracts (MCP / LLM output) evolve separately from the internal persisted shapes. The engine is the layer that hides storage heterogeneity from the operations above it and the surfaces above that.

Heterogeneous storage **makes the engine boundary load-bearing** (no surface could touch files+DB+blobs directly and stay consistent), which is exactly why engine-with-surfaces is the right shape here.

## 3. Storage: SSOT + projection, decided per component

Two independent decisions per component, and projections come in three flavors:
- **ephemeral** (gitignored, rebuilt) — query/perf;
- **tracked** (committed) — at-rest consumption without tooling;
- **on-demand** (generated at export) — portability when needed.

| Component | SSOT | Projection |
|---|---|---|
| Notes + hubs | **Files (OKF)** | → DB (ephemeral index + constraint/read model) |
| Digests (machine wiki) | **Files (OKF)** | → DB (ephemeral index) |
| Capabilities | **Files (OKF)** | → DB registry (ephemeral) + `ai-catalog.json` (tracked) |
| Catalog (records/entities/graph) | **SQLite** | → `references.bib` (required tracked materialization for bibliography-changing writes, drift-checked) + optional OKF export |
| Operational journal (`requested`…`resolved`) | **SQLite** | → optional JSONL/log export |
| Raw | **Blob files** (content-addressed) | none; hash in the DB |
| Index / derived views | *(pure projection)* | SQLite, rebuilt from files + SQLite SSOT tables |

**SQLite is intentionally an SSOT for Memoria's working state.** The split is explicit: the user-owned knowledge and capabilities are files; the relational catalog and operational machinery are SQLite. This is the alpha.12 simplicity tradeoff. SQLite gets ordinary backup/integrity handling, but it is not part of the minimum takeout promise.

**Catalog DB shape:** source identity lives in SQLite tables, not catalog JSON: `sources(source_id, type, title, resource, identifiers, citekey_aliases, csl_json, metadata_status, check_status, content_hash, raw_hash, provenance refs)`, `entities(entity_type, id, metadata, check_status)`, and `catalog_links(subject_id, predicate, object_id, status, evidence/run refs, timestamps)` for `version_of`, dedup/merge decisions, and source↔entity links. `references.bib` is generated from `sources.csl_json`; every transaction that changes BibTeX-visible source fields schedules `references.bib` as a file-owned materialization before the operation can complete. Drift checks prove the tracked file still matches the SQLite source rows.

**Cross-store reference rule:** when **any keep-set Concept** (`note`, `digest`, *or* `hub`) references a catalog source, **denormalize the survival data into the file** — its Citation carries the **CSL/DOI**, not just the live `source_id`. The catalog can die (or be left behind when only the keep-set is kept) and every keep-set citation is still meaningful. `source_id` is the live handle; the embedded CSL is the survival copy. This is what makes §1's self-sufficiency invariant hold for digests and hubs, not just notes.

**The journal splits by the test:** the *human-meaningful* provenance — citations, the anchored notes' verbatim quotes, the `evidence_set`, supersession status — is **embedded in the notes' markdown** (keep-set). The *operational* trace — `requested` / `run` / `model_call` / `derived` / `check-fired` / `checked` / `materialized` / `resolved` — lives in **SQLite journal tables** with FK/transaction semantics, plus optional JSONL/log export for inspection. `requested` makes accepted work durable; `derived` records an output candidate; `check-fired` records a checker invocation and failures/warnings; `checked` means required checks passed and is **not** a consumability state; `materialized` records that a file-owned output was committed to git and its commit hash was recorded; `resolved` records a later flag/rollback resolution. Consumability is a derived read-barrier predicate: DB-owned rows need `check_status='checked'`; file-owned outputs need `check_status='checked'` **and** `materialization_status='materialized'`. The notes carry their own grounding; SQLite carries the operational audit.

**Backup posture (kept honest):** git backs the user takeout set (`knowledge/`, `capabilities/`, materialized `references.bib`). SQLite is authoritative for Memoria working state, so a full Memoria backup needs ordinary SQLite backup/snapshot handling. Alpha.12 does **not** promise full working-state portability without SQLite; it promises the user's completed knowledge and bibliography can leave.

**Optional remote backup adapter:** Turso/libSQL may be used as a **backup/sync adapter for the SQLite working-state DB**, especially to preserve the rollback graph and operational audit if the local DB is lost. It is not the storage abstraction, not required for alpha.12, and not part of the ownership promise. Local SQLite remains the required baseline; Turso must be spike-gated because it adds provider auth, network behavior, restore UX, compatibility limits, and free-tier/PITR terms that must be rechecked before adoption. Turso does not back up content-addressed staging; if a restored DB contains pending file materializations whose staged payloads are gone, recovery marks them failed/non-consumable or rolls back DB-owned rows.

## 4. The database (clean-slate role)

One embedded **SQLite**, four jobs:
1. **Retrieval** — FTS5 + sqlite-vec (the hybrid index over the knowledge graph).
2. **Catalog + operations SSOT** — source/entity/link rows, operational journal, queue, flags, check status, and rollback metadata.
3. **Graph + relational queries** — Concepts/Links/journal as tables; **recursive CTEs** give transitive closure: cascade-rollback blast-radius and retraction propagation become SQL, not hand-rolled walks.
4. **Write-boundary constraints** — inside the same worker transaction, SQLite **rejects on violation**: referential integrity (the Citation's target **exists** — `checked` *or* same-operation staged), uniqueness (no duplicate `source_id`/DOI), valid enums/types, DAG well-formedness. This converts the **DB-checkable invariants from detection to prevention**.

It is the **SSOT for catalog and ops**, and the **projection engine** for search/derived views over file-owned knowledge and capabilities. Drop it and you can still read/take the knowledge bundles and `references.bib`; you cannot recover the full Memoria working state unless you restore the SQLite backup.

**Prevention vs detection, drawn honestly:** *prevent* (inside SQLite/worker validation) everything a database could — FK / uniqueness / type / DAG. *Detect* (post-promote, behind the read barrier) only what nothing can — grounding, crafted injection, semantic warrant (the irreducible residual). Files don't "lose constraints"; the worker/SQLite boundary prevents everything a DB could and detects only the semantic slice.

**Transactions: SQLite first, file promotion second.** The worker records `requested`, catalog rows, journal events, queue/flags/status, rollback edges, and expected file hashes in one SQLite transaction. File-owned outputs (`knowledge/`, `capabilities/`, and `references.bib`) are staged, checked, then promoted with a coupled git commit whose hash is recorded in SQLite by appending `materialized`. On crash, SQLite decides what to resume, roll back, or re-promote; staged files are retry inputs, not authoritative state. If staging is missing after restore, the pending output is failed/non-consumable and the DB-owned part is rolled back where possible.

**SQLite-as-SSOT for the keep-set, CRDTs, and a hosted primary DB are rejected** — the keep-set remains plain markdown; CRDT/hosted-primary storage breaks the local single-user simplicity target. A Turso/libSQL remote copy is allowed only as an optional backup/sync adapter for the SQLite working-state DB.

## 5. Operations & the runner (pydantic-ai)

An **operation** is the runtime boundary: `operation_id`, allowed tools/paths/network, runner + model + routing, prompt version, **typed I/O (a Pydantic model)**, risk class, required checks, audit events. pydantic-ai implements the typed call: structured Pydantic output + validation + bounded retries + model-agnostic routing.

- **Runner: borrow the pattern, not the framework.** The default (and likely *only*) runner is a **direct OpenAI-compatible API** call via pydantic-ai (gateway candidate Kilo, swappable; raw-SDK fallback). The **Karpathy wiki *pattern* is borrowed as a prompt + SCHEMA + workflow conventions into a pydantic-ai operation — not as a Hermes runtime**, because most of the wiki skill's value is text (conventions), not engine. Embedding **Hermes as a library** (fresh `AIAgent`, `quiet_mode`, `skip_memory`, bounded, no canonical-write tools) is a **spike-gated contingency** — taken *only if* a spike proves the compounding agent-loop is genuinely beyond pydantic-ai + the worker; otherwise its cost (heavy dependency, runtime fragility, constant state-fencing, two runner paths) outweighs the reuse. **Checks/evals never route through Hermes** (certified-model pin). All runners sit behind Memoria's runner abstraction.
- **One canonical model, adapters per role:** the domain model has adapters for LLM output, write-request validation, SSOT shape (markdown frontmatter *or* SQLite row), and the MCP tool schema — each versioned independently so external contracts (MCP / LLM) aren't coupled to internal persistence. The shared *core* is why pydantic-ai earns its place (one typed object → multiple materializations); the adapters keep the four roles from drifting into one brittle artifact.
- **generator ≠ checker** maps onto the layers: pydantic-ai validates *shape*; SQLite constraints validate *graph* inside the worker transaction before any output is exposed.
- **The interview is an operation behavior, not a surface feature:** the `ingest` operation conducts it (asks clarifying questions, won't compile a digest without them), so the first-party Co-PI gets grounded results now and a future BYO-MCP agent gets the same behavior when the MCP server ships. Its **suspend/resume + collected-answers state is engine state** (worker/journal), surface-agnostic — **never** pydantic-ai's session and **never** `pydantic-graph`.

## 6. The uniform boundary (one worker protocol, mechanisms per store)

**Write lifecycle — one worker protocol, store-specific commits.** A write moves through ordered steps; SQLite coordinates the operation, while file-owned outputs still promote through git:
1. `requested` — the accepted write-request is inserted into SQLite (`journal_events` + queue), so the queue survives a crash.
2. **stage** — compute file-owned outputs into content-addressed staging; compute DB-owned catalog/ops rows in memory.
3. **validate** — shape (Pydantic adapter) + graph (SQLite FK/UNIQUE/CHECK/DAG constraints, against committed state **+ the staged set in this unit**).
4. **check** — structural checks run on the staged output; machine file output is checked *before* it can be promoted.
5. **DB commit** — one SQLite transaction writes catalog/ops rows, journal events (`derived`, `check-fired`, `checked` as check-passed), queue/flag/status, rollback edges, and the expected hashes for any file-owned outputs. Bibliography-changing catalog writes include `references.bib` in that expected-output set. File-owned outputs remain `materialization_status: pending` and non-consumable until promoted.
6. **file promote** — the worker git-commits checked markdown/capability outputs and required `references.bib` updates from staging, then records the commit hash, appends `materialized`, and flips `materialization_status: materialized` in SQLite. `resolved` is appended only by later flag/rollback-resolution transactions.
7. **project** — update FTS/vector/dashboard views from SQLite + files.

**Durability is SQLite for Memoria working state and git for the user-owned files.** A DB commit with pending file materialization is not consumable; on crash, the worker resumes promotion from content-addressed staging or marks the operation failed and rolls back DB-owned rows. A remote DB backup may preserve pending rows without staged payloads; those rows are recovery metadata only, never consumable outputs. Machine content reaches `knowledge/` only at step 6 (`checked` + `materialized`), so the completed keep-set is checked at rest (§1); **live unchecked visibility is limited to PI edits** (re-checked on import).

**The spine is uniform at the worker boundary, not in the storage mechanism:**
- **Staging → promote:** staged file outputs → checks → git commit; DB-owned catalog/ops rows → SQLite transaction.
- **Foreign-write defense:** a worker commit is recognized by its matching SQLite materialization row plus commit trailer/hash. Any git commit without that match is foreign. If it changes only PI-editable keep-set markdown, the worker imports it as a `pi_edit` candidate, re-checks it, and backfills the journal on success; if it touches generated/worker-owned paths, machine-authored markers, `references.bib`, capabilities, or fails import checks, it is quarantined.
- **Rollback:** inverse SQLite transaction for DB-owned rows and inverse git commit for file-owned Concepts; **blast-radius is a recursive CTE over the SQLite rollback graph**, spanning markdown- and catalog-backed Concepts alike.

**The read barrier is a *machine-consumer* guarantee** — agents, retrieval, gap-analysis, and export consume the `consumable_outputs` view: DB-owned rows with `check_status='checked'`, and file-owned outputs with `check_status='checked'` plus `materialization_status='materialized'`. `checked` alone is never enough for file-owned outputs. The **human reading/editing `knowledge/` in Obsidian sees live content, including their own not-yet-reimported edits** — fine, because the human is trusted and it's their own edit (re-checked on import). So Obsidian direct-read is safe *because the human is trusted*, not because everything on disk is `checked`.

Everything else (catalog, journal) is engine-mediated. Plugin/agent surfaces never write storage directly; native PI markdown editing in Obsidian is the explicit observed exception, backfilled by the worker. Future **MCP/BYO agents never see the storage heterogeneity** — they call typed operations whose adapters hide which store backs each Concept.

## 7. Building the engine: composition over a framework

**There is no drop-in OSS framework that *is* the integrity core** — the cascade rollback (machine-auto-revert / human-flag), the read barrier, staging→promote, the prevention-vs-detection split, and the check suite over a knowledge graph are Memoria's contribution, not a library. Build the core; compose the primitives; don't reinvent them.

**Structure it as a SQLite-backed local engine, not CQRS-over-git or full event-sourcing** — the durable **working state** is SQLite (catalog + ops journal + queue/flags/check state), while the durable **user-owned artifacts** are git-tracked markdown/capability files and `references.bib`. A Python **event-sourcing library** (`eventsourcing`) drops out: SQLite transactions, ordinary tables, and a small recovery state machine are clearer for a single-user local tool. **Decision: hand-roll the worker state machine and projection refreshes over SQLite + git; no event-sourcing framework.** The load-bearing pattern is still append-only audit + checked promotion + rollback graph; the framework is not.

**Primitives to compose (do not hand-roll):**
- **SQLite + FTS5 + sqlite-vec** — the catalog/ops SSOT, constraint engine (CHECK/UNIQUE/FK), recursive-CTE graph queries (rollback blast-radius, retraction), and hybrid index. The engine itself, not a wrapper.
- **git** (`pygit2` / `dulwich`) — the content-addressed, versioned store for user-owned markdown/capability files, `references.bib`, and the small journal-head anchor file.
- **Turso/libSQL** — optional backup/sync adapter for the SQLite working-state DB; never the required primary and never the source of the keep-set.
- **Pydantic / pydantic-ai** — typed operations + I/O + validation (§5).
- **`prov`** (W3C PROV) — optional, for PROV-native derivation docs; PROV-DM-shaped JSON is small enough to hand-roll.
- **NetworkX** or SQLite recursive CTE — DAG traversal.
- **RapidFuzz** + NLI (behind a HANS gate) + retraction feeds — the check suite, composed.

**Deliberately *not* used (dead ends for this core):**
- **Workflow engines** (Temporal / Prefect / Dagster / Airflow) — the worker is one queue, not a DAG orchestrator; the interview's suspend/resume is engine state, not a workflow product. (Already in the deliberately-absent list.)
- **Verifiable-log servers** (immudb / Trillian / Sigstore-Rekor) — they solve cross-org cryptographic tamper-*proofing*. Alpha.12 uses a local append-only SQLite journal (`event_id`, `prev_hash`, `row_hash`, append-only triggers) plus a git-tracked `.memoria/audit/journal-heads.jsonl` anchor (`tx_id`, event range, head hash, git HEAD) committed at operation-completion checkpoints. That gives enough practical tamper-*evidence* for the single-user threat model; verifiable-log servers are overkill and break local-first.
- **Dolt** (git-for-data SQL) — SQLite is the simpler local relational SSOT for alpha.12; Dolt adds a versioned-SQL engine and merge semantics the single-machine design does not need.
- **Policy engines** (OPA / Rego, Cerbos) — those gate *authorization*; our gates are *knowledge-integrity* checks. Wrong shape.
- **LangChain / LlamaIndex** — heavy framework-owned state; already rejected.

**Turso/libSQL posture:** useful as an optional managed safety copy for the SQLite DB, particularly the rollback graph and audit rows. It does **not** simplify the engine boundary: SQLite+git cross-store materialization still needs `materialization_status`, recovery, staged-payload handling, journal-head anchoring, and read-barrier checks. It also adds provider dependency, credentials, network/offline behavior, restore testing, and compatibility checks, so alpha.12 treats it as an adapter spike, not a baseline dependency.

**Same rule for Hermes (§5):** borrow the **wiki *pattern*** (a prompt + workflow conventions), not the **framework**. Embed Hermes-as-library only as a spike-gated contingency if the compounding loop proves beyond pydantic-ai; never as the runtime, and never for checks.

**Takeaway:** the integrity core is a *thin composition over solid primitives* (SQLite + git + Pydantic + NetworkX), structured as a SQLite-backed local engine with checked file promotion, **plus the domain semantics no library provides** — which is exactly the part that is the moat.

## 9. What this buys, what it costs

**Buys:**
- Real **prevention** (FK/unique/type/DAG enforced in SQLite) + SQL graph queries + normal DB transactions for the relational majority.
- Simpler alpha.12 implementation: no catalog JSON SSOT, no git projector for catalog/ops, no event-sourcing framework, fewer drift surfaces.
- A **self-sufficient takeout set for completed operations** — `knowledge/` + `capabilities/` + materialized `references.bib`; your knowledge survives Memoria's death, grounded, in plain markdown with portable bibliography.
- A clean **engine boundary** that any surface (Co-PI, MCP, CLI) drives safely; the typed Concept model threads all four layers.
- Scale headroom for reads and writes (relational state lives in SQLite, not as many small JSON files).
- **The custom core is the moat** — no OSS does the trace + check + cascade-rollback + read-barrier composition; building it (over solid primitives) is the point, not a cost to avoid (§7).

**Costs (kept honest):**
- **SQLite backup matters** for full Memoria working state. The minimum takeout survives without it, but detailed catalog graph, operational trace, queue/flags, and rollback history do not.
- **Cross-store materialization recovery** — a DB transaction can commit before file promotion finishes, so `materialization_status`, non-consumable pending rows, staged-payload recovery/failure, and worker recovery must be correct.
- **Reduced ownership promise** — `references.bib` is portable but lossy relative to the catalog DB; this is accepted for alpha.12 simplicity.
- **No sync** (accepted) — single machine.
- **Digest file-count** — digests stay markdown in `knowledge/` (keep-set), so 800 sources ≈ 800+ files; git's role shrinks to versioning that folder.
- **Blobs aren't in git** — `raw/` is content-addressed and gitignored, so it needs an out-of-band copy (or Zotero re-fetch) for a complete backup; it's the one thing git doesn't cover.
- **Optional Turso adds ops surface** — remote auth, network/offline behavior, restore UX, provider terms, SQLite/libSQL compatibility checks, and pending-materialization restore behavior. It can reduce DB-loss risk, but it does not remove the need for local recovery logic.

## 10. Decisions (this iteration)

- **SQLite is SSOT for catalog + ops.** File SSOT is limited to the keep-set (`knowledge/`, `capabilities/`) plus tracked `references.bib` as bibliography takeout (§3/§4).
- **SQLite transactions are durable for working state; git commits are durable for user-owned files.** Cross-store operations use `materialization_status` and worker recovery rather than pretending SQLite+git are one atomic store (§4/§6).
- **Journal = SQLite tables, full vocabulary** `requested / run / model_call / derived / check-fired / checked / materialized / resolved`; optional JSONL/log export is inspection-only (§3/§6). `checked` means checks passed, not consumable; `materialized` records git-backed file availability; `resolved` belongs to later flag/rollback-resolution transactions, not every successful write.
- **Catalog = SQLite SSOT + required tracked `references.bib` materialization.** Zotero only re-seeds raw bibliography; `references.bib` is enough for completed-operation takeout but not a full catalog graph backup (§3).
- **Write lifecycle** = `requested → stage → validate → check → DB commit → file promote → project` — one worker protocol; machine content promotes into `knowledge/` only when `checked` and materialized; bibliography-changing catalog writes promote `references.bib` the same way (§6).
- **SQLite constraints are staged-aware** — a citation may target a `checked` row *or* a same-operation staged row; `checked`-ness is a read-barrier property, not write-validity (§4).
- **Spine is uniform at the worker boundary**; rollback = inverse SQLite transaction + inverse git commit where needed, with recursive-CTE blast radius over the SQLite rollback graph (§6).
- **Grounding** = every keep-set Concept (note/digest/hub) embeds the source's CSL/DOI survival copy, not just `source_id` (§3).
- **One canonical domain model + role adapters** (LLM / file / DB / MCP), versioned independently (§2/§5).
- **Read barrier = a machine-consumer guarantee over `consumable_outputs`**; the human sees live content in Obsidian (§6).
- **MCP-ready now; MCP server deferred** (post-alpha). alpha.12 surfaces = Co-PI + CLI (§2).
- **No event-sourcing framework** — hand-roll the SQLite-backed worker state machine and projection refreshes (§7).
- **Turso/libSQL is optional** as a backup/sync adapter for the SQLite working-state DB. Local SQLite is the required baseline; Turso is not a hosted primary, not required for alpha.12, and not part of takeout ownership (§3/§7).

## 11. Open questions

- **Self-describing-citation payload:** full CSL-JSON vs DOI+title+authors — enough to be meaningful standalone without bloating every note.
- **digest adopt semantics under files-SSOT:** a human "adopt/rewrite" of a digest is a markdown edit (PI path) → re-checked on import; confirm this stays detection (human-owned), not prevention.
- **Turso/libSQL backup adapter:** does the current installed/client stack support the needed local-first flow, restore UX, and SQLite compatibility without making Turso a required hosted primary? Recheck current pricing/PITR/embedded-replica terms before any implementation dependency.
