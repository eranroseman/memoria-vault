Date: 2026-07-01
Status: working design for `0.1.0-alpha.15`. This is a **delta** on
`0.1.0-alpha.14`, not a rewrite — but it is an **integrity/architecture delta**,
not a pure clarification. It does not rewrite the product or the alpha.14

dev-environment decision from first principles, (b) the small set of section
overrides that follow from it, (c) a sharper work enrichment design derived
from OSS comparison, (d) a native gap-analysis design derived from OSS
comparison, (e) a paper-framing/readiness layer for Project Concepts, (f) a

(§6/§11): one commit per completed operation, a fixed tracked/gitignored file
list, and `init` establishing the repo + `.gitignore`, (t) an explicit
cross-store ACID contract (§6): operation-level atomicity via 2PC + recovery,
serializable isolation via a workspace-scoped lock, and WAL + write-ahead fsync
ordering for durability, and (u) a minimal runner policy (§4/§7): two
OpenAI-compatible providers (`local`, `gateway`), a per-operation **test +

the JSON read API is the sole read path for every surface, UIs never touch raw
SQLite/files, every read projection carries its verdict, and read-API

(§1/§2): a thin control-surface UI plus conversation via the user's own agent as

per-Co-PI read scoping, and an agent API implemented as one engine + two

conversation), a plugin compatibility filter, and a clean-slate thin-client
Obsidian plugin design (transport, architecture, integrity mapping), and (x) a CopilotKit
evaluation as an alternative UX approach (Appendix H) — rejected as the approach,
its AG-UI/declarative-UI patterns kept as reference. §S6 records the alpha.15
implementation scope (in-15 vs defer): the engine + integrity core ships now,
enhancements and adapters defer.

Method: alpha.14 assumed Python (`pip install -e .`, pydantic-ai, SQLite) by
inheritance from the current implementation. Alpha.15 re-derives the language
choice from the requirements, confirms Python, and upgrades the *tooling around
it* to best practice. It also sharpens — without altering — the integrity
thesis alpha.14 already holds. The enrichment override comes from a scan of
nearby OSS systems: biblio-glutton, OpenCitations Meta, GROBID, Zotero, JabRef,
Manubot, and Citation.js. The paper-framing override comes from practical
research-writing guidance: understand the assignment/venue and audience first
(Purdue OWL), keep one central contribution (PLOS), and make the paper's gap,
evidence, limitations, and target venue explicit before drafting (PLOS/Nature).
The gap-analysis override comes from a scan of adjacent OSS systems: Local
Citation Network, citationchaser, litstudy, bibliometrix, ASReview, revtools,
PaperQA2, Open Knowledge Maps/Headstart, and pyBibX.
The alpha.14 lifecycle, read-barrier semantics, and rollback model are not
rewritten; the three new mechanisms above extend them, they do not replace them.

## 0. Decision

Everything in `0.1.0-alpha.14-design.md` §0–§15 carries forward unchanged
**except** the sections listed in §S3 below. Where this document and alpha.14
disagree, this document wins; everywhere else alpha.14 is authoritative and is
not restated here.

The added decision:

**Memoria is a Python tool, and that choice is earned from the requirements,
not inherited from the current code.** The clean-slate improvement is in the
distribution, CLI, and persistence tooling, not the language.

**Memoria helps a project become paper-shaped before it writes prose.** The
baseline adds a thin paper-framing plan and readiness checks; it does not add an
autonomous Writer workflow.

**Memoria's gap engine stays native.** OSS tools provide useful heuristics for
citation expansion, coverage matrices, screening, and grounded explanations,
but no surveyed tool owns Memoria's checked-only, claim-aware gap taxonomy.

## S1. Why Python (first principles)

The decisive question is not "which language writes a nice CLI" — every
candidate does. It is what the product is *made of* once framing is stripped
away. Read alpha.14 §7, §9, §10 as a shopping list and the core value-add lives
in one ecosystem:

- NLI / entailment for contradiction, supersession, dedup (tri-class
  SUPPORTED / REFUTED / NOTENOUGHINFO) — §9.
- Embeddings for retrieval and graph neighborhoods — §8/§10.
- Full-text extraction with page/span/offset anchors — the quote-anchor and
  citation-survival checks have no target without it — §8/§9.
- Typed structured LLM output + validation + retries (pydantic-ai) — §7.
- Scholarly metadata (CSL-JSON, DataCite, CFF/CodeMeta, BibTeX) — §4/§8.

That stack — `transformers`/`sentence-transformers`, PDF-anchor extraction,
CSL/`citeproc` tooling, pydantic-ai/instructor — is Python-native. It is the
product's differentiator, not incidental. Everything else (SQLite authority,
subprocess to qmd/git/pandoc, allowlisted HTTP, a hash-chained journal) is
plumbing any language does.

Alternatives, judged on requirements not habit:

- **Rust** — best fit for the crash-safe lifecycle, hash-chain, and
  single-binary distribution. But NLI/embeddings/PDF-anchoring still shell out
  to Python, so you ship and debug two runtimes and lose pydantic-ai's
  structured-output ergonomics. The single-binary win is real but `uv tool
  install` (§S2) closes most of the gap.
- **Go** — excellent CLI and distribution, weakest ML/scholarly ecosystem of
  the four, same Python-subprocess tax, worse structured output. No.
- **TypeScript/Node** — tempting because qmd already drags Node into the
  runtime, so it could unify. But the NLI/entailment/PDF-anchor/scholarly-
  metadata layer is not there; you would still subprocess to Python. Unifying
  on the weaker ML side is the wrong direction.

Python's classic weaknesses are irrelevant here: single-user, one write-lock
worker at a time (alpha.14 §6), no throughput hot path — the concurrency/GIL
argument never bites. Its one real cost is distribution plus no compiled
integrity primitives, and neither outweighs the one-directional ecosystem
gravity: every non-Python choice ends up embedding Python anyway.

Language baseline: **Python 3.12+**.

## S2. Clean-slate stack

Best-practice choices, each stopped at the first rung that holds.

| Concern | Choice | Reason |
| --- | --- | --- |
| Runtime + distribution | **`uv`**; user install is `uv tool install memoria` | Reproducible lockfile; single-command install answers the "no single binary" objection without leaving Python. |
| Schemas | **Pydantic v2** | Already the boundary for frontmatter, request envelope, and structured LLM output — one validator everywhere. |
| LLM runner | **pydantic-ai**, OpenAI-compatible endpoint | As alpha.14 §7; typed calls, retries, no framework-owned state. |
| CLI | **Typer** | ~60 resource/action commands (alpha.14 §5) with uniform `--json` and stable exit codes justify it over stdlib `argparse`. |
| Database | **stdlib `sqlite3`, no ORM**; WAL mode; FK/CHECK/UNIQUE in schema DDL | The integrity schema is hand-designed (alpha.14 §6); an ORM hides the constraints that *are* the product and buys nothing at single-user scale. |
| Migrations | **plain `.sql` files gated by `PRAGMA user_version`** | Single-user, no live-migration problem; no Alembic. |
| NLI / embeddings | pinned local-capable models run **out-of-process**, separate from the generation runner | Already mandated by alpha.14 §7/§9. |
| qmd / git / pandoc | stdlib **`subprocess`** with absolute resolved paths | Already required by alpha.14 §10/§11. |

Rejected additions (ponytail): ORM (SQLAlchemy/SQLModel), Alembic, a settings
framework beyond Pydantic + `providers.yaml`, and any dependency added for what
a few lines of stdlib do. Add only against a concrete failing gate.

## S3. Section overrides on alpha.14

These are the only alpha.14 sections this document changes.

### §2/§9 Universal quarantine-and-verify invariant

Make explicit and binding what alpha.14 §2/§9 and §S5 already imply: no changed
knowledge/capability content is machine-consumable until a verification pass
dispositions it. This unifies the pieces (workspace scan, read barrier,
act/ask/drop) into one invariant and closes the status-vs-hash gap.

Invariant:

> Every mutation to `knowledge/` or `capabilities/`, regardless of origin
> (human editor, machine operation, external agent, scheduled job), is
> non-consumable by machines until a verification pass dispositions it. The
> disposition vocabulary is exactly {act, ask, drop}. No path grants
> `check_status = checked` without that pass.

Two layers uphold it, with an asymmetry:

- **`workspace scan` is the sole verifier and promoter.** It imports a changed
  file as a first-class request, classifies it (PI-edit vs foreign-write),
  journals the `observed_external_edit`, sets derivation inputs so cascade
  rollback still reaches origins, runs the check suite, and dispositions the
  change **act/ask/drop**. Promotion back to `checked` happens **only here**.
- **The read barrier is a fail-closed, demote-only guard.** At machine
  consumption, the barrier verifies the on-disk hash equals the recorded
  `checked` hash. A mismatch **refuses to serve, demotes the row to
  `unchecked`, and enqueues `workspace scan`** — it never classifies, promotes,
  or runs checks. It can only ever take `checked` away, never grant it. A cheap
  dirty pre-check (mtime + size) gates the SHA so hot reads of unchanged files
  pay near-zero.

Asymmetry, stated plainly: **promotion to `checked` is always `workspace scan`'s
job; demotion can happen at either point; consumption never serves a file whose
hash does not match the recorded `checked` hash.** This makes quarantine
effectively immediate-on-consumption without a daemon: the scheduled/manual scan
does the real verification, and the barrier guarantees nothing unverified is
consumed in the window before it runs.

Interim scope (holds through beta.1): detection is scan-triggered plus the
consumption guard — there is no filesystem watcher. **beta.2 must add real-time
change detection** (an event/watch-triggered verification pass that fires on
write instead of waiting for scan or consumption). The consumption guard is not
retired by that work; it remains the fail-closed correctness backstop even once
real-time detection exists. Until beta.2, the guard is what makes the invariant
hold, not a watcher.

### §3 Storage And Authority — addition: allocation rule and recovery classes

Make the DB/file/blob split a derivable rule rather than a per-datum table.
Three questions decide where any datum lives, and jointly fix its authority and
its recovery source:

- **Q1 — fit.** Relational / graph / dedup / transactional / high-cardinality
  → DB. Human-readable / diffable / editable-standalone → file. Large opaque
  bytes → content-addressed blob.
- **Q2 — human access.** Authority follows **authoring**, not keeping. Does the
  human **author** it (→ file is authority)? A file the human **keeps but does not
  author** — a generated export like `references.bib` — is **DB-authoritative +
  file-recoverable**, not file-authoritative; direct edits to it are quarantined
  (§2). A **view-only** datum is DB-authoritative, and any file copy is a
  generated projection, never trusted on read.
- **Q3 — recovery if the DB is lost.** Three-valued, not yes/no: recoverable
  **from files**, recoverable **by replay from blobs**, or **accepted loss**
  (working/audit state born in the DB).

Governing invariant (this *is* the keep-test, re-derived):

> **No human knowledge is ever in the accepted-loss class.** Everything a human
> authored or keeps is file-recoverable; only machine working/audit state may be
> unrecoverable.

Allocation rule:

```text
Q2 = human authors it?  ── yes ──▶ FILE is authority; add a DB projection iff Q1 wants query.
        │ no  (generated-but-kept, e.g. references.bib → DB authority + file recovery;
        │      view-only → DB authority, file copy is an ignored-on-read projection)
        ▼
Q1 = opaque bytes? ── yes ──▶ content-addressed BLOB (itself a recovery source).
        │ no
        ▼
Q3 = derivable (file or blob-replay)? ── yes ──▶ DB rebuildable CACHE
                                        │          (incl. provider-derived records: blob-replay).
                                        │          Human overrides on records are file-recoverable
                                        │          via the git-tracked override log, not this class.
                                        └─ no  ──▶ DB-only IRREDUCIBLE state
                                                   (journal/requests/checks/verdicts only);
                                                   the only accepted-loss class, and it holds no human knowledge.
```

Authority + recovery matrix. **Authority (trusted on read)** and **recovery
source (survives DB loss)** are *distinct* columns — a datum can be
file-recoverable without being file-authoritative. The split-authority rule is
**file owns meaning, DB owns verdicts**.

| Datum | Authority (trusted on read) | Recovery source |
| --- | --- | --- |
| Concept bodies, steering, vocabulary, authored facets/links, compact `citations[]` | file | file |
| note claims/anchors | file (DB caches for checks) | file re-parse |
| **human `work update` overrides/dispositions** (corrected ids/facets, retraction disposition, classification) | **DB** | **file — a git-tracked override log** |
| `references.bib` | **DB (SQLite Works)** — generated projection; direct edits quarantined (§2) | file (portable keep-set export; also rebuildable) |
| `check_status` / `materialization` / derived signals (`confidence`/`contested`/`contradictions[]`) | **DB** | **accepted loss → fail closed to `unchecked`** |
| Works/Authors/graph, field provenance, **provider-derived** merge decisions | **DB** | **replay from blobs** |
| requests / queue / checks / journal / materializations / rollback edges | **DB** | accepted loss (`.memoria/journal-head` in git detects the gap) |
| provider payloads, acquired full text | blob | it *is* the recovery source |

Frontmatter `check_status` supersedes the alpha.14 contract. `check_status`
(and `materialization`/derived signals) is **DB-authoritative**. Any copy in
Concept frontmatter is a **generated projection, never trusted on read, and
ignored by `workspace rebuild`** — a forged `check_status: checked` in a file
confers nothing (§S5). This explicitly overrides alpha.14 §4, where
`check_status` reads as a plain frontmatter field.

Human overrides are file-recoverable, not accepted-loss. `work update` records
(§8) are human judgments — **not** derivable from provider blobs — so they cannot
sit in the provider-derived (blob-replay) class without violating the governing
invariant. They are split from provider-derived records and materialized to a
**git-tracked override log at `.memoria/overrides.jsonl`** (append-only; written
and staged in the `work update` operation's commit, §6/§11), so DB-loss recovery
is: replay blobs → provider-derived records, **then apply the tracked overrides**.
Provider-derived merge decisions replay from blobs; human dispositions replay from
the override log.

Two recovery properties this makes binding:

- **DB loss fails closed to `unchecked`.** Because verdicts are DB-authoritative
  and accepted-loss, a workspace rebuilt from files alone starts **fully
  unchecked** — you keep the knowledge, you lose the claim it was verified, and
  you never fail open to false-`checked`. A human editing a file changes
  *meaning* (→ re-quarantine + re-check per the §2/§9 invariant); it can never
  change a *verdict*.
- **The records graph is blob-replayable, not accepted-loss.** The §8 gate
  "deterministic replay from blobs reconstructs the same normalized records" is
  precisely the Q3 guarantee that keeps Works/Authors out of the accepted-loss
  class. Without deterministic replay they would be unrecoverable, which would be
  wrong — they are derivable.

Physical consequence for the schema: the **rebuildable concept mirror**
(projected content: title/type/facets/links/claims) is separated from the
**verdict state** (`check_status`/materialization) into distinct tables, because
they sit in different recovery classes. `workspace rebuild` may truncate and
repopulate the mirror from files; it must never touch the verdict/journal/records
state. Full DDL is implementation; the binding decisions are this rule, the
authority/recovery matrix, and the fail-closed-to-`unchecked` property.

### §4 Project Concepts — addition: outcome frame

Every Project Concept may carry an `outcome_frame` block. It is the general case
of paper framing: the durable "what does done look like, and what's next" that
Smart Loop keeps in an ad-hoc outcome note, made a typed field on the project
Concept Memoria already owns. It is planning state, not authority or prose.

```yaml
outcome_frame:
  definition_of_done: ""
  success_criteria: []
  next_actions: []
```

- `definition_of_done` and `success_criteria` are the readiness target for any
  project, paper or not.
- `next_actions` is advisory and does not become a second source of truth: it is
  reconciled with `attention worklist` and the `status --json`
  next-recommended-action, never authoritative over them.
- `paper_plan` (below) is the paper-shaped specialization of `outcome_frame` for
  projects targeting a paper; a project may carry both. `project gaps`,
  `project ask`, and `status` read `outcome_frame` the same way they read
  `paper_plan`.

This adds no engine: the project Concept, request lifecycle, and read barrier
already exist. It only names two fields so "definition of done" and "next
actions" are first-class instead of living in prose.

### §4 Project Concepts — addition: paper framing

A Project Concept may carry a `paper_plan` block. This is a planning/readiness
surface, not generated prose and not a new authority. It records the paper's
compass so `project gaps`, `project ask`, argument-health operations, and
`project export` can reason about whether the project is ready to become a
paper.

```yaml
paper_plan:
  target_venue: ""
  audience: ""
  article_type: research_article
  research_question: ""
  central_contribution: ""
  gap_statement: ""
  section_order: [introduction, methods, results, discussion]
  claim_evidence_map: []
  figure_plan: []
  limitations: []
  status: framing
```

`status` is one of `framing`, `outlined`, `evidence-mapped`, or `export-ready`.
`claim_evidence_map` links project claims to checked Works, notes, digests,
figures, or missing-evidence markers. `figure_plan` records the conclusion each
figure/table must communicate, not just a filename. `limitations` is required
before export-ready status because limitations are part of the argument, not
submission garnish.

Schema contract (both blocks):

- **Optional.** `outcome_frame` and `paper_plan` are optional blocks on the
  `project` Concept. A project without them is valid and unchanged; consumers
  treat absence as "no frame."
- **Additive, no migration.** Adding these fields requires no migration:
  existing project Concepts simply have no block, and the `PRAGMA user_version`
  schema is unaffected (these are Markdown frontmatter, not new SQLite columns
  beyond the projected read model).
- **Enums.** `paper_plan.status ∈ {framing, outlined, evidence-mapped,
  export-ready}`; `article_type` is an open controlled term (vocabulary, §4).
  `outcome_frame` carries no enum. Unknown enum values fail validation like any
  facet.
- **Ownership.** Canonical ownership is the `project` Concept schema (Pydantic +
  frontmatter, §S2); the SQLite read model projects them, it does not own them.
- **Consumers/tests to update:** `project gaps`, `project ask`, `status`, and
  `project export --ready-only` — no other surface. There are **no Base or
  dashboard artifacts to update**: UI is adapter-deferred (§0); glance is
  `status --json` + `knowledge/_views/`.

### §4/§9 Graph edges — addition: deterministic edge extraction

On Concept materialization, Memoria deterministically extracts typed links from
the Concept body/frontmatter into graph edges with **zero LLM calls** (the
GBrain borrow, Appendix D; reshaped to the read barrier). This is a cheap,
model-free, injection-safe graph-density win — it extracts only what is literally
written — and it is additive to, not a replacement for, explicit `note link`
curation. It is strictly scoped so it never mints a false argument edge:

- **Explicit-typed links only auto-mint.** A link that declares its type in
  syntax (e.g. `[[supports::note-x]]`) extracts a typed edge candidate. A bare
  `[[wikilink]]` extracts at most a weak untyped `mentions`/`related` candidate,
  never a `supports`/`contradicts`/`extends` edge — a bare link usually means
  "see also," and auto-typing it would corrupt the argument graph.
- **Extracted edges are unchecked candidates, never checked graph.** They emit
  attention/worklist rows, not checked edges. §S5 stays intact: extracted ≠
  trusted.
- **act/ask/drop routing bounds the noise, on a fixed allowlist.** Only edge
  types on the **act-promotable allowlist** — `{mentions, related, cites,
  version-of}` — may be auto-promoted by `act`, and only when the target
  resolves and the type is explicit. Everything else routes to `ask`; bare-mention
  noise below the allowlist routes to `drop` (logged, not shown).
- **Argument-class edges are excluded from `act` entirely.** `supports` /
  `contradicts` / `extends` feed argument health (§4) and are **never
  act-promotable**, regardless of resolvability or explicit typing: an
  extracted `[[supports::x]]` becomes an unchecked candidate routed to `ask` /
  `note link`, never an act-promoted checked edge. This removes the collision
  between "trivially-valid typed edge → act" and "argument edges never checked
  from extraction": the allowlist is the single source of truth for what `act`
  may promote, and it excludes the argument classes by construction.
- **Reconciliation is add-only with provenance.** Re-extraction on edit adds
  current-body edges; removal of a link retracts its extracted candidate only
  when the edge carries extraction provenance, so hand-curated `note link` edges
  are never clobbered by a re-chunk (the GBrain reindex-wipes-tags failure class,
  designed out).
- **Resolution is explicit.** Bare-basename cross-folder resolution is off by
  default; an unresolved link target yields no edge, not a guessed one.

Value is contingent on keep-set link density; where notes are prose-heavy and
link-sparse, extraction is a no-op rather than a cost. It earns retrieval/
navigation density, not argument-graph authority.

### §5 CLI Surface — addition: project frame-paper

Add one operation alias:

```text
memoria project frame-paper --workspace <path> <project-id> --json
```

It asks for and records only the paper-shaping fields that must exist before
drafting: target venue/assignment, audience, research question, central
contribution, gap statement, claim-evidence map, figure plan, and limitations.
It routes through the normal request lifecycle and writes the `paper_plan`
block; it does not draft sections.

Also add one eval subcommand for model selection (§4/§7):

```text
memoria eval select-models --workspace <path> [--operation <id>] [--mode test|production] --json
```

It runs each operation's bar against cost-ranked candidates and pins the
cheapest that passes into `runner.test`/`runner.production`, emitting a selection
record. It is human-triggered, not an autonomous loop.

Existing operations consume the plan:

- `project gaps` reports paper gaps: missing research question, vague central
  contribution, missing gap statement, claims without checked evidence, figures
  without a stated conclusion, unresolved contradictions/retractions, and absent
  limitations.
- `project ask` answers "what should I write next?" from `paper_plan`, checked
  graph coverage, and argument health.
- `analyze-project-argument` maps Toulmin claims into `claim_evidence_map`.
- `project export --ready-only` refuses export unless the `paper_plan` is
  `export-ready` and all required readiness checks pass.

### §2/§5 Engine read-API completeness (UI as thin client)

The engine boundary already routes all *writes* through the request envelope
(§2). Make the symmetric rule for *reads* explicit, because it is the gating
work for any UI and the guardrail against re-coupling: **the engine's JSON read
API is the sole read path for every surface — CLI, TUI, browser, Obsidian
plugin, MCP — over both stores.** A UI is a thin client of that API; the engine
stays fat.

- **No surface reads behind the engine.** No adapter opens `memoria.sqlite` or
  reads keep-set files outside the engine's read model. Raw-DB access is a side
  channel around the boundary and re-creates the coupling the CLI-first pivot
  removed (a UI bound to the internal schema fights it as the schema evolves).
  DB-authoritative data reaches the UI only through read commands.
- **The `--json` read surface is the UI's API.** The CLI read commands —
  `status`, `request list/show`, `attention list/show`/`worklist`,
  `journal list/show`, work/records-graph queries, `project gaps`, `project
  trace` — are the read API. They cover the DB-only data no markdown editor can
  show (inbox, records graph, provenance, gaps, audit, queue/verdicts) and the
  file layer alike.
- **Read-API completeness is an explicit engine requirement.** If a UI needs
  data the read commands do not expose, the fix is to **add a read command /
  `--json` projection to the engine**, never a raw-DB backdoor. "Does the read
  API expose everything a UI needs?" is tracked as a completeness checklist over
  the data classes a UI surfaces (inbox, graph, gaps, provenance, journal,
  requests, records, project state).
- **Read barrier at the presentation layer.** Every read projection carries the
  item's `check_status`; a client must render the verdict — `unchecked` /
  `quarantined` visibly distinct from `checked` — and may show unchecked data for
  triage but must **never present it as verified** (§S5). The verdict is part of
  every read payload, not something a UI infers.
- **Versioned contract, encapsulated schema.** `--json` fields and exit codes are
  a stable, versioned contract so UIs do not break on internal change; SQLite
  stays free to evolve behind it.

Sequencing consequence: read-API completeness is the **first investment for any
UI** — build and validate it (a small standalone read surface is the cheapest
test) before a daily UI, so the daily UI (e.g. an Obsidian plugin) is a clean
client of a proven API rather than the place the API's gaps are discovered.
Writes from any UI still go through the request envelope; reads through this API;
the adapter owns no state and touches neither store behind the engine's back.

### §1/§2 Co-PI agent contract + UX strategy

Resolve "CLI-only isn't good UX" without re-coupling to a framework. Two moves:
a thin **control-surface UI** for the visual/deterministic work (inbox, records
graph, gaps, curation, triggering operations, browsing), and **conversation
delegated to the user's own AI agent** as a Co-PI. Memoria builds neither a
chatbot nor a heavy app.

Co-PI, reused from the alpha.14 concept and generalized:

> A **Co-PI** is any delegated agent — Memoria's own worker *or* the user's
> bring-your-own agent (Claude Code, Cursor, ChatGPT/MCP) — that **reads and
> writes only through Memoria**, never touching SQLite or files directly. The
> user's agent is one Co-PI instance; it is not special-cased.

The existing invariants already govern it — no new trust model:

- **Universally quarantined.** Everything a Co-PI writes lands `unchecked` and
  passes the same checks (§2/§9, §S5 equal-suspects). An injection-compromised
  Co-PI **cannot inject checked knowledge** — its writes are gated exactly like a
  human edit or Memoria's worker.
- **Bounded to the API.** Reads via the verdict-tagged read API (§2/§5); writes
  via the request envelope. Memoria grants a Co-PI only that bounded surface —
  never file/terminal/exec/send. The sandbox principle (Memoria never *grants*
  dangerous tools) is intact; a BYO agent's *own* tools were never Memoria's
  grant.
- **Guarantees hold at the write boundary and returned data, not the
  conversation.** A Co-PI's synthesis is *conversation* — exploratory, not
  verified — until filed back and passed through the checks (§10 explorations-
  compound). The answer contract (§10) is *provided* as structured data (verdicts,
  edges, gaps, a Memoria skill/system-prompt) for a cooperating agent; it is not
  enforced on an external one, and it does not need to be, because nothing the
  agent says becomes knowledge without passing the checks.
- **Two LLM layers, distinct.** Memoria's runner (§7) calls the pinned providers
  directly for its own operations (digest, extraction, checks); the Co-PI is a
  separate, external conversational agent calling Memoria as a tool. Delegating
  conversation does not mean Memoria stops using LLMs for its own labor.
- **Per-Co-PI least-privilege read scope — deferred (arrives with the MCP
  transport).** In alpha.15 the Co-PI is the **CLI**, **all-vault scoped**: a
  shell agent driving `memoria … --json` reads the whole workspace, which is the
  accepted single-user-on-own-vault trust boundary. Per-Co-PI read scoping (and
  the out-of-scope-read refusal it implies) ships with the MCP transport later,
  to bound exfiltration through a scoped agent's own outbound tools (the AgentDojo
  "own tools suffice" residual). What ships in alpha.15 regardless: returning
  untrusted text as **data, never instructions** (§9), so injection does not
  amplify. Beyond that, a BYO agent's own tool set is the single-user's trust
  boundary.

Scope note: the control-surface UI and the MCP transport (with per-Co-PI read
scoping and Memoria-granted tool restriction) are post-baseline. What is an
alpha.15 engine property is the **CLI Co-PI contract** — read-API + request-
envelope + quarantine exposed to a shell caller, all-vault scoped. Scoping and
MCP-granted-tool restriction defer with the MCP transport (§S6).

Implementation — one engine API, two transports:

- **The engine API is a set of typed Python functions** (read_* returning
  verdict-tagged projections, write_* creating request envelopes). This is the
  single source; the CLI and any agent transport are thin wrappers over it, never
  parallel implementations (the GBrain "CLI + MCP from one contract" pattern,
  Appendix D).
- **Transport 0 — the CLI `--json` is already an agent API.** A shell-capable
  agent (Claude Code, Codex) drives Memoria today via `memoria … --json`; no new
  code. This is the minimal Co-PI path.
- **Transport 1 — a thin MCP server** over the same engine functions, for MCP
  clients (ChatGPT, Cursor, Claude Desktop) and for scoped/remote access. Stdio
  for local (subprocess, no auth); HTTP + token/OAuth for remote (deferred). Use
  the Python MCP SDK; do not hand-roll a protocol.
- **Small, task-oriented tool surface**, not all ~60 commands: read tools
  (checked retrieval / ask-context, `status`, `attention`, `project gaps`,
  `journal`, records-graph, work/note lookup) + write tools (`work capture`,
  `note propose/accept`, `operation run`, `attention resolve`). Every read is
  verdict-tagged; every write creates a quarantined request envelope; the MCP
  session carries the Co-PI identity + read scope, enforced before dispatch.

Concrete shape:

- **Engine function layer** (`engine/api.py`): `read_*(…, *, scope) -> <typed,
  verdict-tagged model>` and `write_*(…, *, actor) -> Request` (a *queued,
  unchecked* request — a write never returns "done," because nothing is checked
  at write time, §S5). `scope` and `actor` are supplied by the transport and
  enforced *inside* the engine before the read model is touched. The read barrier
  lives in the return type, not the caller.
- **CLI wrapper**: Typer command → engine function → `--json`. This is
  transport 0 and it already works; ship a `CLAUDE.md`/`AGENTS.md` Co-PI protocol
  (brain-first, writes are quarantined, cite only from returned edges) so a shell
  agent behaves.
- **MCP wrapper** (FastMCP, ~one file): each tool marshals args → the *same*
  engine function → `model_dump()`. Wrappers stay dumb; all logic and enforcement
  are in `engine/api` (guarded by the no-parallel-implementation gate).
- **Untrusted text is returned as data, never instructions**: acquired work text
  is wrapped in a tagged field with provenance (`{text, provenance, is_data:
  true}`), never inlined as a bare string that reads as a prompt (§9).
- **No dangerous tool is ever exposed to a Co-PI**: the tool list is exactly
  read-API + request-envelope; there is no file/shell/send tool in Memoria's
  server (a BYO agent's own tools are its own, not Memoria's grant).

Two implementation watch-outs: keep the transport wrappers pure marshalling (no
logic behind MCP that the CLI lacks), and make `scope` a first-class engine
parameter from day one — retrofitting scope enforcement after the read functions
exist is where exfiltration bounding leaks.

Build order: (0) CLI `--json` + Co-PI protocol file — works now; (1) the FastMCP
stdio server when non-shell agents or read-scoping are needed; (2) HTTP + OAuth
remote transport and richer scoping later.

Obsidian as a concrete adapter instance. If Obsidian is used as a client, the two
halves of the UX strategy map onto two plugins, both thin clients:

- **`memoria-inspector` = the control surface** — read projections + enqueue-only
  writes (request envelopes). Rewired from the old Hermes/MCP-profile control
  plane to the `memoria` engine read-API + request-envelope.
- **`agent-client` (ACP) = the Co-PI conversation** — an in-Obsidian chat pane to
  the user's own agent, which reaches Memoria **via MCP as a scoped, quarantined
  Co-PI**. Not via `obsidian-local-rest-api` (direct vault access bypasses read
  scoping + verdict-tagging); that plugin's "expose the vault" job is exactly what
  the engine boundary replaces, so it is dropped.

Plugin compatibility filter (the adapter contract, applied to Obsidian): a plugin
is admissible iff it **(a) only reads/displays** the markdown + engine outputs, or
**(b) routes every write through the engine** (CLI/MCP) or as a plain note-edit
that `workspace scan` quarantines — and **never becomes a second authority** for
state, frontmatter, checks, or commits. Consequences: view/edit/nav/style plugins
carry over; `obsidian-git` is view-only with **auto-commit off** (the engine owns
commits, §6/§11); a frontmatter formatter (obsidian-linter) is banned harder
because `check_status` is an engine-owned projection ignored on read (§S5); and an
in-Obsidian vector index (smart-connections) is not the retrieval-of-record —
checked, verdict-tagged retrieval is the engine's `ask`. The old inventory's
"required" plugins were required because Obsidian hosted the control plane; under
the engine boundary none are required, and the survivors are view/edit clients
plus the two above.

Obsidian plugin design (clean-slate, thin-client). Post-baseline (adapter), but
specified so the plugin, when built, is a faithful renderer + intent router and
never a framework. Obsidian's own best practices coincide with Memoria's
integrity rules — notably "never `innerHTML` with external content" *is* §9's
"don't amplify untrusted text."

- **Transport — `requestUrl` to a local loopback HTTP engine API.** Not
  `child_process` (desktop-only, mobile-breaking) and not a stdio MCP client in
  the renderer (MCP is the *agent's* transport). The plugin is an HTTP client of
  the same `engine/api` functions the CLI and MCP wrap; the HTTP server is the
  adapter's on-demand backend (`memoria serve --http`, loopback + token) launched
  while Obsidian is open — not a daemon (§5).

  ```text
  plugin ──requestUrl──▶ Memoria HTTP API (loopback+token) ──▶ engine/api  ◀──MCP── Co-PI agent
  ```

- **Architecture — one client, two views, nothing else.**
  - `engine/client` — the **single boundary**; every read/write goes through it
    (typed `requestUrl`, token, API-version check, engine-down handling,
    verdict-tagged returns). Nothing else touches the engine or reaches behind it.
  - Inspector `ItemView` (sidebar) — the **control surface**: inbox/attention,
    `status`, `project gaps`, records-graph, journal — read + **enqueue-only**
    writes (each action creates a request envelope).
  - Co-PI chat `ItemView` — the `agent-client`/ACP pane; the agent reaches the
    engine over **MCP** (scoped, quarantined), not via vault access.
  - a `renderVerdict()` helper + an untrusted-text render helper + a settings tab
    (base URL, token, read-scope, interval, enabled views).

- **Obsidian best practices baked in:** `registerView`/`registerEvent`/
  `addCommand` for auto-cleanup; never hold view refs (use `getLeavesOfType`);
  never `detachLeavesOfType` in `onunload`; Vault/Editor API not Node `fs`;
  `createEl`/`createSpan` not `innerHTML`; CSS classes + Obsidian variables;
  mobile-safe (no Node/Electron APIs).

- **Integrity mapping (presentation layer):** every engine item renders through
  `renderVerdict()` (checked/unchecked/quarantined visible — §S5); work excerpts
  render via `createEl`/`textContent` with a provenance label, never as
  instruction/active content (§9); the plugin **never** writes `check_status`,
  machine frontmatter, or commits (the obsidian-linter and auto-commit failure
  classes designed out); human note edits use Obsidian's editor and flow through
  `workspace scan`.

- **Skip (ponytail):** no in-plugin retrieval/vector index (retrieval-of-record
  is engine `ask`), no state store or cache-as-truth, no generative-UI framework
  (declarative rendering of engine JSON), no git, no direct file/frontmatter
  writes. Net size: `engine/client` + two `ItemView`s + a verdict helper +
  settings — a few hundred lines; everything hard stays in the engine.

The same shape serves a future TUI or web UI: all are HTTP clients of the one
engine API, differing only in rendering.

### §4/§7 Runner policy — per-operation model pin (no role layer)

How a prompt selects its model, in the minimal form the requirements actually
justify. A "prompt" is an `operation` capability file, so **model-per-prompt is
model-per-operation**: each manifest pins its own runner.

- **Two providers, both OpenAI-compatible.** `providers.yaml` holds exactly two
  connection entries — `local` and `gateway` (Kilo) — each just `{url,
  key_env}`. Both use the OpenAI-compatible client; a provider differs only in
  base URL and key. This is connection config, not an abstraction layer.

  ```yaml
  providers:
    local:   { url: http://localhost:11434/v1, key_env: null }
    gateway: { url: https://<kilo>/v1,          key_env: KILO_KEY }
  ```

- **Each operation pins two models — one for test, one for production —
  selected by run mode.**

  ```yaml
  runner:
    test:       { provider: local,   model: qwen2.5:7b,         temperature: 0 }
    production: { provider: gateway, model: <cheapest-passing>, temperature: 0 }
  ```

  A global `--mode test|production` (default `test` in CI/dev, `production`
  live) picks the branch. Both are pinned in the manifest; `provider` picks one
  of the two connections. The test model is normally `local` (free, offline,
  deterministic — the test suite runs constantly); the production model is
  whatever clears the bar most cheaply (below).

- **No semantic-role / preset layer, and no shared "checker" model.** Checks are
  heterogeneous tasks — NLI wants an NLI model (a separately-pinned comparator,
  §7), extraction-faithfulness may want a small constrained local model,
  reasoning checks may want a stronger one, and structural checks
  (quote-anchor, year-order, citekey-resolves) use no model at all. There is no
  first-principles reason checks share a model, so nothing forces them to.

- **Cheapest-passing selection (`memoria eval select-models`).** For each
  operation × mode, candidate models are ranked by cost (a per-model cost table
  in `providers.yaml`; `local` = 0). The procedure runs the operation's **bar**
  — its seeded-error slice for a check, a task-eval fixture for a generative
  op — against candidates **cheapest-first, and pins the cheapest that passes**
  into `runner.test` / `runner.production`. This is §9 baseline discipline
  applied to model choice: the smallest/cheapest model that clears the bar wins
  (and §7's "constrain the output, not the model" means that is often a small
  one). No viable candidate → the op fails selection and raises attention. The
  procedure is **explicit, human-triggered, and inspectable** (it writes a
  manifest pin + a selection record); it is not an autonomous online optimizer
  (§7 forbids self-improvement loops). Re-run it when models, prompts, bars, or
  prices change.

- **Calibration is per-`(operation, model, mode)`.** The test suite runs on the
  **test** model — a conservative floor: a weaker local model passing implies the
  stronger production model very likely passes (on-box qwen numbers are pessimistic
  floors). But the §9 seeded-error verdict for non-sandbox use must **independently
  pass against the `production` model** (§11) — passing on the test model does not
  license production. Changing either model re-triggers that mode's verdict.

- **Recorded per run.** The resolved mode/provider/model/params + prompt hash are
  written to the request/journal (§4 provenance), so which model executed which
  prompt in which mode is auditable, and the manifest is the hard ceiling — a
  prompt cannot run on a model its policy did not declare.

Add a preset/role layer later only if LLM-using operations proliferate to where a
shared model edit measurably hurts; until then it is YAGNI.

### §8 Work Enrichment — replacement of the merge design

Keep `work enrich` Memoria-native. No surveyed OSS project is a drop-in fit:
biblio-glutton is the closest identifier lookup/aggregation service, while
OpenCitations Meta is the best provenance/dedup model. Both are references, not
dependencies.

Catalog vs OpenAlex (Appendix G): Memoria borrows OpenAlex's **entity
vocabulary** (Works/Authors/Sources/Institutions/Topics/Publishers/Funders) and
its **topic hierarchy**, but OpenAlex is a **provider and taxonomy source, not
the canonical schema**. Canonical form is **CSL-JSON** (+ DataCite/CFF/CodeMeta
per `item_type`); OpenAlex's own JSON is a content-addressed provider payload
merged by field authority. The catalog adds the two things OpenAlex has no
concept of — **per-field provenance** and the **integrity/read-barrier layer** —
which is why it is not just a local OpenAlex mirror.

Pipeline:

```text
identifier(s)
  -> resolve canonical work key
  -> fetch provider payloads
  -> normalize each provider record
  -> merge by field authority
  -> detect conflicts / retractions / missing required coverage
  -> write raw payloads + canonical work + field provenance
  -> promote checked only if required gates pass
```

Core tables:

- `works`
- `external_ids`
- `provider_payloads`
- `enrichment_runs`
- `normalized_provider_records`
- `field_provenance`
- `work_relations`
- `attention_items`

Provider adapters are plain functions, not a class hierarchy:

```python
fetch_openalex(id) -> RawPayload
normalize_openalex(payload) -> ProviderRecord
```

Add an abstraction only when it deletes duplicated code.

Merge policy is a static field-authority table, never "last provider wins":

| Field | Primary | Secondary |
| --- | --- | --- |
| DOI / identifiers | Crossref / DataCite | OpenAlex, Semantic Scholar, PubMed |
| title / issued / venue | Crossref | OpenAlex, PubMed |
| OA location | Unpaywall | OpenAlex |
| citations / references | OpenAlex | Semantic Scholar, Crossref references |
| biomedical metadata / MeSH | PubMed | OpenAlex |
| authors / institutions | OpenAlex | Crossref, PubMed, ORCID/ROR |

High-authority disagreement writes the best current value, records
`field_provenance`, marks `metadata_status = unverified`, and creates an
attention item. It never silently overwrites.

For DOI scholarly works:

| Role | Providers |
| --- | --- |
| Required | Crossref, OpenAlex, Unpaywall/OA locator |
| Conditional | PubMed when PMID/PMCID or biomedical signal exists |
| Optional | Semantic Scholar, arXiv, ORCID/ROR hardening |

Required-provider failure leaves the work `unchecked`. A human may accept
degraded enrichment through `work update`, but acceptance confers no `checked`
status (§S5): it only relaxes the **provider-coverage policy gate** — a scope
threshold, not a correctness check — records the decision as a disposition, and
sets `metadata_status = unverified`. Every remaining content check (structural,
provenance, quote-anchor, retraction/contested) still gates `checked`
independently; degraded acceptance bypasses none of them. Retraction or
contested status blocks checked promotion until resolved regardless of any human
acceptance.

Raw provider responses are content-addressed blobs:

```text
<workspace>/.memoria/blobs/provider-payloads/<sha256>.json
```

SQLite stores hashes, normalized facts, provenance, and merge decisions. This
keeps enrichment replayable without making provider payload JSON the query
surface.

### §9 Contradiction / dedup / supersession — addition: layered proposer

Sharpen alpha.14 §9's comparator into an explicit two-tier proposer. The rule is
not "NLI instead of an LLM judge"; it is that **no mechanism decides — each is a
bounded, auditable proposer, and the human sets the directional link.** The
distinction that matters is discipline (abstain + adversarial gate + FP budget),
not model class.

- **Tier 1 — NLI + abstain + HANS, the high-volume filter.** A pinned,
  local-capable NLI comparator screens dedup/supersession/contradiction
  candidates with a tri-class SUPPORTED / REFUTED / NOTENOUGHINFO verdict and a
  recorded warrant sentence. NOTENOUGHINFO/abstain is first-class with a tunable
  abstain threshold and a false-positive budget. Dedup keys on canonical IDs.
  NLI may gate a candidate only after passing a **HANS-style acceptance set**
  (high lexical overlap, opposite meaning) — because NLI shares embedding
  cosine's lexical-overlap blind spot, and without the gate it is no better than
  cosine on the cases that justify it.
- **Tier 2 — LLM judge, a gated escalation proposer.** An LLM judge is
  **admissible only** on Tier-1 abstain/hard cases (the world-knowledge-mediated
  and multi-source contradictions NLI is structurally blind to), and only when
  disciplined the same way: retrieval-grounded (forced to a source span),
  temperature 0, abstain-first, self-consistency as a one-sided uncertainty
  signal, and a recorded warrant. It is **never a contradiction/dedup
  authority** — the GBrain failure (Appendix D). Its output is a candidate for
  the human, not a checked edge.
- **Baseline discipline applies to both.** Each tier must beat a declared cheap
  baseline (cosine / BM25 / whole-read) on the fixture or it does not ship (§9).
  Neither tier auto-accepts; supersession/contradiction/dedup edges become
  checked only through curation or an `act`-safe deterministic promotion.

### §6/§9 Git vs journal — history division of labor

Make explicit which history git owns and which the SQLite journal owns, so
Memoria neither reinvents file history nor expects git to do what it structurally
cannot. Contract:

> **Git owns file-tree history and the tamper-evidence root; the SQLite journal
> owns operation, derivation, and verdict history.**

Division:

- **Git (file layer).** The keep-set markdown is git-tracked; per-file
  diff/blame/restore and the Merkle commit chain are git's, not reimplemented.
  The git-tracked `.memoria/journal-head` anchor (a tracked **file**, not a DB
  table) uses git as the integrity root at each
  completed operation (as §9 already specifies).
- **Journal (DB layer).** Everything git has no model of: SQLite row mutations,
  verdicts (`check_status`), checks, requests, the derivation graph, and
  cascade-rollback blast radius. This is structured and queryable (SQL), not
  recoverable from `git log`.

Two consequences that reduce custom code:

- **The journal stores git refs for the file side, not file content.** A journal
  event's file-side before/after is the **commit SHA / blob hash**, not the
  bytes — git already holds the bytes. The journal witnesses the event, its
  metadata (operation, runner, model, prompt hash, input refs, checks, routing),
  and which commit materialized it; git holds the diff. No duplication of file
  history in the DB.
- **File rollback is a git path-restore paired with the DB inverse.** A
  cross-store rollback is the inverse SQL transaction (DB side) **plus** a git
  restore of the specific paths to the pre-materialization commit (file side).
  The file half is a git operation, not bespoke inverse-materialization code.

Why not git-as-sole-history: the records graph, verdicts, and journal are
DB-authoritative and in the accepted-loss recovery class (§3) — deliberately not
file-recoverable. Making git the sole authority would force that high-cardinality
relational state into tracked files, which the keep-test rejects. Git is the
authority for files; it cannot be the authority for state that is not files.

### §6/§11 Git workspace commit contract

Pin the commit behavior the lifecycle leaves implicit. The workspace root is a
git repo; the runtime commits to it automatically as part of materialization —
this is the user's vault repo, not the Memoria source tree, and it is not
user-driven.

- **Granularity: one commit per completed operation**, made at the file-
  materialize step (§6 step 6). That commit is what `.memoria/journal-head`
  anchors and what the journal references by SHA (§6/§9); the commit↔operation
  mapping is 1:1. The commit message is a summary (operation id + request id);
  structured audit stays in the journal, not parsed from `git log`.
- **Stage only the operation's own paths — never `git add -A`.** The commit
  stages **exactly** the operation's materialized paths plus
  `.memoria/journal-head`. It must not sweep the whole worktree: an unrelated
  dirty tracked path (a direct human edit) is a foreign write that belongs to
  `workspace scan`, not to this operation's commit. Committing it here would
  bypass the §2/§9 quarantine — the edit would land committed but never
  quarantined or checked. Unrelated dirty paths are left uncommitted, detected
  by scan, and held non-consumable by the read-barrier guard until scanned; the
  operation may proceed (staging only its own paths) or, under a strict-worktree
  policy, block and enqueue scan first.
- **Tracked:** `knowledge/`, `capabilities/`, `references.bib`,
  `.memoria/journal-head`, `.memoria/overrides.jsonl` (the human-override
  recovery log, §3), and kept projections (`knowledge/_views/`,
  `capabilities/_generated/`).
- **Gitignored:** `.memoria/memoria.sqlite` (DB-authoritative binary),
  `.memoria/blobs/` (opaque, backed up separately), `.memoria/index/qmd/`
  (rebuildable), and secrets in `config/` (§8). These are the DB /
  opaque / rebuildable layers; a full backup is git + SQLite + blobs (§3).
- **`init` establishes the repo.** `memoria init` runs `git init` on the
  workspace if it is not already a repo and writes the `.gitignore` covering the
  exclusions above. Git is a hard baseline dependency (§0), so this is not
  optional; `init --dry-run` reports the repo/`.gitignore` setup without writing.

### §6 Cross-store ACID contract

A Memoria operation spans four stores (SQLite, files, blobs, git). ACID is not
inherited from one transaction; it is engineered at the operation boundary. Make
the guarantee explicit and its currently-implicit preconditions binding.

- **Atomicity — operation-level, via two-phase commit + recovery.** DB commit and
  file materialization are separate steps; a crash between them is completed
  idempotently by `workspace recover` or left non-consumable. The read barrier
  (`checked AND materialized`) makes any half-done operation unobservable as
  complete — atomicity holds for observers by fail-closed, not by a single
  transaction.
- **Consistency — DB constraints + read barrier.** FK/CHECK/UNIQUE/DAG enforce
  DB invariants; the read barrier and consumption-hash guard prevent observing an
  inconsistent DB/file pair. Stronger than plain ACID consistency.
- **Isolation — serializable via a single workspace-scoped write lock.** One
  mutating worker holds a **workspace-wide** lock (DB + files + blobs + git) for
  the **whole operation**, not merely a SQLite lock. Writes are serialized; WAL
  permits concurrent readers.
- **Durability — WAL + write-ahead ordering.** SQLite WAL for the DB; git commit
  for files; content-addressed temp-write-then-rename for blobs. Ordering is
  binding: the **durable payload is fsynced before the DB commit** that
  references it, and the **DB commit is durable before materialization** begins.
  A hash-only pending row without a replayable payload is invalid.

Two safety details the guarantee depends on:

- **The write lock is workspace-scoped and held for the entire operation** — a
  DB-only lock would let a process mutate files outside the transaction and break
  cross-store isolation.
- **Demote-on-read is an idempotent, serialized flip.** The §2/§9 consumption
  guard's `checked → unchecked` demotion is a write on the read path; it routes
  through the worker (or a safe compare-and-set) so it never races the mutating
  worker.

### §10 Search, Gap Analysis, And Ask — replacement of gap design

Keep gap analysis Memoria-native. Do not adopt a bibliometric, screening, or
RAG tool as the gap engine; each covers one surface but not Memoria's checked
graph plus project argument model.

The operation reads only checked current inputs, except for explicit discovered
candidate rows:

```text
checked Works + provider topics
checked digests
checked notes/claims
citation/reference neighborhoods
full-text acquisition state
project thesis / paper_plan / Toulmin graph
  -> coverage matrix + argument scan
  -> gap findings + candidate actions
```

Gap kinds:

| Kind | Detection |
| --- | --- |
| `new-topic` | Project seed/topic exists, but no checked Works or notes cover it. |
| `undigested` | Works/digests are dense, notes/claims are thin. |
| `under-warranted` | Notes/claims exist, but checked Work support is thin. |
| `citation-neighborhood` | Seed Works have uncaptured high-signal cited/citing neighbors. |
| `full-text-missing` | Metadata exists, but no acquired text/digest supports analysis. |
| `argument-unsupported` | Project claim has no checked evidence. |
| `argument-uncountered` | Claim has support but no counterevidence/limitations. |
| `argument-fragile` | Evidence is retracted, contradicted, stale, or low-authority. |

Scoring is deterministic on a fixed finite scale — no hidden floats, so the
gate's "stable severities" requirement is testable:

```text
each input ∈ {0, 1, 2}           # ordinal, rule-assigned (below)
score = impact * confidence * actionability   # 0..8
severity = high if score >= 4 else medium if score >= 2 else low
```

- `impact ∈ {0,1,2}`: 0 peripheral, 1 relevant, 2 central — by project
  centrality, `paper_plan`/`outcome_frame` relevance, and citation-neighborhood
  centrality.
- `confidence ∈ {0,1,2}`: by checked source count, provider agreement, and
  topic/facet match (e.g. 0 = no checked source, 1 = single checked source,
  2 = corroborated).
- `actionability ∈ {0,1,2}`: 0 = no concrete next operation (advisory only),
  1 = a next operation exists, 2 = an `act`-safe deterministic next operation.

The `{0,1,2}` band rules and the `{2,4}` thresholds are the calibration knobs;
they calibrate on the gap fixture (as the §8 DOI thresholds do), but the scale
and formula are fixed so a given input tuple always yields the same severity.

`project gaps --json` emits findings and proposed actions; it does not write new
knowledge. Writes are generated attention/worklist rows plus optional unchecked
candidate Works.

Minimum JSON shape:

```json
{
  "project_id": "p1",
  "summary": {"high": 2, "medium": 4, "low": 3},
  "gaps": [
    {
      "kind": "under-warranted",
      "severity": "high",
      "target": "claim:central-contribution",
      "why": "3 checked notes, 0 checked works",
      "next_actions": ["capture candidate works", "mark claim missing-evidence"],
      "candidate_work_ids": []
    }
  ]
}
```

Borrowed patterns, reimplemented inside Memoria:

- Local Citation Network / citationchaser: bounded forward/backward citation
  expansion and "top cited/top citing" candidate Works.
- litstudy / bibliometrix / pyBibX: topic, co-word, citation, and temporal
  coverage matrices as advisory views.
- ASReview / revtools: PI-labeled screening can later rank candidate Works, but
  labels are not checked knowledge.
- PaperQA2: `ask` may explain a gap with cited evidence, but RAG output never
  becomes the gap authority.

### §14 Pre-implementation Gates — added gates

Write these failing tests before implementation code, alongside the alpha.14
gate list:

  wal`.

Integrity-principle gates (§S5):

- Promotion to `check_status = checked` is driven only by checks passing /
  warrants resolving; no code path sets `checked` on the basis of a human

- No baseline schema description, command help, `--json` field, or generated
  view labels `checked` as "approved" / "verified" / "trusted" (an `rg` gate

- A calibration fixture asserts the human checkpoint is measured against a
  no-checkpoint baseline (§9 seeded-error harness) and is thinned when it does
  not beat it; a checkpoint may not be introduced without this measurement.

Enrichment gates (§8):

- Identical provider payloads produce identical canonical work rows,
  `field_provenance`, and `references.bib` payload/hash.
- Each provider adapter calls the shared allowlist/rate-limit path before
  network I/O; a direct HTTP call in an adapter fails a test.
- DOI scholarly promotion requires Crossref, OpenAlex, and Unpaywall/OA locator
  unless the run records explicit degraded acceptance; degraded acceptance sets
  `metadata_status = unverified` and a fixture proves it does not bypass the
  structural/provenance/quote-anchor/retraction checks (no human-acceptance path
  sets `checked`).
- A Crossref/OpenAlex/PubMed high-authority conflict leaves the work
  `unchecked` or `unverified` and creates an attention item with payload hashes.
- A retracted DOI is never promoted to checked without an explicit human
  disposition recorded by `work update`.
- Dropping provider payload blobs and replaying from stored hashes fails loudly;
  replay from present blobs reconstructs the same normalized records.

Gap-analysis gates (§10):

- `project gaps --json` reads only checked current Works, digests, notes/claims,
  project state, and explicit unchecked candidate rows; unchecked knowledge does
  not affect severity except as a candidate action.
- A fixture covering empty, undigested, under-warranted, mature, and
  fragile-argument areas produces stable gap kinds and severities.
- Citation-neighborhood discovery writes only unchecked candidate Works or
  attention/worklist rows; no discovered Work is promoted by gap analysis.
- `project gaps` includes `why`, `next_actions`, and source identifiers for each
  finding; findings without an actionable next operation are advisories, not
  blocking gaps.
- `project ask` may summarize gaps, but the same gap state must be reproducible
  from deterministic read-model data without asking an LLM.

Engine read-API gates (§2/§5):

- No adapter/UI code path opens `memoria.sqlite` or reads keep-set files outside
  the engine read model; an `rg`/architecture assertion fails on a raw-DB or
  raw-file read from adapter code.
- Every read command emits stable `--json` and every returned item carries its
  `check_status`; a fixture asserts the verdict is present on every read
  projection (no unlabeled item).
- A read-completeness checklist covers the UI data classes (inbox, records
  graph, gaps, provenance, journal, requests, project state); a gap is closed by
  a read command, and the checklist failing blocks the "UI-ready" milestone.

Co-PI agent-API gates (§1/§2) — alpha.15 (CLI Co-PI):

- A CLI Co-PI write lands `unchecked` and passes the same checks as a
  human/worker write; a fixture with an injected Co-PI proves it cannot produce a
  `checked` artifact without passing the checks.
- The CLI holds no read/write logic outside `engine/api`; a test asserts the CLI
  is a pure wrapper (so the deferred MCP transport is later a skin, not a
  refactor).
- Untrusted work text is returned in a data-tagged field, never as bare
  instruction text (§9), on the CLI `--json` surface.

Deferred with the MCP transport (§S6) — not alpha.15 gates:

- A Co-PI session carries an identity + read scope; an out-of-scope read is
  refused (exfiltration bounding). *(Deferred; alpha.15 CLI Co-PI is all-vault.)*
- Memoria grants a Co-PI only read-API + request-envelope tools — never
  file/terminal/exec/send. *(Deferred; a Memoria-granted tool surface exists only
  once the MCP server does — a CLI shell agent runs `memoria` itself.)*
- The CLI and the MCP server dispatch to the **same** engine functions; no
  parallel implementation exists behind the MCP transport.

Runner policy gates (§4/§7):

- `providers.yaml` holds exactly the two OpenAI-compatible connections (`local`,
  `gateway`) plus a per-model cost table; each operation manifest pins **both**
  `runner.test` and `runner.production` (`{provider, model, params}`), and a
  manifest missing either branch fails loading.
- `--mode test|production` selects the branch; a fixture proves the test suite
  runs `runner.test` and the production path runs `runner.production`. The
  resolved mode/provider/model is recorded per request; the manifest is the hard
  ceiling (a prompt cannot run on an undeclared model).
- `memoria eval select-models` pins the cheapest candidate that passes an
  operation's bar: a fixture proves a cheaper failing candidate is rejected, a
  passing cheaper candidate is chosen over a costlier one, and "no candidate
  passes" raises attention rather than pinning a failing model.
- The §9 seeded-error verdict for non-sandbox use runs against
  `runner.production`; passing on `runner.test` alone does not license
  production. Changing either model re-triggers that mode's verdict, keyed to the
  per-`(operation, model, mode)` configuration, not a shared constant.

Storage allocation / recovery gates (§3):

- A workspace rebuilt from files alone (DB discarded) starts **fully
  `unchecked`**: a fixture drops the DB, runs `workspace rebuild`, and asserts no
  Concept is `checked` until re-verified — recovery fails closed, never open to
  false-`checked`. Rebuild **ignores any frontmatter `check_status`** value; a
  file forged with `check_status: checked` still rebuilds `unchecked`.
- `workspace rebuild` repopulates the concept mirror from files but does not
  fabricate verdict/journal/records rows; a fixture proves the mirror is
  rebuildable and the verdict state is not.
- Records recovery is two-layer: dropping the records tables and replaying
  present provider payload blobs reconstructs identical provider-derived records
  (shared with the §8 replay gate), **then** applying the git-tracked override
  log `.memoria/overrides.jsonl` restores human `work update`
  corrections/dispositions. A fixture with a human retraction disposition proves
  it survives DB loss via the override log, and that the log is tracked, not
  gitignored.
- No human-authored or human-decision datum is in the accepted-loss class: an
  `rg`/schema assertion proves every human-authored field and every `work update`
  override has a file recovery source (file body or override log).

Quarantine-and-verify invariant gates (§2/§9):

- A previously-`checked` file edited directly on disk (no `workspace scan` run)
  fails closed on the next machine read: the barrier refuses to serve it,
  demotes it to `unchecked`, and enqueues scan. A machine never receives the
  divergent-but-believed-checked content.
- No consumption path sets `check_status = checked`; promotion occurs only
  through the `workspace scan` → checks pipeline.
- A changed knowledge/capability file of any origin lands non-consumable until a
  verification pass dispositions it act/ask/drop; a fixture exercises human
  editor, machine operation, external-agent, and scheduled-job origins.
- The dirty pre-check (mtime + size) short-circuits the SHA on unchanged files;
  an unchanged checked file is served without rehashing.

Git/journal division gates (§6/§9):

- A journal event's file-side before/after is a git commit/blob ref, not stored
  file content; a fixture asserts the journal does not duplicate file bytes.
- A file-backed rollback restores the affected paths via git to the
  pre-materialization commit and applies the DB inverse transaction; a fixture
  proves the file half is a git restore, not custom content rewriting.
- The `.memoria/journal-head` git anchor (tracked file) is written at operation
  completion and a broken git/journal linkage fails a check (shared with the §9
  journal-chain check).
- A completed operation produces exactly one workspace commit anchoring its
  journal event; a fixture asserts the commit↔operation mapping is 1:1.
- The commit stages only the operation's materialized paths plus
  `.memoria/journal-head`; a fixture with an unrelated dirty tracked file
  (a direct human edit) proves that file is **not** swept into the operation's
  commit and is instead left for `workspace scan` (non-consumable until scanned).
  No code path uses `git add -A`.

Cross-store ACID gates (§6):

- A crash injected at **each** lifecycle boundary (post-stage, post-DB-commit,
  mid-materialize, post-materialize/pre-git-commit) recovers to a state that is
  either fully complete or non-consumable — never a consumable partial. A
  fixture exercises every boundary.
- The durable payload is fsynced before the referencing DB commit; a fixture with
  the payload absent after a DB commit fails loudly (invalid hash-only pending
  row).
- Concurrent write attempts serialize on the workspace-scoped lock; a second
  mutating worker blocks or fails rather than interleaving DB/file writes.
- A demote-on-read triggered concurrently with the worker does not corrupt state:
  the flip is idempotent and serialized.
- `memoria init` leaves a git repo with a `.gitignore` that excludes
  `memoria.sqlite`, `blobs/`, `index/qmd/`, and secrets, and tracks
  `knowledge/`, `capabilities/`, `references.bib`, `.memoria/journal-head`, and
  `.memoria/overrides.jsonl`; `init --dry-run` reports this without writing.

Contradiction/dedup proposer gates (§9):

- The NLI comparator must pass a HANS-style overlap-but-opposite acceptance set
  before it may gate dedup/supersession; failing it, NLI is treated as no better
  than cosine and does not gate.
- An LLM judge runs only on Tier-1 abstain/hard cases, is retrieval-grounded and
  abstain-first, and never writes a checked contradiction/dedup edge; its verdict
  is a human-routed candidate.
- Neither tier auto-accepts: a fixture proves no contradiction/dedup/supersession
  edge becomes checked without curation or an `act`-safe deterministic promotion.
- Each tier beats its declared cheap baseline (cosine/BM25) on the fixture or is
  disabled.

Graph edge-extraction gates (§4/§9):

- Edge extraction is deterministic: identical Concept content yields identical
  edges, with no LLM call on the extraction path.
- A bare `[[wikilink]]` never produces a `supports`/`contradicts`/`extends`
  edge; only explicit-typed link syntax does.
- `act` promotes only edge types on the fixed allowlist
  (`{mentions, related, cites, version-of}`); an argument-class edge
  (`supports`/`contradicts`/`extends`) is never act-promoted even when explicit
  and resolvable — a fixture asserts an extracted `[[supports::x]]` lands
  unchecked and routes to `ask`/`note link`.
- Extracted edges land unchecked; a checked graph edge requires either an
  `act`-promoted allowlisted edge or explicit `note link` curation. No
  extracted edge is machine-consumable before promotion.
- Removing a link from a body retracts only extraction-provenance edges; a
  hand-curated `note link` edge survives a re-chunk/reindex.
- An unresolved link target produces no edge (no guessed basename resolution
  unless explicitly enabled).

Outcome-frame and paper-framing gates (§4/§5):

- `outcome_frame` round-trips through the request lifecycle; `next_actions`
  render in `status`/`attention` without becoming a competing authority (the
  next-recommended-action is still computed from queue/attention/graph state,
  not read verbatim from `next_actions`).
- `project frame-paper` creates or updates `paper_plan` through the request
  lifecycle; direct edits imported by `workspace scan` land unchecked until the
  same schema/readiness checks pass.
- A project cannot reach `paper_plan.status = export-ready` without
  `target_venue`, `audience`, `research_question`, `central_contribution`,
  `gap_statement`, at least one checked evidence item in `claim_evidence_map`,
  explicit `limitations`, and no unresolved retraction/contradiction affecting
  mapped evidence.
- Every mapped claim references either checked evidence or an explicit
  missing-evidence marker surfaced by `project gaps`.
- Every planned figure/table has a conclusion sentence and at least one linked
  evidence or analysis source.
- `project export --ready-only` fails closed when the paper plan is absent,
  incomplete, stale against argument state, or contains unchecked evidence.

## S5. Integrity principle — no write-time correctness oracle

This restates the integrity thesis alpha.14 already holds (§1, §9) in the
sharpest form, and fixes any wording — in the baseline or in downstream docs —
that reads as "checked = approved." It changes no mechanism; it removes an
error of interpretation. Source: `_papers/REVIEW-REFUTATIONS.md`, whose
adversarial pass left the traceability/reversibility/verbatim-warrant skeleton
standing and killed every strong claim layered on top, including every
write-time correctness oracle.

The principle:

> **Correctness is not conferred at write time by any approver — human or
> machine. A human-authored note and an LLM digest are epistemically equal:
> both are fallible claims, not facts. No mechanism ensures correctness at
> creation — not the human gate, not the NLI checker, not self-consistency, not
> the structural checks. What the traceable, reversible graph buys is that
> errors become detectable, attributable, and reversible instead of permanent
> and invisible. Correctness is an emergent, revisable property discovered over
> time, never a stamp applied at creation.**

Why the human gate is not the oracle (evidence, not assertion):

- Human + machine ≯ machine. Jacobs 2021 (N=220, within-subject):
  clinician+ML 0.356 ≈ clinician-alone 0.357, both far below ML-alone 0.667;
  a wrong recommendation dragged the human *below* their own no-recommendation
  baseline (0.299 vs 0.357, p=0.0018). The reviewer did not filter the error;
  the error contaminated the reviewer. "PI approves" therefore does not
  automatically improve correctness.
- Explanations can make it worse (Jacobs 2021: incorrect rec + explanation was
  the lowest-accuracy condition; confidence scores were uncalibrated).

Why the graph is not the oracle either (the same error one level up):

- Automatic contradiction/supersession is the least reliably automatable memory
  capability (ADR-10's own framing; ConCoRD ~28% wrong belief flips).
- NLI is high-precision/low-recall, miscalibrated, and shares cosine's
  lexical-overlap blind spot (HANS <10% on the near-duplicate-but-opposite
  case).
- Self-consistency is blind to *stable* fabrication and is not a confidence
  signal at test-fixture scale.

So no single check — human or machine — is trusted alone. Safety is relocated
from *gating creation* to *surviving error*.

Design consequences (already latent in alpha.14; made binding here):

1. **The read barrier gates on checks-passing, not on human approval.**
   `check_status = checked` MUST mean "structural/graph checks passed and its
   warrants resolve" (model-free: verbatim span present, citekey/DOI resolves,
   record-type match, year order, no opposite-edge, no unresolved
   contradiction). It MUST NOT mean "a human blessed it." No baseline artifact,
   schema field, doc, or command help may describe `checked` as an approval
   state.
2. **Human and machine writes are equal suspects.** Both enter through the same
   request envelope, pass the same checks, land in the same hash-chained
   journal, and carry the same rollback edges. Provenance records *who
   authored*; *who* confers no trust — only resolved warrants and clean
   re-analysis do. A PI edit imported by `workspace scan` is `unchecked` until
   the same checks re-promote it, exactly as a machine write is (§2 already
   specifies this; it is the tell that the model treats human writes as
   suspect).
3. **Defense in depth, not a decision point.** Correctness rests on an ensemble
   of cheap, model-free checks plus continuous whole-graph re-analysis, no one
   of which is trusted alone. Each check must beat a declared cheap baseline
   (no-checks / BM25 / whole-read) or it is dropped — the §9 baseline
   discipline, applied to the checks themselves.
4. **Reversibility is the safety net, not the gate.** Being right at write time
   is not required when any error is cheaply undoable and its blast radius is a
   graph query. Cascade rollback and the tamper-evident journal are
   load-bearing; the human gate is not.
5. **The human checkpoint is calibrated or thinned.** A human is routed to
   (act/ask/drop, consequence-first) only where checkpointed items are
   *measurably* more correct than the same items un-checkpointed. A checkpoint
   that cannot beat its no-checkpoint baseline is theatre and gets thinned —
   Jacobs 2021 turned into a gate on the gate. Never per-write approval.

This also sharpens the contrast with review-queue tools (e.g.
llm-wiki-compiler): a review queue *is* the gate-as-quality-filter the corpus
refutes. Memoria's advantage is not a better gate — it is not relying on a gate
for correctness at all.

Terminology fix (binding on the baseline and downstream docs): prefer
"checks passed" / "warrants resolve" / "promoted to checked" over "approved" /
"verified" / "trusted" wherever the subject is an artifact's read-barrier
status. "Approve" survives only for genuinely irreversible outward actions, of
which the baseline has none.

minimum end-to-end product loop (capture → enrich → digest → notes → gaps →

the loop, adapters, and futures.** A correct engine can gain the deferred set
later without reopening the boundary. (Gap analysis is in because the loop ends in
re-gap and the first-integrated smoke includes `project gaps`; it is loop-minimum,

  in 15 as the loop's export step; only the paper-readiness framing defers.)

Deferred (fast-follow) — enhancements a correct engine gains later:

- LLM-judge Tier-2 contradiction proposer (§9) — its own calibration burden;
  Tier-1 must clear the seeded-error bar in 15 regardless.
- Edge-extraction act-promotion/allowlist tuning (§4/§9) — needs real-data
  attention-noise calibration you do not have pre-launch.
- Per-Co-PI read scoping (§1/§2) — enforcement work; single-user-on-own-vault is
  a weak exfil threat, and "don't-amplify / data-not-instructions" ships anyway.

Deferred (adapters/futures):

- UX surfaces: control-surface UI, Obsidian plugin, `agent-client`, TUI/web
  (§1/§2) — gated on read-API completeness; a full UI in 15 is the "surface
  before engine" mistake §S6 guards against.
- HTTP + OAuth remote transport (§1/§2).
- **beta.2 real-time change detection** (§2/§9) — bindingly deferred; the
  consumption guard covers beta.1.

Rationale: the ADR-worthy mechanisms split cleanly — quarantine-and-verify,
ACID, git/journal, storage-recovery, and NLI Tier-1 are integrity/correctness
properties 15 is *wrong* without, so they ship, plus the minimum loop (enrichment,
digest, notes, gap analysis, Markdown export, recover). Raw edge extraction moves
in because it reuses 15 machinery at near-zero risk; a minimal MCP and
`select-models` are pull-ins only when cheap. Everything else — paper/outcome
framing (expansion beyond the loop), LLM-judge escalation, edge-extraction tuning,
per-Co-PI scoping, UX adapters, remote transport, real-time detection — is product
expansion, an enhancement with its own calibration/enforcement burden, or a
surface, and rides on the correct engine later.

## Appendix D. GBrain borrow / adapt / reject

`GBrain` (TypeScript/Bun; PGLite or Postgres+pgvector; markdown git repo as
system of record; MCP-first, agent-operated) is the closest **full-system**
analogue to Memoria — a personal/'company' knowledge brain with a self-wiring
graph, a synthesis+gap-analysis answer layer, and an overnight enrichment
("dream") cycle. It ships more than alpha.14 designs. It also independently
re-derives a striking amount of Memoria's shape while making the **opposite bet
on the integrity backbone**: no read barrier, no checked/unchecked promotion, no
cascade rollback, no tamper-evident journal, no adversarial seeded-error gate.
That contrast is the whole value of the comparison.

### Feature → Memoria map

| GBrain | Memoria | Verdict |
| --- | --- | --- |
| Markdown git repo as system of record, synced into Postgres for retrieval | Markdown keep-set authority + SQLite read model | Same shape (§3); confirms keep-test posture. |
| `gbrain search` (raw hybrid retrieval) vs `gbrain think` (synthesis + citations + gap analysis) | `ask` answer contract (cite / unknowns / stale / contradictions / missing) | **Direct** — `think`'s "heads up: what the brain doesn't know" *is* the §10 answer contract. Confirms it. |
| Self-wiring graph: every write extracts entity refs → typed edges (`works_at`, `invested_in`, …) with **zero LLM calls** | `graph_links` + note typed links (`supports`/`contradicts`/`extends`) via `note link` | **Borrow the mechanism** (below); benchmarked +31.4 P@5. |
| `create_safety` hint (exists / probable / unknown) on results | §4 "orient before creating" synthesis discipline | **Borrow** — mechanizes duplicate-avoidance as a retrieval-time signal. |
| Evidence tag (why it matched) + `--explain` per-stage attribution | attributability (§9) | Borrow the transparency affordance. |
| Dream cycle: overnight dedup / citation-fix / contradiction-find / salience | scheduled maintenance (queue drain, scan, check, rebuild, retraction refresh) | Same shape (§5); GBrain's mutate directly, Memoria's run through the lifecycle. |
| Minions job queue: two-phase `pending→done` durable subagents, crash-safe | request lifecycle + `workspace recover` (DB commit → materialize) | Same idea (§6); confirms two-phase durability. |
| `content_hash` short-circuit on re-sync | content-addressed blobs | Confirms. |
| Schema packs (detect → suggest → review-candidates human gate; agent-authored) | fixed opinionated Concept types + controlled `vocabulary`/`facets` | Mostly **reject** (opinionated fixed taxonomy is a feature), but the detect→gate path could inform vocabulary intake. |
| `suspected-contradictions`: query-conditioned **LLM judge** | Layered proposer: NLI+abstain+HANS filter, LLM judge as gated escalation (§9 override) | **Adapt, don't adopt as authority** — admissible only as a HANS-gated, retrieval-grounded escalation proposer on NLI's abstain cases; the human sets the link. |
| MCP-first, "installed and operated by an AI agent" | CLI-first, human-directed (PI directs); MCP dropped to adapter | **Reject the posture** (§0) — opposite of PI-direction. |
| Multi-user company brain, OAuth read/write/admin scopes | single-user | Out of scope (alpha.14 non-goal: multi-writer). |
| `skillopt`: SKILL.md as a trainable parameter, optimizer keeps only edits that score higher | operations are checked capability files; §7 rejects self-improvement loops | **Adapt with care** — prompt A/B gated on a held-out judge is compatible *if* it stays external/inspectable (edits to checked operation files, journaled), never preference-into-weights. |

### The one strong borrow: zero-LLM typed-edge extraction

GBrain's highest-value idea, and the one most aligned with Memoria's own
"deterministic first" rule (§9): **every page write deterministically extracts
entity refs and wikilinks into typed graph edges with no LLM call.** It is cheap,
model-free, and benchmarked to carry the retrieval lift.

Memoria today grows typed links mainly through explicit `note link` curation and
provider enrichment. Adopt GBrain's auto-extraction **reshaped to the read
barrier**: extraction on materialization emits **unchecked edge candidates**
(attention/worklist rows), which `note link` / checks promote to checked graph —
never checked graph directly. This borrows the cheap deterministic wiring while
keeping §S5 intact (extracted ≠ trusted). **Adopted** as the §4/§9 override
above (deterministic edge extraction), scoped to explicit-typed links only.

### Reject — where GBrain's bet diverges from Memoria's thesis

| GBrain choice | Why Memoria rejects it |
| --- | --- |
| No read barrier / no checked-unchecked | Machines would consume unverified writes; §9 read barrier is load-bearing. |
| No cascade rollback / no tamper-evident journal | GBrain "survives error" via read-time gap-analysis **without reversibility**; Memoria keeps rollback + hash-chain as the actual safety net (§S5.4). |
| No adversarial seeded-error gate | GBrain evals measure retrieval quality/consistency; Memoria gates non-sandbox use on 0 false-clean + recall bars (§9). |
| LLM-judge as contradiction *authority* | Unreliable per REVIEW-REFUTATIONS; admissible only as a HANS-gated escalation *proposer* on NLI's abstain cases (§9 override), never as the decider. |
| Postgres/pgvector + Bun runtime | SQLite authority + keep-test (§3); Python (§S1). PGLite is the interesting road-not-taken, not a reason to switch. |
| Agent-operated, MCP-first | PI-directed, CLI-first; MCP is an optional adapter (§0). |

### Synthesis

GBrain is the strongest existence proof that Memoria's *shape* is right — markdown
system-of-record, self-wiring typed graph, synthesis-with-gap-analysis answers,
overnight enrichment, two-phase durable jobs, content-hash sync all show up
independently. It is also the sharpest illustration of §S5's dividing line:
GBrain agrees with "don't gate creation" and "surface staleness/contradictions
at read time," but stops there — it keeps **none** of the reversibility, read
barrier, or adversarial gate that make errors *survivable* rather than merely
*visible*. GBrain optimizes a usable strategic moat; Memoria optimizes defensible
scholarship. Borrow its **zero-LLM graph wiring** and **orient-before-create
signal**; reject its integrity posture wholesale. Same axis as Appendices A–C,
at full-system scale.

## Appendix E. OpenSearch borrow / adapt / reject

`OpenSearch` (Java/Lucene, Apache 2.0, Linux Foundation; distributed
search-and-analytics engine, an Elasticsearch fork) is a different *kind* of
reference from A–D. Those are **brains** with weak integrity; OpenSearch is
**infrastructure one layer below** — not a competitor to Memoria the product, but
an alternative to **qmd**, the §10 retrieval substrate. So the question is narrow:
should the search layer be OpenSearch instead of qmd? It also breaks the §S5
through-line usefully: OpenSearch is *not on the integrity axis at all*. It has no
checked/unchecked, no read barrier, no claim provenance, no derivation rollback;
its security (auth, doc/field ACLs, TLS, audit) is **access control, not epistemic
integrity**. The integrity semantics live entirely above it, which confirms qmd's
role — the search layer is never truth (§10).

### Borrow — retrieval patterns, reimplemented over qmd + SQLite

| Component | Why |
| --- | --- |
| Hybrid score fusion (BM25 + vector via a normalization / RRF processor) | Clean reference for how §10 fuses qmd keyword/semantic results with the SQLite graph read model. Borrow the method, not the impl. |
| Analyzer/tokenizer discipline (stemming, language analysis) | Confirms the keyword side of hybrid retrieval; qmd already owns it. |

### Adapt — good idea, baseline-gated, must stay local

| Component | Reshape into |
| --- | --- |
| Cross-encoder **reranking** stage (OpenSearch: Cohere/Bedrock) | An optional local rerank after hybrid retrieval — but it must beat no-rerank on the fixture (§9 baseline discipline) and stay local-capable (§7), or it does not ship. §10 does not currently rerank. |
| Neural-sparse / hybrid-dense encoding (SPLADE-style) | A candidate embedding option for efficiency, same baseline gate. |
| ML inference in ingest pipelines | Only as Memoria's lifecycle-gated enrichment (§6/§8) — never OpenSearch's auto-write-on-ingest model, which has no checks. |

### Reject — the engine, the surface, and everything server-shaped

| Component | Why |
| --- | --- |
| OpenSearch as the retrieval engine (JVM + Lucene cluster, always-on daemon, ports) | §0 drops the always-on control plane; §5 has no internal daemon; §S1/§S2 are Python + embedded SQLite + qmd subprocess. Running a search server to index a ~10K-Concept single-user vault is textbook over-build. |
| Index as the store; ISM lifecycle, snapshots, cross-cluster replication | §3 keep-test: markdown + SQLite are authority; qmd/search is a rebuildable projection, never state. Memoria's journal + cascade rollback own lifecycle, not ISM. |
| Dashboards UI, SQL/PPL, aggregations engine | Memoria answers glance/analytics via `status --json` + `knowledge/_views/`; §0 defers UI to adapters. |
| Doc/field-level security, multi-tenancy | Single-user (§0); access control is not the integrity model. Relevant only to a future multi-user tier (below). |

### The one legitimate re-entry: scale-out substrate

If Memoria ever grows the **multi-user / team "company-brain" shape** (the GBrain
direction, currently an alpha.14 non-goal), the local qmd + SQLite substrate stops
fitting — the same axis as SQLite→Postgres. At that point OpenSearch (or a
pgvector Postgres) becomes a legitimate scale-out retrieval substrate, **behind
the same read barrier, never as an integrity layer.** That is a beta+/post-
multi-user decision, explicitly not now.

Synthesis: OpenSearch is a substrate comparison, not a product one. Borrow the
hybrid **score-fusion** pattern (confirms §10), adapt an optional baseline-gated
**local rerank / sparse encoding**, and reject the **server-shaped engine and
surface** wholesale — they violate local + no-daemon + keep-test. Its real lesson
is the clean separation it embodies: retrieval is pure substrate below the read
barrier; all epistemic integrity is Memoria's, above it.

### Entity model — near-direct borrow

| OpenAlex | Memoria catalog | Note |

| Works / Authors / Sources / Institutions / Topics / Publishers / Funders | `works`, `entities(kind=…)` | The 7-entity taxonomy is deliberately identical — legibility + interop. |
| `ids` object (openalex/doi/orcid/ror/issn/mag…) | `external_ids` rows | Same idea, **normalized** into queryable rows. |
| `referenced_works` / `related_works` | `graph_links(edge_type=citation/related)` | Direct. |
| `roles` (one org as publisher+funder+institution) | shared ROR `external_id` across `entities` kinds | Memoria does it more normalized. |
| Topics hierarchy (domain→field→subfield→topic) | topic records + controlled facets | Import as taxonomy source for topic-coverage gaps (§10). |
| `concepts` (deprecated) | — | Correctly dropped. |

### Structural divergences (the point)

- **Canonical form.** OpenAlex-JSON is bespoke; Memoria's canonical is
  **CSL-JSON** (+ DataCite/CFF/CodeMeta per `item_type`). OpenAlex's response is
  a content-addressed **payload**, not the record.
- **Provenance.** OpenAlex exposes a merged value with ~no per-field lineage;
  Memoria's `field_provenance` + merge-by-authority is the core add — *answers
  plus their derivation.*
- **Integrity.** OpenAlex has `is_retracted` as a field that gates nothing;
  Memoria adds `check_status`, the read barrier, retraction-blocks-promotion, and
  rollback.
- **Normalization.** OpenAlex denormalizes for 250M-work read-scale (embedded
  authorships/topics, `abstract_inverted_index` for copyright); Memoria
  normalizes (separate `entities` + `graph_links`) for single-user scale, clean
  rollback, and provenance — and stores **real acquired full text as blobs with
  anchors**, because the §9 quote-anchor checks need actual text an inverted index
  cannot provide.

### Borrow / adapt / reject

| Verdict | Item | Why |
| --- | --- | --- |
| Borrow | 7-entity taxonomy + IDs-object pattern | Already adopted; legibility + OpenAlex interop. |
| Borrow | Topics hierarchy (domain→field→subfield→topic) | Ready-made controlled taxonomy for topic-coverage gap analysis (§10). |
| Borrow | `roles` cross-linking | Represent via shared ROR `external_id`. |
| Adapt | `counts_by_year` / `cited_by_count` / `summary_stats` | Cache as **volatile provider fields** (§15 cadence), never integrity-critical. |
| Adapt | `open_access` / `best_oa_location` | Feeds Unpaywall-role OA locating (§8), provenance-tracked, not authority. |
| Reject | OpenAlex-JSON as canonical form | CSL-JSON is canonical; OpenAlex is a payload blob. |
| Reject | `abstract_inverted_index` | Memoria acquires real full text (blobs + anchors) for quote-checks. |
| Reject | Denormalized embedded authorships/topics | Normalized graph + rollback wants separate rows. |
| Reject | Bibliometric stats as core schema | Volatile; cache-only. |
| Reject | Provenance-free / integrity-free posture | The reason Memoria isn't a local OpenAlex mirror. |

Synthesis: the catalog is **OpenAlex-shaped for legibility, provenance-and-
integrity-shaped for correctness.** OpenAlex supplies the entity vocabulary and
topic taxonomy and lives inside the catalog as a normalized provider; canonical
storage is CSL with per-field provenance under the read barrier.

## Appendix H. CopilotKit as an alternative UX approach

`CopilotKit` (TypeScript/React, MIT; "the frontend stack for agents & generative
UI") is a mature toolkit for building **agent-native apps**: an embedded copilot,
generative UI (agents render components at runtime), shared agent↔UI state,
human-in-the-loop, over the AG-UI protocol, integrating agent frameworks
(LangGraph, CrewAI, PydanticAI, …). It is the alternative to the §1/§2
control-surface + BYO-Co-PI strategy — and it is the **wrong direction for
Memoria**, for the same reasons the pivot happened.

### Why it's the wrong direction

- **It inverts the boundary.** CopilotKit is "one agent backend, every frontend"
  — the **agent drives the UI** via **shared, mutable app state** both read and
  write. That is precisely the "adapter owns state" pattern §2/§S5 forbid;
  Memoria's engine owns state/workflow/writes and the agent is a *quarantined
  Co-PI*, not the driver. Shared mutable state has no read barrier.
- **It's a framework you build *on*.** Adopting it means Memoria sits on top of a
  framework that owns the UI/agent interaction model — the exact
  Obsidian/Hermes-as-framework coupling the CLI-first pivot removed.
- **Stack mismatch.** TypeScript/React frontend + a CopilotKit runtime server
  (+ optional CopilotKit Cloud) reintroduces the web-app/always-on-server stack
  §0/§S1/§S2 avoid — the open-notebook direction (Appendix B) rejected.
- **Generative UI conflicts with §9.** Open-ended agent-generated UI (MCP
  Apps/JSON) is the opposite of §9's "agent-rendered UI, if any, is **declarative
  and graph-bound**." Only its *declarative* (A2UI) mode is even in bounds.
- **CLHF conflicts with §7.** CopilotKit's "self-learning / continuous learning
  from human feedback / in-context RL" is exactly the scalar self-improvement
  loop §7 rejects; and it's cloud-gated, against local-first.

### What is worth borrowing as reference (only if a rich UI is ever built)

| Item | As |
| --- | --- |
| **AG-UI protocol** | A reference for a thin agent↔UI streaming transport — *if* a rich generative UI is ever built. As a transport only, with the agent still a quarantined Co-PI and the UI reading the **engine**, never shared agent state. |
| **Declarative, graph-bound generative UI** (A2UI mode) | The acceptable form of any future agent-rendered UI (§9); reject the open-ended mode. |
| **Human-in-the-loop** | Maps to Memoria's native attention + act/ask/drop — already present; CopilotKit adds nothing here. |

### Verdict

CopilotKit optimizes **agent-drives-rich-web-UI**; Memoria optimizes
**engine-owns-boundary + thin adapters + BYO quarantined Co-PI.** For a
single-user local tool that already has markdown files, an engine, and a Co-PI
contract, CopilotKit is a large framework pulling in the boundary-inversion,
framework-coupling, web/server stack, and self-learning that the design
deliberately rejects. **Reject as the approach**; keep AG-UI + declarative
graph-bound UI as reference patterns for a possible future rich UI, always with
the engine as the boundary and the agent as a Co-PI. The Obsidian
Inspector + `agent-client` adapter (§1/§2) delivers the same control-surface +
conversation UX with thin clients and none of the coupling.
