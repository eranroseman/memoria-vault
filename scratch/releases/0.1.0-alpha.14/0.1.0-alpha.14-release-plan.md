## Status

Draft sandbox release plan for the alpha.14 standalone CLI + engine cut. This is a
full architecture cut, not a partial phase: Memoria becomes a standalone local CLI
tool with a SQLite engine, and Obsidian, Zotero, Hermes, MCP, installed profiles, and
lane/fleet are dropped from the baseline. The alpha.12 SQLite working-state boundary
(ADR-122) and alpha.13 DOI enrichment (ADR-123) already landed on `main`, so the
engine exists. Current `main` through `cc154cdd` / PR #1122 has the `memoria`
console script, grouped CLI surface, request-envelope dispatch for wired mutating
commands, workspace-local qmd checks/rebuild support, `doctor --repair`,
expanded `init --dry-run --json`, journal filters, `workspace check --assert
no-legacy-work-markdown`, capability-backed `operation list`, hardened network
allowlist matching, stable work IDs for Co-PI interview runs, and DB-only product
capture operation routing. Digest/interview operation evidence, inputs, and hub
members now use Work ID refs, and integrity/provenance/quote-anchor checks resolve
those refs through SQLite catalog rows plus acquired blob text. Generated
`references.bib` and catalog-index projections now render checked SQLite catalog
rows only. Lower-level capture helper entry points now route through SQLite
catalog rows plus `.memoria/blobs/source-content/<work-id>/...` payloads and no
longer create legacy Work markdown, legacy raw/content folders, derived entity
markdown, or capture-time `references.bib`. pydantic-ai is now the sole accepted
operation runner, all checked operation manifests declare `runner: pydantic-ai`,
and the generated capability index is regenerated. Capability Concepts now use
file-per-capability operation manifests and optional `adapter` Concepts under
`capabilities/adapters/`; `mcp` is no longer a capability type. The old operation
implementation port is landed: all 26 legacy `.memoria/operations/` Python
modules now live under `src/memoria_vault/runtime/subsystems/`, tests import the
runtime package path, `eval-run` has worker dispatch and a capability manifest,
and direct Hermes transport/env assumptions were removed from the copied
reconcile/eval/ingest paths. PR #1118 removed the vacated
`vault-template/.memoria/operations/` tree, moved the template pre-commit hook to
`vault-template/.githooks/`, rewrote current callers to runtime package modules,
and added `scripts/alpha14_negative_gate.py` for operation-home references. PR
#1119 removed shipped installed profiles, lane overrides, the profile tool
registry, profile rendering, live Hermes/L2 smoke scripts, and profile/Hermes
installer modes; current docs and agent guidance now describe the standalone CLI
baseline, and the negative gate covers the deleted profile/lane/tool-registry
surfaces. PR #1120 removed the shipped `vault-template/.memoria/mcp/` adapter
home, moved policy-hook enforcement into `src/memoria_vault/runtime/policy/`,
kept optional adapter/plugin entry points around the installed runtime package,
rewrote CI/installer/docs/generated references away from `.memoria/mcp`, and
expanded the negative gate so `.memoria/mcp` cannot reappear in runtime,
installer, tests, or current docs. PR #1121 removed legacy Work Markdown source
fallbacks from runtime digest, integrity, rollback, projection, and seeded-error
source paths; checked SQLite catalog rows are now the source authority. PR #1122
removed the shipped fleet-health dashboard, Fleet nav target, and Memoria
Inspector lane-metric/Fleet trust panel, then expanded the negative gate for
those contracts. The main blockers are full parity/runtime-local gate coverage,
full-text/graph/retrieval/gap/discovery completion, installed runner smoke
evidence, and removal of the remaining dropped-baseline residue
(Obsidian-required paths, Zotero-specific import/operation surfaces,
optional-adapter lane policy residue, and stale current-doc/ADR contradictions).

## Scope

Alpha.14 delivers, in `Memoria-test`, a grouped `memoria` CLI over the existing
SQLite engine that runs the thin end-to-end cycle - `init`, `work capture`,
`work enrich`, `work update`, `work interview`, `work digest`,
`note propose/accept/link`, `memoria ask`, `project ask`, `project export`, and
`workspace recover` - with no Obsidian, Zotero, Hermes, MCP, or profile
dependency. Capture becomes SQLite-only (no legacy Work Markdown, no OpenAlex
record Markdown). The runner becomes pydantic-ai over an OpenAI-compatible
endpoint. Operation manifests become file-per-capability with optional same-stem
asset folders only for oversized prompts, schemas, or examples. Existing
operation logic is imported into the new runtime before obsolete operation homes
are removed. The dropped-baseline surfaces are removed from runtime, installer,
current docs, and tests, and an `rg` negative gate proves it. The full
user-facing parity floor (palette actions plus the manifest-derived operation
set, qmd-backed checked-only keyword/semantic retrieval, full-text acquisition,
first-order citation/topic graph, embeddings, discovery candidates,
attention/worklists, linter/session summaries, diagnostics, and export) runs
through CLI/engine.

Dropped from the baseline (not deferred): Obsidian as required surface, Zotero as
Work authority, Hermes, MCP as required operation API, installed profiles,
lane/fleet, `direct_api` as a product runner, backward-compatibility/migration
branches, directory-only capability manifests, and legacy markdown Work support.

Later / deferred: ISBN and URL-primary depth beyond the full-text/digest
contract; provider-specific Semantic Scholar, PubMed, and arXiv hardening beyond
the OpenAlex/full-text baseline; exhaustive citation expansion; non-baseline
embedding jobs; topic expansion beyond OpenAlex/controlled facets; full Writer
agent and automated draft/verify workflow; RO-Crate/BagIt and in-toto/SLSA; any
optional Obsidian/MCP adapter or Zotero adapter around generic BibTeX/CSL files
and the stable CLI/engine.

## Known Limitations

- Limitation: sandbox-only; no installed-workspace migration target and no
  backward-compatibility mode. Impact: alpha.14 starts from fresh disposable workspaces
  only. Workaround: test only in `~/Memoria-test`. Tracking: alpha.14 parent.
- Limitation: provider breadth. Impact: alpha.14 requires qmd, full text,
  first-order citation/topic graph, embeddings, and discovery for the DOI/OA plus
  user-supplied URL/PDF/text baseline, but deeper provider-specific hardening and
  exhaustive graph expansion remain later.
  Workaround: use OpenAlex/Unpaywall/user-supplied text first; schedule provider
  hardening after the baseline gates pass. Tracking: alpha.14 follow-ups.
- Limitation: no Obsidian/Zotero/MCP product surface. Impact: the CLI is the
  only product surface this checkpoint; portable BibTeX/CSL files are the
  supported bibliography import path, and adapters are future optional edges.
  Workaround: operate via `memoria`, a plain editor, and generic bibliography
  files. Tracking: adapter follow-up.
- Limitation: re-enrichment cadence and workspace-export packaging beyond Markdown +
  `references.bib` remain open. Impact: they do not block the first checked work,
  digest, ask, project export, or recovery path. Workaround: keep volatile-field
  rechecks manual or scheduled explicitly until cadence is decided. Tracking:
  alpha.14 open questions.
