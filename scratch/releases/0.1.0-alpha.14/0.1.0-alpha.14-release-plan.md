## Status

Draft sandbox release plan for the alpha.14 standalone CLI + engine cut. This is a
full architecture cut, not a partial phase: Memoria becomes a standalone local CLI
tool with a SQLite engine, and Obsidian, Zotero, Hermes, MCP, installed profiles, and
lane/fleet are dropped from the baseline. The alpha.12 SQLite working-state boundary
(ADR-122) and alpha.13 DOI enrichment (ADR-123) already landed on `main`, so the
engine exists; the main blockers are that there is no `memoria` CLI, that
`capture-source` still dual-writes markdown catalog + entity Concepts alongside
SQLite, the old operation implementations still live under obsolete transport
homes, and the dropped-baseline surfaces (Hermes runner, 46 profile files,
lane/override system, MCP-as-required, `direct_api`-as-runner) are still shipped.

## Scope

Alpha.14 delivers, in `Memoria-test`, a `memoria` CLI over the existing SQLite engine
that runs the thin end-to-end cycle - init, capture-source, enrich-source,
compile-digest, ask, recover - with no Obsidian, Zotero, Hermes, MCP, or profile
dependency. Capture becomes SQLite-only (no `catalog/sources/*/source.md`, no entity
markdown). The runner becomes pydantic-ai over an OpenAI-compatible endpoint. Operation
manifests become directory-per-capability and flat markdown stops loading. Existing
operation logic is imported into the new runtime before obsolete operation homes are
removed. The dropped-baseline surfaces are removed from runtime, installer, current
docs, and tests, and an `rg` negative gate proves it. The full user-facing parity floor
(palette actions plus the manifest-derived operation set) runs through CLI/engine.

Dropped from the baseline (not deferred): Obsidian as required surface, Zotero as
authority, Hermes, MCP as required operation API, installed profiles, lane/fleet,
`direct_api` as a product runner, backward-compatibility/migration branches, flat
operation manifests, and `catalog/sources/*/source.md` support.

Later / deferred: ISBN and URL-primary depth, Semantic Scholar, PubMed, arXiv, citation
graph, embeddings, topics, full-text acquisition (unless a scheduled gate is added);
RO-Crate/BagIt and in-toto/SLSA; any optional Obsidian/Zotero/MCP adapter around the
stable CLI/engine.

7. Drop-baseline cleanup sweep + `rg` negative gate (Hermes, lanes, 46 profiles, MCP,

## Known Limitations

- Limitation: sandbox-only; no installed-vault migration target and no
  backward-compatibility mode. Impact: alpha.14 starts from fresh disposable workspaces
  only. Workaround: test only in `~/Memoria-test`. Tracking: alpha.14 parent.
- Limitation: DOI-first enrichment breadth. Impact: ISBN, URL-primary, Semantic Scholar,
  PubMed, arXiv, citation graph, embeddings, and full text remain deferred behind a
  scheduled gate. Workaround: use the DOI/CSL/BibTeX paths; schedule follow-ups.
  Tracking: alpha.14 follow-ups.
- Limitation: no Obsidian/Zotero/MCP surface. Impact: the CLI is the only product
  surface this checkpoint; adapters are future optional edges. Workaround: operate via
  `memoria` and a plain editor. Tracking: adapter follow-up.
- Limitation: re-enrichment cadence and minimal export beyond Markdown +
  `references.bib` remain open. Impact: they do not block the first checked source,
  digest, ask, or recovery path. Workaround: keep volatile-field rechecks manual or
  scheduled explicitly until cadence is decided. Tracking: alpha.14 open questions.
