# Memoria - 0.1.0-alpha.14 design (standalone CLI + engine)

Date: 2026-06-30
Status: working design for `0.1.0-alpha.14`. This design reconciles the best
parts of alpha.11, alpha.12, and alpha.13 into one CLI-first product direction.
It supersedes earlier scratch claims that make Obsidian, Zotero, Hermes, MCP, or
installed profiles part of the required architecture.

Method: carry forward the product thesis and integrity model from alpha.11, the
SQLite-backed engine and keep-test from alpha.12, the OpenAlex-style work enrichment and
clean concept boundary from alpha.13, and the working implementation primitives
that already prove useful behavior.

---

## 0. Decision

Memoria is a standalone local CLI tool and engine.

Required baseline:

- `memoria` CLI.
- Local workspace on disk.
- SQLite for the OpenAlex-style graph, operations, queue, journal, checks, and
  rollback graph.
- Markdown files for the user keep-set.
- Content-addressed blobs for raw work and provider payloads.
- OpenAI-compatible model runner, local or remote, behind Memoria's runner API.
- qmd CLI for checked-only keyword and semantic retrieval.
- Git CLI for checked file materialization and journal anchoring.

Outside the baseline:

- Obsidian.
- Zotero.
- Hermes.
- MCP servers.
- Installed profiles.
- A plugin, browser UI, companion app, hosted service, or always-on control
  plane.

Obsidian UI, a Zotero adapter, or MCP/BYO-agent access may exist later only as
optional adapters around the CLI/engine and generic exported files. Hermes,
installed profiles, lane/fleet runtime, and `direct_api` as a product runner are
dropped, not deferred. No
external adapter ever owns state, workflow, policy, checks, or the write
boundary.

Cutover rule:

- Alpha.14 is a full architecture cut, not another partial phase.
- Default to implementing the final baseline now.
- Defer only when alpha.14 cannot safely decide or implement the behavior with
  available evidence or local prerequisites.
- Every deferral names the temporary behavior, the gate that proves it, and the
  condition that removes it.
- No competing product paths. Old mechanisms are either ported behind the new
  CLI/engine boundary or dropped from the baseline.
- There is no production vault, so alpha.14 has no backward-compatibility mode,
  no migration path, and no legacy runtime branch. Existing compatibility code,
  tests, docs, and schema paths are removed or reclassified as historical/import
  notes before the design is called implemented.
- Current runtime, installer, schema, tests, and current docs must not carry
  alpha.11/alpha.12/alpha.13 compatibility branches. Historical scratch and ADR
  provenance may describe old layouts; product code may not support them.

## 1. Product Thesis

Carry forward from alpha.11:

Memoria is an opinionated authorial thinking-and-writing tool. The human directs
the research program; the LLM and deterministic workers do labor under that
direction. Safety comes from traceability, checks, read barriers, and rollback,
not from requiring a human to approve every generated write.

Genesis: Karpathy's LLM wiki — an LLM-driven, compounding markdown store —
crossed with the historian's Zettelkasten method (works -> atomic notes ->
synthesis, driven by the scholar's judgment). The wiki alone is a knowledge
*base*; adding the ZK graph turns it into knowledge *production*. Integrity then
rests on the graph itself — traceability and reversibility over a checked
derivation graph — not on approval gates.

The core loop stays:

```text
open project -> seed -> gap analysis -> capture work -> enrich work
-> guided conversation -> digest + hub compounding -> notes -> re-gap
```

The Co-PI is an engine behavior, not a UI dependency. In alpha.14 the first
surface is the CLI:

```bash
memoria operation run <operation-id>
memoria request answer <request-id> key=value
memoria request resume <request-id>
```

An editor can still open the markdown files. The product must remain fully
operable if the editor is only `vim`, `nano`, or a file browser.

## 2. Architecture

```text
CLI       memoria commands, stdin/stdout, JSON mode, exit codes
Engine    request envelope, queue, worker, checks, read barrier, recovery
Ops       typed operation specs, pydantic-ai runner, deterministic helpers
Storage   SQLite authority for OpenAlex-style graph + ops; Markdown keep-set; blob store
Export    references.bib, OKF Knowledge Bundle takeout, optional JSONL reports
Adapters  future optional importers/UIs/APIs that call the CLI/engine
```

The engine is the product boundary. All surfaces create the same request
envelope and all writes pass through the same worker lifecycle.

Minimum request envelope:

```text
request_id
operation_id
args
idempotency_key
actor
input_refs[]
output_intents[]
primary_target?
precondition_hashes{}
causal_refs[]
provenance
schedule_id?
```

Work enters exactly three ways:

- CLI command: a `memoria ...` invocation typed by a human or a script.
- File change: `memoria workspace scan` classifies a direct edit to keep-set
  markdown.
- Scheduled task: a normal `memoria ... --schedule-id <job>` invocation fired
  by cron, a systemd timer, launchd, or Task Scheduler.

There is no dedicated ingress-type field in the request envelope. The command
name, actor, provenance, and optional `schedule_id` are enough. Only
`memoria workspace scan` has distinct behavior: the scan import/quarantine path
below. No ingress path may write storage directly.

`memoria workspace scan` is the only bridge for direct editor/file-browser edits.
It journals an `observed_external_edit`, creates a request row for the observed
edit, and marks the changed Concept `unchecked` and not machine-consumable until
the normal checks promote it again. If the changed file was previously checked,
the old checked materialization remains in the journal as history, but the new
hash does not inherit that status.

Foreign-write classification. `scan` compares each changed path against the
materialization journal. A changed keep-set Concept is imported as a PI edit,
with `inputs = prior upstream inputs + the prior checked hash from HEAD` so
cascade rollback still reaches its machine-derived origins. A change to a
generated projection, a worker-owned path, `references.bib`, a capability
manifest, or a machine-authored marker has no PI-edit path and is quarantined,
never silently re-consumed. A commit with no matching materialization row is
foreign by the same rule.

## 3. Storage And Authority

Carry forward alpha.12's keep-test:

> The thing the user keeps if Memoria dies must be markdown files plus a
> portable bibliography export. Memoria working state may be SQLite.

| Component | Authority | Projection / export |
| --- | --- | --- |
| Knowledge bundle | OKF Knowledge Bundle under `knowledge/` | SQLite read model, `knowledge/index.md`, `knowledge/_views/` |
| Capability bundle | OKF Knowledge Bundle under `capabilities/` | `capabilities/index.md`, `capabilities/_generated/capability-index.json` |
| OpenAlex-style graph records | SQLite in the workspace aux state | `references.bib`, optional JSON/OKF export |
| Operations/journal/queue | SQLite in the workspace aux state | optional JSONL/report export |
| Raw/provider payloads | content-addressed blobs in the workspace aux state | hashes in SQLite |
| Search/views/qmd | derived from checked Concepts, acquired text, and SQLite graph read models | `knowledge/_views/`, rebuilt SQLite views, `.memoria/index/qmd/{checked,config,index.sqlite}` |

There is no markdown work-graph folder in the alpha.14 baseline. Works,
Authors, OpenAlex Sources, Institutions, Topics, Publishers, Funders, provider
payloads, external IDs, and graph links are records in SQLite. A Work's keep-set
presence is its digest.

The portable takeout set is:

```text
knowledge/
capabilities/
references.bib
```

This is enough to keep thinking and writing outside Memoria for completed,
checked operations. It is not a full backup of Memoria's working state. A full
Memoria backup also needs SQLite and blobs.

Runtime workspace layout:

```text
<workspace>/
  knowledge/
  capabilities/
  references.bib
  .memoria/
    memoria.sqlite
    config/providers.yaml
    blobs/
    index/qmd/checked/
    index/qmd/config/
    index/qmd/index.sqlite
    journal-head
```

No runtime path is named `vault-template/`. The repository may seed this
skeleton during install, but the installed workspace is self-contained.
`.memoria/` is only Memoria-owned auxiliary state: database, config, blobs, and
journal anchoring. It is not a profile tree, MCP home, operation host, or
adapter runtime.

OKF bundle contract:

- `knowledge/` and `capabilities/` are OKF Knowledge Bundles: each is a
  directory tree of Markdown Concepts plus a generated bundle-root `index.md`.
- A Concept ID is the path without `.md`, relative to its bundle root.
- Every Concept is YAML frontmatter followed by a Markdown body. Frontmatter is
  strict Memoria schema; body headings are type-owned Markdown, with OKF
  reserved headings such as `# Schema`, `# Examples`, and `# Citations` used
  only where that content belongs.
- Generated views and indexes are projections, not Concepts, unless they live
  as bundle `index.md`.

## 4. Object Model

Use alpha.13's clean boundary, without making any UI part of the model.

| Class | Storage | Owns |
| --- | --- | --- |
| Records | SQLite | Works, Authors, OpenAlex Sources, Institutions, Topics, Publishers, Funders, links, provider evidence, field provenance |
| Knowledge Concepts | Markdown in `knowledge/` | digest, note, hub, project |
| Capability Concepts | Markdown in `capabilities/` | operation, skill, workflow, adapter/tool |
| Projections | Generated files | references, indexes, reports |

Concept frontmatter baseline, applied to both bundles:

```yaml
id: <stable-id>
type: digest | note | hub | project | steering | vocabulary | asset | operation | skill | workflow | adapter
title: <human title>
check_status: unchecked | checked | quarantined
standing: current | superseded | retracted | archived
description: <one-sentence summary>
links: {}
tags: []
facets:
  research_area: []
  methodology: []
  topics: []
```

`facets` are typed vocabulary fields, not free tags. Works and Concepts use the
same controlled terms where applicable; unknown values fail validation until the
PI adds or renames them through the vocabulary contract.

Knowledge Concepts:

- `digest`: one Work or Work bundle's checked machine summary, keyed by
  `work_id`, carrying `confidence`, `contested`, and `contradictions[]` quality
  signals.
- `note`: one durable human-meaningful idea. `note_kind` stays an optional open
  string, but the schema carries the payloads the §9 checks anchor to: an
  optional anchor (`quote`, `text_sha256`, `work_id`, `annotation_ref` = a
  W3C Web-Annotation selector subset preserving page/span/offset) and an optional
  Toulmin-shaped claim block (`claim_text`; `evidence_set` grounds; `warrant`;
  `qualifier`; `rebuttal`/`backing` links; temporal scope). Quote-anchor,
  citation-survival, and unwarranted-claim checks have no target without these,
  so they are modeled now, not deferred.
- `hub`: curated map of an area, with machine suggestions and human direction,
  carrying the same `confidence`/`contested`/`contradictions[]` signals.
- `project`: output-oriented working memory: thesis, scope, decisions, gaps,
  outputs, links, and argument state. With a thesis it carries an argument graph
  — the Toulmin picture of the project's knowledge (claims with
  grounds/warrant/qualifier/rebuttal, support/challenge edges, tensions,
  falsifiability, refutation status) — that the §5 argument-health operations read
  and update, so you can see the shape of the knowledge while working. This is the
  directed/writing half of the loop, not just coverage.
- `steering`: program-level PI priorities and constraints read by operations
  before project-local hints. There is one current steering Concept per
  workspace.
- `vocabulary`: the visible, PI-editable controlled vocabulary for `tags` and
  typed `facets`.
- `asset`: research code, notebooks, figure scripts, supplementary data, and
  other project outputs kept with provenance links. Large files live in a
  same-stem asset folder under `knowledge/assets/`; the Markdown Concept owns
  identity, hashes, citations, and project links.

Minimum body contracts:

- `note` body: claim, evidence, links, open questions. The frontmatter carries
  machine-checkable anchors; the body keeps the human reading shape.
- `hub` body: scope/framing, curated map, machine suggestions, planning/open
  questions. A hub is consumable only after human curation evidence is present;
  machine-only suggestions stay unchecked.
- `steering` and `vocabulary` bodies are human-editable, but `workspace scan`
  validates and projects them into SQLite before machines consume them.

Synthesis discipline (from the Karpathy-wiki and ZK lineage):

- Orient before creating. A synthesis operation reads the index and the existing
  graph for the OpenAlex records/Concepts in play before writing, so it updates existing
  Concepts and cross-references instead of spawning duplicates.
- Creation thresholds. Create a note or hub when a concept appears in two or more
  Works or is central to one; a passing mention is a link, not a new Concept.
  Oversized Concepts split; superseded ones move to `standing: archived`.
- Controlled tags. `tags` are facets drawn from a schema-owned taxonomy,
  add-before-use, so the vocabulary does not sprawl.
- Weak claims must not silently harden. Lint surfaces `confidence: low`,
  `contested: true`, and single-work claims for review before they read as
  settled knowledge.

Capability Concepts:

- `operation`: runnable operation policy and prompt/contract body. Current
  prompt operations use YAML frontmatter plus a `# Pattern` body; alpha.14 keeps
  one Markdown file as the Concept. A same-stem asset folder is allowed only when
  a prompt, schema, or example is too large for the main file.
  Every operation declares `method_class: deterministic | hybrid | generative`.
  Hybrid/generative operations must name the deterministic baseline they beat,
  required checks, allowed paths, allowed network, runner/model policy, risk, and
  output intents. LLM use without that manifest evidence fails closed.
- `skill`: reusable capability metadata and trust information.
- `workflow`: ordered operation composition and required checks.
- `adapter` / `tool`: optional external surface metadata. Current code/docs name
  the equivalent type `mcp`; alpha.14 must either rename it, drop it, or keep it
  only as an optional adapter Concept, never as required operation API.

Records:

- `works`: identity, `item_type`, identifiers, metadata status, check status,
  CSL JSON, content/blob hashes, citekey aliases. Metadata standard follows
  `item_type` (CSL-JSON for scholarly works, DataCite for datasets, CFF/CodeMeta
  for software/repositories), with SPDX license IDs where a license exists. Only
  the dataset/software provider adapters (DataCite, Zenodo) are deferred; the
  record schema and standards posture ship now.
- `authors`, `openalex_sources`, `institutions`, `topics`, `publishers`, and `funders`:
  OpenAlex-named record classes, with external IDs and provider provenance.
- `graph_links`: work/author/openalex_source/institution/topic/citation/version/dedup
  links.
- `external_ids`, `enrichment_runs`, `provider_payloads`,
  `field_provenance`.

Keep-set citation rule:

Any digest, note, or hub that references a Work stores both `work_id` and a
compact survival citation:

```yaml
citations:
  - work_id: doi-10-example
    title: Example title
    authors: [A. Researcher]
    issued: 2026
    doi: 10.0000/example
```

SQLite owns full CSL and provenance. Markdown embeds enough to remain meaningful
without SQLite.

Capability manifest layout:

```text
capabilities/
  index.md
  operations/<id>.md
  operations/<id>/prompt.md        # optional overflow asset
  operations/<id>/schema.json      # optional overflow asset
  operations/<id>/examples/*.md    # optional overflow assets
  adapters/<id>.md
  skills/<id>.md
  workflows/<id>.md
  _generated/capability-index.json
```

The Markdown file is the capability Concept and owns identity, lifecycle,
policy, and references to any assets. Asset folders are not Concepts; a folder
without the matching `<id>.md` file is invalid. The loader enumerates only
capability Markdown files and resolves asset paths only inside the same-stem
asset folder.

Pattern library handling:

- Prompt patterns are not a separate runtime object in alpha.14. They are
  ordinary checked `operation` capability files under `capabilities/operations/`.
- `# Pattern` remains the small-prompt body convention. Oversized prompts,
  schemas, or examples move to the same-stem asset folder and are referenced from
  the operation file.
- The old `patterns_list` and `patterns_run` compatibility tools are removed
  from the required product surface. Listing is `memoria operation list`; running
  is `memoria operation run <operation-id>`.
- The file stem is the `operation_id`; old `pattern_id` wording is compatibility
  input only.
- `output_target` is replaced by `output_intents[]` in the request envelope.
  Operation policy may still name allowed homes and materialization classes, but
  no prompt operation owns a direct canonical write path.
- The current prompt operations carry forward if still useful:
  `analyze-claims`, `check-falsifiability`, `compare-and-contrast`,
  `extract-claim-stubs`, `propose-note-candidates`, `red-team-argument`,
  `summarize-for-recall`, `surface-tensions`, and the digest operation behind
  `memoria work digest`. Their old work-facing inputs are renamed to OpenAlex
  terminology such as `checked-work` and `work-note`.
- Pattern provenance remains, but it is operation provenance: request row,
  operation id/version, input refs, runner/model, prompt hash, output intents,
  checks, and journal evidence. A separate `patterns.jsonl` log is not required.

## 5. CLI Surface

The first complete surface is `memoria`. Its grammar is resource-oriented:

```text
memoria [global-options] <resource> <action> [action-options]
```

Global options:

```text
--workspace PATH       required except for help/version and init
--json                 stable machine-readable output
--quiet                suppress non-error human output
--verbose              write diagnostics to stderr
--yes                  non-interactive confirmation
--dry-run              plan without mutation
--idempotency-key KEY  script-supplied idempotency
--schedule-id KEY      scheduler-supplied identity; does not change behavior
```

Use `--workspace`, not `--vault`: alpha.14 is a standalone workspace, not an
Obsidian vault adapter.

Required command surface:

```text
memoria init
memoria status
memoria doctor
memoria doctor bundle
memoria doctor self-test
memoria ask
memoria work capture
memoria work import
memoria work enrich
memoria work digest
memoria work interview
memoria work update
memoria note capture
memoria note propose
memoria note accept
memoria note reject
memoria note link
memoria project ask
memoria project trace
memoria project gaps
memoria project suggest-hubs
memoria project export
memoria request answer
memoria request amend
memoria request cancel
memoria request retry
memoria request resume
memoria request list
memoria request show
memoria attention list
memoria attention show
memoria attention resolve
memoria attention worklist
memoria operation list
memoria operation run
memoria steering show
memoria steering edit
memoria vocabulary list
memoria vocabulary add
memoria vocabulary rename
memoria journal list
memoria journal show
memoria workspace scan
memoria workspace run
memoria workspace recover
memoria workspace rollback
memoria workspace check
memoria workspace rebuild
memoria workspace export
memoria eval run
memoria eval seeded-error-verdict
```

Only these names are top-level alpha.14 commands. Capability-specific work that
does not earn a human-facing alias runs through `memoria operation run
<operation-id>`. This keeps the CLI small while still covering the full manifest
set.

Command options own variants:

- `work capture` accepts `--doi`, `--url`, `--pdf`, or `--file`; there are no
  separate `capture-url` or `capture-pdf` commands.
- `work import` accepts `--format bibtex|csl --file <path>`; Zotero-specific
  `zotero-export` is not an alpha.14 format. Users may export generic BibTeX or
  CSL outside Memoria and import those files.
- `work interview` records the guided Work conversation used by digest
  compilation; multi-turn prompts still use `request answer` and `request
  resume`.
- `work update` owns Work metadata and lifecycle changes: archive/restore,
  retraction/contested standing, human classification, and corrected identifiers.
  There are no separate archive, restore, or classify commands.
- `note accept`, `note reject`, and `note link` own note-candidate and typed-link
  curation; direct file edits go through `workspace scan`, not silent curation.
- `project export` renders a checked project draft/composition to Markdown by
  default and to Pandoc-backed formats when Pandoc is installed. It is a
  deterministic export step, not a Writer agent.
- `memoria ask` is workspace-scoped checked retrieval; `project ask` adds project
  memory, thesis, and argument context.
- `attention worklist` emits/list/updates aggregate worklists so high-cardinality
  review does not become one prompt per row.
- `request retry`, `request cancel --reason`, and `request amend` are the safe
  recovery controls for blocked or stale queue work.
- `steering show/edit` exposes program-level steering memory independent of any
  one project.
- `vocabulary list/add/rename` is the PI-editable path for schema-owned facets
  and controlled tags.
- `journal list/show` filters audit/session events by request, path, decision,
  date, and operation.
- `workspace run` drains pending queue work; scheduled wrappers call the same
  command with `--schedule-id queue-drain`.
- `workspace check` runs the structural/lint checks.
- `workspace rebuild` refreshes derived projections and `references.bib`.
  `workspace rebuild --search --embeddings` additionally rebuilds the
  checked-only qmd source tree, qmd keyword index, and qmd semantic index.
- `doctor --check runner --provider <name>` checks runner dispatch.
- `doctor bundle` emits a redacted local support bundle; `doctor self-test`
  exercises diagnostic capture/redaction without external services.

Scheduled maintenance:

- There is no internal daemon. A scheduler invokes normal CLI commands with
  `--schedule-id`; the request envelope records the schedule id but has no
  ingress-type field.
- Required alpha.14 jobs are queue drain, file scan, structural check, projection
  rebuild, retraction/provider refresh, and eval. They may be cron, systemd
  timers, launchd, Task Scheduler, or a human-run script.
- Old board export and metrics aggregation become `status --json`, journal
  summaries, and generated `knowledge/_views/` reports; no board daemon survives.

Rejected command names for alpha.14: `work capture-url`,
`work capture-pdf`, `work import-bibtex`, `work import-csl`,
`work import-zotero-export`, `work ingest`, `workspace run-pending`,
`workspace run-scheduled`, `workspace lint`, `projection regenerate`,
`reference regenerate`, `search rebuild`, `diagnostics runner`, and
`diagnostics quarantine`.

Minimal multi-turn syntax:

```bash
memoria project ask --workspace <path> <project-id> --question "..." --json
memoria request answer --workspace <path> <request-id> key=value --json
memoria request resume --workspace <path> <request-id> --json
```

`project ask` is for user questions and Co-PI-style clarification, not operation
startup. Operation conversations start with `memoria operation run
<operation-id>` and then use `request answer` / `request resume` with the
returned request id.

CLI rules:

- Every mutating command creates a durable request row before work starts.
- Every workspace-scoped command has `--workspace`; every command supports
  `--json` where output is emitted and has stable nonzero exit codes.
- Commands import without Obsidian, Zotero, Hermes, MCP, profile env vars, or
  GUI state.
- `memoria status` is read-only.
- `memoria doctor` checks local prerequisites and never repairs without an
  explicit flag.
- The CLI is usable by a human and scriptable by future adapters.

`status --json` is the replacement for dashboard glance surfaces. It reports
request counts, queue state, attention/worklist counts, failed checks, projection
freshness, search freshness, last scheduled runs, provider config status,
journal-chain status, and next recommended action. Generated views under
`knowledge/_views/` cover daily glance, structural health, operational health,
and synthesis agenda as files, not UI cards.

The CLI is not "operator-only." It is the product surface for alpha.14.

The alpha.14 product loop is complete only when the CLI can run a thin end-to-end
cycle: import or capture a Work, enrich it, ask/resume the interview, create a
Work digest, suggest hub updates, propose notes, curate accepted notes/links, run
qmd-backed retrieval and gap analysis with unchecked discovery candidates, export
a checked project draft, and recover from an interrupted materialization.

First integrated smoke, not a product phase: CLI imports, `work capture`,
`work enrich`, `work digest`, `workspace rebuild --search --embeddings`,
`memoria ask`, `project gaps`, unchecked discovery candidates, and
`workspace recover`. That smoke is useful for merge order, but alpha.14 is not
implemented until the parity, cleanup, seeded-error, and no-compatibility gates
below pass.

Current user-facing parity floor:

Alpha.14 is non-regressing only when every current Obsidian command-palette
action and every current Inspector worker action has a CLI/engine equivalent
that runs without Obsidian, Zotero, Hermes, MCP, or installed profiles.
Obsidian-specific shell repair remains adapter-only; it is not core product
functionality.

| Current user-facing action | Alpha.14 equivalent |
| --- | --- |
| `Memoria: capture note` | `memoria note capture` writes one unchecked PI note Concept. |
| `Memoria: open Inbox` | `memoria attention list` shows unresolved attention. |
| `Memoria: record exploration trace` | `memoria project trace <project-id>` writes a project-local trace note. |
| `Memoria: resolve inbox card` | `memoria attention resolve <id> --outcome resolved`. |
| `Memoria: dismiss inbox card` | `memoria attention resolve <id> --outcome dismissed`. |
| `Memoria: restore shell on startup` | Optional Obsidian adapter macro only; no CLI core behavior. |
| Memoria Inspector enqueue actions | `memoria operation run <operation-id>` plus named aliases above. |

The operation parity set is not hand-copied into this design. The gate
enumerates final `capabilities/operations/*.md` manifests at test
time; every manifest file stem must be runnable through
`memoria operation run <operation-id>` or explicitly marked adapter-only/dropped
with a reason. Current operation files in the implementation repository are
conversion input until they pass the alpha.14 file-per-capability contract. The
gate proves directory-only capability manifests do not load.

Existing operation implementations are import inputs, not disposable history.
The legacy ingest, eval, linter, retraction, merge, project, cleanup, and shared
library modules must be imported into the new runtime engine before their old
transport homes are removed. Import means: the behavior runs through the request
envelope, file capability manifests, SQLite state, and the same tests or
replacement tests. No old operation is deleted just because its old host was
Hermes/MCP/profile-shaped.

Directed-mode argument operations are a capability family reachable through
`memoria operation run`: analyze-project-argument and its canvas projection, plus
extract-claim-stubs, analyze-claims, surface-tensions, compare-and-contrast,
red-team-argument, and check-falsifiability. They read and update the project
argument state (§4) and carry over from the current implementation — argument-
health lenses on a thesis project, not new machinery.

`work import --format zotero-export` is not preserved as a product alias.
Portable bibliography import means generic BibTeX/CSL files only; a future
Zotero adapter can produce those files or call the same CLI.
Historical palette commands that current docs mark as removed are not
compatibility commitments: sample-vault load/remove, start project, map corpus,
draft section, verify draft, old delegation/per-lane commands, and retired
workspace switches stay absent. If an underlying capability remains current, it
is exposed through the CLI operation model, not by resurrecting the old command.

## 6. Operation And Write Lifecycle

Carry forward alpha.12's lifecycle:

```text
requested -> stage -> validate -> check -> DB commit -> file materialize -> project
```

Steps:

1. `requested`: insert request, queue row, idempotency key, actor, and inputs.
2. `stage`: produce DB row candidates and durable file payloads.
3. `validate`: Pydantic shape checks plus SQLite FK/UNIQUE/CHECK/DAG checks,
   evaluated against committed state plus this operation's staged set. A citation
   may target a `checked` row or a same-operation staged row; `checked`-ness is a
   read-barrier property, not a write-validity one.
4. `check`: structural and policy checks run before any machine output is
   consumable.
5. `DB commit`: write graph/ops/journal/check rows and file payload metadata.
6. `file materialize`: write checked markdown/projections from durable payloads,
   commit/anchor them, then record `materialized`.
7. `project`: refresh indexes/views from SQLite + files.

Crash rule:

A DB commit with pending file materialization is not complete and not
consumable. `memoria workspace recover` resumes from SQLite and the durable payload. A
hash-only pending row is invalid because it cannot replay a lost file payload.

Rollback:

- DB-backed rows roll back with inverse SQLite transactions.
- File-backed Concepts roll back with inverse materializations.
- Blast radius is a recursive graph query over graph links, derivation edges,
  citations, and materialization records.

Request controls:

- `request retry` replays a failed or cancelled request only when its
  idempotency key, inputs, and precondition hashes still match.
- `request cancel --reason` moves pending/blocked work to terminal cancelled
  state without deleting its evidence.
- `request amend` supplies missing answers or corrected inputs to a blocked
  request and records the amendment as a new journal event.
- One mutating worker holds the workspace write lock at a time. Pending,
  blocked, and attention queue limits surface through `status --json` before
  the workspace silently piles up work.

## 7. Runner Model

The runner is a Memoria implementation detail.

Baseline:

- pydantic-ai for typed LLM calls, structured output, validation, and retries.
- OpenAI-compatible endpoint shape for local models, gateways, or direct
  vendors.
- Operation policy pins model/provider/routing for checks and evals.
- Direct deterministic helpers are preferred whenever they work.
- Extraction and check operations run a faithfulness contract: closed-domain (no
  claim not present in the Work), verify the target exists before extracting it,
  forced verbatim grounding, temperature 0, and self-consistency sampling where
  disagreement across samples becomes the uncertainty flag. Constrain the output
  (typed schema, constrained decoding), not the model; a small constrained model
  matches scale at single-user size.
- Auxiliary comparators — the NLI/entailment checker (contradiction, supersession,
  dedup) and embeddings — are pinned, local-capable components, separate from the
  generation runner.

Rejected as baseline:

- Hermes runtime or Hermes-as-library.
- LangChain/LlamaIndex-style framework-owned state.
- Workflow engines.
- Agent session memory as authority.
- Provider auto-routing for calibrated checks.
- Scalar or learned self-improvement loops that replace human judgment:
  co-trained generator+reviewer, tournament/evolution ranking, tree-search over
  synthesis, and confidence-routed bypass of the human checkpoint.
- Preference internalization into model weights; personalization stays external
  and inspectable (workspace content, config), never trained in.

Borrow patterns, not runtimes. The Karpathy wiki pattern is prompt/schema/workflow
material inside a Memoria operation, not a dependency on a wiki engine.

## 8. Work Capture And Enrichment

Carry forward alpha.13's capture/enrich split.

`work capture`:

- Stores raw/content hashes, initial identifiers, minimal work row, request
  and journal rows.
- Leaves `metadata_status = partial | unverified`.
- Leaves `check_status = unchecked`.
- Does not update `references.bib` for DOI/ISBN works.
- Does not create markdown Work, Author, OpenAlex Source, or Institution records.

Portable seed imports:

- `work import --format bibtex` and `work import --format csl` accept local
  BibTeX/CSL files as standalone work seed inputs.
- They create unchecked work rows and queued enrichment work.
- They are not Zotero import paths, do not read a reference-manager database, and
  do not make citekeys authoritative.

`work interview`:

- Records the guided conversation for one Work or Work bundle.
- Stores interview answers as checked request/journal state only after normal
  validation.
- Feeds `work digest`; digest compilation fails if it requires an interview that
  is missing or blocked.

`work update`:

- Applies human-corrected metadata, `standing`, active/archive state, typed
  `facets`, identifier corrections, and contested/retraction dispositions.
- Recomputes active views, retrieval filters, and `references.bib` if the change
  affects citation output.
- Resolves any linked attention item in the same request, so classification
  review and correction are one auditable operation.

`work enrich`:

- Resolves identifiers and work branch.
- Fetches required providers.
- Caches raw provider payloads as content-addressed blobs under
  `<workspace>/.memoria/blobs/provider-payloads/`.
- Acquires legal digestable work text: user-supplied PDF/text first, otherwise
  provider-discovered open-access PDF/HTML/text. Abstracts and metadata are not
  enough for a checked digest.
- Extracts text with stable anchors (page/span/offset where available) and stores
  content-addressed text blobs for digest, quote-anchor checks, retrieval, and
  rollback.
- Normalizes provider records.
- Merges canonical CSL with per-field provenance.
- Reconciles identities with external IDs.
- Imports the first-order citation/reference/topic graph available from required
  providers and marks discovered candidate Works as unchecked attention/worklist
  rows, not checked knowledge.
- Builds or queues embeddings for acquired text, digests, notes, hubs, compact
  citations, and graph neighborhoods.
- Checks conflicts, required-provider coverage, and retraction/contested status.
- Promotes to consumable `checked` only after required checks pass.
- Stores durable `references.bib` payload/hash and materializes it before the
  operation completes.

Branches:

| Branch | Required for checked promotion | Optional / async |
| --- | --- | --- |
| DOI scholarly | Crossref; OpenAlex; Unpaywall/OA locator; acquired full text or user-supplied text; first-order citation/topic graph; embeddings | S2, PubMed, arXiv, ORCID/ROR, DataCite, retraction hardening |
| ISBN book | Open Library or Google Books; acquired full text or user-supplied text before digest | other book provider, Wikidata |
| URL / primary work | URL metadata extraction; acquired page/PDF text; embeddings | Wayback/archive, Wikidata |

Dataset and software works (DataCite/Zenodo-keyed, CFF/CodeMeta metadata) are a
deferred branch: the `item_type` and per-type metadata standards (§4) ship now;
the provider adapters do not.

Provider policy:

- `<workspace>/.memoria/config/providers.yaml` is runtime data config for
  provider roles, timeouts, token buckets, required/optional sets, and env var
  names. The repository may seed this file at install; the runtime never reads
  provider policy from a repository checkout or template directory.
- The operation manifest is the hard ceiling for network access.
- Every HTTP call passes a shared operation allowlist check before network I/O.
- Secrets and mailto/email values live outside tracked config.

Decided policies (were §15 open questions):

- Identifier resolution uses multi-signal corroboration, not a single score:
  auto-accept a resolved DOI only when at least two of {title similarity, author
  overlap, issued-year match} agree; otherwise abstain and create an attention
  item. Exact thresholds calibrate on the seeded-error fixture.
- Digest text is required: a checked digest is compiled only from acquired full
  work text or a user-supplied equivalent. Abstracts and metadata may create a
  preview, gap, or attention item, but they cannot produce a checked digest.
  Full-text acquisition is therefore baseline alpha.14 functionality, not a
  deferred enrichment.

## 9. Integrity And Security

Carry forward alpha.11's integrity thesis, with alpha.12 mechanisms.

Guarantees:

- Attributable: every output has operation, request, input hashes, runner, model,
  prompt version, and materialization evidence.
- Reversible: rollback works by output/record, not whole run.
- Detected: structural checks run on write and on schedule.
- Read-barriered: machines consume only checked DB rows and checked +
  materialized file outputs.
- Audited: every allowed mutation records before/after evidence in an append-only
  journal that is hash-chained (`event_id`, `prev_hash`, `row_hash`) and anchored
  by a git-tracked journal-head record at each completed operation — single-user
  tamper-evidence without a verifiable-log server.

Read barrier:

```text
DB row:       check_status = checked
File output:  check_status = checked AND materialization_status = materialized
```

Anything missing a status fails closed for machine consumption.

Flag routing (not per-write approval). A fired check is routed, not queued for
blanket approval: `act` = a deterministic reversible fix the worker applies and
records (re-check, retrace, auto-rollback a clearly-bad machine write); `ask` =
high-risk or provenance-triggered (freshly-ingested, uncorroborated external
Work) surfaced to the human as attention; `drop` = a low-value candidate, logged
not shown. This bounds human attention without restoring an approval queue.

Attention contract:

- Attention items preserve the current honesty-card shapes: finding, proposal,
  verification, and work prompt. Each carries loudness, certainty, argument
  target, source request, suggested action, and due/blocked state.
- Worklists are aggregate attention objects with per-row decisions and dedupe
  keys. They exist when a batch is the honest unit of review.
- `attention resolve` records accept/reject/dismiss/defer outcomes; accepted
  note and link candidates route to `note accept` / `note link` so curation is
  explicit and journaled.

Check/linter surface:

- `workspace check` is the alpha.14 linter entrypoint. It preserves the current
  user-visible findings even where old paths disappear: schema/frontmatter,
  broken links, misplaced Concepts, citation survival, quote anchors,
  journal-chain integrity, package-baseline drift, projection/search freshness,
  vocabulary/facet validity, stale-as-current evidence, retraction/contested
  evidence, and generated-view drift.
- Old detector names do not have to survive when their old Obsidian/profile
  target is gone; the finding class and blocking/advisory behavior must survive.
- Scheduled checks also write deterministic request/session summaries: inputs,
  outputs, checks, decisions, touched paths, final hashes, and failed-open/closed
  status.

Check construction:

- Deterministic first. Warrant checks are model-free where possible — quote
  substring presence, OpenAlex record-type match, year ordering, no opposite-edge, citekey
  resolves — so a cheap engine cannot post-rationalize provenance.
- Contradiction, supersession, and dedup run on entailment (NLI), not embedding
  cosine, which is blind to negation. The verdict is tri-class SUPPORTED /
  REFUTED / NOTENOUGHINFO with a recorded warrant sentence; NOTENOUGHINFO/abstain
  is first-class, and a tunable abstain threshold with a false-positive budget
  keeps a false "duplicate"/"contradiction" from being the costly error. Dedup
  keys on canonical IDs, since models mint near-duplicates.

Security controls:

- External text is untrusted data.
- Data/instructions are separated.
- Raw work text is never system prompt material.
- Network access is operation-scoped and allowlisted.
- Generation and checking are separate operation roles.
- A retracted or contested work is never silently checked.
- Capability import is supply chain and starts quarantined.
- Agent-rendered UI, if any later exists, is declarative and graph-bound.

Residual risk:

No structural check proves semantic truth. Seeded-error evaluation remains the
gate to real use. If recall is weak for crafted injection or unwarranted claims,
add a targeted human checkpoint for that class. The checkpoint is itself
calibrated: checkpointed items must be measurably more correct than the same
items un-checkpointed, or it is theatre and gets thinned. Under baseline
discipline, every check, retrieval, and synthesis capability must beat a declared
cheap baseline (no-checks, BM25, whole-read) before it counts as a design claim.

Provisional alpha.14 seeded-error bars:

- 100% detection recall on the frozen seeded-error fixture for structural
  classes: citation survival, quote anchor, link target, provenance checkpoint,
  counterfactual faithfulness (a claim re-cited into a decoy span is flagged),
  retraction/contested evidence, stale-as-current evidence, and rollback
  completeness.
- Prompt-injection persistence and well-formed-but-unwarranted claim fixtures
  must either be detected or routed to a targeted human checkpoint before machine
  consumption. For non-sandbox use, sampled clean controls for those semantic
  classes must stay at >=90% pass rate after checkpoint routing.
- 0 false-clean outcomes: no seeded bad output may become machine-consumable.
- <=10% false-positive rate on clean control fixtures.
- 100% rollback completeness for every injected corruption with a rollback
  edge.

Any miss blocks non-sandbox use until that error class gets either a stronger
check or a targeted human checkpoint, then the fixture is rerun.

Audit and diagnostics:

- The journal is forensic truth; `request show` and `journal show` are
  user-facing summaries over it, not a second log.
- `journal list/show` supports request, path, decision, operation, date, and
  check filters.
- `doctor bundle` creates a redacted local bundle containing config shape, recent
  journal summaries, failed checks, projection freshness, provider status, and
  diagnostic stderr. It excludes secrets, raw provider payloads, raw Work text,
  and personal contact/payment identifiers.
- `doctor self-test` proves the capture/redaction path on a synthetic fixture.

## 10. Search, Gap Analysis, And Ask

The search/index layer is a rebuilt interface, not truth.

Baseline:

- qmd is the required local search backend. Memoria owns the checked-only input
  tree, read barrier, qmd collection registration, rebuild command, and result
  filtering; qmd owns keyword and semantic ranking.
- The qmd executable is resolved to an absolute path at install/doctor time
  because other packages can also ship a `qmd` binary.
- Memoria runs qmd with workspace-local config/index state:
  `QMD_CONFIG_DIR=<workspace>/.memoria/index/qmd/config` and
  `INDEX_PATH=<workspace>/.memoria/index/qmd/index.sqlite`. It must not set
  workspace-local `XDG_CACHE_HOME`, because qmd uses that for model files and
  per-workspace model downloads make runtime gates slow and brittle. The
  repo-local `package.json` qmd dependency is contributor tooling, not the
  runtime source of truth.
- `workspace rebuild --search --embeddings` rebuilds
  `.memoria/index/qmd/checked/` from checked current Concepts and acquired text,
  registers collection `memoria-checked` with mask `**/*.md`, runs `qmd update`,
  and, with `--embeddings`, runs `qmd embed --chunk-strategy auto`. CPU vector
  build is acceptable; a missing binary, wrong collection, update failure, or
  embed failure fails the command.
- qmd indexes acquired work text, digests, notes, hubs, compact citations, and
  graph-neighborhood projection documents.
- Hybrid retrieval combines qmd keyword/semantic results with the SQLite graph
  read model. SQLite remains graph/ops authority; qmd is never state authority
  and never writes keep-set files.
- First-order citation/reference graph, OpenAlex Topics, controlled facets, and
  discovered candidate Works are part of the read model.
- Query expansion over citations, topics, facets, and argument terms is baseline;
  LLM-generated expansion still has to beat qmd+graph retrieval before
  it counts.
- Retrieval hides superseded and retracted content by default; historical lookup
  is explicit. Supersession is a deterministic filter over the append-only claim
  trace, not just a dashboard, so the store does not silently surface outdated
  facts as it grows.

Gap analysis reads both:

- Work coverage.
- Knowledge graph coverage.
- Mismatch between work density and note/claim density.
- Full-text coverage.
- Citation/reference neighborhood coverage.
- Topic/facet coverage.
- Discovery candidates not yet captured, enriched, or digested.

Gap states:

- works thin / notes thin: early area, propose seed.
- works dense / notes thin: undigested corpus.
- works thin / notes dense: under-warranted claim cluster.
- works dense / notes dense: mature area.

With a thesis, gap analysis also reads argument health from the project's Toulmin
graph: a claim that is unsupported, uncountered, or fragile (thin or contested
warrant) is a gap even where work/note coverage looks dense.

Automated discovery is baseline but bounded: `project gaps` and scheduled
maintenance can propose candidate Works from OpenAlex citation/reference/topic
neighborhoods, full-text mentions, and controlled facets. Discovery writes
unchecked attention/worklist rows. It never silently imports a discovered Work as
checked knowledge and never bypasses the provider allowlist.

Ask answer contract:

- cite works;
- state unknowns;
- state stale/superseded evidence;
- state contradictions;
- state missing evidence.

A confident answer with no edges is a defect.

`memoria ask` answers over the checked workspace graph without requiring a
project id. `memoria project ask` adds project steering, thesis, decisions,
argument state, and output context. Both obey the same read barrier and answer
contract.

Supersession is tested as behavior, not principle: retrieval over a
superseded/current pair is measured by Temporal Coverage@k and Temporal
Precision@k (top-k spans both the baseline and the current window), and the
stale-as-current seeded class exercises it.

Explorations compound: a substantial Ask answer — a comparison, a synthesis, a
discovered connection — can be filed back as a note or hub Concept through the
normal write lifecycle, so it is not lost to chat history. Trivial lookups are not
filed.

Project export:

- `project export` renders checked project drafts/compositions from project
  Concepts, notes, hubs, argument state, and `references.bib`.
- Markdown export is baseline. Pandoc-backed docx/pdf/odt export is available
  only when Pandoc is installed; the command fails with a clear prerequisite
  error otherwise.
- Export is terminal and deterministic: users edit the source composition and
  re-export, not the generated artifact in place.

- Paywalled full-text bypass or credentialed publisher scraping.

Deferred-boundary posture:

- Export stays deterministic and plain: keep-set Markdown, `references.bib`,

  only when third-party capability import becomes real.

  What stays deferred is exhaustive/deep expansion beyond the local workspace's
  digest, retrieval, gap-analysis, and discovery needs.
- Local SQLite and the git CLI are the boring baseline. Turso/libSQL, pygit2,
  dulwich, and verifiable-log servers are considered only after a concrete
  failing baseline gate proves they are needed.

## 14. Pre-implementation Gates

Write small failing tests before implementation code:

- `memoria --help` imports without Obsidian, Zotero, Hermes, MCP, profiles, or
  GUI env vars.
- `memoria init` creates a usable workspace from scratch.
- `memoria init --dry-run` reports the workspace skeleton, DB, package baseline,
  generated targets, steering/vocabulary Concepts, qmd collection, and provider
  config without writing them.
- `doctor --check qmd` fails when Node is too old, qmd is missing, qmd resolves
  only through ambiguous `PATH`, qmd config/index state is outside the
  workspace, collection `memoria-checked` is missing, or the collection root/mask
  is not `.memoria/index/qmd/checked/` + `**/*.md`.
- `doctor --repair` restores a corrupted packaged capability/skeleton fixture
  only when explicitly requested; plain `doctor` reports without repair.
- DOI capture creates an unchecked SQLite work and no work markdown.
- DOI capture does not update `references.bib`.
- Enrichment required-provider success materializes `references.bib` before
  completion.
- Required-provider failure leaves work unchecked and creates attention.
- Full-text acquisition failure leaves digest blocked and creates attention; no
  checked digest is produced from abstract-only or metadata-only input.
- Acquired text is content-addressed, anchored, and replayable from fixtures.
- URL page/PDF capture and user-supplied PDF/text fixtures can produce checked
  digests only after acquired anchored full text exists.
- BibTeX/CSL imports create unchecked Work rows for DOI and ISBN/book examples
  without Zotero or legacy Work markdown; book digest remains blocked until full
  text or user-supplied equivalent exists.
- First-order citation/reference/topic graph import creates graph rows and
  unchecked discovery candidates without silently checking any discovered Work.
- qmd rebuild covers acquired work text, digests, notes, hubs, compact citations,
  and graph neighborhoods using collection `memoria-checked`; retrieval tests
  prove qmd+graph beats BM25-only on the fixture before it counts as product
  behavior.
- Provider endpoint enabled in config must be covered by operation
  `allowed_network`.
- Adapter HTTP bypassing shared allowlist fails closed.
- Provider payloads are content-addressed and deterministic remerge is stable.
- High-authority DOI/title conflict blocks checked promotion.
- Retraction/contested evidence blocks checked promotion.
- Keep-set work reference without compact `citations[]` fails validation.
- Crash after DB commit but before file materialization recovers from durable
  payload.
- Lookup, digest compilation, references, attention, rollback, and recovery work
  by `work_id` without a legacy markdown Work path.
- `work update` archives/restores a Work, applies a human classification
  correction, refreshes active views/retrieval filters, and resolves the linked
  attention item in the same request.
- `work interview` records guided answers and `work digest` consumes them without
  reading profile/Hermes memory.
- A keep-set edit imports as a PI edit whose derivation inputs keep the prior
  upstream plus the prior checked hash, so cascade rollback still reaches its
  origins; an edit to a generated projection, worker-owned path, `references.bib`,
  or capability manifest is quarantined, not consumed.
- A `note` validates its optional anchor and Toulmin claim block; a `quote` that
  does not resolve to its work span fails the quote-anchor check.
- `note accept`, `note reject`, and `note link` create curation journal rows;
  typed links validate `supports`/`contradicts`/`extends` targets.
- A claim re-cited into a decoy span is flagged by the counterfactual-faithfulness
  check on the frozen fixture.
- A citation targeting a same-operation staged row validates; write validity does
  not require `checked`-ness.
- Journal events are hash-chained and a git-tracked journal-head anchor is written
  at operation completion; a broken chain fails a check.
- A fired check routes act/ask/drop; no blanket per-write approval queue exists.
- Attention fixtures preserve finding/proposal/verification/work-prompt shapes,
  loudness, certainty, suggested action, and aggregate worklist behavior.
- Queue fixtures prove one mutating worker per workspace, bounded pending/blocked
  counts, and `request retry|cancel|amend` behavior.
- The project argument operations run through `operation run`, update project
  argument state, and render the Toulmin argument picture/canvas.
- The seeded-error harness measures the human checkpoint against a no-checkpoint
  baseline, and each check/retrieval/synthesis capability declares and beats a
  cheap baseline before it counts.
- `digest`/`hub` carry `confidence`/`contested`/`contradictions[]`, and lint

- A synthesis operation orients against the existing graph and updates an existing
  Concept instead of creating a duplicate for a concept already present.

  lifecycle; a trivial lookup is not.
- `tags` outside the schema-owned taxonomy fail validation (add-before-use).

  read barrier mechanics, cross-store recovery, and typed operation runner.

  small runner constraints to port into the alpha.14 engine.
- This alpha.14 design resolves their surface/runtime drift: Memoria is a
  standalone CLI + engine first. External tools are optional edges, not the core.
- Reference scans: Karpathy's LLM-wiki gist and the Hermes `research-llm-wiki`
  skill supply the compounding markdown-wiki pattern — ingest/query/lint
  operations, one Work touching many pages, index+log navigation,
  confidence/contested quality signals, orient-before-create, and filing query
  answers back so explorations compound. Razlogova's *Doing History with Zotero
  and Obsidian* supplies the Zettelkasten method: one Work's annotations
  distilled into several atomic notes that land in different parts of the
  argument, anchored back to the work span, linked into a graph. Together they
  turn a knowledge *base* into knowledge *production*; the Toulmin model gives the
  note/argument shape. The ~400-paper literature review behind the
  correctness-via-integrity bet lives in `_papers/` — `_papers/REVIEW-REFUTATIONS.md`
  (durable state, verbatim warrants, sparse review, deterministic grounding,
  supersession-as-precondition) and `_papers/REVIEW-SUMMARY.md` — with the
  ~47-system agent-research slice judged in the borrow/adapt/ignore provenance
  table. Its dominant result: independent research lines re-derive Memoria's bets
  — generator/verifier sample-and-rank (Cobbe, Perez), FEVER evidence-grounded NLI
  verification, TEMPO temporal coverage, mixed-initiative act/ask/drop (Horvitz,
  Bernstein), and injection-as-untrusted-code (Greshake, AgentDojo) confirming the
  least-privilege sandbox. alpha.14 borrows each mechanic and strips the
  scalar-optimization loop layered on top (co-trained reviewer, tournament,
  tree-search, confidence-routed bypass, preference-into-weights). The survey docs'
  pre-alpha.14 profile/lane/board/MCP framing is not adopted — only the
  architecture-agnostic findings are.
