# Implementation status

The design docs in `docs/` describe the **intended** Memoria system. The starter vault (under `vault/` in this repo) is a **v0.1 scaffold** — some components are shipped and present in the vault, and a significant portion of the v0.1 design is pending implementation. Nothing is approved yet. This ledger is the single source of truth for the build state of every artifact, so a gap reads as a known status — not a documentation error.

When a design doc describes an artifact, tag it with one of the status values below (inline, e.g. *(status: pending)*). When in doubt, this ledger wins.

## Status values

| Value | Meaning |
| --- | --- |
| **approved** | In `vault/`, tested, and verified to work end-to-end. |
| **shipped** | In `vault/`, but not yet verified end-to-end (dependencies missing or testing not done). |
| **pending** | Designed and required — but not yet built. A build gap; the requirement stands. |
| **deferred** | Consciously excluded from this release. A scope decision; absence is intentional. |

Authoritative runtime signal: `install.ps1` requires `SOUL.md` + `config.yaml` + `mcp.json` + `distribution.yaml` per profile. All four now ship for all seven profiles (the v0.2 wiring landed), so the installer no longer skips them — it will stage, substitute `{{VAULT_PATH}}`, and call `hermes profile install`. If the installer and a design doc disagree, the installer is right and the doc needs a status tag.

## How to use this ledger

- Adding or moving an artifact? Update its row here in the same change.
- Writing a design doc for something not yet built? Tag it *(status: pending)* and link this ledger.
- Writing a design doc for something intentionally out of scope? Tag it *(status: deferred)*.
- A `plugin-config-drift` / structural-detector pass can read this file to distinguish pending (build gap) from deferred (scope decision) from shipped (present, unverified) from approved (present, verified).
