# Implementation status

The design docs in `developer-guide/` describe the **intended** Memoria system. The starter vault (`memoria-vault/`) is an early **v0.1 scaffold** that implements part of it. This ledger is the single source of truth for *what is actually built today* versus *what is designed but not yet shipped*, so a gap reads as a known status — not a documentation error.

When a design doc describes an artifact, it should tag it with one of the status values below (inline, e.g. *(status: deferred — v0.2)*). When in doubt, this ledger wins.

## Status values

| Value | Meaning |
| --- | --- |
| **shipped** | Present and working in the starter vault today. |
| **accepted-pending** | An ADR accepted the design, but the implementation has not landed yet. The decision is final; the build is not. |
| **deferred** | Designed and documented, intentionally not built yet. Scheduled to a later version/phase or "adopt-on-demand." Absence is expected, not a bug. |

Authoritative runtime signal: `install.ps1` ships only `SOUL.md` per profile and treats `config.yaml`, `mcp.json`, `distribution.yaml` as the "v0.2 wiring." If the installer and a design doc disagree, the installer is right and the doc needs a status tag.

## How to use this ledger

- Adding or moving an artifact? Update its row here in the same change.
- Writing a design doc that describes something not yet built? Tag it *(status: deferred)* or *(status: accepted-pending — ADR-NN)* and link this ledger.
- A future `plugin-config-drift` / structural-detector pass can read this file to distinguish "missing because deferred" from "missing in error."
