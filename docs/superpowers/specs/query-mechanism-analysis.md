# Memoria Query Architecture: Lexical, Semantic, Structural, and Graph Retrieval Over a Unified Store

*Prerequisite analysis for the data-structure design in `data-structure-analysis.md` Part 8. Answers "what is the best way to query the data" — a separate question from, and an input to, "how should the data structure itself be designed to be queryable." Produced via multi-agent research + adversarial verification; kept as its own document per the project owner's direction rather than embedded inline.*

## 1. Requirements

The requirements below are restated from independent sourcing, not reverse-engineered from current code. Each is tagged with its origin.

- **Q0 [user-emphasized, central]** — Memoria is a knowledge-*production* tool, not a knowledge-*storage* tool. This is the north star every other requirement is subordinate to, and the design is judged first against it. Three binding facets: **(a) topic surfacing, not narrow Q&A** — "I am thinking about X" must return all relevant knowledge about X, with the full grounding context (claims, sources, relations) exposed alongside or instead of any single synthesized paragraph; **(b) universal claim/concept-level addressability** — every claim, question, tension, hypothesis, or concept must be independently retrievable with its own context, and tensions specifically are a first-class retrievable object, which requires checking whether the schema can represent a contradiction at all; **(c) idea production is the payoff, but the human ideates, for now** — the PI ideates from what the system surfaces; automated tension/gap/idea detection is explicitly out of scope for this pass (a beta.1 scope-discipline decision made by the project owner), and this pass's job is to build the foundation — tension recordable/surfaceable, evidence-thinness cheaply visible — that a later automated layer could use, not the automation itself. This scope line is supported two ways: (i) the project owner's explicit instruction, and (ii) confirmatory literature — [ClawArena](https://arxiv.org/abs/2604.04202)'s multi-source-plus-revision round scored 0% exact-match across all 8 model configurations tested (every atomic fact recalled correctly, but agents failed to aggregate/revise into the required compound claim), [ConCoRD](https://arxiv.org/abs/2211.11875)-style automated NLI contradiction resolution flips ~28–29% of beliefs incorrectly even with per-problem-tuned hyperparameters, and Memoria's own ADR-10 independently calls automatic supersession detection "the least reliably automatable memory capability" — cited as confirmatory context for a decision already made on its own terms, not as the justification itself.
- **Q13 [derived from literature]** — the retrievable unit defaults to the verbatim claim in its actual surrounding passage, not a decontextualized atomic fact. [LongMemEval (Wu et al., ICLR 2025)](https://arxiv.org/abs/2410.10813) supports preserving source context rather than relying only on extracted facts or summaries: in Table 3, round text indexed as itself has recall@5 0.582 vs. 0.530 when the same round value is keyed only by extracted facts, and session text indexed as itself has recall@5 0.706 vs. 0.572 for session summaries and 0.642 for session facts. This is a long-term chat-memory result, not a direct Memoria benchmark, so it is used only as evidence against fact-only storage, with Memoria's own fixture still deciding retrieval defaults (Q8).
- **Q1 [user-requested]** — one query combining relevance-ranked free text with structural graph constraints in a single pass (e.g., "open questions about X still uncited," "checked sources on Y not yet referenced by any note").
- **Q2 [user-requested]** — semantic/embedding retrieval is first-class, not deferred.
- **Q3 [user-requested, floor requirement]** — "QMD + LightRAG equivalent over the entire data structure (db+files)": hybrid lexical+vector retrieval (Q1/Q2) combined with a LightRAG-equivalent dual-level graph retrieval — local-level (entity/claim + immediate neighborhood) and global-level (theme/topic neighborhood) — fused with vector similarity, over both DB and files as one structure. The scoping distinction: adopt LightRAG's *retrieval method*, not its *construction pipeline* — Memoria's graph is already curated and checked, a better input than LLM auto-extraction, and building an auto-extraction pipeline would itself be the automation Q0(c) scoped out. Multi-hop traversal, co-citation, bibliographic coupling, centrality, and community detection remain in scope as concrete techniques feeding local/global retrieval. [MemoryAgentBench (Hu, Wang & McAuley)](https://arxiv.org/abs/2507.05257) found graph-structured memory systems (GraphRAG 0.4, Cognee 2.3, Zep 4.2 — not LightRAG specifically) scoring below plain BM25 (19.0) on its Long-Range Understanding synthesis task, on a different task (general agent memory over auto-extracted graphs) and different graph provenance than Memoria's curated one; this does not override the floor — it is exactly what Memoria's own fixture (Q8) must measure before/instead of trusting a third-party number.
- **Q4 [derived]** — markdown stays the human-editable, Obsidian-live-edited, OKF-export-compatible source of truth. No design collapses authorship into a DB-only representation.
- **Q5 [derived]** — the checked/verdict gate (concept_verdicts, check_status, quarantine) is inviolable; retrieval must never surface unchecked or quarantined content as checked.
- **Q6 [derived]** — single local user, no server process, no ops team.
- **Q7 [derived from corpus scale]** — hundreds to low thousands of documents/rows today; evaluate at this scale, not web scale.
- **Q8 [derived from ADR-125/ADR-129, reinforced by the project owner]** — any new retrieval substrate is benchmarked against a declared cheap baseline on a committed fixture before replacing the current one; any third-party benchmark number is reported as claimed-on-that-source's-fixture, never as directly transferable.
- **Q9 [explicit]** — no implementation privileged by inertia; SQLite/Python/current layout are candidates to keep or replace strictly on merit, in both directions.
- **Q10 [derived, refined by the project owner]** — offline/no-network is the *default preference* at retrieval time, chosen for speed (latency), not an absolute prohibition. A network call is admissible for a specific capability if the value it adds is weighed against, and justifies, the added latency — evaluated per capability against a stated bar, not banned outright. Where this document recommends a local-only mechanism (e.g., 4.9's reranking/synthesis), that is because the local option already clears the bar at Q7's scale, not because network calls are forbidden in principle.
- **Q11 [process rule, corrected]** — migration cost is not merely a tie-breaker here: at this stage there is nothing deployed to migrate (pre-1.0, no external vault depends on the current shape), so migration cost is currently zero, full stop. Where this document describes "migration phases," read those as build/sequencing steps for a fresh schema, not as data-preserving conversion of an existing production corpus — there is no compatibility shim, no phased rollout, no dual-write period to design around.
- **Q12 [required output]** — name the concrete condition that would flip the recommendation toward a different engine or a rewrite.

A methodological rule applies throughout: Memoria's own product history (what was tried and removed, what was never built) is never cited as evidence for or against a design choice, in either direction. Wherever Memoria's current implementation is mentioned at all — principally Section 6's gap/migration analysis, and 4.11's Q0(b)-required check of whether the present schema can represent a tension — it is bracketed explicitly as descriptive, non-evidentiary context: a statement of present fact used to size the gap to the target design, never a premise offered for or against any design choice in this document.

## 2. Prior Art

**Rank fusion.** Reciprocal Rank Fusion originates with [Cormack, Clarke & Buettcher, SIGIR 2009](https://doi.org/10.1145/1571941.1572114) ("Reciprocal Rank Fusion outperforms Condorcet and Individual Rank Learning Methods"). It is now the default hybrid-search fusion mechanism in production lexical+vector systems specifically *because* it never compares raw scores across retrievers: Elasticsearch's native [`rrf` retriever](https://www.elastic.co/docs/reference/elasticsearch/rest-apis/retrievers/rrf-retriever), OpenSearch's [`score-ranker-processor`](https://docs.opensearch.org/latest/search-plugins/search-pipelines/score-ranker-processor/) (2.19+), Weaviate's [`rankedFusion`](https://docs.weaviate.io/weaviate/concepts/search/hybrid-search), Qdrant's [`rrf` fusion method](https://qdrant.tech/documentation/search/hybrid-queries/), Azure AI Search's [default hybrid-query fusion](https://learn.microsoft.com/en-us/azure/search/hybrid-search-ranking), and Vespa's [`hybrid-rrf` rank profile](https://docs.vespa.ai/en/learn/tutorials/hybrid-search.html) all implement `score(d) = Σ 1/(k + rank(d))`, k≈60, over each ranked list.

**Score incomparability.** [OpenSearch's own docs](https://docs.opensearch.org/latest/search-plugins/search-pipelines/normalization-processor/) state BM25 and k-NN "use different scales to calculate relevance scores," so it's "beneficial to normalize them"; [Qdrant's docs](https://qdrant.tech/documentation/search/hybrid-queries/) note (verbatim) "a fixed alpha over raw scores tends to be dominated by whichever retriever has larger raw magnitudes on a given query"; [Azure's docs](https://learn.microsoft.com/en-us/azure/search/hybrid-search-ranking) tabulate BM25 (no upper limit) against cosine (0.333–1.00) to make the same point. Where systems do want to preserve score magnitude, they normalize explicitly rather than summing raw scores: OpenSearch's [min-max/L2/z-score normalization-processor](https://docs.opensearch.org/latest/search-plugins/search-pipelines/normalization-processor/), Weaviate's [`relativeScoreFusion`](https://docs.weaviate.io/weaviate/concepts/search/hybrid-search), Qdrant's [DBSF](https://qdrant.tech/documentation/search/hybrid-queries/) (mean ± 3σ), Vespa's [`atan`-based BM25 squashing](https://docs.vespa.ai/en/learn/tutorials/hybrid-search.html).

**Reranking.** [SBERT's Retrieve & Rerank pattern](https://sbert.net/examples/sentence_transformer/applications/retrieve_rerank/README.html), [Pinecone's](https://docs.pinecone.io/guides/search/rerank-results) and [Cohere's](https://docs.cohere.com/reference/rerank) rerank products all implement the same two-stage shape: cheap high-recall first-stage retrieval, expensive cross-encoder rescoring of only the top-N. [SBERT's own MS MARCO benchmark table](https://www.sbert.net/docs/pretrained-models/ce-msmarco.html) is the one hard, primary-source number found: `ms-marco-MiniLM-L6-v2` at 74.30 NDCG@10, with lighter models trading a few NDCG points for an order-of-magnitude throughput gain — a real, quantified precision/latency curve, not a vendor-blog percentage.

**Grounded generation.** [Lewis et al. (NeurIPS 2020)](https://arxiv.org/abs/2005.11401) established retrieve-then-generate. [Anthropic's Citations API](https://platform.claude.com/docs/en/build-with-claude/citations) returns response blocks with exact quoted source text and character-offset pointers "guaranteed to contain valid pointers to the provided documents." Google's Vertex AI [grounded generation](https://docs.cloud.google.com/generative-ai-app-builder/docs/grounded-gen) maps answer segments to specific grounding chunks and exposes a separate [Check Grounding API](https://docs.cloud.google.com/generative-ai-app-builder/docs/check-grounding) returning a 0–1 entailment/support score. [ALCE (Gao et al.)](https://arxiv.org/abs/2305.14627) formalizes citation-recall and citation-precision as automatic metrics; [Self-RAG (Asai et al.)](https://arxiv.org/abs/2310.11511) trains reflection tokens that critique whether generated segments are actually supported.

**Graph-augmented retrieval.** [Microsoft GraphRAG (arXiv:2404.16130)](https://arxiv.org/abs/2404.16130) solves corpus-wide sensemaking via LLM entity/relation extraction, Leiden community detection, and per-community summarization — evaluated at ~1M-token scale. Its indexing cost is documented, separately from the arXiv paper, in [Microsoft's own methods docs](https://microsoft.github.io/graphrag/index/methods/): graph extraction is roughly 75% of total indexing cost. **A specific $33,000/5GB-corpus figure previously cited here for that cost could not be re-verified against any Microsoft source** — it traces only to an uncited third-party post, so it's removed rather than left as an unsourced number. [LazyGraphRAG](https://www.microsoft.com/en-us/research/blog/lazygraphrag-setting-a-new-standard-for-quality-and-cost/) is Microsoft's own admission that this cost was overkill for most cases, reporting ~0.1% of the indexing cost and >700x lower query cost for global queries by deferring all LLM use to query time over an NLP-only (no-LLM) noun-phrase graph. [LightRAG (arXiv:2410.05779)](https://arxiv.org/abs/2410.05779) is the direct floor-requirement referent: dual-level retrieval (low-level specific entities/facts, high-level abstract themes) fused with vector embeddings, runnable fully embedded with file-based storage — but its graph *construction* still requires an LLM extraction step at indexing time, which is the one piece of it this design does not adopt. [HippoRAG (arXiv:2405.14831)](https://arxiv.org/abs/2405.14831) runs Personalized PageRank — a lightweight, LLM-free, in-memory graph algorithm — over an LLM-extracted graph for multi-hop QA, beating prior methods by up to 20% at 10–30x lower cost than iterative retrieval; the PPR step itself needs no LLM call. [Neo4j's `neo4j-graphrag-python`](https://neo4j.com/docs/neo4j-graphrag-python/current/user_guide_rag.html) separates a `HybridRetriever` (vector+full-text fusion, no graph traversal — its own headline example, a plot-text keyword-vs-vector case, is a lexical/semantic fusion win, not a graph win) from a `VectorCypherRetriever` (vector-seeded Cypher traversal, the actual graph-augmented pattern) — both require a live Neo4j server. [Graphiti/Zep (arXiv:2501.13956)](https://arxiv.org/abs/2501.13956) is a hosted, multi-tenant, bi-temporal knowledge-memory service backed by Neo4j, solving a genuinely different problem (many concurrent agent sessions) than a single local vault. ["RAG vs. GraphRAG" (arXiv:2502.11371)](https://arxiv.org/abs/2502.11371) finds graph structure earns its cost specifically on complex multi-hop synthesis questions; for localized factual questions, plain RAG is competitive and cheaper.

**PKM tools.** Across Obsidian core, Dataview/Datacore, Logseq (file-graph), Dendron, and Foam, none of the first-party/core tools cross from ranked-list retrieval into grounded-answer synthesis — that capability exists only in third-party AI plugins layered on top ([Obsidian Copilot's Vault QA](https://github.com/logancyang/obsidian-copilot), confirmed shipping RAG with [inline footnote citations](https://github.com/logancyang/obsidian-copilot/releases/tag/3.0.3); [Smart Chat](https://github.com/brianpetro/obsidian-smart-connections), retrieval-then-generation behavior confirmed, though its current docs don't use the literal word "RAG"). [Logseq's 2025–2026 split](https://github.com/logseq/logseq) into a markdown-source ["OG" version](https://github.com/logseq/og) and a SQLite-source-of-truth "DB" version is real prior art for the *opposite* choice (database becomes authoritative, markdown becomes an export artifact) — cited here as a documented cautionary contrast, not a precedent to follow, and not evidence about Memoria's own history.

## 3. Storage Substrate Decision

Compared against Q1 (native structural+relevance join), Q2 (first-class semantic), Q4 (must not force markdown out of authorship), Q6 (no server), Q7 (hundreds–low-thousands rows), Q9 (no incumbency privilege), Q10 (offline):

**SQLite + FTS5 + [sqlite-vec](https://github.com/asg017/sqlite-vec).** FTS5 is first-party, compiled into the SQLite amalgamation, and present in most standard Python 3.10+ `sqlite3` builds — but this is a build-time compile flag (`SQLITE_ENABLE_FTS5`), not a universal guarantee: the amalgamation supports omitting it, so a given Python distribution's bundled SQLite could in principle lack it. Confirmed present on this session's local `python3`, not confirmed as a property of every Python 3.10+ build everywhere; the design should verify FTS5 availability with a startup capability check (`PRAGMA compile_options` or a trial `CREATE VIRTUAL TABLE`), not assume it. sqlite-vec is third-party, pre-1.0 (stable v0.1.9, ANN still alpha), brute-force KNN only in its stable release — but at Q7's scale, brute-force over a few thousand vectors is sub-millisecond-to-low-millisecond, so the "no ANN yet" gap is not a real cost here. The core is public-domain, single-file, and its recursive-CTE support is a direct, native fit for an adjacency-list citation graph. This is the only candidate where lexical, vector, and relational/graph live in one file with zero additional services.

**DuckDB (+ [duckdb-fts](https://duckdb.org/docs/current/core_extensions/full_text_search) + core array-distance functions / [duckdb-vss](https://duckdb.org/docs/current/core_extensions/vss)).** Vector search is arguably *more* first-party than SQLite's: `array_cosine_distance` etc. are core functions needing no extension at all, and exact brute-force search at Q7 scale needs nothing beyond that. But `duckdb-fts`'s own docs state plainly that the FTS index "will not update automatically when the input table changes" — it requires a manual `PRAGMA create_fts_index` rebuild after every write. A vault edited incrementally in Obsidian generates continuous small writes; requiring an explicit reindex step after each is a real operational cost SQLite FTS5 does not have (FTS5's content-table+trigger pattern updates transparently). DuckDB is also a columnar/OLAP engine whose design center is bulk analytical reads, not the single-row insert/update pattern of a note vault. Recursive CTEs are supported, so the graph story is equivalent. DuckDB is the strongest competitor on the vector axis specifically, and loses on write-pattern fit and FTS-freshness.

**LanceDB.** Vector search is the native purpose of the project and its ergonomics are the best of the four — brute-force exact k-NN is explicitly the recommended small-data path, no index required. But it fails Q1 mechanically: [Lance/LanceDB's own docs](https://docs.lancedb.com/search/sql) state that filters, aggregations, and **joins across tables** require reaching for a bolted-on DuckDB-over-Lance extension or LanceDB Enterprise's DataFusion SQL endpoint — native LanceDB queries do not support joins, and a citation graph is fundamentally a join (or recursive traversal) between a nodes table and an edges table. It also fails the single-file property Q6 implicitly favors: a Lance dataset is a directory tree (`data/`, `_versions/`, `_deletions/`), not one file. Winning the vector axis while losing native graph/relational join support directly against Q1 rules it out as the primary substrate.

**Tantivy.** Not a database at all — a Rust FTS library. No vector search feature exists ([an open GitHub issue since April 2020](https://github.com/quickwit-oss/tantivy/issues/815), unresolved) and no relational/join layer exists (it is an index, not a store). It fails two of the three stated axes outright; adopting it would mean building the vector layer and the relational/graph layer yourself on top of it, strictly more integration work than SQLite+extensions.

**Decision: SQLite + FTS5 + sqlite-vec wins**, because it is the only candidate that natively covers all three retrieval modes — lexical (first-party FTS5), semantic (sqlite-vec, the one non-first-party piece, but sufficient at Q7 scale), and structural/graph (native recursive CTE and joins) — in one file, no server, no bolted-on second engine for any of the three axes. This does not privilege SQLite by incumbency (Q9): DuckDB's vector story is evaluated as *more* first-party and still loses on FTS freshness and write pattern; LanceDB's vector ergonomics are evaluated as the best of the four and still loses on the join axis that Q1 makes non-negotiable. Q4 is not satisfied *by* choosing SQLite — it is satisfied by the design in Section 4 keeping markdown files as the authoritative store and the database as a derived mirror, a decision independent of substrate choice and argued on its own terms below.

The specific condition under which this flips (elaborated with full triggers in Section 5): sqlite-vec's brute-force KNN is a real limitation, dormant only because Q7's row count keeps it dormant. It is not a limitation of the *decision method* — it is a scale-dependent fact that a larger corpus would surface on Memoria's own fixture, at which point LanceDB's native ANN or DuckDB's HNSW (if its persistence has left experimental status by then) become the honest re-comparison.

## 4. Query Design

### 4.1 Unified schema, one file

All of the following live in one `.sqlite` file:

- The existing structural/graph tables — `catalog_sources`, `work_graph_edges(work_id, relation_type, target_id, ...)`, `concepts(concept_id, concept_type, store)`, `concept_verdicts(concept_id, check_status)` — kept because Q1's join/traversal requirement is native to a relational table, independent of any claim about their current implementation.
- A new `passages` table: one row per retrievable claim-in-context. Schema sketch (illustrative, not asserting exact current column names — **`data-structure-analysis.md` Part 8 is the resolved schema; it renames this column `origin` with four values `note-body`/`source-span`/`work-aspect`/`db-mirror`, not the two-value `source_kind` sketched here**): `passage_id, concept_id, file_path, byte_start, byte_end, text, check_status, source_kind ('file'|'db-mirror'), indexed_mtime`. `text` is the verbatim passage — the paragraph/section the claim actually sits in — not an extracted fact string (Q13, Section 4.14). Rows come from two origins: (i) parsing real markdown files (notes/, hubs/, projects/, digests/) into passage-sized spans, and (ii) mirroring the synthetic markdown-shaped text already generated on the fly from `catalog_sources`/`work_graph_edges` rows, so DB-native claims (a source's description, an edge's relation) are equally addressable passages — this is required by Q0(b)'s universal addressability, not an incidental convenience. `check_status` is copied onto every passage row at index time from its owning concept's verdict, so the gate (Q5) travels with the retrievable unit itself rather than requiring a separate join at query time to enforce.
- An FTS5 virtual table over `passages.text`, using the content-table + trigger pattern so it stays synchronized on every insert/update/delete — the property DuckDB's FTS extension explicitly lacks.
- A `vec0` virtual table (sqlite-vec) with one row per `passage_id`, holding a locally-computed embedding (Q10: the embedding model runs locally at index time, never a required network call at query time).

*(Note: this table sketch is illustrative from the query-mechanism analysis. Whether it should be a genuinely new table, or generalize the real, already-existing `work_aspects`/`evidence_sets` tables, is a schema question resolved in `data-structure-analysis.md` Part 8, not here.)*

### 4.2 Query-time, mtime-gated lazy reindex — no daemon

Before executing a query, a cheap step compares each markdown file's on-disk mtime against a stored `indexed_mtime` in a small `file_index_state` table. Any file newer than its stored value is re-parsed into `passages` (+ FTS5 + embedding) synchronously, in-process, before the query runs. There is no background watcher process and no always-on daemon (Q6); this also avoids the "stale until reindex" failure mode documented directly in the PKM research — Obsidian's own Linux subfolder-watch gap, and Obsidian Copilot's Vault QA requiring an explicit manual "Force Reindex Vault" after out-of-process edits. At Q7 scale, reparsing the handful of files that actually changed since the last query is *expected* to be a low-single-digit-millisecond cost — an estimate, hardware- and corpus-shape-dependent, not a measured number; per Q8, this is exactly the kind of claim Section 6's committed fixture must confirm before the no-daemon design is trusted, not asserted as settled here.

### 4.3 The four retrieval modes, mechanically

- **Lexical**: `passages_fts MATCH ?` ranked by FTS5's built-in `bm25()`.
- **Semantic**: for an unconstrained "find similar passages" query, sqlite-vec's `vec0` virtual table's native KNN operator (`WHERE embedding MATCH :qvec AND k = :n`) is the right tool — a fast, whole-table top-k search. This is the baseline case; see 4.4 for why a *filtered* semantic query uses a different mechanism.
- **Structural**: plain SQL predicates over `work_graph_edges`/`concept_verdicts`/`catalog_sources` — e.g. Q1's "checked sources on Y not yet referenced by any note" is `catalog_sources JOIN concept_verdicts ... LEFT JOIN work_graph_edges e ON e.target_id = catalog_sources.source_id AND e.relation_type = 'references' WHERE e.target_id IS NULL AND concept_verdicts.check_status = 'checked'`.
- **Graph**: `WITH RECURSIVE` over `work_graph_edges` for bounded multi-hop traversal; self-joins on shared reference targets/sources for co-citation and bibliographic coupling; `GROUP BY target_id, COUNT(*)` for degree centrality. All plain SQL, no library (detailed in 4.8).

### 4.4 R1 mechanically: one statement, filter-before-rank, not fetch-then-discard

The mechanism that makes Q1's "single pass" real: structural predicates (check_status, relation_type existence/absence, edge joins) are applied in an *earlier* CTE than the ranking computation, and the ranking computation runs only over the rows that CTE already narrowed to — not as an application-code filter applied after fetching a ranked page. This distinction matters differently for the lexical and semantic legs of the fusion:

- **Lexical**: FTS5's `bm25()` is a scalar ranking function computable over any row set, so `passages_fts MATCH ?` can sit inside a CTE that is then joined against the structural predicates, with `bm25()` and `LIMIT` both evaluated last. This composes cleanly in one statement.
- **Semantic**: `vec0`'s native `MATCH ... AND k = N` operator is a whole-table top-k operation — it does not compose with an arbitrary `WHERE`/`JOIN` applied afterward without either over-fetching `k` and discarding non-matching rows after the fact (recreating exactly the problem this design eliminates elsewhere) or narrowing first. The correct mechanism for a *filtered* semantic query is therefore not the KNN operator but sqlite-vec's scalar distance functions (e.g. `vec_distance_cosine`/`vec_distance_L2` — exact function names to confirm against the installed sqlite-vec version at implementation time), computed as an ordinary `SELECT ... ORDER BY vec_distance_cosine(embedding, :qvec)` over the already structurally-filtered candidate set. At Q7 scale a full scan of the filtered subset is exact and cheap, so no index or approximation is needed for this path.

Because the structural filter is applied before either ranking computation runs, in both legs, the final `LIMIT` already reflects the true qualifying set; nothing fetched is later discarded because nothing was fetched before the predicates ran.

**Q5 rides through this by construction, not by a separate check**: every CTE in the fusion selects `check_status` alongside `passage_id`, and every query mode defaults to filtering to `check_status = 'checked'` for anything presented as ground truth. A PI can explicitly opt into an "include unchecked" view (useful for finding work still needing review), but unchecked/quarantined rows are never merged into a checked result set indistinguishably — they are either excluded by default or explicitly and visibly labeled when included on request.

### 4.5 Fusing incomparable rankings: RRF

Per the fusion-patterns findings (Section 2), summing FTS5's unbounded `bm25()` score directly against sqlite-vec's bounded cosine/L2 distance is exactly the pitfall OpenSearch's, Qdrant's, and Azure's own docs warn against — the retriever with larger raw magnitude would dominate regardless of true relevance. The fusion mechanism is Reciprocal Rank Fusion, computed over rank position, not raw score: `score(passage) = Σ_lists 1/(k + rank_in_list(passage))`, k=60 (the default cited by every production system surveyed), summed across the FTS5 rank list, the semantic rank list, and — per 4.6 — the local- and global-level graph-expansion rank lists. This sidesteps score comparability entirely, as intended.

### 4.6 Q3 floor: the LightRAG-equivalent dual-level method

Given a query, the base hybrid step (4.4/4.5) produces a fused set of seed passages/concepts. Two additional graph-expansion passes run from those seeds and are fused in as two more ranked lists:

- **Local level** — from each seed concept/passage, a bounded-depth (`WITH RECURSIVE`, depth ≤ 2–3) traversal over `work_graph_edges` (bibliographic) and `concept_edges` (`data-structure-analysis.md` Part 8 — a derived mirror of `links:` frontmatter plus PI-confirmed `tension` edges) pulls the immediate neighborhood: cited sources, directly related concepts, notes sharing a wiki-link. **Correction: an earlier draft of this section treated "notes sharing a wiki-link" as resolvable through `work_graph_edges`/"wiki-links" without naming how — it isn't, without `concept_edges`; `links:` frontmatter has no other SQL representation, and a query-time frontmatter parse would break Q1's one-statement requirement for exactly this edge type.** This is "specific entity/claim + immediate neighborhood," LightRAG's local level, expressed as a bounded CTE over both tables.
- **Global level** — passages/concepts sharing a `topic` or `keyword` relation_type edge are grouped into a theme cluster; a query matching one member of a theme surfaces the theme's other members. This substitutes Memoria's already-curated topic/keyword taxonomy for LightRAG's/GraphRAG's algorithmic community-detection step as the *default* global-level mechanism — a cheaper, already-checked input, not an equivalent-but-different construction.
- **Fusion** — both expansion passes are ranked lists (e.g., by traversal depth / shared-edge count) and folded into the same RRF sum as the lexical and vector lists in 4.5, so a passage reachable only through graph expansion still competes on equal footing, by rank, with a passage found by keyword or embedding match — this rank-fused ordering governs *presentation order within a result set*; whether that set is itself capped by a `LIMIT` or returned in full depends on which of the two query shapes in 4.7 is in play.

**The scoping distinction, stated explicitly**: this adopts LightRAG's *retrieval method* — dual-level local/global resolution fused with vector search — not its *construction pipeline*, which auto-extracts entities/relations from raw text via an LLM. Memoria's graph (`work_graph_edges`, `concepts`, human-authored wiki-links) is already curated and has already passed Q5's checked-gate; it is a higher-quality retrieval input than LLM auto-extraction would produce over the same text, not merely a cheaper substitute. This is precisely why adopting the method does not reopen the Q0(c) deferral: the method is a new way of *reading* structure that is already human-produced and already checked; it introduces no new automated content-generation or judgment step, and building an auto-extraction pipeline instead would itself be exactly the kind of automation Q0(c) scoped out for this pass.

### 4.7 Two PI-facing query shapes: this is how Q0(a) is actually met

Everything in 4.3–4.6 is machinery — lexical match, semantic match, structural predicate, graph expansion, RRF ordering. The PI-facing behavior built from that machinery has **two distinct shapes**, and conflating them is exactly how a design ends up satisfying Q1 while failing Q0(a):

**Shape 1 — targeted lookup** (Q1's own examples: "checked sources on Y not yet referenced by any note"). The question is narrow and finite *by construction* — an explicit structural predicate already bounds the answer set (a source either is or isn't referenced; a concept either does or doesn't have a given property). Here a ranked, `LIMIT`-bounded result list is the correct and sufficient shape, exactly as designed in 4.3–4.5: the predicate itself defines completeness, and ranking exists only to order or paginate a set that may still be large.

**Shape 2 — topic surfacing** (Q0(a)'s own example: "I am thinking about X"). This is explicitly *not* a request for a ranked top-k list — the requirement states a single synthesized paragraph "is not sufficient by itself." The design for this shape is:

1. Resolve X to one or more seed concepts/passages via the lexical+semantic hybrid (4.3–4.5) — this is a *seeding* step, not the final result.
2. From those seeds, run the full local+global expansion of 4.6 to closure: every passage/claim/source/concept within the bounded local-level neighborhood, plus every member of any global-level (topic/keyword) cluster the seeds belong to, plus every `relation_type='tension'` edge touching any node in that set (Section 4.11).
3. **Completeness is bounded by structural reachability and a relevance floor, not by an arbitrary result-count `LIMIT`** — bounded, not literally "all." The boundary of the returned set is: within traversal depth ≤ the configured bound (2–3 hops) *or* passing the FTS5/semantic match threshold — not "the top N by fused score," and not "every relevant fact in existence" either. **Precise contract, stated so it's testable rather than aspirational: "all indexed, checked items within the stated reachability bound and match threshold," with the bound and threshold reported alongside the result, not silently assumed.** A passage two hops beyond the configured depth, or below the match threshold, is real checked knowledge this query will not surface — a stated limit of the design, not a shortfall against an unbounded "all relevant knowledge" bar no query contract could actually test. RRF ranking (4.5–4.6) is applied to *order* this set for presentation — most-central/most-relevant first — never to cut it down to a page size.
4. The result is grouped and returned by kind (claims/passages, sources, concepts, tensions, gap-visibility flags for under-evidenced sub-areas of X per 4.12) rather than flattened into one undifferentiated ranked list, so the PI can see the shape of what is known about X, not just an ordered pile of hits. If the set is large, pagination within each group is a presentation concern; the underlying query does not truncate before the PI has seen the full set exists. **Each group states its own denominator** (e.g. "14 checked passages within 2 hops") so the PI can judge coverage rather than assume the count shown is exhaustive.
5. Any synthesized paragraph (4.9) is generated *from* this complete set and displayed alongside it, never in place of it — directly implementing Q0(a)'s "exposed alongside or instead of any synthesized text."

This is the concrete difference between "returns a ranked document list" (the bar Q0 states is not sufficient) and "surfaces all the relevant knowledge" (Q0(a)'s own words, restated here in testable terms): Shape 1 answers a bounded question with a ranked, capped list; Shape 2 answers an open topic with the complete connected neighborhood *within an explicitly reported reachability bound and match threshold*, ranked only for legibility, coverage always stated rather than implied.

### 4.8 Cheap SQL vs. real graph library — and what this does and doesn't gate

**Cheap, zero-new-dependency SQL** (ship now, part of the floor): co-citation (self-join `work_graph_edges` on shared `target_id` where `relation_type='references'`), bibliographic coupling (self-join on shared referenced target from two different `work_id` sources), degree centrality (`GROUP BY target_id, COUNT(*)`), bounded multi-hop traversal (`WITH RECURSIVE` with an explicit depth column and a `MAX depth` guard).

**Requires a real graph library**: community detection (Leiden/Louvain, as used by Microsoft GraphRAG) is not expressible as SQL and needs an actual library (e.g. `python-igraph` or `networkx`). Per Q11, this is gated on demonstrated need, not adopted speculatively: the global level ships now using the cheap topic/keyword-edge grouping (4.6), with algorithmic community detection as a later, optional upgrade to the global-level clustering step, adopted only if Section 6's fixture shows the curated-topic grouping under-performs. **This gate applies to the algorithm inside the global level, not to whether the dual-level method ships.** The method — local+global, fused with lexical+vector — is the Q3 floor and is **built** now regardless, per Q3's explicit floor-requirement status; **its activation as the default, replacing-the-baseline query path is still Q8-gated, not "ships regardless" in the sense of skipping the benchmark.** Concretely (detailed in Section 6, phase 3): the method is coded and run against the fixture before it becomes the default; if the fixture shows a regression against the Phase-2 hybrid baseline, the graph-expansion legs stay built but not defaulted-on — available for further tuning, not silently trusted as the PI's ground-truth result set — rather than shipping into the default path unconditionally. Community detection is one candidate future implementation of one sub-step within the method, phased separately per Q11's tie-breaking role, never used to demote the method itself to optional; the method's *build* is not optional, its *default activation* is Q8-gated.

### 4.9 Reranking and grounded-answer-synthesis: decided in scope now, not deferred

**Reranking: in scope now.** Q0 demands the PI not be confused by a poorly-ordered fused candidate set; a cross-encoder rerank is a precision pass with a real, measured tradeoff curve (SBERT's MS MARCO numbers). For **Shape 1** (targeted lookup, already `LIMIT`-bounded), the RRF-fused candidate set entering reranking is at most low-hundreds of passages; a CPU-only local cross-encoder over that size is *expected* to add on the order of tens of milliseconds and needs no server — an estimate dependent on the actual model and hardware chosen at implementation time, to be confirmed on Q8's fixture alongside the retrieval-quality benchmark, not asserted here as a measured fact. **Correction for Shape 2, where an earlier draft of this paragraph wrongly assumed the same bound applies:** Shape 2's own design (4.7) explicitly does not cap the candidate set at low-hundreds — a broad or well-connected topic's complete bounded neighborhood can be substantially larger, and Shape 2 must not truncate before the PI has seen the full set exists. Reranking a set of that size with a single cross-encoder pass would not clear the "tens of milliseconds" estimate and risks becoming exactly the kind of implicit truncation Shape 2 forbids if a size cap were quietly added to make it fast. The fix: **rerank per group, not the flattened whole** — each of 4.7's returned groups (claims/passages, sources, concepts, tensions) is reranked independently, each bounded by its own group's real size, not by an assumed global cap; if one group is itself large, rerank within a bounded window (e.g. its own top-N by RRF score) and state that window's denominator alongside the group's own (4.7 point 4) rather than silently reranking only part of it. If it clears Q10's speed-vs-value bar once measured, no network call is being foregone; if it doesn't, that is itself the trigger to reconsider, not a premise this document treats as already settled. Reranking's role stays strictly: improve presentation order — never to cut down what Shape 2 surfaces.

**Grounded-answer-synthesis: in scope now, narrowly.** Q0(a) itself anticipates synthesized text ("full grounding context ... exposed alongside or instead of any synthesized text") — it does not forbid it, it forbids it being *sufficient by itself*. Given this, and given Q7's scale is *expected* to make even a local-LLM synthesis call cheap relative to the query itself (a claim dependent on the local model and hardware chosen at implementation time, to be confirmed on Q8's fixture rather than asserted as already trivial here), a synthesis step is included, scoped strictly as: (i) always secondary — it never replaces the full retrieved+ranked context, which is always shown per 4.7's Shape 2; (ii) strictly extractive/entailment-bound — every generated sentence must carry at least one `passage_id` citation, following the Anthropic Citations / ALCE citation-precision-recall pattern, and sentences without a resolvable citation are rejected, not shown; (iii) check-status-aware — the synthesis input carries each passage's `check_status` (already riding along per 4.4), and the presentation must not let an unchecked-sourced sentence read as checked (Q5); (iv) not a required step of the base query path — the base retrieval loop (4.3–4.7) returns full results with zero LLM/network calls, satisfying Q10 unconditionally, and synthesis is a separate, PI-invoked enrichment layer on top of an already-complete result; (v) **for Shape 2, synthesis reads from a stated, bounded subset, not the whole set, with its own denominator** — the same "large candidate set" problem reranking has: a coherent paragraph cannot cite from thousands of passages, so synthesis draws from an explicitly labeled top-K (post-rerank, per group or overall) and states as much ("synthesized from the top 20 of 340 checked passages, ranked highest") rather than reading as if it drew on everything shown. This is a labeling requirement, not a truncation of the underlying result — the complete set from (i) is unaffected and still fully shown.

This is explicitly distinguished from Q0(c)'s deferred automation: a synthesis paragraph that restates and cites only what was already retrieved and already passed the checked-gate is a presentational compression of existing, human-authored material — categorically different from the system proposing a new tension, a new gap, or a new idea on its own initiative, which remains out of scope for this pass.

### 4.10 Q8 discipline, restated for this section specifically

Every number cited in this document — qmd's self-reported fixture scores (not directly examined here but referenced by the floor requirement's own framing), LightRAG's and HippoRAG's paper results, MemoryAgentBench's cross-system comparison, Zep's DMR/LongMemEval numbers, SBERT's MS MARCO NDCG table, Anthropic's "+15% recall" citations claim — is claimed by that source, on that source's fixture, unverified against Memoria's own corpus. None of them decide anything here. The only number that decides whether the LightRAG-equivalent dual-level layer (or any heavier algorithm phased into it, e.g. community detection) earns its keep is Memoria's own committed precision/recall/MRR fixture (Section 6), run after the method is built, measured against the plain lexical+vector hybrid baseline. This is what the fixture is for — not grounds to hedge the Q3 floor before it exists.

### 4.11 Q0(b): can the schema represent a tension today?

No. `work_graph_edges.relation_type` is a fixed, CHECK-constrained enum: `references, related, topic, keyword, authorship, institution, source` — the real, current, as-built values (`data-structure-analysis.md` Part 8 renames `source` to `published_in`; read `source` here as today's fact being gap-measured, not the target name). None of these carry contradiction semantics — `related` is the closest and is generic/associative, not adversarial; none of the others fit at all. **A new enum value is required** — `relation_type = 'tension'` — as an edge between two concept rows representing "these two claims are in unresolved tension." **Correction: this value does not land in `work_graph_edges`, an earlier draft of this section notwithstanding.** Every existing `work_graph_edges` row is bibliographic — `work_id`/`target_id` are catalog-work/OpenAlex identifiers, and the non-key columns (`target_title`, `target_doi`, `source_provider`, `raw_json`) are OpenAlex-provenance-shaped. A tension is ordinarily between two `mode: claim` notes, whose id is a `concept_id`, not a catalog `work_id` — a different id space, with no use for any of those provenance columns. Storing a `concept_id` there would work mechanically (the columns are unconstrained `TEXT`) but conflates two identity spaces in one table for no reason once a better home exists: `relation_type='tension'` instead goes into a new `concept_edges` table (`data-structure-analysis.md` Part 8), whose `concept_id`/`target_concept_id` columns already match what a tension edge connects, with no unused provenance columns and no namespace widening to document as a wart.

`concept_edges` has no per-edge `check_status` column, and `concept_verdicts` is concept-keyed, not edge-keyed — so a tension edge cannot reuse the existing per-row verdict gate directly. The clean resolution consistent with Q5 and with the existing propose/PI-decides pattern: **proposed tensions never touch `concept_edges`.** They remain exactly where they are today — journaled to the PI-review inbox. Only once the PI confirms a candidate does a `concept_edges` row get written with `relation_type='tension'`. Edge *existence* is therefore the confirmation signal — there is no unconfirmed/provisional row in the structural graph at any point, which is a stricter, simpler satisfaction of Q5 than adding a status column would be. **A write-path invariant this correction also removes, not just relocates:** because tension never lands in `work_graph_edges`, `replace_work_graph_edges` (`state.py:1003–1029`, which deletes every row for a `work_id` before reinserting on each OpenAlex re-enrichment) needs no guard against deleting a confirmed tension — there is no tension row in that table for it to ever touch. Under the original, now-corrected `work_graph_edges` placement, that guard would have been a required, easy-to-miss write-path change; relocating the value to `concept_edges` removes the risk instead of merely documenting it.

This directly closes Memoria's own described `surface_tensions`/`_compare_claims` gap (referenced here purely as the current-state fact the design must close, not as justification for the design): today it journals candidate findings to a PI-review inbox but, by its own docstring, never writes a persisted contradiction link — so a PI-confirmed tension has no queryable representation afterward. With `relation_type='tension'` added to `concept_edges`, confirming a journaled finding through the existing propose/PI-decides inbox pattern writes that row instead of merely closing a journal entry. Once written, that edge participates in local-level graph expansion (4.6) alongside `work_graph_edges` and `concept_edges`'s `links:`-mirror rows: any query touching either concept surfaces the linked contradicting claim as a first-class neighbor, which is precisely Q0(b)'s "tensions are a first-class retrievable object."

The same current-state description notes that the candidate-pair-finding step in `_compare_claims` scans rather than retrieves — it does not use BM25 or embeddings to find candidate pairs today. Using the new hybrid retrieval to feed that candidate-finding step better (e.g., proposing topically-close claim pairs via vector similarity) is **explicitly out of scope for this pass and not something this design builds now.** It is named here only as a possible, separately-scoped future extension, because it would mean extending the automated proposer itself — precisely the layer Q0(c) defers — not merely adding a recordability or surfacing primitive. What this pass builds and stops at is: the `relation_type='tension'` primitive above, so a tension has somewhere to live once confirmed; and the surfacing primitives in 4.12–4.13. Whether, or how, the existing candidate-finding step is ever improved is a decision for that later, separately-scoped automation pass, not this one.

**Questions and hypotheses** (also named in Q0(b)) reach baseline *addressability* with no schema change: a question or hypothesis a PI writes in a note is prose, and any prose span is already a `passages` row (4.1, 4.14) — retrievable by lexical/semantic match like any other claim, with no new type needed. **Correction: addressability is not the same bar as Q1's own example, and an earlier draft of this paragraph conflated them.** Q1's stated example is "open questions about X still uncited" — a *structural* filter (is this a question, is it still open, does it lack citations), not merely a lexical/semantic match on "X." `passages` as designed (4.1) carries `check_status` per row but not `mode`/`question_status` — a note's frontmatter fields — so answering Q1's example as one statement needs those two fields available as structured, queryable columns, not just retrievable prose. The fix, sized in `data-structure-analysis.md` Part 8: mirror `mode` (and `question_status`, when `mode='question'`) onto `passages` for `note-body`-origin rows, refreshed at the same reindex pass that already keeps `check_status` current (§4.2) — the same shape as that existing column, not a new mechanism. Without this, Q1's own example is not actually a one-statement query, only a lexical-match approximation of one. Separately, and smaller: treating a hypothesis as a first-class structural *node* that can be graph-linked to supporting/opposing evidence the way a concept can (analogous to the tension-edge gap above) is not covered even with the `mode`/`question_status` columns above — flagged as an identified, not-required-now extension, should the PI later want hypotheses to be graph-traversable nodes rather than only full-text-retrievable, structurally-filterable prose.

### 4.12 Gap-visibility primitive (evidence-thinness, not detection)

A plain `COUNT`/`GROUP BY` query, not an algorithm:

```sql
SELECT c.concept_id, c.concept_type,
       COUNT(DISTINCT p.passage_id) AS checked_evidence_count
FROM concepts c
LEFT JOIN work_graph_edges e
       ON e.target_id = c.concept_id
      AND e.relation_type IN ('references', 'topic', 'related')
LEFT JOIN passages p
       ON p.concept_id = e.work_id
      AND p.check_status = 'checked'
GROUP BY c.concept_id
ORDER BY checked_evidence_count ASC;
```

(Column names illustrative — see the schema design document for the real join shape.) This surfaces which concepts/topics have little or no checked evidence attached; it counts and sorts, it does not infer, threshold, or flag anything algorithmically. The PI reads the ordered list and judges thinness themselves — a retrieval primitive, exactly as scoped by Q0(c).

### 4.13 Cross-topic juxtaposition primitive

A query the PI invokes, not a background scan: find concept pairs that share a `topic`/`keyword` edge (topically adjacent, using the same global-level grouping as 4.6) but are *not* connected within a bounded traversal depth in the citation/reference graph (structurally distant) — a self-join on shared topic-edge target, anti-joined against a bounded-depth reachability set computed via the same `WITH RECURSIVE` pattern as 4.8. This surfaces candidate pairs of the form "about the same thing, but not already linked to each other" for the PI's own judgment — the same shape as `_compare_claims`'s candidate-pair scan, expressed as an on-demand queryable primitive instead of a background process, and, like 4.12, stops at surfacing the pair — it proposes nothing about what relation (if any) should follow.

### 4.14 Q13: the retrieval unit is the passage, not the fact

`passages.text` stores the verbatim passage a claim actually appears in — the paragraph or heading-delimited section from the source markdown file, addressed by `file_path` + byte offsets, not a paraphrased or extracted fact string. This is what gets indexed into FTS5 and embedded into `vec0`. A claim's identity (its `concept_id`) is linked many-to-many with the passage(s) that evidence it, so a claim is simultaneously retrievable as itself (concept-level addressability, Q0(b)) and in its exact original grounding context (passage-level verbatim text, Q13). This directly implements the conservative lesson from LongMemEval: extracted facts/summaries can be useful keys, but they must not replace the original text unit.

## 5. Trade-offs and the Flip Condition

**What this design gives up:**
- No ANN index — brute-force KNN/scalar-distance only, acceptable at Q7 scale, a real cost if the corpus grows substantially.
- Community detection deferred by default — global-level clustering runs on curated topic/keyword edges, which may under-cluster if that taxonomy is sparse or inconsistent, until measured need justifies the library.
- Single-writer concurrency (SQLite's model) — a non-issue at Q6's single-user scope, a real constraint the moment a second concurrent writer exists.
- Synthesis correctness depends on citation/entailment faithfulness, not provable correctness — mitigated (mandatory per-sentence citation, check-status-aware presentation) but not eliminated.
- sqlite-vec is the one non-first-party, pre-1.0 piece of the stack; its on-disk format and function surface could change across versions. Mitigated by embeddings being a cheaply regenerable derived cache from `passages.text` (source of truth), never a unique data store.

**Flip condition (Q12), concrete triggers:**
1. Corpus grows roughly 100x (low-thousands → ~100k–1M+ passages) *and* the committed fixture measures brute-force/scalar-scan latency exceeding an interactive threshold (e.g. >200ms/query) — re-run the substrate comparison against LanceDB's native ANN or DuckDB's HNSW (contingent on its persistence graduating out of documented-experimental status).
2. A second concurrent writer/user appears — SQLite's single-writer transactional model becomes a real constraint, triggering evaluation of a client-server engine (e.g. Postgres+pgvector) — a change to Q6 itself, not a tuning decision within it.
3. An always-on server process becomes acceptable (e.g. a sync service, a companion client) — this removes Q6's no-server constraint and makes Neo4j-class graph databases or hosted vector stores newly admissible on their own merits, not before.
4. A capability need arises that no embedded engine offers today — e.g. transactionally durable persistent ANN with crash-safe recovery (DuckDB VSS's HNSW persistence is explicitly experimental with documented data-loss risk on crash; sqlite-vec's ANN line is alpha) — triggered by a measured correctness incident traceable to this gap, not a hypothetical one.

None of these conditions currently hold at Q7's stated scale; the recommendation is not a default, it is the earned outcome of Section 3's comparison, and it is falsifiable by the above.

## 6. Gap to Current Implementation and Migration Sketch

*(This section, together with 4.11's schema-capability check, is where Memoria's current implementation is referenced — strictly for gap-measurement, never as justification for any design choice above.)*

- **R1 gap**: the current `propose_project_slice` over-fetches `k = max(limit*4, limit)` BM25 results and discards non-matching rows in application code, because relevance-ranking and structural filtering cannot currently be expressed in one query. Section 4.4's design removes this by construction — structural predicates are applied in an earlier CTE than either ranking computation, so `LIMIT` already reflects the final filtered set.
- **No embeddings today**: the current search is a hand-rolled deterministic BM25 in pure Python over a corpus built from real markdown plus synthetic markdown-shaped rows generated on the fly from `catalog_sources`/`work_graph_edges`. Adding the `passages`/FTS5/`vec0` schema (4.1) plus an initial indexing pass computing embeddings for whatever corpus already exists is a build step, not a data-preserving migration — per the corrected Q11, there is no deployed vault depending on the current shape, so this is sequencing only, not a cost that trades off against the target design.
- **No reranking or synthesis today**: current queries return ranked document/row lists only. Section 4.9's decision adds both, phased after the hybrid retrieval is stable (below).
- **`surface_tensions`/`_compare_claims` today**: finds real tensions via a lexical-overlap heuristic plus an LLM NLI-style classifier, journals candidates to a PI-review inbox, but (by its own docstring) never writes a persisted contradiction link — so a PI-confirmed tension today has no queryable representation afterward. Section 4.11 closes this one gap: add `relation_type='tension'`, written only on PI confirmation (never as a provisional row). Improving the candidate-pair-finding step itself (e.g., with vector similarity) is explicitly not part of this migration — per 4.11, that would extend the automated proposer, which stays out of scope for this pass and is named only as a possible later, separately-scoped extension.
- **Schema migration mechanic**: `work_graph_edges.relation_type`'s CHECK constraint cannot be altered in place in SQLite (no `ALTER TABLE ... ADD CHECK`); the `source`→`published_in` rename requires the standard SQLite table-rebuild pattern (create new table with the edited CHECK list, copy rows, drop old, rename) inside a single transaction. `tension` no longer triggers this rebuild at all — it lands in the new `concept_edges` table (a fresh `CREATE TABLE`, nothing to migrate), per the correction in 4.11.
- **Checked-gate mechanism**: `trusted_writer.py`'s `mark_checked`/`_write_checked` and `integrity.py`'s cascade/quarantine writers are purely structural today (they verify citations trace and the schema is valid, not claim truth — that judgment stays with the PI). **Correction: this does not extend unchanged to `passages.check_status` — that claim understated real, new work.** `passages.check_status` is a value copied at index time, not a live join; without an explicit same-transaction cascade write from these existing writers into every `passages` row sharing the changed concept's `concept_id`/`work_id`, a later demotion/quarantine/recheck leaves stale, wrong `check_status` values in `passages` until an unrelated file edit happens to trigger a reparse — which may never happen, since a verdict change touches no file and so trips no mtime gate (Section 4.2). The fix (detailed in `data-structure-analysis.md` Part 8's `passages` design): every write path that changes a check_status value must also update `passages` in the same transaction. This is new schema-adjacent work this design requires, not a restatement of an existing mechanism. Tension edges deliberately do not need this extension (per 4.11, their gating is existence-based, not status-based).

**Migration phases** (ordering may be influenced by cost per Q11; the target shape is not). **Correction: the check_status cascade (line above) was described in prose but never scheduled as an actual phase in an earlier draft of this list — a real gap, since Q5 is inviolable, not optional, closed here rather than left implicit:**

1. Add `passages` + FTS5 (content-table/trigger pattern); backfill from the existing real+synthetic markdown corpus. **In the same phase, not a later one:** wire `trusted_writer.py`'s `mark_checked`/`_write_checked` and `integrity.py`'s cascade/quarantine writers to also `UPDATE passages SET check_status = ?` for every row sharing the changed concept's `concept_id`/`work_id`, in the same transaction as the verdict write — Q5's gate must hold from the moment `passages` exists, not be retrofitted after other phases have already shipped queries that trust `passages.check_status`. Benchmark lexical-only retrieval on the new schema against the current hand-rolled BM25 on the committed fixture (sanity check before adding anything else).
2. Add `vec0`, a local embedding model, one-time backfill; add RRF fusion of FTS5+semantic (4.5), including the filter-before-rank scalar-distance path (4.4) for structurally-constrained semantic queries. Benchmark hybrid vs. lexical-only baseline on the fixture (Q8).
3. Add `concept_edges` (`data-structure-analysis.md` Part 8) — a derived mirror of `links:` frontmatter (`supports`/`contradicts`/`extends`), synced in the same query-time reindex pass as `passages` — closing the gap named below (§4.6's "notes sharing a wiki-link" hop) before local-level expansion is built, not after.
