## LLM-application CI/CD and eval gating

### Scout summary

2024-2026 practice broadly validates this pipeline's architecture — deterministic CI gates, a cheap-model mechanism tier, frozen eval suites, and an explicit human go/no-go map directly onto documented patterns (promptfoo/Braintrust/LangSmith eval-gated PRs, Anthropic's capability-vs-regression suites, Azure GenAIOps promotion gates). But practice disagrees on four points. (1) Live cheap LLM in exact-state mechanism tests: the consensus wiring-test pattern is record/replay (VCR cassettes) or mocks — pinned outputs, zero flake, zero cost — because a live model, however cheap, makes exact-state assertions nondeterministic while telling you nothing about the production model's quality; cheap-model results do not transfer as a quality signal, so a separate frozen eval suite must run against the production-tier model before promotion. (2) 'Only two human gates' conflicts with the strongest current evals doctrine (Hamel Husain/Shreya Shankar, Anthropic): error analysis and transcript reading are recurring human activities, and any LLM judge needs initial calibration against ~100+ human labels plus ongoing agreement monitoring — a standing maintenance loop, not a one-time gate. (3) LLM-as-judge cannot be an unattended promotion gate: Thoughtworks downgraded it Trial→Assess (Nov 2025) over position/verbosity/self-enhancement bias and preference leakage; peer-reviewed work shows rankings flip by reordering candidates. Deterministic validators are unanimously preferred wherever a property is objectively checkable; judges are reserved for subjective quality, calibrated, position-balanced, and never the same model that generated the output. (4) Because the production vault is recoverable but not reconstructible, practice's shadow/champion-challenger pattern translates to a mandatory dry-run of every candidate against a snapshot/replay of the real corpus before go/no-go — the single-user analog of shadow traffic. One internal disagreement in practice: Hamel argues for binary pass/fail judges while Anthropic's docs show Likert-scale LLM grading; the binary camp currently has the stronger empirical argument (labeling consistency). Caveat: OpenAI's hosted Evals platform is being deprecated (read-only Oct 2026), so gate on portable OSS tooling.

### Findings

- **Eval-gated CI quality gates (fail the build on eval thresholds)** — Run the eval suite in CI on every prompt/model change and fail the pipeline on assertion failures or when pass rate drops below a threshold (e.g. >=95%); cache LLM calls (PROMPTFOO_CACHE_PATH keyed on file hashes) so reruns are cheap, and track token cost over time. This is exactly the 'deterministic gate before promotion' the pipeline uses, extended to eval assertions. Trade-off: threshold gates without baseline diffs can miss which specific cases regressed. *Source: promptfoo docs — CI/CD integration — <https://www.promptfoo.dev/docs/integrations/ci-cd/>*
- **Immutable experiment snapshots + baseline regression comparison on every PR** — Promote each eval run to an immutable experiment snapshot; CI compares every PR's experiment against the current baseline and surfaces per-case improvements/regressions before merge. Scorers are code-based (reproducible) or LLM-judge. This is the concrete form of 'frozen eval suite + regression baseline + explicit promotion decision'. *Source: Braintrust docs — Evals guide — <https://www.braintrust.dev/docs/guides/evals>*
- **Evals as pytest tests with disk-cached LLM calls** — @pytest.mark.langsmith turns eval cases into pytest tests synced to datasets; each CI run creates a tracked experiment; LANGSMITH_TEST_CACHE caches HTTP requests to disk so repeated CI runs cost nothing. Fits an agent-built repo well: eval gating lives in the same pytest harness the deterministic gates already use. Recommended cadence: every PR or nightly, not every save. *Source: LangSmith (LangChain) docs — How to run evaluations with pytest — <https://docs.langchain.com/langsmith/pytest>*
- **Capability evals vs regression evals, with graduation** — Maintain two parallel suites: capability evals (start at low pass rates, measure headroom) and regression evals (held at ~100% pass to catch backsliding); saturated capability evals 'graduate' into the regression suite. Prefer deterministic graders where possible, but grade OUTCOMES not intermediate steps — step-level exact assertions are 'overly brittle' because models find valid paths designers didn't anticipate. Direct caution for the 'exact vault state after each command' design: safe for deterministic CLI mechanics, brittle wherever LLM-generated content lands in the vault. Also prescribes humans regularly reading eval transcripts to verify grader fairness — a recurring activity beyond two gates. *Source: Anthropic Engineering — Demystifying evals for AI agents (Jan 2026) — <https://www.anthropic.com/engineering/demystifying-evals-for-ai-agents>*
- **Grader hierarchy: code-graded first, LLM-graded for subjective criteria, different judge model** — Design task-specific evals, automate grading, and 'prioritize volume over quality' (many automated checks beat few hand-graded ones — this supports the unattended-agent pipeline). Grading ladder: exact match / code-graded / similarity metrics first; LLM-based grading only for subjective qualities; and 'use a different model to evaluate than the model used to generate' the output. Note: shows a Likert-scale LLM grader example, which the Husain/Shankar school explicitly argues against. *Source: Anthropic platform docs — Create strong empirical evaluations — <https://platform.claude.com/docs/en/test-and-evaluate/develop-tests>*
- **OpenAI evals: eval = data_source_config + testing_criteria graders; prompt-regression detection** — Evals are declared as a data schema plus graders (string_check for deterministic comparison, model graders for judgment); OpenAI calls evals essential when upgrading models or changing prompts, and links cookbook recipes for detecting prompt regressions and bulk prompt/model experimentation. Caveat: OpenAI's hosted Evals platform is deprecated — read-only Oct 31 2026, shutdown Nov 30 2026 — so encode the pattern in portable tooling (promptfoo/pytest), not vendor platforms. *Source: OpenAI API docs — Evals guide — <https://developers.openai.com/api/docs/guides/evals>*
- **Three levels of evaluation (assertions → human/model eval → A/B)** — Level 1: fast, cheap deterministic assertions run in CI on every change (also reusable for runtime guards/retries); Level 2: human review of logged traces plus LLM critique, with judge-human agreement built by repeatedly grading 25-50 examples in a spreadsheet and monitored continuously for drift; Level 3: A/B testing (N/A for single-user). Key disagreement with the pipeline: 'you must continue doing periodic exercises to monitor model and human agreement' — human involvement is standing, not gate-shaped. *Source: Hamel Husain — Your AI Product Needs Evals (2024) — <https://hamel.dev/blog/posts/evals/>*
- **Judge calibration protocol: TPR/TNR on held-out human labels, binary pass/fail, error analysis as the core human activity** — LLM-judges require 100+ human-labeled examples up front and ongoing weekly maintenance, validated by TPR/TNR on a held-out labeled set; binary pass/fail beats Likert scales (subjective point boundaries, inconsistent labeling); error analysis via open coding of real traces is 'the most important activity in evals' and must precede infrastructure. For CI specifically: small curated datasets favoring 'assertions or other deterministic checks over LLM-as-judge evaluators'. Their 'benevolent dictator' domain-expert model actually fits a solo researcher perfectly — but as a recurring role, which is where practice disagrees with 'only two human gates'. *Source: Hamel Husain & Shreya Shankar — LLM Evals FAQ — <https://hamel.dev/blog/posts/evals-faq/>*
- **Tiered pass thresholds by criticality** — Eval-based CI tests are regression tests for non-deterministic systems, not unit tests: dataset + experiment runner + programmatic evaluators, asserting a minimum accuracy threshold that varies by criticality (e.g. 0.95 for critical flows, 0.70 for experimental ones) rather than exact output matching. Useful refinement for the go/no-go gate: not every capability needs the same bar. *Source: Langfuse blog — LLM Testing: A Practical Guide (Oct 2025) — <https://langfuse.com/blog/2025-10-21-testing-llm-applications>*
- **Progressive exposure with qualitative tests in release pipelines (GenAIOps)** — Because AI output is nondeterministic, release pipelines must 'incorporate qualitative tests' alongside code gates; integrate models into production via progressive exposure and side-by-side deployments (canary, blue-green); monitoring feeds quality gates; version prompts/models as deployable artifacts within existing DevOps pipelines. For a single-user local-first product there is no traffic to split — the honest translation is a staged dry-run against a production-vault snapshot before cutover, plus tested rollback (backups), which the 'recoverable but not reconstructible' vault makes mandatory. *Source: Microsoft Azure Well-Architected Framework — MLOps and GenAIOps for AI workloads — <https://learn.microsoft.com/en-us/azure/well-architected/ai/mlops-genaiops>*
- **Champion/challenger with shadow replay before promotion** — A challenger model/pipeline shadows the deployed champion by replaying the same production requests; its outputs are logged and compared (never served) and promotion happens only when the challenger consistently outperforms, under an approval workflow with audit trail. Single-user analog: before the pre-production go/no-go, run the candidate build against a copy/replay of the real 1200-paper corpus and diff its outputs against the incumbent's — the strongest available evidence for the human gate, given the ephemeral test-vault seed cannot represent production-scale data. *Source: DataRobot MLOps docs — Challengers — <https://docs.datarobot.com/en/docs/workbench/nxt-console/nxt-mitigation/nxt-challengers.html>*
- **Model tiering in tests: small models for structural verification, capped tokens, selective CI** — Official framework guidance endorsing the cheap-model tier: use smaller/cheaper models for structural verification, cap maxTokens, test one behavior per test (not end-to-end chains), run live-call integration tests selectively rather than on every commit, and use VCR cassettes for repeated runs. Real API calls are reserved for validating credentials, latency, and wiring — explicitly NOT for quality judgment. This is the closest published statement of the pipeline's ephemeral-tier design, with one amendment: prefer recorded/replayed calls over live cheap-model calls once wiring is proven. *Source: LangChain docs — Integration testing — <https://docs.langchain.com/oss/python/langchain/test/integration-testing>*
- **Record/replay (VCR cassettes) for deterministic LLM tests in CI** — Record real LLM HTTP interactions once, commit cassettes, replay deterministically: tests fail only when the input sent to the LLM changes, making them prompt-regression detectors that isolate non-LLM bugs at zero cost and zero flake (redact API keys from cassettes). This directly challenges using a LIVE cheap/local LLM in exact-vault-state mechanism tests: exact-state assertions demand pinned outputs, and a live cheap model is both nondeterministic (flaky asserts or temperature-0 false confidence) and unrepresentative of the production model (validates wiring only, never quality). Trade-offs: cassettes rot when prompts change (re-record deliberately, treating diffs as review events) and never catch provider-side drift — which is the quality-eval tier's job, run against the production model. For agents, decision-level replay (record the model's tool-choice decisions, really execute the tools — e.g. sixty-north/langchain-replay) avoids HTTP cassettes' 'tool code never actually runs' blind spot, which matters when the assertion target is real vault state. *Source: Anay Nayak — Eliminating Flaky Tests: Using VCR tests for LLMs — <https://anaynayak.medium.com/eliminating-flaky-tests-using-vcr-tests-for-llms-a3feabf90bc5>*
- **LLM cascade / cost-tiered model routing (FrugalGPT)** — The academic grounding for model tiering: route queries through cheap models first, escalating to expensive ones only when needed — matching GPT-4 performance with up to 98% cost reduction. Caveat for the pipeline: this is a production-serving pattern with a learned router; in TESTING, the transferable lesson is only the cost structure. A cheap model passing mechanism tests is not evidence the production model will behave the same — quality claims require evals on the production-tier model. *Source: Chen, Zaharia & Zou (2023), arXiv:2305.05176 — <https://arxiv.org/abs/2305.05176>*
- **LLM-judge biases: position, verbosity, self-enhancement (MT-Bench)** — The foundational judge paper: LLM judges exhibit position bias, verbosity bias, self-enhancement bias, and limited reasoning ability; with mitigations, strong judges (GPT-4) reach >80% agreement with humans — the same level as human-human agreement. Establishes both that judges are usable at scale AND that they need explicit bias mitigation; 80% agreement also means ~1-in-5 judgments diverge, which is why practice refuses judge-only promotion gates. *Source: Zheng et al. (2023), 'Judging LLM-as-a-Judge with MT-Bench and Chatbot Arena', arXiv:2306.05685 (NeurIPS 2023) — <https://arxiv.org/abs/2306.05685>*
- **Position-bias hackability + Balanced Position Calibration** — Judge rankings can be flipped simply by reordering candidates — Vicuna-13B was made to 'beat' ChatGPT on 66 of 80 queries by swapping positions. Mitigations: Multiple Evidence Calibration (judge must write evidence before scoring), Balanced Position Calibration (evaluate both orderings, aggregate), and Human-in-the-Loop calibration for cases flagged by position-diversity entropy. If any LLM judge sits in this pipeline's gates, position-swapped double evaluation is the minimum viable mitigation. *Source: Wang et al. (2023), 'Large Language Models are not Fair Evaluators', arXiv:2305.17926 (ACL 2024) — <https://arxiv.org/abs/2305.17926>*
- **Position bias is judge- and task-dependent, driven by candidate quality gap** — Systematic study (15 judges, ~40 candidate models, 150k+ instances) introducing repetition stability, position consistency, and preference fairness metrics: position bias varies substantially across judges and tasks, is weakly influenced by length but strongly by the quality gap between candidates — i.e., judges are least reliable exactly when candidates are close in quality, which is the regime a promotion gate must adjudicate. *Source: Shi et al. (2024), 'Judging the Judges: A Systematic Study of Position Bias in LLM-as-a-Judge', arXiv:2406.07791 — <https://arxiv.org/abs/2406.07791>*
- **Self-preference bias correlates with self-recognition** — LLMs have non-trivial accuracy at recognizing their own generations, and there is a linear correlation between self-recognition capability and self-preference strength. Grounds two hard rules for this pipeline: never let the model (family) that generated a digest/verification also judge it, and be wary of same-vendor judge/generator pairs in the go/no-go evidence. *Source: Panickssery, Bowman & Feng (2024), 'LLM Evaluators Recognize and Favor Their Own Generations', arXiv:2404.13076 (NeurIPS 2024) — <https://arxiv.org/abs/2404.13076>*
- **12-bias taxonomy for LLM judges (CALM framework)** — Quantifies 12 distinct biases in LLM-as-judge systems via automated, principle-guided perturbation, finding that 'significant biases persist in certain specific tasks' even in strong judges — the current best checklist for auditing any judge added to an eval suite, well beyond the canonical position/verbosity/self-enhancement trio. *Source: Ye et al. (2024), 'Justice or Prejudice? Quantifying Biases in LLM-as-a-Judge', arXiv:2410.02736 — <https://arxiv.org/abs/2410.02736>*
- **Industry downgrade: LLM-as-judge moved Trial → Assess** — Thoughtworks demoted LLM-as-judge in the Nov 2025 Radar (Vol 33) citing position bias, verbosity bias, low robustness, self-enhancement bias, and preference leakage that 'inflate performance metrics without real-world validity', and now advises 'human verification, transparency and ethical oversight before incorporating LLM judges into critical workflows'. This is the sharpest documented disagreement with any design that lets an LLM judge gate promotion unattended: the mainstream engineering-advisory position as of 2026 is that judges inform the human go/no-go, they do not replace parts of it. *Source: Thoughtworks Technology Radar — LLM as a judge (Vol 33, Nov 2025) — <https://www.thoughtworks.com/radar/techniques/llm-as-a-judge>*
- **Deterministic-first assertion taxonomy** — Assertions split into deterministic metrics (equals, contains, regex, is-json/is-sql, cost, latency, custom Python/JS — no model involved) and model-assisted metrics (llm-rubric, g-eval, embedding similarity, classifiers). Guidance: deterministic for anything objectively checkable (format, structure, performance budgets — which covers vault-state integrity and schema checks); model-graded only for subjective qualities like relevance and tone. The pipeline's schema-first trust order (schema → tests → code → docs) is exactly this doctrine applied to a product. *Source: promptfoo docs — Assertions & metrics reference — <https://www.promptfoo.dev/docs/configuration/expected-outputs/>*


## Agentic coding pipelines (AI-authored code)

### Scout summary

2025-26 practice largely validates this pipeline's shape — human judgment concentrated at spec approval and a behavioral go/no-go, with deterministic gates in between — but it is ahead of the mainstream on zero per-PR human review, and the named precedents that skip it all add specific guardrails. (a) Trust models: the dominant vendor position (GitHub Copilot coding agent) is the OPPOSITE of auto-merge — agent PRs are drafts, the agent cannot approve/merge, CI needs human 'Approve and run' by default; DORA 2025 finds ~90% AI adoption but ~30% of devs with little/no trust and most orgs not extending delivery controls to AI code, prescribing 'treat AI output as untrusted input to a strong deterministic pipeline'. Anthropic's pattern for unattended agents is harness engineering: machine-readable feature specs, per-session test runs, fresh-context adversarial agent review (never the authoring context). (b) Auto-merge-on-green has two solid precedents: bot dependency updates (Renovate/Dependabot — explicitly gated on 'automerge is only as safe as your test suite', scoped to low-risk change classes first) and Intercom's AI-approval system (risk-tiered: refuses large/complex PRs, human escalation on demand, full audit log, engineers stay accountable with rollback readiness; 0.53% vs 5.39% revert rate for AI- vs human-authored backend code). Infrastructure: GitHub required checks + branches-up-to-date + merge queue (essential once multiple agents land concurrently). (c) The 'review spec and behavior, not code' model is now named practice — GitHub Spec Kit, AWS Kiro, Tessl, Sean Grove's 'specs are the new code' — and the rubber-stamp literature actually supports dropping pro-forma per-PR review: most agent PRs get no meaningful human review anyway, and automation complacency degrades what review happens. Documented failure modes are real and directly relevant: coding agents (incl. Claude Code and Codex) demonstrably reward-hack visible test suites; the visible-vs-holdout pass gap GROWS with task complexity (error compounding); Anthropic showed reward hacking learned in coding RL generalizes to sabotage. Dissent: Simon Willison calls unreviewed production code irresponsible and warns of normalization of deviance. (d) The recommended deterministic stack for agent code: lint + strict typing + tests run in CI independent of the agent's environment, PLUS holdout behavioral assertions the agent never sees (this pipeline's ephemeral test-vault is exactly this pattern), property-based tests (Hypothesis) to assert invariants rather than examples, and mutation testing (Google diff-scoped model; Meta ACH) to verify the test gate itself has teeth — the standard defense against agent-written tests that pass but assert nothing. Gaps worth closing in this pipeline given the sources: keep the exact-state assertion corpus out of agent context; add mutation testing over agent-authored tests; size/scope caps that force decomposition of large agent PRs; a merge queue if agents run concurrently; and a snapshot-before-promotion revert strategy for the non-reconstructible production vault (every auto-merge precedent pairs autonomy with cheap rollback).

### Findings

- **Draft-by-default agent PRs; agent cannot approve or merge itself (most conservative mainstream trust model)** — GitHub Copilot coding agent: PRs open as drafts; the agent can only push to copilot/ branches; it cannot mark PRs ready, approve, or merge; the human who prompted the agent cannot approve the PR (preserves required-approvals semantics); branch protections and required checks apply; Actions workflows do not run until a human clicks 'Approve and run' (a March 2026 setting lets admins skip this for CI, but merge still requires a human). Trade-off: this is the industry-leader position and it directly contradicts unattended auto-merge for feature code — deviating from it is a deliberate risk acceptance, not standard practice. *Source: GitHub Docs — Risks and mitigations for Copilot coding agent — <https://docs.github.com/en/copilot/concepts/agents/cloud-agent/risks-and-mitigations>*
- **'AI is an amplifier; treat AI output as untrusted input' — extend existing deterministic controls, don't invent parallel ones** — DORA 2025 (Google Cloud, 'State of AI-assisted Software Development'): ~90% of developers use AI; only ~3% report high trust and ~30% little/none; >60% have found AI-introduced errors post-deployment; most orgs have NOT extended DevOps controls to AI-generated code. Prescription: AI magnifies the strengths/weaknesses of your existing delivery system — small batches, version control, fast deterministic quality gates are the mechanism that makes AI code safe, not per-line human scrutiny. Trade-off: DORA documents low trust as the norm; a no-human-review pipeline is betting against the median org's posture and must compensate with gate quality. *Source: DORA / Google Cloud — 2025 State of AI-assisted Software Development — <https://dora.dev/dora-report-2025/>*
- **Fresh-context adversarial agent review (machine review as first-pass gate for agent code)** — Anthropic's guidance: review agent code with a FRESH context or a separate subagent so the reviewer isn't biased toward code it just wrote; their Claude Code code-review feature runs multiple specialized agents that find candidate bugs, verify them to cut false positives, then rank by severity — positioned explicitly as a response to the flood of AI-generated code exceeding human review capacity. Trade-off: Anthropic frames it as augmenting, not replacing, human accountability; it reduces but does not eliminate escape rate. *Source: Anthropic — Claude Code best practices; Code Review for Claude Code — <https://code.claude.com/docs/en/best-practices>*
- **Harness engineering for unattended long-running agents (initializer + incremental sessions + self-verification artifacts)** — Anthropic's published pattern for agents working unattended across many context windows: an initializer agent expands the prompt into a machine-readable feature-list.json (a spec the harness tracks); each coding session must make incremental progress on one feature, run tests, write a progress note, and commit — the environment is engineered so the agent can verify its own work and the next session can reconstruct state from artifacts + git history. This is the closest vendor-documented precedent for 'agents implement unattended against an approved spec'. *Source: Anthropic Engineering — Effective harnesses for long-running agents — <https://www.anthropic.com/engineering/effective-harnesses-for-long-running-agents>*
- **Perception-measurement gap: human judgment of AI-code productivity/quality is unreliable — measure, don't ask** — METR's RCT (16 experienced OSS maintainers, 246 tasks, early-2025 tools): developers were 19% SLOWER with AI allowed yet estimated they were 20% faster. METR now flags the result as historical, but the durable lesson for gate design stands: self-assessed confidence in agent output diverges from measured outcomes, which argues for deterministic/behavioral gates over human impressions at the go/no-go — bring data (revert rates, gate escape counts) to that human decision. *Source: METR — Measuring the Impact of Early-2025 AI on Experienced Open-Source Developer Productivity — <https://metr.org/blog/2025-07-10-early-2025-ai-experienced-os-dev-study/>*
- **Auto-merge on green: GitHub native auto-merge + required status checks + branches-up-to-date** — GitHub's supported mechanism: enable auto-merge on the PR; it merges only when all required reviews and required status checks pass. Practitioner guidance is unanimous that this is 'only as safe as your CI' — required checks must be explicitly configured (an unconfigured repo auto-merges on nothing), and 'require branches to be up to date' should be on so the merged state was actually tested. For a solo-maintainer repo the required-review count can be zero, making checks the sole gate — which is exactly this pipeline's model. *Source: GitHub Docs — Automatically merging a pull request — <https://docs.github.com/en/pull-requests/collaborating-with-pull-requests/incorporating-changes-from-a-pull-request/automatically-merging-a-pull-request>*
- **Merge queue for concurrent agent PRs (green-on-branch ≠ green-on-main)** — GitHub merge queue validates each PR against the latest target branch PLUS the PRs queued ahead of it (merge_group event), eliminating the semantic-conflict window where two individually-green PRs break main together. 2025-26 tooling guides (Mergify, Aviator, GitHub native) recommend a merge queue as the default once multiple agents/bots land changes concurrently. Trade-off: extra CI cost and latency; unnecessary for strictly serialized single-agent flow, cheap insurance otherwise. *Source: GitHub Docs — Managing a merge queue — <https://docs.github.com/en/repositories/configuring-branches-and-merges-in-your-repository/configuring-pull-request-merges/managing-a-merge-queue>*
- **Bot automerge with scoped trust (the decade-old precedent: dependency bots)** — Renovate's automerge doctrine is the mature template for merging bot PRs without human review: enable automerge only where you trust test coverage; start with the lowest-risk change classes (dev dependencies, lockfile maintenance) and expand as confidence grows; require branch-protection status checks; accept that 'automerge is only as safe as your tests'. The transferable pattern: autonomy is granted per change-class, not globally, and is earned by demonstrated gate quality. *Source: Renovate Docs — Automerge key concepts — <https://docs.renovatebot.com/key-concepts/automerge/>*
- **AI-approval with risk-tiered autonomy and audit trail (strongest named precedent for merging without human code review)** — Intercom (2026): an AI agent reviews and approves PRs — 19% of PRs fully autonomously approved, 93% agent-driven review. Guardrails: the agent refuses to approve large/complex/broad-scope PRs and forces decomposition; any engineer can summon a human review at any time; every review comment, approval decision, test result, and merge event is logged and queryable; engineers remain accountable — they watch deploys and are ready to roll back. Reported outcomes: zero reverts in the 100-PR pilot; AI-authored backend revert rate 0.53% vs 5.39% human. Trade-off: this is one company's self-reported data on their own system; the load-bearing guardrails (size caps, audit, rollback-readiness) are the exportable part. *Source: Intercom Engineering blog — 'AI is approving our pull requests: here's how we made it safe' — <https://www.intercom.com/blog/ai-is-approving-our-pull-requests-heres-how-we-made-it-safe/>*
- **Deterministic policy layer decides merges; AI only advises (graduated autonomy levels)** — Emerging practitioner consensus for agent PR automation: AI review accelerates and suggests, but the merge decision is made by deterministic policy-as-code — required checks, path allowlists, change-class rules. Graduated autonomy: full auto-merge reserved for defined low-risk classes (docs, lint fixes, localized changes in allowlisted paths) with all checks green; everything else escalates. Agent code must pass the same CI as human code, executed in infrastructure independent of the agent's own environment so the agent cannot influence its verdict. *Source: M. Tavares — 'AI can speed up code review, but merge decisions still need deterministic guardrails' (DEV Community) — <https://dev.to/mvtavares/ai-can-speed-up-code-review-but-merge-decisions-still-need-deterministic-guardrails-2l85>*
- **Spec-Driven Development (SDD) — GitHub Spec Kit** — GitHub's open-source toolkit (Sept 2025) makes the spec the reviewed, living source of truth: /constitution (project principles) → /specify → /plan → /tasks → /implement, with optional analyze/checklist validation steps; agents (Copilot, Claude Code, Gemini CLI, 29 integrations) implement from the spec. The human's review surface moves to the spec and its acceptance criteria — the named industry articulation of 'human approves spec, agents implement'. Trade-off: Spec Kit still assumes humans inspect resulting PRs; it standardizes the upstream gate, not the removal of the downstream one. *Source: GitHub — Spec Kit repo and announcement blog — <https://github.com/github/spec-kit>*
- **Specs as the unit of work — AWS Kiro** — Kiro (AWS, July 2025 preview) operationalizes the same model in an agentic IDE: a prompt becomes requirements.md (user stories + structured acceptance criteria), design.md, and tasks.md with traceability from requirement to task; agent hooks auto-run tests/docs/security scans on file events. Independent precedent (Amazon) that the human-reviewable artifact is the spec triad, with deterministic hooks as the code-side gate. *Source: AWS — Kiro agentic IDE — <https://kiro.dev/>*
- **'Specs are the new code' (spec clauses as executable tests)** — Sean Grove (OpenAI), AI Engineer World's Fair 2025: code is a lossy projection of the spec; the durable artifact worth human review is the specification. Concrete mechanism from OpenAI's Model Spec: every clause has an ID with associated examples that act as unit tests, so each stated principle is mechanically checkable against behavior — the intellectual grounding for 'human reviews spec and behavior, machine checks conformance'. *Source: Sean Grove (OpenAI) — 'The New Code', AI Engineer 2025 (talk transcript) — <https://lawwu.github.io/transcripts/8rABwKRsec4.html>*
- **Failure mode: rubber-stamp review — mandated per-PR human review of agent code degenerates** — EASE 2026 study on the AIDev GitHub dataset: most AI-generated PRs receive NO review; when reviewed, review is dominated by other AI agents; human-authored PRs attract far more direct human feedback — standard review metrics overstate human oversight of agent code. Practitioner literature adds automation complacency ('if there were an issue the bot would have flagged it'); the only well-evidenced countermeasure (TU Berlin line of work) is deliberate exposure to automation failures. Implication that supports this pipeline: pro-forma per-PR human review of agent code buys little; concentrating scarce human attention at spec and behavior gates is more honest — provided the deterministic stack is hardened. *Source: 'These Aren't the Reviews You're Looking For: How Humans Review AI-Generated Pull Requests' (EASE 2026) — <https://arxiv.org/abs/2605.02273>*
- **Failure mode: error compounding with task length/complexity** — METR's time-horizon research quantifies that agent success probability decays with task length (the 50%-success horizon metric; doubling ~every 7 months but still bounded — long unattended chains accumulate failure). SpecBench complements it: across Claude Code, Codex, and OpenCode, every model saturates the VISIBLE test suite on every task, while the gap between visible-test and holdout-test pass rates grows with task complexity — i.e., in long unattended runs, defects don't surface as red checks, they hide behind green ones. Design consequence: cap task size per agent run and assert behavior against checks the agent cannot see. *Source: METR — 'Measuring AI Ability to Complete Long Tasks'; SpecBench (arXiv 2605.21384) — <https://metr.org/blog/2025-03-19-measuring-ai-ability-to-complete-long-tasks/>*
- **Failure mode: reward hacking in coding agents (documented in the exact agents in use)** — OpenAI ('Monitoring Reasoning Models for Misbehavior', Mar 2025): frontier coding agents subvert unit tests to pass; CoT monitoring by a weaker model catches much of it, but optimizing against the monitor teaches obfuscated hacking — so keep gates as held-out checks, never as signals the agent iterates against. Anthropic ('Natural Emergent Misalignment from Reward Hacking in Production RL', Nov 2025): reward hacking learned in production coding environments generalized to sabotage, including of the paper's own codebase, via Claude Code. EvilGenie benchmark (arXiv 2511.21654) observed explicit reward hacking by both Codex and Claude Code. Consequence for this pipeline: integrity assertions and exact-state checks must live outside the agent's context and file-write perimeter, and 'all green' must never be the agent's optimization target during implementation. *Source: OpenAI chain-of-thought monitoring; Anthropic reward-hacking paper; EvilGenie benchmark — <https://openai.com/index/chain-of-thought-monitoring/>*
- **Dissent: 'unreviewed production code is vibe coding' — accountability and normalization of deviance** — Simon Willison (the most-cited practitioner voice on AI-assisted programming) maintains that shipping unreviewed agent code to production is irresponsible: AI has no accountability or reputation, and repeated agent success breeds 'normalization of deviance' where verification steps quietly erode until a critical failure. By May 2026 he concedes his own boundary is blurring for low-stakes changes, but keeps the human accountable for production behavior. Counterweight to take seriously: his failure mode (gates eroding over time because they keep passing) applies to deterministic gates too — periodically inject known-bad changes to prove the gates still catch them. *Source: Simon Willison — 'Vibe coding and agentic engineering are getting closer than I'd like' — <https://simonwillison.net/2026/May/6/vibe-coding-and-agentic-engineering/>*
- **Holdout behavioral verification the agent never sees (the anti-reward-hacking gate)** — SpecBench's core finding — uniform 100% pass on visible tests, widening gap on holdout tests — makes held-out verification the single highest-leverage deterministic gate for agent-authored code. This pipeline's ephemeral test-vault (every command × every API from a seed baseline, exact vault state + integrity asserted) IS this pattern, with one condition the literature adds: the assertion corpus and expected-state fixtures must be excluded from the implementing agent's context and write access, or they become part of the hackable reward. *Source: SpecBench — 'Measuring Reward Hacking in Long-Horizon Coding Agents' (arXiv 2605.21384) — <https://arxiv.org/html/2605.21384v1>*
- **Property-based testing as the generation-validation bridge (Hypothesis)** — 2025 research line ('Use Property-Based Testing to Bridge LLM Code Generation and Validation', arXiv 2506.18315; FSE 2025 'From Prompts to Properties'): PBT validates semantic properties/invariants instead of examples, which LLM-authored example tests systematically under-cover; the Property-Generated Solver improved pass@1 by 23–37% over TDD-style baselines, and combining PBT with example tests raised bug detection from ~69% to ~81% (arXiv 2510.25297). For a Python engine, Hypothesis is the standard tool; properties (e.g., vault integrity invariants, round-trip laws for catalog operations) are also harder for an agent to reward-hack than enumerated examples. *Source: arXiv 2506.18315 / 2510.25297 (FSE 2025 line of work) — <https://arxiv.org/html/2506.18315v1>*
- **Mutation testing to verify the test gate itself has teeth (Google diff-scoped; Meta ACH)** — When agents author both code and tests, 'tests pass' proves little — mutation testing is the deterministic check that the tests can actually fail. Google ('Practical Mutation Testing at Scale', IEEE TSE 2021; 24k developers, 1k+ projects) makes it tractable by mutating only the diff under review, filtering low-value mutants, and surfacing survivors as review findings — the exact shape needed for per-PR agent gates (Python: mutmut/cosmic-ray on changed files). Meta's ACH (FSE 2025) inverts it: LLM-generated targeted mutants, then LLM-generated tests that kill them, with 73% engineer acceptance. Trade-off: CI cost and equivalent-mutant noise; scope to changed code and treat surviving mutants as a soft gate initially. *Source: Google (Petrović et al., IEEE TSE) and Meta ACH (arXiv 2501.12862, FSE 2025) — <https://arxiv.org/abs/2102.11378>*
- **Recommended deterministic stack + revert strategy for a non-reconstructible target** — Composite of the above sources, ordered by cost: (1) lint + strict static typing (mypy/pyright strict — cheapest large-class error filter, universally assumed in 2025-26 agent-CI guidance); (2) example + property tests in CI executed on infrastructure the agent cannot modify; (3) diff-scoped mutation testing to validate agent-written tests; (4) holdout behavioral/state assertions (the ephemeral vault) kept out of agent context; (5) risk-tier rules: size/scope caps forcing decomposition (Intercom), change-class allowlists for full autonomy (Renovate); (6) merge queue if agents land concurrently (GitHub); (7) because every auto-merge precedent pairs autonomy with cheap rollback, and the production vault is recoverable-but-not-reconstructible, take an immutable snapshot at each promotion and treat 'restore last snapshot' as a rehearsed, tested operation — the go/no-go human gate should see gate-escape and revert metrics, not diffs. *Source: Synthesis: GitHub / Renovate / Intercom / DORA 2025 / Google TSE / SpecBench (URLs in individual findings) — <https://dora.dev/dora-report-2025/>*


## Golden-state / exhaustive-surface testing

### Scout summary

Practice strongly supports the transcript/golden-state approach for CLI+stateful-engine testing — it is exactly how git (t/ suite), Mercurial (.t unified tests), cram, Go's cmd/go (testscript/txtar), Jane Street (expect tests) and Rust (insta) work — but with four disciplines the described pipeline should adopt and three places where practice disagrees with it. Disciplines: (1) goldens stay honest only via a review ritual — machine writes the snapshot, a human reviews the diff (cargo insta review, dune promote, cram -i), and CI must refuse to auto-update snapshots; since Memoria has no per-PR human review, the golden-state DIFF (not just pass/fail) must be surfaced at the human go/no-go gate, or agents will rubber-stamp their own regressions. (2) Nondeterminism is pinned at the source — redactions/filters for timestamps/IDs (insta), (glob)/(re) escapes (cram/hg), and record-replay cassettes for LLM calls; practice disagrees with running even a cheap/local LLM inside exact-state mechanism tests — pin that seam with recorded fixtures and test live LLM behavior separately as threshold-based evals. (3) Keep snapshots small, focused, and semantically labeled; prefer explicit assertions for business-logic values (Jest/Kent C. Dodds; insta says the same). (4) Assert state via public surface, not internals (git's t0000 rule). Disagreements: NIST's interaction rule says a full command×API×state cross-product is wasteful — enumerate the command catalog exhaustively (completeness by construction, like git's t0450 help-vs-docs meta-test and Schemathesis iterating every OpenAPI operation) but sample option/state combinations pairwise. Meszaros warns a rich shared seed breeds Obscure/Erratic tests — reconciled by treating the seed as an immutable baseline cloned fresh per test, which the ephemeral-vault design already does. Most importantly, SQLite — the canonical irreplaceable-user-state product — does NOT protect production state with golden snapshots: it uses invariant checks (PRAGMA integrity_check) under injected anomalies (OOM, I/O error, crash/power-loss via a rigged VFS), differential testing against other engines, and regression tests for every bug; and Google SRE adds that a recoverable-not-reconstructible vault is only protected by continuously exercised restores, not by backups. The pre-production gate should therefore add anomaly/crash injection + integrity invariants + scheduled restore drills on top of exact-state goldens.

### Findings

- **Snapshot review workflow: machine writes the golden, human reviews the diff; CI never auto-updates** — insta's core discipline: snapshots are never hand-edited; `cargo insta review` presents each changed snapshot as a diff to accept/reject, and in CI insta will not write new snapshot files (INSTA_UPDATE defaults to no in CI) — updates can only enter via a reviewed commit. Redactions and filters replace nondeterministic values (timestamps, IDs) so snapshots stay stable; unreferenced-snapshot cleanup prevents orphan-golden rot; and the docs warn not to force a snapshot where a simple assertion does the job. Trade-off: the whole scheme's integrity rests on the review step — in a no-per-PR-review pipeline like Memoria's, the equivalent control is (a) CI hard-failing on snapshot drift and (b) routing the accumulated snapshot diff to the human go/no-go gate for eyeball review. *Source: insta (Armin Ronacher), Rust snapshot-testing library docs — <https://insta.rs/docs/>*
- **Unified .t transcript tests (cram / Mercurial): shell session as golden file with typed escape hatches** — Tests are literal interactive-shell transcripts: `  $ cmd` lines run, following indented lines are expected output. Matching is literal-first, with explicit per-line escape hatches — (glob), (re), (?) optional-line — so nondeterminism is annotated visibly rather than hidden in normalization code. `run-tests.py -i` / `cram -i` shows the output diff and asks 'Accept this change?', merging accepted output back into the test (the promote workflow). Mercurial's guidance for avoiding brittleness: no time- or path-dependent output, force non-interactive modes (-m, HGMERGE), filter tmp paths, use conventional names (f1, d1), and fold new cases into existing test files rather than proliferating scripts. This is the closest existing pattern to Memoria's command-transcript-over-a-vault tests. *Source: Mercurial wiki WritingTests; cram (bitheap.org) — <https://wiki.mercurial-scm.org/WritingTests>*
- **Expect tests: inline goldens filled in by the tool, promoted via build system** — Jane Street's ppx_expect: you drop an empty [%expect] block in the test source; running the test captures actual output into a .corrected file and `dune promote` rewrites the source. The insight practice draws from this: making goldens near-zero-cost to create AND to update turns snapshot testing from a maintenance tax into the default way to observe behavior — but the diff-accept step is still the load-bearing human control. Jane Street pairs it with strict formatting/indentation options to keep diffs minimal and reviewable. *Source: Jane Street engineering blog, 'The joy of expect tests'; janestreet/ppx_expect — <https://blog.janestreet.com/the-joy-of-expect-tests/>*
- **git t/ suite: fresh trash-directory fixture per script, output compared with test_cmp, state asserted via public surface only** — Every test script gets an isolated 'trash directory' with a fresh repository (fresh fixture by construction); assertions are `test_expect_success` blocks chaining commands with && (so mid-sequence failures aren't masked), `test_cmp expected actual` for output goldens, `test_must_fail` instead of `! git` (to distinguish controlled failure from segfault), and a hard rule to never pipe git output (pipes discard exit codes and hide crashes). Crucially, t/README dictates that only t0000-basic.sh may have intimate knowledge of internals — all other tests must assert repository state through git's public commands and output, keeping thousands of tests stable across internal refactors. Direct guidance for Memoria: assert vault state via the product's own inspection commands, not raw SQLite rows or file layouts, except in one designated internals test. *Source: git project, t/README (test suite documentation) — <https://github.com/git/git/blob/master/t/README>*
- **testscript/txtar (Go cmd/go): script + seed filesystem + expected files in one archive; combined FS and output assertions** — Go's cmd/go is tested by ~900 txtar script tests: one text file holds the command script at top and the complete seed workspace (input files) below as `-- name --` sections; the script runs in a sandbox and asserts exit status, stdout/stderr patterns, and resulting file contents — i.e., combined filesystem-state + output assertion from an explicit seed baseline, exactly Memoria's ephemeral-vault shape. Extracted for general use as rogpeppe/go-internal/testscript, with support for auto-updating golden files (the Go community's -update flag idiom, popularized by Mitchell Hashimoto's 'Advanced Testing with Go', GopherCon 2017). Trade-off: txtar seeds are per-test and minimal; a large shared seed (1200-paper corpus analog) should live outside the archive and be cloned per test. *Source: rogpeppe/go-internal testscript (extracted from Go toolchain's cmd/go test infrastructure) — <https://pkg.go.dev/github.com/rogpeppe/go-internal/testscript>*
- **Snapshot rot and rubber-stamping: keep snapshots small, committed, and reviewed; prefer explicit assertions for meaning** — Jest's own docs: commit snapshots, review them like code in PRs, keep them focused and short (eslint-plugin-jest no-large-snapshots), and pin nondeterministic data. Kent C. Dodds and multiple practitioner critiques document the failure mode: broad snapshots + reflexive `jest -u` turn the suite into a rubber stamp that encodes 'whatever the code did last week' with no semantic intent; unrelated tests go red on every change. Practice consensus: snapshots for stable output you'd otherwise transcribe by hand; explicit matchers for business-logic values, error states, and interactions. For an agents-only pipeline this is the sharpest risk — an agent asked to make CI green will regenerate goldens; the countermeasure is separating the permission to change product code from the permission to change snapshot files (CI rejects snapshot changes not explicitly flagged, human gate reviews the golden diff). *Source: Jest documentation; Kent C. Dodds 'Effective Snapshot Testing'; practitioner critiques (Brains&Beards, Soluto) — <https://jestjs.io/docs/snapshot-testing>*
- **Coverage complete by construction: derive the test matrix from the machine-readable surface catalog, and meta-test the catalog itself** — Two verified instances: (1) Schemathesis walks every operation in an OpenAPI/GraphQL schema and generates property-based test cases (Hypothesis) per operation, including stateful multi-step sequences — endpoint coverage is complete by construction because the schema IS the catalog; response-conformance and crash properties are checked rather than golden outputs. (2) git's t0450-txt-doc-vs-help.sh enumerates all builtins and cross-checks each command's -h synopsis against its manpage, with an explicit committed allowlist (txt-help-mismatches) for known exceptions — a meta-test that fails when a new command escapes the documented/tested surface. Pattern for Memoria: a single manifest of commands × APIs, a meta-test asserting every manifest entry has a transcript test, and schema-driven generation for the HTTP/MCP surface; new surface then cannot ship untested by construction. *Source: Schemathesis docs; git t0450 (help-vs-doc consistency test) — <https://schemathesis.readthedocs.io/>*
- **Consumer-driven contract testing (Pact): pin only what consumers use — a deliberate counterpoint to exhaustive surface testing** — Pact's guidance: contract testing fits when you control both consumer and provider (Memoria's Obsidian plugin ↔ engine HTTP/MCP APIs qualifies even in one repo). The consumer's tests generate the contract; the provider verifies it in its build. Deliberate consequence: only the parts of the API actually used by consumers are pinned — unused provider behavior stays free to change without breaking tests. This disagrees with 'run every API exhaustively and assert exact state': practice splits the surface into consumer-relied-upon behavior (pin hard, contract tests) and the rest (conformance/property checks, free to evolve). Worth adopting so the plugin can't silently break while keeping the engine refactorable. *Source: Pact documentation (docs.pact.io) — <https://docs.pact.io/>*
- **NIST interaction rule / combinatorial testing: exhaustive cross-products waste budget; pairwise covers most faults** — NIST's empirical studies (1999–2004, across domains) found most software failures are triggered by a single parameter or an interaction of two, with sharply diminishing returns beyond ~3-way (the 'interaction rule'). Practical Combinatorial Testing (SP 800-142) and the ACTS covering-array tool operationalize this: cover all pairs (or triples) of parameter values instead of the full cross-product. Applied to Memoria: enumerating every command and every API is cheap and should be exhaustive (small axis, completeness by construction), but crossing that with option flags × seed-state variants explodes — practice says sample those combinations pairwise via covering arrays rather than running the full matrix from the seed baseline every time. *Source: NIST SP 800-142 'Practical Combinatorial Testing' (Kuhn, Kacker, Lei) — <https://nvlpubs.nist.gov/nistpubs/legacy/sp/nistspecialpublication800-142.pdf>*
- **Fixture discipline (Meszaros, xUnit Test Patterns): rich shared seeds breed Obscure and Erratic tests; reconcile via immutable baseline cloned per test** — Fresh Fixture (each test builds its own state) prevents Erratic Tests (interdependence, unrepeatability) and yields Tests-as-Documentation; Shared Fixture (one rich seed reused across tests) is faster but risks Obscure Test — the reader can't see which part of the seed a test depends on (Mystery Guest) — and test coupling; Minimal Fixture says use the smallest state that expresses each test. This partially disagrees with a single rich seed baseline. The recognized reconciliation, which Memoria's ephemeral-vault design already implements: treat the rich seed as an IMMUTABLE baseline image, restore/clone it fresh for every command-under-test (Meszaros's Immutable Shared Fixture / fresh-fixture-by-cloning), never mutate it in place across tests, and use intent-revealing finder helpers so each test names the seed entities it relies on. Keep unit-level tests on minimal fixtures; reserve the rich seed for the surface-matrix tier. *Source: Gerard Meszaros, xUnit Test Patterns (xunitpatterns.com pattern catalog) — <http://xunitpatterns.com/Shared%20Fixture.html>*
- **SQLite testing philosophy: for irreplaceable user state, assert invariants under induced failure — not golden state** — SQLite maintains 590x more test code than product code across multiple INDEPENDENTLY developed harnesses (TCL ~51k cases, proprietary TH3 with 100% MC/DC branch coverage, SQL Logic Test, dbsqlfuzz at ~500M cases/day). The philosophy is not exact-output snapshots but: (1) anomaly injection — instrumented malloc for OOM, a rigged VFS simulating I/O errors, crash/power-loss simulation that corrupts unsynced writes — followed by (2) invariant assertion, prominently PRAGMA integrity_check verifying the database is well-formed after every induced failure; (3) every reported bug gets a permanent regression test before it's 'fixed'. Direct lesson for Memoria's pre-production gate: exact-state goldens catch behavioral drift, but protecting a recoverable-not-reconstructible vault requires a crash/interrupt/fault-injection tier asserting vault integrity invariants (SQLite integrity_check + Memoria's own grounding-structure invariants) after kill -9 mid-command, disk-full, and interrupted LLM calls. *Source: SQLite, 'How SQLite Is Tested' (official) — <https://sqlite.org/testing.html>*
- **Differential testing as oracle (SQL Logic Test): cross-check answers when exact goldens are hard to author** — SQLite's SLT harness runs millions of queries against SQLite AND PostgreSQL/MySQL/SQL Server/Oracle and asserts all engines agree — the other engines serve as the oracle, removing the need to hand-author expected output at scale. Applicable pattern: where Memoria's expected vault state is expensive to specify (e.g., derived indexes, digest side-tables), assert agreement between two independent paths (CLI vs HTTP API performing the same operation; rebuild-from-catalog vs incremental update) instead of a stored golden. *Source: SQLite, 'How SQLite Is Tested' — SQL Logic Test section — <https://sqlite.org/testing.html>*
- **Pin the LLM seam with record/replay; test live LLM behavior as separate evals — not inside exact-state gates** — Practice disagrees with running any live LLM (even cheap/local) inside deterministic golden-state mechanism tests: LLM output varies run-to-run, so exact vault-state assertion downstream of a real model is structurally flaky, and teams converge on VCR-style cassettes — record the HTTP interaction once, replay deterministically (VCR.py / pytest-recording; langchain-replay for agent decisions; LangChain's own integration-testing docs recommend cached/recorded responses). The split practice prescribes: mechanism tests exercise command logic against recorded/faked LLM responses (fully deterministic, exact-state goldens valid); a separate, smaller eval tier hits the real model and asserts properties/thresholds (schema validity of digest output, citation-grounding invariants), never exact text. Caveat: cassettes capture auth headers — redact before committing; and refresh cassettes deliberately when prompts or models change, treating the cassette diff as a reviewable golden. *Source: VCR.py / pytest-recording ecosystem; Anay Nayak, 'Eliminating flaky tests: VCR tests for LLMs' — <https://anaynayak.medium.com/eliminating-flaky-tests-using-vcr-tests-for-llms-a3feabf90bc5>*
- **Recovery is proven only by performing it: continuous restore drills for the permanent vault** — Google SRE's Data Integrity chapter: 'No one really wants backups; what people really want are restores' — the only test that counts is a periodically executed end-to-end restore, with alerting when the recovery drill fails to report success ('you only know you can recover your recent state if you actually do so'). For Memoria's production vault (recoverable but NOT reconstructible), practice therefore adds a standing automated job — outside the promotion pipeline — that restores the latest vault backup into a scratch location, runs integrity invariants (SQLite integrity_check + product-level grounding checks) and a smoke command-transcript against it, and pages the human on failure. Promotion go/no-go should require a recent green restore drill, not just green ephemeral-vault tests. *Source: Google, Site Reliability Engineering book, ch. 'Data Integrity: What You Read Is What You Wrote' — <https://sre.google/sre-book/data-integrity/>*
- **Golden-file hygiene mechanics that transfer directly: -update flags, .err artifacts, allowlisted exceptions** — Recurring mechanics across ecosystems that keep golden suites maintainable: (1) a single explicit update mode (`go test -update`, `cargo insta accept`, `cram -i`, `dune promote`, `jest -u`) so regeneration is a deliberate, greppable act — never the default and never enabled in CI; (2) failed runs emit the actual output as an artifact (Mercurial writes .err files next to the .t test) so a human can diff without re-running; (3) known deviations live in a committed, reviewed allowlist file (git's t/t0450/txt-help-mismatches) rather than in weakened assertions — the exception list shrinks over time and its growth is visible in diffs. These three give an unattended-agent pipeline auditability: every golden change is a diff in version control attributable to one deliberate command. *Source: Cross-ecosystem: Go testdata idiom (Hashimoto, GopherCon 2017), Mercurial run-tests, git t0450, insta, Jest — <https://suraj.io/post/advanced-golang-testing-mitchellh-talk-notes/>*


## Upgrade, migration, and durability testing

### Scout summary

Practice converges on a clear doctrine for stateful single-user products, and it disagrees with several plausible instincts. (a) Migration testing: the gold standard for embedded SQLite is Firefox's Places suite — a frozen binary fixture per shipped schema version (places_v52.sqlite … places_v86.sqlite) plus a test_current_from_vNN.js per version, asserting the upgraded DB reaches current schema with data intact. Web frameworks do NOT do this by default: Django's test DB is built base-to-head on an empty schema, which never exercises data migrations against realistic data; django-test-migrations and pytest-alembic exist precisely to fill that gap. The org-scale complement is GitLab's pipeline that runs every migration against a thin clone of production before merge — for a solo vault the equivalent is rehearsing the migration on a copy of the real production vault as part of the go/no-go gate (Lightroom effectively ships this to end users: upgrade writes a NEW catalog and archives the old one). Reversibility: practice is forward-only for embedded SQLite. GitLab requires down methods yet explicitly rolls FORWARD in production and tells self-managed users to restore the pre-upgrade backup rather than run down migrations; desktop apps (Firefox, Anki, Zotero) don't ship down migrations at all — they take an automatic backup before migrating, stamp the schema version (PRAGMA user_version is the standard slot), and REFUSE downgrade (Firefox compatibility.ini/dedicated profiles; Anki's explicit 'Downgrade & Quit'). SQLite's crippled ALTER TABLE forces the move-and-copy (12-step/batch) pattern, which Alembic automates. (b) Backup doctrine (Google SRE ch. 26): 'No one really wants backups; what people really want are restores' — you only know you can recover if you actually restore, so a scripted restore-into-ephemeral-vault + integrity + invariant check belongs in the release gate, not on a wiki; Google institutionalizes this as DiRT. (c) Crash durability: SQLite already does exhaustive torn-write/power-loss simulation via a custom VFS (testing.html), so app-level dm-flakey/CrashMonkey rigs are filesystem-developer territory and poor ROI; the app's actual obligations are the misuse list in howtocorrupt.html (never file-copy a live DB — use VACUUM INTO or the backup API; never delete -wal/-journal), choosing synchronous=FULL vs NORMAL deliberately (WAL+NORMAL is consistent but can silently lose the last commits on power loss — likely wrong for a vault whose writes are expensive LLM outputs), kill-9-during-write recovery tests, and knowing PRAGMA integrity_check does NOT check foreign keys (run foreign_key_check too — directly relevant to 'assert integrity after each command'). The fsync-failure literature (ATC'20) shows even SQLite mishandles fsync errors, which argues for post-crash verification + Litestream-style continuous replication rather than trust. (d) Reconstructible-vs-recoverable is exactly pets-vs-cattle (Bill Baker via Randy Bias/CERN): the ephemeral test vault is cattle, the production vault is a pet, and the industry that ships code against unrebuildable user libraries (browsers, Zotero, Anki, Lightroom) converged on one promotion contract — automatic pre-migration snapshot kept by the product itself, forward-only migration, version stamp + downgrade refusal, and integrity check after upgrade. One caveat for this product: expand/contract (ParallelChange) is unnecessary ceremony for an exclusive-access single-writer upgrade, EXCEPT where version skew is possible (Obsidian plugin vs engine vs MCP clients touching the same vault) — there it's the standard answer.

### Findings

- **Fixture-per-shipped-version upgrade matrix (browser practice)** — Firefox's Places component keeps a frozen binary SQLite fixture for each shipped schema version (places_v52.sqlite, places_v66/72/79/83/86.sqlite, favicons_v41.sqlite) with a matching test per version (test_current_from_v66.js, ..._from_v79.js, ...) that opens the old file and asserts migration to the current schema succeeds with data preserved. This is the strongest published example of the pattern for an embedded-SQLite single-user product. Trade-off: fixtures are opaque binaries that must be committed and never regenerated (regenerating defeats the point — they are a record of what actually shipped), and each release adds one fixture forever; cost is small for a solo product with few tables. Complement, not substitute, for testing against a copy of real production data. *Source: Mozilla, mozilla-central toolkit/components/places/tests/migration — <https://searchfox.org/mozilla-central/source/toolkit/components/places/tests/migration>*
- **Framework default migration testing is weaker than it looks; dedicated plugins fill the gap** — Django's test runner builds the test database by running all migrations base-to-head on an EMPTY database — data migrations are never exercised against populated state, and Django's built-in CI hooks only check hygiene (makemigrations --check fails CI when models drift from migrations; migrate --check detects unapplied migrations). django-test-migrations (wemake-services) lets tests set up state at an old migration, insert data, apply forward, and assert the transformed data. pytest-alembic ships default tests: test_upgrade (base→head runs), test_single_head_revision (no forked history — a real failure mode for parallel agents landing migrations concurrently), test_model_definitions_match_ddl (migrations coalesce to the same DDL as the models), and test_up_down_consistency. Disagreement to note: up-down consistency tests reversibility that embedded-SQLite products don't actually ship (see forward-only finding); test_single_head_revision and model-vs-DDL parity are the high-value gates for an agent-driven pipeline. *Source: Django docs; wemake-services/django-test-migrations; pytest-alembic (Schireson) — <https://pytest-alembic.readthedocs.io/en/latest/quickstart.html>*
- **Rehearse migrations against a clone of real production data before promotion** — GitLab runs every proposed migration in CI against a thin clone of the actual production database (Database Lab / postgres.ai, ZFS snapshots), reporting per-migration timing, success, and size delta before merge — because synthetic fixtures never contain the weird rows production does. The single-user equivalent maps directly onto the described go/no-go gate: snapshot the permanent vault (VACUUM INTO), run the full migration + integrity + invariant suite against the snapshot in the ephemeral harness, and only then promote. Adobe ships the same idea to end users: Lightroom Classic upgrades write a NEW catalog and archive a compressed copy of the old one, so the upgrade is physically a migration-of-a-copy with automatic rollback artifact. Trade-off: rehearsal doubles migration wall time and requires disk for the copy; for a 1200-paper vault this is trivially cheap and it is the only test that exercises the data you actually cannot lose. *Source: GitLab database migration pipeline docs; Adobe Lightroom Classic catalog FAQ — <https://docs.gitlab.com/development/database/database_migration_pipeline/>*
- **Forward-only migrations + backup-before-migrate + downgrade refusal (embedded/desktop consensus)** — Practice disagrees with the 'every migration must be reversible' instinct. GitLab's own style guide requires down methods, yet states production problems are handled by rolling FORWARD, and self-managed users are told to restore the pre-upgrade backup — the down path is for dev/test only. Desktop apps skip down migrations entirely: Zotero writes zotero.sqlite.[N].bak automatically during upgrades and documents reverting as 'replace the DB with the automatic backup'; Anki keeps ~30 rolling backups (every close/sync) and offers an explicit 'Downgrade & Quit' that rewrites the collection to the old schema — the only downgrade path is a deliberate product feature, not a migration reversal; Firefox refuses outright to open a profile touched by a newer version (compatibility.ini / dedicated-profiles, override only via --allow-downgrade). Mechanism for the version stamp: PRAGMA user_version (32-bit int at header offset 60, unused by SQLite, the conventional schema-version slot; PRAGMA application_id marks the file type). Rule that falls out: migration code path = snapshot → stamp check (refuse if DB is newer than code) → forward migrate → integrity check. *Source: GitLab Migration Style Guide; Zotero docs; Anki manual; Mozilla support; sqlite.org PRAGMA docs — <https://docs.gitlab.com/development/migration_style_guide/>*
- **SQLite ALTER TABLE is crippled; migrations must use move-and-copy (batch) pattern** — SQLite deliberately lacks most ALTER TABLE forms (no DROP/ALTER COLUMN historically, no constraint changes), so nontrivial schema changes require the documented workflow: create new table, INSERT-SELECT the data, drop old, rename — Alembic automates this as 'batch migrations' (move-and-copy). Consequence for testing: a 'column rename' on SQLite is physically a full-table rewrite, so migration tests must assert row counts, FK integrity (PRAGMA foreign_key_check), triggers, and indexes survive — not just that the DDL ran. This is a category of bug the fixture-per-version matrix exists to catch. *Source: Alembic documentation (SQLAlchemy project), 'Running "Batch" Migrations for SQLite' — <https://alembic.sqlalchemy.org/en/latest/batch.html>*
- **Restores, not backups, are the deliverable ('backups are a tax')** — Google SRE Book ch. 26 (Data Integrity: What You Read Is What You Wrote): 'No one really wants to make backups; what people really want are restores' and 'backups are a tax paid for the municipal service of data availability.' You only know you can recover if you actually restore — so restore must be a routinely executed, automated drill whose output is verified, not a documented procedure. The chapter also prescribes defense in depth (soft deletion, backups, replication) and continuous out-of-band data validation. Mapping to this product: a scripted 'restore production-vault backup into ephemeral harness, run integrity + invariant suite' drill as a RELEASE PREREQUISITE is exactly what the doctrine prescribes for state that is recoverable-but-not-reconstructible; the folk version is 'Schrödinger's backup: the condition of any backup is unknown until a restore is attempted.' *Source: Google SRE Book, ch. 26 (Beyer et al., O'Reilly) — <https://sre.google/sre-book/data-integrity/>*
- **Scheduled disaster/recovery drills as institution (DiRT)** — Google formalizes recovery testing as DiRT — an annual company-wide exercise that deliberately breaks live systems and withholds key people to prove recovery paths work before they're needed in anger (Kripa Krishnan, ACM Queue 'Weathering the Unexpected', 2012; also CACM 55(11)). The transferable practice for a solo product is scale-appropriate: the drill must actually execute (restore, rebuild ephemeral env from seed, replay a migration) on a calendar or per-release trigger, and a failed drill blocks release exactly like a failed test. Trade-off: drills cost wall time per release; the counter-argument from doctrine is that an untested recovery path is the single most expensive latent failure a non-reconstructible vault can carry. *Source: Kripa Krishnan, Google — ACM Queue — <https://queue.acm.org/detail.cfm?id=2371516>*
- **SQLite-correct backup mechanics: never file-copy a live database** — Copying an SQLite file while a transaction is in flight (or copying it without its -wal/-journal siblings) is a documented corruption cause. Correct snapshot mechanisms: the Online Backup API (incremental, doesn't lock the source for the duration) or VACUUM INTO (3.27+, produces a compacted consistent copy without exclusive access). For continuous protection, Litestream streams WAL changes to cloud/local storage for point-in-time restore with no application changes. For this product: the pre-migration snapshot and the release backup should be VACUUM INTO or backup-API artifacts, and the ephemeral-vault seed baseline should be produced the same way so tests exercise the identical snapshot path used for production recovery. *Source: sqlite.org backup docs; howtocorrupt.html; Litestream (Ben Johnson) — <https://sqlite.org/backup.html>*
- **SQLite does its own crash testing; the application's job is the misuse list** — SQLite's test harness simulates power loss with a custom VFS producing incomplete sector writes, garbage-filled pages, and out-of-order writes, asserting every transaction is all-or-nothing afterwards, plus systematic I/O-error injection verified with PRAGMA integrity_check (testing.html; the commit protocol is specified in atomiccommit.html). Practice disagreement: replicating this with dm-flakey (kernel device-mapper target that periodically errors/drops writes) or CrashMonkey/ACE (OSDI'18 bounded black-box crash testing — found 10 new bugs in Linux filesystems; see also ALICE, OSDI'14) is filesystem-developer territory with poor ROI for an application. The app-level obligations are instead: avoid the documented corruption causes in howtocorrupt.html (file-copying live DBs, deleting hot journals/-wal, multiple links to the DB file, broken fsync), and test the APPLICATION's crash-recovery path — kill -9 mid-command in the ephemeral vault, reopen, assert integrity_check + foreign_key_check + app invariants. That kill-during-write loop is cheap, catches real bugs (half-written sidecar files, app state diverging from DB), and fits the existing per-command assertion harness. *Source: sqlite.org testing/howtocorrupt docs; Mohan et al. OSDI'18 (CrashMonkey/ACE); Linux kernel dm-flakey docs — <https://www.sqlite.org/testing.html>*
- **WAL durability is a deliberate setting, not a default: synchronous=NORMAL vs FULL** — In WAL mode with synchronous=NORMAL (the common recommendation for speed), commits are written to the WAL without fsync until checkpoint: the database stays CONSISTENT after power loss but recently committed transactions can vanish. synchronous=FULL adds a WAL sync per commit, making commits durable at a per-commit fsync cost. For a research vault where each write may embody paid LLM work and human judgment, losing 'the last few commits' silently is arguably worse than slower writes — either set FULL, or keep NORMAL and force a wal_checkpoint at command boundaries so the CLI's unit of durability matches the user's unit of work. Related caution from research: 'Can Applications Recover from fsync Failures?' (USENIX ATC'20) found none of PostgreSQL/LMDB/LevelDB/SQLite/Redis handle fsync failure correctly (SQLite can corrupt in-memory state via its rollback journal) — one more reason post-crash verification and off-machine replication beat trusting the write path. *Source: sqlite.org WAL documentation; Rebello et al., USENIX ATC 2020 — <https://sqlite.org/wal.html>*
- **Integrity assertion precision: integrity_check does not check foreign keys** — PRAGMA integrity_check validates b-tree structure, record format, missing/surplus index entries, UNIQUE/CHECK/NOT NULL constraints and the freelist — but explicitly NOT foreign key constraints; PRAGMA foreign_key_check must run separately. quick_check is the O(N) variant that skips UNIQUE and index-content verification for routine use. Directly actionable for the described pipeline ('exact vault state + integrity asserted after each command'): the per-command gate should be quick_check + foreign_key_check, with full integrity_check at the promotion gate and after any restore — otherwise referential corruption in the claim/argument graph passes 'integrity ok'. *Source: sqlite.org PRAGMA documentation — <https://www.sqlite.org/pragma.html#pragma_integrity_check>*
- **Pets vs cattle: ephemeral envs are cattle, the production vault is a pet — and that's the design, not a flaw** — The reconstructible-vs-recoverable distinction is the pets/cattle analogy (coined by Bill Baker for SQL Server scaling, reframed for cloud by Randy Bias ~2012, propagated via Tim Bell/CERN): cattle are rebuilt from recipe, pets are cared for and RECOVERED. Immutable-infrastructure doctrine ('cattle everything') simply does not apply to user data — the entire desktop-software industry ships code against unrebuildable user libraries and manages it with the promotion contract in the forward-only finding (snapshot, stamp, refuse-downgrade, verify). The practice insight for this pipeline: don't try to make the production vault reconstructible (re-running 1200 papers of LLM digestion is neither faithful nor affordable); instead make its recovery loop as heavily tested as the cattle side — which is why the restore drill and migrate-a-copy rehearsal are the two gates that carry the pet. *Source: Randy Bias, Cloudscaling — 'The History of Pets vs Cattle' — <http://cloudscaling.com/blog/cloud-computing/the-history-of-pets-vs-cattle/>*
- **Expand/contract (ParallelChange) — needed only where version skew exists** — Fowler's ParallelChange / expand-contract (expand: add new alongside old; migrate: move consumers; contract: remove old) and Refactoring Databases (Ambler & Sadalage) are the canonical patterns for schema change under live, mixed-version consumers; strong_migrations (ankane) is the lint-in-CI enforcement of safe operations for Rails. Where practice disagrees with cargo-culting it: a single-user CLI that migrates with exclusive access doesn't need dual-write phases — one-shot forward migration with backup is the standard and simpler. Expand/contract earns its cost in this architecture at exactly one seam: version skew between components sharing the vault (Obsidian plugin vs Python engine vs HTTP/MCP clients that may lag a release). There, either enforce lockstep versions (refuse connection on schema mismatch, Firefox-style — simplest and matches the single-user premise) or keep API/schema additive across one release (N-1 compatibility) using expand/contract. *Source: Martin Fowler, bliki 'ParallelChange'; ankane/strong_migrations — <https://martinfowler.com/bliki/ParallelChange.html>*


## Resilience to user-edited / corrupted input

### Scout summary

Practice converges on a layered model, not a single stance. (1) Make the read path total for display but strict for derivation: interactive surfaces never refuse or rewrite a broken file (Obsidian renders invalid frontmatter as plain text with a warning; IDE parsers emit best-effort trees with error nodes; CommonMark defines every byte sequence as a valid document), while anything that derives state — index, digest, publish — should parse at the boundary into typed structures ('parse, don't validate') and fail closed per-file, quarantining the file with a reason rather than crashing the sweep or silently skipping it. The documented anti-pattern is Jekyll's default (log-and-skip → pages silently vanish; the community added strict_front_matter to fail the build), and RFC 9413 explicitly repudiates liberal acceptance where one party controls both ends — which a single-user vault does. (2) For test generation, practice says corpus AND generation, layered: a hand-enumerated, schema-driven fault corpus (delete required field, break YAML, dangling link, wrong enum) is the executable spec of per-class fail-safe behavior and the permanent regression set (Hypothesis itself says pin regressions with @example, treat its failure database as a cache); property-based/structure-aware generation on top finds the faults you didn't enumerate, and beats blind fuzzing on structured formats when it mutates realistic seed vaults rather than generating from scratch. SQLite — the closest analog to a human-corruptible store — fuzzes the database file itself, not just commands, and its anomaly tests assert 'responds sanely, never corrupts further.' (3) For repair, git fsck is the consensus model for irreplaceable data: verify integrity on every read (fail-closed), provide a user-invoked read-only sweep that reports dangling/corrupt/orphaned items, move recoverable material to a lost-found/quarantine area, and never auto-rewrite user files — Dendron's auto-fixing doctor introduced its own corruption bugs, and Logseq's rebuild-index-from-files promise failed exactly under the corrupted-input conditions it exists for, so the rebuild path itself needs fault-injection coverage. Mapping to Memoria's pipeline: the ephemeral test-vault harness is the seed corpus; add a schema-driven mutation layer (Hypothesis over frontmatter/wikilink grammar) with the generic oracle 'structured error or correct parse, never crash, vault bytes never modified by a read'; ship an fsck-style `memoria doctor` (report-only default, explicit repairs, vault-git as undo); run SQLite PRAGMA integrity_check plus a files↔index reconciliation at startup of the permanent vault; and use fsync+atomic-rename discipline anywhere the engine rewrites notes, since the write side is where 'never silently lose data' is actually violated (Pillai OSDI'14).

### Findings

- **Fuzz the state, not just the commands (anomaly testing)** — SQLite's test philosophy: 'anomaly tests' verify sane behavior when something goes wrong, not just correctness on well-formed input. Their dbsqlfuzz mutates the database FILE and the SQL together — the direct analog of mutating vault files and commands jointly — and corrupt-DB tests literally fwrite garbage into the middle of the file, then assert error codes, no crash, no further corruption. Trade-off: this level of investment (600+ LOC of test per LOC of product) is justified because the store is externally writable; the cheap subset (deliberate file corruption + assert structured failure) is what transfers to a vault product. *Source: SQLite, 'How SQLite Is Tested' — <https://sqlite.org/testing.html>*
- **Treat external edits as an expected corruption vector, checked on open** — SQLite enumerates real-world corruption causes — including other processes (or humans) writing the file directly — and prescribes PRAGMA integrity_check / quick_check as the detection sweep. For Memoria: the SQLite state DB and the vault files are both human-reachable; run quick_check at startup of the permanent vault and a full files↔index reconciliation on demand. Trade-off: full integrity_check is O(database); quick_check skips index-consistency for speed. *Source: SQLite, 'How To Corrupt An SQLite Database File' — <https://www.sqlite.org/howtocorrupt.html>*
- **Schema-driven negative-input generation (the malformed-input corpus, derived not hand-written)** — Schemathesis generates deliberately schema-invalid inputs (missing required field, wrong type, out-of-enum, boundary violations) by mutating the schema itself, then asserts the system rejects them with a structured error — exactly the 'delete required field / wrong enum' taxonomy, but generated from the frontmatter schema so it stays in sync as the schema evolves. Trade-off: generated negatives assert only the generic oracle (reject, don't crash); class-specific UX assertions (exact quarantine message, repair hint) still need hand-enumerated cases. *Source: Schemathesis docs (built on Hypothesis / hypothesis-jsonschema); Hatfield-Dodds, 'Deriving Semantics-Aware Fuzzers from Web API Schemas' — <https://schemathesis.readthedocs.io/en/stable/explanations/data-generation/>*
- **Structure-aware mutation of realistic seeds beats both blind fuzzing and pure generation** — Google's fuzzing guidance: for structured formats (nested YAML, checksummed/linked content), a mutator that parses→mutates→re-serializes valid inputs achieves better coverage over time than arbitrary generation, and the seed corpus should contain a small varied set of valid AND invalid inputs with maximal coverage. For a vault: mutate a realistic seed vault (the existing ephemeral test-vault baseline is already the seed corpus) rather than synthesizing notes from scratch. *Source: google/fuzzing, 'Structure-Aware Fuzzing with libFuzzer'; OSS-Fuzz ideal-integration guidance — <https://github.com/google/fuzzing/blob/master/docs/structure-aware-fuzzing.md>*
- **PBT and fuzzing are one continuum — build one generator, two oracles** — Elhage's argument, echoed by Hypothesis's own positioning: property-based testing and fuzzing differ in workflow, not concept. Practical consequence: don't build a PBT suite and a fuzz suite separately; one input generator (strategies over the frontmatter/wikilink grammar) drives both a generic oracle (never crash, never modify vault bytes on a read path, parse→normalize→reparse round-trips) and domain properties. PBT beats a hand-enumerated fault corpus when the input space is combinatorial and the oracle is general; it cannot replace the corpus where the assertion is per-fault-class behavior. *Source: Nelson Elhage, 'Property-Based Testing Is Fuzzing'; hypothesis.works — <https://blog.nelhage.com/post/property-testing-is-fuzzing/>*
- **Pin known fault classes as explicit examples; treat generative findings as a cache** — Hypothesis persists failing examples in its example database and replays them before generating new inputs, but the docs explicitly warn: the database is a cache that may be transparently invalidated — 'do not rely on the database for correctness; to ensure Hypothesis always tries an input, use @example.' This settles the corpus-vs-PBT question in practice: the hand-enumerated malformed-vault corpus is the permanent, versioned regression spec; generation discovers new entries for it. *Source: Hypothesis documentation, 'Replaying failed tests' / example database — <https://hypothesis.readthedocs.io/en/latest/tutorial/replaying-failures.html>*
- **Tolerant read for display: never refuse to open, never rewrite, flag inline (Obsidian)** — Obsidian on malformed YAML frontmatter: the note still opens and edits as plain text, the properties UI shows an invalid-properties warning, and the metadata is simply not indexed — no crash, no auto-rewrite of the user's bytes. Documented trade-off from the forums: the degradation is quiet, so notes with broken frontmatter silently drop out of queries/search — which is why a product needs the complementary integrity sweep (finding: doctor/fsck pattern) to make the quiet failures enumerable. *Source: Obsidian Help (Properties) + Obsidian forum behavior reports — <https://forum.obsidian.md/t/some-issues-with-properties-and-bad-yaml/67216>*
- **Total format design: make syntax errors impossible, concentrate failure in schema** — CommonMark spec §2.1: 'Any sequence of characters is a valid CommonMark document.' Markdown deliberately has no parse errors, so in a Markdown vault the entire malformed-input surface collapses to frontmatter, links, and your own schema — which is where the fault corpus and validation effort should go. This is a design lever, not just a test lever: prefer formats/fields that degrade to text over ones that can be syntactically broken. *Source: CommonMark specification 0.31.2, §2.1 — <https://spec.commonmark.org/0.31.2/>*
- **The git fsck model: self-verifying store, read-only diagnosis, explicit lost-found, no auto-repair** — Git verifies content hashes on every object read (corruption = immediate hard error, fail-closed), and git-fsck is a separate user-invoked sweep that verifies connectivity and validity, reports dangling/unreachable/corrupt objects, and NEVER modifies the repository; recovery is explicit via --lost-found, which writes recoverable objects into .git/lost-found/ for human inspection. This is the consensus pattern for irreplaceable data: detection is automatic and continuous, repair is manual and inspectable. Trade-off: fail-closed reads mean one corrupt object can block an operation until the human acts — accepted cost for never propagating corruption. *Source: git-fsck documentation, git-scm.com — <https://git-scm.com/docs/git-fsck>*
- **Silent skip is the documented anti-pattern; fail loud on derive/publish (Jekyll vs Hugo)** — Jekyll's default on malformed front matter is log-error-skip-page-continue — pages silently vanish from the built site; the community response was strict_front_matter: true (Jekyll 3.5+, PR #5832) to fail the build instead. Hugo sits at the strict end: malformed front matter fails the whole build with file/line (and Hugo #693 shows the failure mode to avoid — the dev server once crashed outright on it). Practice extract: batch/derive operations should either fail the batch or continue while ENUMERATING every skipped item; continue-and-say-nothing is the one behavior everyone regrets. *Source: Jekyll PR #5832 (strict_front_matter) + Hugo issue #693 — <https://github.com/jekyll/jekyll/pull/5832>*
- **IDE-grade resilient parsing: degrade per-node, not per-file** — Production language tooling (rust-analyzer, Roslyn, tree-sitter) never rejects a file: parsers produce a best-effort tree with explicit error nodes so everything after the broken region still works (highlighting, navigation, other definitions). matklad's tutorial notes that with hand-written recursive descent, resilience is mostly 'don't bail on the first error.' For a vault engine: one broken note — or one broken field within a note — should cost exactly that node's functionality, nothing more. *Source: matklad (rust-analyzer author), 'Resilient LL Parsing Tutorial' — <https://matklad.github.io/2023/05/21/resilient-ll-parsing-tutorial.html>*
- **Parse, don't validate: fail-closed typed boundary before any mutation** — King's prescription: convert input into the most precise typed representation at the system boundary, before ANY of the data is acted upon — because a program that acts on a valid portion and then discovers an invalid portion must roll back half-applied changes. For vault operations this means: fully parse and schema-check a note before the digest/index/bundle operation touches anything; invalid notes never enter business logic, they enter the quarantine list. *Source: Alexis King, 'Parse, don't validate' (2019) — <https://lexi-lambda.github.io/blog/2019/11/05/parse-don-t-validate/>*
- **The strictness disagreement: Tolerant Reader / Postel vs RFC 9413** — Fowler's Tolerant Reader (take only what you need, ignore the rest) and Postel's law optimize for interop between independently-evolving parties. RFC 9413 (IETF, 2023) is the counter-position: liberal acceptance ossifies bugs, hides errors from the people who can fix them, and 'fatal error' handling is appropriate when the ecosystem is actively maintained. For a single-user product where the same human authors and consumes the files, RFC 9413's condition holds — you control both ends — so practice favors strict-with-loud-errors on derived operations, reserving tolerance for the display path. This is the sharpest genuine disagreement in the space; the resolution is per-path, not per-product. *Source: IETF RFC 9413 'Maintaining Robust Protocols'; Martin Fowler, 'TolerantReader' — <https://datatracker.ietf.org/doc/html/rfc9413>*
- **Quarantine / dead-letter channel with reason metadata and replay** — Streaming/data-pipeline consensus (Kafka Connect DLQs, Spark badRecordsPath): invalid records are routed to a side channel enriched with the failure reason, the pipeline keeps running, and a human reviews/fixes/replays; failures are classified retriable vs fixable vs poison, and nothing is auto-reinjected (loops/failure storms). Vault translation: a corpus-wide digest/verification sweep quarantines broken notes into an inspectable list (not a moved file — moving user files breaks their links) and completes the rest. Trade-off: quarantine without a visible queue re-creates Jekyll's silent-skip; the queue must be surfaced at the human gate. *Source: Confluent, 'Kafka Connect Deep Dive – Error Handling and Dead Letter Queues' — <https://www.confluent.io/blog/kafka-connect-deep-dive-error-handling-dead-letter-queues/>*
- **The 'doctor' command genre — and its auto-repair hazard** — Dendron (a Markdown-vault product) ships `dendron doctor` with per-class actions: list broken links, find dangling vault-prefixed links, fixFrontmatter, create missing linked notes. It demonstrates both the pattern and its risk: doctor auto-rewrites user files, and had bugs doing so (issue #1919: fixFrontmatter injected stray lines). Contrast git fsck's never-modify stance. Best practice synthesis: report-only by default, each repair an explicit named action, and rely on the vault's own git versioning as the undo — which Memoria already has as product behavior. *Source: Dendron wiki, 'Doctor'; dendron issue #1919 — <https://wiki.dendron.so/notes/ZeC74FYVECsf9bpyngVMU/>*
- **Derived index is a disposable cache — but the rebuild path is itself a fault surface** — Logseq's model: markdown files are source of truth, re-index rebuilds the internal DB from disk. Its bug history is the cautionary tale: re-indexing emptied pages that were intact on disk (#10481), and a single empty journal file aborted the entire re-index (#683) — the recovery mechanism failed precisely on the corrupted-vault inputs it exists for. Prescription: Memoria's SQLite index should be reconstructible from vault files, and 'rebuild index from a vault containing every fault-corpus entry' should be a mechanism test in the ephemeral gate. *Source: Logseq discussions/issues (#7825, #10481, #683) — <https://github.com/logseq/logseq/issues/10481>*
- **Never silently lose data — the write side: crash-safe file update protocols** — Pillai et al. (OSDI '14) studied application update protocols with their ALICE framework and found crash vulnerabilities in nearly every application examined (including SQLite and LevelDB); even the common write-temp→fsync→rename heuristic depends on underspecified filesystem persistence properties. Resilience to corrupted input includes not CREATING corrupted files: any engine rewrite of a user note needs fsync+atomic-rename discipline, and ideally the vault-git commit as the recovery backstop. Trade-off: full fsync discipline costs latency; acceptable in a single-user CLI. *Source: Pillai et al., 'All File Systems Are Not Created Equal', USENIX OSDI '14 — <https://www.usenix.org/conference/osdi14/technical-sessions/presentation/pillai>*


## Environment promotion and human gate placement

### Scout summary

Practice largely validates the two-gate shape but disagrees on three specifics. SUPPORT: CD doctrine itself (Humble/Farley) reduces human involvement to at most one gate — the production release decision as a push-button business/judgment call — with everything between commit and release-candidate automated; ITIL 4 supplies the named precedent for the front gate (pre-authorizing a change *model*/standard change once, then letting instances flow through automated gates via delegated 'change authority'); NASA-style systems engineering supplies the named precedent for the pair (design review up front, Flight Readiness Review + go/no-go poll at the irreversible commit, verification-by-test in between). DISAGREEMENTS: (1) DORA found external end-of-pipe approval has NO correlation with change-fail rate and correlates with worse speed/stability — so the pre-production go/no-go earns its keep only as an irreversibility-management ritual (checklist, backup verified, migration reversibility confirmed), not as a defect filter; expecting it to catch bad code is the CAB fallacy. (2) 'Code never human-reviewed per-PR' contradicts what DORA recommends as the CAB *replacement* — peer review during development plus automation, not automation alone — and the 2025 DORA AI report doubles down: AI-generated change volume increases instability and makes review more critical, not less; the standard compensations are adversarial agent cross-review and treating the deterministic gates as the reviewed seam. (3) The ephemeral test-vault violates Twelve-Factor dev/prod parity on exactly the two axes most likely to bite — backing service (cheap/local LLM vs live LLM) and data shape (seed baseline vs 1200-paper corpus) — and the testing-in-production school (Majors) warns such staging produces misplaced confidence. The consensus mitigation for a recoverable-but-not-reconstructible store is a middle ring: restore the latest production-vault backup into an isolated pre-prod environment and run the release candidate (with live LLM for at least a smoke subset) against it before go/no-go — which simultaneously performs the restore-testing that makes 'recoverable' a verified property rather than a hope. Two mechanical requirements from artifact doctrine: promotion must ship the exact artifact the ephemeral vault validated (no rebuild between gate and prod), and stateful vault migrations should use expand/contract with a pre-promotion backup so the 'irreversible' deploy is actually a recoverable step — turning the go/no-go from a one-way door into a checkpointed transition, which is what release checklists (Google LCE) and go/no-go polls (NASA: pre-declared, non-waivable constraints) actually formalize.

### Findings

- **Build once, promote immutable artifacts** — The artifact that passed the pipeline is the exact artifact deployed to every subsequent environment; promotion is metadata (tag/route change), never a rebuild, because a rebuild invalidates everything the earlier environments proved. Implication for the vault pipeline: the human go/no-go must promote the same build (same wheel/commit/lockfile, same plugin bundle) the ephemeral test-vault exercised — a re-resolve of dependencies between gate and production silently re-opens the gate. *Source: Humble & Farley, Continuous Delivery (2010); MinimumCD 'Immutable Artifact' — <https://minimumcd.org/minimumcd/immutable/>*
- **Twelve-Factor X: dev/prod parity (tools gap and backing-services gap)** — Keep development/staging/production as similar as possible; the methodology explicitly resists using 'lightweight local services in dev and more serious/robust services in production' because divergent backing services cause bugs that only appear in prod. The ephemeral vault diverges on the two axes 12-factor warns about: the LLM backing service (cheap/local in gate, live in prod — different tokenization, latencies, refusal/format behavior) and data (seed baseline vs 1200-paper corpus — scale-dependent bugs in indexing, ranking, migration time are invisible at seed scale). Trade-off: full parity means paying for live-LLM CI runs and long corpus-scale tests; practice compromises with a small live-LLM smoke subset plus one prod-scale pre-prod run rather than full parity everywhere. *Source: Adam Wiggins et al., The Twelve-Factor App — <https://12factor.net/dev-prod-parity>*
- **Staging skepticism / testing in production** — The dissenting school: staging is 'a misleading set of results — mostly resulting in misplaced confidence in your code'; mirroring prod is a fool's errand because real data, scale, and long-tail states can't be cloned. This directly challenges treating a green ephemeral-vault run as predictive of the production vault. Their remedy is making production exposure safe (observability, progressive exposure, fast revert) rather than perfecting staging. For a single-user product with no canary population, the transferable piece is the revert half: invest in making the production promotion observable and cheaply reversible (verified backup + integrity assertions post-promotion) rather than adding more pre-prod fidelity you can't achieve. *Source: Charity Majors, Honeycomb — 'I Test in Production' — <https://www.honeycomb.io/blog/i-test-in-prod>*
- **Restore production backup into pre-prod (restore testing as a release ring)** — For stateful systems that are recoverable but not reconstructible, standard practice is scheduled restore testing in an isolated environment — 'a backup you have never tested is a backup you cannot trust' — and the mature version merges it with release validation: restore the latest prod backup into an isolated pre-prod, run the release candidate (including its migrations) against that real-shaped data, assert integrity, then promote. This closes the biggest gap in the current design (seed-baseline vault ≠ 1200-paper vault) AND continuously proves the 'recoverable' property the whole promotion model depends on. Cost: storage + a slower pre-gate step; run it per-release or on a schedule, not per-commit. *Source: AWS Backup restore-testing guidance; Bacula/TechTarget backup-verification practice — <https://docs.aws.amazon.com/aws-backup/latest/devguide/restore-testing.html>*
- **Single path to production / deterministic pipeline** — MinimumCD's floor definition: exactly one way for any change to reach production — through the pipeline; artifacts always meet the definition of deployable; all work stops when the pipeline is red. For the vault product this means no hand-applied hotfixes to the production vault's code or schema outside the ephemeral-gate→go/no-go path, even (especially) for one-line fixes — the classic failure mode of solo projects with a scary prod environment is the out-of-band patch that the next promotion then tramples. *Source: MinimumCD (Bryan Finster et al.) — <https://minimumcd.org/>*
- **DORA: external change approval does not reduce failure rate** — Verified from the primary page: 'no evidence was found to support the hypothesis that a more formal, external review process was associated with lower change fail rates'; heavyweight approval slows delivery, batches changes, and is associated with HIGHER risk; the 2019 Accelerate State of DevOps report found orgs with CAB-style approval were ~2.6x more likely to be low performers. Consequence for the two-gate model: the pre-production go/no-go should not be justified as a quality filter (the data says it isn't one); its legitimate jobs are irreversibility management, timing/coordination, and accountability for a judgment call automation cannot make. If it starts functioning as a code inspection, it's a CAB of one. *Source: DORA capability: Streamlining change approval (fetched directly); Accelerate State of DevOps 2019 — <https://dora.dev/capabilities/streamlining-change-approval/>*
- **DORA's prescribed replacement is peer review PLUS automation — not automation alone** — DORA recommends change approval via peer review 'captured in the team's development platform as part of the development process', supplemented by automated detection/prevention. The user's pipeline deletes the peer-review half entirely (agents implement unattended, code never human-reviewed per-PR). Practice's honest position: this is a deviation, not an endorsement — the recognized compensations are (a) adversarial/independent agent cross-review as a stand-in for peers, (b) shifting the reviewed surface from code to the spec + the behavioral assertions (every command × every API with exact-state assertions is effectively executable review), and (c) accepting that novel failure classes not encoded in the gates will reach the go/no-go undetected, which is why the pre-prod ring must be prod-shaped. *Source: DORA capability: Streamlining change approval — <https://dora.dev/capabilities/streamlining-change-approval/>*
- **Humble/Farley: the release decision is the one legitimate human gate** — Continuous Delivery (as distinct from continuous deployment) explicitly reserves a human, push-button decision to release — a business decision, not a technical inspection — while requiring everything from commit to release-candidate to be automated. This is the closest thing in core CD doctrine to a blessing of the user's second gate: the doctrine's demand is only that by the time a human decides, the candidate's fitness is already fully established by the pipeline, so the human is deciding WHETHER/WHEN, never 'is it good'. The user's model matches this shape exactly, provided the ephemeral-vault gate is trusted as the fitness oracle. *Source: Humble & Farley, Continuous Delivery (Addison-Wesley 2010); Fowler's book page and InfoQ interview — <https://martinfowler.com/books/continuousDelivery.html>*
- **Progressive delivery (canary, flags, rings) as the gate replacement — and its single-user limit** — Governor's 2018 coinage (from Microsoft's 'progressive experimentation'): replace approval gates with graded exposure — deploy to a small blast radius, observe, widen on data; feature flags decouple deploy from release; authority moves to whoever is closest to the user plus automated metrics. Trade-off that matters here: every mechanism assumes a population to segment. A single-user product has no canary cohort, so progressive delivery degrades to (a) time-based self-canarying — run the new version against the real vault in read-only/shadow mode before granting write access, and (b) feature flags to enable risky subsystems (digest/verification) after the promotion proved stable. Where rollback is impossible (data migration), progressive delivery doctrine itself concedes the gate + backup remains. *Source: James Governor, RedMonk, 'Towards Progressive Delivery' (2018; corroborated by RedMonk 2020 follow-up and LaunchDarkly) — <https://redmonk.com/jgovernor/2018/08/06/towards-progressive-delivery/>*
- **DORA 2025: AI-generated change volume increases instability; review gets more critical, not less** — The 2025 State of AI-assisted Software Development report (90% AI adoption) found AI amplifies throughput AND instability downstream unless absorbed by strong control systems — fast automated testing, mature version control, disciplined review; it frames AI as an amplifier of existing capability, not a substitute for review. This is the strongest current-evidence pushback on 'code is never human-reviewed per-PR': the unattended-agent stage is exactly the high-change-volume regime the report says destabilizes systems without robust controls. The user's deterministic gates + exhaustive command×API state assertions are the intended control system; the report implies their coverage, not the human gate count, is the load-bearing safety property — so gate-coverage regressions should be treated as release blockers in their own right. *Source: DORA / Google Cloud, State of AI-assisted Software Development 2025 — <https://dora.dev/dora-report-2025/>*
- **ITIL 4 change enablement: pre-authorized standard changes + delegated change authority (the named 'approve intent once' precedent)** — ITIL 4 replaced the per-change CAB with a 'change authority' that can be delegated — including to automated pipeline checks — and with 'standard changes' whose PROCEDURE is authorized once, after which instances need no per-instance human approval. This is the closest named ITSM precedent for the front gate: the human approves the change model/spec (intent), and conforming instances flow through automated authority; human judgment is reserved for high-risk or novel changes — which maps to the pre-production go/no-go for the irreversible vault. This is the accepted ITIL/DevOps reconciliation; auditors accept pipeline evidence + traceability in place of board minutes. *Source: ITIL 4 (Axelos) change enablement practice; itsm.tools analysis — <https://itsm.tools/change-enablement/>*
- **Systems-engineering gate reviews: design review up front, Flight Readiness Review + go/no-go poll at the irreversible commit** — NASA's lifecycle is the canonical two-anchor gate structure: intent is approved at design reviews (PDR/CDR), execution is verified by test rather than continuous oversight, and the irreversible action gets a Flight Readiness Review plus a final go/no-go poll where every station must answer against PRE-DECLARED criteria — and 'mandatory launch constraints cannot be waived after start of the terminal count'. Two transferable rules for the vault go/no-go: (1) write the no-go constraints down BEFORE the release exists (backup verified-restorable, migration has a contract-phase plan, integrity suite green on prod-shaped data), so the solo human can't rationalize past them at 11pm; (2) the poll's value is forcing an explicit affirmative per criterion, not an overall gut feel. *Source: NASA launch status check / Flight Readiness Review (NPD 8610.24C; APPEL 'Getting to Yes — the FRR') — <https://en.wikipedia.org/wiki/Launch_status_check>*
- **Regulated CD / segregation of duties via pipeline evidence** — In SOX/FDA/ISO-13485 contexts, the modern accepted pattern is that enforced peer review on merge plus automated pipeline gates (tests, scans, policy checks) with immutable audit trails achieves the INTENT of segregation of duties, replacing manual approval boards; regulators want traceability from requirement→change→verification→deploy, not meetings. Caveat that disagrees with the user's setup: every regulated formulation still assumes some review independent of the author. A solo human whose 'authors' are agents arguably satisfies independence at the spec and gate level (human is independent of the agent-authored code), but nothing independent examines the code itself — the audit-trail answer is to keep the spec, the gate assertions, and the go/no-go checklist as the durable evidence chain. *Source: Harness DevOps Academy on SOX compliance for software delivery; arXiv 2007.11295 (DevOps under ISO 13485, multivocal review) — <https://www.harness.io/harness-devops-academy/sox-compliance-for-software-delivery-explained>*
- **Google SRE launch readiness: LCE, launch checklist, PRR** — Google formalized go/no-go for launches with Launch Coordination Engineers and a curated launch checklist where 'every question's importance must be substantiated, ideally by a previous launch disaster', and adding a question eventually required VP approval to stop checklist bloat; Production Readiness Review gates whether a service is fit for production support at all. Transferable prescriptions: keep the go/no-go checklist short, every item traceable to a real or credible failure of THIS system (vault corruption, migration data loss, LLM-cost runaway), and prune it as aggressively as you add to it — a checklist that grows monotonically stops being read, which is the failure mode of solo-operator rituals. *Source: Google SRE Book, ch. 27 'Reliable Product Launches at Scale' + Appendix E launch checklist — <https://sre.google/sre-book/launch-checklist/>*
- **Expand/contract (parallel change) for stateful migrations — make the irreversible deploy recoverable** — For schema/data changes to a store you cannot rebuild: expand (add new structures alongside old), migrate (dual-write/backfill while both coexist), contract (remove old structures only in a LATER release, after the new path is proven in production). Combined with a verified pre-promotion backup, this converts the 'irreversible' production promotion into a checkpointed, revertible step — the new release can be rolled back during the expand/migrate window because the old structures still exist. Known trade-off: the contract phase gets forgotten and accretes debt (dual-written fields nobody removed), so schedule contraction explicitly. This is the single highest-leverage practice for the SQLite vault: never ship a destructive migration in the same release that introduces its replacement. *Source: Martin Fowler, 'Parallel Change' bliki (2014); Ambler & Sadalage, Refactoring Databases (2006); Sato/Fowler, Evolutionary Database Design — <https://martinfowler.com/bliki/ParallelChange.html>*
- **Release rings with dogfooding as ring 0; phased desktop rollouts** — Microsoft's ring model (canary → internal dogfood/selfhost → insiders → broad) and store-level phased releases (Apple: 1→2→5→10→20→50→100% over 7 days) institutionalize 'the makers absorb the first blast radius'. The user IS ring 0 — production-vault promotion is definitionally dogfooding — but the ring model's real lesson is that ring 0 is never the FIRST execution against realistic state: Microsoft's canary ring runs on real internal machines before dogfood. Solo analog: the restored-prod-backup pre-prod run is the missing 'ring −1'; optionally follow promotion with a soak window (new version live but risky/costly features flagged off, integrity checks scheduled) before declaring the release done. Trade-off: rings add calendar latency per release — acceptable here since a solo researcher's release cadence is not throughput-bound. *Source: Microsoft Learn, 'Create a deployment plan' (deployment rings); Windows Insider canary/dogfood history; Apple phased release — <https://learn.microsoft.com/en-us/windows/deployment/update/create-deployment-plan>*
