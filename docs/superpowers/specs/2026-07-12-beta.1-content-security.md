# Memoria 0.1.0-beta.1 — content-security spec

Date: 2026-07-12. Status: **derived spec** — folds the content-security /
egress layer the alpha.20 restart lost. Source: the archive gap analysis
(`2026-07-12-archive-gap-analysis.md`), which found this layer specified in
depth in the 2026-07-03 information-flow design + owner rulings but compressed
in the consolidation to one line (`prompt-injection-defense`) — while the
*live* requirements (`0.1.0-beta.1-requirements.md:359`) still assert an
"exfiltration linter" as **inherited-from-alpha.16 substrate** that `grep`
proves absent in alpha.20 code. Milestone: `0.1.0-alpha.21` for the substrate
pieces; the rest beta.1.

## Why this is its own package

The consolidation's X1 `prompt-injection-defense` is a *provenance-triggered
checkpoint on writes*. That is one control in a layer of eight. The layer has
one thesis: **untrusted text (source PDFs, enrichment metadata, foreign file
edits) must never gain authority over writes, egress, the human-owned region,
or the reader's browser** — and it fails *closed*. Three of these are asserted
as already-present substrate in a live spec but do not exist; two are
three-reader-consensus gaps; none has a home in the consolidation §2/§4. They
build together because they share the VaultWriter perimeter and the
render/export boundary.

Trust rule governing the whole package (from the 07-03 requirements charter,
Overreliance row): **a trust-incident defense may be credited only to a
vigilance-independent mechanism** — taint, the manifest gate, revert, the
export gate, these neutralizations — **never to the researcher noticing a
marker.** "The user will see the taint dot" is not a defense.

## CS1 · Exfiltration linter (VaultWriter-owned) — alpha.21

The single most-cited gap: a live requirement (`requirements.md:359`, "Inherited
From Alpha.16") with no implementation (`grep -rin exfiltrat src/**/*.py` → 0;
the integrity linter dir has `detectors.py`/`hub_handoff.py`/`session_summary.py`
but no link-neutralization linter).

- **Content-layer neutralization at apply AND export**, default-deny: in any
  machine-written region containing model- or third-party-derived text, escape
  Markdown image embeds (`![`), strip/escape raw HTML, and render external URLs
  as **non-clickable code spans**; leave only vault-internal wikilinks live.
- Runs inside `VaultWriter` (before the write lands) and again at export.
- **Canary drill** in the test suite: a seeded work title carrying an image
  embed must not survive apply or export as a live beacon.
- Distinct from Guard-brokered egress (network-layer, built) and X1
  prompt-injection (write-gate): this is *content-layer* neutralization of text
  that already passed those gates.

## CS2 · Renderer-side beacon neutralization — beta.1 (U3 plugin)

The Obsidian reading view is an unguarded egress surface: a poisoned title with
an image embed **beacons on open**. CS1 neutralizes at write; CS2 covers what
CS1 can't (already-stored content, third-party renderers): the plugin renders
machine-written third-party text with embeds/HTML inert and external URLs
non-clickable. Ships with U3.

## CS3 · Out-of-band change witness — alpha.21 (pairs F2)

Three-reader consensus (R-3 / S5.4 / info-flow). beta.1 is the first version
with *no* authorship-blind change witness (the alpha line carried one
continuously, alpha.1 `vault-hash-drift` → alpha.15 workspace scan). Two
fail-open holes:

- **Foreign edits to human-class files** by non-VaultWriter writers (the
  researcher's own file-tooled agent, a sync process) are silently trusted as
  human (absence-of-`mem-*` = human).
- **Removal of a restriction key** (`superseded`, `local-only`) fails open —
  silently re-admits a retracted work to Ask and past the export gate.

Fix: extend the session-start / periodic reconciliation into an integrity
sweep that (a) recomputes each manifest path's `human_hash` and diffs the vault
against the last run's commit to detect out-of-band human-file edits, surfacing
them as `observe-file-event` Review items; (b) explicitly witnesses
restriction-key removal; (c) runs graph-integrity verification between
consumption points. **Flag-and-route, never block** (consistent with the
derive-on-write model); human-content quarantine is deliberately not restored.
Detection is already paid for — the manifest hash is file-recomputable and each
run's pre-tag commit materializes the diff. Folds beside F2 `journal-chain-verifier`,
K3 `human-edit-boundary`, X1 `drift-detection-mutable-sources` (which targets
external *sources*, not vault human files).

## CS4 · Import resolution as a zero-LLM injection defense — beta.1 (O2)

Untrusted PDF/BibTeX text may only **nominate** an identifier; a **trusted
registry API authors** the metadata:

- Nominate via deterministic patterns only (Crossref DOI regex, arXiv id);
  never let document text author metadata.
- Acceptance requires the registry-returned title **fuzzy-match the PDF text
  ≥ 0.95** (Ratcliff/Obershelp).
- No identifier → Crossref `query.bibliographic` with **validation-scoring
  (SBMV), never rank-1 acceptance**.
- **GROBID disqualified** for authoring identity fields (8–16% silent title
  error poisons a work-keyed graph).

This is the security property behind the consolidation's bare "land-then-resolve";
it is a build constraint on the O2 pipeline that feeds F1 provenance integrity.

## CS5 · Enrichment metadata is an injection channel — beta.1 (X1)

Abstracts/titles/TLDRs are partly author-controlled (documented in-the-wild
attacks, e.g. arXiv:2507.06185). They cross into the digest prompt as **fenced
data only**, and carry a cheap **instruction-likeness flag** on suspicious
metadata. Extends the untrusted-input boundary beyond PDF body text to
enrichment-derived fields.

## CS6 · Anti-laundering on promote-to-note — beta.1 (W2)

`extract-to-note` is PI-gated (captured), but the gate alone does not prevent
laundering: one-tap promotion of machine sentences derived from adversarial
document text would move injectable prose into the human-owned region, where
downstream analysis then treats it as "what the researcher wrote." Fix: machine
pre-fill on a promoted stub stays machine-attributed (`suggested-*` keys +
fenced block) until the human edits/confirms each element; a promoted note with
**zero human-typed characters is flagged `unauthored`**.

## CS7 · `local-only` fail-closed egress/privacy marker — beta.1

A user-facing privacy control, distinct from untrusted-input defense and from
secret-management (K4, which covers API keys):

- Per-file/per-folder `local-only: true` frontmatter key + `local-only/` folder;
  **fail-closed** (unparseable frontmatter treated as local-only; transitive
  through marked ancestors).
- Excludes content from any LLM/provider egress during prompt assembly; **drafts
  default local-only**.
- Paired invariant: `bibliography.bib` is a keep-test survival artifact and is
  **never** subject to `local-only` (distinct from a per-export cited-only
  `references.bib`, which *is* local-only by inheritance).

## CS8 · Secret read-denylist for the sandbox — beta.2 (code-exec)

Deferred with the code-execution lane, but named here so its beta.2 spec carries
it: sandboxed runs must **deny read** on `~/.ssh`, `~/.aws`, the OS keychain,
and the vault DB — write-scoping alone still leaks secrets via read (Anthropic
sandbox-runtime finding). beta.2-scope §3 currently lists only write-side
hardening. → carried into `0.1.0-beta.2-scope.md` §3.

## Acceptance (content-security)

1. Canary: a seeded embed/HTML/external-URL payload in a machine-written region
   does not survive apply or export as live (CS1) and renders inert (CS2).
2. A foreign edit to a human-class file, and removal of a `superseded`/`local-only`
   key, each surface as a Review item within one reconciliation sweep (CS3).
3. Import of a PDF whose embedded text asserts a false DOI does not mint that
   DOI's metadata without the ≥0.95 title match (CS4).
4. A `local-only`-marked note never appears in any provider prompt payload; a
   draft is local-only by default; `bibliography.bib` is never excluded (CS7).
5. No safety acceptance check credits a defense to the human noticing a marker
   (trust rule).

## Sequencing

CS1 + CS3 are alpha.21 (substrate/trust repairs, beside F1/F2). CS4–CS7 are
beta.1 (ride O2/W2/X1 and the export path). CS2 rides U3. CS8 is beta.2.
